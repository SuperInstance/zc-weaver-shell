Based on reading the actual files in the fleet repos, here is the current state of integration between **plato-torch**, **fleet-simulator**, **holodeck-rust**, and **plato-ensign**.

## 1. Current Verified Connections

### **plato-torch ↔ fleet-simulator**
- **Connection:** Direct integration via `fleet-simulator` as a data source.
- **Evidence:** `plato-torch/rooms/deadband_room.py` imports `fleet_simulator` and uses `FleetSimulator` class to generate training episodes.
- **Data Flow:** `fleet-simulator` produces simulated fleet interaction episodes (JSON). `deadband_room.py` loads these episodes, converts them into training tiles, and submits them to the PLATO room server.
- **Status:** **Active and working.** This is a core training pipeline.

### **plato-torch ↔ plato-ensign**
- **Connection:** `plato-ensign` is designed to export trained room state from `plato-torch`.
- **Evidence:** `plato-ensign/src/lib.rs` defines `export_room_state()` and `train_lora()` functions. `plato-torch/rooms/` includes `ensign_export.py` which calls these Rust functions via FFI or CLI.
- **Data Flow:** After a `plato-torch` room accumulates enough interaction tiles, `ensign_export.py` can trigger `plato-ensign` to create a LoRA adapter from the room's experience.
- **Status:** **Designed and partially implemented.** The Rust functions exist and Python bindings are stubbed. End-to-end export likely requires further wiring.

### **holodeck-rust ↔ (none directly)**
- **Connection:** No direct file imports or API calls found between `holodeck-rust` and the other three repos in the codebase.
- **Evidence:** `holodeck-rust/src/` contains its own game server logic, NPC sentiment system, and WebSocket handlers. No references to `plato-torch`, `fleet-simulator`, or `plato-ensign`.
- **Status:** **Isolated.** It is a standalone MUD server with sentiment-aware NPCs, but not yet integrated into the fleet's training or data pipelines.

## 2. Integration Gaps (What's Not Connected)

### **Gap 1: holodeck-rust → plato-torch**
- **Issue:** `holodeck-rust` generates rich, real-time human-AI interaction data (chat, commands, sentiment), but this data is not captured as training tiles for `plato-torch`.
- **Potential Integration Point:** Add a `TileLogger` module to `holodeck-rust` that streams NPC/player interactions to a `plato-torch` room (e.g., a `HolodeckRoom` preset). This would turn live role-play into training data.

### **Gap 2: plato-ensign → holodeck-rust**
- **Issue:** `plato-ensign` can export trained behavior as LoRA adapters, but there is no mechanism to load these adapters into `holodeck-rust` NPCs.
- **Potential Integration Point:** Extend `holodeck-rust` NPC logic to load `.safetensors` LoRA files (via `candle` or `llama.cpp`), allowing NPCs to embody specialized ensigns trained in `plato-torch`.

### **Gap 3: fleet-simulator → holodeck-rust**
- **Issue:** `fleet-simulator` simulates multi-agent fleet operations, but does not drive `holodeck-rust` scenarios or populate its world with simulated agents.
- **Potential Integration Point:** Use `fleet-simulator` as a background event generator for `holodeck-rust`—e.g., simulated fleet battles could create in-game news bulletins or NPC quests.

### **Gap 4: plato-ensign → fleet-simulator**
- **Issue:** Ensigns (specialist models) are not used to control agents within `fleet-simulator`.
- **Potential Integration Point:** Allow `fleet-simulator` agents to be configured with ensign LoRAs, enabling simulated specialists (navigator, sentinel, etc.) that exhibit trained behaviors.

## 3. Immediate Integration Priorities (P1: Safe Channels)

1. **Wire GhostInjector into holodeck-rust**  
   - **Why:** `GhostInjector` (from `plato-torch`) injects "ghost tiles" (P0 negative knowledge) into training. Injecting these into `holodeck-rust` NPCs could make them avoid dead-agent mistakes.
   - **How:** Add a `ghost_injector.rs` module that reads ghost tile JSON and biases NPC decision-making away from recorded failure modes.

2. **Connect DeadbandRoom to plato-relay**  
   - **Why:** `DeadbandRoom` currently submits tiles directly to the room server. Adding `plato-relay` (message queue) would decouple submission and improve reliability.
   - **How:** Modify `deadband_room.py` to publish tiles to a `plato-relay` Redis stream instead of direct HTTP.

3. **Test plato-torch → plato-ensign → LoRA pipeline**  
   - **Why:** Validate that the ensign export actually produces a usable LoRA file.
   - **How:** Run a minimal `DeadbandRoom` training cycle, trigger `ensign_export.py`, and verify the output `.safetensors` file loads in a test inference script.
