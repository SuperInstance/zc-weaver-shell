## 1. Verified Current Connections (What's Connected)

Based on reading the actual code in the fleet repos:

### **plato-torch → plato-ensign** (Strong Connection)
- **plato-torch/rooms/DeadbandRoom.py** imports from `plato_ensign`:
  ```python
  from plato_ensign import export_room_experience, LoRAAdapter
  ```
- `export_room_experience()` is called at the end of training cycles.
- LoRA adapters are saved to `./ensigns/` directory.
- **Connection Type:** Direct Python import, functional pipeline.

### **fleet-simulator → plato-torch** (Data Flow Connection)
- **fleet-simulator/src/simulator.rs** generates training episodes (JSONL).
- Output files are written to `fleet-simulator/data/` (e.g., `episodes_20240419.jsonl`).
- **plato-torch/rooms/DeadbandRoom.py** has a `load_simulator_data()` method that reads these JSONL files.
- **Connection Type:** File-based data flow (simulator writes, room reads).

### **holodeck-rust → (No Direct Code Integration Found)**
- **holodeck-rust/src/main.rs** is a standalone MUD server.
- No imports of `plato-torch`, `fleet-simulator`, or `plato-ensign` in the Rust code.
- However, **fleet-knowledge/HOLODECK-CONNECTIONS.md** mentions:
  > "Ghost tiles from dead agents can be injected into holodeck NPCs."
- This suggests a *planned* integration, not yet implemented in code.

### **plato-ensign → (Export Only)**
- **plato-ensign/export.py** is a library called by `plato-torch`.
- It does not import other fleet components.
- **Connection Type:** Library dependency (one-way).

## 2. Integration Gaps (What's Not Connected)

### **Gap 1: holodeck-rust ↔ plato-torch (Ghost Tile Injection)**
- **Problem:** The concept of "ghost tiles" exists in fleet doctrine, but no code moves P0 negative-space lessons from plato-torch rooms into holodeck NPCs.
- **Evidence:** `HOLODECK-CONNECTIONS.md` describes the intent, but `holodeck-rust/src/npc.rs` has no ghost-tile loading logic.
- **Actionable Integration:** Build `GhostInjector` tool that reads `plato-torch/ghost_tiles/` and pushes sentiment patterns to holodeck NPCs.

### **Gap 2: fleet-simulator → holodeck-rust (Live Scenario Feed)**
- **Problem:** Simulator generates training episodes, but holodeck cannot use them for real-time scenario generation.
- **Evidence:** `fleet-simulator` outputs JSONL; `holodeck-rust` reads static map files (`maps/tavern.yaml`).
- **Actionable Integration:** Add a REST endpoint to `holodeck-rust` that accepts simulator episodes and spawns dynamic NPCs.

### **Gap 3: plato-ensign → fleet-simulator (Feedback Loop)**
- **Problem:** Ensigns (LoRA adapters) are exported but not used to improve the simulator's AI behavior.
- **Evidence:** `fleet-simulator/src/agent.rs` uses hard-coded behavior trees, not adaptable LoRAs.
- **Actionable Integration:** Modify simulator to load the latest ensign from `./ensigns/` and apply it to agent decision-making.

### **Gap 4: DeadbandRoom → plato-relay (Live Tile Streaming)**
- **Problem:** Tiles are saved locally but not streamed to the fleet's central relay for real-time training.
- **Evidence:** `plato-torch/rooms/DeadbandRoom.py` writes tiles to `./tiles/` directory only.
- **Actionable Integration:** Implement `submit_tile_to_relay()` method that POSTs tiles to `plato-relay` service (if service exists).

## 3. Priority Order (P0 → P1 → P2)

**P0 (Negative Space — What NOT to do):**
- Do not break existing connections (plato-torch → plato-ensign, fleet-simulator → plato-torch).
- Do not modify core room training logic without testing.

**P1 (Safe Channels — Where to work):**
- Gap 1 (GhostInjector) is safe: it adds new functionality without touching core systems.
- Gap 4 (Tile Streaming) is safe: adds optional network call, can fail gracefully.

**P2 (Optimize — Best path):**
- Start with Gap 1 (GhostInjector) because:
  1. It aligns with fleet doctrine (ghost tiles as P0 knowledge).
  2. It's a standalone tool, low risk.
  3. It demonstrates integration value quickly.
