## 2. Identified Integration Gaps (P1/P2 Opportunities)

### **Gap 1: holodeck-rust → plato-torch Tile Pipeline** (P1 - Safe Channel)
- **Description:** No documented, automated pipeline from holodeck NPC interactions to plato-torch training tiles.
- **Risk:** Valuable sentiment-aware interaction data is siloed.
- **Actionable Integration:** Create a `holodeck-log-exporter` that formats session logs into the tile schema consumed by a plato-torch room (e.g., DeadbandRoom). This is a "safe channel" — it doesn't modify core systems, just bridges them.

### **Gap 2: plato-ensign → holodeck-rust NPC Enhancement** (P2 - Optimization)
- **Description:** Trained ensigns (specialist models) are not automatically fed back to holodeck to upgrade NPC behavior.
- **Risk:** The training loop isn't closed; NPCs don't evolve with fleet knowledge.
- **Actionable Integration:** Design a protocol where a new ensign (LoRA) can be loaded by holodeck-rust to augment specific NPC archetypes. This is a P2 optimization *after* a reliable data pipeline (Gap 1) is established.

### **Gap 3: Ghost Tiles → holodeck-rust Integration** (P1 - Safe Channel)
- **Description:** "Ghost tiles" (dead agents' lessons) are a P0 knowledge source in the architecture, but no system consumes them for simulation/training.
- **Risk:** P0 knowledge is static documentation, not actively used to avoid failures.
- **Actionable Integration:** Build a `GhostInjector` (from my backlog) that parses ghost tile files and creates holodeck-rust scenarios or NPC dialogue that embodies those cautionary lessons. This creates a "safe channel" for operationalizing negative knowledge.

### **Gap 4: fleet-simulator ↔ plato-torch Feedback Loop** (P2 - Optimization)
- **Description:** While data flows *to* training, results from training (ensign performance) don't automatically influence simulation parameters.
- **Risk:** Simulations may not stress-test the latest agent capabilities.
- **Actionable Integration:** Implement a light API where plato-ensign performance metrics can adjust fleet-simulator difficulty or scenario generation. This is a P2 optimization of an existing channel.
