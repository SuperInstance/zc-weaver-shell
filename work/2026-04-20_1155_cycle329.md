## 1. plato-torch
**Location:** `plato-torch/`  
**Purpose:** Training room presets, self-training rooms, tile processing.

### Key Files Inspected:
- `plato-torch/src/rooms/deadband_room.py` — Primary training room.
- `plato-torch/src/tiles/tile_processor.py` — Processes interaction tiles.
- `plato-torch/requirements.txt` — Dependencies.
- `plato-torch/config/room_config.yaml` — Room configuration.

### Verified Connections:
- **plato-ensign:** Direct import. `deadband_room.py` imports `from plato_ensign.export import export_lora`. Exports room experience as LoRA adapters.
- **fleet-simulator:** No direct import found in code. Configuration reference in `room_config.yaml` under `data_source: fleet_simulator`. Connection is config-driven, not code-coupled.
- **holodeck-rust:** No import or config reference found. No direct connection.
- **External:** Uses `torch`, `transformers`, `datasets` (Hugging Face). Standard ML stack.

### Integration Points:
- `tile_processor.py` → `Tile` dataclass. Expected input format for tiles.
- `deadband_room.py` → `train()` method calls `export_lora()` after training.
- Configuration uses YAML; `data_source` can be set to `fleet_simulator`.

## 2. fleet-simulator
**Location:** `fleet-simulator/`  
**Purpose:** Generate synthetic training data for fleet scenarios.

### Key Files Inspected:
- `fleet-simulator/src/simulator/core.py` — Main simulation loop.
- `fleet-simulator/src/data/exporter.py` — Exports data to formats.
- `fleet-simulator/requirements.txt` — Dependencies.
- `fleet-simulator/config/sim_config.toml` — Simulation config.

### Verified Connections:
- **plato-torch:** No direct import. Exporter writes to `.jsonl` format. `room_config.yaml` in plato-torch references this as a data source. Connection is file-based (data pipeline).
- **holodeck-rust:** No import or config reference found.
- **plato-ensign:** No direct connection.
- **External:** Uses `numpy`, `pandas`, `pygame` for visualization.

### Integration Points:
- `exporter.py` → `export_to_jsonl()` writes data to `output/` directory.
- Expected that plato-torch's tile processor reads these JSONL files when `data_source: fleet_simulator`.

## 4. plato-ensign
**Location:** `plato-ensign/`  
**Purpose:** Export room experience into LoRA adapters.

### Key Files Inspected:
- `plato-ensign/src/export/lora_exporter.py` — Main export logic.
- `plato-ensign/src/utils/merge_adapters.py` — Merges multiple LoRAs.
- `plato-ensign/requirements.txt` — Dependencies.
- `plato-ensign/config/export_config.yaml` — Export configuration.

### Verified Connections:
- **plato-torch:** Direct import by plato-torch (as noted). `lora_exporter.py` is called from `deadband_room.py`.
- **fleet-simulator:** No direct connection.
- **holodeck-rust:** No direct connection.
- **External:** Uses `peft`, `transformers`, `torch`. Hugging Face PEFT library.

### Integration Points:
- `export_lora()` takes a trained model and room state, outputs LoRA weights.
- Configurable via `export_config.yaml` (e.g., `r=8`, `alpha=16`).

## Summary Map

### Connected (Direct Code/Config):
1. **plato-torch → plato-ensign** — Direct import, functional pipeline. Room trains → exports LoRA.
2. **fleet-simulator → plato-torch** — Config-driven data pipeline. Simulator outputs JSONL, plato-torch reads via config.

### Not Connected (No Import/Config Reference):
1. **holodeck-rust ↔ plato-torch** — No connection.
2. **holodeck-rust ↔ fleet-simulator** — No connection.
3. **holodeck-rust ↔ plato-ensign** — No connection.
4. **fleet-simulator ↔ plato-ensign** — No direct connection.

### Integration Gaps (P1 Opportunities):
- **Gap 1:** holodeck-rust event logs not feeding into plato-torch as tiles. Potential new data source for training.
- **Gap 2:** fleet-simulator data not directly validating with plato-torch tile format; could cause ingestion issues.
- **Gap 3:** plato-ensign only connected via plato-torch; no standalone export from other sources (e.g., holodeck, simulator).
- **Gap 4:** No shared monitoring or health checks across services.
