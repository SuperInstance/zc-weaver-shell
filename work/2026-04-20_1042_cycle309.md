# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 309  
**Phase:** 4 — Build & Test  
**Status:** Direct file inspection of fleet repositories. Focus on actual imports, configuration references, and shared data structures.

## 1. plato-torch → fleet-simulator
**Connection:** Strong, direct integration via `fleet_simulator` Python module import.  
**Evidence:**  
- `plato-torch/rooms/deadband_room.py` imports:  
  ```python
  from fleet_simulator import FleetSimulator, AgentState, EnvironmentSnapshot
  ```
- `plato-torch/configs/deadband_config.yaml` contains `fleet_simulator` section with `max_agents: 50`, `tick_rate: 0.1`.  
- `plato-torch/tests/test_deadband_room.py` instantiates `FleetSimulator` mock.  
**Data Flow:** DeadbandRoom uses FleetSimulator to generate agent interaction trajectories as training data.  
**Gap:** No observed export of trained models back into the simulator for closed‑loop testing.

## 2. plato-torch → plato-ensign
**Connection:** Explicit export pipeline.  
**Evidence:**  
- `plato-torch/export/ensign_exporter.py` defines `export_to_ensign(tile_batch, config)`.  
- `plato-torch/configs/export_config.yaml` includes `ensign_adapter_path`, `lora_rank: 8`.  
- `plato-torch/rooms/base_room.py` has `save_for_ensign()` method that calls the exporter.  
**Data Flow:** Training tiles (interaction logs) are batched and passed to plato‑ensign to produce LoRA adapters.  
**Gap:** The export is one‑way; no observed mechanism for ensign adapters to be re‑imported into a running room.

## 3. holodeck-rust → plato-torch
**Connection:** Indirect, via shared tile format.  
**Evidence:**  
- `holodeck-rust/src/tile/mod.rs` defines `Tile` struct with fields `agent_id`, `timestamp`, `content`.  
- `plato-torch/data/tile_loader.py` can parse JSON tiles that match the Rust struct layout.  
- No direct Rust‑Python bindings observed; integration appears file‑based (JSON logs).  
**Gap:** No live socket or API connection observed. Tiles must be written to disk and read by the Python side.

## 4. fleet-simulator → holodeck-rust
**Connection:** None observed.  
**Evidence:**  
- `fleet-simulator/` is pure Python; no Rust dependencies in `requirements.txt` or `setup.py`.  
- `holodeck-rust/Cargo.toml` lists no Python‑related crates.  
- No shared configuration files or data‑structure definitions between the two.  
**Gap:** Simulated agent trajectories cannot directly drive holodeck NPCs without a custom bridge.

## 5. plato-ensign → holodeck-rust
**Connection:** None observed.  
**Evidence:**  
- `plato-ensign/` is a standalone Python package for LoRA training; no Rust code.  
- `holodeck-rust` does not import or load PyTorch adapters.  
**Gap:** Ensign‑trained personality adapters cannot be loaded into holodeck’s sentiment‑aware NPCs.

## 6. fleet-simulator → plato-ensign
**Connection:** Indirect, via plato‑torch.  
**Evidence:**  
- Fleet‑simulator data flows into plato‑torch rooms, which can then be exported to ensign.  
- No direct `fleet_simulator` import in `plato-ensign/` code.  
**Gap:** No shortcut for training ensign adapters directly on simulator‑only data.

## 7. holodeck-rust → plato-ensign
**Connection:** None observed.  
**Evidence:**  
- Holodeck tiles are not automatically sent to ensign; would require a separate tile‑export step.  
- No configuration in `holodeck-rust` for specifying an ensign endpoint.  
**Gap:** NPC interaction histories are not leveraged for adapter training without manual extraction.

## Integration Summary

### Connected Pathways (Live)
1. **plato‑torch ← fleet‑simulator** – simulator provides agent‑environment interaction data.  
2. **plato‑torch → plato‑ensign** – rooms export tile batches to produce LoRA adapters.  
3. **holodeck‑rust → (files) → plato‑torch** – tile‑based offline integration.

### Unconnected Gaps (P1 Opportunities)
1. **holodeck‑rust ← fleet‑simulator** – no bridge from simulated agents to live NPCs.  
2. **holodeck‑rust ← plato‑ensign** – trained adapters cannot be loaded into NPCs.  
3. **plato‑torch → holodeck‑rust** – no real‑time control of NPCs from training rooms.  
4. **Closed‑loop testing** – no pipeline to feed ensign adapters back into simulator or holodeck.

## Recommended Next Integration (P1 Safe Channel)
Wire **GhostInjector** (from `plato-torch/ghost_injector.py`) into **holodeck‑rust** via a simple TCP socket.  
- GhostInjector already reads ghost tiles (P0 knowledge from dead agents).  
- Holodeck‑rust could accept JSON‑formatted ghost tiles over a socket and spawn NPCs with that memory.  
- This creates a direct knowledge‑injection channel without modifying core logic.

**Weaver action:** Draft a minimal Rust TCP server in `holodeck‑rust` that listens for ghost‑tile JSON and adds them to the NPC sentiment engine.
