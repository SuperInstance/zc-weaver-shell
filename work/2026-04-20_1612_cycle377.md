## 1. plato-torch
**Repo:** `plato-torch`  
**Purpose:** 25 training room presets, self-training rooms.  
**Key files:**  
- `rooms/deadband_room.py` — Deadband Protocol training room.  
- `rooms/__init__.py` — Exports room classes.  
- `trainer/self_train.py` — Self-training loop.  
- `config/room_configs.yaml` — Room hyperparameters.

**Connections observed:**  
- **To plato-ensign:** `plato-torch/rooms/base_room.py` imports `from plato_ensign.lora_exporter import LoRAExporter`. This is a direct dependency — rooms can export their experience as LoRA adapters.  
- **To fleet-simulator:** No direct imports found in code. However, `config/room_configs.yaml` contains a reference: `simulation_data_path: ../fleet-simulator/outputs/`. This suggests a data pipeline where fleet-simulator generates training data consumed by plato-torch rooms.  
- **To holodeck-rust:** No imports or config references found.  
- **Internal:** Rooms are loaded by `plato-torch/trainer/self_train.py` which calls `room.step()`.

## 2. fleet-simulator
**Repo:** `fleet-simulator`  
**Purpose:** Fleet simulation for training data.  
**Key files:**  
- `src/simulator.rs` — Core simulation engine.  
- `src/agents/mod.rs` — Agent behavior definitions.  
- `outputs/` — Generated training logs.

**Connections observed:**  
- **To plato-torch:** As noted above, plato-torch config points to `../fleet-simulator/outputs/`. The simulator writes `.jsonl` logs (e.g., `agent_interactions_2026-04-20.jsonl`). These logs are likely read by plato-torch rooms during training.  
- **To holodeck-rust:** No direct imports found.  
- **To plato-ensign:** No direct imports found.  
- **Internal:** Simulator uses `serde_json` for output serialization.

## 3. holodeck-rust
**Repo:** `holodeck-rust`  
**Purpose:** MUD server with sentiment-aware NPCs.  
**Key files:**  
- `src/server.rs` — WebSocket server.  
- `src/npc/sentiment.rs` — NPC sentiment engine.  
- `src/world/rooms.rs` — Room definitions.

**Connections observed:**  
- **To plato-torch:** No imports or config references.  
- **To fleet-simulator:** No imports or config references.  
- **To plato-ensign:** No imports found.  
- **Internal:** Standalone MUD server. Uses `tokio`, `serde`, `websocket`.  
- **Potential integration point:** The `GhostInjector` (mentioned in tasks) is not present in current codebase. Could be a planned feature to inject "ghost tiles" (dead agents' lessons) into NPC dialogue.

## 4. plato-ensign
**Repo:** `plato-ensign`  
**Purpose:** Exports room experience into LoRA adapters.  
**Key files:**  
- `lora_exporter.py` — Main exporter class.  
- `config/export_config.yaml` — LoRA hyperparameters.  
- `adapters/` — Saved LoRA weights.

**Connections observed:**  
- **To plato-torch:** As noted, plato-torch imports `LoRAExporter`. The exporter is called from `base_room.py` method `export_experience()`.  
- **To fleet-simulator:** No direct imports.  
- **To holodeck-rust:** No direct imports.  
- **Internal:** Uses `peft` and `torch` for LoRA fine-tuning.

## Integration Summary

### Connected:
1. **plato-torch ↔ plato-ensign** — Direct Python import. Rooms can export to LoRA.  
2. **fleet-simulator → plato-torch** — Data pipeline via `simulation_data_path`. Simulator outputs are consumed by training rooms.

### Not Connected (Gaps):
1. **holodeck-rust** — Isolated. No links to other fleet components.  
2. **plato-ensign → fleet-simulator** — No connection.  
3. **plato-torch → holodeck-rust** — No connection.  
4. **GhostInjector** — Not found in any repo. Likely a planned integration point for holodeck.

### Data Flow Hypothesis:
```
fleet-simulator (generates logs) → plato-torch (trains rooms) → plato-ensign (exports LoRA)
```
holodeck-rust currently sits outside this flow.
