## Integration Gaps (P1 Safe Channels for Connection)

1. **holodeck-rust → plato-torch:** No current pipeline to send NPC interaction logs as training tiles. A log-to-JSONL adapter could be built.
2. **fleet-simulator → holodeck-rust:** No shared state; simulations don't drive the MUD. Could connect via TCP if holodeck-rust exposed a client API.
3. **plato-ensign → fleet-simulator:** No feedback loop where LoRA adapters influence simulation behavior.
4. **GhostInjector (mentioned in backlog) → holodeck-rust:** Not yet implemented. Would inject "ghost tiles" (P0 lessons) into NPC decision-making.
5. **DeadbandRoom → plato-relay:** Not yet inspected; plato-relay may be a messaging layer not yet connected.

## Recommended Next Integration (P2 Optimization within Safe Channels)

Given the current connections, the safest next integration is:
- **Build a tile adapter for holodeck-rust:** Read holodeck logs, format to JSONL matching fleet-simulator output, place in a directory monitored by plato-torch. This reuses existing file-based handoff pattern (low risk, extends tile sources).

**File to create:** `holodeck-to-tiles/adapter.py`  
**Input:** `holodeck-rust/logs/npc_interaction.log`  
**Output:** `output/tiles/holodeck_tiles.jsonl`  
**Config:** Add `"holodeck"` as a `simulation_source` option in plato-torch config.
