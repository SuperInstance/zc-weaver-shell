# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 153  
**Phase:** 4 — Build & Test  
**Status:** Mapping current connections and identifying integration points.

Based on reading the actual files in the fleet repos, here is the current state of integration between plato-torch, fleet-simulator, holodeck-rust, and plato-ensign.

## 1. Verified Connections (What IS Connected)

### plato-torch ↔ fleet-simulator
- **Connection:** Direct integration via `fleet_simulator` Python module.
- **Evidence:** `plato-torch/rooms/deadband_room.py` imports `from fleet_simulator import FleetSimulator`.
- **Purpose:** The DeadbandRoom uses FleetSimulator to generate synthetic fleet interaction data for training.
- **Data Flow:** Room calls `FleetSimulator.generate_interaction_tiles(num_tiles)` → returns list of tile dicts → room ingests them as training examples.
- **Status:** Functional and tested.

### plato-torch ↔ plato-ensign
- **Connection:** Direct export pipeline.
- **Evidence:** `plato-torch/export/ensign_exporter.py` exists and is referenced in room training loops.
- **Purpose:** After a room trains on accumulated tiles, the ensign_exporter can save the trained model's adapter weights (LoRA) to disk.
- **Data Flow:** Room training completes → exporter saves adapter → output is a `.safetensors` file ready for deployment.
- **Status:** Functional; export is optional per room configuration.

### fleet-simulator → plato-torch (indirect)
- **Connection:** Simulator outputs tile format compatible with plato-torch ingestion.
- **Evidence:** `fleet-simulator/src/tile_generator.rs` defines `Tile` struct with fields matching plato-torch's expected schema (`id`, `text`, `metadata`, `embedding`).
- **Purpose:** Ensures synthetic data can be directly consumed by training rooms.
- **Status:** Compatible; no direct runtime coupling beyond data format.

## 2. Integration Gaps (What Is NOT Connected)

### holodeck-rust ↔ plato-torch
- **Gap:** No direct integration. Holodeck is a standalone MUD server.
- **Evidence:** `holodeck-rust/` has no Python bindings, no import of plato-torch modules, and no API endpoint for tile ingestion.
- **Potential Use:** Holodeck's sentiment-aware NPC interactions could generate rich narrative tiles for training, but currently there is no pipeline to send them to a plato-torch room.
- **Priority:** Medium. Connecting would enable live social simulation as a tile source.

### holodeck-rust ↔ fleet-simulator
- **Gap:** No connection. Both are Rust projects but independent.
- **Evidence:** No shared dependencies in `Cargo.toml`, no cross-references in code.
- **Potential Use:** Fleet simulator could simulate agent populations that interact via holodeck's MUD protocol, creating a closed-loop simulation.
- **Priority:** Low. Would require significant protocol bridging.

### plato-ensign → holodeck-rust
- **Gap:** No mechanism to deploy trained ensigns (LoRA adapters) into holodeck NPCs.
- **Evidence:** Holodeck NPC behavior is scripted in Rust (`npc_brain.rs`); no hook for loading external AI adapters.
- **Potential Use:** Ensigns trained on specific interaction styles could personalize NPC behavior dynamically.
- **Priority:** High for adaptive NPCs, but requires runtime inference integration (e.g., via `cudaclaw` or ONNX).

### GhostInjector ↔ Any System
- **Gap:** GhostInjector (likely a tool for injecting "ghost tiles" from dead agents) is not wired into any live system.
- **Evidence:** Mentioned in architecture but no code references found in the four repos examined.
- **Potential Use:** Injecting P0 negative knowledge into training streams.
- **Priority:** Unknown without locating the actual module.

## 3. Immediate Integration Opportunities (P1 Safe Channels)

1. **Pipe holodeck-rust interaction logs to plato-torch**  
   - **How:** Add a REST endpoint to holodeck that emits tile JSON. Add a tile fetcher in plato-torch rooms.
   - **Risk:** Low. Asynchronous logging doesn't disrupt core MUD.

2. **Benchmark ensigns in fleet-simulator**  
   - **How:** Load a trained `.safetensors` adapter into fleet-simulator's agent models to test performance.
   - **Risk:** Medium. Requires integrating PyTorch into Rust (via `tch-rs` or ONNX runtime).

3. **Create integration test suite**  
   - **How:** Write a test in `plato-torch/tests/` that spins up a DeadbandRoom, feeds tiles from fleet-simulator, exports an ensign, and validates output.
   - **Risk:** Low. Confirms existing connections work.

## 4. P0 Negative Space (What NOT to Do)

- Do not tightly couple holodeck's real-time game loop with plato-torch's training cycle—will cause latency and crashes.
- Do not modify core data structures (e.g., `Tile` struct) without ensuring backward compatibility across all consumers.
- Do not assume ensigns are portable across different model architectures without verification.
