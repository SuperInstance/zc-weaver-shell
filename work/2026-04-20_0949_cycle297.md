# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 297

**Status:** Direct file inspection of fleet repositories. Focus on actual imports, configuration references, and shared data structures.

## 1. plato-torch
**Repo:** `plato-torch`  
**Purpose:** Training room presets, self‑training rooms, tile processing.  
**Key files:** `DeadbandRoom.py`, `GhostInjector.py`, `plato_relay.py`, `tile_processor.py`.

### Verified connections:
- **To plato‑ensign:** Exports room experience via `export_to_ensign()` method in `DeadbandRoom.py`. Creates LoRA adapters from accumulated tiles.
- **To fleet‑simulator:** `DeadbandRoom.py` imports `from fleet_simulator import FleetSimulator` and uses it to generate training data. The simulator is instantiated as `self.simulator`.
- **To holodeck‑rust:** `GhostInjector.py` imports `holodeck_client` and calls `inject_ghost_tile()` to send ghost tiles into the MUD. Connection is configurable via `HOLODECK_HOST` and `HOLODECK_PORT`.
- **Internal:** `tile_processor.py` is used by `DeadbandRoom` to validate and store tiles. `plato_relay.py` handles communication with the PLATO server.

### Integration gaps:
- **GhostInjector → holodeck:** Currently only injects tiles; no bidirectional state sync. The MUD does not push sentiment‑aware NPC states back to the room.
- **plato‑relay:** Standalone module; not wired into the main training loop. Room tiles are submitted directly via HTTP, not through the relay.

## 2. fleet‑simulator
**Repo:** `fleet‑simulator`  
**Purpose:** Generate synthetic fleet interaction data for training.  
**Key files:** `simulator.py`, `scenarios/`.

### Verified connections:
- **From plato‑torch:** Imported and instantiated in `DeadbandRoom.py`. Provides `generate_scenario()` method that returns a dict of agent interactions.
- **To constraint‑theory‑core:** Not directly imported. The simulator generates geometric constraints as text descriptions, not as live snapping operations.
- **To cudaclaw / flux‑runtime:** No imports or references found.

### Integration gaps:
- **Output format:** Simulator returns Python dicts; not yet aligned with tile schema used by `tile_processor.py` (requires `agent_id`, `timestamp`, `content`, `metadata`).
- **Real‑time integration:** Runs on‑demand when room requests data; no continuous background simulation.

## 3. holodeck‑rust
**Repo:** `holodeck‑rust`  
**Purpose:** Multi‑user dungeon server with sentiment‑aware NPCs.  
**Key files:** `src/server.rs`, `src/npc/sentiment.rs`, `src/api.rs`.

### Verified connections:
- **From plato‑torch:** Receives ghost tiles via `POST /inject_ghost` endpoint (defined in `api.rs`). The tile is stored in the NPC memory system.
- **To plato‑ensign:** No direct export path. NPC memories stay within the MUD; no LoRA adapter creation.
- **To fleet‑simulator:** No reference.

### Integration gaps:
- **Bidirectional flow:** NPC sentiment states are not exposed to external rooms. The API only accepts injections, does not publish state changes.
- **Tile validation:** Injected tiles are stored but not validated against the PLATO server’s rules.

## 4. plato‑ensign
**Repo:** `plato‑ensign`  
**Purpose:** Export room experience into LoRA adapters.  
**Key files:** `ensign_exporter.py`, `lora_builder.py`.

### Verified connections:
- **From plato‑torch:** Called via `DeadbandRoom.export_to_ensign()`. Passes tile history and room config.
- **To external models:** Uses `peft` and `transformers` libraries to fine‑tune base models (e.g., `deepseek‑coder`). Output is a `.safetensors` adapter file.
- **To fleet‑simulator / holodeck‑rust:** No imports.

### Integration gaps:
- **Ghost tile incorporation:** The exporter uses room tile history; ghost tiles from holodeck are not automatically included.
- **Incremental updates:** Exporter runs on full tile history each time; no delta‑update mechanism.
