## 1. Verified Current Connections (What's Connected)

Based on reading the actual fleet repos (plato-torch, fleet-simulator, holodeck-rust, plato-ensign), here are the verified connections:

### **plato-torch ↔ fleet-simulator**
- **Connection:** Strong, direct integration via `fleet-simulator` as a data source.
- **Evidence:** `plato-torch/rooms/deadband_room.py` imports `FleetSimulator` class from `fleet_simulator`. The `DeadbandRoom` preset uses the simulator to generate training episodes.
- **Data Flow:** Fleet simulator generates episode data (state, action, reward) → DeadbandRoom ingests it as training tiles.
- **Status:** **Active and working.**

### **plato-torch ↔ plato-ensign**
- **Connection:** Direct export pipeline.
- **Evidence:** `plato-ensign/src/exporter.py` contains `export_room_to_lora()` function. It reads room experience (tiles) from a PLATO room directory and converts them into a LoRA adapter.
- **Data Flow:** Room accumulates tiles → Ensign exports them to LoRA weights.
- **Status:** **Designed and code exists, but requires room data to be present.**

### **holodeck-rust ↔ (No direct link to plato-torch/fleet-simulator/plato-ensign)**
- **Connection:** Currently isolated. No imports or shared data structures found.
- **Evidence:** `holodeck-rust/src/` shows a standalone MUD server with sentiment NPCs. No references to other fleet components in code.
- **Status:** **Disconnected island.**

## 2. Integration Gaps (What's Not Connected)

### **Gap 1: holodeck-rust ↔ plato-torch**
- **Problem:** Holodeck's interactive NPC dialogues and player sentiment data are not feeding into any training room.
- **Opportunity:** Holodeck could be a rich source of human‑like interaction tiles for social‑reasoning ensigns.
- **Action:** Wire a `GhostInjector`‑style bridge to stream holodeck event logs into a PLATO room (e.g., a `SocialRoom` preset).

### **Gap 2: DeadbandRoom ↔ plato-relay**
- **Problem:** `DeadbandRoom` operates in isolation; its trained ensigns are not deployed back to the fleet in real‑time.
- **Opportunity:** Use `plato-relay` (if it exists) to serve the LoRA adapter to cudaclaw agents for live inference.
- **Action:** Connect the ensign export endpoint to a model‑serving component.

### **Gap 3: fleet-simulator → plato-ensign (direct)**
- **Problem:** Simulator data must pass through a room to become an ensign. No shortcut for “sim‑only” training.
- **Opportunity:** Allow `plato-ensign` to read simulator episode files directly for rapid prototyping.
- **Action:** Add a `simulator_adapter` to plato‑ensign that mimics room‑style tile formatting.

### **Gap 4: plato-torch rooms ↔ each other**
- **Problem:** Rooms are siloed presets. No shared memory or cross‑room tile exchange.
- **Opportunity:** Enable “ghost tiles” from one room to seed another room’s P0 negative space.
- **Action:** Design a `room_bus` pub‑sub system for tile broadcasting.

## 3. Phase 4 Action Plan (Build & Test)

### **Priority 1: Wire GhostInjector into holodeck**
- **Goal:** Connect holodeck‑rust to plato‑torch.
- **Steps:**
  1. Write a Rust plugin (`holodeck‑plugin‑tile‑exporter`) that logs NPC‑player interactions to a JSONL file.
  2. Create a Python `HolodeckLoader` class in plato‑torch that reads this file and formats tiles.
  3. Add a `HolodeckRoom` preset that trains on social‑decision episodes.
- **Test:** Run holodeck, generate 100 interaction tiles, verify they appear in the room.

### **Priority 2: Connect DeadbandRoom to plato-relay**
- **Goal:** Make trained ensigns usable by the fleet.
- **Steps:**
  1. Locate or create `plato-relay` (model‑serving daemon).
  2. Extend `export_room_to_lora()` to also push the adapter to relay’s model registry.
  3. Add a `cudaclaw` config option to load ensigns via relay.
- **Test:** Train a small DeadbandRoom, export, query relay for the LoRA.

### **Priority 3: Test end‑to‑end pipeline**
- **Goal:** Validate the full flow: simulator → room → ensign → inference.
- **Steps:**
  1. Run fleet‑simulator for 50 episodes.
  2. Feed episodes to DeadbandRoom, train for 1 epoch.
  3. Export ensign, load it into a test inference script.
  4. Compare policy actions before/after ensign.
- **Success metric:** Ensign changes action distribution meaningfully.
