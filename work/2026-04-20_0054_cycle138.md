# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 138  
**Phase:** 4 — Build & Test  
**Status:** Mapping current connections and identifying integration points.

Based on reading the actual files in the fleet repos, here is the current integration state between plato-torch, fleet-simulator, holodeck-rust, and plato-ensign.

## 1. plato-torch → fleet-simulator
**Connection:** ✅ **Connected**  
**How:**  
- `plato-torch/src/rooms/deadband_room.py` imports `fleet_simulator` as a module.  
- The `DeadbandRoom` class uses `fleet_simulator.simulate_fleet()` to generate training data.  
- The simulator returns a list of `FleetStep` objects which are converted into training tiles.  
**Data Flow:**  
`fleet_simulator.simulate_fleet(config)` → `List[FleetStep]` → `DeadbandRoom._tiles_from_simulation()` → `List[TrainingTile]`  
**Integration Gap:** None — this is a working, direct dependency.

## 2. plato-torch → plato-ensign  
**Connection:** ✅ **Connected**  
**How:**  
- `plato-torch/src/export/ensign_exporter.py` provides the `export_room_to_ensign()` function.  
- It loads a trained room (`.pt` checkpoint), extracts LoRA weights, and packages them into a `.safetensors` file with metadata.  
- The exporter is called from the CLI (`plato-torch/scripts/export_ensign.py`).  
**Data Flow:**  
`Room checkpoint` → `ensign_exporter.extract_lora_weights()` → `LoRA weights + config` → `.safetensors` file  
**Integration Gap:** The export is optional and manual — no automatic trigger when a room reaches a certain performance threshold.

## 3. plato-torch → holodeck-rust  
**Connection:** ⚠️ **Partial — via shared tile format**  
**How:**  
- Both systems use the same `TrainingTile` JSON schema (defined in `plato-torch/src/data/training_tile.py`).  
- `holodeck-rust/src/npc/sentiment_engine.rs` can read tiles generated by plato-torch to influence NPC behavior.  
- No direct API calls or library imports between them.  
**Integration Gap:**  
- No live connection: tiles must be manually copied or shared via file system.  
- No bidirectional interaction: holodeck cannot send real‑time events back to plato‑torch for training.

## 4. fleet-simulator → holodeck-rust  
**Connection:** ❌ **Not connected**  
**How:**  
- `fleet-simulator` outputs `FleetStep` structs (Rust).  
- `holodeck-rust` expects `TrainingTile` JSON.  
- No adapter exists to convert simulation steps into holodeck‑readable tiles.  
**Integration Gap:** Missing converter: `FleetStep` → `TrainingTile`. This would allow simulated fleet interactions to directly populate holodeck scenarios.

## 5. holodeck-rust → plato-ensign  
**Connection:** ❌ **Not connected**  
**How:**  
- `plato-ensign` only exports from plato‑torch room checkpoints.  
- `holodeck-rust` has no mechanism to export its NPC sentiment models or interaction logs as LoRA adapters.  
**Integration Gap:** No pipeline to capture holodeck NPC “experience” into an ensign.

## 6. fleet-simulator → plato-ensign  
**Connection:** ❌ **Not connected**  
**How:**  
- The simulator generates training data, not trained models.  
- `plato-ensign` exports trained room weights, not raw simulation data.  
**Integration Gap:** No direct path — simulation data must first go through a plato‑torch room to be turned into an ensign.

## Priority Integration Gaps (P1 Channels)

1. **Live tile stream from holodeck to plato‑torch** — would allow real‑time training from NPC interactions.
2. **Simulation‑to‑holodeck adapter** — turn `FleetStep` into `TrainingTile` for automated scenario generation.
3. **Automatic ensign export trigger** — when a room’s loss falls below threshold, auto‑export an ensign.
