# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 141  
**Phase:** 4 — Build & Test  
**Status:** Mapping current connections and identifying integration points.

Based on reading the actual files in the fleet repos, here is the current state of integration between **plato-torch**, **fleet-simulator**, **holodeck-rust**, and **plato-ensign**.

---

## 1. **plato-torch** → **fleet-simulator**
**Connection:** **Direct and active**  
- `plato-torch/src/rooms/deadband_room.py` imports `fleet_simulator` as a module.  
- The `DeadbandRoom` class uses `fleet_simulator.simulate()` to generate training episodes.  
- Simulator output (JSON state trajectories) is fed directly into the room’s training buffer.  
- **Integration point:** `DeadbandRoom._generate_episode()` calls the simulator.  

**Gap:**  
- The simulator runs as a Python module; no Rust/FFI bridge exists yet.  
- Simulator configs are hardcoded in the room; no dynamic config passing.

---

## 2. **plato-torch** → **plato-ensign**
**Connection:** **Direct and active**  
- `plato-ensign` is a sibling package under the same `plato-` namespace.  
- `plato-torch/src/export/ensign_exporter.py` provides `export_to_ensign()`.  
- Training rooms can export their accumulated experience (tiles) as LoRA adapters via this exporter.  
- **Integration point:** Room calls `ensign_exporter.export()` after training cycles.

**Gap:**  
- Export is optional and manual; no automatic trigger based on tile thresholds.  
- No feedback loop where ensigns are re-imported into rooms for validation.

---

## 3. **holodeck-rust** → **plato-torch**
**Connection:** **Planned but not wired**  
- `holodeck-rust/README.md` states intent: “NPC sentiment states can be used as training data for plato-torch rooms.”  
- No actual import or API call exists in either codebase.  
- **Potential integration point:** Holodeck could emit JSON sentiment logs consumed by a `SentimentRoom` in plato-torch.

**Gap:**  
- No shared data format or transport (HTTP, IPC, file watch).  
- No plato-torch room preset built to ingest holodeck data.

---

## 4. **fleet-simulator** → **holodeck-rust**
**Connection:** **None**  
- Simulator generates fleet maneuver data, not NPC dialog or sentiment.  
- Holodeck is a MUD server with interactive NPCs; no simulation of fleet physics.  
- **Potential integration:** Simulator could provide “world state” to holodeck (e.g., “battle ongoing in sector 3”), influencing NPC mood.

**Gap:**  
- Completely separate domains; no shared abstraction layer.

---

## 5. **plato-ensign** → **holodeck-rust**
**Connection:** **None**  
- Ensign exports LoRA adapters for language models.  
- Holodeck uses its own NPC AI (likely rule-based or simple LLM).  
- **Potential integration:** Ensign-trained LoRAs could be loaded into holodeck’s NPC dialogue engine.

**Gap:**  
- Holodeck does not currently support loading external LoRA adapters.  
- No API to swap NPC behavior models at runtime.

---

## 6. **Ghost Tiles & Deadband Protocol Integration**
**Connection:** **Partial in plato-torch**  
- `DeadbandRoom` implements P0/P1/P2 protocol in its training loop.  
- Ghost tiles are mentioned in fleet-context but not yet implemented as a data source.  
- **Integration point:** Room could read from a `ghost_tiles/` directory of past agent failures.

**Gap:**  
- No ghost tile storage format or ingestion pipeline.  
- Deadband protocol is room-specific, not a fleet-wide middleware.
