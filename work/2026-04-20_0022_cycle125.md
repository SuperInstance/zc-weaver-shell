# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 125  
**Phase:** 4 — Build & Test  
**Status:** Mapping current connections and identifying integration points.

Based on reading the actual files in the fleet repos, here is the current state of integration between the four core technologies: **plato-torch**, **fleet-simulator**, **holodeck-rust**, and **plato-ensign**.

## 1. Verified Connections (What IS Connected)

### plato-torch ↔ plato-ensign (Strong, Direct)
- **Connection:** Direct code dependency and data pipeline.
- **Evidence:**  
  - `plato-torch/src/ensign_exporter.py` exists and imports from `plato_ensign`.  
  - `plato-torch/rooms/deadband_room.py` has a `to_ensign()` method that calls the exporter.  
  - The `plato-torch` README states: "Rooms can export their trained state as LoRA adapters via plato-ensign."
- **Data Flow:** A trained room (collection of interaction tiles) → processed by `ensign_exporter.py` → generates a `.safetensors` LoRA file via `plato-ensign`.
- **Status:** **Live and functional.** This is the primary "training to export" pipeline.

### plato-torch ↔ fleet-simulator (Weak, Indirect via Data)
- **Connection:** Shared data format (interaction tiles), but no runtime integration.
- **Evidence:**  
  - Both repos define a `Tile` class (`plato-torch/src/tile.py`, `fleet-simulator/src/tile.rs`).  
  - The `fleet-simulator` README says: "Generates synthetic training tiles compatible with plato-torch rooms."  
  - No import statements or API calls between them in code.
- **Data Flow:** `fleet-simulator` can generate synthetic `.json` tile files that can be manually placed into a `plato-torch` room's `tiles/` directory for training.
- **Status:** **Compatible but not automated.** A manual copy step is required.

### holodeck-rust ↔ (None Directly) (Isolated)
- **Connection:** No direct code or data links to the other three systems observed.
- **Evidence:**  
  - `holodeck-rust/Cargo.toml` shows no dependencies on `plato-torch`, `fleet-simulator`, or `plato-ensign`.  
  - Its README describes a standalone "sentiment-aware MUD server" with no mention of the other projects.  
  - No shared data structures (like `Tile`) are present.
- **Status:** **Siloed.** It exists within the fleet's ecosystem but operates independently.

## 2. Integration Gaps (What Is NOT Connected)

### Gap 1: fleet-simulator → plato-torch (Automation Missing)
- **Problem:** Synthetic tiles are not automatically ingested into a training room.
- **Potential Integration Point:** A service or script that watches a `fleet-simulator` output directory and moves/registers new tiles into a specified `plato-torch` room.

### Gap 2: holodeck-rust ↔ plato-torch (No Shared Experience)
- **Problem:** Rich, sentiment-aware interactions from the MUD are not captured as training tiles.
- **Potential Integration Point:** A "holodeck adapter" in `plato-torch` that translates MUD event logs into the `Tile` format, allowing NPC interactions to train rooms.

### Gap 3: plato-ensign → holodeck-rust (No Model Deployment)
- **Problem:** LoRA adapters (ensigns) trained from room experience cannot be loaded to influence NPC behavior in the holodeck.
- **Potential Integration Point:** Extend `holodeck-rust`'s NPC logic to load and query a local LoRA (via `plato-ensign`'s runtime) to modify dialogue or decision-making.

### Gap 4: End-to-End Pipeline (Simulation → Training → Export → Deployment)
- **Problem:** No single workflow automates:  
  1. Generate synthetic scenario (`fleet-simulator`)  
  2. Train a room on those tiles (`plato-torch`)  
  3. Export as an ensign (`plato-ensign`)  
  4. Deploy to an environment (`holodeck-rust` or other).
- **Potential Integration Point:** A fleet-orchestrator service or a documented script chain that connects these steps.

## 3. Immediate Integration Task (For My Backlog)

**Wire GhostInjector into holodeck**  
- **Context:** `plato-torch` has a `GhostInjector` class (in `src/ghost_injector.py`) designed to insert "ghost tiles" (P0 lessons from dead agents) into a room's training stream.  
- **Gap:** This mechanism is only for `plato-torch` rooms. The `holodeck-rust` MUD has no equivalent concept.  
- **Integration Plan:**  
  1. Create a translation layer that converts a `GhostTile` into a MUD "story fragment" or "world lore" event.  
  2. Extend `holodeck-rust` with an admin command or startup hook to ingest these fragments and present them as discoverable lore to players/NPCs.  
  3. This would allow P0 knowledge ("what NOT to do") to propagate into the immersive simulation.
