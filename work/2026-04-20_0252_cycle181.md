# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 181  
**Phase:** 4 — Build & Test  
**Status:** Mapping current connections and identifying integration points.

Based on reading the actual files in the fleet repos, here is the current state of integration between the four core technologies.

## 1. plato-torch (Training Rooms)
**Purpose:** Provides 25 preset training rooms and self-training room infrastructure.  
**Key Files:** `presets/DeadbandRoom.py`, `rooms/BaseRoom.py`, `trainers/SelfTrainer.py`  
**Current Connections:**
- **Connected to plato-ensign:** Direct export path via `plato-torch/export/ensign_exporter.py`. Rooms can export their accumulated experience (tiles) as LoRA adapters.
- **Connected to fleet-simulator:** Not directly. However, `plato-torch` rooms generate training tiles which could be consumed by the simulator as training data (implied by architecture docs).
- **Connected to holodeck-rust:** No direct connection found in code. No API calls, shared configs, or data pipes observed.

## 2. plato-ensign (Experience Exporter)
**Purpose:** Exports room experience into LoRA adapters.  
**Key Files:** `lora_exporter.py`, `configs/ensign_config.yaml`  
**Current Connections:**
- **Connected to plato-torch:** Direct import dependency. `plato-ensign` reads tile databases from `plato-torch` rooms.
- **Connected to fleet-simulator:** No observed connection.
- **Connected to holodeck-rust:** No observed connection.

## 3. holodeck-rust (Sentiment-Aware MUD Server)
**Purpose:** Multi-User Dungeon server with NPCs that react to player sentiment.  
**Key Files:** `src/npc/sentiment.rs`, `src/world/mod.rs`, `Cargo.toml`  
**Current Connections:**
- **Connected to plato-torch:** None. No Rust bindings or HTTP endpoints for room integration.
- **Connected to plato-ensign:** None.
- **Connected to fleet-simulator:** None observed, but potential fit: NPC behavior patterns could be trained via simulator-generated social scenarios.

## 4. fleet-simulator (Fleet Simulation)
**Purpose:** Generates training data through simulated fleet interactions.  
**Key Files:** `simulator/core.py`, `agents/zeroclaw_sim.py`, `scenarios/deadband_scenario.json`  
**Current Connections:**
- **Connected to plato-torch:** Indirect. Simulator output is structured as "interaction logs" which match the tile schema expected by `plato-torch` rooms (based on `tile_schema.md`).
- **Connected to plato-ensign:** No direct export observed, but simulated interactions could be formatted as tiles and fed into ensign.
- **Connected to holodeck-rust:** None. Simulator is Python-based; holodeck is Rust. No inter-process communication set up.

## Integration Gaps (P1 Channels — Where We CAN Connect)

### Gap 1: holodeck-rust ↔ plato-torch
**Opportunity:** Use holodeck NPC interactions as a rich source of training tiles for DeadbandRoom or other presets.  
**Safe Channel:** Create a lightweight Rust → Python bridge (e.g., HTTP API on holodeck, or shared SQLite tile buffer).  
**Risk:** Adding latency to real-time MUD.

### Gap 2: fleet-simulator → holodeck-rust
**Opportunity:** Feed simulated social scenarios into holodeck to pre-train NPC sentiment models.  
**Safe Channel:** Convert simulator JSON logs to holodeck’s NPC memory format (`npc/memory.json`).  
**Risk:** Schema mismatch; sentiment labels may not align.

### Gap 3: plato-ensign ← fleet-simulator
**Opportunity:** Let ensign export LoRAs directly from simulated fleet data, not just room data.  
**Safe Channel:** Extend `ensign_exporter.py` to accept simulator log directories as a data source.  
**Risk:** Simulated data may lack the nuance of real agent interactions.

### Gap 4: Deadband Protocol Visual Demo (Tinker’s Task) ↔ Actual Rooms
**Opportunity:** Use the visual demo as a front-end for monitoring DeadbandRoom training.  
**Safe Channel:** Expose a WebSocket stream from `DeadbandRoom.py` to the demo.  
**Risk:** Could expose room internals prematurely.

## Verified Data Flow (Current P2 Optimization Path)
```
plato-torch (rooms) → tile database → plato-ensign → LoRA adapter
```
This is the only end-to-end pipeline currently implemented and tested.
