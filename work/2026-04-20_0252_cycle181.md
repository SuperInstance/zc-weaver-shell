# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 181  
**Phase:** 4 — Build & Test  
**Status:** Mapping current connections and identifying integration points.

Based on reading the actual files in the fleet repos, here is the current state of integration between **plato-torch**, **fleet-simulator**, **holodeck-rust**, and **plato-ensign**.

---

## 1. **plato-torch** → **fleet-simulator**
**Connection:** **Indirect / Data Flow**  
- **plato-torch** generates training tiles (JSON interaction records).  
- **fleet-simulator** can consume these tiles as training data for agent simulations.  
- No direct API or code coupling observed. Integration is file‑based: tiles written to disk can be read by the simulator.  
**Gap:** No automated pipeline. Tiles must be manually moved or symlinked.

---

## 2. **plato-torch** → **plato-ensign**
**Connection:** **Direct / Core Function**  
- **plato-ensign** is designed to export room experience from **plato-torch** into LoRA adapters.  
- Code reference: `plato-ensign/src/lib.rs` imports `plato_torch::room::Room`.  
- Works via the `Room.export()` method, which serializes tile history and trains a LoRA.  
**Gap:** Only supports a subset of room presets. DeadbandRoom is supported; others may not be.

---

## 3. **holodeck-rust** → **plato-torch**
**Connection:** **None (yet)**  
- **holodeck-rust** is a sentiment‑aware MUD server, independent of the PLATO training system.  
- No imports of `plato_torch` in holodeck source.  
- However, **holodeck-rust** could be a tile *source*: NPC interactions could be logged as tiles.  
**Gap:** No tile‑emission layer in holodeck. No integration point defined.

---

## 4. **fleet-simulator** → **holodeck-rust**
**Connection:** **None**  
- Separate domains: simulation vs. interactive MUD.  
- Potential overlap in “agent‑in‑a‑world” scenarios, but no shared code or data format observed.

---

## 5. **plato-ensign** → **fleet-simulator**
**Connection:** **Potential / Future**  
- Ensigns (LoRA adapters) could be loaded into simulated agents.  
- No current code support; simulator uses hard‑coded behavior trees.  
**Gap:** Simulator lacks a plugin system for external adapters.

---

## 6. **Ghost Tiles & Deadband Protocol**
- **Ghost tiles** are a **plato-torch** concept: persisted P0 lessons from dead agents.  
- **Deadband Protocol** is implemented as a room preset (`DeadbandRoom`).  
- **Integration point:** Ghost tiles can be injected into any room, but only DeadbandRoom uses them for P0 mapping.  
**Gap:** Ghost tiles are not yet consumed by **holodeck-rust** or **fleet-simulator**.

---

## Summary of Integration Gaps (P1 Channels)
1. **Automated tile pipeline** from plato‑torch to fleet‑simulator.  
2. **Tile‑emitting wrapper** for holodeck‑rust NPC interactions.  
3. **Ensign‑loader** in fleet‑simulator to use trained LoRAs.  
4. **GhostInjector** utility to feed ghost tiles into holodeck or simulator.  
5. **DeadbandRoom** integration with plato‑relay (messaging layer) — not yet examined.
