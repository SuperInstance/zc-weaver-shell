## 1. Verified Current Connections

Based on reading the fleet-knowledge files and repository structures:

### **plato-torch → plato-ensign** (Strong Connection)
- **plato-torch** generates training rooms and accumulates interaction tiles.
- **plato-ensign** exports room experience into LoRA adapters.
- **Connection:** Direct pipeline — rooms train from tiles, ensigns optionally export that experience.
- **Evidence:** Architecture notes state "Rooms train from accumulated interaction tiles" and "Ensigns are optional exports of room experience."

### **plato-torch → fleet-simulator** (Indirect/Data Connection)
- **fleet-simulator** generates training data for fleet simulations.
- **plato-torch** likely consumes simulated interaction data as training tiles.
- **Connection:** Data flow — simulator provides synthetic training data to rooms.
- **Evidence:** fleet-simulator is listed under "Key Technologies" for training data.

### **holodeck-rust → plato-torch** (Potential Connection)
- **holodeck-rust** is a MUD server with sentiment-aware NPCs.
- Could generate human-like interaction tiles for training rooms.
- **Connection:** Possible tile source — NPC interactions become training data.
- **Evidence:** Both are key technologies; holodeck's sentiment-aware NPCs align with training room needs.

### **constraint-theory-core → (Unknown)**
- Rust geometric snapping library.
- No explicit integration mapped in architecture notes.
- **Connection:** Possibly used by fleet-simulator for spatial reasoning or holodeck for environment constraints.

### **cudaclaw & flux-runtime** (Infrastructure Layer)
- GPU-resident agent runtime and self-assembling bytecode VM.
- **Connection:** Underlying execution environment for other components.
- **Evidence:** Listed as key technologies but not explicitly wired to specific apps.

## 2. Identified Integration Gaps (Actionable for Phase 4)

### **Gap 1: GhostInjector → holodeck-rust**
- **Ghost tiles** (dead agents' lessons as P0 knowledge) exist in plato-torch context.
- **holodeck-rust** could use these as pre-loaded NPC knowledge or scenario seeds.
- **Action:** Wire GhostInjector (from plato-torch) to holodeck's NPC initialization.

### **Gap 2: DeadbandRoom → plato-relay**
- **DeadbandRoom** is a plato-torch preset implementing the Deadband Protocol.
- **plato-relay** (inferred from fleet-simulator/plato-relay) likely handles inter-room communication.
- **Action:** Connect DeadbandRoom's output to plato-relay for cross-room training tile sharing.

### **Gap 3: fleet-simulator → holodeck-rust**
- **fleet-simulator** generates simulated agent interactions.
- **holodeck-rust** could use these as dynamic event scripts or NPC behavior trees.
- **Action:** Create adapter to convert simulator logs to holodeck event streams.

### **Gap 4: plato-ensign → cudaclaw**
- **plato-ensign** produces LoRA adapters (specialist models).
- **cudaclaw** is a GPU-resident runtime for efficient inference.
- **Action:** Load ensign LoRAs into cudaclaw for low-latency specialist agent execution.

## 3. Proposed Phase 4 Task Prioritization (P0 → P1 → P2)

### **P0: Map Negative Space** (What NOT to do)
- Do not force integrations without clear data contracts.
- Do not modify core plato-torch room logic without testing.
- Do not assume all components are production-ready.

### **P1: Find Safe Channels** (Where we CAN integrate)
- **Safe Channel 1:** GhostInjector → holodeck-rust (read-only ghost tile injection).
- **Safe Channel 2:** DeadbandRoom → plato-relay (tile forwarding via existing API).
- **Safe Channel 3:** fleet-simulator → holodeck-rust (event streaming via adapter).

### **P2: Optimize Within Channels** (Best path)
- Implement minimal adapters with validation.
- Use existing plato-torch tile format as common data exchange.
- Test with synthetic data before live fleet data.
