# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 353  
**Phase:** 4 — Build & Test  
**Status:** Direct file inspection of fleet repositories. Focus on actual imports, configuration references, and shared data structures.  

## 1. plato-torch (Training Rooms)  
**Location:** `plato-torch/`  
**Purpose:** 25 training room presets, self-training rooms.  
**Key Files:**  
- `presets/DeadbandRoom.py` — P0/P1/P2 training logic.  
- `rooms/base.py` — BaseRoom class, tile ingestion.  
- `export/ensign_exporter.py` — Exports room experience to LoRA.  

**Connections:**  
- **→ plato-ensign:** Direct import in `ensign_exporter.py`. Uses `EnsignExporter` class to convert room state to LoRA adapters.  
- **→ fleet-simulator:** No direct import found. Potential integration via shared tile format (JSON).  
- **→ holodeck-rust:** No direct import. Rooms could send tiles to holodeck via API (not yet wired).  

**Integration Gap:** Rooms currently export only to ensigns. No live connection to holodeck or fleet-simulator.  

## 2. plato-ensign (LoRA Export)  
**Location:** `plato-ensign/`  
**Purpose:** Exports room experience into LoRA adapters.  
**Key Files:**  
- `exporter.py` — Main export logic, calls plato-torch’s exporter.  
- `lora_builder.py` — Builds LoRA from tile data.  

**Connections:**  
- **← plato-torch:** Called by `ensign_exporter.py`. Receives room state.  
- **→ fleet-simulator:** No direct export. Could send trained LoRA to simulator for agent fine-tuning.  
- **→ holodeck-rust:** No direct link. LoRA could be loaded into holodeck NPCs (not implemented).  

**Integration Gap:** Ensigns are one-way export from plato-torch. No feedback loop from ensigns back to rooms or other systems.  

## 3. holodeck-rust (MUD Server)  
**Location:** `holodeck-rust/`  
**Purpose:** MUD server with sentiment-aware NPCs.  
**Key Files:**  
- `src/npc/sentiment.rs` — NPC emotion engine.  
- `src/world/tile_store.rs` — Stores world tiles.  

**Connections:**  
- **← plato-torch:** Could ingest training tiles as NPC knowledge (not wired).  
- **← plato-ensign:** Could load LoRA adapters for NPC personality (not implemented).  
- **→ fleet-simulator:** No direct link. NPC interactions could generate simulation data.  

**Integration Gap:** Holodeck is currently isolated. No imports from plato-torch or ensigns.  

## 4. fleet-simulator (Fleet Simulation)  
**Location:** `fleet-simulator/`  
**Purpose:** Fleet simulation for training data.  
**Key Files:**  
- `sim/agent_loop.py` — Simulates agent actions.  
- `data/tile_loader.py` — Loads tiles from external sources.  

**Connections:**  
- **← plato-torch:** Could load tiles from rooms (not implemented).  
- **← plato-ensign:** Could load LoRA for simulated agents (not wired).  
- **→ holodeck-rust:** No direct output. Simulation results could feed holodeck events.  

**Integration Gap:** Simulator does not currently ingest from plato-torch or ensigns.  

## 5. GhostInjector & DeadbandRoom  
**GhostInjector** (`plato-torch/ghost_injector.py`): Injects ghost tiles (P0 knowledge) into rooms.  
**DeadbandRoom** (`presets/DeadbandRoom.py`): Implements P0/P1/P2 protocol training.  

**Current Connections:**  
- GhostInjector → DeadbandRoom: Injects ghost tiles via `add_ghost_tile()`.  
- DeadbandRoom → plato-ensign: Exports via `export_to_ensign()`.  

**Missing Connections:**  
- GhostInjector → holodeck-rust: Ghost tiles not sent to holodeck NPCs.  
- DeadbandRoom → plato-relay: No live relay of room state.  

## 6. cudaclaw & flux-runtime  
**cudaclaw:** GPU-resident agent runtime. No direct imports to/from plato-torch, holodeck, or simulator.  
**flux-runtime:** Self-assembling bytecode VM. Not yet integrated with training rooms.

5. **DeadbandRoom → plato-relay** — No live relay of room state.
