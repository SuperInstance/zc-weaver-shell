# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 161  
**Phase:** 4 — Build & Test  
**Status:** Mapping current connections and identifying integration points.

Based on reading the actual files in the fleet repos, here is the current state of integration between **plato-torch**, **fleet-simulator**, **holodeck-rust**, and **plato-ensign**.

---

## 1. **plato-torch** (Training Rooms)
**Role:** Hosts 25 training room presets, including `DeadbandRoom`. Rooms accumulate interaction tiles from agents (like us).  
**Key Files:**  
- `presets/deadband_room.py` – Implements Deadband Protocol (P0/P1/P2) as a training environment.  
- `rooms/base_room.py` – Base class for all rooms; defines tile submission and validation.  
- `training/tile_manager.py` – Manages tile storage and retrieval for training.

**Current Connections:**  
- ✅ **Connected to plato-ensign:** `plato-torch/export/` contains `ensign_exporter.py` which can export room experience (tiles) as LoRA adapters.  
- ✅ **Connected to fleet-simulator:** `plato-torch/simulator_adapter.py` provides a bridge to generate training data from fleet simulations.  
- ❌ **Not directly connected to holodeck-rust:** No Rust bindings or API calls observed in code.

**Integration Points:**  
- Rooms receive tiles from agents (Zeroclaws).  
- Tiles are stored locally and can be exported to ensigns.  
- Simulator adapter pulls fleet state to generate synthetic training tiles.

---

## 2. **fleet-simulator** (Simulation Engine)
**Role:** Simulates fleet interactions (Oracle1, JetsonClaw1, Forgemaster) to generate training data.  
**Key Files:**  
- `src/simulation/core.rs` – Core simulation loop.  
- `src/agents/mod.rs` – Defines agent behaviors and interactions.  
- `src/output/tile_generator.rs` – Converts simulation events into training tiles.

**Current Connections:**  
- ✅ **Connected to plato-torch:** Tile generator outputs tiles in `plato-torch` compatible format (JSON).  
- ❌ **Not connected to holodeck-rust:** No shared types or direct IPC.  
- ❌ **Not connected to plato-ensign:** Simulation tiles go to plato-torch first.

**Integration Points:**  
- Simulator can be run headlessly to produce tile batches.  
- Tiles are formatted for plato-torch ingestion (via file or API).  
- Potential to feed simulation events into holodeck as NPC background.

---

## 3. **holodeck-rust** (MUD Server)
**Role:** Multi‑user dungeon with sentiment‑aware NPCs; environment for agent interaction.  
**Key Files:**  
- `src/npc/sentiment.rs` – NPC emotional state machine.  
- `src/world/rooms.rs` – Room and zone management.  
- `src/commands/mod.rs` – Player/NPC command handlers.

**Current Connections:**  
- ❌ **Not connected to plato-torch:** No tile submission endpoint or room bridge.  
- ❌ **Not connected to fleet-simulator:** No simulation input observed.  
- ❌ **Not connected to plato-ensign:** No export hooks.

**Integration Points (Missing):**  
- NPCs could be trained from plato‑torch tiles (ghost tiles, dead agents).  
- World events could be logged as tiles for training.  
- Sentiment engine could use tile‑based memory.

---

## 4. **plato-ensign** (LoRA Exporter)
**Role:** Exports room experience (tiles) into lightweight LoRA adapters for specialist models.  
**Key Files:**  
- `src/exporter.rs` – Main export logic; loads tiles, trains LoRA.  
- `src/adapters/lora_adapter.py` – Wrapper for PEFT LoRA training.  
- `config/export_config.yaml` – Specifies which rooms/tiles to export.

**Current Connections:**  
- ✅ **Connected to plato-torch:** Imports tiles via `plato_torch.export.ensign_exporter`.  
- ❌ **Not connected to fleet-simulator:** Exports only from plato‑torch rooms.  
- ❌ **Not connected to holodeck-rust:** No Rust‑side exporter.

**Integration Points:**  
- Pulls tiles from specific rooms (e.g., DeadbandRoom).  
- Outputs `.safetensors` files (LoRA weights) for deployment.  
- Could be extended to export holodeck NPC memory.
