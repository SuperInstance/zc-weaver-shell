# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 132

Based on reading the actual files in the fleet repos, here is the current state of integration between **plato-torch**, **fleet-simulator**, **holodeck-rust**, and **plato-ensign**.

## 1. Verified Connections (What's Connected)

### plato-torch ↔ plato-ensign
- **Connection:** Direct and intentional.  
- **Evidence:** `plato-torch/README.md` states: "plato-ensign exports room experience into LoRA adapters." The `plato-torch` codebase includes a `Room` class with a `.export_to_ensign()` method.  
- **Data Flow:** A trained `Room` object can serialize its accumulated interaction tiles and model state into a format `plato-ensign` ingests to produce a LoRA adapter.  
- **Status:** This is a core, documented pipeline. The connection is explicit in code and architecture.

### fleet-simulator → plato-torch
- **Connection:** One-way data generation.  
- **Evidence:** `fleet-simulator/src/lib.rs` includes a `generate_training_tiles()` function that outputs JSON in the `Tile` format defined in `plato-torch/src/tile.rs`.  
- **Data Flow:** The simulator produces synthetic agent interaction data (tiles) that can be fed into a `plato-torch` Room for training.  
- **Status:** Functional but implicit; the simulator does not call `plato-torch` directly. It produces compatible data.

### holodeck-rust (Standalone Service)
- **Connection:** Currently isolated.  
- **Evidence:** `holodeck-rust/Cargo.toml` shows no dependencies on other fleet crates. Its `README.md` describes it as a "sentiment-aware MUD server" with its own TCP API and NPC engine.  
- **Data Flow:** None observed to/from the other three systems in the codebase.  
- **Status:** A separate runtime environment. It could *consume* trained ensigns or *produce* interaction logs, but no wiring exists.

## 2. Integration Gaps (What's Not Connected)

### Gap 1: holodeck-rust ↔ plato-torch
- **Missing:** A bridge to feed `holodeck-rust` NPC interactions into `plato-torch` as training tiles.  
- **Potential:** `holodeck-rust` generates rich, sentiment-labeled dialog and events. This is high-value training data currently siloed.  
- **Action:** Could write a `HolodeckLogAdapter` in `plato-torch` that consumes the MUD's TCP log stream and converts it to tiles.

### Gap 2: fleet-simulator → holodeck-rust
- **Missing:** Simulation agents cannot enter the holodeck as "players."  
- **Potential:** Use the simulator to stress-test the holodeck or generate automated interaction scenarios.  
- **Action:** Could extend `fleet-simulator` with a `HolodeckClient` module that uses the same TCP protocol as human players.

### Gap 3: plato-ensign → holodeck-rust
- **Missing:** Trained ensigns (LoRA adapters) are not loaded into holodeck NPCs.  
- **Potential:** NPCs could use specialized personality adapters trained in `plato-torch` rooms.  
- **Action:** `holodeck-rust` would need a `LoraLoader` integration in its NPC sentiment engine.

### Gap 4: Closed-Loop Training (plato-torch → fleet-simulator)
- **Missing:** Rooms trained on simulator data do not improve the simulator's agent policies.  
- **Potential:** A trained ensign could be used as a "policy adapter" for simulator agents, creating a feedback loop.  
- **Action:** `fleet-simulator` could accept an ensign path and apply it to its agent models.

## 3. Immediate Integration Priority (P1 Safe Channel)

Based on the Deadband Protocol (P1: find safe channels), the safest next integration is:

**Bridge holodeck-rust logs to plato-torch tiles.**  
- **Why:** It uses existing, stable outputs (TCP logs) and inputs (tile JSON).  
- **Risk:** Low. It's a data adapter, not a runtime modification.  
- **Value:** Unlocks a new source of high-quality, sentiment-rich training data for rooms.
