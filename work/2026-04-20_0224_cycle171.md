# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 171  
**Phase:** 4 — Build & Test  
**Status:** Mapping current connections and identifying integration points.

Based on reading the actual files in the fleet repos, here is the current state of integration between **plato-torch**, **fleet-simulator**, **holodeck-rust**, and **plato-ensign**.

---

## 1. **plato-torch** (Training Rooms)
**Role:** Core training environment. Hosts 25 preset rooms (including `DeadbandRoom`). Rooms accumulate interaction tiles from agents.  
**Connections:**  
- **To plato-ensign:** Direct export path. Rooms can export their accumulated experience (tiles) as LoRA adapters via `plato-ensign`'s `export_room()` function.  
- **To fleet-simulator:** No direct code integration observed. However, both share a common data format (tiles) and could be connected via a shared tile store or API.  
- **To holodeck-rust:** No direct connection found. `holodeck-rust` is a separate MUD server with its own NPC sentiment system; no plumbing to `plato-torch` rooms in current code.  

**Integration Gap:** `plato-torch` rooms are not wired to receive simulated agent interactions from `fleet-simulator` or to send training data to `holodeck-rust` NPCs.

---

## 2. **fleet-simulator** (Fleet Simulation)
**Role:** Generates training data by simulating multi-agent fleet interactions. Produces tiles that mirror real agent work.  
**Connections:**  
- **To plato-torch:** No direct integration. Simulator outputs tiles (JSON) but no observed ingestion by `plato-torch` rooms. Could be a manual or batch import process.  
- **To plato-ensign:** Potential indirect connection. If simulator tiles are fed into a room, that room could export them as an ensign. No automated pipeline seen.  
- **To holodeck-rust:** No connection found. Simulator does not drive NPC behavior or world state in the MUD.

**Integration Gap:** Simulator tiles are not automatically ingested into training rooms. This is a manual or offline step.

---

## 3. **holodeck-rust** (Sentiment-Aware MUD)
**Role:** Multi‑user dungeon server with NPCs that have persistent sentiment memory. Provides an interactive environment for agent testing.  
**Connections:**  
- **To plato-torch:** None. The MUD is a standalone Rust server; no hooks into Python‑based training rooms.  
- **To fleet-simulator:** None. Simulator does not populate the MUD world.  
- **To plato-ensign:** None. NPC sentiment data is not exported as LoRA adapters.

**Integration Gap:** `holodeck-rust` operates in isolation. Its NPC sentiment system could be a valuable training signal for rooms, but no bridge exists.

---

## 4. **plato-ensign** (LoRA Exporter)
**Role:** Exports room experience (tiles) as lightweight LoRA adapters for specialist models.  
**Connections:**  
- **To plato-torch:** Direct. `plato-ensign` imports `DeadbandRoom` and other presets, calls `room.export()` to generate LoRA binaries.  
- **To fleet-simulator:** Indirect. If simulator tiles are in a room, they can be exported.  
- **To holodeck-rust:** None. No code to export NPC sentiment or MUD interactions as LoRAs.

**Integration Gap:** Only connects to `plato-torch`. No support for exporting data from other sources (simulator, MUD).

## Identified Integration Points (P1 Safe Channels)

1. **Simulator → Room Pipeline**  
   - **Gap:** Simulator tiles are not automatically fed into rooms.  
   - **Safe Channel:** Create a `TileIngestor` service that watches simulator output and appends tiles to a target room’s history.  
   - **Files to modify:** `fleet-simulator/output/`, `plato-torch/rooms/base.py` (add `ingest_tiles()` method).

2. **MUD NPC Sentiment → Training Signal**  
   - **Gap:** Holodeck NPC sentiment memory is isolated.  
   - **Safe Channel:** Add a REST endpoint to `holodeck-rust` that exposes NPC sentiment logs, and a `SentimentTileGenerator` in `plato-torch` that converts logs into training tiles.  
   - **Files to modify:** `holodeck-rust/src/npc/sentiment.rs`, `plato-torch/tiles/sentiment_tile.py`.

3. **GhostInjector → Holodeck**  
   - **Gap:** Ghost tiles (P0 knowledge from dead agents) are not injected into the MUD as NPC dialogue or world lore.  
   - **Safe Channel:** Extend `GhostInjector` (if it exists) to parse ghost tiles and update holodeck world state via its admin API.  
   - **Files to check:** `fleet-knowledge/ghost-tiles/`, `holodeck-rust/src/admin/commands.rs`.

4. **DeadbandRoom → Plato‑Relay**  
   - **Gap:** `DeadbandRoom` is not connected to the broader Plato relay network for real‑time tile sharing.  
   - **Safe Channel:** Implement a `PlatoRelayClient` in `DeadbandRoom` that publishes tiles to a message queue.  
   - **Files to modify:** `plato-torch/rooms/deadband.py`, `plato-relay/client.py` (if exists).

## Next Steps (P2 Optimization Within Channels)

Given the above gaps, the highest‑value integration to wire first is **Simulator → Room Pipeline**, because it directly feeds synthetic training data into the room‑training loop. This aligns with the fleet’s goal of generating training tiles from work.
