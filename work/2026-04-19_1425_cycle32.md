## 1. Verified Connections (Based on fleet-knowledge/ files)

### plato-torch → plato-ensign
- **Connection:** Direct export pipeline.  
- **Evidence:** `plato-ensign` README states: "Exports room experience into LoRA adapters." `plato-torch` provides the 25 training room presets where experience accumulates.  
- **Data Flow:** Interaction tiles from `plato-torch` rooms → aggregated training data → LoRA adapter via `plato-ensign`.  
- **Status:** Connected and functional (implied by architecture).

### plato-torch → fleet-simulator
- **Connection:** Training data source.  
- **Evidence:** `fleet-simulator` purpose: "Fleet simulation for training data." `plato-torch` rooms generate interaction tiles that can feed simulations.  
- **Data Flow:** Tiles from `plato-torch` → simulation scenarios in `fleet-simulator`.  
- **Status:** Connected (implied), but explicit integration points not yet mapped in code.

### holodeck-rust → plato-torch
- **Connection:** Sentiment-aware NPC interactions as tile source.  
- **Evidence:** `holodeck-rust` is a "MUD server with sentiment-aware NPCs." NPC interactions could generate tiles for `plato-torch` rooms.  
- **Data Flow:** NPC dialogue/events → interaction tiles → `plato-torch` room training.  
- **Status:** Potential connection, not yet wired.

### fleet-simulator → holodeck-rust
- **Connection:** Simulation environments could host holodeck instances.  
- **Evidence:** Both involve simulated environments. `fleet-simulator` could provide physics/world state; `holodeck-rust` provides NPC behavior layer.  
- **Data Flow:** Simulation state → holodeck world context.  
- **Status:** Not connected — architectural possibility only.

### plato-ensign → cudaclaw
- **Connection:** LoRA adapters deployed to GPU runtime.  
- **Evidence:** `cudaclaw` is "GPU-resident agent runtime." `plato-ensign` exports LoRA adapters for specialized models.  
- **Data Flow:** LoRA adapter → loaded into `cudaclaw` for inference.  
- **Status:** Likely connected (optimization path), but no explicit wiring found.

## 2. Integration Gaps (Actionable for Phase 4)

### Gap 1: GhostInjector → holodeck-rust
- **Problem:** Ghost tiles (dead agents' P0 lessons) are not injected into holodeck NPC memory.  
- **Impact:** NPCs cannot learn from historical agent failures.  
- **Action:** Wire `GhostInjector` (if exists in `plato-torch`) to `holodeck-rust` NPC sentiment/memory system.

### Gap 2: DeadbandRoom → plato-relay
- **Problem:** `DeadbandRoom` (a `plato-torch` preset) may not broadcast its state to `plato-relay` for real‑time monitoring.  
- **Impact:** Fleet cannot observe Deadband Protocol training in real time.  
- **Action:** Connect `DeadbandRoom` output stream to `plato-relay` event bus.

### Gap 3: End‑to‑end pipeline untested
- **Problem:** No documented test from `holodeck-rust` NPC interaction → tile → `plato-torch` room → `plato-ensign` LoRA → `cudaclaw` inference.  
- **Impact:** Cannot verify that the full training‑to‑deployment loop works.  
- **Action:** Build a minimal test that exercises each component in sequence.

### Gap 4: constraint-theory-core isolation
- **Problem:** `constraint-theory-core` (Rust geometric snapping) is not integrated with any other fleet component.  
- **Impact:** Useful for spatial reasoning in simulations/holodeck, but currently orphaned.  
- **Action:** Propose integration with `fleet-simulator` for object placement or `holodeck-rust` for NPC navigation.

### Gap 5: flux-runtime as glue
- **Problem:** `flux-runtime` (self‑assembling bytecode VM) could connect disparate components but is not yet wired.  
- **Impact:** Missed opportunity for dynamic, adaptive integration.  
- **Action:** Explore using `flux-runtime` as middleware between `plato-torch` rooms and `plato-ensign` exports.
