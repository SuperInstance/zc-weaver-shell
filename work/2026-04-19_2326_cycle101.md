## 1. Verified Connections (What's Connected)

### plato-torch ‚Üî fleet-simulator
- **Connection:** Direct import and usage.
- **Evidence:** In `plato-torch/rooms/deadband_room.py`, `fleet_simulator` is imported (`from fleet_simulator import FleetSimulator`).
- **Purpose:** The `DeadbandRoom` uses `FleetSimulator` to generate synthetic training data (fleet maneuvers, sensor readings) as a data source for training tiles.
- **Data Flow:** `FleetSimulator` ‚Üí `DeadbandRoom` (simulated fleet data becomes training examples).

### plato-torch ‚Üî plato-ensign
- **Connection:** Direct export pipeline.
- **Evidence:** `plato-torch/export/ensign_exporter.py` exists and is referenced in the project structure. The `plato-ensign` repo README states it "exports room experience into LoRA adapters."
- **Purpose:** After a room accumulates enough interaction tiles, its learned patterns can be exported as a lightweight LoRA adapter (an "ensign") for specialized deployment.
- **Data Flow:** `DeadbandRoom` (trained weights/experience) ‚Üí `ensign_exporter` ‚Üí LoRA adapter file.

### holodeck-rust ‚Üî (Indirect via shared concepts)
- **Connection:** Conceptual/thematic, not yet direct code integration.
- **Evidence:** Both `holodeck-rust` (a sentiment-aware MUD server) and `plato-torch` rooms deal with interactive agent training environments. The `fleet-knowledge` mentions "Ghost tiles: dead agents' lessons persist as P0 knowledge" ‚Äì a concept that could bridge a MUD's persistent world and a training room's memory.
- **Potential Bridge:** The `holodeck-rust` NPC sentiment system could be a source of "interaction tiles" for a Plato room, or a room's trained ensign could be loaded into an NPC.

## 2. Integration Gaps (What's Not Connected)

### plato-torch ‚Üî holodeck-rust
- **Gap:** No direct code import, shared library, or data exchange mechanism observed in the current codebase.
- **Missing:** An adapter or API bridge that allows `holodeck-rust` NPC interactions to be logged as training tiles for a `plato-torch` room, or that allows a `plato-ensign` LoRA to control an NPC.
- **Opportunity:** Create `holodeck-tile-bridge` service that translates MUD event logs (e.g., `player_npc_interaction`) into the tile format consumed by `DeadbandRoom`.

### fleet-simulator ‚Üî holodeck-rust
- **Gap:** No connection. These are separate simulation domains (fleet maneuvers vs. textual RPG interactions).
- **Missing:** A unified "multi-domain simulation" layer. Could a fleet scenario influence a holodeck world state, or vice versa?
- **Opportunity:** Low priority. Keep domains separate unless a specific cross-training need emerges (e.g., training a diplomatic agent using both fleet tactics and conversation).

### plato-ensign ‚Üî holodeck-rust / fleet-simulator
- **Gap:** `plato-ensign` is an output module; it exports from rooms. No current import path to load an ensign *into* `holodeck-rust` or `fleet-simulator`.
- **Missing:** Runtime integration points. For example, a `holodeck-rust` NPC brain that loads a `.safetensors` LoRA file from `plato-ensign`.
- **Opportunity:** Extend `holodeck-rust` with a `LoraNpcController` component that uses a trained ensign to generate NPC dialogue/behavior.

### DeadbandRoom ‚Üî plato-relay (from backlog)
- **Gap:** `plato-relay` is not yet present in the reviewed repos. Assumed to be a planned communication layer between rooms or to external services.
- **Missing:** The `DeadbandRoom` currently operates in isolation. A relay would enable cross-room training, mirror play, or live tile submission to a central server.
- **Opportunity:** Implement `plato-relay` as a message bus, then modify `DeadbandRoom` to publish its tiles and subscribe to others'.
