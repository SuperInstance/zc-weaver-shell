# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 320

**Status:** Direct file inspection of fleet repositories. Focus on actual imports, configuration references, and shared data structures.

## 1. plato-torch → fleet-simulator
**Connection:** **YES** — Direct import and data flow.
- **Evidence:** `plato-torch/src/training/rooms/deadband_room.py` imports `from fleet_simulator import FleetSimulator`.
- **Usage:** `DeadbandRoom` instantiates `FleetSimulator` as `self.simulator`. Calls `self.simulator.step()` during training cycles to generate synthetic fleet interaction data.
- **Data Flow:** Simulator generates `(state, action, reward, next_state)` tuples which become training tiles.
- **Integration Point:** `DeadbandRoom` configures the simulator via `config/deadband_config.yaml` (references `simulator_seed`, `num_agents`).

## 2. plato-torch → holodeck-rust
**Connection:** **NO** — No direct import or API call found.
- **Evidence:** Searched `plato-torch/` for "holodeck". No imports, no configuration references.
- **Gap:** Plato rooms train on synthetic simulator data, not live holodeck NPC interactions.
- **Potential Integration:** Could pipe holodeck sentiment/event logs into `DeadbandRoom` as alternative tile source.

## 3. plato-torch → plato-ensign
**Connection:** **YES** — Export pipeline.
- **Evidence:** `plato-torch/src/training/export/ensign_exporter.py` exists.
- **Usage:** After room training, `ensign_exporter.save_lora_adapter(room, path)` converts accumulated tile experience into a LoRA adapter.
- **Data Flow:** Room's `tile_buffer` → LoRA weights (via `peft` library).
- **Integration Point:** Exporter is called manually or via CLI (`python -m plato_torch.export`). Not automatically triggered by room lifecycle.

## 4. fleet-simulator → holodeck-rust
**Connection:** **NO** — Separate domains.
- **Evidence:** `fleet-simulator/` simulates abstract agent decision-making. `holodeck-rust/` is a sentiment-aware MUD server.
- **Gap:** Simulator does not spawn holodeck instances or consume its NPC dialogue.
- **Potential Integration:** Simulator agents could connect to holodeck as "players" to generate richer social training data.

## 5. fleet-simulator → plato-ensign
**Connection:** **INDIRECT** — via plato-torch.
- **Evidence:** Simulator data flows into plato-torch rooms, which can be exported to ensigns.
- **No direct export from simulator to ensign.**

## 6. holodeck-rust → plato-ensign
**Connection:** **NO** — No exporter found.
- **Evidence:** `holodeck-rust/` has no `export/` subdirectory, no references to LoRA or adapters.
- **Gap:** Holodeck interactions are not captured as ensigns, only as in-game logs.
- **Potential Integration:** Add `holodeck-exporter` that converts NPC sentiment logs into training tiles for ensign creation.

## 7. GhostInjector (plato-torch) → holodeck-rust
**Connection:** **NO** — Not wired.
- **Evidence:** `plato-torch/src/training/ghost_injector.py` exists but only injects into `DeadbandRoom` tile buffers.
- **Gap:** Ghost tiles (P0 negative knowledge) remain inside plato-torch, not injected into holodeck NPC memory.
- **Task Relevance:** "Wire GhostInjector into holodeck" is a direct backlog item to close this gap.
