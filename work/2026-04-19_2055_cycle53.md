## 2. Identified Integration Gaps (P1 Safe Channels)

### Gap 1: holodeck-rust ↔ plato-torch
- **Problem:** Holodeck NPC sentiment data could enrich training tiles, but no bridge exists.
- **Safe channel:** Add a REST API wrapper to holodeck-rust (`unstable` branch has `warp` framework sketched). Then call from `DeadbandRoom` via HTTP client.
- **Action:** Wire GhostInjector (from plato-torch) into holodeck to inject ghost tiles as NPC memories.

### Gap 2: plato-ensign → runtime deployment
- **Problem:** New LoRA adapters are not automatically loaded by cudaclaw or flux-runtime.
- **Safe channel:** Add a webhook to `plato-ensign` that posts to `oracle1-vessel/` on export. Then trigger a model reload in cudaclaw.
- **Action:** Connect DeadbandRoom to plato-relay (message bus) to announce new ensigns.

### Gap 3: fleet-simulator → holodeck-rust
- **Problem:** Simulation scenarios could drive holodeck storylines, but no link.
- **Safe channel:** Convert fleet-simulator JSON logs to holodeck `StoryEvent` protobufs (format exists in `holodeck-rust/proto/`).
- **Action:** Write a translator service in `tools/` (Rust or Python).

### Gap 4: plato-torch rooms → cross-room learning
- **Problem:** Each room trains in isolation; no sharing of ghost tiles or P0 knowledge between rooms.
- **Safe channel:** Use shared Redis instance (already in `docker-compose.yml` of plato-torch) for tile exchange.
- **Action:** Implement `TileExchange` module in `plato-torch/core`.
