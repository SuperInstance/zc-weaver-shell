Based on reading the actual files in the fleet repos, here is the current integration state between plato-torch, fleet-simulator, holodeck-rust, and plato-ensign.

## 1. Verified Connections (What's Connected)

### plato-torch ↔ plato-ensign (Strong, Direct)
- **Connection:** plato-ensign is a direct export module of plato-torch.
- **Evidence:** `plato-ensign/README.md` states: "Exports room experience from plato-torch into LoRA adapters."
- **Mechanism:** plato-torch training rooms accumulate interaction "tiles." plato-ensign processes these tiles to generate LoRA weights.
- **Data Flow:** Tiles (JSON) → plato-torch room state → plato-ensign export → `.safetensors` LoRA file.
- **Status:** This is a core, working pipeline. The connection is explicit and designed.

### plato-torch ↔ fleet-simulator (Indirect, Potential)
- **Connection:** fleet-simulator generates training data; plato-torch consumes training data.
- **Evidence:** `fleet-simulator/README.md` describes it as a "Fleet simulation for training data." `plato-torch` presets (like DeadbandRoom) are designed to train on interaction data.
- **Mechanism:** No direct API call observed in code. However, the architecture suggests fleet-simulator could output simulation logs (text/JSON) that become "tiles" for plato-torch rooms.
- **Data Flow:** Simulation events → log/tile format → ingested by a plato-torch room.
- **Status:** This is a logical, intended connection, but no observed import/export code or configuration currently wires them together.

### holodeck-rust ↔ (plato-torch / fleet-simulator) (Conceptual, Not Wired)
- **Connection:** holodeck-rust is a sentiment-aware MUD server. Its NPC interactions could be a source of training "tiles" or simulation events.
- **Evidence:** `holodeck-rust/README.md` describes "sentiment-aware NPCs" and a MUD server. It is a standalone application.
- **Mechanism:** No observed code or configuration shows holodeck-rust sending data to plato-torch or receiving from fleet-simulator.
- **Status:** Isolated component. Its "sentiment" system and interactive logs are a potential rich data source, but currently not integrated.

## 2. Integration Gaps (What's Not Connected)

### Gap 1: fleet-simulator → plato-torch (Data Pipeline Missing)
- **Problem:** No observed ingestion mechanism in plato-torch to consume fleet-simulator output.
- **Specifics:** `fleet-simulator` likely outputs simulation state (agent actions, events). `plato-torch` rooms expect "tiles" (user/agent interaction snippets). A transformer or adapter would be needed.
- **Potential Integration Point:** Create a `SimulationTileAdapter` that converts fleet-simulator event logs into the tile schema expected by a plato-torch room.

### Gap 2: holodeck-rust → plato-torch (NPC Interaction Tiles)
- **Problem:** holodeck-rust's NPC dialogues and sentiment events are not captured as training tiles.
- **Specifics:** holodeck-rust runs as a TCP MUD server. Its internal event stream (player-NPC interactions) could be tapped and formatted into tiles.
- **Potential Integration Point:** Add a "tile sink" plugin to holodeck-rust that forwards JSON-formatted interaction events to a plato-torch room via HTTP or file watch.

### Gap 3: plato-ensign ← holodeck-rust / fleet-simulator (Multi-Source Ensigns)
- **Problem:** plato-ensign currently only exports from plato-torch rooms. It could be extended to create LoRAs from diverse data sources (simulation logs, MUD interactions).
- **Specifics:** plato-ensign's tile processing is tied to plato-torch room state. To use other sources, either feed them into a room first, or extend plato-ensign to accept generic tile streams.
- **Potential Integration Point:** Generalize `plato-ensign`'s input to a `TileStream` interface, allowing it to pull from holodeck-rust, fleet-simulator, or plato-torch.

### Gap 4: holodeck-rust ↔ fleet-simulator (Shared Agent Simulation)
- **Problem:** Two separate simulation environments: holodeck-rust (text-based MUD) and fleet-simulator (agent fleet sim). They do not share agent state or environments.
- **Specifics:** Could be valuable to have fleet-simulator agents "enter" the holodeck MUD as characters, or for holodeck NPCs to be driven by fleet-simulator behavior models.
- **Potential Integration Point:** A bridge service that translates between holodeck-rust's protocol and fleet-simulator's API, allowing agent crossover.

## 3. Immediate Integration Task (P1: Safe Channel)

**Task:** Wire GhostInjector into holodeck-rust.
- **Context:** GhostInjector (likely part of plato-torch or fleet-simulator) injects "ghost tiles" (lessons from dead agents) into training streams.
- **Action:** Modify holodeck-rust to accept a GhostInjector feed, blending historical "ghost" interactions with live NPC dialogue to create richer training tiles.
- **Why P1:** It uses existing components (GhostInjector, holodeck-rust) without modifying core protocols. It adds value by enhancing tile quality with P0 (negative space) knowledge.
