## 1. plato-torch → fleet-simulator
**Connection:** **Strong**  
- `plato-torch/rooms/DeadbandRoom.py` imports `fleet_simulator` directly:  
  ```python
  from fleet_simulator import FleetSimulator, FleetConfig
  ```
- Uses `FleetSimulator` to generate training episodes:  
  ```python
  self.simulator = FleetSimulator(config=FleetConfig(...))
  episodes = self.simulator.generate_episodes(n=10)
  ```
- Shared data structure: `Episode` objects (list of `Step` tuples) flow from simulator to room.

**Integration Gap:**  
- Simulator output is consumed but not fed back. No closed loop where room decisions affect simulator state.
- No shared configuration file; room hardcodes `FleetConfig`.

---

## 2. plato-torch → holodeck-rust
**Connection:** **Weak / Indirect**  
- No direct import of `holodeck-rust` in plato-torch Python code.
- Potential link via **shared protocol**: `holodeck-rust` defines a MUD server with sentiment-aware NPCs; `plato-torch` rooms could connect as clients.
- In `plato-torch/configs/room_presets.yaml`, preset `"HolodeckTraining"` exists but is commented out:
  ```yaml
  # HolodeckTraining:
  #   room_class: "ExternalRoom"
  #   endpoint: "ws://localhost:8080/holodeck"
  ```
- **Actual connection:** Not live. Preset suggests planned integration.

**Integration Gap:**  
- No active WebSocket client in plato-torch to connect to holodeck-rust server.
- No shared data schema for NPC sentiment or game state.

---

## 3. plato-torch → plato-ensign
**Connection:** **Strong**  
- `plato-torch/exporters/ensign_exporter.py` imports `plato_ensign`:
  ```python
  from plato_ensign import LoRAAdapter, export_room_experience
  ```
- Called in `DeadbandRoom.on_training_complete()`:
  ```python
  adapter = export_room_experience(self.tiles, self.config)
  adapter.save(f"ensigns/{self.room_id}.safetensors")
  ```
- Shared data: `Tile` objects (serialized interactions) from room → ensign.

**Integration Gap:**  
- Export is one-way (room → ensign). No mechanism for ensign to influence room behavior.
- No validation that exported LoRA adapters are loadable or functional.

---

## 4. fleet-simulator → holodeck-rust
**Connection:** **None**  
- `fleet-simulator/src/lib.rs` has no mention of `holodeck`.
- Simulator models fleet dynamics (agents, tasks, comms), not MUD environments.
- Different domains: simulator is strategic/coordination; holodeck is interactive narrative.

**Integration Gap:**  
- No shared interface. Could be bridged if holodeck-rust used simulator agents as NPCs, but no code exists.

---

## 5. fleet-simulator → plato-ensign
**Connection:** **None**  
- No direct reference. Simulator generates episodes for training, not LoRA adapters.
- Plato-ensign works on room tiles, not simulator episodes.

**Integration Gap:**  
- Simulator data not formatted for ensign export. Missing `Tile` conversion step.

---

## 6. holodeck-rust → plato-ensign
**Connection:** **Planned**  
- `holodeck-rust/README.md` states:  
  > "NPC sentiment logs can be exported to plato-ensign for fine-tuning."
- No exporter code present in repo.
- Could be a future feature: sentiment logs → tiles → LoRA.

**Integration Gap:**  
- No actual exporter implementation. Sentiment logs exist but not connected to ensign pipeline.

---

## 7. plato-ensign → Others
**Connection:** **Sink**  
- Plato-ensign is an output module; it receives tiles from rooms (or planned from holodeck) and produces LoRA adapters.
- No references to other fleet components in its code.

**Integration Gap:**  
- No feedback loop. Ensigns are not loaded back into rooms or simulator.

---

## 8. Ghost Tiles & Mirror Play
**Connection:** **Architectural, Not Coded**  
- Ghost tiles: referenced in `FLEET-CONTEXT.md` as "dead agents' lessons persist as P0 knowledge."
- No file `GhostInjector` found in searched repos.
- Mirror Play: "two PLATO systems iterating screen-to-screen" — conceptual, no implementation found.

**Integration Gap:**  
- Ghost tile injection not implemented.
- Mirror Play not coded; could be a future training mode.
