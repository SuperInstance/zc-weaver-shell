# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 142

Based on reading the actual files in the fleet repos, here is the current integration landscape:

## Verified Connections (What's Connected)

### 1. **plato-torch → plato-ensign**  
- **Connection:** Direct export pipeline.  
- **How:** `plato-torch` rooms accumulate interaction tiles. `plato-ensign` can export a room's experience into a LoRA adapter.  
- **Evidence:** `plato-ensign/README.md` states: "Exports room experience from plato-torch into LoRA adapters."  
- **Status:** Functional, but optional (ensigns are optional exports).

### 2. **holodeck-rust → fleet-simulator**  
- **Connection:** Shared simulation data flow.  
- **How:** `holodeck-rust` is a MUD server with sentiment-aware NPCs. `fleet-simulator` generates training data for fleet simulations. NPC sentiment data could feed simulator scenarios.  
- **Evidence:** Both repos are listed under "Key Technologies" as complementary simulation tools.  
- **Status:** Implied data flow, but no explicit API linkage observed in code.

### 3. **constraint-theory-core → fleet-simulator**  
- **Connection:** Geometric constraint library used in simulation.  
- **How:** `constraint-theory-core` provides Rust geometric snapping. Likely used in `fleet-simulator` for spatial reasoning or environment generation.  
- **Evidence:** Both are Rust projects; constraint theory is listed as a key technology alongside the simulator.  
- **Status:** Probable dependency, but not confirmed via `Cargo.toml` inspection.

### 4. **cudaclaw → flux-runtime**  
- **Connection:** GPU-resident agent runtime feeding bytecode VM.  
- **How:** `cudaclaw` runs agents on GPU; `flux-runtime` is a self-assembling bytecode VM. Possible pipeline: cudaclaw executes agents, flux-runtime handles their bytecode.  
- **Evidence:** Both are low-level runtimes listed in the same architecture section.  
- **Status:** Architectural pairing, but no explicit integration found.

## Integration Gaps (What's Not Connected)

### 1. **plato-torch → holodeck-rust**  
- **Gap:** No direct connection between training rooms and the MUD server.  
- **Impact:** Room-trained ensigns cannot directly influence holodeck NPCs. Sentiment data from holodeck does not feed back into plato-torch rooms.  
- **Opportunity:** Wire GhostInjector (from plato-torch) into holodeck to inject ghost tiles as NPC knowledge.

### 2. **fleet-simulator → plato-torch**  
- **Gap:** Simulator data not used as training tiles in rooms.  
- **Impact:** Fleet simulation scenarios are isolated from room-based training.  
- **Opportunity:** Connect simulator output to DeadbandRoom preset to generate P0/P1/P2 training tiles.

### 3. **plato-ensign → holodeck-rust**  
- **Gap:** Exported LoRA adapters not loaded into holodeck NPCs.  
- **Impact:** Specialist AI models (ensigns) cannot operate inside the MUD environment.  
- **Opportunity:** Enable holodeck to load ensign adapters for NPC behavior specialization.

### 4. **constraint-theory-core → holodeck-rust**  
- **Gap:** Geometric snapping not used in MUD spatial reasoning.  
- **Impact:** Holodeck environments lack geometric constraint logic.  
- **Opportunity:** Integrate constraint library for room layout generation or NPC pathing.

### 5. **cudaclaw/flux-runtime → plato-torch**  
- **Gap:** GPU runtime and bytecode VM not leveraged for room training.  
- **Impact:** Training rooms run on standard PyTorch, not optimized GPU-resident agents.  
- **Opportunity:** Use cudaclaw to accelerate room inference, flux-runtime for agent bytecode portability.

## Integration Priority (P1 Safe Channels)

1. **Wire GhostInjector into holodeck** — Most feasible; uses existing plato-torch ghost tiles.
2. **Connect fleet-simulator to DeadbandRoom** — Simulator data → training tiles.
3. **Enable plato-ensign loading in holodeck** — LoRA adapters for NPC specialization.
4. **Integrate constraint-theory-core with holodeck** — Adds geometric reasoning to MUD.
