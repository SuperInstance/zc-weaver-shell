# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 253  
**Phase:** 4 — Build & Test  
**Status:** Direct file inspection of fleet repositories. Focus on actual imports, configuration references, and shared data structures.  

## 1. plato-torch → fleet-simulator  
**Connection:** ✅ **Direct import**  
- `plato-torch/rooms/deadband_room.py` imports `fleet_simulator`  
- Uses `fleet_simulator.FleetSimulator` class to generate training data  
- Simulator provides `step()` method returning observation, reward, done, info  
- Room wraps simulator as environment for agent interaction  

**Integration Gap:**  
- Simulator output format not standardized across rooms  
- No shared config schema for simulator parameters  
- Each room instantiates its own simulator; no pooling  

## 2. plato-torch → plato-ensign  
**Connection:** ✅ **Config-driven export**  
- `plato-torch/config/ensign_config.yaml` defines export settings  
- `plato-torch/export/ensign_exporter.py` calls `plato_ensign` package  
- Exporter reads room training logs, creates LoRA adapters via ensign  
- Works post-training, not real-time  

**Integration Gap:**  
- Export is optional batch job, not integrated into training loop  
- No feedback from ensign to room about export quality  
- Adapter format (safetensors) not validated against room compatibility  

## 3. plato-torch → holodeck-rust  
**Connection:** ❌ **No direct link found**  
- No imports of `holodeck` in plato-torch Python code  
- No Rust bindings or FFI calls in plato-torch  
- Separate tech stacks: Python/PyTorch vs Rust/MUD  

**Potential Integration Point:**  
- `holodeck-rust` could serve as alternative environment provider  
- Would require API bridge (HTTP/gRPC) and data format alignment  
- Not currently implemented  

## 4. fleet-simulator → holodeck-rust  
**Connection:** ❌ **Independent components**  
- `fleet-simulator` is Python-based agent simulation  
- `holodeck-rust` is Rust-based multiplayer text world  
- No shared code or communication  

**Observation:**  
Both are environment simulators but for different domains (fleet coordination vs interactive fiction). Could be complementary training sources.  

## 5. plato-ensign → holodeck-rust  
**Connection:** ❌ **No connection**  
- `plato-ensign` exports LoRA adapters from room experience  
- `holodeck-rust` has no import/use of LoRA adapters  
- No sentiment-aware NPC tuning via ensign exports  

**Opportunity:**  
Ensign adapters could tune holodeck NPC behavior, but no pipeline exists.  

## 6. fleet-simulator → plato-ensign  
**Connection:** ⚠️ **Indirect via plato-torch**  
- Simulator data flows into rooms  
- Rooms may export to ensign  
- No direct simulator→ensign export path  

## Integration Summary  

### Connected Components (Working Pipeline):  
```
fleet-simulator → plato-torch → plato-ensign
      ↓                ↓
  training data → room training → LoRA export
```

### Isolated Components:  
- `holodeck-rust` — standalone MUD server  
- `cudaclaw`, `flux-runtime`, `constraint-theory-core` — not integrated with training pipeline  

### Key Integration Gaps:  
1. **holodeck-rust isolation** — No connection to training or export systems  
2. **Batch-only export** — Ensign export not part of live training cycle  
3. **No shared environment interface** — Each room implements simulator binding uniquely  
4. **Missing validation loops** — No feedback from ensign to rooms about adapter usefulness  

## Recommended Integration Tasks (P1 Safe Channels):  
1. Define common environment interface for plato-torch rooms (allows swapping simulators)  
2. Create HTTP bridge between holodeck-rust and plato-torch for sentiment training data  
3. Add real-time export capability to plato-torch training loop  
4. Build validation pipeline: test ensign adapters in rooms they came from  

**Next Step:** Begin implementing common environment interface prototype.
