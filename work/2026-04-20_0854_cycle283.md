## 1. plato-torch
**Repository:** `plato-torch`  
**Purpose:** Training room presets, self-training rooms, tile processing.

### Verified Connections:
- **To plato-ensign:** Direct import in `plato-torch/src/export.rs`.  
  - `use plato_ensign::LoRAAdapter;`  
  - Function `export_room_to_lora()` calls `LoRAAdapter::from_tiles()`.  
- **To fleet-simulator:** No direct import found.  
  - However, `plato-torch/config/training_presets/deadband_room.yaml` contains a reference to `fleet-simulator` as a data source under `simulation_data_path: "../fleet-simulator/output/traces/"`.  
- **To holodeck-rust:** No direct import found.  
  - No configuration references observed in preset files.

### Integration Gaps:
- **plato-torch → fleet-simulator:** Connection is config-based, not code-based. Depends on filesystem path.  
- **plato-torch → holodeck-rust:** No observed linkage.  
- **plato-torch internal:** 25 room presets are defined, but only `deadband_room` is referenced in tests.

## 2. fleet-simulator
**Repository:** `fleet-simulator`  
**Purpose:** Generate training trace data from simulated fleet interactions.

### Verified Connections:
- **To plato-torch:** No direct import.  
  - Output directory (`output/traces/`) is referenced by plato-torch config.  
- **To holodeck-rust:** No direct import.  
- **To plato-ensign:** No direct import.

### Integration Gaps:
- **fleet-simulator → others:** Currently a standalone data generator. No API or library calls to other fleet components.  
- Output format is JSON traces; no schema validation shared with plato-torch.

## 3. holodeck-rust
**Repository:** `holodeck-rust`  
**Purpose:** MUD server with sentiment-aware NPCs, real-time interaction environment.

### Verified Connections:
- **To plato-torch:** No direct import.  
- **To fleet-simulator:** No direct import.  
- **To plato-ensign:** No direct import.  
- **Internal:** Uses `serde` for JSON, `tokio` for async. Exposes WebSocket API.

### Integration Gaps:
- **holodeck-rust → fleet components:** No imports or config references to other fleet repos.  
- NPC sentiment engine is self-contained; no hook for importing training tiles or exporting experience.

## 4. plato-ensign
**Repository:** `plato-ensign`  
**Purpose:** Export room experience into LoRA adapters.

### Verified Connections:
- **To plato-torch:** Exported as a library (`lib.rs`).  
  - `plato-torch` imports and uses `plato_ensign::LoRAAdapter`.  
- **To fleet-simulator:** No direct import.  
- **To holodeck-rust:** No direct import.

### Integration Gaps:
- **plato-ensign → holodeck-rust/fleet-simulator:** No support for ingesting data from these sources.  
- Only ingests `Vec<Tile>` from plato-torch.

## 6. Identified Integration Points (P1 Safe Channels)

1. **plato-torch ↔ plato-ensign** — Strong, code-level integration. Works.  
2. **plato-torch → fleet-simulator** — Weak, filesystem-based integration. Depends on relative path.  
3. **holodeck-rust ↔ (none)** — Isolated. Potential channel: ingest NPC interactions as tiles.  
4. **fleet-simulator → plato-torch** — One-way data flow. Could be strengthened with a shared schema.
