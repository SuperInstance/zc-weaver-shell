# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 179  
**Phase:** 4 — Build & Test  
**Status:** Mapping current connections and identifying integration points.

Based on reading the actual files in the fleet repos, here is the current state of integration between **plato-torch**, **fleet-simulator**, **holodeck-rust**, and **plato-ensign**.

---

## 1. **plato-torch** → **fleet-simulator**
**Connection:** **Direct & Active**  
- **plato-torch/rooms/deadband_room.py** imports `fleet_simulator` directly.  
- Uses `fleet_simulator.FleetSimulator` to generate synthetic training data (agent interactions, P0/P1/P2 decisions).  
- Simulated data is formatted into "tiles" and fed into the room's training loop.  
- **Integration point:** `DeadbandRoom._generate_simulated_tiles()` calls the simulator.  

**Gap:**  
- The simulator currently generates generic agent interactions. No direct hook for **holodeck-rust** NPC sentiment data or **plato-ensign** adapter influence.

---

## 2. **plato-torch** → **plato-ensign**
**Connection:** **Optional Export**  
- **plato-ensign** is a separate tool that can export a trained room’s experience (weights, tiles) into a LoRA adapter.  
- **plato-torch/rooms/base_room.py** has a `save_ensign()` method that calls `plato_ensign.export_room()`.  
- This is not automatic — it’s a manual export step after room training.  

**Gap:**  
- No real-time integration during training. The ensign is a post-hoc snapshot, not a live participant in the training loop.

---

## 3. **holodeck-rust** → **plato-torch**
**Connection:** **None (yet)**  
- **holodeck-rust** is a sentiment-aware MUD server written in Rust.  
- It generates NPC interactions with emotional states.  
- Currently no import or API call from **plato-torch** to **holodeck-rust**.  

**Opportunity:**  
- Could feed NPC dialogue + sentiment tiles into `DeadbandRoom` as another data source alongside the fleet simulator.

---

## 4. **fleet-simulator** → **holodeck-rust**
**Connection:** **None**  
- The simulator models fleet agents, not MUD NPCs.  
- No shared data format or protocol between them.

---

## 5. **plato-ensign** → **fleet-simulator** / **holodeck-rust**
**Connection:** **None**  
- Ensign adapters are currently standalone LoRA files.  
- No mechanism to load an ensign into the simulator or holodeck to influence their behavior.

---

## 6. **Ghost Tiles & GhostInjector**
**Location:** **plato-torch/rooms/ghost_injector.py**  
**Status:** Isolated utility.  
- Reads "ghost tiles" (persisted lessons from dead agents) from disk.  
- Injects them into a room’s training buffer as P0 examples.  
- **Not connected** to **holodeck-rust** or **fleet-simulator** — only works with local tile files.

---

## Identified Integration Gaps (P1 Channels)

1. **holodeck-rust → plato-torch**  
   - Add a tile generator in holodeck that formats NPC interactions (dialogue + sentiment) into the same tile JSON schema used by plato-torch.  
   - Extend `DeadbandRoom` to optionally pull from a holodeck HTTP endpoint or shared queue.

2. **plato-ensign → fleet-simulator**  
   - Allow the simulator to load an ensign adapter to influence simulated agent behavior (e.g., bias toward P1 channels observed by previous agents).

3. **GhostInjector → holodeck-rust**  
   - Let holodeck NPCs "learn" from ghost tiles — inject P0 negative examples into NPC decision logic.

4. **Unified tile bus**  
   - Currently each component produces tiles in isolation. A shared tile bus (Redis, Kafka, or simple file queue) could let plato-torch train from multiple live sources simultaneously.
