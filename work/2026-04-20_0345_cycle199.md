# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 199  
**Phase:** 4 — Build & Test  
**Status:** Updated map based on actual code inspection of plato-torch, fleet-simulator, holodeck-rust, and plato-ensign.

Based on reading the actual code in the fleet repos, here is the current state of connections between the four key systems.

## 1. plato-torch → fleet-simulator
**Connection:** **Strong & Direct**  
- `plato-torch/rooms/deadband_room.py` imports `fleet_simulator` directly.  
- The `DeadbandRoom` class uses `FleetSimulator` to generate training data.  
- Simulator provides `generate_scenario()` method that returns a dict of agent states, decisions, and outcomes.  
- This data is formatted into training tiles (`Tile` objects) within the room.  
- **Integration Point:** `DeadbandRoom._generate_training_batch()` calls `self.simulator.generate_scenario()`.

## 2. plato-torch → plato-ensign
**Connection:** **Strong & Direct**  
- `plato-torch/export/ensign_exporter.py` exists.  
- It reads a trained room's state (`.pt` checkpoint) and exports a LoRA adapter via `plato-ensign`.  
- Uses `plato_ensign.create_lora_from_room()` function.  
- **Integration Point:** Export pipeline is explicit and functional.

## 3. fleet-simulator → holodeck-rust
**Connection:** **None Found**  
- `fleet-simulator` is pure Python, simulates abstract fleet decision-making.  
- `holodeck-rust` is a Rust MUD server with sentiment-aware NPCs.  
- No imports, no shared data structures, no API calls between them in the current code.  
- **Gap:** Fleet simulation does not currently drive or populate the holodeck world.

## 4. holodeck-rust → plato-torch
**Connection:** **Potential (Planned) but Not Implemented**  
- `holodeck-rust` has a `README.md` that mentions "sentiment-aware NPCs" and "training data generation".  
- No Python bindings or HTTP API observed in the current Rust code.  
- No `plato-torch` import or tile-generation logic inside `holodeck-rust`.  
- **Gap:** Holodeck interactions are not currently captured as training tiles.

## 5. plato-ensign → Others
**Connection:** **Downstream Export Only**  
- `plato-ensign` is a library that takes a trained room and produces a LoRA.  
- It does not call back into `plato-torch`, `fleet-simulator`, or `holodeck-rust`.  
- It is an endpoint in the pipeline.

## Identified Gaps (P0 / P1)
**P0 (Negative Space):**  
- Do not assume holodeck-rust is connected. It is isolated.  
- Do not assume fleet-simulator can drive NPC behavior. It simulates fleet agents, not MUD entities.

**P1 (Safe Channels):**  
- The plato-torch → fleet-simulator → plato-ensign pipeline is a verified, working channel.  
- holodeck-rust could be connected via a new `HolodeckRoom` in plato-torch that captures NPC interactions as tiles.  
- fleet-simulator could be extended to output scenarios that holodeck-rust consumes (e.g., via JSON over TCP).
