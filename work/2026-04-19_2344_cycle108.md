# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 108  
**Phase:** 4 — Build & Test  
**Status:** Mapping current connections and identifying integration points.  

Based on reading the actual fleet repositories (plato-torch, fleet-simulator, holodeck-rust, plato-ensign), here is the current state of integration.

## Current Connections (What's Connected)

### 1. **plato-torch ↔ fleet-simulator**  
- **Connection:** Strong, bidirectional data flow.  
- **Mechanism:** `fleet-simulator` generates synthetic training data (agent interactions, environment states). This data is formatted into **training tiles** and ingested by `plato-torch` rooms.  
- **Evidence:**  
  - `fleet-simulator/src/tile_generator.rs` exports `Tile` structs matching `plato-torch`'s `Tile` schema.  
  - `plato-torch/rooms/deadband_room.py` includes a `SimulatorTileLoader` class that reads from `fleet-simulator` output directories.  
- **Purpose:** Provides high-volume, diverse interaction data for room training without requiring live human input.

### 2. **plato-torch ↔ plato-ensign**  
- **Connection:** Direct export pipeline.  
- **Mechanism:** A trained `plato-torch` room can export its learned weights and experience into a **LoRA adapter** via `plato-ensign`.  
- **Evidence:**  
  - `plato-ensign/src/exporter.py` imports `Room` from `plato-torch` and calls `room.export_to_lora()`.  
  - `plato-torch/rooms/base_room.py` defines the `export_to_lora()` method, which packages room state and tile history.  
- **Purpose:** Creates portable, specialized AI models (ensigns) from room training.

### 3. **holodeck-rust → plato-torch** (One-way, potential)  
- **Connection:** Conceptual, not yet fully implemented.  
- **Mechanism:** `holodeck-rust` is a MUD server with sentiment-aware NPCs. Its interaction logs *could* be formatted into tiles for `plato-torch`.  
- **Evidence:**  
  - `holodeck-rust/src/npc/sentiment.rs` tracks NPC emotional state—potential rich tile metadata.  
  - `plato-torch/docs/integration_ideas.md` lists "holodeck logs" as a future tile source.  
- **Purpose:** Would provide human-like, narrative-driven interaction data.

## Integration Gaps (What's Not Connected)

### 1. **holodeck-rust ↔ fleet-simulator**  
- **Gap:** No direct connection. Their generated data flows are parallel, not integrated.  
- **Impact:** `holodeck-rust`'s sentiment-rich narratives and `fleet-simulator`'s structured agent simulations are not cross-pollinating.  
- **Opportunity:** Could feed `holodeck-rust` NPC personalities into `fleet-simulator` agents, or use simulator events to trigger holodeck story arcs.

### 2. **plato-ensign → holodeck-rust**  
- **Gap:** No mechanism to deploy an ensign (LoRA) as an NPC in the holodeck.  
- **Impact:** Ensigns remain static exports; cannot be "live-tested" in an interactive environment.  
- **Opportunity:** Integrate `plato-ensign` adapters into `holodeck-rust` NPC brain architecture, allowing ensigns to role-play in real-time.

### 3. **GhostInjector (plato-torch) → holodeck-rust**  
- **Gap:** The `GhostInjector` (which inserts dead agents' lessons as P0 tiles) currently only injects into `plato-torch` rooms.  
- **Evidence:** `plato-torch/utils/ghost_injector.py` has no `holodeck-rust` target.  
- **Impact:** Holodeck NPCs cannot learn from fleet "ghost" knowledge.  
- **Opportunity:** Extend `GhostInjector` to format ghost tiles as holodeck lore or NPC memory snippets.

### 4. **DeadbandRoom (plato-torch) ↔ plato-relay**  
- **Gap:** `DeadbandRoom` (a training room preset) is not connected to `plato-relay` (the real-time tile routing system).  
- **Evidence:** `plato-relay/src/routes.rs` does not list `deadband` as a destination.  
- **Impact:** Deadband Protocol training operates in isolation, not benefiting from live fleet tile streams.  
- **Opportunity:** Connect `DeadbandRoom` to `plato-relay` to enable real-time P0/P1/P2 tile ingestion from active agents.

## Integration Priority (P0/P1/P2)

**P0 (Negative Space — Avoid):**  
- Do not force integration where data schemas clash fundamentally (e.g., raw simulator telemetry into holodeck dialogue without transformation).  
- Do not break existing `plato-torch` ↔ `fleet-simulator` pipeline—it's currently the primary training data source.

**P1 (Safe Channels — Where We Can Be):**  
1. **GhostInjector → holodeck-rust:** Low-risk, adds P0 knowledge without disrupting core loops.  
2. **DeadbandRoom → plato-relay:** Adds real-time data, but requires careful schema alignment.  
3. **holodeck-rust logs → plato-torch:** One-way ingestion, enriches tile diversity.

**P2 (Optimize — Best Path):**  
- **plato-ensign → holodeck-rust:** Higher complexity, but enables live ensign testing—closes the loop between training and deployment.
