## 1. plato-torch → fleet-simulator
**Connection:** Strong, direct dependency.  
**Evidence:**  
- `plato-torch/requirements.txt` includes `fleet-simulator>=0.1.0`  
- `plato-torch/src/training/rooms/deadband_room.py` imports `from fleet_simulator import FleetSimulator`  
- `plato-torch/configs/room_presets/deadband.yaml` references `simulator_config: fleet_simulator/configs/basic_fleet.yaml`  

**Data Flow:**  
DeadbandRoom instantiates a FleetSimulator object. Room steps call `simulator.step()`. Agent observations from the simulator are wrapped into training tiles.

**Integration Gap:**  
No bidirectional control flow observed. The simulator does not call back into plato-torch. It's a passive data source.

## 2. plato-torch → plato-ensign
**Connection:** Optional export pipeline.  
**Evidence:**  
- `plato-torch/src/training/exporter.py` defines `EnsignExporter` class  
- `plato-torch/scripts/export_ensign.py` loads a trained room checkpoint and calls `exporter.export_to_lora()`  
- `plato-ensign/` is a separate repo; plato-torch does not import it directly. The exporter writes LoRA weights to a shared directory format defined by plato-ensign.

**Data Flow:**  
After room training, an explicit export script is run. It reads the room's accumulated experience (tiles) and model state, then produces a `.safetensors` file following plato-ensign's adapter spec.

**Integration Gap:**  
Live, incremental export not implemented. Export is a separate batch job.
