# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 251  
**Phase:** 4 — Build & Test  
**Status:** Direct file inspection of fleet repositories. Focus on actual imports, configuration references, and shared data structures.

## 1. plato-torch → fleet-simulator
**Connection:** Strong, direct integration via `fleet-simulator` as a data source.
- **Evidence:** `plato-torch/rooms/deadband_room.py` imports `from fleet_simulator import FleetSimulator`.
- **Data flow:** `DeadbandRoom` instantiates `FleetSimulator` to generate training episodes (simulated fleet interactions).
- **Shared structures:** `FleetSimulator` returns `List[Dict]` episodes; `DeadbandRoom` converts them to training tiles.
- **Gap:** No observed reverse dependency (fleet-simulator does not import plato-torch). This is a one‑way producer‑consumer relationship.

## 2. plato-torch → plato-ensign
**Connection:** Direct export pipeline.
- **Evidence:** `plato-torch/export/ensign_exporter.py` exists and is referenced in `plato-torch/README.md`.
- **Data flow:** After a room accumulates tiles, `ensign_exporter` can package the room’s experience into a LoRA adapter.
- **Shared structures:** Exporter reads the room’s `tile_history` and `model_checkpoints`.
- **Gap:** The exporter appears to be an optional post‑training step, not a live bidirectional link. No observed imports from plato‑ensign back into plato‑torch.

## 3. holodeck-rust → (plato-torch, fleet-simulator, plato-ensign)
**Connection:** Currently isolated.
- **Evidence:** `holodeck-rust/Cargo.toml` shows no dependencies on other fleet crates or Python packages.
- **Data flow:** None observed. Holodeck is a standalone MUD server with sentiment‑aware NPCs.
- **Shared structures:** None.
- **Gap:** Holodeck does not feed into or consume from the plato‑torch training pipeline. It is a separate simulation environment.

## 4. fleet-simulator → holodeck-rust
**Connection:** None observed.
- **Evidence:** `fleet-simulator/` (Python) has no Rust bindings or references to holodeck.
- **Gap:** No shared episode generation. Fleet‑simulator simulates fleet‑agent interactions; holodeck simulates human‑like NPC interactions. These are parallel, unconnected simulators.

## 5. plato-ensign → holodeck-rust
**Connection:** None observed.
- **Evidence:** `plato-ensign/` (Python) has no references to holodeck‑rust.
- **Gap:** Ensign exports are derived from plato‑torch rooms only, not from holodeck sessions.

## 6. GhostInjector (mentioned in backlog)
**Location:** `plato-torch/injectors/ghost_injector.py`.
- **Connection to holodeck:** None yet. GhostInjector reads `memory/` files (dead agents’ lessons) and injects them as P0 tiles into a room. It does not currently interface with holodeck’s NPC system.
- **Integration opportunity:** GhostInjector could supply “ghost memories” to holodeck NPCs, making them sentiment‑aware of past fleet failures.

## 7. DeadbandRoom → plato-relay
**Connection:** Not yet wired.
- **Evidence:** `plato-relay/` exists as a separate repo (message‑passing bus). `deadband_room.py` does not import it.
- **Gap:** DeadbandRoom currently trains locally; it does not broadcast tiles or receive commands via plato‑relay.
