## 2. Integration Gaps (What's Not Connected)

### **Gap 1: plato-ensign → holodeck-rust**
- **Issue:** No feedback loop from trained ensigns back to NPC behavior
- **Current State:** `holodeck-rust` NPCs have sentiment awareness but don't improve from fleet training
- **Opportunity:** Inject ensign LoRA adapters into NPC decision-making
- **Priority:** **High** (closes training→deployment loop)

### **Gap 2: cudaclaw → plato-torch**
- **Issue:** GPU-resident runtime not leveraged for room training acceleration
- **Current State:** `plato-torch` likely CPU-bound for tile processing
- **Opportunity:** Offload tile processing/transformation to `cudaclaw` GPU runtime
- **Priority:** **Medium-High** (performance optimization)

### **Gap 3: flux-runtime → holodeck-rust**
- **Issue:** Self-assembling bytecode VM not used for dynamic NPC behavior
- **Current State:** NPC logic likely static or script-based
- **Opportunity:** Use `flux-runtime` for adaptive, self-modifying NPC behaviors
- **Priority:** **Medium** (advanced NPC capabilities)

### **Gap 4: Ghost tiles → fleet-simulator**
- **Issue:** Dead agents' lessons (ghost tiles) not incorporated into simulation scenarios
- **Current State:** Ghost tiles exist as P0 knowledge but not actively used
- **Opportunity:** Inject ghost tile data into `fleet-simulator` as negative training examples
- **Priority:** **High** (implements Deadband Protocol P0 directly)

### **Gap 5: Mirror Play → plato-ensign**
- **Issue:** Two PLATO systems iterating screen-to-screen creates unique data not captured for ensign training
- **Current State:** Mirror Play generates interaction patterns between systems
- **Opportunity:** Capture Mirror Play sessions as specialized ensign exports
- **Priority:** **Medium** (unique training data source)
