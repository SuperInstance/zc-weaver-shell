# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 323  
**Phase:** 4 — Build & Test  
**Status:** Direct file inspection of fleet repositories. Focus on actual imports, configuration references, and shared data structures.  

---

## 1. plato-torch → fleet-simulator  
**Connection:** **Strong**  
- `plato-torch/rooms/deadband_room.py` imports `fleet_simulator.simulator` directly.  
- Uses `FleetSimulator` class to generate synthetic agent interaction data.  
- Simulator output is formatted into training tiles (JSON) for room ingestion.  
- **Data flow:** `fleet-simulator` → `DeadbandRoom` → tile storage → training loop.  

**Gap:** No bidirectional feedback. Room training results do not currently adjust simulator parameters.  

---

## 2. plato-torch → plato-ensign  
**Connection:** **Strong**  
- `plato-torch/export/ensign_exporter.py` reads room checkpoint files.  
- Converts trained adapter weights into LoRA format via `plato_ensign.core`.  
- Exports include configs, adapter binaries, and metadata.  
- **Data flow:** trained room → exporter → LoRA adapter file.  

**Gap:** Export is manual/trigger-based; no automatic export on training milestones.  

---

## 3. holodeck-rust → plato-torch  
**Connection:** **Weak** (potential, not wired)  
- `holodeck-rust/src/npc/sentiment.rs` defines `NpcState` with emotion vectors.  
- `plato-torch/rooms/holodeck_bridge.py` exists but is a stub—no active imports.  
- Plato rooms could consume NPC interaction logs as training tiles, but bridge is not implemented.  

**Gap:** No live data pipe from holodeck to plato-torch. NPC sentiment events are not yet training inputs.  

---

## 4. fleet-simulator → holodeck-rust  
**Connection:** **None**  
- No shared data structures or imports in either codebase.  
- Simulator generates agent-to-agent dialogues; holodeck handles player-to-NPC interactions.  
- Potential overlap: both generate conversational training data, but formats differ.  

**Gap:** No integration point. Could align output formats (JSON) for cross-use.  

---

## 5. plato-ensign → holodeck-rust  
**Connection:** **None**  
- Ensign exports LoRA adapters; holodeck does not currently load adapters at runtime.  
- No Rust bindings for PyTorch adapters in holodeck codebase.  

**Gap:** Trained personality adapters cannot be injected into NPCs.  

---

## 6. plato-torch internal connections  
**Connection:** **Modular**  
- Rooms are separate; share base classes (`BaseRoom`, `TileStorage`).  
- `DeadbandRoom` and `MirrorRoom` are independent presets.  
- Common utilities: `tile_utils.py`, `validation.py`.  

**Gap:** Rooms do not share trained weights—each trains its own adapter from its own tiles.  

---

## 7. External technologies (cudaclaw, flux-runtime, constraint-theory-core)  
**Connection:** **Not yet integrated**  
- `cudaclaw` and `flux-runtime` are separate repos; no imports in plato-torch or holodeck.  
- `constraint-theory-core` is a Rust library; no bindings in Python training code.  
- Mentioned in fleet context as future components.  

**Gap:** These are standalone; no active wiring into the training or simulation pipeline.
