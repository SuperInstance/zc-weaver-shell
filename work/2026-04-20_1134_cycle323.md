# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 323  
**Phase:** 4 — Build & Test  
**Status:** Direct file inspection of fleet repositories. Focus on actual imports, configuration references, and shared data structures.  

## 1. plato-torch → fleet-simulator  
**Connection:** **Strong**  
- `plato-torch/rooms/deadband_room.py` imports `fleet_simulator` directly:  
  ```python
  from fleet_simulator import FleetSimulator, FleetState
  ```
- Uses `FleetSimulator` to generate synthetic training data (agent interactions, P0/P1/P2 trajectories).  
- `DeadbandRoom` passes simulator outputs as training tiles to Plato server.  
- **Data flow:** Simulated fleet states → room training tiles.  

**Integration Gap:** None observed — this is a core, working dependency.  

## 2. plato-torch → holodeck-rust  
**Connection:** **Weak / Indirect**  
- No direct Python imports of `holodeck-rust` in `plato-torch` codebase.  
- However, `plato-torch/configs/room_presets.yaml` includes a preset named `"holodeck"` with `room_type: "SentimentRoom"`.  
- `SentimentRoom` (in `plato-torch/rooms/sentiment_room.py`) appears designed to connect to a sentiment‑aware MUD server — which matches `holodeck-rust`’s description.  
- **Likely integration:** Config‑driven; the room expects a running `holodeck-rust` server at a configured endpoint.  

**Integration Gap:** No explicit client‑side Rust/Python binding observed. Connection likely via HTTP/gRPC (not yet implemented in current `plato-torch`).  

## 3. plato-torch → plato-ensign  
**Connection:** **Strong**  
- `plato-torch/export/ensign_exporter.py` is dedicated to exporting room experience to LoRA adapters.  
- Uses `peft` (Parameter‑Efficient Fine‑Tuning) library to save LoRA weights.  
- Called from `plato-torch/train/trainer.py` after room training reaches a checkpoint.  
- **Data flow:** Trained room model → LoRA adapter (`plato-ensign` output).  

**Integration Gap:** None — exporter is fully integrated into the training pipeline.  

## 4. fleet-simulator → holodeck-rust  
**Connection:** **None observed**  
- `fleet-simulator` is a Python‑based synthetic data generator.  
- No Rust dependencies, no HTTP client code, no references to `holodeck` or MUD in its source.  
- Simulator focuses on fleet‑level state transitions, not per‑agent sentiment/NPC interaction.  

**Integration Gap:** These two components serve different domains (simulation vs. interactive environment). No current integration; could be a future bridge.  

## 5. holodeck-rust → plato-ensign  
**Connection:** **None observed**  
- `holodeck-rust` is a standalone MUD server (Rust).  
- `plato-ensign` is a Python‑based LoRA exporter.  
- No shared code, no configuration cross‑references.  

**Integration Gap:** Significant. Holodeck interactions are not currently captured as training tiles, so they cannot be exported to ensigns.  

## 6. fleet-simulator → plato-ensign  
**Connection:** **Indirect via plato-torch**  
- `fleet-simulator` data flows into `plato-torch` rooms.  
- Rooms can export to `plato-ensign`.  
- No direct simulator‑to‑ensign pathway.  

**Integration Gap:** Not a problem — the pipeline is designed to flow through rooms.
