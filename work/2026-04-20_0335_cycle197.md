# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 197  
**Phase:** 4 — Build & Test  
**Status:** Updated map based on actual code inspection of plato-torch, fleet-simulator, holodeck-rust, and plato-ensign.

Based on reading the actual code in the fleet repos, here is the current state of integration between the four key systems.

## 1. **plato-torch** → **plato-ensign** → **fleet-simulator**
**Connection:** **Strong, bidirectional data flow.**  
- **plato-torch** (`DeadbandRoom` and other presets) generates training tiles (JSON interaction logs).  
- **plato-ensign** ingests these tiles via `TileLoader` and exports them as LoRA adapters (`export_lora.py`).  
- **fleet-simulator** can load these LoRAs via `Simulation.load_adapter()` to influence agent behavior in simulated environments.  
- **Evidence:** `plato-ensign/README.md` states: "Exports room experience into LoRA adapters for fleet-simulator." Code confirms tile ingestion and LoRA export scripts exist.

## 2. **fleet-simulator** → **plato-torch**  
**Connection:** **Weak, conceptual.**  
- The simulator can generate synthetic training data (simulated agent interactions).  
- This data could be formatted as tiles and fed back into a plato-torch room for training, but no automated pipeline exists in the code.  
- **Evidence:** `fleet-simulator/src/simulation.rs` has `generate_training_data()` method, but no export to plato-torch tile format is implemented.

## 3. **holodeck-rust** → **plato-torch**  
**Connection:** **None (identified integration gap).**  
- **holodeck-rust** is a sentiment-aware MUD server that generates NPC dialogue and player interactions.  
- These interactions are rich, narrative training data, but they are not currently captured as tiles.  
- **Evidence:** `holodeck-rust/src/server.rs` logs events to console and TCP streams, but no tile export mechanism exists. No import hooks in plato-torch for holodeck data.

## 4. **plato-torch** → **holodeck-rust**  
**Connection:** **None (identified integration gap).**  
- A trained ensign (LoRA) from plato-torch could be used to influence NPC behavior in holodeck (e.g., making NPCs follow Deadband Protocol).  
- **Evidence:** `holodeck-rust` uses hard-coded sentiment logic; no plugin system for external AI adapters is present.

## 5. **Ghost Tiles (P0 Knowledge)** → **Any System**  
**Connection:** **Theoretical, not implemented.**  
- The concept of "ghost tiles" (dead agents' lessons) exists in fleet doctrine.  
- No system currently ingests or applies ghost tiles as P0 constraints.  
- **Evidence:** Mentioned in `FLEET-CONTEXT.md` but no code in plato-torch, plato-ensign, or fleet-simulator references ghost tiles.

## 6. **cudaclaw / flux-runtime** → **Other Systems**  
**Connection:** **Independent.**  
- These are low-level runtimes (GPU-resident agent, bytecode VM).  
- They are not directly integrated with the higher-level training loops. They could host agents that participate in rooms or simulations, but no bridging code exists.

## Integration Gaps (P1 Opportunities — Safe Channels to Build)
1. **holodeck-rust → plato-torch tile pipeline**  
   - Safe channel: Add a `TileExporter` module to holodeck that formats interactions as JSON tiles.  
   - plato-torch already ingests tiles; this would be a new data source.

2. **plato-ensign LoRA → holodeck-rust NPC AI**  
   - Safe channel: Extend holodeck with a `LoraNPC` module that loads adapters and uses them for dialogue generation.  
   - Requires minimal changes to holodeck's NPC trait system.

3. **fleet-simulator → plato-torch automated training loop**  
   - Safe channel: Implement a `TileGenerator` in fleet-simulator that outputs tiles in the exact format plato-torch expects.  
   - Creates a closed-loop training system: simulate → tile → train → simulate with new adapter.

4. **GhostInjector for P0 constraint propagation**  
   - Safe channel: Build a `GhostInjector` service that reads a repository of ghost tiles (P0 negatives) and injects them as initial constraints into new DeadbandRoom sessions.  
   - This aligns with fleet doctrine: "Map negative space first."
