## 1. plato-torch (Training Rooms)

**Location:** `plato-torch/`  
**Purpose:** 25 training room presets, self-training rooms.  
**Key Files:**  
- `rooms/deadband_room.py` — Deadband Protocol training room.  
- `rooms/__init__.py` — Room registry.  
- `trainer/self_train.py` — Self‑training loop.  
- `config/room_config.yaml` — Room hyperparameters.  

**Connections:**  
- **plato‑ensign:** Exports room experience to LoRA adapters.  
  - `plato‑torch/export/lora_exporter.py` imports `plato_ensign.core.AdapterBuilder`.  
  - `plato‑torch/config/export_config.yaml` references `ensign_adapter_path`.  
- **fleet‑simulator:** Provides synthetic training data.  
  - `plato‑torch/data/simulated_dataset.py` imports `fleet_simulator.scenarios`.  
  - Room configs include `use_simulated_data: true`.  
- **holodeck‑rust:** Not directly imported. No Rust bindings in Python code.  
- **cudaclaw:** GPU‑resident runtime referenced in `plato‑torch/hardware/cuda_utils.py` for tensor operations.  
- **flux‑runtime:** Not referenced.  

**Integration Gaps:**  
- No live connection to `holodeck‑rust`. Rooms train on static datasets, not live MUD interactions.  
- `flux‑runtime` bytecode VM not used for room orchestration.  
- `constraint‑theory‑core` geometric snapping not integrated.

## 2. fleet‑simulator (Synthetic Training Data)

**Location:** `fleet‑simulator/`  
**Purpose:** Generate fleet simulation data for training.  
**Key Files:**  
- `scenarios/deadband_scenario.rs` — Deadband Protocol simulations.  
- `agents/sim_agent.rs` — Simulated agent behaviors.  
- `output/json_exporter.rs` — Export to JSON for Python consumption.  

**Connections:**  
- **plato‑torch:** Provides training data via JSON export.  
  - `fleet‑simulator/Cargo.toml` includes `serde_json` for interoperability.  
  - Python side: `plato‑torch/data/simulated_dataset.py` reads `*.json` outputs.  
- **holodeck‑rust:** Shared Rust crates (`serde`, `tokio`), but no direct API calls.  
- **plato‑ensign:** No direct link. Simulator does not export LoRA adapters.  
- **constraint‑theory‑core:** Used in `fleet‑simulator/scenarios/constraint_scenario.rs` for geometric agent movement.  

**Integration Gaps:**  
- Simulator runs offline; no real‑time data stream to training rooms.  
- No direct coupling with `plato‑ensign`—training data passes through plato‑torch first.

## 3. holodeck‑rust (Sentiment‑Aware MUD Server)

**Location:** `holodeck‑rust/`  
**Purpose:** Multi‑user dungeon server with NPC sentiment modeling.  
**Key Files:**  
- `src/npc/sentiment.rs` — NPC emotional state machine.  
- `src/world/room.rs` — World geometry and navigation.  
- `src/protocol/telnet.rs` — Telnet server for client connections.  

**Connections:**  
- **constraint‑theory‑core:** Used in `holodeck‑rust/src/world/pathfinding.rs` for spatial snapping.  
- **fleet‑simulator:** Shared Rust dependencies (`tokio`, `serde`), but no runtime integration.  
- **plato‑torch:** No connection. Holodeck is standalone Rust server.  
- **plato‑ensign:** No connection.  
- **flux‑runtime:** Not referenced.  

**Integration Gaps:**  
- Holodeck operates in isolation; no training data flows to plato‑torch.  
- NPC sentiment engine not exposed as a service for other fleet components.  
- No GPU integration (`cudaclaw`).

## 4. plato‑ensign (LoRA Adapter Export)

**Location:** `plato‑ensign/`  
**Purpose:** Export room experience into LoRA adapters.  
**Key Files:**  
- `core/adapter_builder.py` — Builds LoRA adapters from training logs.  
- `export/export_hook.py` — Integration hook for plato‑torch.  
- `config/adapter_config.yaml` — Adapter hyperparameters.  

**Connections:**  
- **plato‑torch:** Primary consumer.  
  - `plato‑ensign/export/export_hook.py` registered as callback in `plato‑torch/rooms/base_room.py`.  
  - Adapter configs shared via YAML.  
- **fleet‑simulator:** No direct link.  
- **holodeck‑rust:** No connection.  
- **cudaclaw:** Used for adapter fine‑tuning on GPU (`plato‑ensign/core/cuda_train.py`).  

**Integration Gaps:**  
- Only works with plato‑torch rooms; cannot export from holodeck or simulator directly.  
- No adapter versioning or registry service.

## 6. Identified Integration Gaps (P0 / P1)

**P0 (Negative Space — What NOT to do):**  
- Do not force‑integrate holodeck‑rust with plato‑torch without a clear data contract.  
- Do not assume flux‑runtime is required—currently unused.  

**P1 (Safe Channels — Where to Integrate):**  
- **GhostInjector → holodeck:** Add a `ghost_injector` module to holodeck‑rust that reads ghost tiles (JSON) and influences NPC sentiment.  
- **DeadbandRoom → plato‑relay:** Extend `deadband_room.py` to publish training metrics to a message queue (plato‑relay).  
- **Holodeck → Training Data:** Add a `holodeck_log_exporter` that streams NPC interactions to plato‑torch as incremental datasets.  

**P2 (Optimization — Best Path):**  
- Implement the above integrations as optional plugins to avoid breaking existing functionality.  
- Use shared JSON schemas for data interchange.
