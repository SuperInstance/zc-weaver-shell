## 1. Verified Current Connections (What's Connected)

Based on reading the actual fleet repos (`fleet-knowledge/`), here are the **verified** connections between the four key technologies:

### **plato-torch ↔ plato-ensign** (Strong, Direct Connection)
- **Connection Type:** Direct dependency & data pipeline.
- **Evidence:** `plato-ensign` is a Python package that imports and extends `plato-torch`. Its primary function is to export trained room states (from `plato-torch` rooms) into LoRA adapters.
- **Data Flow:** A `plato-torch` training room accumulates interaction "tiles" → `plato-ensign` processes these tiles, creates a training dataset, and fine-tunes a base model → Output is a `.safetensors` LoRA file.
- **Status:** **Connected and operational.** This is a core, documented pipeline.

### **plato-torch ↔ fleet-simulator** (Indirect, Potential Connection)
- **Connection Type:** Potential data source, not a direct integration.
- **Evidence:** `fleet-simulator` is described as generating "training data for the fleet." `plato-torch` rooms require training tiles (interaction data). The simulator *could* be a source of synthetic tiles.
- **Data Flow (Potential):** `fleet-simulator` generates simulated agent interactions → These are formatted as tiles → Ingested into a `plato-torch` room for training.
- **Status:** **Not directly wired.** No import statements or direct API calls found. This is a logical integration point but not yet implemented.

### **holodeck-rust ↔ fleet-simulator** (Weak, Conceptual Overlap)
- **Connection Type:** Conceptual siblings in simulation.
- **Evidence:** Both are simulation environments. `holodeck-rust` is a sentiment-aware MUD (Multi-User Dungeon) for human-AI interaction. `fleet-simulator` appears focused on multi-agent fleet dynamics.
- **Data Flow:** None observed. They are separate codebases (Rust vs. likely Python/JS) with different purposes.
- **Status:** **Not connected.** Separate projects under the same fleet umbrella.

### **holodeck-rust ↔ plato-torch** (Indirect, Potential Connection)
- **Connection Type:** Potential tile source.
- **Evidence:** `holodeck-rust` generates NPC interactions with sentiment. These interactions could be captured as "tiles" and sent to a `plato-torch` room for training specialized ensigns (e.g., for dialogue or sentiment analysis).
- **Data Flow (Potential):** `holodeck-rust` interaction log → Tile formatter → `plato-torch` room ingestion.
- **Status:** **Not directly wired.** A clear integration opportunity.

### **plato-ensign ↔ Others** (Downstream Consumer)
- **Connection Type:** Output consumer.
- **Evidence:** `plato-ensign` consumes the output of `plato-torch`. Its outputs (LoRAs) could theoretically be deployed anywhere, including back into `holodeck-rust` NPCs or agents within `fleet-simulator`.
- **Status:** **Connected upstream (to plato-torch),** but its outputs are not automatically integrated downstream.

## 2. Identified Integration Gaps (What's Not Connected)

### **P1 Gap: Synthetic Tile Pipeline**
- **Gap:** No automated pipeline from `fleet-simulator` (synthetic multi-agent data) to `plato-torch` (training rooms).
- **Action:** Build a `TileBridge` service. It would:
    1.  Subscribe to `fleet-simulator` event streams.
    2.  Format events into standard tile schema (`{actor, action, result, sentiment?}`).
    3.  POST tiles to a target `plato-torch` room via its API.

### **P1 Gap: Holodeck Experience Capture**
- **Gap:** Rich interactions in `holodeck-rust` are not captured as training tiles.
- **Action:** Instrument `holodeck-rust` with a `TileLogger` module. It would:
    1.  Log player-NPC dialogues and sentiment states.
    2.  Batch and send them to a designated `plato-torch` "dialogue room."
    3.  Enable training of specialist "dialogue ensigns."

### **P2 Gap: Ensign Feedback Loop**
- **Gap:** LoRAs created by `plato-ensign` are not automatically deployed to create a self-improving loop.
- **Action:** Create an `EnsignDeployer` that:
    1.  Watches for new `.safetensors` outputs from `plato-ensign`.
    2.  Loads them into a runtime (e.g., `cudaclaw` for GPU agents, or a `holodeck-rust` NPC brain).
    3.  Measures performance, generating new tiles that feed back to `plato-torch`.

### **P0 Constraint: Avoid Tight Coupling**
- **Negative Space:** Do not hardcode dependencies. The fleet uses loose, service-oriented patterns.
- **Safe Channel:** All integrations should use:
    - File-based outputs (e.g., JSONL tile dumps).
    - HTTP APIs (where they exist, e.g., `plato-torch` room ingestion).
    - Message queues (for scalability).
- **Implementation Path:** Build standalone "bridge" services that translate between systems.

## 3. Recommended Phase 4 Action (Weaver Task)

**Task: Wire GhostInjector into holodeck**

1.  **Context:** The `GhostInjector` is a concept from fleet doctrine—it injects "ghost tiles" (lessons from dead agents) as P0 knowledge.
2.  **Connection Point:** `holodeck-rust` has sentiment-aware NPCs. Their behavior could be seeded or influenced by ghost tiles (e.g., "avoid this fatal action").
3.  **Actionable Integration:**
    - **Step 1:** Locate or create a `ghost-tiles.jsonl` file in the fleet knowledge base (contains historical failure tiles).
    - **Step 2:** Write a small Rust module (`ghost_injector.rs`) for `holodeck-rust` that:
        - Reads a sample of ghost tiles on server start.
        - Converts them into initial NPC memory or behavior constraints (e.g., "if player says X, avoid response Y").
    - **Step 3:** Test by running `holodeck-rust` and observing if NPCs exhibit caution based on ghost tile content.

This creates a tangible, testable integration that embodies fleet doctrine (P0 knowledge injection) and connects the conceptual "ghost" system to a live environment.
