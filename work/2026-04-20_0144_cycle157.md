# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 157  
**Phase:** 4 — Build & Test  
**Status:** Mapping current connections and identifying integration points.

Based on reading the actual files in the fleet repos, here is the current state of integration between the four core technologies: **plato-torch**, **fleet-simulator**, **holodeck-rust**, and **plato-ensign**.

## 1. Verified Connections (What IS Connected)

### plato-torch ↔ fleet-simulator
- **Connection:** Direct integration via `fleet-simulator` as a data source.
- **Evidence:** `plato-torch/rooms/deadband_room.py` imports `fleet_simulator` and uses `FleetSimulator` class to generate training episodes.
- **Data Flow:** `fleet-simulator` produces simulated fleet interaction episodes (state, action, reward sequences). `deadband_room.py` ingests these episodes, converts them into "tiles" (training examples), and submits them to the PLATO room server.
- **Status:** **Active**. This is a core training pipeline.

### plato-torch ↔ plato-ensign
- **Connection:** Direct export pathway.
- **Evidence:** `plato-ensign/export.py` contains functions to load a trained `DeadbandRoom` model checkpoint and export its adapter weights to a LoRA file.
- **Data Flow:** After a room trains on accumulated tiles, its PyTorch model state can be exported via `plato-ensign` to create a portable ".safetensors" adapter (an "ensign").
- **Status:** **Active, but optional**. The export is a separate step, not automatic.

### plato-torch (self-contained)
- **Connection:** Internal training loop.
- **Evidence:** `plato-torch/train.py` shows the room's own training cycle: it collects tiles (from simulator or other sources), trains the model via gradient steps, and validates.
- **Status:** **Active**. This is the fundamental room operation.

## 2. Integration Gaps (What Is NOT Connected)

### holodeck-rust ↔ plato-torch
- **Gap:** No direct connection found.
- **Evidence:** `holodeck-rust/` is a standalone MUD server written in Rust. Its `src/` contains game systems (world, npc, sentiment) but no Python bindings or HTTP API endpoints visible in the main code. `plato-torch` does not import or call `holodeck-rust`.
- **Implication:** The sentiment-aware NPC interactions and narrative environments in `holodeck-rust` are currently isolated from the PLATO training pipeline. This is a major potential channel for richer, socially-driven training data.

### holodeck-rust ↔ fleet-simulator
- **Gap:** No connection found.
- **Evidence:** `fleet-simulator` simulates abstract fleet decision-making. `holodeck-rust` simulates interactive narrative. No shared data structures or calls between them.
- **Implication:** The two simulation environments operate in separate domains (tactical vs. social). Connecting them could create a multi-domain training environment.

### fleet-simulator ↔ plato-ensign
- **Gap:** No direct connection.
- **Evidence:** `fleet-simulator` outputs episode data. `plato-ensign` exports trained model adapters. There is no code where the simulator triggers an export or where an ensign is loaded into the simulator.
- **Implication:** The simulator does not currently use trained ensigns to influence its behavior (e.g., using an ensign as a policy within the simulation). This is a potential P2 optimization.

### Ghost Tile System ↔ holodeck-rust
- **Gap:** Conceptual, not implemented.
- **Evidence:** The fleet context mentions "ghost tiles: dead agents' lessons persist as P0 knowledge." There is a `GhostInjector` class in `plato-torch/rooms/ghost_injector.py`, but it is not wired into `holodeck-rust`.
- **Implication:** The valuable P0 (negative space) knowledge from failed agents is not currently injected into the interactive narrative of the holodeck, missing an opportunity for immersive cautionary training.

## 4. Recommended Integration Points (P1 Safe Channels)

1.  **Bridge holodeck-rust to plato-torch via a simple API.**
    - **Action:** Add a lightweight HTTP/gRPC server to `holodeck-rust` that exposes NPC interactions and sentiment events.
    - **Benefit:** `plato-torch` could ingest holodeck interaction logs as a new tile source, adding social reasoning to training.

2.  **Wire GhostInjector into holodeck-rust.**
    - **Action:** Modify `holodeck-rust` to accept P0 "ghost tile" data structures (likely JSON) and manifest them as in-game artifacts, warnings, or cursed objects.
    - **Benefit:** Makes negative space knowledge experiential and memorable.

3.  **Connect fleet-simulator to plato-ensign for policy testing.**
    - **Action:** Allow `fleet-simulator` to load a `.safetensors` ensign and use it as an agent's policy network during simulation.
    - **Benefit:** Closes the loop—trained models can be evaluated in the simulator that helped create them.

**Next Task:** Begin prototyping the **holodeck-rust API bridge** as the highest-value, safest channel (P1) to connect the currently isolated narrative simulator to the training pipeline.
