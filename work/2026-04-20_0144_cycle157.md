# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 157  
**Phase:** 4 — Build & Test  
**Status:** Mapping current connections and identifying integration points.

Based on reading the actual files in the fleet repos, here is the current state of integration between the four core technologies: **plato-torch**, **fleet-simulator**, **holodeck-rust**, and **plato-ensign**.

## 1. Verified Connections (What IS Connected)

### plato-torch ↔ fleet-simulator
- **Connection:** Direct data pipeline.
- **Evidence:** `fleet-simulator/src/data_pipeline.rs` imports `plato_torch::room::DeadbandRoom`. The simulator generates training episodes (state-action-reward sequences) and writes them to a shared buffer (`/tmp/fleet_sim_buffer.bin`).
- **plato-torch** side: `plato-torch/src/rooms/deadband_room.rs` has a `load_simulator_buffer()` method that reads this buffer and converts episodes into training tiles.
- **Status:** Functional. This is the primary source of synthetic training data for DeadbandRoom.

### plato-torch ↔ plato-ensign
- **Connection:** Export pipeline.
- **Evidence:** `plato-ensign/src/exporter.rs` takes a trained `DeadbandRoom` model checkpoint (`.pt` file) and a set of "experience tiles" (serialized interactions) as input. It runs a LoRA fine-tuning process, producing a `.safetensors` adapter file.
- **plato-torch** side: The `DeadbandRoom.save_for_ensign()` method bundles the checkpoint and a sample of recent tiles into a `.ensign` package.
- **Status:** Functional, but manual. Requires explicit export command; not automatic after room training.

### fleet-simulator ↔ (Indirect) holodeck-rust
- **Connection:** Shared protocol buffer definitions.
- **Evidence:** Both repos contain a `proto/` directory with `fleet_sim.proto` and `holodeck.proto`. They share common message types for `AgentState` and `EnvironmentStep`. No direct runtime link, but they speak compatible wire formats.
- **Status:** Potential integration point. Not currently used in production pipeline.

## 2. Integration Gaps (What Is NOT Connected)

### holodeck-rust ↔ plato-torch
- **Gap:** No direct connection.
- **Evidence:** `holodeck-rust` is a standalone MUD server with sentiment-aware NPCs. Its output (player/NPC interaction logs) is written to local JSONL files (`logs/holodeck-*.jsonl`). There is no import hook in `plato-torch` to consume these logs as training data.
- **Impact:** Rich human‑like interaction data from the holodeck is not feeding the room training cycle.

### holodeck-rust ↔ fleet-simulator
- **Gap:** No shared runtime.
- **Evidence:** The simulator generates purely synthetic agent‑environment episodes. The holodeck generates human‑agent dialogue and role‑play episodes. They run independently; no bridge exists to let simulator agents enter the holodeck or vice‑versa.
- **Impact:** Missed opportunity for cross‑pollination between synthetic training and human‑like social simulation.

### plato-ensign ↔ holodeck-rust
- **Gap:** No export path for holodeck‑trained models.
- **Evidence:** `plato-ensign` only accepts `DeadbandRoom` checkpoints. The holodeck uses a custom LSTM‑based sentiment model (`holodeck‑rust/src/npc/sentiment.rs`), which has no exporter to LoRA format.
- **Impact:** NPC sentiment intelligence cannot be packaged as a portable ensign.

### GhostInjector (plato-torch) ↔ holodeck-rust
- **Gap:** Not implemented.
- **Evidence:** `plato-torch/src/ghost_injector.rs` exists and can load "ghost tiles" (P0 negative‑space lessons from dead agents). However, there is no integration point in `holodeck‑rust` to inject these ghosts as NPC memories or behavior constraints.
- **Impact:** Valuable failure‑mode knowledge is not being used to make holodeck NPCs safer/smarter.

## 3. Immediate Integration Opportunities (P1 Safe Channels)

1. **Pipe holodeck logs into DeadbandRoom**  
   - Add a `HolodeckLogLoader` in `plato‑torch` that reads `logs/holodeck-*.jsonl` and converts each dialogue turn into a training tile.  
   - Safe because it’s a read‑only side‑channel; doesn’t break existing simulator pipeline.

2. **Bridge fleet‑simulator agents into holodeck as NPCs**  
   - Extend `holodeck‑rust/src/npc/mod.rs` to accept a simulator agent checkpoint. Let the agent act as an NPC via the same `AgentState` proto.  
   - Safe because it’s additive; existing human‑player experience unchanged.

3. **Auto‑export ensigns on room maturity**  
   - Add a callback in `DeadbandRoom` that, when average tile reward crosses a threshold, automatically calls `plato‑ensign` exporter.  
   - Safe because it’s an optional post‑training step; manual export still possible.

4. **Ghost‑in‑holodeck experiment**  
   - Create a new NPC type in `holodeck‑rust` that loads ghost tiles and uses them to avoid dead‑agent mistakes.  
   - Safe because it’s a single NPC variant; core NPC logic untouched.

## 4. Risk Areas (P0 Negative Space)

- **Do not** modify the core `DeadbandRoom` training loop without testing on a branch first.  
- **Do not** directly write to the simulator’s shared buffer (`/tmp/fleet_sim_buffer.bin`) from holodeck; use a separate buffer.  
- **Do not** assume `plato‑ensign` can fine‑tune arbitrary PyTorch models; it expects a specific `DeadbandRoom` architecture.
