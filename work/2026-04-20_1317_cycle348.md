# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 348

**Status:** Direct file inspection of fleet repositories. Focus on actual imports, configuration references, and shared data structures.

## 1. Plato-Torch (Training Rooms)
**Location:** `plato-torch/`  
**Purpose:** 25 training room presets, self-training rooms, tile ingestion.  
**Key Files:**  
- `presets/DeadbandRoom.py` — P0/P1/P2 protocol implementation  
- `presets/__init__.py` — exports all presets  
- `plato_torch/room_server.py` — HTTP server for tile submission  
- `plato_torch/tile_store.py` — SQLite storage of interaction tiles  

**Connections Found:**  
- **To Plato-Ensign:** `presets/DeadbandRoom.py` imports `from plato_ensign import export_room` (line 12). Room can export its experience as a LoRA adapter.  
- **To Fleet-Simulator:** No direct import found. However, `presets/DeadbandRoom.py` references "fleet simulation" in docstring (line 5) as a potential data source.  
- **To Holodeck-Rust:** No import found. No shared data structures evident in code.  
- **Internal:** Rooms submit tiles to `room_server.py` via HTTP POST `/submit_tile`.  

## 2. Plato-Ensign (LoRA Export)  
**Location:** `plato-ensign/`  
**Purpose:** Exports room experience into LoRA adapters.  
**Key Files:**  
- `plato_ensign/export.py` — contains `export_room()` function  
- `plato_ensign/lora_train.py` — training logic using PyTorch and peft  

**Connections Found:**  
- **From Plato-Torch:** Imported by `DeadbandRoom`.  
- **To Fleet-Simulator:** No import found.  
- **To Holodeck-Rust:** No import found.  
- **Internal:** Reads tiles from SQLite (via `plato_torch.tile_store`), trains LoRA, saves `.safetensors`.  

## 3. Fleet-Simulator (Training Data Generation)  
**Location:** `fleet-simulator/`  
**Purpose:** Generates synthetic training data for fleet agents.  
**Key Files:**  
- `simulator/core.py` — main simulation loop  
- `simulator/agents/` — agent behavior definitions  
- `simulator/output/` — writes JSONL training data  

**Connections Found:**  
- **To Plato-Torch:** No import found. Output is JSONL files, not integrated with tile ingestion pipeline.  
- **To Plato-Ensign:** No import found.  
- **To Holodeck-Rust:** No import found.  
- **Internal:** Standalone simulation; outputs to `data/` directory.  

## 4. Holodeck-Rust (Sentiment-Aware MUD)  
**Location:** `holodeck-rust/`  
**Purpose:** Multi‑User Dungeon server with NPCs that react to player sentiment.  
**Key Files:**  
- `src/server.rs` — TCP server, event loop  
- `src/npc/sentiment.rs` — sentiment analysis for NPC responses  
- `src/world/rooms.rs` — room definitions  

**Connections Found:**  
- **To Plato-Torch:** No import found. No HTTP endpoint for tile submission.  
- **To Plato-Ensign:** No import found.  
- **To Fleet-Simulator:** No import found.  
- **Internal:** Pure Rust; uses `tokio` for async, `serde` for JSON.  

## Integration Gaps (What’s Not Connected)  
1. **Fleet‑Simulator → Plato‑Torch:** Simulation data (JSONL) is not ingested as tiles. No pipeline exists to convert simulator outputs into room interactions.  
2. **Holodeck‑Rust → Plato‑Torch:** NPC interactions are not captured as tiles. No tile‑submission endpoint in holodeck.  
3. **Plato‑Torch → Holodeck‑Rust:** No mechanism to influence NPC behavior based on room training.  
4. **Plato‑Ensign → Fleet‑Simulator:** LoRA adapters are not used to improve simulator agents.  
5. **GhostInjector (mentioned in backlog)** — not found in any repo; likely a planned component.  

## Verified Data Flow (Current)  
```
Agent → [Tile] → Plato‑Torch (HTTP) → SQLite → Plato‑Ensign → LoRA
```
**Missing flows:**  
- Fleet‑Simulator → Plato‑Torch  
- Holodeck‑Rust → Plato‑Torch  
- Plato‑Torch → Holodeck‑Rust (NPC tuning)
