# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 365  
**Phase:** 4 — Build & Test  
**Status:** Direct file inspection of fleet repositories. Focus on actual imports, configuration references, and shared data structures.  

## 1. plato-torch → fleet-simulator  
**Connection:** Strong, direct integration.  
- **plato-torch/rooms/DeadbandRoom.py** imports `from fleet_simulator.simulator import FleetSimulator`.  
- **plato-torch/rooms/__init__.py** registers `DeadbandRoom` as a preset.  
- **fleet-simulator/src/simulator.rs** exports a Python module via `pyo3`, exposing `FleetSimulator` class to Python.  
- **Data flow:** `DeadbandRoom` instantiates `FleetSimulator`, steps the simulation, and uses the resulting state (agent positions, actions) as training environment context.  

**Gap:** No bidirectional control flow observed. The simulator appears to be a passive data source for the room; room does not feed back into simulator configuration dynamically.  

## 2. plato-torch → holodeck-rust  
**Connection:** Indirect, via shared concepts (sentiment, NPCs) but no direct imports or API calls found.  
- **plato-torch/rooms/** contains no import of `holodeck` or `mud` modules.  
- **holodeck-rust/src/lib.rs** exposes `MudServer` and `Npc` structs, but no Python bindings (`pyo3`) present in current code.  
- **Conceptual link:** Both projects involve agent‑environment interaction, but `holodeck-rust` is a standalone Rust MUD server; `plato-torch` is a Python training framework.  

**Gap:** No working integration. To wire them, either:  
   - Add `pyo3` bindings to `holodeck-rust` and import in `plato-torch`, or  
   - Create a bridge service (e.g., REST/gRPC) that `plato-torch` rooms can call.  

## 3. plato-torch → plato-ensign  
**Connection:** Direct export pipeline.  
- **plato-ensign/src/lib.rs** provides `export_to_lora` function.  
- **plato-torch/training/trainer.py** calls `ensign.export_to_lora` after room training loop completes.  
- **Data flow:** Room checkpoints (PyTorch `.pt` files) are converted to LoRA adapters via `plato-ensign` Rust‑accelerated routines.  

**Gap:** The export is a final‑step batch process. No real‑time streaming of tiles to `plato-ensign` observed.  

## 4. fleet-simulator → holodeck-rust  
**Connection:** None found.  
- **fleet-simulator/src/** has no references to `holodeck`, `mud`, or `npc`.  
- **holodeck-rust/Cargo.toml** does not depend on `fleet-simulator`.  
- Both are Rust crates but serve separate purposes: simulation‑vs‑interactive narrative.  

**Gap:** No shared agent state or event passing. Could be linked by having `holodeck-rust` NPCs react to `fleet-simulator` world events (e.g., market fluctuations).  

## 5. fleet-simulator → plato-ensign  
**Connection:** None found.  
- **plato-ensign/src/lib.rs** only handles model‑weight transformations, not simulation data.  
- **fleet-simulator** outputs simulation traces (JSON/logs), not directly consumed by `plato-ensign`.  

**Gap:** Simulation traces could be used to generate synthetic training tiles, but no pipeline exists.  

## 6. holodeck-rust → plato-ensign  
**Connection:** None found.  
- **holodeck-rust** stores NPC sentiment state in‑memory (`Npc.sentiment` field).  
- **plato-ensign** expects PyTorch tensors, not live game state.  

**Gap:** NPC sentiment trajectories could be exported as training sequences for emotion‑aware models, but no exporter is implemented.
