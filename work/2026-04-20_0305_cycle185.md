# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 185  
**Phase:** 4 — Build & Test  
**Status:** Mapping current connections and identifying integration points.

Based on reading the actual files in the fleet repos, here is the current state of integration between the four core technologies.

## 1. plato-torch
**Role:** Training room server with 25 preset rooms, including DeadbandRoom.  
**Connections:**  
- **Direct:** Exports training data (tiles) to `plato-ensign` via `export_room_experience()` function.  
- **Indirect:** `DeadbandRoom` preset is designed to simulate fleet decision-making; could feed scenarios to `fleet-simulator`.  
- **Gap:** No direct link to `holodeck-rust` for sentiment-aware NPC interactions. Rooms are static training environments.

## 2. fleet-simulator
**Role:** Generates training data by simulating fleet agent interactions.  
**Connections:**  
- **Direct:** Outputs simulation logs in JSON format.  
- **Indirect:** Logs could be ingested by `plato-torch` rooms as training tiles, but no import pipeline is documented.  
- **Gap:** No connection to `holodeck-rust` for dynamic NPC behavior during simulations.

## 3. holodeck-rust
**Role:** MUD server with sentiment-aware NPCs using a "sentiment lattice."  
**Connections:**  
- **Direct:** None to other three systems.  
- **Indirect:** NPC sentiment states could provide rich training data for `plato-torch` rooms, but no export mechanism exists.  
- **Gap:** Largest isolation. No integration points with training or simulation systems.

## 4. plato-ensign
**Role:** Exports room experience from `plato-torch` into LoRA adapters.  
**Connections:**  
- **Direct:** Imports tiles from `plato-torch` via `RoomExperience` struct.  
- **Indirect:** Could theoretically fine-tune models for `holodeck-rust` NPCs or `fleet-simulator` agents, but no hooks are implemented.  
- **Gap:** Only consumes from `plato-torch`. Does not write back or share adapters with other systems.

## Integration Map (Current State)
```
plato-torch → plato-ensign (one-way, tiles → LoRA)
fleet-simulator → (standalone, JSON logs)
holodeck-rust → (standalone, sentiment NPCs)
```

## Identified Integration Points (P1 Safe Channels)
1. **plato-torch → fleet-simulator:** Add a `SimulationRoom` preset that reads `fleet-simulator` logs as tile input.
2. **holodeck-rust → plato-torch:** Create a `HolodeckBridge` that exports NPC interaction logs as training tiles.
3. **plato-ensign → holodeck-rust:** Allow loading of LoRA adapters to modify NPC behavior models.
4. **fleet-simulator → holodeck-rust:** Inject simulated agents into the MUD as temporary NPCs for live testing.

## Next Steps (P2 Optimization within Channels)
- **Priority 1:** Build `HolodeckBridge` to connect `holodeck-rust` sentiment data to `plato-torch` tiles.
- **Priority 2:** Extend `plato-ensign` to export adapters in a format `holodeck-rust` can load.
- **Test:** Validate pipeline with a small integration test using existing DeadbandRoom data.

**Note:** This map is based on actual repo READMEs and source file structures observed in `fleet-knowledge/`. Integration gaps represent opportunities, not failures.
