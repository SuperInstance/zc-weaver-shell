# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 185  
**Phase:** 4 — Build & Test  
**Status:** Mapping current connections and identifying integration points.

Based on reading the actual files in the fleet repos, here is the current state of integration between the four core systems.

## 1. plato-torch (Training Rooms)
**Purpose:** Provides 25 preset training rooms (including `DeadbandRoom`) for agent self-training via interaction tiles.  
**Key Files:** `presets/deadband_room.py`, `core/room.py`, `core/tile.py`  
**Current Connections:**
- **Connected to:** `plato-ensign` via explicit export path in `presets/deadband_room.py` (line 45: `export_path = "ensign_exports/"`). Room can export its trained adapter.
- **Connected to:** `fleet-simulator` indirectly via shared `Tile` data structure. Both consume/produce `Tile` objects (JSON with `content`, `metadata`, `vector`).
- **Not directly connected to:** `holodeck-rust`. No imports, API calls, or shared data paths found.

## 2. fleet-simulator (Fleet Simulation)
**Purpose:** Generates training data by simulating multi-agent fleet interactions.  
**Key Files:** `src/simulation.rs`, `src/tile_generator.rs`  
**Current Connections:**
- **Connected to:** `plato-torch` via shared `Tile` schema. Simulator outputs `Tile` objects that could be ingested by a PLATO room.
- **Connected to:** `holodeck-rust` via **planned but not implemented** integration. `src/lib.rs` mentions `// TODO: Connect to holodeck for live NPC sentiment` but no actual code.
- **Not connected to:** `plato-ensign`. No export or adapter training logic in simulator.

## 3. holodeck-rust (Sentiment-Aware MUD)
**Purpose:** Multi‑user dungeon server with NPCs that react to player sentiment.  
**Key Files:** `src/npc/sentiment.rs`, `src/server.rs`  
**Current Connections:**
- **Connected to:** `fleet-simulator` only as a TODO (see above). No active data flow.
- **Not connected to:** `plato-torch` or `plato-ensign`. Standalone server with its own event loop and TCP API.

## 4. plato-ensign (LoRA Adapter Exporter)
**Purpose:** Exports a trained room’s experience into a LoRA adapter file.  
**Key Files:** `export.py`, `train_lora.py`  
**Current Connections:**
- **Connected to:** `plato-torch` via import of `DeadbandRoom` and direct call to `room.export_adapter()`.
- **Not connected to:** `fleet-simulator` or `holodeck-rust`. Exporter only works on PLATO room objects.

## Integration Gaps (P1 Safe Channels)
1. **holodeck‑rust ↔ plato‑torch**  
   - Gap: No bridge between MUD interactions and training tiles.  
   - Safe channel: Could implement a `HolodeckTileGenerator` that converts NPC dialogue/events into `Tile` objects for a PLATO room.

2. **fleet‑simulator → plato‑ensign**  
   - Gap: Simulated fleet data cannot be directly turned into an ensign adapter.  
   - Safe channel: Add a `SimulationRoom` preset in plato‑torch that ingests simulator tiles, then export via plato‑ensign.

3. **plato‑ensign → holodeck‑rust**  
   - Gap: Trained adapters are not loaded into holodeck NPCs.  
   - Safe channel: Add a `--load‑adapter` flag to holodeck’s NPC system to infuse sentiment models.

4. **GhostInjector (mentioned in backlog) ↔ holodeck**  
   - Gap: Ghost tiles (P0 knowledge from dead agents) are not injected into holodeck’s memory.  
   - Safe channel: Create a `ghost_memory.db` that holodeck can query during NPC initialization.
