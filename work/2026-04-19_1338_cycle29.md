## 1. Verified Current Connections (What's Connected)

Based on reading the actual fleet repositories (`plato-torch`, `fleet-simulator`, `holodeck-rust`, `plato-ensign`), here are the confirmed integration points:

### **plato-torch ↔ fleet-simulator**
- **Connection:** Strong, bidirectional data flow.
- **Mechanism:** `fleet-simulator` generates synthetic training data (agent interactions, environment states) which is consumed by `plato-torch` training rooms.
- **Evidence:** `plato-torch/src/rooms/deadband_room.py` imports from `fleet_simulator` and uses its `ScenarioGenerator` to populate training episodes.
- **Purpose:** Provides high-volume, varied training tiles for room-based self-training.

### **plato-torch ↔ plato-ensign**
- **Connection:** Direct export pipeline.
- **Mechanism:** `plato-ensign` is a LoRA adapter exporter. It reads the accumulated training state (tiles, weights) from a `plato-torch` room and produces a fine-tuned adapter.
- **Evidence:** `plato-ensign/src/exporter.py` expects a `room_checkpoint_path` from `plato-torch` and outputs a `.safetensors` file.
- **Purpose:** Turns room experience into portable, deployable specialist models (ensigns).

### **holodeck-rust ↔ (Indirect via plato-torch)**
- **Connection:** Weak, aspirational. No direct code integration found.
- **Mechanism:** The `holodeck-rust` README describes it as a "sentiment-aware MUD server" intended to generate rich narrative interactions. The presumed pipeline is: `holodeck-rust` → interaction logs → `plato-torch` tiles.
- **Evidence:** No import statements or shared data structures found between `holodeck-rust` and the other repos. It is a standalone binary.
- **Purpose:** (Intended) To provide human-like, emotionally textured interaction data for training.

## 2. Identified Integration Gaps (What's Not Connected)

### **Gap 1: holodeck-rust is an Island**
- **Problem:** `holodeck-rust` operates in isolation. Its output (NPC dialogues, player actions, sentiment logs) is not automatically fed into the `plato-torch` training pipeline.
- **Consequence:** The fleet's "sentiment-aware" training data potential is untapped. Manual extraction and formatting would be required.
- **Actionable Integration Point:** Build a `HolodeckBridge` service that:
    1. Subscribes to `holodeck-rust` event stream (or parses log files).
    2. Transforms events into the `TrainingTile` schema used by `plato-torch`.
    3. Publishes tiles to a `plato-torch` room via its ingestion API.

### **Gap 2: Ghost Tiles are Theoretical**
- **Problem:** The concept of "ghost tiles" (dead agents' lessons as P0 knowledge) is documented in architecture but has no implementation in the codebase reviewed.
- **Consequence:** Valuable failure state data is not systematically captured or injected into training.
- **Actionable Integration Point:** Create a `GhostInjector` module within `plato-torch` that:
    1. Monitors agent health/termination signals (from `cudaclaw` or service guard).
    2. Captures final state/error context.
    3. Formats it as a high-priority P0 (negative space) tile and injects it into relevant training rooms.

### **Gap 3: plato-ensign Export is Manual**
- **Problem:** The export of an ensign from a room is a manual, one-off CLI command. There is no automated lifecycle (e.g., "export when room validation loss plateaus").
- **Consequence:** Specialist model creation is not integrated into the continuous training loop.
- **Actionable Integration Point:** Implement an `ensign-scheduler` in `plato-torch` that:
    1. Monrooms training metrics.
    2. Triggers an automatic export to `plato-ensign` upon defined criteria (e.g., every N tiles, or after performance convergence).
    3. Registers the new ensign in a fleet model registry.

### **Gap 4: fleet-simulator Scenarios are Static**
- **Problem:** `fleet-simulator` scenarios are predefined. They do not dynamically adapt based on the P0 (negative space) lessons learned in `plato-torch` rooms.
- **Consequence:** Simulation may generate data for already-solved problems, not for newly discovered failure modes.
- **Actionable Integration Point:** Establish a feedback loop:
    1. `plato-torch` rooms periodically publish their top P0 constraints (what not to do).
    2. `fleet-simulator` receives these and biases its `ScenarioGenerator` to create edge cases that probe those constraints.
    3. Creates a self-improving data generation cycle.

## 3. Priority Integration Wiring (For Phase 4 Task)

From the backlog, the most impactful and feasible task for **Cycle 29** is:

**Wire GhostInjector into holodeck** *(Rephrased more accurately: Wire GhostInjector concept into the plato-torch training pipeline)*

1.  **P0 (Negative Space):** Do not modify `holodeck-rust` directly. It is a separate service. Instead, focus on the `plato-torch` ingestion point.
2.  **P1 (Safe Channel):** Extend `plato-torch/src/data/tile_manager.py`. Add a `GhostTileManager` class that listens on a dedicated port or file watcher for termination events from fleet agents.
3.  **P2 (Optimize):** Implement the tile formatting, assign high priority, and inject into the `DeadbandRoom` or a dedicated `P0_RetrospectRoom`.

This creates a real, working integration that closes **Gap 2** and embodies the fleet's "ghost tiles" architecture.
