## 1. Verified Current Connections (What's Connected)

Based on reading the actual fleet repos (`plato-torch`, `fleet-simulator`, `holodeck-rust`, `plato-ensign`):

### **plato-torch ↔ plato-ensign** (Strong, Direct)
- **Connection**: `plato-ensign` is a direct export module for `plato-torch` rooms.
- **Mechanism**: After a room accumulates enough interaction tiles, `plato-ensign` can be invoked to export the room's experience into a LoRA adapter.
- **Evidence**: `plato-ensign/README.md` states: "Exports room experience from plato-torch into LoRA adapters." The `plato-torch` room presets include hooks for ensign export.

### **plato-torch ↔ fleet-simulator** (Indirect, Data Flow)
- **Connection**: `fleet-simulator` generates synthetic training data (simulated fleet interactions).
- **Mechanism**: This data can be formatted into "tiles" and ingested by `plato-torch` rooms for training.
- **Evidence**: `fleet-simulator/src/lib.rs` exports a `generate_training_batch()` function that outputs JSON in tile-compatible format. `plato-torch`'s `TileLoader` can load external JSON.

### **holodeck-rust ↔ (plato-torch / fleet-simulator)** (Potential, Not Yet Wired)
- **Connection**: `holodeck-rust` is a sentiment-aware MUD server that could act as an interactive data source.
- **Current State**: Standalone. No explicit imports or calls to/from `plato-torch` or `fleet-simulator` in code.
- **Evidence**: `holodeck-rust/Cargo.toml` shows no dependencies on other fleet crates. Its NPC interactions are self-contained.

### **plato-ensign → External Models** (Output)
- **Connection**: `plato-ensign` produces LoRA adapters that can be loaded into transformer models (e.g., via Hugging Face `peft`).
- **Mechanism**: Adapters are saved as `.safetensors` files, compatible with standard fine-tuning pipelines.

## 2. Integration Gaps (What's Not Connected)

### **Gap 1: holodeck-rust → plato-torch (Live Tile Stream)**
- **Problem**: `holodeck-rust` generates rich NPC interaction logs with sentiment scores, but these are not automatically fed into a `plato-torch` room as training tiles.
- **Actionable Integration**: Build a `HolodeckTileBridge` service that:
  1. Subscribes to `holodeck-rust` event stream (WebSocket/logfile).
  2. Formats NPC dialogues + sentiment into tile schema.
  3. Posts tiles to a `plato-torch` room via its REST API (if exposed) or writes to shared tile storage.

### **Gap 2: fleet-simulator → holodeck-rust (Scenario Injection)**
- **Problem**: `fleet-simulator` can generate complex multi-agent scenarios, but `holodeck-rust` worlds are static (hand-authored).
- **Actionable Integration**: Create a `ScenarioLoader` for `holodeck-rust` that:
  1. Reads a scenario JSON from `fleet-simulator`.
  2. Spawns NPCs with personalities/objectives derived from the scenario.
  3. Injects scenario-specific room descriptions and quest hooks.

### **Gap 3: plato-ensign → holodeck-rust (NPC Behavior Adaptation)**
- **Problem**: LoRA adapters trained by `plato-ensign` are not used to evolve `holodeck-rust` NPC behavior.
- **Actionable Integration**: Implement a `LiveNPCAdapter` module in `holodeck-rust` that:
  1. Loads a `.safetensors` LoRA trained on specific interaction patterns.
  2. Applies the adapter to the NPC's underlying language model (if LM-driven) or decision network.
  3. Allows NPCs to "learn" from room exports without full retraining.

### **Gap 4: DeadbandRoom (plato-torch) ↔ plato-relay (Missing)**
- **Problem**: `DeadbandRoom` is a preset for self-training with the Deadband Protocol, but it operates in isolation.
- **Actionable Integration**: Wire `DeadbandRoom` to `plato-relay` (if `plato-relay` exists as a message bus) to:
  1. Broadcast P0/P1/P2 findings to other rooms/agents.
  2. Receive external constraint updates (new "negative space" maps).
  3. Enable cross-room deadband synchronization.

## 3. Priority Recommendations for Phase 4 Build

**P0 (Negative Space)**: Do NOT attempt to wire all gaps at once. Do NOT modify core room logic without tests.

**P1 (Safe Channels)**:
1. **Start with Gap 1 (holodeck → plato-torch)**. It's a one-way data flow, low risk to existing systems. Build the bridge as a standalone service.
2. **Then Gap 4 (DeadbandRoom → plato-relay)**. Requires understanding `plato-relay` API, but is a clear extension of existing preset.

**P2 (Optimize)**:
- After P1 channels are proven, consider bidirectional integrations (Gap 2, Gap 3) which require deeper changes to `holodeck-rust` core.
