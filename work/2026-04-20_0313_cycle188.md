Based on reading the actual files in the fleet repos, here is the current state of integration between the four core systems:

## 1. **plato-torch** (Training Rooms)
- **Connected to:** plato-ensign (via export pipeline)
- **Connection Type:** Direct file export — `plato-torch` generates `.pt` checkpoint files that `plato-ensign` can convert to LoRA adapters.
- **Integration Point:** `plato-torch/rooms/DeadbandRoom.py` has an `export_to_ensign()` method that calls `plato_ensign.convert_checkpoint()`.
- **Not Connected to:** fleet-simulator, holodeck-rust (no imports or API calls found).

## 2. **plato-ensign** (LoRA Adapter Exporter)
- **Connected to:** plato-torch (as above)
- **Connection Type:** One-way dependency — `plato-ensign` expects `plato-torch` checkpoint files as input.
- **Integration Point:** `plato_ensign/cli.py` includes a `--checkpoint` argument pointing to `plato-torch` output.
- **Not Connected to:** fleet-simulator, holodeck-rust.

## 3. **fleet-simulator** (Training Data Generator)
- **Connected to:** None (currently isolated)
- **Integration Gap:** No imports of `plato-torch`, `plato-ensign`, or `holodeck-rust`. Simulates fleet interactions but doesn't feed data to training rooms.
- **Potential Channel:** Could output JSON/Protobuf logs that `plato-torch` rooms ingest as training tiles.

## 4. **holodeck-rust** (Sentiment-Aware MUD)
- **Connected to:** None (currently isolated)
- **Integration Gap:** Standalone MUD server. No hooks for `plato-torch` training or `fleet-simulator` events.
- **Potential Channel:** NPC interaction logs could become sentiment-labeled training data for rooms.
