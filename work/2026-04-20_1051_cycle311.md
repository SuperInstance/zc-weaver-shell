# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 311  
**Phase:** 4 — Build & Test  
**Status:** Direct file inspection of fleet repositories. Focus on actual imports, configuration references, and shared data structures.

## 1. plato-torch
**Location:** `plato-torch/`  
**Purpose:** 25 training room presets, self-training rooms.  
**Key files:**
- `rooms/deadband_room.py` — Deadband Protocol implementation
- `rooms/__init__.py` — exports `DeadbandRoom`, `MirrorRoom`, `GhostRoom`
- `trainer/self_train.py` — self‑training loop
- `config/room_configs.yaml` — preset configurations

**Actual imports (connections outbound):**
- `torch`, `transformers` — external libraries
- `datasets` (Hugging Face) — loads external datasets
- No internal fleet imports found in code.

**Configuration references:**
- `room_configs.yaml` references `"ensign_export_path": "./exports/"` — suggests plato‑ensign as downstream consumer.
- `"mirror_target": "http://localhost:8080"` — suggests Mirror Play setup, but no hardcoded import of another fleet component.

**Shared data structures:**
- `TrainingTile` class (in `rooms/base.py`) — defines `id`, `agent_id`, `content`, `metadata`, `timestamp`. This is the universal tile format used fleet‑wide.

## 2. fleet-simulator
**Location:** `fleet-simulator/`  
**Purpose:** Fleet simulation for training data.  
**Key files:**
- `src/simulator/core.rs` — main simulation engine
- `src/agents/mod.rs` — agent definitions
- `src/export/mod.rs` — data export

**Actual imports (connections outbound):**
- `serde`, `serde_json` — serialization
- `tokio` — async runtime
- No Rust `use` of other fleet crates.

**Configuration references:**
- `config/simulator.toml` contains:
  ```toml
  output_dir = "./data/tiles"
  format = "json"
  ```
  Output directory matches plato‑torch’s expected tile location (`./data/tiles`), but no direct import.

**Shared data structures:**
- `SimulationTile` struct (in `src/export/tile.rs`) — has `agent_id: String`, `content: String`, `metadata: Value`, `timestamp: DateTime<Utc>`. This aligns with plato‑torch’s `TrainingTile` schema.

## 3. holodeck-rust
**Location:** `holodeck-rust/`  
**Purpose:** MUD server with sentiment‑aware NPCs.  
**Key files:**
- `src/server/mod.rs` — MUD server
- `src/npc/sentiment.rs` — sentiment logic
- `src/world/mod.rs` — world state

**Actual imports (connections outbound):**
- `actix-web`, `tokio` — web server
- `sqlx` — database (SQLite)
- No internal fleet imports.

**Configuration references:**
- `config/holodeck.toml` includes:
  ```toml
  tile_source = "../fleet-simulator/data/tiles"
  ```
  Explicitly reads from fleet‑simulator’s output directory.

**Shared data structures:**
- `Tile` struct (in `src/tile.rs`) — same fields as `TrainingTile`/`SimulationTile`.
- Database schema (`tiles` table) stores these tiles.

## 4. plato-ensign
**Location:** `plato-ensign/`  
**Purpose:** Exports room experience into LoRA adapters.  
**Key files:**
- `ensign/export.py` — exports LoRA adapters
- `ensign/train.py` — training from tiles
- `config/export_config.yaml` — export settings

**Actual imports (connections inbound):**
- `from plato_torch.rooms import DeadbandRoom` — **direct import from plato‑torch**.
- `torch`, `peft` — LoRA libraries.

**Configuration references:**
- `export_config.yaml`:
  ```yaml
  source_room: "DeadbandRoom"
  tile_path: "./data/tiles"
  output_dir: "./adapters"
  ```
  Reads tiles from the shared `./data/tiles` directory.

**Shared data structures:**
- Uses `TrainingTile` from plato‑torch (via import).

## Verified Connections
1. **plato‑torch → plato‑ensign** — direct Python import. Ensign consumes DeadbandRoom’s output.
2. **fleet‑simulator → holodeck‑rust** — configuration link: holodeck reads `tile_source` from simulator’s output directory.
3. **Shared tile schema** — all four components use the same tile structure (`agent_id`, `content`, `metadata`, `timestamp`).

## Integration Gaps (What’s Not Connected)
1. **plato‑torch → fleet‑simulator** — no import or configuration link. They run independently.
2. **plato‑torch → holodeck‑rust** — no connection. Holodeck reads from simulator, not from plato‑torch.
3. **plato‑ensign → holodeck‑rust** — no connection. Ensign exports LoRAs; holodeck consumes tiles.
4. **GhostInjector** — referenced in fleet‑context but not found in any of these four repos.
5. **plato‑relay** — not found in these repos; likely separate component.

## P0 Negative Space (What NOT to Do)
- Do not assume plato‑torch and fleet‑simulator share state — they don’t.
- Do not modify the tile schema without checking all four components.
- Do not add new dependencies without verifying they don’t break existing imports.

## P1 Safe Channels (Where You CAN Be)
- Extend plato‑ensign to also read tiles from holodeck’s database (SQLite).
- Create a small adapter that copies plato‑torch tiles to fleet‑simulator’s output directory.
- Add a health‑check endpoint in holodeck that verifies tile‑source exists.

## P2 Optimization (Best Path)
1. Write a `tile_sync.py` script that watches plato‑torch’s export directory and copies new tiles to `./data/tiles` (shared location).
2. Add a configuration flag in holodeck to also read from plato‑torch’s export path as a fallback.
3. Document the data flow:  
   **plato‑torch → (tile_sync) → shared tile directory ← fleet‑simulator → holodeck‑rust**  
   **plato‑torch → plato‑ensign**
