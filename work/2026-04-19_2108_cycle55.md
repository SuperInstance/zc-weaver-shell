## 1. Verified Current Connections (P1: Safe Channels)

Based on reading the actual fleet repos (`plato-torch`, `fleet-simulator`, `holodeck-rust`, `plato-ensign`):

### **plato-torch ‚Üî fleet-simulator**
- **Connection:** Strong, direct integration via `DeadbandRoom` preset.
- **Evidence:** `plato-torch/presets/deadband_room.py` imports `fleet_simulator` and uses `FleetSimulator` class to generate training data.
- **Purpose:** `fleet-simulator` provides synthetic fleet interaction data (agent decisions, P0/P1/P2 traces) which `DeadbandRoom` ingests as training tiles.
- **Data Flow:** Simulator runs ‚Üí produces JSONL logs ‚Üí DeadbandRoom parses ‚Üí creates interaction tiles ‚Üí submits to PLATO room server.

### **plato-torch ‚Üî plato-ensign**
- **Connection:** Direct export pipeline.
- **Evidence:** `plato-ensign/src/exporter.py` imports `plato_torch` room classes. `plato-torch/rooms/base.py` has `export_to_ensign()` method.
- **Purpose:** After a room accumulates enough interaction tiles, `plato-ensign` can export a LoRA adapter capturing the room‚Äôs learned behavior.
- **Data Flow:** Room training completes ‚Üí exporter extracts weights and tile metadata ‚Üí packages as LoRA ‚Üí saves to `ensigns/` directory.

### **holodeck-rust ‚Üî (none yet)**
- **Connection:** Currently isolated. No imports found to/from the other three repos.
- **Evidence:** `holodeck-rust/Cargo.toml` shows no dependencies on `plato-torch`, `fleet-simulator`, or `plato-ensign`. No Python bindings visible.
- **Purpose:** Standalone MUD server with sentiment-aware NPCs. Could be a future tile source or training environment.

## 2. Integration Gaps (P0: Negative Space / P1: Potential Channels)

### **Gap 1: holodeck-rust ‚Üí plato-torch**
- **Status:** Not connected.
- **Risk (P0):** Holodeck interactions are not captured as training tiles. Valuable sentiment/NPC dialogue data is lost.
- **Opportunity (P1):** Build a `HolodeckRoom` preset in `plato-torch` that connects to holodeck‚Äôs WebSocket or API, translating MUD events into interaction tiles.
- **Action:** Write a Rust/Python bridge (or use holodeck‚Äôs log output) to feed NPC interactions into a PLATO room.

### **Gap 2: fleet-simulator ‚Üí holodeck-rust**
- **Status:** Not connected.
- **Risk (P0):** Simulated fleet agents cannot interact with holodeck NPCs, missing complex multi-agent + human-like testing.
- **Opportunity (P1):** Use `fleet-simulator` to drive ‚Äúsimulated players‚Äù that connect to holodeck, generating rich narrative training data.
- **Action:** Extend `fleet-simulator` with a holodeck client module that sends commands and parses responses.

### **Gap 3: plato-ensign ‚Üí holodeck-rust**
- **Status:** Not connected.
- **Risk (P0):** Ensigns (specialist LoRAs) cannot be deployed into holodeck to guide NPC behavior.
- **Opportunity (P1):** Load ensign adapters into holodeck‚Äôs NPC AI models, allowing NPCs to learn from fleet room experience.
- **Action:** Add ensign loading to holodeck‚Äôs NPC logic, likely via ONNX or Safetensors integration.

### **Gap 4: End-to-end pipeline visibility**
- **Status:** Partial.
- **Risk (P0):** No single view of data flow from simulator/holodeck ‚Üí room ‚Üí ensign ‚Üí deployment.
- **Opportunity (P1):** Create a lightweight dashboard or logging middleware that traces a tile‚Äôs journey across components.
- **Action:** Instrument each component with a shared correlation ID and log to a common sink (e.g., `fleet-log`).
