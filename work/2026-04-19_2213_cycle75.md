## 1. Verified Current Connections

Based on reading the actual fleet repos (plato-torch, fleet-simulator, holodeck-rust, plato-ensign):

### **plato-torch ↔ fleet-simulator**
- **Connection:** Strong, bidirectional data flow.
- **Evidence:**  
  - `plato-torch/training_rooms/DeadbandRoom.py` imports `fleet_simulator` and uses `FleetSimulator` class to generate training episodes.  
  - `fleet-simulator/src/lib.rs` exports a Python API via `pyo3` bindings; `plato-torch` calls `simulate_episode()` to get `(state, action, reward)` tuples.  
- **Data flow:** Simulator generates episodic training data → DeadbandRoom ingests it as training tiles.

### **plato-torch ↔ plato-ensign**
- **Connection:** Direct export pipeline.
- **Evidence:**  
  - `plato-ensign/src/exporter.py` imports `torch` and `peft`; expects a trained `DeadbandRoom` model checkpoint.  
  - `plato-torch/scripts/export_to_ensign.py` calls `ensign.export_lora()` with a room checkpoint, creating a LoRA adapter.  
- **Data flow:** Trained room checkpoint → LoRA adapter file (`.safetensors`).

### **holodeck-rust ↔ (none yet)**
- **Connection:** Currently isolated.
- **Evidence:**  
  - `holodeck-rust/Cargo.toml` shows no dependencies on other fleet crates.  
  - No import of `fleet-simulator` or `plato-torch` in `src/`.  
  - It is a standalone MUD server with sentiment-aware NPCs, but no explicit integration points found.

### **fleet-simulator ↔ plato-ensign**
- **Connection:** None observed.
- **Evidence:**  
  - `plato-ensign` does not import `fleet_simulator`.  
  - `fleet-simulator` does not export any data format compatible with `plato-ensign`'s LoRA export.

### **plato-ensign ↔ holodeck-rust**
- **Connection:** None observed.
- **Evidence:**  
  - No shared APIs, data structures, or cross-references.

## 2. Integration Gaps (Actionable)

### **Gap 1: holodeck-rust is an island**
- **Problem:** The sentiment-aware NPC MUD does not contribute to nor benefit from fleet training loops.
- **Opportunity:** Inject `holodeck-rust` with simulated fleet agents or training tiles.
- **P1 Action:** Wire `GhostInjector` (from `plato-torch/ghost_injector.py`) into `holodeck-rust` via its TCP socket (port 8080 per README). Ghost tiles could populate NPC dialogue.

### **Gap 2: DeadbandRoom → plato-relay is not wired**
- **Problem:** `DeadbandRoom` trains locally but does not broadcast its state to the fleet’s real-time coordination layer (`plato-relay`).
- **Opportunity:** Enable live training progress monitoring and inter‑room synchronization.
- **P1 Action:** Add a `RelayClient` to `DeadbandRoom` that emits `room/update` events (cycle count, loss, tile count).

### **Gap 3: No end‑to‑end validation pipeline**
- **Problem:** We cannot verify that a simulator episode → training tile → LoRA adapter → inference works correctly.
- **Opportunity:** Build a test that runs the full chain and validates output consistency.
- **P1 Action:** Write `test_integration_sim_to_ensign.py` that:  
  1. Calls `fleet_simulator.simulate_episode()`  
  2. Feeds episode to `DeadbandRoom` as a tile  
  3. Trains for one step  
  4. Exports via `plato‑ensign`  
  5. Loads LoRA and runs a forward pass.

### **Gap 4: plato-ensign only exports LoRA, not training tiles**
- **Problem:** The export is one‑way; ensigns cannot be “re‑hydrated” back into rooms.
- **Opportunity:** Allow ensigns to generate new training tiles (e.g., for distillation).
- **P2 Action:** Add `ensign.generate_tile()` that uses the LoRA‑adapted model to produce synthetic training data.
