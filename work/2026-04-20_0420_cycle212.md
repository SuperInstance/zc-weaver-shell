## 1. plato-torch (Training Rooms)
**Location:** `plato-torch/`  
**Purpose:** 25 training room presets, self-training rooms.  
**Key Files:**  
- `rooms/deadband_room.py` — Deadband Protocol training room.  
- `rooms/__init__.py` — Exports room classes.  
- `trainer/room_trainer.py` — Trains from interaction tiles.  
- `config/room_configs.yaml` — Room configurations.

**Connections:**  
- **plato-ensign:** Exports room experience via `plato-torch/export/ensign_exporter.py`. Direct import: `from plato_ensign import LoRAExporter`.  
- **fleet-simulator:** Not directly imported. No file references found in code.  
- **holodeck-rust:** Not directly imported. No Rust bindings or API calls in Python code.  
- **External:** Uses PyTorch, transformers, datasets. Logs to TensorBoard.

**Integration Points:**  
- `ensign_exporter.py` calls `LoRAExporter.save_adapter()` after room training.  
- Room configurations can load custom datasets (potentially from fleet-simulator, but not linked).  
- Tiles are saved to `data/tiles/` — could be consumed by other services.

## 2. fleet-simulator (Fleet Simulation)
**Location:** `fleet-simulator/`  
**Purpose:** Generate training data via multi-agent simulation.  
**Key Files:**  
- `src/simulator/core.rs` — Core simulation engine.  
- `src/agents/mod.rs` — Agent definitions.  
- `examples/deadband_simulation.rs` — Deadband Protocol example.

**Connections:**  
- **plato-torch:** No direct connection. No Python bindings or shared data format observed.  
- **plato-ensign:** No direct connection.  
- **holodeck-rust:** Not directly imported. Both are Rust crates but no `Cargo.toml` dependencies found.  
- **External:** Uses `rand`, `serde`, `tokio`. Outputs JSON logs.

**Integration Points:**  
- Simulation logs (`logs/simulation_*.json`) could be parsed into training tiles for plato-torch.  
- Agent behaviors could align with Deadband Protocol (P0/P1/P2) — example exists but not integrated.

## 3. holodeck-rust (MUD Server)
**Location:** `holodeck-rust/`  
**Purpose:** MUD server with sentiment-aware NPCs.  
**Key Files:**  
- `src/server/mod.rs` — Async MUD server.  
- `src/npc/sentiment.rs` — NPC sentiment system.  
- `src/world/rooms.rs` — Room definitions.

**Connections:**  
- **plato-torch:** No direct connection.  
- **fleet-simulator:** No direct connection.  
- **plato-ensign:** No direct connection.  
- **External:** Uses `tokio`, `serde`, `websocket`. Listens on port 8080.

**Integration Points:**  
- NPC sentiment data could train plato-torch rooms (sentiment as reward signal).  
- Room interactions could generate tiles — no tile-saving mechanism observed.  
- Could connect via WebSocket to external agents (like fleet-simulator agents).

## 4. plato-ensign (LoRA Exporter)
**Location:** `plato-ensign/`  
**Purpose:** Export room experience into LoRA adapters.  
**Key Files:**  
- `src/exporter.rs` — LoRA export logic.  
- `src/adapters/lora.rs` — LoRA configuration.  
- `examples/export_deadband.rs` — Example.

**Connections:**  
- **plato-torch:** Directly connected via `ensign_exporter.py` import.  
- **fleet-simulator:** No connection.  
- **holodeck-rust:** No connection.  
- **External:** Uses `safetensors`, `transformers`.

**Integration Points:**  
- Called by plato-torch after room training.  
- Outputs `.safetensors` files to `adapters/` directory.  
- Could be triggered by other systems (not currently).
