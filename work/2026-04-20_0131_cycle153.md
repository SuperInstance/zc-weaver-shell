# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 153

**Status:** Mapping current connections and identifying integration points.

Based on reading the actual files in the fleet repos, here is the current integration state between the four core technologies:

## 1. **plato-torch** ↔ **plato-ensign** — **CONNECTED**
- **Connection:** Direct, explicit integration via `export_ensign()` method in `DeadbandRoom` class.
- **Evidence:** `plato-torch/rooms/deadband_room.py` contains:
  ```python
  def export_ensign(self, adapter_path: str = "ensign_lora"):
      """Export room experience as LoRA adapter."""
      from plato_ensign import export_lora_from_tiles
      tiles = self.get_training_tiles()
      export_lora_from_tiles(tiles, adapter_path)
  ```
- **Data Flow:** Room accumulates interaction tiles → passed to `plato_ensign.export_lora_from_tiles()` → generates LoRA adapter.
- **Integration Status:** **Live**. This is the primary training-to-export pipeline.

## 2. **plato-torch** ↔ **fleet-simulator** — **NOT CONNECTED**
- **Evidence:** No import statements, function calls, or shared data structures found in either codebase.
- **plato-torch** focuses on room-based training with tile accumulation.
- **fleet-simulator** (based on `fleet-simulator/README.md`) generates synthetic training data through multi-agent simulations.
- **Potential Integration Point:** Fleet simulator could generate training tiles that feed into a plato-torch room. Currently separate data pipelines.

## 3. **plato-torch** ↔ **holodeck-rust** — **PARTIALLY CONNECTED**
- **Connection:** Indirect via shared concept of "tiles" but no direct API integration.
- **Evidence:** 
  - `holodeck-rust/src/protocol/tile.rs` defines `Tile` struct with `content`, `metadata`, `sentiment_score`.
  - `plato-torch` uses `TrainingTile` objects with similar fields (`content`, `metadata`, `reward`).
- **Integration Gap:** No code found that passes tiles between the two systems. Holodeck appears to be a standalone MUD server with sentiment-aware NPCs that could generate tiles, but no exporter to plato-torch rooms.

## 4. **fleet-simulator** ↔ **holodeck-rust** — **NOT CONNECTED**
- **Evidence:** Completely separate codebases, different languages (Python vs Rust), no shared protocols.
- **fleet-simulator** uses Python-based agent simulations.
- **holodeck-rust** is a Rust MUD server with WebSocket API.
- **Integration Gap:** No bridge between simulation environments.

## 5. **plato-ensign** ↔ **holodeck-rust** — **NOT CONNECTED**
- **Evidence:** `plato-ensign` only imports from `plato-torch` and PyTorch libraries.
- **Integration Gap:** No mechanism to export holodeck interactions as ensign adapters.

## 6. **plato-ensign** ↔ **fleet-simulator** — **NOT CONNECTED**
- **Evidence:** No imports or function calls between the two.
- **Integration Gap:** Fleet simulator data cannot currently be used to train ensign adapters.
