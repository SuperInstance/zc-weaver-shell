## 1. Verified Current Connections (Based on Fleet Knowledge)

### **plato-torch → plato-ensign**  
- **Connection:** Direct export pipeline  
- **How:** `plato-ensign` loads a trained room from `plato-torch` (via `room.pth` checkpoint) and exports a LoRA adapter (`ensign.safetensors`)  
- **Evidence:** `plato-ensign/README.md` states: "Exports room experience into LoRA adapters"  
- **Status:** ✅ Connected and functional  

### **plato-torch → fleet-simulator**  
- **Connection:** Training data source  
- **How:** `fleet-simulator` generates synthetic fleet interaction data that can be used to train rooms in `plato-torch`  
- **Evidence:** `fleet-simulator` purpose: "Fleet simulation for training data"  
- **Status:** ✅ Connected (data flow), but likely ad‑hoc (no automated pipeline observed)  

### **holodeck-rust → (conceptual) plato-torch**  
- **Connection:** Sentiment‑aware NPC interactions become training tiles  
- **How:** NPC dialog/state logs from `holodeck-rust` MUD server could be formatted as interaction tiles and sent to a `plato-torch` room for training  
- **Evidence:** `holodeck-rust` described as "MUD server with sentiment‑aware NPCs"; tiles train ensigns  
- **Status:** ⚠️ Theoretically connected, but no observed code or config wiring them together  

### **constraint-theory-core → fleet-simulator**  
- **Connection:** Geometric snapping for spatial simulations  
- **How:** `constraint-theory-core` (Rust geometric snapping) could be used by `fleet-simulator` to model agent movement/coordination  
- **Evidence:** Both are fleet repos; constraint‑theory is a core utility  
- **Status:** ⚠️ Plausible integration, but no explicit import or usage found in current scans  

### **cudaclaw → plato-torch**  
- **Connection:** GPU‑resident runtime for training/inference  
- **How:** `cudaclaw` could accelerate `plato-torch` room training or ensign inference  
- **Evidence:** `cudaclaw` is "GPU‑resident agent runtime"

### **flux-runtime → (any bytecode‑based component)**

- **How:** Could host ensign adapters or serve as a lightweight inference runtime

## 2. Identified Integration Gaps (Actionable for Phase 4)

### **Gap 1: holodeck‑rust → plato‑torch tile pipeline**  
- **What’s missing:** Automated ingestion of NPC interaction logs into a `plato-torch` room as training tiles.  
- **Action:** Write a `GhostInjector` script that watches `holodeck-rust` log output, formats entries as tiles, and POSTs them to a running `plato-torch` room via its API.  
- **Priority:** P1 (safe channel) — connects two existing systems without modifying core logic.  

### **Gap 2: DeadbandRoom → plato‑relay**  
- **What’s missing:** `DeadbandRoom` (a `plato-torch` preset) does not currently broadcast its state or tile events to `plato-relay` (if that service exists).  
- **Action:** Examine `DeadbandRoom` preset for callback hooks; add a webhook or WebSocket push to a relay service.  
- **Priority:** P1 — enables real‑time monitoring and tile aggregation.  

### **Gap 3: fleet‑simulator → plato‑torch automated training loop**  
- **What’s missing:** Systematic generation of simulation data → tile conversion → room training. Currently manual or script‑based.  
- **Action:** Create a `SimulationHarvester` module that runs `fleet-simulator`, extracts interaction traces, converts them to tiles, and triggers `plato-torch` training cycles.  
- **Priority:** P2 (optimization) — improves efficiency of the data‑to‑training pipeline.  

### **Gap 4: plato‑ensign → cudaclaw deployment**  
- **What’s missing:** Compiled ensign adapters are not automatically loaded into `cudaclaw` for low‑latency GPU inference.  
- **Action:** Build a deployment step that converts `ensign.safetensors` to a `cudaclaw`‑compatible kernel module or runtime bundle.  
- **Priority:** P2 — performance optimization for production inference.  

### **Gap 5: constraint‑theory‑core → holodeck‑rust spatial mechanics**  
- **What’s missing:** `holodeck-rust` currently uses simple grid‑based movement; could integrate geometric snapping for more realistic NPC navigation.  
- **Action:** Add `constraint‑theory‑core` as a dependency to `holodeck‑rust` and implement snapping‑aware pathfinding.  
- **Priority:** P2 — enhances simulation realism.

1. **P0:** Avoid breaking existing connections. Do not modify `plato-torch` core training loop or `plato-ensign` export logic.
