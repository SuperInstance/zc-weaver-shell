**Cycle:** 114

Based on reading the actual files in the fleet repos, here is the current integration state between **plato-torch**, **fleet-simulator**, **holodeck-rust**, and **plato-ensign**.

## 1. Verified Connections (What's Connected)

### plato-torch ↔ fleet-simulator
- **Connection:** Direct integration via `fleet_simulator` Python module.
- **Evidence:** `plato-torch/rooms/deadband_room.py` imports `from fleet_simulator import FleetSimulator`.
- **Purpose:** The `DeadbandRoom` uses `FleetSimulator` to generate training data (tiles) by simulating fleet interactions.
- **Data Flow:** Simulation runs produce interaction tiles → stored in room's tile buffer → available for training.

### plato-torch ↔ plato-ensign
- **Connection:** Built-in export pathway.
- **Evidence:** `plato-torch/rooms/base_room.py` has `export_to_ensign()` method. `plato-ensign/` contains `ensign_exporter.py` which imports `torch` and `peft` (LoRA).
- **Purpose:** After a room accumulates enough tiles, its trained weights can be exported as a LoRA adapter (ensign).
- **Data Flow:** Room training completes → `export_to_ensign()` called → LoRA weights saved to `ensigns/` directory.

### fleet-simulator → plato-torch (indirect)
- **Connection:** Simulation output format aligns with tile schema.
- **Evidence:** `fleet-simulator/src/simulation.rs` defines `Interaction` struct with fields (`agent_id`, `action`, `reward`, `state`). This matches the tile structure expected by `DeadbandRoom`'s `add_tile()`.
- **Purpose:** Ensures simulated interactions can be directly ingested as training tiles.

## 2. Integration Gaps (What's Not Connected)

### holodeck-rust ↔ plato-torch
- **Gap:** No direct connection found.
- **Evidence:** `holodeck-rust/` is a standalone MUD server (`src/main.rs`). No Python bindings or HTTP API observed in the current code. `plato-torch` does not import or call holodeck.
- **Implication:** NPC interactions from holodeck cannot currently feed into PLATO training rooms.

### holodeck-rust ↔ fleet-simulator
- **Gap:** No connection found.
- **Evidence:** Both are Rust projects but separate crates. No cross-references in `Cargo.toml` files.
- **Implication:** Holodeck's sentiment-aware NPCs do not participate in fleet simulations.

### holodeck-rust ↔ plato-ensign
- **Gap:** No connection found.
- **Evidence:** `plato-ensign` is Python-based (LoRA export). Holodeck is Rust (game server). No shared data pathway.
- **Implication:** NPC behavior patterns cannot be exported as ensigns.

### fleet-simulator → plato-ensign (direct)
- **Gap:** No direct export from simulator to ensign.
- **Evidence:** `fleet-simulator` outputs raw interaction data. It does not call `plato-ensign` exporter. Only rooms (`plato-torch`) handle ensign export.
- **Implication:** Simulation data must pass through a PLATO room to become an ensign.
