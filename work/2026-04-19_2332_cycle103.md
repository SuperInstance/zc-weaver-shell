## 1. Verified Connections (What's Connected)

### plato-torch → plato-ensign (Direct Export)
- **Connection:** Strong, direct integration via `export_ensign()` method in `DeadbandRoom` class.
- **Mechanism:** `plato-torch/src/rooms/deadband_room.py` calls `plato_ensign.export_lora()` with accumulated training tiles.
- **Data Flow:** Room interaction tiles → LoRA adapter file (.safetensors).
- **Status:** Functional. Verified by `test_deadband_room.py` which tests ensign export.

### fleet-simulator → plato-torch (Training Data Source)
- **Connection:** Indirect data pipeline.
- **Mechanism:** `fleet-simulator` generates simulation logs (agent interactions, decisions). These logs can be formatted into training tiles compatible with `plato-torch` tile schema.
- **Data Flow:** Simulation logs → tile JSON → ingested by `DeadbandRoom.add_tile()`.
- **Status:** Potential connection, not yet wired. No import/export code exists in either repo.

### holodeck-rust → (No Direct Fleet Integration)
- **Connection:** Isolated MUD server.
- **Mechanism:** `holodeck-rust` is a standalone Rust application with sentiment-aware NPCs. It has its own TCP server and protocol.
- **Data Flow:** Internal only. No current hooks to `plato-torch` or `fleet-simulator`.
- **Status:** Siloed. Could be a rich source of human-agent interaction tiles if integrated.

## 2. Integration Gaps (What's Not Connected)

### Gap 1: fleet-simulator → plato-torch (Missing Bridge)
- **Problem:** No code converts fleet-simulator output into `plato-torch` training tiles.
- **Location:** Both repos lack an adapter module.
- **Potential Wiring:** Create `sim2tile` converter in `fleet-simulator/tools/` or `plato-torch/ingest/`.

### Gap 2: holodeck-rust → plato-torch (No Tile Export)
- **Problem:** `holodeck-rust` NPC interactions are trapped in the MUD. No mechanism to export them as training tiles.
- **Location:** `holodeck-rust/src/` has no export functionality.
- **Potential Wiring:** Add a `TileLogger` system that captures interactions and writes to a tile-compatible JSON file.

### Gap 3: plato-ensign → fleet-simulator (No Ensign Loading)
- **Problem:** Ensigns (LoRA adapters) are exported but not used by `fleet-simulator` to influence agent behavior.
- **Location:** `fleet-simulator` agents are hard-coded; no hook to load a `.safetensors` adapter.
- **Potential Wiring:** Integrate `plato-torch` inference into simulator agents, allowing them to use trained ensigns.

### Gap 4: Deadband Protocol P0/P1/P2 Enforcement
- **Problem:** The Deadband Protocol is documented but not programmatically enforced across components.
- **Location:** Each component implements its own logic; no shared constraint-checking layer.
- **Potential Wiring:** Create `constraint-theory-core` bindings in each repo to snap decisions to safe channels.
