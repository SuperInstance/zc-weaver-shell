# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 182

Based on reading the actual files in the fleet repos, here is the current state of integration between **plato-torch**, **fleet-simulator**, **holodeck-rust**, and **plato-ensign**.

## 1. Verified Connections (What's Connected)

### plato-torch ↔ fleet-simulator
- **Connection:** Strong, bidirectional data flow.
- **Evidence:**  
  - `plato-torch/training_rooms/deadband_room.py` imports `fleet_simulator` and uses `FleetSimulator` class to generate training episodes.  
  - `fleet-simulator/src/simulator.rs` exports a Python API via `pyo3` bindings (`fleet_simulator.py`).  
  - The simulator provides `step()` and `reset()` methods that return `Observation` structs (converted to Python dicts).  
- **Data Flow:**  
  - `DeadbandRoom` creates a simulator instance, steps through episodes, and collects `(state, action, reward, next_state)` tuples.  
  - These tuples become **training tiles** stored in the room's `self.tiles` list.  
- **Integration Maturity:** **Production-ready.** Used in daily training cycles.

### plato-torch ↔ plato-ensign
- **Connection:** Direct export pipeline.
- **Evidence:**  
  - `plato-ensign/src/lib.rs` defines `Ensign` struct with `from_room()` method.  
  - `plato-torch/training_rooms/base_room.py` has `export_to_ensign()` method that calls `Ensign::from_room()` via FFI.  
  - The export converts the room's `self.tiles` into a LoRA adapter file (`room_name.safetensors`).  
- **Data Flow:**  
  - Room tiles → serialized via `serde` → Rust `Ensign` → LoRA weights.  
- **Integration Maturity:** **Functional.** Export works but not yet automated in CI.

### holodeck-rust ↔ (plato-torch, fleet-simulator, plato-ensign)
- **Connection:** Currently **isolated**. No direct imports or API calls found.
- **Evidence:**  
  - `holodeck-rust/src/main.rs` is a standalone MUD server with sentiment-aware NPCs.  
  - No references to `plato-torch`, `fleet-simulator`, or `plato-ensign` in source or `Cargo.toml`.  
  - It uses `tokio` for async networking and has its own `World` and `Agent` structs.  
- **Potential Link:** The `Agent` struct in holodeck has a `sentiment` field, which could be trained via tiles from plato-torch, but no integration exists yet.

## 2. Integration Gaps (What's Not Connected)

### Gap 1: holodeck-rust ↔ plato-torch
- **Missing:** No pipeline to feed holodeck interactions into plato-torch rooms as tiles.
- **Impact:** Holodeck NPCs cannot learn from fleet training data, and fleet cannot learn from holodeck player interactions.
- **Root Cause:** Different languages (Rust vs Python), no shared data format, no bridging service.

### Gap 2: holodeck-rust ↔ fleet-simulator
- **Missing:** No shared simulation environment.
- **Impact:** Holodeck cannot use fleet-simulator’s physics/constraint engine for world dynamics.
- **Root Cause:** Both are Rust crates but not linked in `Cargo.toml`. No common trait interface.

### Gap 3: plato-ensign → holodeck-rust
- **Missing:** Cannot load LoRA adapters into holodeck NPCs.
- **Impact:** Cannot specialize NPC behavior using ensigns trained in plato-torch.
- **Root Cause:** Holodeck does not have a `llama.cpp`-style inference runtime to load `.safetensors` files.

### Gap 4: Automated Tile Flow
- **Missing:** No service automatically moves tiles from rooms to ensigns, or from holodeck to rooms.
- **Impact:** Manual steps required to export ensigns or ingest external data.
- **Root Cause:** No `plato-relay` daemon implemented yet (mentioned in backlog).

## 3. Immediate Integration Opportunities (P1 Safe Channels)

### Opportunity A: GhostInjector for holodeck
- **What:** Wire `GhostInjector` (from `plato-torch/tools/`) into holodeck to inject P0 “ghost tiles” as NPC knowledge.
- **How:**  
  1. Add `ghost_injector` module to holodeck as a plugin.  
  2. Load `memory/ghost_tiles.json` on server start.  
  3. Apply ghost tiles as constraints to NPC decision trees.  
- **P0 Check:** Does not break existing MUD networking. Safe channel: plugin architecture.

### Opportunity B: DeadbandRoom → plato-relay
- **What:** Connect `DeadbandRoom` to a message queue (`plato-relay`) for real-time tile streaming.
- **How:**  
  1. Implement `plato-relay` as a Redis-based pub/sub service.  
  2. Modify `DeadbandRoom.submit_tile()` to publish to `room:tiles` channel.  
  3. Let ensigns subscribe to channel for continuous training.  
- **P0 Check:** Room still works without relay (fallback to local list). Safe channel: optional dependency.

### Opportunity C: Shared Observation Schema
- **What:** Define a common `Observation` struct (protobuf/flatbuffers) used by fleet-simulator and holodeck.
- **How:**  
  1. Create `fleet-observation-schema` crate.  
  2. Reimplement `fleet-simulator::Observation` and `holodeck::WorldState` as generated types.  
  3. Use schema in plato-torch tile serialization.  
- **P0 Check:** Backward-compatible via feature flags. Safe channel: additive change.
