## 1. plato-torch

**Location:** `plato-torch/`  
**Purpose:** Training room presets, self-training rooms, tile processing.

**Key Files Inspected:**
- `plato-torch/src/rooms/deadband_room.py` — Primary Deadband Protocol implementation.
- `plato-torch/src/tile_processor.py` — Processes interaction tiles into training data.
- `plato-torch/requirements.txt` — Dependencies: `torch`, `transformers`, `numpy`, `pydantic`.

**Connections Found:**
- **plato-ensign:** Direct import in `deadband_room.py`:
  ```python
  from plato_ensign import export_lora, load_room_experience
  ```
  Room can export experience to LoRA adapters via `export_lora()`.
- **fleet-simulator:** No direct import found. Potential indirect link via tile data schema.
- **holodeck-rust:** No direct import. No Rust interop visible in Python code.
- **External:** Uses `transformers` for model fine-tuning, `torch` for training loops.

**Integration Points:**
- `DeadbandRoom` emits `room_experience` dict (P0/P1/P2 logs, tile batches).
- `TileProcessor` outputs `training_batch` tensors compatible with `plato-ensign`'s adapter format.

## 2. plato-ensign

**Location:** `plato-ensign/`  
**Purpose:** Exports room experience into LoRA adapters.

**Key Files Inspected:**
- `plato-ensign/src/exporter.py` — Core LoRA export logic.
- `plato-ensign/src/schemas.py` — Defines `RoomExperience` Pydantic model.
- `plato-ensign/pyproject.toml` — Dependencies: `peft`, `torch`, `pydantic`.

**Connections Found:**
- **plato-torch:** Strong coupling. `RoomExperience` model expects `tiles: List[Tile]` and `deadband_logs: Dict[str, List[str]]` matching plato-torch output.
- **fleet-simulator:** Not referenced.
- **holodeck-rust:** Not referenced.
- **External:** Uses `peft` (Parameter-Efficient Fine-Tuning) library for LoRA.

**Integration Points:**
- Accepts `room_experience` dict from `DeadbandRoom`.
- Outputs `lora_weights.safetensors` and `adapter_config.json`.

## 5. GhostInjector

**Location:** `plato-torch/src/ghost_injector.py`  
**Purpose:** Injects ghost tiles (dead agents' lessons) into training.

**Connections Found:**
- **plato-torch:** Integrated within tile processing pipeline. Called by `TileProcessor`.
- **holodeck-rust:** Not connected. Ghost tiles currently sourced from file system (`ghost_tiles/` directory).
- **fleet-simulator:** Not connected.

**Integration Gap:** GhostInjector reads static files. Could pull from holodeck’s NPC memory or fleet-simulator agent obituaries.

## 6. cudaclaw & flux-runtime

**Location:** `cudaclaw/`, `flux-runtime/`  
**Note:** Not part of immediate integration map task. Inspected for completeness.

**Findings:**
- `cudaclaw/` is a CUDA runtime for GPU-resident agents. No imports to/from plato-torch or holodeck.
- `flux-runtime/` is a self-assembling bytecode VM. Independent project.
- Both are standalone, not wired into the four core repos.

## Summary: What’s Connected vs. What’s Not

### Connected:
1. **plato-torch ↔ plato-ensign** — Strong, direct integration via Python imports and shared data schema.
   - Working pipeline: `DeadbandRoom` → `RoomExperience` → LoRA adapter.

### Not Connected (Integration Gaps):
1. **fleet-simulator ↔ plato-torch** — No bridge. Simulation logs not consumed for training.
2. **holodeck-rust ↔ plato-torch** — No bridge. NPC dialogue not converted to tiles.
3. **GhostInjector ↔ holodeck-rust** — Ghost tiles static; could use NPC memory.
4. **cudaclaw / flux-runtime** — Not integrated with any of the four.

### Potential Safe Channels (P1):
- Build a Rust → Python bridge (`simulation_events.json` → `TileProcessor`).
- Add SQLite listener in holodeck to export dialogue logs as tiles.
- Extend GhostInjector to query holodeck’s NPC memory table.
