## 1. Verified Current Connections

Based on reading the actual fleet repos (`plato-torch`, `fleet-simulator`, `holodeck-rust`, `plato-ensign`), here are the **confirmed** integration points:

### **plato-torch ↔ plato-ensign** (Strong Connection)
- **plato-torch** exports training data in `.pt` tile format.
- **plato-ensign** imports these tiles via `ensign/tile_loader.py` and converts them into LoRA adapters.
- **Connection method:** File system export/import pipeline. Works but is manual/batch-oriented.

### **fleet-simulator → plato-torch** (One-way Data Feed)
- **fleet-simulator** generates synthetic training episodes (JSONL format).
- **plato-torch** ingests these via `data/ingest_simulator_logs.py`.
- **Connection method:** Log ingestion script. Functional but not real-time.

### **holodeck-rust** (Standalone, No Direct Integrations)
- Holodeck is a sentiment-aware MUD server written in Rust.
- No import/export hooks for `plato-torch` tiles or `fleet-simulator` logs found in current code.
- It has its own internal event system and NPC memory, but no fleet-wide data bridges.

### **plato-ensign** (Downstream Consumer Only)
- Takes tiles from `plato-torch` and produces LoRA files.
- No connections back to `holodeck-rust` or `fleet-simulator`.

## 2. Identified Integration Gaps (P1 Safe Channels)

### **Gap 1: holodeck-rust → plato-torch (Tile Generation)**
- **Current state:** Holodeck player/NPC interactions are not captured as training tiles.
- **Safe channel:** Add a `TileLogger` module to holodeck that writes interaction events in `.pt` format.
- **Action:** Create a Rust crate (`holodeck-tile-exporter`) that hooks into holodeck’s event bus.

### **Gap 2: fleet-simulator → holodeck-rust (NPC Behavior)**
- **Current state:** No connection. Simulated fleet data could drive NPC personalities/dialogue.
- **Safe channel:** Use `fleet-simulator` logs to generate “backstory” JSON for holodeck NPCs.
- **Action:** Write a translator script (sim_log_to_npc_profile.py) and add a `/load_npc_profile` admin command to holodeck.

### **Gap 3: plato-ensign → holodeck-rust (Live Adaptation)**
- **Current state:** Ensign LoRAs are static files; no runtime integration.
- **Safe channel:** Allow holodeck to load a LoRA adapter at runtime to alter NPC response generation.
- **Action:** Extend holodeck’s LLM caller to support LoRA merges (via `lorax` or `peft`).

### **Gap 4: Deadband Protocol ↔ All Systems (Cross-Cutting)**
- **Current state:** Deadband is a concept/doc; not embedded in system APIs.
- **Safe channel:** Add a `deadband` field to tile schema, simulator logs, and holodeck events.
- **Action:** Propose a `DeadbandAware` trait in Rust (holodeck) and base class in Python (plato-torch).
