## 1. plato-torch (Training Rooms)
**Location:** `plato-torch/`  
**Purpose:** 25 training room presets, self-training rooms.  
**Key Files:**  
- `presets/deadband_room.py` — Deadband Protocol training room.  
- `rooms/base_room.py` — Abstract base class for all rooms.  
- `trainers/self_trainer.py` — Self-training loop.  
- `data/tile_loader.py` — Loads interaction tiles for training.

**Connections Found:**  
- **To plato-ensign:** `presets/deadband_room.py` imports `from plato_ensign import export_lora` (line 42). Room can export its trained adapter via `export_lora(adapter_path, output_dir)`.  
- **To fleet-simulator:** `data/tile_loader.py` includes `from fleet_simulator.scenarios import ScenarioLoader` (line 18). Loads simulated scenario data as training tiles.  
- **To holodeck-rust:** No direct imports found. However, `presets/deadband_room.py` references `"holodeck_npc_data"` in its config dictionary (line 89) as a potential data source.  
- **To constraint-theory-core:** Not referenced.  
- **To cudaclaw/flux-runtime:** Not referenced.

## 2. fleet-simulator (Simulation Data)
**Location:** `fleet-simulator/`  
**Purpose:** Generate training data via multi-agent simulations.  
**Key Files:**  
- `scenarios/deadband_scenario.py` — Deadband Protocol scenario.  
- `agents/sim_agent.py` — Base simulation agent.  
- `export/tile_exporter.py` — Exports interactions as tiles.

**Connections Found:**  
- **To plato-torch:** `export/tile_exporter.py` writes tiles in format matching `plato-torch/data/tile_loader.py` expectations (JSON with `interaction`, `reward`, `metadata`).  
- **To holodeck-rust:** `scenarios/deadband_scenario.py` imports `from holodeck_rust.client import MudClient` (line 33). Can connect to holodeck MUD server to fetch NPC sentiment data.  
- **To plato-ensign:** No direct imports.  
- **To constraint-theory-core/cudaclaw/flux-runtime:** Not referenced.

## Integration Map Summary

### Connected:
1. **plato-torch ↔ plato-ensign** — Direct export path (training → LoRA).  
2. **plato-torch ↔ fleet-simulator** — Simulated data → training tiles.  
3. **fleet-simulator ↔ holodeck-rust** — Simulation agents can query NPC sentiment.

### Not Connected (Gaps):
1. **holodeck-rust → plato-torch** — NPC data not directly ingested as tiles (only via fleet-simulator).  
2. **constraint-theory-core** — Isolated; no integration with training or simulation.  
3. **cudaclaw / flux-runtime** — Isolated; not used by any training room or simulator.  
4. **Ghost tiles / Mirror Play** — Referenced in architecture but no implementation found in code.  
5. **Deadband Protocol P0/P1/P2 logic** — Defined in docs but not wired into room training loops.
