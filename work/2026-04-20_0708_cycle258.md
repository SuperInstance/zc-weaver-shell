## 1. plato-torch → fleet-simulator
**Connection:** **Strong**  
- `plato-torch/src/rooms/deadband_room.py` imports `fleet_simulator` directly:  
  ```python
  from fleet_simulator import FleetSimulator, FleetConfig
  ```
- `DeadbandRoom` instantiates a `FleetSimulator` object (`self.simulator`) and calls `self.simulator.step()` each training cycle.  
- `DeadbandRoom` passes its own `self.agent` (the training model) into the simulator via `self.simulator.set_agent(self.agent)`.  
- Shared data: The simulator returns a `step_result` dict containing `reward`, `done`, `info`. This is used as training signal.

**Integration Gap:**  
- The simulator’s internal state (fleet positions, rock locations) is not directly exposed to the room’s logging or tile‑export system.  
- No two‑way configuration sync: room hyperparameters (learning rate, batch size) are separate from simulator parameters (fleet size, rock density).

---

## 2. plato-torch → plato-ensign
**Connection:** **Strong**  
- `plato-torch/src/export/ensign_exporter.py` is dedicated to converting room experience into LoRA adapters.  
- It imports `peft` (Parameter‑Efficient Fine‑Tuning) and `torch`.  
- The `EnsignExporter` class reads the room’s `memory_buffer` (accumulated interaction tiles) and fine‑tunes the room’s model, producing a `.safetensors` file.  
- The export is triggered manually or by a room‑completion hook.

**Integration Gap:**  
- Export is optional and off by default. No automatic trigger based on tile‑count or performance threshold.  
- The exported adapter is not automatically registered or versioned in a fleet‑wide model zoo.

---

## 3. plato-torch → holodeck-rust
**Connection:** **None (direct)**  
- No import of `holodeck` in any Python file under `plato-torch/`.  
- No Rust bindings or FFI calls observed.  
- However, both systems share a conceptual link:  
  - `plato-torch` trains agents via simulated interaction.  
  - `holodeck-rust` is a MUD server meant to deploy trained agents into a persistent, sentiment‑aware world.  
- Potential integration point: a trained `ensign` (LoRA) could be loaded into a `holodeck` NPC.

**Integration Gap:**  
- No code‑level bridge between Python training pipeline and Rust‑based MUD server.  
- No shared session‑data format (tiles vs. MUD event logs).

---

## 4. fleet-simulator → holodeck-rust
**Connection:** **None**  
- `fleet-simulator` is a pure‑Python training environment.  
- `holodeck-rust` is a standalone game server.  
- No imports, no shared protocols.

**Integration Gap:**  
- Simulator’s fleet‑navigation tasks could be mirrored as `holodeck` quests, but no mapping exists.  
- Simulator’s `FleetConfig` does not reference any MUD‑world configuration.

---

## 5. fleet-simulator → plato-ensign
**Connection:** **Indirect**  
- `fleet-simulator` generates the training data (state, action, reward) that fills `plato-torch`’s memory buffer.  
- `plato-ensign` exports that buffer.  
- No direct calls between simulator and ensign exporter.

**Integration Gap:**  
- Simulator does not tag its output with metadata (e.g., “this tile came from rock‑avoidance scenario”), which could help ensigns specialize.

---

## 6. holodeck-rust → plato-ensign
**Connection:** **None**  
- `holodeck-rust` does not import any Python modules.  
- `plato-ensign` exports PyTorch adapters, not Rust‑compatible formats.

**Integration Gap:**  
- No loader for `.safetensors` files in `holodeck-rust`.  
- No mechanism to inject a fine‑tuned behavior into an NPC.

---

## 7. Ghost Tiles & Mirror Play
**Observed in code:**  
- `plato-torch/src/rooms/ghost_injector.py` exists. It reads `memory/ghost_tiles/` and injects them into a room’s memory buffer as P0 examples (negative space).  
- `mirror_play` is referenced in `PROJECT-VISION.md` but not yet implemented as a module.

**Integration Gap:**  
- `ghost_injector` is not wired into any default training loop; must be manually enabled.  
- No `mirror_play` code found in inspected repos.

---

## 8. cudaclaw & flux-runtime
**Connection to above systems:** **Not yet visible**  
- `cudaclaw` is a GPU‑resident agent runtime (C++/CUDA).  
- `flux-runtime` is a self‑assembling bytecode VM.  
- Neither is imported by `plato-torch`, `fleet-simulator`, `holodeck-rust`, or `plato-ensign` in the current code snapshots.

**Integration Gap:**  
- Training pipeline runs on standard PyTorch CPU/GPU, not on `cudaclaw`.  
- No bytecode compilation of trained behaviors for `flux-runtime`.
