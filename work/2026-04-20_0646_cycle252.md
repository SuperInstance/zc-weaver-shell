## 1. plato-torch
**Repository:** `plato-torch`  
**Purpose:** Training room presets, self-training rooms, tile processing.

### Verified Connections:
- **plato-ensign:** Direct import.  
  - `plato-torch/src/ensign_exporter.py` imports `from plato_ensign.lora_adapter import LoRAAdapter`.  
  - `plato-torch/config/room_presets/deadband_room.yaml` contains `ensign_export_path: ../plato-ensign/exports/`.  
- **fleet-simulator:** Indirect via shared tile format.  
  - `plato-torch/src/tile_processor.py` defines `Tile` dataclass with fields `agent_id`, `timestamp`, `content`, `metadata`.  
  - `fleet-simulator/src/simulation/tile_generator.rs` uses identical JSON structure for tile output.  
- **holodeck-rust:** No direct import found.  
  - No Rust dependencies or Python imports referencing `holodeck`.  
  - However, `plato-torch/docs/room_design.md` mentions "sentiment-aware NPCs from holodeck" as a future integration.

### Integration Gaps:
- No live socket or API connection to `holodeck-rust` observed in code.  
- Tile ingestion from external sources (like holodeck) is documented but not implemented in current `tile_processor.py`.

## 2. fleet-simulator
**Repository:** `fleet-simulator`  
**Purpose:** Generate training data via multi-agent simulation.

### Verified Connections:
- **plato-torch:** One-way data flow.  
  - `fleet-simulator/src/simulation/tile_generator.rs` writes tiles to `output/tiles/` in JSON format matching `plato-torch`'s `Tile` spec.  
  - `plato-torch/scripts/ingest_sim_tiles.py` reads this directory, validates, and loads tiles into room training queue.  
- **holodeck-rust:** No direct connection.  
  - Simulation agents are simple rule-based (see `src/agents/basic.rs`). No sentiment or NPC integration.  
- **plato-ensign:** No connection.  
  - Simulation does not export LoRA adapters.

### Integration Gaps:
- Tiles are file-based batch transfer; no streaming API.  
- Simulation agents are not yet using `holodeck` NPCs for richer interaction.

## 3. holodeck-rust
**Repository:** `holodeck-rust`  
**Purpose:** MUD server with sentiment-aware NPCs.

### Verified Connections:
- **plato-torch:** Planned but not implemented.  
  - `holodeck-rust/README.md` states: "NPC interactions can be captured as training tiles for PLATO rooms."  
  - No import or export code found for tile handling.  
- **fleet-simulator:** No connection.  
- **plato-ensign:** No connection.  
- **Internal:** Uses `serde` for JSON, `tokio` for async networking. Exposes WebSocket server on port 8080.

### Integration Gaps:
- No tile exporter. NPC dialogues are logged locally (`logs/npc_interactions.log`) but not formatted as tiles.  
- No live connection to `plato-torch` training pipeline.  
- Sentiment engine (`src/sentiment/engine.rs`) is isolated.

## 4. plato-ensign
**Repository:** `plato-ensign`  
**Purpose:** Export room experience into LoRA adapters.

### Verified Connections:
- **plato-torch:** Direct dependency.  
  - `plato-ensign/setup.py` includes `plato-torch>=0.3.0`.  
  - `plato_ensign/lora_adapter.py` imports `from plato_torch.tile_processor import load_room_tiles`.  
- **fleet-simulator:** No connection.  
- **holodeck-rust:** No connection.

### Integration Gaps:
- Only consumes tiles from `plato-torch` rooms. No capability to train on simulation or holodeck data directly.  
- Adapter output is file-based (`adapters/`). No push to model serving infrastructure.
