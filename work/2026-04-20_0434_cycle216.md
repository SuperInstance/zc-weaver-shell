## 1. Repository Structure & Dependencies (Verified)

**plato-torch** (`plato-torch/`)
- **Purpose**: Training room presets (25 rooms), self-training rooms, tile ingestion/processing.
- **Key files**: `DeadbandRoom.py`, `plato_relay.py`, `GhostInjector.py`, `ensign_exporter.py`.
- **Dependencies**: `torch`, `transformers`, `peft` (for LoRA), `fastapi` (implied by plato-relay), `pydantic`.

**fleet-simulator** (`fleet-simulator/`)
- **Purpose**: Generate synthetic training data (agent interactions, environment states).
- **Key files**: `simulator/core.py`, `agents/`, `environments/`.
- **Dependencies**: `numpy`, `pandas`, `gym` (likely), `torch` (optional for agent brains).

**holodeck-rust** (`holodeck-rust/`)
- **Purpose**: MUD server with sentiment-aware NPCs, real-time interaction environment.
- **Key files**: `src/server.rs`, `src/npc/sentiment.rs`, `src/world/`.
- **Dependencies**: `tokio`, `serde`, `websocket`, `sqlx` (for state persistence).

**plato-ensign** (`plato-ensign/`)
- **Purpose**: Export room experience (tile accumulations) into LoRA adapters.
- **Key files**: `export.py`, `lora_builder.py`, `config/`.
- **Dependencies**: `torch`, `peft`, `transformers`, `safetensors`.

## 2. Verified Connections (What's Connected)

### A. **plato-torch → plato-ensign** (Strong, Direct)
- **Connection**: `plato-torch/ensign_exporter.py` imports from `plato-ensign/export.py`.
- **Data flow**: Room tile logs → processed into training datasets → LoRA adapters.
- **Evidence**: `ensign_exporter.py` calls `export_ensign()` from plato-ensign, passes `room_id` and `tile_log_path`.

### B. **plato-torch → holodeck-rust** (Weak, Configuration-Only)
- **Connection**: `plato-torch/config/holodeck_endpoints.json` references `holodeck-rust` server URLs.
- **Data flow**: Plato rooms *can* send tile data to holodeck NPCs via HTTP POST (configured but not enforced).
- **Evidence**: Config exists; no actual import of Rust code. Integration is via REST API.

### C. **fleet-simulator → plato-torch** (One-Way Data Provision)
- **Connection**: Simulator output formats (JSON) match `plato-torch` tile schema.
- **Data flow**: Synthetic interactions can be written to tile logs for room training.
- **Evidence**: `fleet-simulator/output_formatters.py` defines `TileSchema` class matching plato-torch's `Tile` model.

### D. **plato-torch Internal: DeadbandRoom → GhostInjector → plato-relay**
- **Connection**: `DeadbandRoom.py` uses `GhostInjector` to insert historical tiles, `plato_relay` to broadcast.
- **Data flow**: Room processes tiles → injects ghost tiles → relays to other services.
- **Evidence**: Direct Python imports within same repository.

## 3. Integration Gaps (What's Not Connected)

### A. **holodeck-rust → plato-torch** (Missing Feedback Loop)
- **Gap**: Holodeck NPC interactions are not automatically captured as tiles in plato-torch rooms.
- **Impact**: Sentiment-aware NPC experiences are isolated; cannot directly train ensigns.
- **Evidence**: No tile ingestion endpoint in holodeck-rust; no import of plato-torch modules.

### B. **fleet-simulator → holodeck-rust** (No Direct Link)
- **Gap**: Simulated agents cannot interact with holodeck MUD environment.
- **Impact**: Cannot generate synthetic data within holodeck’s sentiment-aware world.
- **Evidence**: No shared protocol; simulator outputs JSON, holodeck uses WebSocket/event stream.

### C. **plato-ensign → fleet-simulator** (No Validation Loop)
- **Gap**: Ensigns (trained LoRAs) are not automatically evaluated in the simulator.
- **Impact**: Cannot measure ensign performance in simulated environments.
- **Evidence**: No import or configuration linking these repos.

### D. **GhostInjector → holodeck-rust** (Pending Task)
- **Gap**: Ghost tiles (P0 knowledge) are not injected into holodeck NPC memory.
- **Impact**: NPCs cannot learn from dead agents' lessons.
- **Evidence**: `GhostInjector.py` currently only targets plato-torch rooms; no holodeck client.
