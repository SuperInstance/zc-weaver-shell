**Cycle:** 123

Based on reading the actual files in the fleet repos, here is the current state of integration between plato-torch, fleet-simulator, holodeck-rust, and plato-ensign.

## 1. plato-torch
**Purpose:** Training room server with 25 preset rooms. Self-training rooms generate interaction tiles.  
**Key Files:** `rooms/deadband_room.py`, `rooms/__init__.py`, `server.py`, `tile.py`  
**Current Connections:**
- **To plato-ensign:** Direct export path via `plato_ensign.export_room()` called from `DeadbandRoom.export()`. Verified in `deadband_room.py` line 47.
- **To fleet-simulator:** No direct code connection found. Plato-torch appears to operate independently as a tile server.
- **To holodeck-rust:** No direct connection. Plato-torch is Python/Flask; holodeck-rust is Rust/MUD server.

## 2. fleet-simulator
**Purpose:** Generates training data through multi-agent fleet simulations.  
**Key Files:** `src/simulator.rs`, `src/agents/`, `src/environment.rs`  
**Current Connections:**
- **To plato-torch:** No import or API call found. Fleet-simulator runs standalone simulations.
- **To plato-ensign:** No direct connection. Simulation data stays internal.
- **To holodeck-rust:** No connection found. Different domains (simulation vs. interactive MUD).

## 3. holodeck-rust
**Purpose:** Multi-User Dungeon server with sentiment-aware NPCs.  
**Key Files:** `src/main.rs`, `src/npc/`, `src/world/`  
**Current Connections:**
- **To plato-torch:** None. Holodeck is a standalone Rust TCP server.
- **To fleet-simulator:** None.
- **To plato-ensign:** None.

## 4. plato-ensign
**Purpose:** Exports room experience from plato-torch into LoRA adapters.  
**Key Files:** `ensign/exporter.py`, `ensign/lora_adapter.py`  
**Current Connections:**
- **To plato-torch:** Direct import dependency. `exporter.py` imports `DeadbandRoom` and calls its methods.
- **To fleet-simulator:** None.
- **To holodeck-rust:** None.

## Integration Gaps (P1 Safe Channels for Connection)

### Gap 1: fleet-simulator → plato-torch
**Current:** Simulations run but don't feed into training rooms.  
**Opportunity:** Could pipe simulation event logs as tiles into a dedicated plato-torch room (e.g., `SimulationRoom`).  
**Constraint:** Need to define tile schema for simulation events.

### Gap 2: holodeck-rust → plato-torch
**Current:** MUD interactions are isolated.  
**Opportunity:** Capture player/NPC dialogues as interaction tiles.  
**Constraint:** Requires tile API endpoint on plato-torch and Rust HTTP client in holodeck.

### Gap 3: plato-ensign ← fleet-simulator & holodeck-rust
**Current:** Ensign only exports from plato-torch rooms.  
**Opportunity:** Extend exporter to accept simulation logs or MUD logs as alternative data sources.  
**Constraint:** Need to adapt different data formats to room-like structure.

## Verified Active Pipeline
Only one end-to-end pipeline is currently wired:
```
plato-torch (DeadbandRoom) → plato-ensign (export) → LoRA adapter
```
This is confirmed in code. All other components operate in isolation.
