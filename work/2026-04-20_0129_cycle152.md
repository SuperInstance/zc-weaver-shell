## 1. Current Verified Connections

### **plato-torch ↔ fleet-simulator**
- **Connection:** Direct integration via `fleet-simulator` as a data source for training rooms.
- **Evidence:**  
  - `plato-torch/rooms/deadband_room.py` imports `from fleet_simulator import FleetSimulator`.  
  - The `DeadbandRoom` class initializes a `FleetSimulator` instance (`self.simulator`) to generate training episodes.  
  - Simulator episodes are converted into training tiles (`self._episode_to_tiles()`).  
- **Data Flow:**  
  `fleet-simulator` → generates episodic data → `DeadbandRoom` → converts to tiles → trains room model.

### **plato-torch ↔ plato-ensign**
- **Connection:** Optional export pathway for trained room experiences.
- **Evidence:**  
  - `plato-ensign/export.py` contains `export_room_to_lora()` function.  
  - `plato-torch/rooms/base_room.py` has a `save()` method that can serialize room state.  
  - Documentation mentions ensigns as “optional exports of room experience.”  
- **Data Flow:**  
  Trained room → serialized state → `plato-ensign` → LoRA adapter.

### **holodeck-rust ↔ (indirect)**
- **Connection:** No direct imports found between holodeck-rust and the other three repos in code.
- **Evidence:**  
  - `holodeck-rust/src/` shows a standalone MUD server with sentiment-aware NPCs.  
  - No Python bindings or Rust FFI hooks visible in the current commit.  
  - README describes it as a “sentiment-aware NPC MUD server” without explicit integration points listed.

- **Status:** Not connected.

### **Gap 3: plato-ensign ↔ fleet-simulator**
