## 1. Plato-Torch (Training Rooms)

**Location:** `plato-torch/`  
**Purpose:** 25 training room presets, self-training rooms.  
**Key Files:**  
- `rooms/deadband_room.py` — Deadband Protocol training room.  
- `rooms/__init__.py` — Room registry.  
- `trainer/self_trainer.py` — Self‑training loop.  
- `data/tile_loader.py` — Loads interaction tiles for training.

**Connections:**  
- **To Plato‑Ensign:** Exports room experience via `export_adapter()` in `plato-torch/ensign_exporter.py`. Creates LoRA adapters from trained checkpoints.  
- **To Fleet‑Simulator:** Not directly imported. However, `tile_loader.py` can load tiles from any source; simulator tiles could be ingested if formatted correctly.  
- **To Holodeck‑Rust:** No direct imports. Potential integration via tile format (JSON) or REST API calls to holodeck’s sentiment‑aware NPCs.

**Integration Gap:**  
- Plato‑torch does not call fleet‑simulator or holodeck‑rust within its codebase.  
- Tiles are loaded from local filesystem (`data/tiles/`). No live streaming from other services.

---

## 2. Plato‑Ensign (LoRA Export)

**Location:** `plato-ensign/`  
**Purpose:** Exports room experience into LoRA adapters.  
**Key Files:**  
- `export.py` — Main export script.  
- `lora_builder.py` — Builds LoRA from PyTorch checkpoints.  
- `config/export_config.yaml` — Specifies source checkpoint, output path.

**Connections:**  
- **From Plato‑Torch:** Expects a trained checkpoint file (`.pt` or `.safetensors`) produced by a plato‑torch room. Called via `plato-torch/ensign_exporter.py`.  
- **To Fleet‑Simulator / Holodeck‑Rust:** No direct links. Export is a post‑training step; it does not pull data from simulator or holodeck.

**Integration Gap:**  
- Export is a one‑way street: room → checkpoint → LoRA. No feedback loop from ensign back to rooms or other services.

---

## 3. Fleet‑Simulator (Training Data Generation)

**Location:** `fleet-simulator/`  
**Purpose:** Generates synthetic training data for fleet scenarios.  
**Key Files:**  
- `simulator/core.py` — Main simulation loop.  
- `agents/zeroclaw_agent.py` — Simulated Zeroclaw agent.  
- `output/tile_writer.py` — Writes simulation results as tiles.

**Connections:**  
- **To Plato‑Torch:** Tiles written by `tile_writer.py` follow the same JSON schema as plato‑torch’s `tile_loader.py`. Could be placed in `plato-torch/data/tiles/` for training.  
- **To Holodeck‑Rust:** No direct imports. Both generate agent‑interaction data, but formats may differ.  
- **To Plato‑Ensign:** No direct link. Simulator does not trigger LoRA exports.

**Integration Gap:**  
- Simulator runs independently. No automatic pipeline to plato‑torch training rooms. Manual file copy required.

---

## 4. Holodeck‑Rust (Sentiment‑Aware NPC MUD)

**Location:** `holodeck-rust/`  
**Purpose:** Multi‑user dungeon server with NPCs that react to player sentiment.  
**Key Files:**  
- `src/server.rs` — MUD server.  
- `src/npc/sentiment.rs` — NPC sentiment engine.  
- `src/protocol.rs` — Network protocol.

**Connections:**  
- **To Plato‑Torch:** None in code. However, holodeck could produce interaction logs that, if converted to tile format, could train plato‑torch rooms.  
- **To Fleet‑Simulator:** Both simulate agents, but no shared code or data exchange.  
- **To Plato‑Ensign:** No connection.

**Integration Gap:**  
- Holodeck is isolated. Its NPC interactions are not captured as training tiles unless a custom logger is added.

## 5. Constraint‑Theory‑Core, Cudaclaw, Flux‑Runtime

**Location:**  
- `constraint-theory-core/` — Rust geometric snapping.  
- `cudaclaw/` — GPU‑resident agent runtime.  
- `flux-runtime/` — Self‑assembling bytecode VM.

**Connections:**  
- **To Plato‑Torch / Fleet‑Simulator / Holodeck‑Rust:** No imports found in the inspected Python/Rust files. These appear to be standalone infrastructure components.

**Integration Gap:**  
- These are lower‑level systems not yet wired into the high‑level training/data pipeline.

## 6. GhostInjector & DeadbandRoom

**GhostInjector** (mentioned in task backlog) is not present in the inspected repos. Likely a planned component to inject “ghost tiles” (dead agents’ lessons) into training streams.

**DeadbandRoom** is a preset in plato‑torch (`rooms/deadband_room.py`). It implements the Deadband Protocol (P0/P1/P2) as a training environment.

**Integration Gap:**  
- DeadbandRoom is not connected to `plato-relay` (if that exists) or to live data sources.
