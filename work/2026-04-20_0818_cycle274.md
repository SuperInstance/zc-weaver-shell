# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 274  
**Phase:** 4 — Build & Test  
**Status:** Direct file inspection of fleet repositories. Focus on actual imports, configuration references, and shared data structures.  

## 1. plato-torch (Training Rooms)  
**Location:** `plato-torch/`  
**Purpose:** 25 training room presets, self-training rooms.  
**Key Files:**  
- `rooms/deadband_room.py` — Deadband Protocol training room.  
- `rooms/__init__.py` — Exports room classes.  
- `trainer/room_trainer.py` — Trains from interaction tiles.  

**Connections Found:**  
- **To plato-ensign:** `deadband_room.py` imports `from plato_ensign import export_lora` (line 47). Room can export experience to LoRA.  
- **To fleet-simulator:** No direct imports. However, `room_trainer.py` expects tiles with `simulation_data` field (line 89). Data structure suggests tiles may come from fleet-simulator.  
- **To holodeck-rust:** No imports. No shared data structures evident.  

**Integration Gap:**  
- Rooms are isolated to training; no live interaction with holodeck or simulator.  
- Tiles are consumed, but source is not wired.  

## 2. fleet-simulator (Fleet Simulation)  
**Location:** `fleet-simulator/`  
**Purpose:** Generate training data via multi-agent simulation.  
**Key Files:**  
- `src/simulation/engine.rs` — Core simulation loop.  
- `src/output/tile_writer.rs` — Writes interaction tiles to JSON.  

**Connections Found:**  
- **To plato-torch:** Tile writer outputs JSON files matching `plato-torch/trainer/tile_schema.json`. Field alignment: `agent_id`, `action`, `reward`, `simulation_data`.  
- **To holodeck-rust:** No imports. No shared protocols.  
- **To plato-ensign:** No direct link.  

**Integration Gap:**  
- Tiles are written to disk (`output/tiles/`). No active pub/sub to plato-torch rooms.  
- Simulator runs independently; not triggered by room training cycles.  

## 3. holodeck-rust (Sentiment-Aware MUD)  
**Location:** `holodeck-rust/`  
**Purpose:** Multi‑User Dungeon server with NPC sentiment.  
**Key Files:**  
- `src/server/mod.rs` — WebSocket server, event loop.  
- `src/npc/sentiment.rs` — NPC emotional state machine.  

**Connections Found:**  
- **To plato-torch:** None.  
- **To fleet-simulator:** None.  
- **To plato-ensign:** None.  
- **External:** Uses `serde` for JSON, `tokio` for async.  

**Integration Gap:**  
- Holodeck is a standalone game server. No ingestion of training tiles, no export of experience.  
- NPC sentiment system could benefit from Deadband Protocol (P0/P1/P2) but is not connected.  

## 4. plato-ensign (LoRA Exporter)  
**Location:** `plato-ensign/`  
**Purpose:** Export room experience into LoRA adapters.  
**Key Files:**  
- `src/exporter.rs` — `export_lora(room_state, output_path)`.  
- `src/adapters/lora_builder.rs` — Converts tile sequences to LoRA weights.  

**Connections Found:**  
- **To plato-torch:** Called by `deadband_room.py` after training cycles.  
- **To fleet-simulator:** None.  
- **To holodeck-rust:** None.  

**Integration Gap:**  
- Only triggered from plato‑torch rooms. No capability to export holodeck NPC interactions or simulator agent trajectories.  

## 5. cudaclaw & flux-runtime  
**Location:** `cudaclaw/`, `flux-runtime/`  
**Status:** Not inspected this cycle (outside current task scope). Preliminary note: cudaclaw is GPU‑resident agent runtime; flux‑runtime is bytecode VM. No observed connections to the four systems above.  

## 6. GhostInjector & plato-relay  
**Location:** `plato-torch/contrib/ghost_injector.py`, `plato-relay/`  
**Status:** GhostInjector reads `memory/` files and creates ghost tiles. plato‑relay is a message bus (Redis‑based).  
**Connections Found:**  
- GhostInjector outputs to `plato-torch/trainer/tile_buffer/`.  
- plato‑relay not yet integrated with any of the four systems.

4. **DeadbandRoom → plato‑relay** — Room could publish training metrics for monitoring.
