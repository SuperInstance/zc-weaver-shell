## 1. plato-torch (Training Rooms)

**Key files inspected:**
- `plato-torch/rooms/deadband_room.py`
- `plato-torch/rooms/__init__.py`
- `plato-torch/config/room_configs.py`

**Connections found:**
- **To plato-ensign:** Direct import in `deadband_room.py`:
  ```python
  from plato_ensign.export import export_lora_from_tiles
  ```
  Used in `export_training_data()` method to create LoRA adapters from accumulated tiles.
- **To fleet-simulator:** No direct imports found. However, `room_configs.py` contains a preset named `"fleet_simulator_scenario"` that references simulation parameters, suggesting configuration-level linkage.
- **To holodeck-rust:** No Python imports. A comment in `deadband_room.py` mentions `"holodeck_npc_sentiment"` as a tile metadata field, indicating data schema alignment.

**Integration point:**  
`plato-torch` rooms accumulate interaction tiles and can export them via `plato-ensign`. It expects tile metadata to include fields compatible with `holodeck-rust` NPC sentiment tracking.

## 2. fleet-simulator (Fleet Simulation)

**Key files inspected:**
- `fleet-simulator/src/simulator/core.rs`
- `fleet-simulator/Cargo.toml`

**Connections found:**
- **To plato-torch:** No Rust imports of Python modules. However, `core.rs` defines a `SimulationTile` struct with fields `room_id` and `agent_id`, matching the tile schema used in `plato-torch`.
- **To plato-ensign:** No direct reference.
- **To holodeck-rust:** No direct imports. Shared `Cargo.toml` dependencies include `serde` and `tokio`, but no crate references.

**Integration point:**  
`fleet-simulator` produces `SimulationTile` data that could be ingested by `plato-torch` rooms if formatted correctly. Currently no automated pipeline exists.

## 3. holodeck-rust (MUD Server)

**Key files inspected:**
- `holodeck-rust/src/npc/sentiment.rs`
- `holodeck-rust/src/protocol/mod.rs`

**Connections found:**
- **To plato-torch:** No direct imports. The `Sentiment` struct in `sentiment.rs` includes `tile_id: Option<String>`, which could link to tiles generated in `plato-torch` rooms.
- **To fleet-simulator:** No references.
- **To plato-ensign:** No references.

**Integration point:**  
NPC sentiment data in `holodeck-rust` can be tagged with a `tile_id`, allowing sentiment traces to be attached to training tiles. No active exporter exists.

## 4. plato-ensign (LoRA Exporter)

**Key files inspected:**
- `plato-ensign/src/export.rs`
- `plato-ensign/pyproject.toml`

**Connections found:**
- **To plato-torch:** Directly called by `plato-torch` rooms via Python import. `export.rs` expects a vector of `Tile` structs matching `plato-torch`'s tile format.
- **To fleet-simulator:** No references.
- **To holodeck-rust:** No references.

**Integration point:**  
`plato-ensign` is a downstream consumer of `plato-torch` tile data. It does not currently ingest data from `fleet-simulator` or `holodeck-rust`.

## 6. Identified Integration Gaps (P0/P1 Channels)

**P0 (Negative Space — what NOT to do):**
- Do not assume data flows automatically between repos. No streaming or event bus exists.
- Do not modify tile schemas without checking all consumers (`plato-torch`, `plato-ensign`, `holodeck-rust` metadata).

**P1 (Safe Channels — where you CAN be):**
- **plato-torch → plato-ensign:** Working export pipeline. Safe to extend with new tile types.
- **holodeck-rust → plato-torch:** Sentiment data can be attached via `tile_id`. Safe to implement a sentiment injector.
- **fleet-simulator → plato-torch:** Simulation tiles share schema. Safe to build a simulator-to-room adapter.

**P2 (Optimization — best path):**
- Build `GhostInjector` to read `holodeck-rust` sentiment logs and attach them to existing tiles in `plato-torch`.
- Create `SimulationRoom` adapter in `plato-torch` that consumes `fleet-simulator` output as training data.
- Add `plato-relay` integration to broadcast tile exports to multiple ensigns.
