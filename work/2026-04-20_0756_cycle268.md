## 1. plato-torch
**Repo:** `plato-torch`  
**Purpose:** Training room presets, self-training rooms.  
**Key Files:** `deadband_room.py`, `presets/`, `trainers/`.

### Verified Connections:
- **To plato-ensign:**  
  - `deadband_room.py` imports `from plato_ensign import export_lora`.  
  - Contains `export_ensign()` method that calls `export_lora(room_state, adapter_path)`.  
  - Direct, functional integration for exporting LoRA adapters from room state.

- **To fleet-simulator:**  
  - No direct imports found in `plato-torch` source.  
  - However, `presets/deadband_preset.yaml` references `fleet_simulator_data_path: ../fleet-simulator/output/training_scenarios/`.  
  - Config-level dependency for loading training scenarios.

- **To holodeck-rust:**  
  - No imports or config references found.  
  - No shared data structures identified.

## 2. fleet-simulator
**Repo:** `fleet-simulator`  
**Purpose:** Fleet simulation for training data.  
**Key Files:** `simulator/core.py`, `output/`, `scenarios/`.

### Verified Connections:
- **To plato-torch:**  
  - `simulator/core.py` does not import plato-torch.  
  - Outputs JSON training scenarios to `output/training_scenarios/`.  
  - One-way data flow: fleet-simulator → plato-torch (via file path).

- **To holodeck-rust:**  
  - No imports found.  
  - No references in configs or outputs.

- **To plato-ensign:**  
  - No direct connection.  
  - Simulation data could be used as training input for ensigns, but no explicit link.

## 4. plato-ensign
**Repo:** `plato-ensign`  
**Purpose:** Exports room experience into LoRA adapters.  
**Key Files:** `export.py`, `lora_builder.py`.

### Verified Connections:
- **To plato-torch:**  
  - `export.py` is imported by `plato-torch` (as above).  
  - `lora_builder.py` expects `room_state` dict with specific keys (`interactions`, `embeddings`).  
  - Tight integration: plato-torch provides data, plato-ensign processes it.

- **To fleet-simulator:**  
  - No imports or references.

- **To holodeck-rust:**  
  - No references.
