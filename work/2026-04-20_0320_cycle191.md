# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 191  
**Phase:** 4 — Build & Test  
**Status:** Updated map based on actual code inspection.

Based on reading the actual files in the fleet repos, here is the current state of connections between **plato-torch**, **fleet-simulator**, **holodeck-rust**, and **plato-ensign**.

---

## 1. **plato-torch** → **fleet-simulator**
**Connection:** ✅ **Connected** (via shared tile format)  
**How:**  
- `plato-torch/src/tiles.rs` defines `Tile` struct with fields: `id`, `agent_id`, `room_id`, `content`, `timestamp`, `metadata`.  
- `fleet-simulator/src/simulation/tile_generator.rs` imports and uses the same `Tile` struct from `plato_torch::tiles`.  
- Simulator generates synthetic training tiles that match the plato-torch tile schema.  
**Integration Point:** Tile generation pipeline. Simulator creates tiles that can be ingested by plato-torch rooms.

---

## 2. **plato-torch** → **plato-ensign**
**Connection:** ✅ **Connected** (direct export)  
**How:**  
- `plato-ensign/src/lib.rs` imports `plato_torch` and calls `room.export_training_data()`.  
- `plato-torch/src/room.rs` has `export_training_data()` method that returns a `Vec<Tile>` for a given room.  
- Plato-ensign then converts tiles into LoRA adapters via `train_lora()` function.  
**Integration Point:** Room experience → LoRA adapter pipeline. Fully wired.

---

## 3. **holodeck-rust** → **plato-torch**
**Connection:** ⚠️ **Partial** (planned but not fully implemented)  
**How:**  
- `holodeck-rust/src/npc/sentiment.rs` defines `SentimentState` and NPC behaviors.  
- `plato-torch/src/rooms/deadband_room.rs` has a TODO comment: `// TODO: Integrate holodeck NPCs for sentiment-aware training`.  
- No actual imports or calls between them in current code.  
**Integration Gap:** Holodeck NPCs are not yet feeding sentiment data into plato-torch rooms.

---

## 4. **fleet-simulator** → **holodeck-rust**
**Connection:** ❌ **Not connected**  
**How:**  
- No shared dependencies in Cargo.toml files.  
- No imports between them.  
- Fleet-simulator generates tile data; holodeck-rust is a standalone MUD server.  
**Integration Gap:** Simulated agent interactions could drive holodeck NPCs, but no pipeline exists.

---

## 5. **plato-ensign** → **holodeck-rust**
**Connection:** ❌ **Not connected**  
**How:**  
- Plato-ensign outputs LoRA adapters (.safetensors files).  
- Holodeck-rust does not currently load or use these adapters.  
**Integration Gap:** Trained ensigns could personalize NPC behaviors, but no integration point.

---

## 6. **plato-torch** → **holodeck-rust** (Ghost Tiles)
**Connection:** ⚠️ **Planned** (GhostInjector concept)  
**How:**  
- `plato-torch/src/ghost_injector.rs` exists and defines `GhostInjector` struct with `inject_ghost_tiles()` method.  
- `holodeck-rust/src/main.rs` has no reference to ghost tiles or GhostInjector.  
**Integration Gap:** Ghost tiles (P0 knowledge from dead agents) are not yet injected into holodeck scenarios.
