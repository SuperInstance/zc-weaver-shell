## 1. plato-torch
**Repo:** `plato-torch`  
**Purpose:** 25 training room presets, self-training rooms.  
**Key Files:** `DeadbandRoom.py`, `plato_torch/rooms/`, `plato_torch/training/`

### Verified Connections:
- **To plato-ensign:** Direct export path. `DeadbandRoom.py` contains `export_to_ensign()` method that saves room state and interaction tiles to a LoRA adapter directory. Uses `torch` and `peft` libraries.
- **To fleet-simulator:** No direct import found. However, `DeadbandRoom` can load simulation data from `.jsonl` files (common format with fleet-simulator output) via `load_training_data()`.
- **To holodeck-rust:** No direct connection in code. Plato-torch is Python/PyTorch; holodeck is Rust. Potential integration via shared tile format or REST API (unverified).

### Integration Gaps:
- No explicit configuration for `holodeck-rust` NPC sentiment data as a training source.
- `fleet-simulator` connection is file-based only (`jsonl`), not live API.

## 2. fleet-simulator
**Repo:** `fleet-simulator`  
**Purpose:** Fleet simulation for training data.  
**Key Files:** `src/simulator.rs`, `src/agents/`, `examples/`

### Verified Connections:
- **To plato-torch:** Outputs `.jsonl` logs of agent interactions. These can be consumed by `DeadbandRoom.load_training_data()`.
- **To holodeck-rust:** No direct import found. Both are Rust projects but separate crates.
- **To plato-ensign:** No direct connection. Simulator output must go through plato-torch first.

### Integration Gaps:
- No live streaming of simulation events to plato-torch (file batch only).
- No integration with holodeck-rust’s NPC sentiment system.

## 3. holodeck-rust
**Repo:** `holodeck-rust`  
**Purpose:** MUD server with sentiment-aware NPCs.  
**Key Files:** `src/server.rs`, `src/npc/sentiment.rs`, `src/world/`

### Verified Connections:
- **To plato-torch:** None found in code. Potential via tile export (unimplemented).
- **To fleet-simulator:** None found.
- **To plato-ensign:** None found.

### Integration Gaps:
- NPC sentiment data is trapped inside the MUD server. No export mechanism to training rooms.
- No API for external systems to inject scenarios or read live interactions.

## 4. plato-ensign
**Repo:** `plato-ensign`  
**Purpose:** Exports room experience into LoRA adapters.  
**Key Files:** `ensign.py`, `export/`, `adapters/`

### Verified Connections:
- **To plato-torch:** Direct import. `ensign.py` imports `DeadbandRoom` and calls `export_to_ensign()`.
- **To fleet-simulator:** Indirect via plato-torch (simulation data → room → ensign).
- **To holodeck-rust:** None.

### Integration Gaps:
- Only works with plato-torch rooms. Cannot directly ingest holodeck or simulator data without a room intermediary.
