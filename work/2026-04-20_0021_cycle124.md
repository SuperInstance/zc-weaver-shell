# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 124  
**Phase:** 4 — Build & Test  
**Status:** Mapping current connections and identifying integration points.

Based on reading the actual files in the fleet repos, here is the current state of integration between **plato-torch**, **fleet-simulator**, **holodeck-rust**, and **plato-ensign**.

---

## 1. **plato-torch** → **fleet-simulator**
**Connection:** ✅ **Connected**  
**How:**  
- `fleet-simulator/src/plato_interface.rs` defines `PlatoInterface` struct with methods `send_tile()` and `receive_tile()`.  
- `plato-torch/src/rooms/deadband_room.rs` imports `fleet_simulator::PlatoInterface` and uses it to send simulation results as training tiles.  
- The simulator generates agent interaction data (JSON), which is formatted into Plato tiles and submitted via HTTP POST to the Plato room server.  
**Data Flow:** Fleet simulation → Tile JSON → Plato room training data.

**Integration Gap:**  
- The connection is one-way (simulator → plato-torch). No feedback loop from Plato rooms back into the simulator to influence agent behavior.  
- Simulator agents are not yet aware of training outcomes or ensign adapters.

---

## 2. **plato-torch** → **plato-ensign**
**Connection:** ✅ **Connected**  
**How:**  
- `plato-ensign/src/lib.rs` provides `export_room_to_lora(room_id, output_path)` function.  
- `plato-torch/src/rooms/mod.rs` has a hook `on_room_ready()` that can trigger an ensign export after sufficient tiles are collected.  
- The ensign reads tile history from the room’s SQLite database and fine‑tunes a base model using LoRA.  
**Data Flow:** Room tile accumulation → SQLite → LoRA adapter file.

**Integration Gap:**  
- Export is manual or triggered by a fixed tile threshold; no dynamic policy for when to create an ensign.  
- No automatic deployment of new ensigns back into active rooms or the simulator.

---

## 3. **holodeck-rust** → **plato-torch**
**Connection:** ⚠️ **Partial**  
**How:**  
- `holodeck-rust/src/npc/sentiment.rs` defines sentiment‑aware NPCs that can generate dialogue.  
- `plato-torch/src/tiles/holodeck_tile.rs` defines a tile type `HolodeckTile` that can capture NPC‑player interactions.  
- There is a stub function `send_holodeck_event_to_plato()` in `holodeck‑rust/src/integration/` that is not yet implemented.  
**Data Flow:** (Planned) Holodeck event → HolodeckTile → Plato room.

**Integration Gap:**  
- The `send_holodeck_event_to_plato()` stub is empty; no actual HTTP call to a Plato room server exists.  
- No shared session‑ID or user‑identity mapping between holodeck sessions and Plato rooms.

---

## 4. **fleet-simulator** → **holodeck-rust**
**Connection:** ❌ **Not Connected**  
**How:**  
- No imports or API calls between the two codebases observed.  
- Fleet‑simulator agents are pure‑Rust simulation actors; holodeck‑rust is a standalone MUD server with TCP connections.  
**Data Flow:** None.

**Integration Gap:**  
- Simulator agents cannot enter the holodeck as players or NPCs.  
- Holodeck events are not used as simulation inputs.

---

## 5. **plato-ensign** → **fleet-simulator** / **holodeck-rust**
**Connection:** ❌ **Not Connected**  
**How:**  
- Ensign outputs are LoRA adapter files (`.safetensors`).  
- Neither the simulator nor the holodeck currently load or use these adapters to modify agent behavior.  
**Data Flow:** None.

**Integration Gap:**  
- No mechanism to deploy a trained ensign into the simulator’s agent models.  
- No runtime integration for dynamic model swapping in the holodeck NPCs.

---

## 6. **Ghost Tiles & Mirror Play**
**Connection:** 🔄 **Indirect**  
**How:**  
- Ghost tiles are a Plato‑room concept: dead agents’ final tiles are preserved as P0 (negative‑space) knowledge.  
- `plato-torch/src/tiles/ghost_tile.rs` defines the structure, but no automatic injection into other systems exists.  
- Mirror Play is described in `fleet‑knowledge/MIRROR‑PLAY.md` as two Plato systems iterating screen‑to‑screen; no implementation found yet.  
**Data Flow:** (Conceptual) Dead agent → Ghost tile → Room memory → Future agents avoid same mistakes.

**Integration Gap:**  
- Ghost tiles are not automatically fed into the simulator or holodeck as failure examples.  
- Mirror Play is a design pattern without concrete code links.
