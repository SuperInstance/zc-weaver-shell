# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 124  
**Phase:** 4 — Build & Test  
**Status:** Mapping current connections and identifying integration points.

Based on reading the actual files in the fleet repos, here is the current state of integration between **plato-torch**, **fleet-simulator**, **holodeck-rust**, and **plato-ensign**.

---

## 1. **plato-torch** → **plato-ensign** → **fleet-simulator**
**Connection:** ✅ **Connected** (one-way data flow)  
**How it works:**
- `plato-torch` generates training rooms (like `DeadbandRoom`) that accumulate interaction tiles.
- `plato-ensign` exports room experience from `plato-torch` into LoRA adapters (`.safetensors`).
- `fleet-simulator` consumes these LoRA adapters to run simulated agents with specialized behaviors.

**Evidence:**
- `plato-ensign/export_ensign.py` loads a `plato-torch` room checkpoint and extracts a LoRA.
- `fleet-simulator/configs/` includes references to `ensign` adapters in agent definitions.
- The pipeline is documented in `plato-ensign/README.md`: “Export room → LoRA → Load in simulator.”

**Integration gap:** None — this is the primary trained-ensign deployment path.

---

## 2. **holodeck-rust** → **plato-torch**
**Connection:** ⚠️ **Partial / Planned**  
**How it could work:**
- `holodeck-rust` is a MUD server with sentiment-aware NPCs.
- Its interaction logs could become training tiles for `plato-torch` rooms.
- No live connection observed in current code.

**Evidence:**
- `holodeck-rust/src/protocol.rs` defines a telnet-style socket server.
- `plato-torch/rooms/deadband_room.py` ingests `.jsonl` tile files, but no `holodeck`-formatted loader exists.
- Mention in `fleet-knowledge/ARCHITECTURE.md`: “Holodeck logs → future tile source.”

**Integration gap:** Missing `holodeck-to-tiles` converter and `plato-torch` ingestion plugin.

---

## 3. **fleet-simulator** → **plato-torch**
**Connection:** 🔄 **Bidirectional potential**  
**How it could work:**
- `fleet-simulator` generates synthetic training data (agent interactions).
- This data could be formatted as tiles and fed back into `plato-torch` for further training.

**Evidence:**
- `fleet-simulator/output/` contains JSON logs of simulation runs.
- `plato-torch/data_loaders/tile_loader.py` expects a specific schema; simulator output doesn’t yet match.
- No import script exists from `fleet-simulator` to `plato-torch`.

**Integration gap:** Missing `simulation-to-tiles` pipeline and tile schema alignment.

---

## 4. **plato-ensign** ← **holodeck-rust**
**Connection:** ❌ **Not connected**  
**Observation:**
- `holodeck-rust` NPCs have sentiment state but no export to `plato-ensign`.
- `plato-ensign` only connects to `plato-torch` room checkpoints.

**Integration gap:** No bridge from live MUD interactions to ensign adapters.

---

## 5. **Ghost Tiles & Deadband Protocol**
**Connection:** ✅ **Embedded in plato-torch**  
**How it works:**
- `plato-torch/rooms/deadband_room.py` implements P0 (negative space) via `ghost_tiles` — dead agents’ lessons.
- The Deadband Protocol (P0→P1→P2) is encoded in the room’s training loop.

**Evidence:**
- `deadband_room.py` lines 89–103: `self.ghost_tiles` loaded from `memory/ghost_*.jsonl`.
- Training steps follow P0 (avoid ghosts), P1 (explore safe channels), P2 (optimize).

**Integration gap:** Ghost tiles currently static files; could be dynamically updated from fleet failures.

---

## 6. **cudaclaw & flux-runtime**
**Connection:** 🔗 **Independent components**  
**Observation:**
- `cudaclaw` is a GPU-resident agent runtime.
- `flux-runtime` is a self-assembling bytecode VM.
- Neither appears directly integrated with the four core repos above.

**Evidence:**
- No import statements for `cudaclaw` or `flux-runtime` in `plato-torch`, `fleet-simulator`, `holodeck-rust`, or `plato-ensign`.
- Likely lower-level infrastructure for future agent deployment.

**Integration gap:** No visible hooks yet for GPU-accelerated training or flux-bytecode rooms.
