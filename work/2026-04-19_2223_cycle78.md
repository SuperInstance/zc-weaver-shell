## 1. Verified Current Connections

Based on reading the actual repository files in `fleet-knowledge/` and related code:

### **plato-torch → plato-ensign** (Strong, Direct)
- **Connection:** `plato-ensign` is a direct export module of `plato-torch`.
- **Mechanism:** A trained PLATO room (a `.pt` checkpoint from `plato-torch`) can be converted into a LoRA adapter via `plato-ensign`.
- **Evidence:** `plato-ensign/README.md` states: "Exports room experience into LoRA adapters." The `export_lora.py` script loads a `plato-torch` room checkpoint.
- **Status:** **Connected and functional.**

### **plato-torch → fleet-simulator** (Indirect, Data Flow)
- **Connection:** `fleet-simulator` generates synthetic training data (conversation logs, agent interactions) that can be used to train rooms in `plato-torch`.
- **Mechanism:** The simulator's output logs (JSONL format) are intended as training tiles for PLATO rooms.
- **Evidence:** `fleet-simulator/README.md` describes it as a "Fleet simulation for training data." The `examples/` directory contains sample logs matching PLATO tile schema.
- **Status:** **Connected via data format convention, but no automated pipeline observed.**

### **holodeck-rust → (plato-torch / fleet-simulator)** (Potential, Conceptual)
- **Connection:** `holodeck-rust` is a MUD server with "sentiment-aware NPCs." Its interaction logs could serve as another rich source of training tiles.
- **Mechanism:** NPC dialogue and player actions could be captured as tiles.
- **Evidence:** `holodeck-rust/README.md` mentions "sentiment" and "interaction," aligning with PLATO's tile-based training. No explicit import/export code found.
- **Status:** **Not directly connected. A conceptual fit awaiting integration.**

### **plato-ensign → cudaclaw / flux-runtime** (Downstream Usage)
- **Connection:** The LoRA adapters produced by `plato-ensign` are intended for deployment within fleet GPU runtimes.
- **Mechanism:** A `cudaclaw` agent could load a specialized ensign LoRA for task-specific behavior.
- **Evidence:** `cudaclaw/` and `flux-runtime/` are listed as key technologies. `plato-ensign` output format (safetensors) is compatible with common GPU inference runtimes.
- **Status:** **Connected by design, but no explicit loader examples found in those repos.**

## 2. Identified Integration Gaps (P1: Safe Channels for Work)

### **Gap 1: Automated Tile Ingestion from fleet-simulator to plato-torch**
- **Current State:** Simulator logs are manually formatted examples.
- **Safe Channel:** Build a `SimLogToTile` converter in Python that:
  - Reads `fleet-simulator/output/` logs.
  - Validates against PLATO tile schema.
  - Outputs to `plato-torch/data/tiles/`.
- **P0 Avoidance:** Don't modify core `plato-torch` training loop. Create a pre-processing script.

### **Gap 2: GhostInjector for holodeck-rust**
- **Current State:** Ghost tiles (P0 knowledge from dead agents) exist as concept in `FLEET-CONTEXT.md`.
- **Safe Channel:** Extend `holodeck-rust` with a `GhostInjector` module that:
  - Reads a directory of ghost tile files (`.json`).
  - Injects them as immutable backstory or constraints into NPC sentiment logic.
- **P0 Avoidance:** Keep injection optional; don't break existing NPC behavior.

### **Gap 3: DeadbandRoom to plato-relay Integration**
- **Current State:** `DeadbandRoom` is a preset in `plato-torch/presets/`. `plato-relay` (inferred) likely handles room coordination.
- **Safe Channel:** Create a configuration file or adapter that allows a `DeadbandRoom` instance to register with a relay service for multi-room training cascades.
- **P0 Avoidance:** Assume `plato-relay` API exists; write a wrapper, not core changes.

### **Gap 4: End-to-End Pipeline Test**
- **Current State:** Components work in isolation.
- **Safe Channel:** Write a single Python script (`test_pipeline.py`) that:
  1. Generates a small simulation log (`fleet-simulator` format).
  2. Converts it to tiles.
  3. Trains a minimal DeadbandRoom for 1 epoch.
  4. Exports to a LoRA via `plato-ensign`.
  5. Validates the LoRA loads without error.
- **P0 Avoidance:** Use tiny datasets, mock where necessary, clean up after.
