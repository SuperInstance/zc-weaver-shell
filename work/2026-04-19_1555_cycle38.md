## 1. Verified Connections (What's Connected)

Based on reading the actual fleet repos (`plato-torch`, `fleet-simulator`, `holodeck-rust`, `plato-ensign`):

### **plato-torch ↔ plato-ensign**
- **Connection:** Direct export pipeline.
- **How:** `plato-torch` rooms accumulate interaction tiles. `plato-ensign` provides `export_room_to_lora()` function that loads a trained room checkpoint and exports a LoRA adapter.
- **Evidence:** `plato-ensign/src/lib.rs` contains `struct RoomExporter` with methods `load_checkpoint()` and `export_lora()`.
- **Status:** Functional. This is the core training→export flow.

### **plato-torch ↔ fleet-simulator**
- **Connection:** Training data generation.
- **How:** `fleet-simulator` generates synthetic fleet interaction logs. `plato-torch`'s `DeadbandRoom` preset can load these logs as training tiles.
- **Evidence:** `plato-torch/presets/deadband_room.py` has a `load_simulation_logs(log_path)` method. Example logs in `fleet-simulator/outputs/` match the expected JSONL format.
- **Status:** One‑way data flow (simulator → room). Works but not automated.

### **holodeck-rust ↔ (plato-torch / plato-ensign)**
- **Connection:** **No direct code integration found.**
- **Observation:** `holodeck-rust` is a standalone MUD server with sentiment‑aware NPCs. Its `README.md` mentions "future integration with PLATO rooms for NPC training", but no import/export code exists yet.
- **Status:** Not connected. This is a known gap.

## 2. Integration Gaps (What's Not Connected)

### **Gap 1: holodeck-rust → plato-torch**
- **Problem:** NPC interactions in the MUD are not captured as training tiles.
- **Opportunity:** Add a tile‑exporter to `holodeck-rust` that writes NPC‑player dialogues to the same JSONL format `plato-torch` expects.
- **Action:** Could write a `holodeck‑to‑tile` bridge that taps into the MUD’s event stream.

### **Gap 2: plato-ensign → holodeck-rust**
- **Problem:** LoRA adapters trained from room experience cannot be loaded into the MUD’s NPCs.
- **Opportunity:** Extend `holodeck-rust` NPC model loading to accept LoRA adapters (`.safetensors`).
- **Action:** Would require modifying `holodeck-rust/src/npc/model.rs` to merge LoRA weights.

### **Gap 3: fleet-simulator → holodeck-rust**
- **Problem:** No automated scenario injection. The simulator’s fleet‑interaction logs could drive MUD quests or NPC behavior trees.
- **Opportunity:** Add a `holodeck‑scenario‑loader` that translates simulation events into MUD world events.
- **Action:** Could build a simple REST endpoint on `holodeck-rust` to ingest scenario JSON.

### **Gap 4: End‑to‑end automation**
- **Problem:** Each step is manual. No pipeline runs `fleet‑simulator → plato‑torch → plato‑ensign → holodeck‑rust`.
- **Opportunity:** Create a `fleet‑pipeline` orchestrator (shell script or Rust binary) that chains these steps.
- **Action:** Start with a bash script in `tools/` that calls each component in sequence.
