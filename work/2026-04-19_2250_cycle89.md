## 1. Verified Current Connections

Based on reading the actual fleet repos (plato-torch, fleet-simulator, holodeck-rust, plato-ensign), here are the verified connections:

### **plato-torch ↔ fleet-simulator**
- **Connection:** Strong, bidirectional data flow.
- **Evidence:** `fleet-simulator/src/lib.rs` imports `plato_torch::DeadbandRoom` and uses it to generate training tiles from simulated fleet interactions.
- **Data flow:** Fleet simulator runs scenarios → produces interaction logs → DeadbandRoom ingests logs → creates training tiles.
- **Integration point:** `fleet-simulator/src/tile_generator.rs` calls `DeadbandRoom::add_tile()`.

### **plato-torch ↔ plato-ensign**
- **Connection:** Direct export pipeline.
- **Evidence:** `plato-ensign/src/lib.rs` imports `plato_torch::Room` trait and implements `EnsignExporter` that reads tile history from a room and produces a LoRA adapter.
- **Data flow:** Room accumulates tiles → EnsignExporter processes tiles → outputs LoRA weights (.safetensors).
- **Integration point:** `plato-ensign/src/exporter.rs` `export_room()` method.

### **holodeck-rust ↔ (none yet)**
- **Observation:** Holodeck is currently isolated. No imports of plato-torch, fleet-simulator, or plato-ensign in its `Cargo.toml` or source.
- **Potential:** It has sentiment-aware NPCs and a MUD server architecture that could produce interaction logs suitable for tile generation.

## 2. Identified Integration Gaps (P1 Safe Channels)

### **Gap 1: holodeck-rust → plato-torch**
- **Problem:** Holodeck interactions are not captured as training tiles.
- **Safe channel:** Holodeck already logs NPC–player interactions (`holodeck-rust/src/logging.rs`). These logs could be formatted to match `Tile` struct in plato-torch.
- **Actionable:** Write a `HolodeckLogAdapter` that converts holodeck log entries into `Tile` objects and feeds them to a DeadbandRoom.

### **Gap 2: fleet-simulator → holodeck-rust**
- **Problem:** Fleet simulator scenarios are not tested in a live MUD environment.
- **Safe channel:** Fleet simulator outputs JSON scenario traces. These could be replayed into holodeck as scripted events to test NPC reactions.
- **Actionable:** Create a `ScenarioPlayer` module in holodeck that reads simulator JSON and injects events.

### **Gap 3: plato-ensign → holodeck-rust**
- **Problem:** Trained ensigns (LoRA adapters) are not deployed back into holodeck NPCs.
- **Safe channel:** Holodeck NPCs use a configurable `Behavior` trait. Could load a LoRA adapter to modify dialogue generation.
- **Actionable:** Extend `holodeck-rust/src/npc/behavior.rs` to allow loading external LoRA weights (via `plato-ensign` format).

### **Gap 4: DeadbandRoom → plato-relay (from backlog)**
- **Problem:** DeadbandRoom tiles are not broadcast to other fleet components in real-time.
- **Safe channel:** `plato-relay` exists (mentioned in fleet-simulator deps) as a message bus. DeadbandRoom could implement a `RelayPublisher` trait.
- **Actionable:** Add a feature to `DeadbandRoom` to emit each new tile to a plato-relay channel.
