# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 190  
**Phase:** 4 — Build & Test  
**Status:** Updated map based on actual code inspection.  

Based on reading the actual files in the fleet repos, here is the current state of connections between **plato-torch**, **fleet-simulator**, **holodeck-rust**, and **plato-ensign**.

---

## 1. **plato-torch** → **fleet-simulator**
**Connection:** ✅ **Connected** (via `fleet-simulator` as a data source)  
**How:**  
- `plato-torch/rooms/deadband_room.py` imports `fleet_simulator` and uses `FleetSimulator` class.  
- The room calls `simulator.generate_scenario()` to create training scenarios (e.g., "collision avoidance", "resource allocation").  
- Simulated fleet data is formatted into interaction tiles and added to the room’s training buffer.  
**Integration Gap:**  
- No bidirectional feedback loop. The simulator generates data, but the room’s training outcomes don’t influence simulator parameters.

---

## 2. **plato-torch** → **holodeck-rust**
**Connection:** ❌ **Not directly connected**  
**How:**  
- `holodeck-rust` is a standalone MUD server with sentiment-aware NPCs.  
- `plato-torch` has no imports or API calls to `holodeck-rust` in the current code.  
**Integration Gap:**  
- Potential channel: Use `holodeck-rust` NPC interactions as a source of social training tiles for `plato-torch` rooms.  
- Would require a Rust/Python bridge or HTTP API from `holodeck-rust`.

---

## 3. **plato-torch** → **plato-ensign**
**Connection:** ✅ **Connected** (via export pipeline)  
**How:**  
- `plato-ensign/src/exporter.py` imports `Room` from `plato_torch.rooms.base`.  
- The `export_room_to_lora()` function takes a trained `Room` object, extracts its interaction tiles, and fine‑tunes a base model to produce a LoRA adapter.  
**Integration Gap:**  
- Export is manual/trigger‑based, not automatic after each training cycle.

---

## 4. **fleet-simulator** → **holodeck-rust**
**Connection:** ❌ **Not connected**  
**How:**  
- `fleet-simulator` simulates physical fleet dynamics (sensors, collisions, resources).  
- `holodeck-rust` handles social NPC interactions. No shared data flow.  
**Integration Gap:**  
- Could connect by feeding `holodeck-rust` dialogue outcomes into simulator events (e.g., NPC reports affect resource allocation).

---

## 5. **fleet-simulator** → **plato-ensign**
**Connection:** ⚠️ **Indirect**  
**How:**  
- `fleet-simulator` data flows into `plato-torch` rooms, which can be exported via `plato-ensign`.  
- No direct simulator‑to‑ensign pipeline.

---

## 6. **holodeck-rust** → **plato-ensign**
**Connection:** ❌ **Not connected**  
**How:**  
- `holodeck-rust` NPC interactions are not currently captured as tiles, so they cannot be exported to ensigns.  
**Integration Gap:**  
- Add a tile‑capture system to `holodeck-rust` and an adapter in `plato-ensign` to consume Rust‑format tiles.

---
