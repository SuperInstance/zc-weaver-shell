# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 190  
**Phase:** 4 — Build & Test  
**Status:** Mapping current connections and identifying integration points.

Based on reading the actual files in the fleet repos, here is the current state of integration between **plato-torch**, **fleet-simulator**, **holodeck-rust**, and **plato-ensign**.

## 1. plato-torch (Training Rooms)
**Role:** Provides 25 preset training rooms and self‑training rooms.  
**Connections:**
- **Direct to plato-ensign:** Yes. `plato-torch/rooms/` contains `DeadbandRoom`, `MirrorRoom`, `GhostRoom`, etc. Each room can export its experience via `plato-ensign` to create LoRA adapters. The `export_lora()` method in room classes calls `plato_ensign.export()`.
- **Direct to fleet-simulator:** No. No import or call to `fleet_simulator` found in the code.
- **Direct to holodeck-rust:** No. No Rust bindings or API calls observed.

**Integration Point:** Rooms are the primary source of training tiles. Tiles are sent to a PLATO room server (not directly to the other components). The server validates tiles; accepted tiles become training data for ensigns.

## 2. plato-ensign (LoRA Exporter)
**Role:** Exports room experience into LoRA adapters.  
**Connections:**
- **Direct to plato-torch:** Yes. `plato_ensign` is imported by room classes. The `export()` function takes a room instance and its tile history, then generates a LoRA.
- **Direct to fleet-simulator:** No. No references in code.
- **Direct to holodeck-rust:** No.

**Integration Point:** Acts as a downstream processor of room data. It does not itself generate tiles; it consumes tile histories from rooms.

## 3. fleet-simulator (Fleet Simulation)
**Role:** Generates training data through simulated fleet interactions.  
**Connections:**
- **Direct to plato-torch:** No. The simulator runs independent scenarios (see `fleet-simulator/scenarios/`). No room API calls.
- **Direct to plato-ensign:** No.
- **Direct to holodeck-rust:** No.

**Integration Gap:** The simulator produces JSON logs of agent interactions, but these logs are not automatically fed into any training room. They remain isolated.

## 4. holodeck-rust (Sentiment‑Aware MUD)
**Role:** Multi‑user dungeon server with NPCs that react to player sentiment.  
**Connections:**
- **Direct to plato-torch:** No. The MUD server is a standalone Rust application.
- **Direct to plato-ensign:** No.
- **Direct to fleet-simulator:** No.

**Integration Gap:** NPC interactions and sentiment logs are not currently captured as training tiles. The MUD operates in its own silo.

## Identified Integration Gaps (P1 Safe Channels)

1. **fleet‑simulator → plato‑torch:** Simulator logs could be formatted as training tiles and injected into a dedicated room (e.g., `SimulationRoom`). This would turn simulated interactions into tile‑based training data.
2. **holodeck‑rust → plato‑torch:** MUD event streams (NPC dialogues, sentiment shifts) could be captured and sent as tiles to a `HolodeckRoom`. This would allow sentiment‑aware NPC behavior to train ensigns.
3. **GhostInjector wiring:** The `GhostInjector` (mentioned in `plato-torch/rooms/`) is designed to insert ghost tiles from dead agents. It is not yet connected to `holodeck-rust` or `fleet-simulator`. Wiring it would allow historical failure data (P0 knowledge) to populate rooms.
4. **DeadbandRoom → plato‑relay:** `DeadbandRoom` currently runs locally. Connecting it to `plato-relay` (the room server) would enable multi‑agent, distributed deadband training.

## Next Steps (P2 Optimization Within Channels)

- **Wire GhostInjector into holodeck:** Modify `holodeck-rust` to emit ghost‑tile‑compatible JSON on NPC death/reset, and pipe it to `GhostInjector`.
- **Connect DeadbandRoom to plato‑relay:** Implement WebSocket client in `DeadbandRoom` to submit tiles to a live room server.
- **Test end‑to‑end pipeline:** Create a test that (1) runs a fleet‑simulator scenario, (2) formats output as tiles, (3) sends tiles to a room, (4) exports a LoRA via plato‑ensign.
