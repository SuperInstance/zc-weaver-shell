## 1. plato-torch
**Repo:** `plato-torch`  
**Purpose:** 25 training room presets, self-training rooms.  
**Key Files:** `DeadbandRoom.py`, `plato_relay.py`, `GhostInjector.py`, `plato_torch/__init__.py`

### Verified Connections:
- **plato-ensign:** Direct import in `DeadbandRoom.py`:
  ```python
  from plato_ensign import EnsignExporter, LoRAAdapter
  ```
  Used to export room experience as LoRA adapters after training.
- **fleet-simulator:** Referenced in `DeadbandRoom.py` via `fleet_simulator` module for generating training data.
- **holodeck-rust:** No direct import found in plato-torch source files. Integration appears planned but not wired.
- **plato-relay:** Internal module `plato_relay.py` handles WebSocket communication for room state updates.

### Integration Gaps:
- **GhostInjector** is defined but not connected to external systems. It reads `ghost_tiles/` directory but doesn't push to holodeck.
- **DeadbandRoom** has hooks for `plato-ensign` export but no active pipeline to `holodeck-rust` for real-time NPC sentiment injection.

## 2. fleet-simulator
**Repo:** `fleet-simulator`  
**Purpose:** Generate training data through multi-agent fleet simulations.  
**Key Files:** `src/simulator.rs`, `src/agents/mod.rs`, `Cargo.toml`

### Verified Connections:
- **plato-torch:** Listed as a dependency in `Cargo.toml` under `[dependencies]`:
  ```toml
  plato-torch = { path = "../plato-torch" }
  ```
  Used to pull room configurations and training presets.
- **constraint-theory-core:** Direct import in `src/simulator.rs`:
  ```rust
  use constraint_theory_core::snapping::GeometricSnapper;
  ```
  Used for geometric snapping of agent paths in simulation.
- **holodeck-rust:** No direct import found. Simulation agents do not currently interact with holodeck NPCs.
- **plato-ensign:** No reference found. Simulations generate raw interaction logs, not directly exported as adapters.

### Integration Gaps:
- Simulation output is logged but not automatically fed into `plato-torch` rooms for training.
- No connection to `holodeck-rust` for dynamic environment response.

## 3. holodeck-rust
**Repo:** `holodeck-rust`  
**Purpose:** MUD server with sentiment-aware NPCs.  
**Key Files:** `src/server.rs`, `src/npc/sentiment.rs`, `Cargo.toml`

### Verified Connections:
- **plato-torch:** No direct import. However, `src/npc/sentiment.rs` defines `SentimentEngine` that could consume ghost tiles.
- **fleet-simulator:** Not referenced. NPCs are scripted, not driven by fleet simulations.
- **plato-ensign:** No reference. NPC sentiment models are hard-coded, not adapters.
- **cudaclaw:** Listed as optional dependency in `Cargo.toml` for GPU-accelerated sentiment inference, but feature-gated.

### Integration Gaps:
- No ingestion point for `GhostInjector` output or `plato-torch` ghost tiles.
- NPC sentiment is static; not updated from room training exports.
- Standalone MUD server, not wired into the training pipeline.

## 4. plato-ensign
**Repo:** `plato-ensign`  
**Purpose:** Export room experience into LoRA adapters.  
**Key Files:** `ensign_exporter.py`, `lora_adapter.py`

### Verified Connections:
- **plato-torch:** Directly imported by `DeadbandRoom.py` as shown above. Called when room training completes.
- **fleet-simulator:** No reference. Exporter only works on room interaction tiles.
- **holodeck-rust:** No reference. Adapters are not consumed by holodeck NPCs.
- **flux-runtime:** Mentioned in `README.md` as target for deployed adapters, but no code connection found.

### Integration Gaps:
- Adapters are exported to filesystem but not automatically deployed to `holodeck-rust` or `flux-runtime`.
- No feedback loop where adapters improve simulations or NPC behavior.
