Based on reading the actual files in the fleet repos, here is the current state of integration between **plato-torch**, **fleet-simulator**, **holodeck-rust**, and **plato-ensign**.

---

## 1. **plato-torch** (Training Rooms)
**Role:** Core training environment with 25 preset rooms.  
**Connections:**  
- **Direct:** Exports training tiles to `plato-ensign` via `export_room()` function.  
- **Indirect:** `DeadbandRoom` preset can be loaded by `fleet-simulator` for agent training.  
- **Gap:** No direct API call to `holodeck-rust` observed in code. Rooms are self-contained Python/ PyTorch environments.

**Integration Points:**  
- `plato-torch/rooms/deadband_room.py` → `plato-ensign/exporter.py` (LoRA export)  
- `plato-torch/presets/` → `fleet-simulator/configs/` (room config sharing)

---

## 2. **fleet-simulator** (Multi-Agent Simulation)
**Role:** Simulates fleet interactions for training data generation.  
**Connections:**  
- **Direct:** Loads `plato-torch` rooms via `room_loader.py` (references `DeadbandRoom`).  
- **Indirect:** Can produce logs that could be consumed by `holodeck-rust` as narrative events.  
- **Gap:** No observed direct connection to `plato-ensign`. Simulation data does not auto-export as LoRA adapters.

**Integration Points:**  
- `fleet-simulator/agents/` → `plato-torch/rooms/` (room instantiation)  
- `fleet-simulator/logs/` → potential → `holodeck-rust/events/` (unimplemented)

---

## 3. **holodeck-rust** (Sentiment-Aware MUD Server)
**Role:** Text-based game server with NPCs that learn from sentiment.  
**Connections:**  
- **Direct:** None observed to `plato-torch` or `fleet-simulator`.  
- **Indirect:** Could receive "ghost tiles" from dead agents (P0 knowledge), but no import mechanism seen.  
- **Gap:** Standalone Rust server; no hooks into the Python training pipeline.

**Integration Points:**  
- `holodeck-rust/src/npc/` → potential ← `plato-ensign` adapters (unused)  
- `holodeck-rust/events/` → potential ← `fleet-simulator/logs/` (unimplemented)

---

## 4. **plato-ensign** (LoRA Adapter Exporter)
**Role:** Exports room experience from `plato-torch` into lightweight LoRA adapters.  
**Connections:**  
- **Direct:** Imports `plato-torch` rooms via `from plato_torch.rooms import DeadbandRoom`.  
- **Indirect:** Could provide adapters to `holodeck-rust` NPCs or `fleet-simulator` agents, but no observed consumption.  
- **Gap:** Only exports; no round-trip re-import of adapters into active training.

**Integration Points:**  
- `plato-ensign/exporter.py` ← `plato-torch/rooms/` (room input)  
- `plato-ensign/adapters/` → potential → `holodeck-rust/npc/` (unused)

## 6. **Identified Integration Gaps (P0/P1)**
**P0 (Negative Space — What NOT to do):**  
- Do not force direct coupling between Rust and Python without a clear serialization bridge.  
- Do not modify `holodeck-rust` NPC logic without sentiment training data.

**P1 (Safe Channels — Where to Integrate):**  
- **Channel A:** `fleet-simulator` → `holodeck-rust` event stream (JSON logs over TCP/WebSocket).  
- **Channel B:** `plato-ensign` adapters → `holodeck-rust` NPC loader (adapter → Rust tensor format).  
- **Channel C:** `plato-torch` ghost tile injection into `holodeck-rust` memory system.

**P2 (Optimization — Later):**  
- Binary serialization for faster Python↔Rust data exchange.  
- Cached adapter loading for hot-swapping NPC behaviors.
