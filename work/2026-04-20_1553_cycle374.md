## 1. plato-torch → fleet-simulator
**Connection:** ✅ **Connected**  
**Mechanism:** Direct import in `plato-torch/rooms/deadband_room.py`  
**Evidence:**
```python
# deadband_room.py
from fleet_simulator.simulator import FleetSimulator
```
**Purpose:** The DeadbandRoom preset uses FleetSimulator to generate synthetic training data (agent interactions, P0/P1/P2 trajectories).  
**Data Flow:** Simulation runs produce "interaction tiles" that feed into the room's training loop.

**Gap:** No reverse dependency found. fleet-simulator does not import plato-torch.

---

## 2. plato-torch → plato-ensign
**Connection:** ✅ **Connected**  
**Mechanism:** Shared module `plato-torch/ensign/exporter.py` is referenced by both.  
**Evidence:**
- `plato-torch/rooms/base_room.py` calls `ensign.export_lora()` after training.
- `plato-ensign/export.py` imports `plato_torch` utilities for adapter extraction.

**Purpose:** Room training accumulates experience; ensign exports that experience as a LoRA adapter (specialist model).  
**Data Flow:** Room state → trained weights → LoRA adapter file (.safetensors).

**Gap:** plato-ensign is optional; rooms can train without exporting.

---

## 3. holodeck-rust → plato-torch
**Connection:** ⚠️ **Indirect / Planned**  
**Mechanism:** No direct import found. Potential integration via "sentiment-aware NPCs" generating tiles for plato-torch rooms.  
**Evidence:**
- `holodeck-rust/README.md` mentions "NPC interactions can be logged as training tiles".
- `plato-torch/rooms/__init__.py` lists `HolodeckRoom` as a preset (commented out).

**Purpose:** Use holodeck as a rich environment for generating human-like interaction data.  
**Data Flow:** Not yet implemented. Could be: holodeck event → tile format → room ingestion.

**Gap:** No actual code connection. This is a documented intention, not a wired integration.

## 4. fleet-simulator → holodeck-rust
**Connection:** ❌ **Not Connected**  
**Mechanism:** No imports or shared data structures found.  
**Evidence:**
- fleet-simulator is pure Python; holodeck-rust is Rust.
- No bridge or adapter between them in either codebase.

**Purpose:** Potential synergy: use holodeck as a visualization front-end for simulations.  
**Data Flow:** None currently.

**Gap:** Language barrier (Python/Rust). Would require an API bridge (e.g., gRPC, sockets).

---

## 5. plato-ensign → holodeck-rust
**Connection:** ❌ **Not Connected**  
**Mechanism:** No reference in either codebase.  
**Evidence:**
- plato-ensign outputs PyTorch adapters; holodeck-rust uses custom entity system.
- No loading of LoRA adapters in holodeck source.

**Purpose:** Could allow holodeck NPCs to use specialized ensign models for behavior.  
**Data Flow:** None.

**Gap:** Architecture mismatch: ensign adapters are for PyTorch LLMs; holodeck uses Rust-native AI.

## 6. Ghost Tiles Integration
**Connection:** ⚠️ **Partial**  
**Mechanism:** `plato-torch/rooms/ghost_injector.py` exists but not wired to holodeck.  
**Evidence:**
- GhostInjector reads `memory/` files (dead agents' lessons) and converts to P0 tiles.
- No reference to GhostInjector in holodeck-rust.

**Purpose:** Inject past failure lessons into holodeck NPCs to avoid repeating mistakes.  
**Data Flow:** Ghost tiles → room training → (maybe) NPC sentiment memory.

**Gap:** GhostInjector is isolated to plato-torch. Not connected to holodeck's NPC system.

---

## 7. cudaclaw / flux-runtime
**Connection:** 🔄 **Orbital**  
**Mechanism:** These are runtime systems, not directly integrated with the four core repos.  
**Evidence:**
- cudaclaw: GPU-resident agent runtime; could host plato-torch models.
- flux-runtime: self-assembling bytecode VM; no imports to/from plato-torch or holodeck.

**Purpose:** Provide execution environments for trained models / simulations.  
**Data Flow:** Potential future: ensign adapters → cudaclaw deployment.

**Gap:** No current wiring. They exist as independent fleet technologies.
