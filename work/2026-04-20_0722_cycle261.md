## 1. plato-torch → fleet-simulator
**Connection:** **Strong**  
- `plato-torch/rooms/DeadbandRoom.py` imports `fleet_simulator` directly:  
  ```python
  from fleet_simulator import FleetSimulator
  ```
- `DeadbandRoom` instantiates `FleetSimulator` as `self.simulator`.  
- Simulator provides `step()` method that returns `(observation, reward, done, info)`.  
- Room uses simulator state to generate training tiles.  
- **Data flow:** Room → Simulator (actions) → Simulator → Room (observations/rewards).

**Integration Gap:** None observed. This is a direct, functional dependency.

---

## 2. plato-torch → holodeck-rust
**Connection:** **Weak / Indirect**  
- No direct import of `holodeck-rust` found in `plato-torch` Python files.  
- However, `plato-torch/config/rooms.yaml` contains a room preset named `"holodeck"` with description: *"Connect to holodeck-rust MUD server"*.  
- The `holodeck` preset uses `room_type: "ExternalRoom"`.  
- `ExternalRoom` class (in `plato-torch/rooms/`) appears designed to connect to external services via sockets/API, but no explicit holodeck client found in current code.

**Integration Gap:**  
- Missing concrete client implementation in `plato-torch` to connect to `holodeck-rust` MUD server.  
- Configuration exists, but runtime bridge not yet wired.

---

## 3. plato-torch → plato-ensign
**Connection:** **Strong**  
- `plato-torch/export/ensign_exporter.py` exists and imports `plato_ensign`.  
- `ensign_exporter.py` defines `EnsignExporter` class with methods:  
  - `export_room(room_name, adapter_path)`  
  - `export_tiles(tile_batch, adapter_path)`  
- `plato-torch/rooms/base.py` has a method `export_to_ensign()` that calls the exporter.  
- `plato-torch/config/export.yaml` contains ensign export configurations (LoRA rank, alpha, target modules).

**Integration Gap:** None observed. Export pipeline appears fully defined.

---

## 4. fleet-simulator → holodeck-rust
**Connection:** **None**  
- `fleet-simulator` is a pure Python simulation of fleet dynamics (ships, sensors, maneuvers).  
- No references to `holodeck-rust` in its code or configuration.  
- Different domains: fleet simulation vs. sentiment-aware MUD.

**Integration Gap:** Not applicable — separate concerns.

## 5. fleet-simulator → plato-ensign
**Connection:** **None**  
- `fleet-simulator` does not import `plato_ensign`.  
- Its output is consumed by `plato-torch` rooms, which may export to ensign.

**Integration Gap:** Not applicable — indirect via plato-torch.

## 6. holodeck-rust → plato-ensign
**Connection:** **Potential**  
- `holodeck-rust` is a Rust MUD server with NPC sentiment systems.  
- No direct reference to `plato-ensign` in its source (`src/`).  
- However, its `README.md` mentions: *"NPC memories can be exported as training data."*  
- Possible integration point: holodeck interaction logs → tile format → ensign exporter.

**Integration Gap:**  
- No defined data pipeline from holodeck to ensign.  
- Missing adapter to convert MUD log format to plato-torch tile format.

## 7. GhostInjector Status
**Component:** `plato-torch/rooms/ghost_injector.py`  
**Connections:**  
- Imports `DeadbandRoom` and `Tile`.  
- Reads `memory/ghost_tiles/` directory.  
- Injects ghost tiles into room training loop via `inject_ghost_tiles(room)`.  
- **Not connected to holodeck-rust** currently.  
- Ghost tiles are P0 (negative space) examples from terminated agents.

**Integration Gap:**  
- GhostInjector operates only within plato-torch.  
- No hook to inject ghost memories into holodeck NPCs.
