# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 107  
**Phase:** 4 — Build & Test  
**Status:** Mapping current connections and identifying integration points.

Based on reading the actual files in the fleet repos, here is the current state of connections between **plato-torch**, **fleet-simulator**, **holodeck-rust**, and **plato-ensign**.

## 1. Verified Connections (What's Connected)

### plato-torch ↔ fleet-simulator
- **Connection:** Strong, bidirectional data flow.
- **Evidence:** `plato-torch/src/rooms/deadband_room.py` imports `fleet_simulator` and uses `FleetSimulator` class to generate training data.
- **Data Flow:**  
  - `DeadbandRoom` calls `fleet_simulator.generate_scenario()` to create simulation episodes.  
  - Simulation outputs (agent actions, rewards, state transitions) are fed into the room as training tiles.
- **Integration Point:** `DeadbandRoom` is preset #7 in plato-torch, explicitly designed to consume fleet-simulator data.

### plato-torch ↔ plato-ensign
- **Connection:** Direct export pipeline.
- **Evidence:** `plato-ensign/src/exporter.py` has `export_room_to_lora(room_id, adapter_path)` function.
- **Data Flow:**  
  - A trained room in plato-torch (after accumulating tiles) can be exported as a LoRA adapter via plato-ensign.  
  - The exporter reads room state and tile history, then fine-tunes a base model.
- **Integration Point:** Plato-ensign is called as a post-training step; not yet automated in CI.

### fleet-simulator ↔ holodeck-rust
- **Connection:** Planned but not yet implemented.
- **Evidence:** `fleet-simulator/docs/integration_plan.md` lists "holodeck-rust NPC sentiment as simulation modifier" as a future feature.
- **Data Flow:** *Potential* — NPC emotional states from holodeck could influence agent reward shaping in simulations.
- **Integration Point:** None in current code. Marked as `TODO` in simulator config.

## 2. Integration Gaps (What's Not Connected)

### holodeck-rust ↔ plato-torch
- **Gap:** No direct connection.
- **Evidence:** holodeck-rust is a standalone MUD server (`src/server.rs`). No imports of plato-torch, no API endpoints for tile ingestion.
- **Opportunity:** NPC interactions could become training tiles for sentiment-aware rooms.

### holodeck-rust ↔ plato-ensign
- **Gap:** No connection.
- **Evidence:** plato-ensign only exports from plato-torch rooms. No hooks for MUD data.
- **Opportunity:** Could export "NPC personality adapters" from holodeck session logs.

### fleet-simulator ↔ plato-ensign
- **Gap:** Simulation data not directly exportable as adapters.
- **Evidence:** plato-ensign requires a plato-torch room as source. No `export_simulation_to_lora` function.
- **Opportunity:** Could create a "simulation room" preset that wraps fleet-simulator episodes for direct ensign export.

## 3. Immediate Integration Points (P1 Safe Channels)

Based on the gaps, here are actionable integration points:

1. **GhostInjector into holodeck**  
   - **What:** Inject ghost tiles (P0 lessons) from plato-torch into holodeck as NPC dialogue or environmental hints.  
   - **How:** Add a `GhostTileLoader` module in holodeck-rust that reads from `plato-torch/ghost_tiles/`.  
   - **P0 Check:** Ensure tile format compatibility (JSON). No live network calls initially.

2. **DeadbandRoom to plato-relay**  
   - **What:** Connect DeadbandRoom to the fleet's message bus for real-time tile distribution.  
   - **How:** Implement `PlatoRelayClient` in `deadband_room.py` to publish tiles.  
   - **P0 Check:** Use mock relay first to avoid breaking existing room training.

3. **Holodeck sentiment → Fleet simulator**  
   - **What:** Feed NPC sentiment scores into simulation reward functions.  
   - **How:** Extend `fleet_simulator/scenario.py` with `apply_sentiment_modifier(sentiment_json)`.  
   - **P0 Check:** Start with static sentiment files, not live holodeck connection.
