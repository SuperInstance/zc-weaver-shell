# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 295  
**Phase:** 4 — Build & Test  
**Status:** Direct file inspection of fleet repositories. Focus on actual imports, configuration references, and shared data structures.

## 1. plato-torch → fleet-simulator
**Connection:** Strong, direct integration via `SimulationRoom` preset.  
**Evidence:**  
- `plato-torch/rooms/simulation_room.py` imports `from fleet_simulator.simulator import FleetSimulator`.  
- `SimulationRoom` initializes a `FleetSimulator` instance and calls `simulator.step()` during training cycles.  
- Shared data: `FleetSimulator` returns `metrics: Dict[str, float]` and `trajectories: List[Dict]` which become training tiles.  
**Gap:** No bidirectional control flow observed. `plato-torch` drives simulation but does not yet allow the simulator to influence room parameters dynamically.

## 2. plato-torch → holodeck-rust
**Connection:** Weak, indirect via shared tile format.  
**Evidence:**  
- `holodeck-rust/src/protocol.rs` defines `Tile` struct with fields `agent_id`, `timestamp`, `content`.  
- `plato-torch/tiles/tile.py` defines `Tile` class with identical fields (`agent_id`, `timestamp`, `content`).  
- No direct imports between codebases. Tiles are exchanged via JSON over HTTP (implied by `plato-relay`).  
**Gap:** No Rust-Python bindings observed. Integration appears mediated by a relay service (not yet inspected).

## 3. plato-torch → plato-ensign
**Connection:** Strong, direct export pipeline.  
**Evidence:**  
- `plato-ensign/export.py` imports `from plato_torch.rooms.base import BaseRoom`.  
- `export_room_to_lora(room: BaseRoom, output_path: str)` function extracts adapter weights from a trained room.  
- `plato-torch/rooms/deadband_room.py` includes `export_hook()` method that calls `plato_ensign.export_room_to_lora()`.  
**Gap:** Export is optional (triggered manually). No automatic export on training completion observed.

## 4. fleet-simulator → holodeck-rust
**Connection:** None observed.  
**Evidence:**  
- `fleet-simulator/src/` contains no Rust code, no references to `holodeck`.  
- `holodeck-rust/Cargo.toml` lists no dependencies related to simulation.  
**Gap:** Simulated agent trajectories not yet feeding into holodeck NPC sentiment system.

## 5. fleet-simulator → plato-ensign
**Connection:** None observed.  
**Evidence:**  
- `plato-ensign` only imports from `plato-torch`. No simulator references.  
- `fleet-simulator` outputs metrics/trajectories but not in LoRA adapter format.  
**Gap:** Simulation data not directly convertible to ensign adapters; requires `plato-torch` room as intermediary.

## 6. holodeck-rust → plato-ensign
**Connection:** None observed.  
**Evidence:**  
- `holodeck-rust` outputs tiles (JSON). `plato-ensign` expects a `BaseRoom` object.  
- No Rust-to-Python room conversion observed.  
**Gap:** Holodeck NPC interactions cannot be directly exported as ensigns.

## Integration Summary
```
plato-torch ────┬───▶ fleet-simulator (drives simulation)
                ├───▶ plato-ensign (exports room → LoRA)
                └───○ holodeck-rust (indirect, via tile format)
fleet-simulator ────○ (no direct links to holodeck or ensign)
holodeck-rust ──────○ (no direct links to simulator or ensign)
```
