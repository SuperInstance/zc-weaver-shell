## 1. Verified Current Connections (What's Connected)

Based on reading the fleet-knowledge files and repository READMEs:

### **plato-torch ↔ plato-ensign** (Strong Connection)
- **plato-torch** generates training rooms and accumulates interaction tiles.
- **plato-ensign** exports room experience (tiles) into LoRA adapters.
- **Connection:** Direct pipeline — rooms train, ensigns export. This is the core training loop.

### **plato-torch ↔ fleet-simulator** (Weak/Planned Connection)
- **fleet-simulator** generates synthetic training data (simulated fleet interactions).
- **plato-torch** could ingest this data as pre‑tiles or simulation‑based rooms.
- **Current gap:** No explicit import mechanism documented. Likely a manual or script‑based transfer.

### **holodeck-rust ↔ (plato-torch / fleet-simulator)** (Indirect Connection)
- **holodeck-rust** is a MUD server with sentiment‑aware NPCs.
- Potential source of rich, narrative interaction tiles for plato‑torch rooms.
- Potential testbed for fleet‑simulator agents (NPCs as simulated agents).
- **Current gap:** No documented tile‑export or live data‑pipe to plato‑torch.

### **constraint-theory-core ↔ fleet-simulator** (Potential Connection)
- **constraint-theory-core** provides geometric snapping (spatial constraints).
- Could be used by fleet‑simulator for environment/agent spatial reasoning.
- **Current gap:** No reference in fleet‑simulator docs.

### **cudaclaw ↔ flux-runtime** (Architectural Symbiosis)
- **cudaclaw** is a GPU‑resident agent runtime.
- **flux-runtime** is a self‑assembling bytecode VM.
- **Connection:** Complementary low‑level execution layers — cudaclaw for GPU‑native agents, flux for portable bytecode.

## 2. Identified Integration Gaps (What's Not Connected)

### **Gap 1: holodeck-rust → plato-torch tile pipeline**
- **Problem:** Holodeck interactions are not automatically captured as training tiles.
- **Action:** Wire a GhostInjector‑style logger to export JSONL interaction logs to a plato‑torch room directory.

### **Gap 2: fleet-simulator → plato-torch data ingestion**
- **Problem:** Simulated fleet data exists in isolation.
- **Action:** Create a `SimulationRoom` preset in plato‑torch that loads fleet‑simulator output as tile batches.

### **Gap 3: DeadbandRoom ↔ plato-relay**
- **Problem:** DeadbandRoom (a plato‑torch preset) is not connected to the fleet’s real‑time messaging layer.
- **Action:** Integrate plato‑relay client into DeadbandRoom to stream P0/P1/P2 decisions to the fleet log.

### **Gap 4: constraint-theory-core → holodeck-rust spatial NPCs**
- **Problem:** NPC movement and interaction lack geometric constraint logic.
- **Action:** Embed constraint‑theory‑core as a library in holodeck‑rust for snapping NPCs to nav‑meshes or object surfaces.

### **Gap 5: cudaclaw/flux-runtime → plato-ensign deployment**
- **Problem:** Ensign LoRAs are trained but not deployed to GPU‑resident or bytecode‑runtime agents.
- **Action:** Add export targets in plato‑ensign for cudaclaw‑compatible adapters and flux‑runtime module formats.
