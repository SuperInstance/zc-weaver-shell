# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 328  
**Phase:** 4 — Build & Test  
**Status:** Direct file inspection of fleet repositories. Focus on actual imports, configuration references, and shared data structures.

## 1. plato-torch
**Location:** `plato-torch/`  
**Purpose:** Training room presets, self-training rooms, tile processing.

### Verified Connections:
- **To plato-ensign:** Direct import in `plato-torch/src/export.rs`.  
  - Exports `RoomExperience` struct to `plato-ensign/src/lora_adapter.rs`.  
  - Uses `serde` for serialization; shared `room_experience.json` schema.  
- **To fleet-simulator:** No direct import found.  
  - However, `plato-torch/config/training_presets.yaml` references `fleet-simulator` as a data source for scenario `deadband_room`.  
- **To holodeck-rust:** No direct import.  
  - `plato-torch/README.md` mentions "sentiment-aware NPCs from holodeck" as a future integration point.

### Integration Gaps:
- No live data pipe from `fleet-simulator` to `plato-torch`. Configuration suggests offline dataset usage.  
- `holodeck-rust` NPCs are not yet wired into training rooms.

---

## 2. fleet-simulator
**Location:** `fleet-simulator/`  
**Purpose:** Generate training data via multi-agent fleet simulations.

### Verified Connections:
- **To plato-torch:** Indirect.  
  - `fleet-simulator/output/` produces JSON logs (`simulation_*.json`).  
  - `plato-torch/scripts/convert_sim_logs.py` reads these logs to create training tiles.  
- **To holodeck-rust:** No direct import.  
  - `fleet-simulator/src/scenarios/mod.rs` includes a placeholder `holodeck_scenario.rs` (empty).  
- **To plato-ensign:** No direct import.

### Integration Gaps:
- Simulation logs are file‑based, not streamed.  
- `holodeck-rust` scenario is stubbed but not implemented.

---

## 3. holodeck-rust
**Location:** `holodeck-rust/`  
**Purpose:** MUD server with sentiment‑aware NPCs, real‑time interaction environment.

### Verified Connections:
- **To plato-torch:** None.  
- **To fleet-simulator:** None.  
- **To plato-ensign:** None.  
- **To cudaclaw:** In `holodeck-rust/Cargo.toml`: `cudaclaw = { path = "../cudaclaw" }`.  
  - Uses `cudaclaw::AgentRuntime` for GPU‑resident NPC logic.

### Integration Gaps:
- Standalone MUD server; not connected to training pipeline.  
- NPC sentiment data is not exported to any training room.  
- No ingestion of fleet‑simulation data.

---

## 4. plato-ensign
**Location:** `plato-ensign/`  
**Purpose:** Export room experience into LoRA adapters.

### Verified Connections:
- **To plato-torch:** Direct import in `plato-ensign/src/lib.rs`.  
  - `use plato_torch::export::RoomExperience;`  
  - Expects `room_experience.json` from a trained room.  
- **To fleet-simulator:** None.  
- **To holodeck-rust:** None.

### Integration Gaps:
- Only consumes static room exports; no live training loop.  
- No integration with `holodeck-rust` NPC interactions.

## 6. Key Integration Gaps (P0/P1)

**P0 (Negative Space):**  
- Do not assume `holodeck-rust` is connected to the training pipeline. It is isolated.  
- Do not stream `fleet-simulator` data directly; it is file‑based.

**P1 (Safe Channels):**  
- `plato-torch` → `plato-ensign` is a verified, working export channel.  
- `fleet-simulator` → `plato-torch` via log conversion script is a safe offline channel.  
- `holodeck-rust` → `cudaclaw` is a live GPU‑runtime channel.

**P2 (Optimization Opportunities):**  
- Wire `holodeck-rust` NPC interactions into `plato-torch` as a real‑time tile source.  
- Stream `fleet-simulator` logs via a message queue (e.g., Redis) instead of files.  
- Connect `plato-ensign` to `holodeck-rust` to export NPC sentiment adapters.
