## 1. plato-torch → fleet-simulator
**Connection:** **Direct import**  
- `plato-torch/src/rooms/deadband_room.py` imports `fleet_simulator` as a module.  
- Line 23: `from fleet_simulator import FleetSimulator, AgentState`.  
- The `DeadbandRoom` class instantiates a `FleetSimulator` object (`self.simulator`) in its `__init__`.  
- The room uses the simulator to generate training episodes (agent trajectories) which become training tiles.  
- **Data flow:** Simulator produces `AgentState` objects → room converts them to interaction tiles → tiles stored in room's memory.

**Integration Gap:**  
- The simulator appears to be a **library dependency**, not a networked service. No HTTP/WebSocket connection observed in code.  
- Configuration sharing unclear: simulator config (e.g., number of agents, environment params) is not explicitly passed from room config files.

## 2. plato-torch → holodeck-rust
**Connection:** **None found**  
- Searched `plato-torch/` for "holodeck" — no imports, no references.  
- No Rust bindings (`pyo3`) observed in Python code.  
- No shared configuration files linking the two.

**Integration Gap:**  
- **Major disconnect.** The MUD server (holodeck-rust) is sentiment-aware NPC environment; plato-torch rooms are training environments. They could be complementary (NPCs as training partners) but currently isolated.  
- Potential integration point: `holodeck-rust` could emit NPC interaction logs → `plato-torch` ingests as training tiles.

## 3. plato-torch → plato-ensign
**Connection:** **Export pipeline**  
- `plato-ensign/` is a separate repo.  
- `plato-torch/src/export/` contains `ensign_exporter.py`.  
- Exporter reads room memory (tiles) → converts to LoRA training dataset → saves adapters.  
- **Data flow:** Room tiles → exporter → LoRA weights (.safetensors).  

**Integration Gap:**  
- Exporter is **optional** and run manually; not automatically triggered by room training cycles.  
- No observed feedback loop: ensign adapters are not loaded back into rooms for self-improvement.

## 4. fleet-simulator → holodeck-rust
**Connection:** **None found**  
- `fleet-simulator/` is Python; `holodeck-rust/` is Rust.  
- No cross-references in code or config.

**Integration Gap:**  
- Simulator generates synthetic agent data; holodeck generates human-like NPC interactions. Could be fused for richer training data (synthetic + organic). Currently separate silos.

## 5. fleet-simulator → plato-ensign
**Connection:** **Indirect via plato-torch**  
- Simulator data flows to ensign only if:  
  1. Simulator → plato-torch room (as tiles)  
  2. Room → ensign exporter (as dataset)  
- No direct simulator → ensign observed.

## 6. holodeck-rust → plato-ensign
**Connection:** **None found**  
- No exporter for holodeck logs to LoRA format observed.

## 7. cudaclaw, flux-runtime, constraint-theory-core
**Connection:** **Not integrated with above**  
- These are standalone technologies listed in fleet context but **no imports or references** in the four repos examined.  
- `cudaclaw/` is a CUDA runtime; `flux-runtime/` is a bytecode VM; `constraint-theory-core/` is geometric snapping library.  
- **Gap:** Specialized capabilities not yet wired into the training pipeline.
