## 1. plato-torch
**Primary role:** Training room server with 25 preset rooms, self-training capability.  
**Key files:** `rooms/deadband_room.py`, `rooms/__init__.py`, `config/room_presets.yaml`.

**Connections found:**
- **To plato-ensign:** Explicit import in `rooms/deadband_room.py`:
  ```python
  from plato_ensign.export import export_room_lora
  ```
  Used in `DeadbandRoom.finalize()` to export LoRA adapters.
- **To fleet-simulator:** No direct imports found in plato-torch codebase. However, `config/room_presets.yaml` includes a preset named `fleet_simulator_scenario` with description: "Training room that ingests fleet-simulator generated trajectories." This indicates configuration-level integration.
- **To holodeck-rust:** No imports or direct references found.

**Data flow observed:**  
Rooms accumulate interaction tiles → tiles stored in room’s memory buffer → on `finalize()`, tiles can be exported via `plato_ensign.export_room_lora()` to produce a LoRA adapter.

## 2. fleet-simulator
**Primary role:** Generates synthetic training data (trajectories, agent interactions) for fleet training.  
**Key files:** `src/simulator/core.rs`, `src/export/mod.rs`.

**Connections found:**
- **To plato-torch:** No Rust imports of Python modules. However, `src/export/mod.rs` contains `export_to_plato_torch()` function that writes `.jsonl` files in a format matching plato-torch’s tile schema (`tile_type`, `content`, `metadata`). This is a one-way data export.
- **To holodeck-rust:** No direct imports found.
- **To plato-ensign:** No references.

**Data flow observed:**  
Simulation runs → trajectories logged → `export_to_plato_torch()` converts them to JSONL → files can be ingested by a plato-torch room (preset `fleet_simulator_scenario`).

## 4. plato-ensign

## Identified Gaps (P0 / P1)

**P0 – Negative space (what NOT to do):**
- Do not assume holodeck‑rust automatically feeds into plato‑torch. No integration exists.
- Do not assume fleet‑simulator runs inside plato‑torch. It is a separate data generator.
- Do not assume plato‑ensign trains from live rooms; it only processes exported tile lists.

**P1 – Safe channels (where integration CAN happen):**
- **plato‑torch ← fleet‑simulator:** Already configured via preset. Safe to extend with more scenario types.
- **plato‑torch → plato‑ensign:** Working export path. Safe to add more export formats or adapter types.
- **holodeck‑rust → plato‑torch:** No current path. Could build an exporter in holodeck that writes tiles in plato‑torch JSONL format.
- **holodeck‑rust → fleet‑simulator:** Could simulate NPC‑agent interactions as training trajectories.

**P2 – Optimization opportunities:**
- Add real‑time tile streaming from holodeck to a plato‑torch room.
- Let fleet‑simulator ingest holodeck logs to improve NPC behavior.
- Cache tokenized tiles between plato‑torch and plato‑ensign to speed up LoRA training.
