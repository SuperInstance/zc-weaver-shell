# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 263  
**Phase:** 4 — Build & Test  
**Status:** Direct file inspection of fleet repositories. Focus on actual imports, configuration references, and shared data structures.

## 1. plato-torch
**Location:** `plato-torch/`  
**Purpose:** Training room presets, self-training rooms, tile processing.

### Verified Connections:
- **plato-ensign:** Direct import.  
  `plato-torch/src/rooms/deadband_room.py` imports `from plato_ensign import export_lora`.  
  `DeadbandRoom` has a method `export_to_ensign()` that calls `export_lora()`.
- **fleet-simulator:** Indirect via data flow.  
  `plato-torch/src/data/tile_loader.py` loads `.jsonl` tile files. Fleet-simulator generates training data in this format (confirmed by `fleet-simulator/README.md`).
- **holodeck-rust:** No direct import found. No Rust bindings or API calls in Python source.
- **constraint-theory-core:** No direct import. Not referenced in code.
- **cudaclaw:** No direct import. Not referenced.
- **flux-runtime:** No direct import. Not referenced.

### Integration Points:
- `DeadbandRoom` is the main preset. It ingests tiles, trains, and can export to ensign.
- Tiles are stored in `plato-torch/data/tiles/`. This is the shared data format.
- Configuration via `plato-torch/config/room_presets.yaml` defines room parameters.

## 2. fleet-simulator
**Location:** `fleet-simulator/`  
**Purpose:** Generate synthetic training data (tiles) for fleet agents.

### Verified Connections:
- **plato-torch:** One-way data flow.  
  `fleet-simulator/src/generators/agent_tile_generator.rs` outputs `.jsonl` files. These match the tile schema expected by `plato-torch/src/data/tile_schema.py`.
- **holodeck-rust:** No direct import. Separate codebase.
- **plato-ensign:** No direct import.
- **constraint-theory-core:** No direct import.
- **cudaclaw:** No direct import.
- **flux-runtime:** No direct import.

### Integration Points:
- Output directory: `fleet-simulator/output/tiles/`. This can be symlinked or copied to `plato-torch/data/tiles/` for training.
- Tile schema alignment is implicit but not enforced by shared library.

## 3. holodeck-rust
**Location:** `holodeck-rust/`  
**Purpose:** MUD server with sentiment-aware NPCs, real-time interaction environment.

### Verified Connections:
- **plato-torch:** No direct import. However, `holodeck-rust/README.md` mentions "tile ingestion planned" — not implemented.
- **fleet-simulator:** No direct import.
- **plato-ensign:** No direct import.
- **constraint-theory-core:** **Direct dependency.**  
  `holodeck-rust/Cargo.toml` includes `constraint-theory-core = { path = "../constraint-theory-core" }`. Used for spatial snapping of NPCs.
- **cudaclaw:** No direct import.
- **flux-runtime:** No direct import.

### Integration Points:
- Uses `constraint-theory-core` for geometry.  
  `holodeck-rust/src/npc/movement.rs` imports `use constraint_theory_core::snap::SnapEngine;`.
- TCP server on port `8080` (configurable). Could be connected to an agent runtime.
- No current tile export or training loop.

## 4. plato-ensign
**Location:** `plato-ensign/`  
**Purpose:** Export room experience into LoRA adapters.

### Verified Connections:
- **plato-torch:** Direct import (as seen above).  
  `plato-ensign/src/lib.rs` provides `pub fn export_lora(...)` called by `DeadbandRoom`.
- **fleet-simulator:** No direct import.
- **holodeck-rust:** No direct import.
- **constraint-theory-core:** No direct import.
- **cudaclaw:** No direct import.
- **flux-runtime:** No direct import.

### Integration Points:
- Called from `plato-torch` after room training.
- Outputs LoRA weights (`.safetensors`) to `plato-ensign/output/`.
- Configurable via `plato-ensign/config/export_config.yaml`.

## 5. Other Technologies
- **constraint-theory-core:** Used only by `holodeck-rust`. Not integrated with others.
- **cudaclaw:** No imports found in any of the four repos. Likely standalone GPU runtime.
- **flux-runtime:** No imports found. Self-assembling bytecode VM, not yet wired.
