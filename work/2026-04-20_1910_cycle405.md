## 1. plato-torch → fleet-simulator
**Connection:** **Yes** — Direct import and data flow.  
**Evidence:**  
- `plato-torch/rooms/deadband_room.py` imports `from fleet_simulator.simulator import FleetSimulator`.  
- The `DeadbandRoom` class initializes a `FleetSimulator` instance (`self.simulator = FleetSimulator()`).  
- Room steps call `self.simulator.step()` and use `self.simulator.get_state()` to generate training observations.  
**Data Flow:** Simulation state → room observation tiles.  
**Integration Gap:** None observed; this is a working, coupled integration.

---

## 2. plato-torch → holodeck-rust
**Connection:** **No** — No direct imports or references found.  
**Evidence:**  
- Searched `plato-torch/` for "holodeck" (no matches).  
- No Rust bindings or foreign‑function interface in plato‑torch.  
- `holodeck‑rust/` is a standalone MUD server (TCP, WebSocket) with no Python dependencies.  
**Integration Gap:** No bridge between Plato training rooms and holodeck’s sentiment‑aware NPCs. Potential channel: tile export/import or shared session IDs.

---

## 3. plato-torch → plato-ensign
**Connection:** **Yes** — Optional export pipeline.  
**Evidence:**  
- `plato‑ensign/export.py` imports `from plato_torch.rooms.base_room import BaseRoom`.  
- `plato‑ensign/train.py` loads room checkpoints and tile logs from `plato‑torch/data/`.  
- The ensign training script expects a `room_class` argument (e.g., `DeadbandRoom`).  
**Data Flow:** Room experience (tiles, checkpoints) → LoRA adapters.  
**Integration Gap:** The export is manual/script‑based; no automatic triggering from room events.

---

## 4. fleet-simulator → holodeck-rust
**Connection:** **No** — No cross‑references.  
**Evidence:**  
- `fleet‑simulator/` is pure Python, no Rust code.  
- `holodeck‑rust/` has no Python bindings or simulation hooks.  
**Integration Gap:** Separate domains: simulation (agent‑fleet training) vs. interactive narrative (MUD). Could be linked via shared agent personas or scenario definitions.

---

## 5. fleet-simulator → plato-ensign
**Connection:** **Indirect** — via plato‑torch.  
**Evidence:**  
- `plato‑ensign` does not import `fleet‑simulator` directly.  
- Simulation data reaches ensign only after being logged as tiles in a Plato room.  
**Integration Gap:** No direct ensign training on raw simulator trajectories; requires room intermediation.

---

## 6. holodeck-rust → plato-ensign
**Connection:** **No** — No references in either direction.  
**Evidence:**  
- `holodeck‑rust/` outputs game‑state logs, not Plato‑compatible tiles.  
- `plato‑ensign/` expects Plato room checkpoints.  
**Integration Gap:** Holodeck session data cannot currently be converted to ensign adapters.
