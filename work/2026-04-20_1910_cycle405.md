## 1. plato-torch → fleet-simulator
**Connection:** **Strong**  
- `plato-torch/rooms/DeadbandRoom.py` imports `fleet_simulator` directly:  
  ```python
  from fleet_simulator import FleetSimulator
  ```
- `DeadbandRoom` instantiates `FleetSimulator` as `self.simulator`.  
- Simulator provides `step()` method called during room training cycles.  
- Data flow: Room → Simulator (agent actions) → Simulator returns observations.  
- **Integration point:** `DeadbandRoom` configures simulator with agent count and scenario.

**Gap:**  
- Simulator output not yet formatted as training tiles. Room logs interactions but tile serialization appears separate.  
- No observed import of `fleet_simulator` in other room presets (only DeadbandRoom).

## 2. plato-torch → holodeck-rust
**Connection:** **None observed**  
- No Python imports of `holodeck` in `plato-torch/`.  
- No Rust bindings or FFI calls found in plato-torch codebase.  
- `holodeck-rust` is a standalone MUD server (TCP, WebSocket).  
- **Potential link:** Both systems generate "interaction data" but formats differ (holodeck: JSON events, plato-torch: tile dictionaries).

**Gap:**  
- No direct data exchange. Could be integrated via:  
  1. `holodeck` emitting events to a `plato-torch` room socket.  
  2. Shared storage (e.g., Redis) for interaction logs.  
- Ghost tiles (P0 knowledge) from holodeck not ingested into plato-torch training.

## 3. plato-torch → plato-ensign
**Connection:** **Strong**  
- `plato-ensign/` contains `export_lora.py` which imports `torch` and `peft`.  
- `plato-torch/rooms/` generates `.pt` checkpoint files (trained weights).  
- `plato-ensign` reads these checkpoints, extracts adapter weights, saves as LoRA (`.safetensors`).  
- **Integration point:** File system — checkpoints written by plato-torch, read by plato-ensign.

**Gap:**  
- No automated pipeline observed. Export is manual run of `export_lora.py`.  
- No configuration sharing (e.g., which rooms to export, hyperparameters).

## 4. fleet-simulator → holodeck-rust
**Connection:** **None observed**  
- `fleet-simulator` is Python-based agent simulation.  
- `holodeck-rust` is a multiplayer text game server.  
- No shared code, imports, or network connections found.  
- Both simulate "agents" but at different abstraction levels (fleet-simulator: abstract decision loops, holodeck: narrative NPCs).

**Gap:**  
- Could connect via:  
  1. `holodeck` as a "scenario provider" for `fleet-simulator` (generate terrain/NPCs).  
  2. `fleet-simulator` agents acting as clients to `holodeck` MUD.

## 5. fleet-simulator → plato-ensign
**Connection:** **Indirect via plato-torch**  
- `fleet-simulator` data flows into `DeadbandRoom`.  
- Room trains, produces checkpoint.  
- Checkpoint exported by `plato-ensign`.  
- No direct simulator→ensign link.

**Gap:**  
- Simulator-specific adapters not extracted separately (e.g., "fleet-tactics LoRA").

## 6. holodeck-rust → plato-ensign
**Connection:** **None observed**  
- `holodeck-rust` outputs JSON event logs.  
- `plato-ensign` expects PyTorch checkpoints.  
- No conversion tool from holodeck logs to training tiles/checkpoints.

**Gap:**  
- Holodeck sentiment-aware NPC interactions could train "empathy adapters" if logged and formatted for plato-torch.
