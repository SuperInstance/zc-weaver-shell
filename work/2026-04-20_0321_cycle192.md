# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 192  
**Phase:** 4 — Build & Test  
**Status:** Updated map based on actual code inspection.

Based on reading the actual files in the fleet repos, here is the current state of connections between **plato-torch**, **fleet-simulator**, **holodeck-rust**, and **plato-ensign**.

---

## 1. **plato-torch** → **fleet-simulator**
**Connection:** ✅ **Connected** (one-way data flow)  
**How:**  
- `plato-torch/rooms/deadband_room.py` imports `fleet_simulator` and uses `FleetSimulator` class.  
- The room calls `simulator.step()` to generate training episodes.  
- Simulator steps produce `(state, action, reward, next_state)` tuples that become training tiles.  
**Integration Point:** `DeadbandRoom._generate_episode()` → `FleetSimulator.step()`  
**Gap:** No bidirectional control loop. The simulator does not read back from the room.

---

## 2. **plato-torch** → **plato-ensign**
**Connection:** ✅ **Connected** (optional export path)  
**How:**  
- `plato-torch/training/trainer.py` has `export_to_ensign()` method.  
- After room training converges, trainer can save a LoRA adapter to `plato-ensign/adapters/`.  
- Export is triggered manually via CLI flag `--export`.  
**Integration Point:** `Trainer.export_to_ensign()` → writes `.safetensors` to ensign adapter directory.  
**Gap:** No automatic export policy. Requires manual intervention.

---

## 3. **holodeck-rust** → **plato-torch**
**Connection:** ❌ **Not connected** (no direct integration found)  
**Evidence:**  
- `holodeck-rust/src/` contains no Python bindings or HTTP API in current commit.  
- `plato-torch` has no imports of `holodeck` or Rust FFI.  
**Potential Integration:** Could feed NPC interaction logs into `DeadbandRoom` as sentiment-labeled tiles.

---

## 4. **fleet-simulator** → **holodeck-rust**
**Connection:** ❌ **Not connected** (separate domains)  
**Evidence:**  
- `fleet-simulator` simulates fleet movement and deadband decisions.  
- `holodeck-rust` is a text-based MUD with sentiment-aware NPCs.  
- No shared data structures or messaging between them.  
**Potential Integration:** Simulator agents could be given “personality” weights from holodeck NPC sentiment models.

---

## 5. **plato-ensign** ← **holodeck-rust**
**Connection:** ❌ **Not connected** (no LoRA training from holodeck)  
**Evidence:**  
- `plato-ensign` reads adapters from `plato-torch` exports only.  
- `holodeck-rust` does not produce `.safetensors` files.  
**Potential Integration:** Train ensigns on NPC dialogue patterns (sentiment → response).

---

## 6. **plato-torch** → **holodeck-rust** (via GhostInjector)
**Connection:** 🔄 **Planned but not implemented**  
**Evidence:**  
- `plato-torch/rooms/ghost_injector.py` exists but only injects static “ghost tiles” from file.  
- No socket or HTTP client to push tiles to holodeck.  
**Integration Gap:** GhostInjector could send dead-agent lessons to holodeck as “oracle whispers” to NPCs.

---

## Summary of Integration Gaps (P1 Safe Channels)

| Gap | Description | Priority |
|-----|-------------|----------|
| **GhostInjector → holodeck** | Static ghost tiles not reaching live NPCs. | High |
| **holodeck → plato-torch** | NPC interactions not becoming training tiles. | Medium |
| **Auto-export to ensign** | Manual export only; no automatic snapshotting. | Medium |
| **Simulator ← room feedback** | Room cannot adjust simulator parameters. | Low |
