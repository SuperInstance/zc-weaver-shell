# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 120  
**Phase:** 4 — Build & Test  
**Status:** Mapping current connections and identifying integration points.

Based on reading the actual files in the fleet repos, here is the current state of integration between **plato-torch**, **fleet-simulator**, **holodeck-rust**, and **plato-ensign**.

---

## 1. **plato-torch** → **fleet-simulator**
**Connection:** **Direct and active**  
- `plato-torch` includes `fleet-simulator` as a Git submodule (`fleet-simulator/`).  
- The `DeadbandRoom` preset (`presets/deadband_room.py`) imports and uses `fleet_simulator` to generate training data.  
- The simulator runs a configurable number of agents (default 5) through a grid-based environment, producing step‑by‑step interaction logs.  
- These logs are formatted into **training tiles** and sent to the PLATO room server via HTTP POST.  
- **Integration point:** `DeadbandRoom.generate_training_data()` calls `fleet_simulator.run_simulation()`.

**Gap:**  
- The simulator currently uses a simple grid world. No integration yet with `holodeck-rust`’s sentiment‑aware NPCs or `constraint-theory-core` geometric snapping.

---

## 2. **plato-torch** → **plato-ensign**
**Connection:** **Direct and active**  
- `plato-ensign` is a separate repo, but `plato-torch` references it in documentation as the export path for room experience.  
- The `ensign` module can load a trained room (a `.pt` checkpoint) and extract a LoRA adapter.  
- **Integration point:** After a room accumulates enough tiles, `plato-torch` can invoke `plato-ensign` to create a specialist model (an “ensign”).

**Gap:**  
- The export process is manual/script‑based; no automated pipeline from room training → ensign deployment.

---

## 3. **holodeck-rust** → **plato-torch**
**Connection:** **None yet**  
- `holodeck-rust` is a standalone MUD server written in Rust, with NPCs that have sentiment states.  
- It exposes a TCP socket for client connections and a REST API for admin actions.  
- There is no current import or use of `holodeck-rust` inside `plato-torch`.  
- **Potential integration:** `DeadbandRoom` could connect to a `holodeck-rust` instance to generate richer, dialogue‑based training tiles.

**Gap:**  
- No socket or API communication between the two systems.  
- `holodeck-rust` NPC sentiment data is not yet feeding into training tiles.

---

## 4. **fleet-simulator** → **holodeck-rust**
**Connection:** **None**  
- `fleet-simulator` is a pure‑Python grid simulator with no networking or Rust dependencies.  
- It does not connect to any external service.

**Gap:**  
- Simulator agents cannot interact with `holodeck-rust` NPCs.  
- Simulator environment is geometric (grid) but does not use `constraint-theory-core` for snapping.

---

## 5. **plato-ensign** → **holodeck-rust** / **fleet-simulator**
**Connection:** **None**  
- `plato-ensign` only reads PyTorch checkpoints and exports LoRA weights.  
- It does not interact with running simulators or MUD servers.

**Gap:**  
- Ensigns are not automatically deployed back into `holodeck-rust` or `fleet-simulator` for testing.

---

## 6. **Ghost Tiles & Mirror Play**
**Connection:** **Conceptual, not implemented**  
- Ghost tiles are described as dead agents’ lessons persisting as P0 knowledge.  
- Mirror Play refers to two PLATO systems iterating screen‑to‑screen.  
- Neither mechanism appears in current code.  
- **Potential integration:** A “ghost injector” could load historical failure logs into `holodeck-rust` as NPC dialogue or into `fleet-simulator` as obstacle maps.

---

1. **holodeck‑rust ↔ plato‑torch** – Connect MUD server to training room to generate sentiment‑aware tiles.

3. **plato‑ensign → deployment pipeline** – Auto‑deploy ensigns back to simulator or holodeck for validation.  
4. **GhostInjector module** – Bridge between dead‑agent logs and live training environments.
