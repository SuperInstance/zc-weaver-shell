## 1. plato-torch

**Location:** `plato-torch/`  
**Purpose:** Training room presets, self-training rooms, tile processing.

**Key Files:**
- `presets/deadband_room.py` — Deadband Protocol training room.
- `presets/__init__.py` — Exports `DeadbandRoom`.
- `tile_processor.py` — Processes incoming tiles (validation, embedding).
- `room_server.py` — Main server handling WebSocket connections.

**Connections:**
- **To plato-ensign:** `presets/deadband_room.py` imports `from plato_ensign import export_room`. Direct integration for LoRA export.
- **To fleet-simulator:** No direct imports found. Potential indirect link via shared tile format.
- **To holodeck-rust:** No direct imports. No Rust interop visible.

**Integration Points:**
- `DeadbandRoom` implements `step()` method that processes agent actions and returns rewards/observations.
- `tile_processor.py` expects JSON tiles with `agent_id`, `action`, `reward`, `metadata`.
- Room server listens on WebSocket (`/ws`) for agent connections.

---

## 2. fleet-simulator

**Location:** `fleet-simulator/`  
**Purpose:** Generate training data via multi-agent fleet simulations.

**Key Files:**
- `src/simulation.rs` — Core simulation engine.
- `src/agents/mod.rs` — Agent definitions (Oracle1, JetsonClaw1, Forgemaster).
- `src/tiles.rs` — Tile generation from simulation events.

**Connections:**
- **To plato-torch:** No Rust→Python bindings found. Tiles written to `output/tiles.jsonl`; could be consumed by plato-torch’s tile processor.
- **To plato-ensign:** No direct link.
- **To holodeck-rust:** No shared crate dependencies.

**Integration Points:**
- Simulator outputs JSONL tiles compatible with plato-torch’s expected format.
- Agents in simulation follow Deadband Protocol (P0/P1/P2) — logic mirrored in `src/agents/deadband.rs`.

## 3. holodeck-rust

**Location:** `holodeck-rust/`  
**Purpose:** MUD server with sentiment-aware NPCs, environment interactions.

**Key Files:**
- `src/server.rs` — TCP server for MUD connections.
- `src/npc/sentiment.rs` — NPC sentiment modeling.
- `src/world/mod.rs` — Room and object management.

**Connections:**
- **To plato-torch:** None. Separate runtime (Rust vs Python).
- **To fleet-simulator:** None.
- **To plato-ensign:** None.

**Integration Points:**
- Exposes TCP socket (`0.0.0.0:8080`). Could be connected via a bridge agent.
- NPCs generate dialogue/action events that could be captured as tiles.

## 4. plato-ensign

**Location:** `plato-ensign/`  
**Purpose:** Export room experience into LoRA adapters for specialist models.

**Key Files:**
- `export.py` — Main export function (`export_room`).
- `lora_builder.py` — Converts room interaction history to LoRA weights.
- `config.yaml` — Model and training configuration.

**Connections:**
- **To plato-torch:** Direct import in `deadband_room.py`. Called when room reaches export threshold.
- **To fleet-simulator:** None.
- **To holodeck-rust:** None.

**Integration Points:**
- `export_room(room_state, output_path)` — expects room state dict from plato-torch.
- Outputs `.safetensors` LoRA file and training log.

## 5. GhostInjector

**Location:** `ghost-injector/`  
**Purpose:** Inject ghost tiles (dead agents’ lessons) into training streams.

**Key Files:**

**Connections:**
- **To plato-torch:** Could be piped into `tile_processor.py` via a wrapper.
- **To holodeck-rust:** Not yet connected — potential task “Wire GhostInjector into holodeck”.

**Integration Points:**
- Ghost tiles have same JSON schema as live tiles, with `ghost: true` flag.
- Injector can be run as a sidecar process feeding plato-torch.

## Summary: What’s Connected vs. What’s Not

### Connected:
- **plato-torch ↔ plato-ensign** — Direct Python import. Functional export pipeline.
- **fleet-simulator → plato-torch** — Indirect via tile format compatibility (JSONL).
- **GhostInjector → plato-torch** — Schema-compatible; needs plumbing.

### Not Connected:
- **holodeck-rust** — Isolated MUD server. No links to other systems.
- **cudaclaw / flux-runtime** — Standalone runtimes, not wired into training loops.
- **fleet-simulator ↔ holodeck-rust** — Separate simulations (fleet vs. MUD).
- **plato-ensign ↔ holodeck-rust** — No mechanism to export MUD interactions as LoRA.
