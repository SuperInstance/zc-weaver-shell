## 1. plato-torch → fleet-simulator
**Connection:** ✅ **Connected via `fleet-simulator` as a data source.**  
- **plato-torch/rooms/deadband_room.py** imports `from fleet_simulator import FleetSimulator`.  
- **plato-torch/rooms/deadband_room.py** line 47: `self.simulator = FleetSimulator(config_path="fleet_simulator/config.yaml")`.  
- **plato-torch/rooms/deadband_room.py** line 89: Calls `self.simulator.generate_scenario()` to produce training scenarios.  
- **Data flow:** FleetSimulator generates synthetic fleet interaction data → DeadbandRoom ingests it as training tiles.

**Integration Gap:**  
- No evidence of plato-torch pushing results back to fleet-simulator. The connection is one‑way (simulator → room).  
- fleet-simulator’s config.yaml is referenced but not verified in the plato-torch repo.

## 2. plato-torch → plato-ensign
**Connection:** ✅ **Connected via explicit export pipeline.**  
- **plato-torch/export/ensign_exporter.py** exists.  
- **plato-torch/export/ensign_exporter.py** line 22: `from plato_ensign import LoRAAdapter, export_room_experience`.

- **plato-torch/config/ensign_config.yaml** defines LoRA hyperparameters (rank=8, alpha=16).

**Integration Gap:**

- No observed import of plato-ensign into plato-torch’s training loop; export is a post‑processing step.

## 4. fleet-simulator → holodeck-rust
**Connection:** ❌ **No direct connection found.**  
- fleet-simulator is Python‑based, generates JSON/YAML scenarios.  
- holodeck-rust is Rust, uses its own protocol and state machine.  
- No shared data formats or imports between them.

**Integration Gap:**  
- fleet-simulator could supply NPC behavior profiles or scenario definitions to holodeck-rust, but no adapter exists.  
- Sentiment‑aware NPCs in holodeck-rust might benefit from simulated fleet interaction data.

## 5. holodeck-rust → plato-ensign
**Connection:** ❌ **No direct connection found.**  
- plato-ensign is a Python library for LoRA adapter export.  
- holodeck-rust has no Python interop in its source (Cargo.toml shows no pyo3 or cffi).  
- No evidence that holodeck-rust can load or use ensign adapters.

**Integration Gap:**  
- holodeck-rust NPCs could be fine‑tuned via ensigns, but would require a Rust‑compatible inference runtime (e.g., candle, tract) and integration glue.

## 6. fleet-simulator → plato-ensign
**Connection:** ❌ **No direct connection found.**  
- fleet-simulator does not import plato-ensign.  
- plato-ensign does not import fleet-simulator.  
- Both are independent data producers (simulator → scenarios, ensign → adapters).

**Integration Gap:**  
- Could connect if ensigns were trained on simulated data, but that would require plato-torch as an intermediary.

## 7. Ghost Tiles & Mirror Play
**Connection:** 🔄 **Conceptual, not implemented.**  
- **FLEET-CONTEXT.md** mentions ghost tiles and mirror play as architectural concepts.  
- No repo currently implements ghost‑tile persistence or screen‑to‑screen mirroring.  
- **plato-torch/rooms/deadband_room.py** has a `ghost_injector` placeholder (line 63) but no implementation.

**Integration Gap:**  
- Ghost tiles would require a shared memory store (e.g., SQLite, Redis) accessible by plato-torch, holodeck-rust, and fleet-simulator.  
- Mirror Play implies two PLATO instances; only one (plato-torch) exists in the fleet.
