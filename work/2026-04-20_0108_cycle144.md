# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 144  
**Phase:** 4 — Build & Test  
**Status:** Mapping current connections and identifying integration points.

Based on reading the actual files in the fleet repos, here is the current state of integration between **plato-torch**, **fleet-simulator**, **holodeck-rust**, and **plato-ensign**.

## 1. Current Verified Connections

### plato-torch ↔ fleet-simulator
- **Connection:** Strong, direct integration via `fleet-simulator` as a data source.
- **Evidence:**  
  - `plato-torch/src/rooms/deadband_room.py` imports `from fleet_simulator import FleetSimulator`.  
  - `DeadbandRoom` initializes a `FleetSimulator` instance (`self.simulator`) and uses it to generate training episodes.  
  - The room calls `self.simulator.run_episode()` to produce `(state, action, reward, next_state)` tuples which become training tiles.
- **Data Flow:** Fleet simulation → Episode data → Training tiles in Plato room.

### plato-torch ↔ plato-ensign
- **Connection:** Direct export pipeline.
- **Evidence:**  
  - `plato-ensign/src/exporter.py` contains `export_room_to_lora(room_path, output_dir)`.  
  - `plato-torch` rooms (including `DeadbandRoom`) accumulate interaction tiles in `.room/` directories.  
  - The ensign exporter reads these tile files and trains a LoRA adapter on the accumulated experience.
- **Data Flow:** Room tiles → Ensign exporter → LoRA adapter file.

### fleet-simulator (standalone)
- **Self-contained:** The simulator generates its own episodic training data using agent policies and environment models. It does not currently read from or write to `holodeck-rust` or external MUD states.

### holodeck-rust (standalone)
- **Self-contained:** A MUD server with sentiment-aware NPCs, using an ECS architecture. No imports of or references to `plato-torch`, `fleet-simulator`, or `plato-ensign` in the source code.

## 2. Identified Integration Gaps (P1 Safe Channels)

### Gap 1: holodeck-rust → plato-torch
- **Current State:** No connection. Holodeck interactions are not captured as training tiles.
- **Opportunity:** Instrument `holodeck-rust` to log player/NPC interactions (state, action, sentiment outcome) in a format compatible with `plato-torch` tile structure.
- **Potential Integration Point:** Add a `TileLogger` system to holodeck that writes to a `.room/` directory, or send events via a local socket to a `plato-torch` ingestion service.

### Gap 2: fleet-simulator → holodeck-rust
- **Current State:** No connection. The simulator does not use the MUD as an environment.
- **Opportunity:** Use `holodeck-rust` as a rich, interactive environment for fleet agents, replacing or augmenting the current simulated grid-world.
- **Potential Integration Point:** Implement a `HolodeckEnvironment` adapter in `fleet-simulator` that connects to the MUD server via TCP and translates agent actions into MUD commands.

### Gap 3: plato-ensign → holodeck-rust
- **Current State:** No connection. Trained ensigns (LoRA adapters) are not loaded into the MUD’s NPCs.
- **Opportunity:** Load exported ensigns into holodeck NPCs to give them specialized behaviors learned from room interactions.
- **Potential Integration Point:** Add a `LoraNPCSystem` to holodeck that loads a `.safetensors` adapter and uses it to modify NPC response generation.

### Gap 4: Cross-room tile sharing
- **Current State:** Rooms accumulate tiles independently. No mechanism for tiles from one room (e.g., a holodeck-logged room) to be used by another room (e.g., DeadbandRoom).
- **Opportunity:** A central tile registry or relay (`plato-relay`) could allow rooms to publish and subscribe to tile streams.
- **Potential Integration Point:** Implement `plato-relay` as a simple pub/sub server, and add `TileRelayClient` to both `plato-torch` rooms and `holodeck-rust`.

## 4. Constraints & Notes
- **plato-torch** uses PyTorch and Python; **holodeck-rust** is Rust. Integration will require an IPC mechanism (sockets, shared files, or REST).
- **plato-ensign** expects a specific tile format (JSON with `state`, `action`, `reward`, `next_state`). Any new tile source must conform.
- The **Deadband Protocol** (P0, P1, P2) should guide integration: first ensure we don’t break existing pipelines (P0), then establish safe data channels (P1), then optimize throughput (P2).
