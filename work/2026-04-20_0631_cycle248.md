## 1. plato-torch → fleet-simulator
**Connection:** **Strong** (direct import, shared data flow)  
- **Evidence:** `plato-torch/rooms/deadband_room.py` imports `from fleet_simulator.simulator import FleetSimulator`.  
- **Data Flow:** The `DeadbandRoom` class instantiates a `FleetSimulator` object (`self.simulator`). It calls `self.simulator.step()` to generate training data (agent trajectories). This data is formatted into interaction tiles.  
- **Integration Point:** `DeadbandRoom` uses the simulator as a **data source**. The simulator's step function returns a list of agent states, which the room converts to tiles.  
- **Gap:** The simulator does not appear to import anything from plato-torch. The dependency is one-way.

---

## 2. plato-torch → plato-ensign
**Connection:** **Strong** (direct import, core functionality)  
- **Evidence:** `plato-torch/export/ensign_exporter.py` exists. `plato-torch/rooms/base_room.py` has a method `export_to_ensign()` that calls the exporter.  
- **Data Flow:** After training, a room can export its accumulated experience (tiles) into a LoRA adapter via `plato-ensign`. The exporter uses `plato_ensign` library functions to create the adapter.  
- **Integration Point:** The `ensign_exporter.py` module is the bridge. It reads the room's tile history, formats it for training, and invokes the ensign training process.  
- **Gap:** `plato-ensign` is a separate library; its setup and model configuration are external dependencies. The integration assumes `plato_ensign` is installed and configured.
