## 1. Verified Current Connections (What's Connected)

Based on reading the actual fleet repos:

### **plato-torch → plato-ensign** (Strong Connection)
- **plato-torch** (`rooms/`) generates training tiles from agent interactions.
- **plato-ensign** (`ensign/`) exports room experience into LoRA adapters.
- **Connection:** `plato-ensign` imports `plato-torch` as a dependency (Cargo.toml). It reads tile data from `plato-torch` rooms and converts them to training datasets.
- **Evidence:** `plato-ensign/src/lib.rs` has `use plato_torch::rooms::TileStore;`.

### **plato-torch → fleet-simulator** (One-Way Data Flow)
- **fleet-simulator** (`simulator/`) generates synthetic training data (agent interactions, sensor logs).
- **plato-torch** can ingest this data as "simulated tiles" for training rooms.
- **Connection:** `fleet-simulator` outputs JSON logs that match `plato-torch` tile schema. `plato-torch` has a `simulation_loader` module (unfinished).
- **Evidence:** `fleet-simulator/README.md` states: "Output compatible with plato-torch tile format."

### **holodeck-rust → plato-torch** (Weak/Planned Connection)
- **holodeck-rust** is a MUD server with sentiment-aware NPCs.
- **Potential:** NPC interactions could become training tiles for `plato-torch`.
- **Current State:** No direct import. `holodeck-rust` has its own event system; no tile exporter yet.
- **Evidence:** `holodeck-rust/src/main.rs` shows custom `Event` enum, not `Tile`.

### **plato-ensign → (External)** (Export Path)
- **plato-ensign** exports LoRA adapters (`.safetensors`) to Hugging Face or local cache.
- **Connection:** Uses `safetensors` and `huggingface-hub` crates. No direct fleet integration beyond that.

## 2. Integration Gaps (What's Not Connected)

### **Gap 1: holodeck-rust → plato-torch (Tile Pipeline)**
- **Problem:** NPC interactions in holodeck are not captured as training tiles.
- **Impact:** Lost training data source; sentiment-aware NPC behaviors not used for ensign training.
- **Actionable:** Build `holodeck-tile-exporter` that converts `Event` streams to `Tile` format.

### **Gap 2: fleet-simulator → holodeck-rust (NPC Behavior)**
- **Problem:** Simulated agent interactions from `fleet-simulator` could drive NPC behavior in `holodeck-rust`.
- **Impact:** NPCs lack dynamic, data-driven behavior; simulation remains isolated.
- **Actionable:** Create adapter that translates simulator logs to holodeck `Command` messages.

### **Gap 3: plato-ensign → holodeck-rust (LoRA-Infused NPCs)**
- **Problem:** Trained LoRAs from `plato-ensign` are not loaded into `holodeck-rust` NPCs.
- **Impact:** NPCs cannot improve or specialize via fleet training.
- **Actionable:** Add `EnsightLoader` to holodeck that loads `.safetensors` and applies them to NPC LLM calls.

### **Gap 4: Deadband Protocol Integration**
- **Problem:** The Deadband Protocol (P0/P1/P2) is implemented in `plato-torch` (`DeadbandRoom`), but not in `holodeck-rust` or `fleet-simulator`.
- **Impact:** Holodeck NPCs and simulated agents do not follow fleet doctrine.
- **Actionable:** Port `DeadbandLogic` to `holodeck-rust` (as NPC decision module) and `fleet-simulator` (as agent policy).

### **Gap 5: Ghost Tiles Injection**
- **Problem:** Ghost tiles (dead agents' lessons) are a `plato-torch` concept, but no pipeline exists to inject them into `holodeck-rust` or `fleet-simulator`.
- **Impact:** P0 knowledge (negative space) not reused across systems.
- **Actionable:** Build `GhostInjector` service that reads ghost tiles and broadcasts them as warnings/constraints to other services.
