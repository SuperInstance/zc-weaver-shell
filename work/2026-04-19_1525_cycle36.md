## 1. Verified Connections (What's Connected)

Based on reading the actual fleet repos:

### **plato-torch ↔ fleet-simulator**
- **Connection:** Direct integration via `fleet-simulator` as a data source.
- **Evidence:** `plato-torch/rooms/deadband_room.py` imports from `fleet_simulator` and uses `FleetSimulator` class to generate training episodes.
- **Purpose:** `fleet-simulator` provides synthetic fleet interaction data (agent states, actions, rewards) which `plato-torch` rooms consume to create training tiles.
- **Data Flow:** Simulated episodes → training tiles → room memory.

### **plato-torch ↔ plato-ensign**
- **Connection:** Built-in export pipeline.
- **Evidence:** `plato-torch/training/ensign_exporter.py` exists. Rooms can be configured to export their trained adapter (LoRA) via `export_adapter()` method.
- **Purpose:** Capture specialized room experience (tiles) into a portable LoRA adapter for deployment.
- **Data Flow:** Room training loop → accumulated gradients → LoRA weights → `.safetensors` file.

### **holodeck-rust ↔ (Indirect)**
- **Current State:** No direct code-level integration found with the other three systems.
- **Potential Link:** `holodeck-rust` is a sentiment-aware MUD server. Its NPC interactions could become a rich source of narrative training tiles for a `plato-torch` room, but this is not yet implemented.
- **Note:** It exists as a standalone service (`holodeck-server`).

## 2. Integration Gaps (What's Not Connected)

### **Gap 1: holodeck-rust → plato-torch**
- **Description:** No pipeline exists to feed holodeck interaction logs (player-NPC dialogues, sentiment events) into a `plato-torch` training room.
- **Impact:** Narrative and social reasoning data from the MUD is isolated and not contributing to fleet training tiles.
- **Actionable Integration:** Build a `HolodeckLogIngester` module in `plato-torch` that subscribes to holodeck event streams (via its API or log file) and formats events into training tiles.

### **Gap 2: fleet-simulator → holodeck-rust**
- **Description:** The simulated fleet agents do not interact with the holodeck MUD world.
- **Impact:** Simulation lacks a rich, persistent social environment; holodeck lacks automated agent participants.
- **Actionable Integration:** Extend `fleet-simulator` agent action space to include holodeck API commands (move, say, emote). Connect simulator to holodeck's WebSocket or TCP interface.

### **Gap 3: plato-ensign → Runtime Deployment**
- **Description:** Exported LoRA adapters (`plato-ensign` output) are not automatically loaded into a running service (e.g., `cudaclaw` runtime, `flux-runtime` VM).
- **Impact:** Trained specialists exist as files, but are not actively used by the fleet without manual intervention.
- **Actionable Integration:** Create an `ensign-loader` service that watches an export directory, validates new `.safetensors` files, and hot-swaps them into a designated runtime (e.g., via `cudaclaw`'s model management API).

### **Gap 4: Ghost Tile Injection**
- **Description:** The concept of "ghost tiles" (P0 knowledge from dead agents) is documented but no system exists to inject them into active training rooms.
- **Impact:** Historical failure lessons are not actively reinforcing the P0 "negative space" mapping for new agents.
- **Actionable Integration:** Implement a `GhostInjector` class in `plato-torch` that periodically samples from a curated corpus of failure logs and formats them as negative example tiles into room training buffers.

## 3. Priority for Phase 4 Work (P0 → P1 → P2)

**P0 (Avoid Rocks):** Do not break existing connections (`plato-torch` ↔ `fleet-simulator`, `plato-torch` ↔ `plato-ensign`).  
**P1 (Find Safe Channels):**  
1. Implement **GhostInjector** (Gap 4) — leverages existing tile infrastructure, low risk.  
2. Design **HolodeckLogIngester** (Gap 1) — read-only consumption of logs, safe.  
**P2 (Optimize):**  
3. Build **ensign-loader** (Gap 3) — requires runtime integration, higher complexity.  
4. Connect **fleet-simulator to holodeck** (Gap 2) — bidirectional integration, highest complexity.
