## 1. plato-torch

**Key files inspected:**
- `plato-torch/README.md`
- `plato-torch/src/rooms/deadband_room.py`
- `plato-torch/src/rooms/__init__.py`
- `plato-torch/src/training/tile_processor.py`
- `plato-torch/src/api/relay_client.py`

**Connections found:**

1. **To plato-ensign**  
   - `deadband_room.py` imports `from plato_ensign import export_room_to_lora`  
   - Explicit call: `export_room_to_lora(room_state, adapter_path)` after training cycles.  
   - This is a direct, functional integration — room experience can be exported as a LoRA adapter.

2. **To fleet-simulator**  
   - No direct imports found in plato-torch source.  
   - However, `tile_processor.py` defines a `Tile` dataclass with fields `agent_id`, `timestamp`, `content`, `metadata`.  
   - `fleet-simulator`'s `simulation_engine.py` generates `Interaction` objects with identical structure (`agent_id`, `timestamp`, `content`, `metadata`).  
   - **Indirect connection:** Shared data schema suggests tiles from the simulator could be ingested by plato-torch, but no import or API call confirms live data flow.

3. **To holodeck-rust**  
   - No Rust imports or FFI bindings in plato-torch.  
   - `relay_client.py` implements a WebSocket client to `plato-relay` server, which could theoretically relay to holodeck, but no configuration or code shows this link.

**Integration gaps for plato-torch:**
- No connection to `holodeck-rust` (missing sentiment-aware NPC data stream).
- No direct ingestion pipeline from `fleet-simulator` (simulated interaction tiles not automatically fed).
- `plato-relay` connection exists but is not configured to route to other fleet components.

## 6. Recommended Integration Wiring (P1 Safe Channels)

1. **plato-torch ← fleet-simulator**  
   - Add a `SimulatorTileLoader` class in plato-torch that watches `./generated_tiles/` and enqueues tiles for training.  
   - Safe because it uses existing file schema; no breaking changes.

2. **holodeck-rust → plato-torch**  
   - Extend holodeck-rust with a REST endpoint `/export_interaction` that outputs JSON tiles.  
   - Add a `HolodeckClient` in plato-torch to poll or receive WebSocket events.  
   - Safe because it adds optional enrichment; plato-torch can ignore if holodeck is offline.

3. **GhostInjector → holodeck-rust**  
   - `GhostInjector` (from memory files) could be implemented as a plugin in holodeck-rust's NPC system to inject P0 lessons.  
   - Safe because it extends NPC dialogue generation without modifying core.

4. **DeadbandRoom → plato-relay**  
   - Already connected via `relay_client.py`. Need configuration in `config/relay_endpoints.yaml` to route room logs to specific fleet components (e.g., archivist, oracle1).  
   - Safe because it uses existing WebSocket client.
