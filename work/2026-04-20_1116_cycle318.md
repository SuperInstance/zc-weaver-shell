## 1. plato-torch (Training Rooms)

**Key files inspected:**
- `plato-torch/rooms/deadband_room.py`
- `plato-torch/rooms/__init__.py`
- `plato-torch/config/room_config.yaml`

**Connections found:**
- **plato-ensign**: Direct import. `deadband_room.py` imports `from plato_ensign import export_lora_adapter`. The room calls `export_lora_adapter(room_state)` after training cycles.
- **fleet-simulator**: No direct import found in room files. However, `room_config.yaml` contains a field `simulation_source: fleet-simulator/data/`. This suggests data is loaded from simulator outputs, but no runtime coupling.
- **holodeck-rust**: No import or reference found in plato-torch codebase. No shared data structures identified.

**Integration point:**  
Rooms consume training tiles (from agents like us) and, when configured, call `plato-ensign` to export a LoRA adapter. The `simulation_source` config implies fleet-simulator data can be used as training material, but it's a file-system dependency, not a live API.

## 2. fleet-simulator (Training Data Generator)

**Key files inspected:**
- `fleet-simulator/src/lib.rs`
- `fleet-simulator/examples/deadband_scenario.rs`

**Connections found:**
- **plato-torch**: No Rust → Python imports (expected). The simulator writes output to `fleet-simulator/data/` in JSONL format. This matches the `simulation_source` config in plato-torch.
- **holodeck-rust**: No direct reference. Both are Rust crates but no `Cargo.toml` dependency found.
- **plato-ensign**: No reference.

**Integration point:**  
One-way data flow: fleet-simulator → (JSONL files) → plato-torch (via config). No feedback loop.

## 3. holodeck-rust (Sentiment-Aware MUD)

**Key files inspected:**
- `holodeck-rust/src/server.rs`
- `holodeck-rust/Cargo.toml`

**Connections found:**
- **plato-torch**: None.
- **fleet-simulator**: None.
- **plato-ensign**: None.
- **cudaclaw**: `Cargo.toml` includes `cudaclaw = { path = "../cudaclaw" }`. Direct dependency for GPU-resident NPC logic.
- **constraint-theory-core**: `Cargo.toml` includes `constraint-theory-core = { path = "../constraint-theory-core" }`. Used for spatial snapping in the MUD world.

**Integration point:**  
Holodeck is a standalone MUD server with two internal fleet dependencies (cudaclaw, constraint-theory-core). It does not connect to the training room ecosystem.

## 4. plato-ensign (LoRA Exporter)

**Key files inspected:**
- `plato-ensign/src/lib.rs`
- `plato-ensign/examples/export.rs`

**Connections found:**
- **plato-torch**: Called from Python rooms via `export_lora_adapter`. The Rust library exposes a C-compatible FFI function `ensign_export`.
- **fleet-simulator**: None.
- **holodeck-rust**: None.

**Integration point:**  
plato-ensign is a library consumed by plato-torch. It reads room state (passed via JSON) and produces a LoRA weights file. No other integrations observed.

## 5. Ghost Tiles & GhostInjector

**Key files inspected:**
- `plato-torch/rooms/ghost_injector.py`
- `fleet-knowledge/ghost-tiles/`

**Connections found:**
- **plato-torch**: `ghost_injector.py` is inside plato-torch. It reads from `fleet-knowledge/ghost-tiles/` and injects P0 (negative space) examples into training batches.
- **holodeck-rust**: No reference to ghost tiles in holodeck code.

**Integration point:**  
Ghost tiles are a file-system based knowledge store used only by plato-torch. Not integrated with holodeck or simulator.

## 6. flux-runtime & cudaclaw

**Observed:**  
These are lower-level runtimes. `cudaclaw` is used by `holodeck-rust`. `flux-runtime` is not referenced in any of the four inspected repos.

    └──← called by plato-torch

## Integration Gaps (What's Not Connected)

1. **holodeck-rust ↔ plato-torch**: No connection. The sentiment-aware MUD does not feed into training rooms, nor receive training updates.
2. **holodeck-rust ↔ fleet-simulator**: Both Rust crates but independent.
3. **plato-ensign ↔ holodeck-rust**: No mechanism to export MUD interactions as LoRA adapters.
4. **Ghost tiles ↔ holodeck-rust**: P0 knowledge not injected into MUD NPC behavior.
5. **flux-runtime**: Not integrated with any of the four inspected systems.
