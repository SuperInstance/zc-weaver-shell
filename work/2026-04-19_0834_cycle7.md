## 1. Current Connections (What IS Connected)

### **plato-torch ↔ fleet-simulator**
- **Connection:** Data pipeline for training tile generation  
- **Mechanism:** `fleet-simulator` generates synthetic agent interaction logs → formatted as training tiles → ingested by `plato-torch` rooms  
- **Evidence:**  
  - `fleet-simulator/src/output.rs` → `export_training_tiles()` writes `.tile.json` files  
  - `plato-torch/rooms/deadband_room.py` → `ingest_tiles()` reads tile directory  
- **Status:** **Active** — simulator output directory mounted to plato-torch container in docker-compose.dev.yml  

### **plato-torch ↔ plato-ensign**
- **Connection:** Room experience export to LoRA adapters  
- **Mechanism:** Trained room checkpoints → `plato-ensign` extracts LoRA weights → exports as `.safetensors`  
- **Evidence:**  
  - `plato-ensign/src/export.py` → `export_room_lora(room_checkpoint_path)`  
  - `plato-torch/scripts/train_room.py` → saves checkpoints to shared volume `/room_exports`  
- **Status:** **Active** — one-way export pipeline (plato-torch → plato-ensign)  

### **holodeck-rust ↔ fleet-simulator** (Indirect)
- **Connection:** Shared NPC behavior patterns  
- **Mechanism:** `fleet-simulator` uses sentiment-aware NPC logic from `holodeck-rust/src/npc/sentiment.rs` as library dependency  
- **Evidence:**  
  - `fleet-simulator/Cargo.toml` → `holodeck-rust = { path = "../holodeck-rust" }`  
  - Simulator NPCs call `SentimentEngine::update()` from holodeck module  
- **Status:** **Active** — compile-time dependency

## 2. Missing Connections (What's NOT Connected)

### **holodeck-rust ↔ plato-torch**
- **Gap:** No direct integration between MUD server and training rooms  
- **Potential:** Holodeck player interactions could become real training tiles (not just synthetic)  
- **Evidence:**  
  - `holodeck-rust` outputs JSON event streams to stdout  
  - No plato-torch ingestion endpoint for live holodeck events  
- **Status:** **Disconnected** — parallel systems  

### **plato-ensign ↔ fleet-simulator**
- **Gap:** Simulator doesn't use exported ensign LoRAs  
- **Potential:** Fleet agents could load specialized ensign adapters during simulation  
- **Evidence:**  
  - `fleet-simulator/src/agent/runtime.rs` uses base DeepSeek model only  
  - No adapter loading mechanism in simulator codebase  
- **Status:** **Disconnected** — one-way export only  

### **holodeck-rust ↔ plato-ensign**
- **Gap:** Holodeck NPCs don't benefit from trained ensigns  
- **Potential:** NPC dialogue models could use room-trained LoRAs  
- **Evidence:**  
  - `holodeck-rust/src/npc/dialogue.rs` uses hard-coded response templates  
  - No model inference integration in current code  
- **Status:** **Disconnected** — static NPC system
