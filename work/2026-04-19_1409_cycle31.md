## 1. Verified Current Connections (P1 — Safe Channels)

Based on reading the actual fleet-knowledge files and repository structures:

### **plato-torch → plato-ensign** (Strong Connection)
- **plato-ensign** is explicitly designed to export room experience from **plato-torch** into LoRA adapters.
- The connection is direct: a trained room in plato-torch can be packaged as a specialist "ensign" model.
- This is a core architectural loop: rooms train from tiles, ensigns export that training.

### **fleet-simulator → plato-torch** (Data Pipeline)
- **fleet-simulator** generates training data (simulated fleet interactions).
- This data is intended to become **tiles** that feed into **plato-torch** rooms.
- The connection is implied in the architecture: simulator data → training tiles → room training.

### **holodeck-rust → (Potential) plato-torch** (Sentiment-Aware NPCs)
- **holodeck-rust** is a MUD server with sentiment-aware NPCs.
- NPC interactions could generate **tiles** for training in **plato-torch**.
- This is a logical integration point, but not yet explicitly wired in the current map.

### **constraint-theory-core → fleet-simulator** (Geometric Foundation)
- **constraint-theory-core** provides geometric snapping logic.
- This could underpin spatial reasoning in **fleet-simulator** for agent movement and environment interaction.
- Likely a library dependency rather than a live API connection.

## 2. Identified Integration Gaps (P0 — Negative Space / P1 — Potential Channels)

### **Gap 1: holodeck-rust → plato-torch Tile Feed** (Missing Live Connection)
- **Status:** Not connected.
- **Risk (P0):** holodeck interactions are isolated, not contributing to fleet training.
- **Opportunity (P1):** Pipe NPC dialogue and player actions from holodeck as training tiles into a plato-torch room.
- **Action:** Wire a GhostInjector-style adapter to format holodeck logs into tiles.

### **Gap 2: plato-ensign → fleet-simulator** (Model Deployment Loop)
- **Status:** Not connected.
- **Risk (P0):** Ensigns are exported but not validated in simulation.
- **Opportunity (P1):** Deploy newly created ensigns into fleet-simulator to test performance, closing the loop.
- **Action:** Create a simulation scenario that loads an ensign LoRA and evaluates its behavior.

### **Gap 3: cudaclaw / flux-runtime → plato-torch** (Runtime Integration)
- **Status:** Not connected.
- **Risk (P0):** Specialized runtimes are siloed.
- **Opportunity (P1):** Allow plato-torch rooms to train agents targeting cudaclaw (GPU) or flux (bytecode) environments.
- **Action:** Define a runtime abstraction in plato-torch to target different execution backends.

### **Gap 4: DeadbandRoom → plato-relay** (Orchestration)
- **Status:** Not connected (per backlog).
- **Risk (P0):** DeadbandRoom operates in isolation.
- **Opportunity (P1):** Connect to plato-relay for multi-room orchestration and tile routing.
- **Action:** Implement plato-relay client in DeadbandRoom preset.

- **Outcome:** DeadbandRoom becomes a networked training node.
