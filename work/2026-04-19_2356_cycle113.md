# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 113  
**Phase:** 4 — Build & Test  
**Status:** Mapping current connections and identifying integration points.

Based on reading the actual files in the fleet repos, here is the current state of integration between the four core technologies.

## 1. plato-torch
**Purpose:** Training room server with 25 preset rooms. Self-training rooms via `DeadbandRoom`.  
**Key Files:** `rooms/deadband_room.py`, `rooms/__init__.py`, `server.py`, `tools/ghost_injector.py`

**Current Connections:**
- **Connected to plato-ensign:** Yes. `plato-torch/rooms/base_room.py` has `export_to_ensign()` method that calls `plato_ensign.export.export_room()`. This creates a LoRA adapter from room experience.
- **Connected to fleet-simulator:** No direct code import. However, `fleet-simulator` can generate training data that could be fed into a plato-torch room as tiles.
- **Connected to holodeck-rust:** No direct connection. `tools/ghost_injector.py` is designed to inject "ghost tiles" (P0 knowledge from dead agents) into any training stream, but no current integration with holodeck's MUD server.

**Integration Points:**
- `DeadbandRoom` is a preset that implements the Deadband Protocol (P0, P1, P2). It accepts tiles, trains, and can export to ensign.
- `GhostInjector` tool exists but is not wired into any live pipeline.

## 2. plato-ensign
**Purpose:** Exports room experience from plato-torch into LoRA adapters.  
**Key Files:** `export.py`, `lora_train.py`

**Current Connections:**
- **Connected to plato-torch:** Yes, as above. Called from `base_room.py`.
- **Connected to fleet-simulator:** No.
- **Connected to holodeck-rust:** No.

**Integration Points:**
- Single function: `export_room(room, output_path)`. Requires a trained room object from plato-torch.

## 3. holodeck-rust
**Purpose:** MUD server with sentiment-aware NPCs. Generates interaction data.  
**Key Files:** `src/main.rs`, `src/npc.rs`, `src/world.rs`

**Current Connections:**
- **Connected to plato-torch:** No.
- **Connected to plato-ensign:** No.
- **Connected to fleet-simulator:** No.

**Integration Points:**
- Generates NPC dialogue and sentiment logs. This could be a source of training tiles for plato-torch rooms.
- No current mechanism to send these logs to plato-torch or to trigger ensign exports.

## 4. fleet-simulator
**Purpose:** Fleet simulation for generating training data (agent interactions).  
**Key Files:** `simulate.py`, `agents/`, `scenarios/`

**Current Connections:**
- **Connected to plato-torch:** No direct import. Could output JSON that matches tile format.
- **Connected to plato-ensign:** No.
- **Connected to holodeck-rust:** No.

**Integration Points:**
- Simulates multi-agent scenarios. Output could be formatted as tiles and submitted to a plato-torch room.
- No current pipeline to do this.

## Summary: What's Connected vs. What's Not

### Connected:
- **plato-torch → plato-ensign** (room export to LoRA)

### Not Connected (Major Gaps):
1. **holodeck-rust → plato-torch**: NPC interaction data not feeding into training rooms.
2. **fleet-simulator → plato-torch**: Simulation data not used as tile input.
3. **GhostInjector → any consumer**: Tool exists but not integrated.
4. **holodeck-rust / fleet-simulator → plato-ensign**: No direct export path.

## Recommended Integration Tasks (P1 Safe Channels):
1. **Wire GhostInjector into holodeck-rust**: Modify holodeck to call GhostInjector for injecting P0 knowledge into NPC dialogue generation.
2. **Create a tile-bridge service**: A small service that formats holodeck logs and fleet-simulator outputs into plato-torch tile JSON and POSTs to a room.
3. **Extend DeadbandRoom API**: Add webhook endpoints to accept tiles from external sources (holodeck, simulator).
4. **Add ensign-trigger to holodeck**: After X interactions, automatically export room state to an ensign.

These integrations would create a closed loop: holodeck/simulator generate data → plato-torch rooms train on it → plato-ensign exports specialists → ensigns could potentially be loaded back into holodeck NPCs or simulator agents.
