## 3. **holodeck-rust** → **plato-torch** — **PARTIALLY CONNECTED**
- **Connection Type:** Shared protocol (WebSocket/MUD)
- **Evidence:** `holodeck-rust/src/protocol/mod.rs` defines `Tile` struct matching `plato-torch/src/types/tile.py`
- **Data Flow:** NPC interactions → sentiment-tagged tiles → potential room input
- **Integration Point:** Manual tile ingestion (no automatic pipeline)
- **Status:** Protocol-compatible but not wired

## 4. **fleet-simulator** → **holodeck-rust** — **NOT CONNECTED**
- **Gap:** Different languages (Rust vs Rust, but separate crates)
- **Evidence:** No `Cargo.toml` dependencies between them
- **Shared Concept:** Both use entity-component systems
- **Potential Integration:** Simulated agents could populate holodeck as NPCs

---

## 5. **plato-ensign** → **holodeck-rust** — **NOT CONNECTED**
- **Gap:** LoRA adapters not consumed by Rust MUD
- **Evidence:** `holodeck-rust` has no ONNX/TorchScript loading capability
- **Potential Integration:** Ensigns could parameterize NPC behavior models
- **Current State:** Export-only, no feedback loop

## 6. **Ghost Tiles Integration** — **PARTIAL**
- **Connection:** `plato-torch/src/utils/ghost_injector.py` reads from `memory/` directory
- **Evidence:** GhostInjector loads `.md` files from dead agents
- **Integration Point:** Injects P0 (negative space) knowledge into training rooms
- **Gap:** Not connected to holodeck-rust (NPCs don't learn from ghosts)
- **Status:** One-way injection into plato-torch only
