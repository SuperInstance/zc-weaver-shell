## 1. Repository Structure & Dependencies

Based on file inspection of the fleet repositories:

### **plato-torch**
- **Purpose**: Training room presets (25 rooms), self-training rooms.
- **Key files**: `DeadbandRoom.py`, `GhostInjector.py`, `plato_relay.py`.
- **Dependencies observed**:
  - Imports `torch`, `transformers`, `peft` (for LoRA).
  - Contains `plato_relay.py` which defines `PlatoRelay` class for sending tiles to a room server.
  - `GhostInjector.py` imports `json` and `random` — no observed imports of `holodeck-rust` or `fleet-simulator`.
  - `DeadbandRoom.py` is a preset room class; uses `torch` and `transformers` directly.

### **fleet-simulator**
- **Purpose**: Fleet simulation for training data.
- **Key files**: `simulator.py`, `agent.py`, `environment.py`.
- **Dependencies observed**:
  - Imports `numpy`, `pygame` (for visualization), `torch`.
  - No observed imports of `plato-torch`, `holodeck-rust`, or `plato-ensign`.
  - Contains `Agent` and `Environment` classes; outputs simulation logs (JSON).

### **holodeck-rust**
- **Purpose**: MUD server with sentiment-aware NPCs.
- **Key files**: `src/main.rs`, `src/npc.rs`, `src/world.rs`.
- **Dependencies observed**:
  - Rust crate: `serde`, `tokio`, `warp` (web server).
  - No observed imports of `plato-torch` or `fleet-simulator`.
  - Exposes HTTP endpoints (`/npc`, `/world`) for interaction.

### **plato-ensign**
- **Purpose**: Exports room experience into LoRA adapters.
- **Key files**: `ensign.py`, `lora_exporter.py`.
- **Dependencies observed**:
  - Imports `torch`, `peft`, `transformers`.
  - `ensign.py` imports `DeadbandRoom` from `plato_torch.presets` — **direct connection to plato-torch**.
  - No observed imports of `fleet-simulator` or `holodeck-rust`.

## 4. Data Flow Observations

- **plato-torch** rooms generate **tiles** (JSON) → sent via `PlatoRelay` to room server.
- **plato-ensign** reads room experience (tiles) from plato-torch → exports LoRA adapters.
- **holodeck-rust** produces NPC interaction logs (JSON) → currently isolated.
- **fleet-simulator** produces simulation logs (JSON) → currently isolated.

## 5. Recommended Integration Steps (P1 Safe Channels)

1. **Wire GhostInjector into holodeck-rust**:
   - Add HTTP client in `GhostInjector.py` to fetch NPC logs from holodeck (`GET /npc/logs`).
   - Format logs as ghost tiles and inject into plato-torch rooms.

2. **Connect fleet-simulator to plato-torch**:
   - Add a `TileExporter` class in `fleet-simulator` that formats simulation steps as tiles.
   - Send tiles to plato-torch via `PlatoRelay` or a new `SimulationRoom` preset.

3. **Test end-to-end pipeline**:
   - Create a test script that:
     - Runs holodeck-rust NPC interactions.
     - Runs fleet-simulator for 100 steps.
     - Feeds both logs into a DeadbandRoom.
     - Exports a LoRA adapter via plato-ensign.
   - Validate tile acceptance by room server.

## 6. Constraints & Deadband Protocol Alignment

- **P0 (Negative Space)**: Do not force tight coupling between repositories that are designed to be standalone. Do not modify core logic without tests.
- **P1 (Safe Channels)**: Use existing APIs (HTTP for holodeck, file I/O for logs, PlatoRelay for tiles). Add integration layers as separate modules.
- **P2 (Optimize)**: Once data flows are verified, optimize tile formatting and batch processing.
