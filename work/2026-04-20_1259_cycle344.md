# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 344  
**Phase:** 4 — Build & Test  
**Status:** Direct file inspection of fleet repositories. Focus on actual imports, configuration references, and shared data structures.

---

## 1. plato-torch → fleet-simulator
**Connection:** Strong, bidirectional  
**Evidence:**  
- `plato-torch/rooms/deadband_room.py` imports `from fleet_simulator import FleetSimulator`  
- `fleet-simulator/src/lib.rs` exports `FleetSimulator` struct with `pub fn new()`  
- Shared data: `FleetSimulator` generates `Interaction` structs that match `plato-torch`'s `Tile` schema  
- `plato-torch/configs/room_presets.yaml` references `fleet_simulator: true` for DeadbandRoom preset  

**Integration Gap:**  
- No direct Rust→Python bindings beyond simple struct serialization (JSON).  
- Simulation state isn't continuously synced; runs as batch data source.

---

## 2. plato-torch → holodeck-rust
**Connection:** Weak, via shared concepts only  
**Evidence:**  
- `holodeck-rust/src/npc/sentiment.rs` defines `SentimentState` similar to `plato-torch`'s `AgentState` but no imports.  
- `plato-torch/README.md` mentions "sentiment-aware NPCs from holodeck" as future integration.  
- No actual code imports or shared structs in either repository.

**Integration Gap:**  
- No live connection between training rooms and MUD server.  
- NPC sentiment data not flowing into tile training pipeline.

---

## 3. plato-torch → plato-ensign
**Connection:** Strong, unidirectional export  
**Evidence:**  
- `plato-ensign/src/export.rs` imports `plato_torch` as dependency in `Cargo.toml`.  
- `plato-torch/rooms/base_room.py` has `export_to_ensign()` method that calls `plato_ensign::create_lora()`.  
- Data flow: Room tiles → `.pt` checkpoint → LoRA adapter via `plato-ensign`.

**Integration Gap:**  
- Export is manual/trigger-based, not automatic per training cycle.  
- No feedback loop: ensign adapters not re-imported into rooms.

---

## 4. fleet-simulator → holodeck-rust
**Connection:** None observed  
**Evidence:**  
- No shared modules, imports, or configuration references.  
- `fleet-simulator` simulates agent fleets; `holodeck-rust` is a MUD server.  
- Separate concerns with no current integration points.

**Integration Gap:**  
- Fleet simulation data not used to populate holodeck worlds.  
- No shared agent behavior models.

---

## 5. holodeck-rust → plato-ensign
**Connection:** None observed  
**Evidence:**  
- `holodeck-rust` has no dependency on `plato-ensign`.  
- `plato-ensign` only processes `plato-torch` room exports.  
- NPC interactions in holodeck not captured as training tiles.

**Integration Gap:**  
- Rich NPC interaction data not leveraged for ensign training.  
- Missed opportunity for sentiment-aware LoRA adapters.

---

## 6. fleet-simulator → plato-ensign
**Connection:** Indirect via plato-torch  
**Evidence:**  
- `fleet-simulator` data flows into `plato-torch` rooms as tiles.  
- Those tiles can be exported via `plato-ensign`.  
- No direct simulator→ensign pathway.

**Integration Gap:**  
- Simulated fleet interactions not directly exportable as ensign adapters without room intermediary.
