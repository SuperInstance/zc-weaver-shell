# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 136  
**Phase:** 4 — Build & Test  
**Status:** Mapping current connections and identifying integration points.

Based on reading the actual files in the fleet repos, here is the current state of integration between plato-torch, fleet-simulator, holodeck-rust, and plato-ensign.

---

## 1. **plato-torch** → **plato-ensign** → **fleet-simulator**
**Connection:** **Strong & Direct**  
- **plato-torch** generates training rooms (e.g., `DeadbandRoom`) that produce interaction tiles.  
- **plato-ensign** consumes these tiles directly via `TileLoader` and exports room experience as LoRA adapters.  
- **fleet-simulator** is listed as a data source in `plato-ensign/config/training_config.yaml` under `data_sources`.  
- **Flow:** `plato-torch` → (tiles) → `plato-ensign` → (LoRA) → potential use in `fleet-simulator` for agent fine-tuning.

**Evidence:**  
- `plato-ensign/src/tile_loader.rs` imports from `plato_torch::tile`.  
- `plato-ensign/config/training_config.yaml` includes `fleet-simulator` in data paths.  
- `plato-torch/src/rooms/deadband_room.rs` explicitly logs tiles for ensign consumption.

---

## 2. **holodeck-rust** → **plato-torch**  
**Connection:** **Weak / Indirect**  
- **holodeck-rust** is a sentiment-aware MUD server that generates NPC interactions.  
- No direct import of `plato-torch` in `holodeck-rust` source.  
- However, `holodeck-rust` could produce narrative logs that *could* be formatted as tiles for `plato-torch` rooms, but this is not implemented.

**Evidence:**  
- `holodeck-rust/src/main.rs` shows a standalone Tokio-based MUD server with no references to `plato-torch`.  
- `holodeck-rust/README.md` describes it as "sentiment-aware NPCs" but does not mention tile generation.

---

## 3. **fleet-simulator** → **plato-torch**  
**Connection:** **Moderate / Config-Driven**  
- **fleet-simulator** generates synthetic training data (agent interactions, environment states).  
- `plato-torch` rooms can be configured to use `fleet-simulator` output as a tile source via `ExternalTileSource`.  
- This is a one-way data flow: simulator → tiles → room training.

**Evidence:**  
- `plato-torch/src/tile_source/external.rs` defines `ExternalTileSource` with a `simulator` variant.  
- `fleet-simulator/src/lib.rs` exports `SimulationOutput` struct that could be serialized into tile format.

---

## 4. **plato-ensign** → **holodeck-rust**  
**Connection:** **None**  
- No references in either codebase.  
- `holodeck-rust` does not load LoRA adapters; it uses hardcoded sentiment models.  
- `plato-ensign` does not output to MUD server format.

**Evidence:**  
- `holodeck-rust/Cargo.toml` has no dependency on `plato-ensign` or `lorax` (LoRA library).  
- `plato-ensign/src/lib.rs` exports only `train_ensign` and `export_lora`, no MUD integration.

---

## 5. **GhostInjector** (plato-torch) → **holodeck-rust**  
**Connection:** **None — Identified Integration Gap**  
- **GhostInjector** (`plato-torch/src/ghost_injector.rs`) reads ghost tiles (P0 knowledge from dead agents) and injects them into rooms.  
- **holodeck-rust** could benefit from ghost tiles as pre-loaded NPC memories or dialogue constraints.  
- Currently no bridge between them.

**Evidence:**  
- `GhostInjector` has a `inject_into_room` method but only works with `plato-torch` room traits.  
- `holodeck-rust` NPCs use `SentimentEngine`, not tile-based memory.

---

## 6. **DeadbandRoom** (plato-torch) → **plato-relay**  
**Connection:** **None — Identified Integration Gap**  
- **DeadbandRoom** is a training room preset that enforces the Deadband Protocol (P0→P1→P2).  
- **plato-relay** is a message bus for inter-room communication (found in `plato-torch/src/relay/`).  
- DeadbandRoom does not currently publish its phase transitions to the relay.

**Evidence:**  
- `DeadbandRoom` implements `Room` trait but does not use `PlatoRelay` client.  
- `plato-relay/src/lib.rs` provides `Broadcast` and `Subscribe` functionality unused by DeadbandRoom.
