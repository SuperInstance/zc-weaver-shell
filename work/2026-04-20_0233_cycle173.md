# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 173  
**Phase:** 4 — Build & Test  
**Status:** Mapping current connections and identifying integration points.

Based on reading the actual files in the fleet repos, here is the current state of integration between the four key technologies: **plato-torch**, **fleet-simulator**, **holodeck-rust**, and **plato-ensign**.

## 1. Verified Connections (What IS Connected)

### plato-torch ↔ plato-ensign
- **Connection:** Direct and explicit.  
- **Evidence:** `plato-torch/rooms/deadband_room.py` imports `plato_ensign` and uses `EnsignExporter` to export room experience as LoRA adapters.  
- **Data Flow:** Room interaction tiles → training loop → LoRA adapter export via `plato_ensign`.  
- **Status:** Live and functional.

### plato-torch ↔ fleet-simulator
- **Connection:** Indirect via shared tile format.  
- **Evidence:** `fleet-simulator/src/simulation.rs` generates `InteractionTile` structs that match the JSON schema expected by `plato-torch` rooms.  
- **Data Flow:** Fleet simulation runs → produce synthetic interaction tiles → tiles can be ingested by any `plato-torch` room for training.  
- **Status:** Compatible but not wired; tiles must be manually moved or piped.

### holodeck-rust ↔ (plato-torch, fleet-simulator)
- **Connection:** Minimal; potential via sentiment-aware NPCs.  
- **Evidence:** `holodeck-rust/src/npc/sentiment.rs` defines `SentimentEngine` that could generate NPC dialogue/actions. No import of `plato-torch` or `fleet-simulator` modules.  
- **Data Flow:** None currently. NPCs operate in isolated MUD environment.  
- **Status:** Standalone; no active integration.

## 2. Integration Gaps (What Is NOT Connected)

### Gap 1: holodeck-rust → plato-torch
- **Missing:** No pipeline to send NPC interactions from holodeck to a `plato-torch` room as training tiles.  
- **Impact:** Holodeck’s sentiment-aware NPCs cannot contribute directly to fleet training.  
- **Potential Integration Point:** Add a `TileExporter` to holodeck that formats NPC interactions as `InteractionTile` JSON and POSTs to a `plato-torch` room’s ingestion endpoint.

### Gap 2: fleet-simulator → holodeck-rust
- **Missing:** No way to seed holodeck scenarios with fleet simulation data.  
- **Impact:** Simulations cannot drive NPC behavior or scenario generation.  
- **Potential Integration Point:** Extend `holodeck-rust` `ScenarioLoader` to read `fleet-simulator` output and spawn NPCs with simulated histories.

### Gap 3: plato-ensign → holodeck-rust
- **Missing:** No mechanism to load a trained ensign (LoRA) into holodeck NPCs.  
- **Impact:** NPCs cannot be personalized or improved with fleet-trained adapters.  
- **Potential Integration Point:** Add `EnsignLoader` to holodeck that loads a `.safetensors` LoRA and applies it to NPC dialogue generation.

### Gap 4: Deadband Protocol ↔ holodeck-rust
- **Missing:** No implementation of P0/P1/P2 logic in NPC decision-making.  
- **Impact:** NPCs do not use fleet doctrine for safe interaction.  
- **Potential Integration Point:** Integrate `deadband_protocol.rs` (if written) into `npc/decision.rs`.

## 3. Immediate Integration Task (P1 Safe Channel)

**Task:** Wire GhostInjector into holodeck.  
**Why:** GhostInjector (from `plato-torch/ghost_injector.py`) injects “ghost tiles” (P0 negative knowledge from dead agents) into training. Holodeck NPCs could use these as cautionary memories.  
**Steps:**  
1. Read ghost tile format from `plato-torch/ghost_tiles/`.  
2. Add `GhostMemory` component to holodeck NPCs.  
3. Inject ghost tiles as “past experiences” that influence sentiment and decisions.  
**Expected Outcome:** NPCs avoid actions marked as P0 violations.
