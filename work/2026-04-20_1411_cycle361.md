## 1. Plato-Torch (Training Rooms)

**Location:** `plato-torch/`

**Key Files:**
- `presets/deadband_room.py` — Deadband Protocol training room
- `presets/__init__.py` — exports 25 presets
- `core/room.py` — base Room class
- `core/tile.py` — Tile data structure
- `core/ensign_exporter.py` — exports room experience to LoRA

**Connections Found:**

1. **To Plato-Ensign (direct import)**
   - `ensign_exporter.py` imports `from plato_ensign import export_lora`
   - Exports trained room state as LoRA adapter
   - One-way flow: plato-torch → plato-ensign

2. **To Fleet-Simulator (configuration reference)**
   - `presets/deadband_room.py` line 47: `"simulator_config": "fleet_simulator/configs/basic.json"`
   - Reference only — no direct import
   - Simulator runs separately, provides training data via file I/O

3. **To Holodeck-Rust (no direct connection)**
   - No imports of holodeck modules
   - No shared data structures in code
   - Potential integration point: sentiment data from NPCs could feed training

## 2. Fleet-Simulator (Training Data Generation)

**Location:** `fleet-simulator/`

**Key Files:**
- `src/simulator.rs` — main simulation engine
- `configs/basic.json` — referenced by deadband_room.py
- `output/training_data.json` — generated scenarios

**Connections Found:**

1. **To Plato-Torch (data provider)**
   - Generates JSON training data
   - Plato-torch reads via file path in config
   - One-way flow: fleet-simulator → plato-torch

2. **To Holodeck-Rust (potential connection)**
   - Both are Rust projects
   - No shared crates in Cargo.toml
   - Could share entity/behavior models

3. **To Plato-Ensign (no connection)**
   - Simulator doesn't know about ensign exports
   - Training data → room → ensign is indirect pipeline
