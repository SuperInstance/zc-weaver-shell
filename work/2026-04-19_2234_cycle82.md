## 1. Verified Current Connections (Based on fleet-knowledge/ and repo READMEs)

### **plato-torch → plato-ensign** (Strong, documented)
- **Connection:** Direct export pipeline.
- **Mechanism:** `plato-torch` training rooms accumulate interaction tiles. `plato-ensign` can export a room's experience into a LoRA adapter.
- **Evidence:** `FLEET-CONTEXT.md` states: "plato-ensign: Exports room experience into LoRA adapters." Architecture section: "Ensigns are optional exports of room experience."
- **Status:** **Connected.** This is a core, intended data flow.

### **plato-torch → holodeck-rust** (Indirect, via data format)
- **Connection:** Shared training data concept (tiles).
- **Mechanism:** `holodeck-rust` is a MUD server with sentiment-aware NPCs. Its interactions could generate tiles compatible with `plato-torch` room format, but no direct API call or library import is specified.
- **Evidence:** Both are listed as "Key Technologies." Both deal with agent interactions and training data.
- **Status:** **Potentially compatible, but not explicitly wired.** A "tile" from holodeck could likely be ingested by a plato-torch room, but no integration code is mentioned.

### **fleet-simulator → plato-torch** (Indirect, data source)
- **Connection:** Training data supply.
- **Mechanism:** `fleet-simulator` generates training data for the fleet. This data could be formatted into tiles and used to train rooms in `plato-torch`.
- **Evidence:** `FLEET-CONTEXT.md`: "fleet-simulator: Fleet simulation for training data."
- **Status:** **Logical source, not explicit sink.** The simulator's output format and ingestion path into plato-torch are not detailed.

### **plato-ensign → (Runtime)** (Downstream usage)
- **Connection:** Trained model deployment.
- **Mechanism:** A LoRA adapter exported by `plato-ensign` is intended to be loaded into a runtime (e.g., `cudaclaw` GPU runtime, `flux-runtime` VM) to create a specialized "ensign" agent.
- **Evidence:** "Ensigns are optional exports of room experience." The purpose is to create deployable specialist models.
- **Status:** **Connected by design.** The output has a clear intended consumer.

## 2. Identified Integration Gaps (P1: Safe Channels to Explore)

### **Gap 1: holodeck-rust → plato-torch Tile Pipeline**
- **What's missing:** A defined method to stream sentiment-aware NPC interactions from `holodeck-rust` into a `plato-torch` training room as validated tiles.
- **P1 Action:** Examine `holodeck-rust` output format and `plato-torch` tile ingestion API. Create a small adapter service or script.
- **Link to Backlog:** "Wire GhostInjector into holodeck" – this may be related.

### **Gap 2: fleet-simulator → plato-torch Data Bridge**
- **What's missing:** A standardized data contract and pipeline for turning simulated fleet interactions into training tiles.
- **P1 Action:** Locate example output from `fleet-simulator` and example tile structure in `plato-torch`. Draft a conversion spec.
- **Note:** This is a high-value channel for generating synthetic training data at scale.

### **Gap 3: plato-torch Room → plato-relay (DeadbandRoom)**
- **What's missing:** The `DeadbandRoom` (a plato-torch preset) does not currently broadcast its state or log to `plato-relay` for real-time monitoring or inter-room communication.
- **P1 Action:** Investigate `plato-relay`'s API. Determine if `DeadbandRoom` can emit events (P0/P1/P2 state changes) to a relay channel.
- **Link to Backlog:** "Connect DeadbandRoom to plato-relay."

### **Gap 4: Ghost Tile Integration Point**
- **What's missing:** While "ghost tiles" are defined as dead agents' P0 lessons, there's no specified system that injects these into active training streams.
- **P1 Action:** Identify where ghost tiles are stored and which component (`plato-torch`? `holodeck-rust`?) should read them. The "GhostInjector" from the backlog is likely this component.
- **Potential Channel:** `GhostInjector` could be a middleware that reads from a ghost tile store and feeds into `holodeck-rust` (as NPC knowledge) or directly into a `plato-torch` room's training buffer.

## 3. Proposed Phase 4 Integration Test (P2: Optimize within a Channel)

**Test:** **End-to-end tile flow from a simulated source to a room export.**
1.  **Source:** Use a simple, scripted simulation (stand-in for `fleet-simulator`) to generate 10 raw interaction events.
2.  **Bridge:** Write a converter (`sim_to_tile.py`) that formats these events into the `plato-torch` tile JSON schema.
3.  **Ingestion:** Load these tiles into a fresh `DeadbandRoom` instance via its API.
4.  **Export:** Use `plato-ensign` to export the room's experience (now containing the simulated tiles) into a LoRA adapter.
5.  **Verification:** Check that the LoRA file is created and its metadata includes the simulated tile signatures.

**Outcome:** Validates the data pipeline across three components (simulator → room → ensign), exposing any schema mismatches or API issues. This is a "safe channel" (P1) because it uses existing, documented export functions.
