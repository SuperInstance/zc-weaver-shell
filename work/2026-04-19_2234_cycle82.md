## 1. Verified Current Connections

Based on reading the fleet-knowledge files and repository structures:

### **plato-torch → plato-ensign** (Strong Connection)
- **plato-torch** generates training rooms and accumulates interaction tiles.
- **plato-ensign** exports room experience (tiles) into LoRA adapters.
- **Connection:** Direct pipeline. Rooms train → ensigns export. This is the core training loop.

### **plato-torch → holodeck-rust** (Potential Connection)
- **plato-torch** has training room presets (e.g., DeadbandRoom).
- **holodeck-rust** is a MUD server with sentiment-aware NPCs.
- **Connection:** Not yet wired. Holodeck could be a training room preset in plato-torch, or its NPC interactions could become tiles.

### **fleet-simulator → plato-torch** (Data Connection)
- **fleet-simulator** generates training data (simulated fleet interactions).
- **plato-torch** uses training data to train rooms.
- **Connection:** Simulator output likely feeds into plato-torch as training tiles or datasets.

### **holodeck-rust → plato-ensign** (Indirect Potential)
- If holodeck interactions were captured as tiles, they could be exported via plato-ensign.
- Currently no direct link.

### **constraint-theory-core → fleet-simulator** (Architectural)
- **constraint-theory-core** provides geometric snapping (Rust).
- **fleet-simulator** may use it for spatial reasoning in simulations.
- **Connection:** Likely a library dependency.

### **cudaclaw & flux-runtime** (Infrastructure)
- **cudaclaw:** GPU-resident agent runtime.
- **flux-runtime:** Self-assembling bytecode VM.
- **Connection:** These are underlying runtimes; plato-torch or holodeck may deploy on them.

## 2. Integration Gaps (Actionable for Phase 4)

### **Gap 1: GhostInjector not wired into holodeck-rust**
- **Ghost tiles** (dead agents' lessons) exist as P0 knowledge.
- **holodeck-rust** NPCs could be seeded with ghost tile sentiment/memory.
- **Action:** Modify holodeck to load ghost tiles as NPC backstories or behavior constraints.

### **Gap 2: DeadbandRoom not connected to plato-relay**
- **DeadbandRoom** is a plato-torch preset.
- **plato-relay** (implied) likely handles tile distribution/validation.
- **Action:** Ensure DeadbandRoom tiles are validated and relayed to the server.

### **Gap 3: End-to-end pipeline untested**
- Current flow: fleet-simulator → plato-torch → plato-ensign.
- Missing: holodeck as a tile source, ghost tile injection, relay validation.
- **Action:** Build a test pipeline: holodeck interaction → tile → plato-torch room → ensign export.

### **Gap 4: Documentation of integration points**
- No central map of APIs, data formats, or handoff points.
- **Action:** Create `INTEGRATION.md` specifying:
  - Tile JSON schema
  - Room preset API
  - Ensign export format
  - Ghost tile injection interface
