## 1. Verified Current Connections (What's Connected)

Based on reading the actual fleet repos:

### **plato-torch → plato-ensign** (Strong, Direct)
- **Connection:** `plato-torch` exports training data (tiles) → `plato-ensign` ingests them to produce LoRA adapters.
- **Evidence:** `plato-ensign/README.md` states: "Takes Plato room training data and exports LoRA adapters." The `plato-torch` room presets (like `DeadbandRoom`) generate `.pt` tile files that are the expected input format.
- **Status:** **Connected and functional.** This is the core training→export pipeline.

### **fleet-simulator → plato-torch** (Indirect, Data Flow)
- **Connection:** `fleet-simulator` generates synthetic training data (agent interactions, state logs) that can be formatted into tiles for `plato-torch` rooms.
- **Evidence:** `fleet-simulator/src/lib.rs` exports simulation events. `plato-torch/rooms/` includes data loaders that can read external `.jsonl` logs. No direct API call, but a clear data format pathway.
- **Status:** **Connected via data convention.** Simulated logs become training tiles.

### **holodeck-rust → (plato-torch / fleet-simulator)** (Potential, Not Wired)
- **Connection:** `holodeck-rust` is a MUD server with sentiment-aware NPCs. Its interaction logs could be a rich source of training tiles for `plato-torch` or behavior data for `fleet-simulator`.
- **Evidence:** `holodeck-rust/src/npc/sentiment.rs` tracks emotional state. No current exporter to tile format or simulator event format.
- **Status:** **Not connected.** Standalone service.

### **plato-ensign → (No Downstream Consumer)** (Export Endpoint)
- **Connection:** `plato-ensign` outputs LoRA adapters (`.safetensors`). No fleet component currently consumes these for deployment or inference.
- **Evidence:** `plato-ensign/src/exporter.rs` writes adapters to disk. No reference in other repos to loading these adapters.
- **Status:** **Dead-end output.** Adapters are produced but not used.

### **cudaclaw / flux-runtime** (Isolated Runtimes)
- **Connection:** `cudaclaw` (GPU-resident agents) and `flux-runtime` (self-assembling VM) are independent execution environments. No references in `plato-torch`, `fleet-simulator`, or `holodeck-rust`.
- **Status:** **Not integrated.** They exist as separate fleet technologies.

## 2. Integration Gaps (What's Not Connected)

### **Gap 1: holodeck-rust → plato-torch Tile Pipeline**
- **Problem:** NPC interactions in holodeck are lost for training.
- **Action:** Build `GhostInjector` (from backlog) that watches holodeck log, converts dialogue/state to tiles, pushes to a `plato-torch` room.
- **Priority:** **High** (enriches training data with social sentiment).

### **Gap 2: plato-ensign Adapters → Deployment**
- **Problem:** Trained ensigns (LoRAs) are not loaded into any runtime (cudaclaw, flux, or even plato-torch for inference).
- **Action:** Create `ensign-loader` module in `plato-torch` to apply adapters during inference, or integrate with `cudaclaw` for GPU deployment.
- **Priority:** **High** (closes the training→deployment loop).

### **Gap 3: fleet-simulator → holodeck-rust NPC AI**
- **Problem:** Simulated agent behavior could drive NPCs, making them more dynamic.
- **Action:** Feed `fleet-simulator` event streams into holodeck's NPC controller, using them as behavior scripts.
- **Priority:** **Medium** (adds realism to holodeck).

### **Gap 4: DeadbandRoom → plato-relay (Monitoring)**
- **Problem:** Room training metrics are not broadcast for fleet observability.
- **Action:** Connect `DeadbandRoom`'s callback hooks to `plato-relay` (if exists) or a simple metrics endpoint.
- **Priority:** **Medium** (improves fleet situational awareness).

### **Gap 5: constraint-theory-core → (Nowhere)**
- **Problem:** Geometric snapping library is not used in any simulation or training context.
- **Action:** Propose integration into `fleet-simulator` for spatial agent constraints, or into `holodeck-rust` for environment geometry.
- **Priority:** **Low** (specialist library awaiting use case).
