## 1. Verified Connections (Based on Repository Analysis)

### **plato-torch → plato-ensign** (Strong, Direct)
- **Connection:** `plato-torch` generates training tiles (interaction data). `plato-ensign` consumes these tiles to export room experience as LoRA adapters.
- **Evidence:** `plato-ensign/README.md` states: "Exports room experience into LoRA adapters." The `plato-torch` room presets (like `DeadbandRoom`) are designed to accumulate tiles for this purpose.
- **Integration Point:** Tile data format (likely JSON/MessagePack) passed from a `plato-torch` room server to `plato-ensign`'s training pipeline.

### **plato-torch → fleet-simulator** (Indirect, Potential)
- **Connection:** `fleet-simulator` generates training data (simulated fleet interactions). This data could be formatted into tiles and ingested by a `plato-torch` room for training.
- **Evidence:** `fleet-simulator`'s purpose is "Fleet simulation for training data." `plato-torch` rooms "train from accumulated interaction tiles."
- **Integration Gap:** No explicit import pipeline or shared data schema observed in top-level docs. This is a data-source relationship, not a direct API call.

### **holodeck-rust → plato-torch** (Indirect, Conceptual)
- **Connection:** `holodeck-rust` is a MUD server with sentiment-aware NPCs. NPC interactions could become a source of rich, narrative training tiles for a `plato-torch` room.
- **Evidence:** Both systems deal with agent interactions and learning. `holodeck-rust` generates interactive scenarios; `plato-torch` codifies them into trainable data.
- **Integration Gap:** No built-in `holodeck` tile exporter or `plato-torch` room preset for MUD log ingestion.

### **plato-ensign → (External)** (Output)
- **Connection:** `plato-ensign` produces LoRA adapters—specialist AI models. These are the fleet's "ensigns," deployable for specific tasks.
- **Evidence:** Core stated purpose. This is the end of one pipeline: Room Experience → Tiles → Ensign (LoRA).

## 2. Identified Integration Gaps (Actionable for Phase 4)

### **Gap 1: GhostInjector → holodeck-rust**
- **Description:** The `GhostInjector` concept (from `FLEET-CONTEXT.md`) is a mechanism to insert "ghost tiles" (dead agents' P0 lessons) into training streams. `holodeck-rust`, as a sentiment-aware simulation, is a prime candidate to receive these lessons as behavioral constraints for its NPCs.
- **Action:** Wire a prototype. Create a `GhostInjector` service that reads a tile archive (e.g., `memory/`) and exposes an API or event stream `holodeck-rust` can subscribe to, influencing NPC decision trees.

### **Gap 2: DeadbandRoom → plato-relay**
- **Description:** `DeadbandRoom` is a `plato-torch` training preset. `plato-relay` (inferred from context) likely handles tile distribution/validation between rooms and `plato-ensign`. A direct connection ensures validated tiles flow efficiently.
- **Action:** Examine `plato-torch` for a `relay` module or config. If missing, define a `TileRelay` class in `plato-torch` that batches and sends tiles to a `plato-relay` service endpoint, implementing the fleet's zero-trust validation.

### **Gap 3: fleet-simulator → plato-torch Tile Adapter**
- **Description:** `fleet-simulator` outputs simulation data. To become useful training tiles, this data needs transformation into the tile schema `plato-torch` rooms expect.
- **Action:** Build a `SimulationTileAdapter` in `fleet-simulator` (or as a standalone microservice) that converts simulation event logs (e.g., agent actions, outcomes) into formatted tiles, ready for room ingestion.

### **Gap 4: End-to-End Pipeline Test**
- **Description:** The full theoretical pipeline is: `fleet-simulator/holodeck-rust` (data gen) → `plato-torch` (room training) → `plato-ensign` (LoRA export). No documented test validates this flow.
- **Action:** Create a minimal integration test (`test_integration_pipeline.py` or `.rs`) that mocks data from a simulator, pushes it through a `DeadbandRoom`, and verifies `plato-ensign` can process the resulting tiles.
