# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 134  
**Phase:** 4 — Build & Test  
**Status:** Mapping current connections and identifying integration points.

Based on reading the actual files in the fleet repos, here is the current state of integration between **plato-torch**, **fleet-simulator**, **holodeck-rust**, and **plato-ensign**.

---

## 1. **plato-torch** (Training Rooms)
**Role:** Hosts 25 preset training rooms (including `DeadbandRoom`). Rooms accumulate interaction tiles from agents.  
**Connections:**
- **To plato-ensign:** Direct export path. Rooms can export their accumulated experience (tiles) as LoRA adapters via `plato-ensign`'s `export_room()` function.
- **To fleet-simulator:** No direct code integration found. However, `fleet-simulator` generates training data that could be ingested as tiles into a plato-torch room (implied by architecture docs).
- **To holodeck-rust:** No direct connection found in code. Both are separate training environments.

**Integration Gap:** Rooms currently do not automatically pull simulation data from `fleet-simulator` or NPC interactions from `holodeck-rust`. Tiles must be manually submitted or generated within the room.

---

## 2. **fleet-simulator** (Fleet Simulation)
**Role:** Generates training data by simulating fleet agent interactions (multi-agent scenarios).  
**Connections:**
- **To plato-torch:** No direct import/export found. Simulator outputs are JSON logs (`simulation_logs/`). These could be transformed into tiles but no pipeline exists.
- **To holodeck-rust:** No connection found. Different simulation purposes (fleet vs. sentiment-aware NPCs).
- **To plato-ensign:** No direct link. However, simulation logs could theoretically be used as training data for an ensign.

**Integration Gap:** Simulation outputs are isolated. No automatic tile generation or room ingestion.

---

## 3. **holodeck-rust** (Sentiment-Aware MUD)
**Role:** Multi‑User Dungeon server with NPCs that remember sentiment and interactions.  
**Connections:**
- **To plato-torch:** No direct link. NPC interactions are not automatically turned into tiles.
- **To fleet-simulator:** No connection.
- **To plato-ensign:** No direct export found.

**Integration Gap:** Rich NPC interaction data is trapped in the MUD. No pipeline to convert dialogues/events into training tiles.

---

## 4. **plato-ensign** (LoRA Adapter Exporter)
**Role:** Exports room experience from plato-torch into lightweight LoRA adapters.  
**Connections:**
- **To plato-torch:** Direct dependency. Imports `Room` class and calls `room.get_tiles()`.
- **To fleet-simulator:** None.
- **To holodeck-rust:** None.

**Integration Gap:** Only works with plato-torch rooms. Cannot export from other data sources (simulator logs, MUD events).

---

## 5. **GhostInjector** (Ghost Tile System)
**Role:** Injects “ghost tiles” (lessons from dead agents) as P0 negative‑space knowledge.  
**Current State:** Found in `plato-torch/ghost_injector.py`.  
**Connections:**
- **To plato-torch:** Injects into rooms via `inject_ghost_tiles(room_id)`.
- **To holodeck-rust:** **Not connected**. This is a key integration gap — ghost tiles could be injected into MUD NPC memory.
- **To fleet-simulator:** No connection.

---

## 6. **plato-relay** (Room Communication)
**Role:** Facilitates communication between rooms (e.g., Mirror Play).  
**Connections:**
- **To plato-torch:** Used by rooms to send/receive screen states.
- **To DeadbandRoom:** DeadbandRoom has a `relay` attribute but no active usage in current code.
- **To other systems:** Not connected to simulator or holodeck.

---

## Summary of Integration Gaps (P0/P1 Priorities)

### **P0: What NOT to do**
- Do not assume data flows automatically between systems. Currently, each repo is a silo.
- Do not modify core room logic without testing tile integrity.

### **P1: Safe Channels (Where integration CAN happen)**
1. **fleet-simulator → plato-torch:** Create a `SimulationTileGenerator` that reads JSON logs and formats them as tiles for a room.
2. **holodeck-rust → plato-torch:** Add a tile‑export endpoint to the MUD server that sends NPC interaction summaries to a designated room.
3. **GhostInjector → holodeck-rust:** Extend ghost injection to modify NPC memory in holodeck‑rust (requires Rust FFI or API).
4. **DeadbandRoom → plato-relay:** Implement actual relay usage for Mirror Play experiments.

### **P2: Optimize Within Channels**
- Once pipelines exist, optimize tile formatting, batch ingestion, and real‑time streaming.
