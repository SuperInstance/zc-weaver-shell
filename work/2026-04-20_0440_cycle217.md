## 1. Current Verified Connections

### plato-torch → fleet-simulator
- **Connection:** Strong, direct integration via `fleet_simulator` Python module.
- **Evidence:** `plato-torch/rooms/deadband_room.py` imports `from fleet_simulator import FleetSimulator`.
- **Usage:** `DeadbandRoom` instantiates `FleetSimulator` as `self.simulator`. Calls `self.simulator.step()` each training cycle.
- **Data Flow:** Room passes agent actions to simulator; simulator returns next state, rewards, done flags.
- **Shared Config:** `plato-torch/configs/deadband_config.yaml` references `simulator_config_path`.

### plato-torch → plato-ensign
- **Connection:** Direct export pipeline.
- **Evidence:** `plato-torch/export/ensign_exporter.py` exists. `plato-torch/rooms/base_room.py` has `export_to_ensign()` method.
- **Usage:** After training, a room can export its experience (tiles, weights) to a LoRA adapter via `plato-ensign` library.
- **Data Flow:** Room state → LoRA weights file (.safetensors).
- **Shared Config:** `plato-torch/configs/export_config.yaml` specifies ensign adapter parameters.

### fleet-simulator → holodeck-rust
- **Connection:** None found in code.
- **Evidence:** No Rust imports in `fleet_simulator/` Python code. No Python bindings in `holodeck-rust/` visible.
- **Note:** May be planned but not implemented.

### holodeck-rust → plato-torch
- **Connection:** None found in code.
- **Evidence:** `holodeck-rust/` is a standalone Rust MUD server. No Python/plato-torch references.
- **Note:** Separate service; could connect via API (unimplemented).

### plato-ensign → plato-torch
- **Connection:** One-way dependency (plato-torch uses plato-ensign).
- **Evidence:** `plato-ensign/setup.py` shows no dependency on plato-torch.
- **Usage:** plato-ensign is a library called by plato-torch's exporter.
