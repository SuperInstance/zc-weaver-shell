## 1. Verified Connections (Based on Code Inspection)

### plato-torch → plato-ensign (Strong Connection)
- **plato-torch/src/ensign_exporter.py** imports from `plato_ensign`:
  - `from plato_ensign import export_room_to_lora, validate_adapter`
- **plato-torch/presets/DeadbandRoom.py** calls `ensign_exporter.export()` after training cycles.
- **Data Flow**: Room training logs → LoRA adapter files (.safetensors) via plato-ensign's export pipeline.

### fleet-simulator → plato-torch (One-Way Data Feed)
- **fleet-simulator/src/loggers/tile_logger.py** writes to `training_tiles/` directory.
- **plato-torch/src/data/loaders.py** includes `SimulationTileLoader` class that reads from `fleet-simulator/output/tiles/`.
- **Connection Type**: File-system based. No direct API calls; simulator outputs tiles, plato-torch ingests them.

### holodeck-rust → (No Direct Code Links Found)
- **holodeck-rust/src/main.rs** shows no imports of plato-torch, fleet-simulator, or plato-ensign.
- **holodeck-rust/Cargo.toml** dependencies: `serde`, `tokio`, `warp` (web server). No fleet crates.
- **Observation**: Standalone MUD server. May integrate via REST API (unverified in current code).

### plato-ensign → plato-torch (Callback Registration)
- **plato-ensign/src/integrations/plato_torch_hooks.rs** defines `register_room_export_hook()`.
- **plato-torch/src/init.py** calls this hook during startup (found in commented initialization block).
- **Status**: Hook system exists but appears partially implemented.
