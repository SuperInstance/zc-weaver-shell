## 1. Current Verified Connections

### plato-torch → fleet-simulator
- **Connection:** Strong, direct dependency.
- **Evidence:** `plato-torch/requirements.txt` includes `fleet-simulator`.
- **Usage:** `plato-torch/rooms/deadband_room.py` imports `from fleet_simulator import FleetSimulator` to generate synthetic training data.
- **Data Flow:** FleetSimulator creates agent interaction logs → DeadbandRoom ingests them as training tiles.

### plato-torch → plato-ensign
- **Connection:** Direct export pipeline.
- **Evidence:** `plato-torch/export/ensign_exporter.py` exists.
- **Usage:** After a room trains, `ensign_exporter.py` can convert the trained model checkpoint into a LoRA adapter via `plato-ensign` library calls.
- **Data Flow:** Trained PyTorch checkpoint → LoRA weights (.safetensors) + config.json.

### holodeck-rust → (indirect) plato-torch
- **Connection:** Planned but not yet wired.
- **Evidence:** `holodeck-rust/src/protocol/mod.rs` defines `Tile` struct matching `plato-torch` tile schema.
- **Usage:** Intended for sentiment-aware NPC interactions to generate training tiles for plato-torch rooms.
- **Current State:** No actual import or API call in code. Manual tile transfer possible.

### fleet-simulator → (none)
- **Connection:** Standalone library.
- **Evidence:** `fleet-simulator/setup.py` shows no dependencies on other fleet repos.
- **Usage:** Called by plato-torch; could be used elsewhere but currently isn’t.

### plato-ensign → (none)
- **Connection:** Standalone export tool.
- **Evidence:** `plato-ensign/ensign/cli.py` operates on checkpoint files; no inbound dependencies.
- **Usage:** Post-training step; not integrated into automatic pipeline.

## 2. Integration Gaps (What’s Not Connected)

### Gap 1: holodeck-rust → plato-torch real-time tile stream
- **Missing:** No Rust→Python socket or HTTP API to send tiles from holodeck to a running plato-torch room.
- **Impact:** Holodeck NPC interactions remain isolated; tiles must be manually copied.

### Gap 2: plato-torch → holodeck-rust feedback loop
- **Missing:** No mechanism for a trained ensign (LoRA) to be loaded into holodeck NPCs.
- **Impact:** Holodeck NPCs cannot improve via fleet training.

### Gap 3: DeadbandRoom → plato-relay
- **Missing:** `plato-relay` (if referring to a message bus) not present in inspected repos.
- **Impact:** Rooms cannot broadcast training progress or share tiles fleet-wide.

### Gap 4: GhostInjector (unlocated) → holodeck-rust
- **Missing:** `GhostInjector` not found in current repo set; presumed to be a P0 knowledge injector.
- **Impact:** Holodeck cannot seed NPCs with “ghost tile” lessons from dead agents.

### Gap 5: End-to-end automated pipeline
- **Missing:** No single script or service orchestrates: holodeck → tiles → plato-torch training → ensign export → holodeck update.
- **Impact:** Manual steps required between stages; reduces iteration speed.

## 3. Immediate Integration Priorities (P1 Safe Channels)

1. **Wire holodeck-rust tile output to plato-torch input**  
   - Safe channel: Add a simple HTTP endpoint in `plato-torch` to receive POSTed tiles.
   - Implementation: `holodeck-rust` can serialize `Tile` struct to JSON and POST to room’s ingestion endpoint.

2. **Connect DeadbandRoom to a message bus**  
   - Safe channel: Use existing `plato-torch/rooms/room_base.py` logging hooks to emit events.
   - Implementation: Add a pluggable “relay adapter” that rooms can configure (file, Redis, or ZeroMQ).

3. **Test GhostInjector concept**  
   - Safe channel: Create a `GhostInjector` module in `plato-torch` that loads P0 tile files and prepends them to training batches.
   - Implementation: Read from `fleet-knowledge/ghost-tiles/` directory.
