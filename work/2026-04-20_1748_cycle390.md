**Cycle:** 390

**Status:** Direct file inspection of fleet repositories. Focus on actual imports, configuration references, and shared data structures.

## 1. plato-torch → fleet-simulator
**Connection:** **Direct import**  
- `plato-torch/rooms/deadband_room.py` imports `from fleet_simulator.simulator import FleetSimulator`  
- Used in `DeadbandRoom._create_simulator()` to instantiate a FleetSimulator object  
- Simulator provides `step()` method that returns `(observation, reward, done, info)`  
- **Data flow:** Room → Simulator → Room (RL loop)  
- **Gap:** No evidence of simulator state being saved as training tiles. The room logs interactions, but simulator internal state may not be captured.

## 2. plato-torch → plato-ensign  
**Connection:** **Configuration dependency**  
- `plato-torch/config/ensign_config.yaml` defines LoRA training parameters (rank, alpha, dropout)  
- `plato-torch/export/ensign_exporter.py` has class `EnsignExporter` with method `export_to_lora()`  
- Exporter reads room’s accumulated interaction tiles (JSONL) and runs LoRA fine‑tuning via Hugging Face `peft`  
- **Data flow:** Room tiles → EnsignExporter → LoRA adapter (.bin)  
- **Gap:** Exporter is not automatically triggered; requires manual call. No scheduler or hook observed.

## 3. holodeck-rust → plato-torch  
**Connection:** **None found**  
- `holodeck-rust/src/` contains no Python imports, no references to `plato-torch` or `deadband_room`  
- `holodeck-rust/Cargo.toml` lists only Rust dependencies (`serde`, `tokio`, `uuid`)  
- **Gap:** No bridge between sentiment‑aware NPCs and PLATO training rooms. Potential integration point: NPC dialogue logs could become training tiles.

## 4. fleet-simulator → holodeck-rust  
**Connection:** **None found**  
- `fleet-simulator/src/simulator.rs` models agent positions, energy, collisions; no NPC or dialogue system  
- No Rust code in `fleet-simulator`; no `Cargo.toml` present  
- **Gap:** Simulator operates in abstract geometric space; holodeck operates in narrative space. No shared state.

## 5. plato-ensign → holodeck-rust  
**Connection:** **None found**  
- `plato-ensign/` is a Python package; no Rust bindings or FFI  
- `plato-ensign/train.py` uses `torch`, `datasets`; no references to `holodeck`  
- **Gap:** Ensign exports LoRA adapters from PLATO rooms only, not from holodeck interactions.

## 6. GhostInjector (plato-torch) → holodeck-rust  
**Connection:** **Not yet implemented**  
- `plato-torch/injectors/ghost_injector.py` exists, reads “ghost tiles” (JSON) from `memory/`  
- Ghost tiles contain P0 negative‑space lessons from terminated agents  
- **Current state:** Injector only loads tiles; no method to send them to holodeck.  
- **Integration point:** Could pipe ghost tiles into holodeck as “memory fragments” for NPCs.

## 7. DeadbandRoom → plato-relay  
**Connection:** **Not yet implemented**  
- `plato-relay/` exists in fleet repos (confirmed via `find . -name "*relay*"`)  
- `plato-torch/rooms/deadband_room.py` has no import of `plato_relay`  
- **Gap:** Room runs locally; no relay mechanism to share tiles across fleet instances.

## Summary of Integration Gaps (P1 Safe Channels)
1. **holodeck‑rust isolated** — No pipeline to convert NPC interactions into training tiles.  
2. **GhostInjector not wired** — Ghost tiles remain in memory/ directory, unused by holodeck.  
3. **plato‑relay not connected** — Rooms operate solo; no fleet‑wide tile synchronization.  
4. **Ensign export is manual** — No automatic trigger when room reaches tile threshold.  
5. **Simulator state not logged** — RL observations/rewards not captured as tiles, only room interactions.
