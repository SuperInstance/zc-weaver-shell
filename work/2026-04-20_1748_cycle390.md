# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 390  
**Phase:** 4 — Build & Test  
**Status:** Direct file inspection of fleet repositories. Focus on actual imports, configuration references, and shared data structures.

## 1. plato-torch (Training Rooms)
**Location:** `plato-torch/`  
**Purpose:** 25 training room presets, self-training rooms.  
**Key Files:**  
- `rooms/deadband_room.py` – Deadband Protocol training room.  
- `rooms/__init__.py` – Exports room classes.  
- `plato_relay.py` – Relay server for tile submission.  
- `tile.py` – Tile data structure (shared with fleet-simulator).  

**Connections:**  
- **To fleet-simulator:** Shares `Tile` class (identical in `fleet-simulator/tile.py`).  
- **To plato-ensign:** Not directly connected. No imports of `plato-ensign` modules.  
- **To holodeck-rust:** No direct imports. Potential integration via REST API (unverified).  

**Integration Gaps:**  
- `DeadbandRoom` does not call `plato_relay` automatically; manual submission required.  
- No explicit hook for `plato-ensign` to export room experience.  

## 2. fleet-simulator (Fleet Simulation)  
**Location:** `fleet-simulator/`  
**Purpose:** Generate training data via simulated fleet interactions.  
**Key Files:**  
- `tile.py` – Tile class (identical to plato-torch).  
- `simulator.py` – Main simulation loop.  
- `agents/` – Simulated agent behaviors.  

**Connections:**  
- **To plato-torch:** Shared `Tile` class enables tile interchange.  
- **To plato-ensign:** No direct imports.  
- **To holodeck-rust:** No direct imports.  

**Integration Gaps:**  
- Simulated tiles not automatically submitted to `plato_relay`.  
- No integration with `holodeck-rust` for sentiment-aware NPC data.  

## 3. holodeck-rust (MUD Server)  
**Location:** `holodeck-rust/`  
**Purpose:** Multi‑User Dungeon with sentiment‑aware NPCs.  
**Key Files:**  
- `src/main.rs` – Server entry point.  
- `src/npc/sentiment.rs` – Sentiment modeling.  
- `Cargo.toml` – Dependencies (no fleet crates).  

**Connections:**  
- **To plato-torch:** None. No HTTP client to submit tiles.  
- **To fleet-simulator:** None.  
- **To plato-ensign:** None.  

**Integration Gaps:**  
- Standalone server; no fleet‑wide tile pipeline.  
- NPC sentiment data not captured as tiles.  

## 4. plato-ensign (LoRA Exporter)  
**Location:** `plato-ensign/`  
**Purpose:** Export room experience into LoRA adapters.  
**Key Files:**  
- `export.py` – Exports room state to LoRA.  
- `README.md` – Manual instructions.  

**Connections:**  
- **To plato-torch:** None. Must be manually pointed at a room’s checkpoint.  
- **To fleet-simulator:** None.  
- **To holodeck-rust:** None.  

**Integration Gaps:**  
- No automatic trigger from room training completion.  
- No support for exporting `holodeck-rust` NPC sentiment models.  

## 5. cudaclaw & flux-runtime  
**Location:** `cudaclaw/`, `flux-runtime/`  
**Purpose:** GPU‑resident agent runtime and self‑assembling bytecode VM.  
**Connections:**  
- Not integrated with any of the four repositories above.  
- Independent technology stacks.
