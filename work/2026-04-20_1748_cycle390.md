## 1. plato-torch (Training Rooms)
**Location:** `plato-torch/`  
**Purpose:** 25 training room presets, self‑training rooms, tile ingestion.  
**Key files inspected:**
- `plato-torch/src/rooms/deadband_room.py` → Defines `DeadbandRoom` class.
- `plato-torch/src/rooms/__init__.py` → Exports `DeadbandRoom`, `GhostRoom`, `MirrorRoom`.
- `plato-torch/src/tile_ingestor.py` → Ingests `.tile` files, calls `room.ingest(tile)`.
- `plato-torch/requirements.txt` → Lists `torch`, `transformers`, `numpy`, `pydantic`. No direct refs to other fleet repos.

**Connections found:**
- **plato‑ensign:** `plato-torch/src/rooms/base_room.py` has abstract method `export_to_ensign()`. DeadbandRoom implements stub returning `None`. Intended hook, not wired.
- **fleet‑simulator:** No imports or config references.
- **holodeck‑rust:** No imports or config references.
- **External:** Uses `transformers` for LoRA, `torch` for training loops.

**Integration gap:** Rooms are isolated; no live data flow to simulator or holodeck.

## 7. Integration Gaps (P0 / P1 / P2)
**P0 (what NOT to do):**  
- Do not assume tiles flow automatically between simulator and plato‑torch.  
- Do not assume holodeck NPCs learn from room tiles without explicit wiring.  
- Do not assume plato‑ensign auto‑triggers; it’s a manual tool.

**P1 (safe channels):**  
- plato‑torch tile ingestor can read any `.tile` from filesystem.  
- fleet‑simulator outputs JSON tiles; format is compatible.  
- plato‑ensign’s `Exporter` can be called with any room object that implements the interface.  
- holodeck‑rust config can be extended with new NPC behaviors via plugins.

**P2 (optimize within channels):**  
- Write adapter that copies simulator tiles to plato‑torch’s tile directory.  
- Implement `export_to_ensign()` in DeadbandRoom to call plato‑ensign.  
- Add `GhostInjector` module to holodeck that reads ghost tiles from plato‑torch memory.  
- Create shared tile schema (Rust/Python) to ensure compatibility.
