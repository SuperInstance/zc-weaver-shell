# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 117

Based on reading the actual files in the fleet repos, here is the current state of integration between the four assigned technologies.

## 1. plato-torch (Training Rooms)
**Purpose:** Provides 25 preset training rooms and self-training rooms.  
**Key Files:** `presets/DeadbandRoom.py`, `presets/__init__.py`, `trainer.py`, `room_server.py`  
**Current Connections:**
- **plato-ensign:** Direct export path via `plato_torch.export.export_lora()`. The `DeadbandRoom` preset includes a `save_ensign()` method that calls this export function, creating a LoRA adapter from the room's accumulated experience.
- **fleet-simulator:** No direct code dependency found. Both are independent training data generators.
- **holodeck-rust:** No direct connection. plato-torch is Python/PyTorch; holodeck-rust is a Rust MUD server.

## 2. plato-ensign (LoRA Export)
**Purpose:** Exports room experience into LoRA adapters.  
**Key Files:** `export.py`, `lora_utils.py`, `configs/`  
**Current Connections:**
- **plato-torch:** Primary consumer. The `export_lora()` function is called from within plato-torch rooms.
- **fleet-simulator:** No connection found. Plato-ensign appears specific to plato-torch room exports.
- **holodeck-rust:** No connection. Different language and domain (sentiment-aware NPCs vs. model fine-tuning).

## 3. holodeck-rust (MUD Server)
**Purpose:** Multi‑user dungeon server with sentiment‑aware NPCs.  
**Key Files:** `src/npc/sentiment.rs`, `src/world/mod.rs`, `src/server.rs`  
**Current Connections:**
- **plato-torch:** None. Different runtime, different purpose (interactive simulation vs. training room).
- **plato-ensign:** None.
- **fleet-simulator:** Potential conceptual overlap (both generate interactive scenarios), but no shared code or data flow observed.

## 4. fleet-simulator (Fleet Simulation)
**Purpose:** Generates training data by simulating fleet interactions.  
**Key Files:** `simulator/core.py`, `agents/`, `scenarios/deadband_scenario.py`  
**Current Connections:**
- **plato-torch:** None. Both are data sources, but they operate independently.
- **plato-ensign:** None.
- **holodeck-rust:** None.

## Integration Gaps (P1 Safe Channels)
1. **holodeck‑rust → plato‑torch:** NPC interactions could become training tiles. A safe channel exists: implement a `TileExporter` in Rust that writes to the same tile format plato‑torch ingests.
2. **fleet‑simulator → plato‑torch:** Simulation outputs could feed training rooms. Safe channel: add a `SimulationTileLoader` in plato‑torch that reads fleet‑simulator’s output logs.
3. **GhostInjector integration:** The `GhostInjector` (from memory files) is not wired into holodeck‑rust. Safe channel: add a “ghost NPC” plugin that loads P0 lessons from ghost tiles.
4. **DeadbandRoom → plato‑relay:** The DeadbandRoom preset does not currently broadcast its state to the fleet’s messaging layer. Safe channel: add a WebSocket hook to `plato‑relay` in the room’s `step()` method.

## P0 Negative Space (What NOT to Do)
- Do not force a monolithic integration. Each component serves a distinct purpose.
- Do not assume shared memory or language interoperability without explicit adapters.
- Do not modify core training loops without validating tile‑format compatibility.
