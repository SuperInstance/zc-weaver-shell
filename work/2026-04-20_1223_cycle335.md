# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 335  
**Phase:** 4 — Build & Test  
**Status:** Direct file inspection of fleet repositories. Focus on actual imports, configuration references, and shared data structures.

## 1. plato-torch → fleet-simulator
**Connection:** **Strong**  
- `plato-torch/rooms/DeadbandRoom.py` imports `fleet_simulator` directly:  
  ```python
  from fleet_simulator import FleetSimulator, FleetState
  ```
- Uses `FleetSimulator` to generate training episodes (P0/P1/P2 trajectories).  
- `DeadbandRoom` passes simulator state to `plato-torch`'s training loop via `self.simulator.step()`.  
- Shared data structure: `FleetState` (contains agent positions, velocities, P0/P1/P2 maps).  

**Integration Gap:** None observed. The simulator is a direct dependency.

## 2. plato-torch → plato-ensign  
**Connection:** **Strong**  
- `plato-torch/training/trainer.py` calls `plato_ensign.export_lora()` after room training completes.  
- Exports a LoRA adapter from the room’s accumulated experience (tiles).  
- Configuration in `plato-torch/config/ensign_config.yaml` specifies export format (e.g., `delta_weights.pth`).  

**Integration Gap:** None observed. Export is a post-training step.

## 3. plato-torch → holodeck-rust  
**Connection:** **Weak / Indirect**  
- No direct Rust imports in Python code.  
- However, `plato-torch/rooms/HolodeckBridgeRoom.py` exists (stub).  
- Design intent: use HTTP or gRPC to connect to `holodeck-rust` MUD server.  
- Current status: `HolodeckBridgeRoom` is a skeleton; connection logic not implemented.  

**Integration Gap:** **Major**. No live bridge between Plato training rooms and holodeck’s sentiment-aware NPCs.

## 4. fleet-simulator → holodeck-rust  
**Connection:** **None**  
- `fleet-simulator` simulates physical fleet kinematics (P0/P1/P2).  
- `holodeck-rust` is a text‑based MUD with NPC emotion engines.  
- No shared data structures or API calls found.  

**Integration Gap:** **Complete**. These services operate in separate domains (kinematic vs. narrative).

## 5. plato-ensign → holodeck-rust  
**Connection:** **None**  
- `plato-ensign` exports LoRA adapters from Plato rooms.  
- `holodeck-rust` does not currently load LoRA weights.  
- No configuration or code references between them.  

**Integration Gap:** **Complete**. No mechanism to apply ensign adapters to holodeck NPCs.

## 6. fleet-simulator → plato-ensign  
**Connection:** **Indirect**  
- `fleet-simulator` generates training data for `plato-torch`.  
- `plato-torch` trains, then `plato-ensign` exports.  
- No direct link between simulator and ensign.  

**Integration Gap:** **Expected**. This is a pipeline: simulator → room → ensign.

## 7. GhostInjector (plato-torch) → holodeck-rust  
**Connection:** **None**  
- `plato-torch/ghost_injector.py` loads “ghost tiles” (P0 lessons from dead agents).  
- Injects them into training rooms as negative examples.  
- No reference to `holodeck-rust` in the injector code.  

**Integration Gap:** **Major**. Ghost tiles are not currently sent to holodeck NPCs.

## Integration Summary

### Connected (Working)
1. **plato-torch ←→ fleet-simulator** – Direct import, shared FleetState.  
2. **plato-torch → plato-ensign** – Direct export call after training.

### Not Connected (Gaps)
1. **plato-torch → holodeck-rust** – Bridge room is a stub.  
2. **fleet-simulator → holodeck-rust** – Separate domains.  
3. **plato-ensign → holodeck-rust** – No LoRA loading in holodeck.  
4. **GhostInjector → holodeck-rust** – Ghost tiles not used in MUD.
