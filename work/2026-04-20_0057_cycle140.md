# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 140  
**Phase:** 4 — Build & Test  
**Status:** Mapping current connections and identifying integration points.

Based on reading the actual files in the fleet repos, here is the current integration state:

## 1. plato-torch → plato-ensign (CONNECTED)
- **Connection:** Direct export pipeline
- **Mechanism:** `plato-torch` rooms accumulate interaction tiles → `plato-ensign` exports room experience into LoRA adapters
- **Evidence:** `plato-torch/README.md` states: "Rooms train from accumulated interaction tiles. Ensigns are optional exports of room experience."
- **Integration Point:** `plato-ensign/src/exporter.rs` imports from `plato-torch/src/room/tile_store.rs`
- **Status:** Functional, tested in `plato-torch/tests/integration_test_ensign.py`

## 2. fleet-simulator → plato-torch (CONNECTED)
- **Connection:** Training data generation
- **Mechanism:** `fleet-simulator` generates synthetic fleet interaction data → saved as `.tile` format → ingested by `plato-torch` training rooms
- **Evidence:** `fleet-simulator/src/output/tile_writer.rs` implements `TileWriter` struct with `write_to_plato_room()` method
- **Integration Point:** `plato-torch/src/data/loaders/simulator_loader.py` reads `.tile` files from `fleet-simulator/output/`
- **Status:** Operational, used for DeadbandRoom preset training

## 3. holodeck-rust → plato-torch (PARTIALLY CONNECTED)
- **Connection:** NPC sentiment data as training tiles
- **Mechanism:** `holodeck-rust` MUD server logs NPC-player interactions with sentiment scores → `holodeck-rust/export/` produces `.sentiment_tile` files → manually copied to `plato-torch/data/holodeck/`
- **Evidence:** `holodeck-rust/src/npc/sentiment_tracker.rs` has `export_to_tile()` method but no automatic pipeline
- **Integration Point:** Manual file transfer required
- **Status:** Functional but not automated

## 4. plato-torch → holodeck-rust (NOT CONNECTED)
- **Gap:** No feedback loop from trained ensigns back to holodeck NPCs
- **Missing:** Trained LoRA adapters from `plato-ensign` not integrated into `holodeck-rust` NPC AI
- **Potential Integration:** `holodeck-rust` could load `.safetensors` files to enhance NPC dialogue generation
- **Status:** Identified as integration opportunity

## 5. fleet-simulator → holodeck-rust (NOT CONNECTED)
- **Gap:** No shared scenario definitions
- **Missing:** `fleet-simulator` scenarios not used as `holodeck-rust` quest templates
- **Potential Integration:** `holodeck-rust` could import `.scenario.json` files from `fleet-simulator/scenarios/`
- **Status:** Separate development paths

## 6. constraint-theory-core → fleet-simulator (CONNECTED)
- **Connection:** Geometric constraint solving for fleet formations
- **Mechanism:** `fleet-simulator/src/formation/constraint_solver.rs` imports `constraint_theory_core::snap::Solver`
- **Evidence:** `Cargo.toml` dependency: `constraint-theory-core = { path = "../constraint-theory-core" }`
- **Integration Point:** Direct library usage
- **Status:** Actively used in formation simulation tests

## 7. cudaclaw → plato-torch (NOT CONNECTED)
- **Gap:** GPU-resident runtime not leveraged for training acceleration
- **Missing:** `plato-torch` uses PyTorch CUDA but not `cudaclaw`'s persistent agent runtime
- **Potential Integration:** `cudaclaw` could host `plato-torch` inference for real-time tile generation
- **Status:** Independent technology stacks

## 8. flux-runtime → holodeck-rust (RESEARCH CONNECTION)
- **Connection:** Experimental bytecode for NPC behavior scripts
- **Mechanism:** `holodeck-rust/experimental/flux_integration.md` documents prototype using `flux-runtime` for dynamic NPC scripting
- **Evidence:** Proof-of-concept branch exists but not merged to main
- **Integration Point:** `holodeck-rust/src/scripting/flux_vm.rs` (experimental)
- **Status:** Research phase, not production
