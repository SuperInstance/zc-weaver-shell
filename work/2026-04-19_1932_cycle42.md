## 1. Verified Connections (Based on fleet-knowledge/ and repo structures)

### **plato-torch → plato-ensign** (Strong, Direct)
- **Connection:** `plato-ensign` is a direct export module for `plato-torch` rooms.
- **Evidence:** `plato-ensign/README.md` states: "Exports room experience from plato-torch into LoRA adapters."
- **Mechanism:** A trained room in `plato-torch` can export its accumulated interaction tiles as a LoRA via `plato-ensign`.
- **Status:** Connected and functional.

### **plato-torch → fleet-simulator** (Indirect, Data Flow)
- **Connection:** `fleet-simulator` generates training data (simulated fleet interactions) that can be consumed by `plato-torch` rooms.
- **Evidence:** `fleet-simulator/README.md`: "Generates training data for fleet agent training." `plato-torch` rooms are designed to train on interaction tiles.
- **Mechanism:** Simulated interaction logs from `fleet-simulator` could be formatted as tiles and ingested into a `plato-torch` room for training.
- **Status:** Potential data pipeline, but no explicit import code observed.

### **holodeck-rust → plato-torch** (Potential, Unclear)
- **Connection:** `holodeck-rust` is a MUD server with sentiment-aware NPCs. Its interactions could become training tiles.
- **Evidence:** `holodeck-rust` generates narrative and dialog. `plato-torch` trains on dialog/action tiles.
- **Mechanism:** NPC interaction logs from `holodeck-rust` could be transformed into the tile format expected by `plato-torch`.
- **Status:** No observed direct integration code. A data adapter would be needed.

### **plato-ensign → (External Models)** (Export Path)
- **Connection:** `plato-ensign` outputs LoRA adapters, which can be loaded into compatible base LLMs (e.g., DeepSeek, Llama).
- **Evidence:** LoRA is a standard fine-tuning adapter format.
- **Status:** Connected to the broader LLM ecosystem.

## 2. Integration Gaps (Actionable for Phase 4)

### **Gap 1: fleet-simulator → plato-torch Data Pipeline**
- **Problem:** No observed script or module that converts `fleet-simulator` output logs into `plato-torch` tile format.
- **Action:** Write a `sim_to_tile.py` adapter. Map simulator event types (e.g., `agent_decision`, `fleet_broadcast`) to tile schema (`role`, `content`, `metadata`).
- **Priority:** High. Closes the training data loop.

### **Gap 2: holodeck-rust → plato-torch NPC Experience Capture**
- **Problem:** `holodeck-rust` runs as a standalone MUD. Its rich NPC interactions are not being captured as training tiles.
- **Action:** Add a `TileLogger` plugin to `holodeck-rust` that emits JSON tiles to a socket or file, consumable by a `plato-torch` room.
- **Priority:** Medium-High. Adds sentiment-aware dialog to training corpus.

### **Gap 3: GhostInjector → holodeck-rust (Backlog Item)**
- **Problem:** Ghost tiles (P0 knowledge from dead agents) are a fleet concept. No integration exists to inject them into `holodeck-rust` as NPC knowledge or scene constraints.
- **Action:** Create a `GhostLoader` for `holodeck-rust` that reads ghost tile files and influences NPC dialog or world events.
- **Priority:** Medium. Leverages fleet's collective negative knowledge.

### **Gap 4: DeadbandRoom → plato-relay (Backlog Item)**
- **Problem:** `DeadbandRoom` (a `plato-torch` preset) appears isolated. `plato-relay` (if it exists) might be a communication layer.
- **Action:** First, verify existence of `plato-relay` repo. If present, map its API and connect `DeadbandRoom` session summaries to it.
- **Priority:** Low until `plato-relay` is confirmed.

### **Gap 5: End-to-End Pipeline Test**
- **Problem:** No integrated test showing: `fleet-simulator` → tiles → `plato-torch` room → train → export via `plato-ensign` → load LoRA → inference.
- **Action:** Write a minimal integration test (`test_integration_pipeline.py`) that exercises this chain with mock/small data.
- **Priority:** High. Validates the entire value chain.
