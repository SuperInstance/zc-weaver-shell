## 1. plato-torch → fleet-simulator
**Connection:** **Yes** — Direct import and usage.  
**Evidence:**  
- `plato-torch/src/rooms/deadband_room.py` imports `from fleet_simulator.simulator import FleetSimulator`.  
- The `DeadbandRoom` class initializes a `FleetSimulator` instance (`self.simulator = FleetSimulator()`).  
- Room methods like `step()` call `self.simulator.step()`.  
- Shared data: The room passes agent actions to the simulator and receives updated environment states.  

**Integration Gap:** None observed. This is a working, wired connection.

## 2. plato-torch → holodeck-rust
**Connection:** **No** — No direct imports or API calls found.  
**Evidence:**  
- Searched `plato-torch/src/` for "holodeck". No imports, no references.  
- `holodeck-rust` is a separate Rust MUD server. No Python bindings or HTTP client visible in plato-torch code.  
- Plato-torch rooms appear to be pure Python training environments; holodeck-rust is a multiplayer game server.

**Integration Gap:** **Major**. These systems are not connected. Potential integration point: Could feed holodeck NPC sentiment/event logs into plato-torch as training tiles.

## 3. plato-torch → plato-ensign
**Connection:** **Yes** — Plato-ensign is a direct consumer of plato-torch output.  
**Evidence:**  
- `plato-ensign/src/tile_processor.py` imports `from plato_torch.rooms.base_room import BaseRoom`.  
- It processes "tiles" (room interaction logs) exported from plato-torch rooms.  
- The ensign pipeline loads room checkpoints and tile datasets to train LoRA adapters.  

**Integration Gap:** **Minor**. The connection is one-way (export). No feedback loop observed where ensign adapters are loaded back into live rooms.

## 4. fleet-simulator → holodeck-rust
**Connection:** **No**.  
**Evidence:**  
- `fleet-simulator/src/` contains no Rust code, no HTTP client, no references to holodeck.  
- It is a Python simulation of fleet dynamics (agents, tasks, P0/P1/P2 logic).  
- Holodeck-rust is a sentiment-aware MUD; no shared protocol.

**Integration Gap:** **Major**. These simulate different domains. Could connect by having fleet-simulator agents "enter" the holodeck as avatars, but no such bridge exists.

## 5. fleet-simulator → plato-ensign
**Connection:** **Indirect via plato-torch**.  
**Evidence:**  
- Plato-ensign processes tiles generated by plato-torch rooms.  
- Those rooms contain fleet-simulator instances, so simulator state is embedded in tile data.  
- No direct ensign dependency on fleet-simulator library.

**Integration Gap:** **Moderate**. Ensign trains on simulator-influenced tiles, but there's no direct hook for simulator events to trigger ensign training.

## 6. holodeck-rust → plato-ensign
**Connection:** **No**.  
**Evidence:**  
- Plato-ensign code does not mention holodeck.  
- Holodeck-rust outputs game logs (likely JSON), but no tile export pipeline seen.  
- No shared data format.

**Integration Gap:** **Major**. Holodeck's rich NPC interactions are not being captured as training tiles.
