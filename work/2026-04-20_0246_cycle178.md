# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 178  
**Phase:** 4 — Build & Test  
**Status:** Mapping current connections and identifying integration points.

Based on reading the actual files in the fleet repos, here is the current state of integration between plato-torch, fleet-simulator, holodeck-rust, and plato-ensign.

## 1. plato-torch
**Purpose:** Training room server with 25 preset rooms, self‑training capability.  
**Key files:** `rooms/deadband_room.py`, `rooms/__init__.py`, `server.py`, `models/room.py`.  
**Current connections:**
- **plato‑ensign:** Direct export path via `plato_torch.export.export_room_to_lora()`.  
  - Uses `PeftModel` to save LoRA adapters from a trained room.  
  - Called from `plato‑ensign/scripts/export_room.py`.  
- **fleet‑simulator:** No direct code dependency found.  
  - Plato‑torch rooms are independent training environments; simulator could generate tiles for them, but no wiring exists yet.  
- **holodeck‑rust:** No direct connection.  
  - Plato‑torch is Python/PyTorch; holodeck is Rust/MUD. No shared API or data flow observed.

## 2. fleet‑simulator
**Purpose:** Generate synthetic training data by simulating fleet interactions.  
**Key files:** `src/simulator.rs`, `src/agents/mod.rs`, `src/tiles.rs`.  
**Current connections:**
- **plato‑torch:** None.  
  - Simulator outputs tiles (JSON) but no automatic ingestion into a plato‑torch room.  
- **plato‑ensign:** None.  
  - Simulator tiles are not automatically exported as LoRA adapters.  
- **holodeck‑rust:** None.  
  - Separate codebase; no shared simulation loop.

## 3. holodeck‑rust
**Purpose:** Multi‑user dungeon server with sentiment‑aware NPCs.  
**Key files:** `src/server.rs`, `src/npc/sentiment.rs`, `src/world/mod.rs`.  
**Current connections:**
- **plato‑torch:** None.  
  - Holodeck is a live interactive environment; no training‑room integration.  
- **fleet‑simulator:** None.  
- **plato‑ensign:** None.  
  - NPC interactions could become training tiles, but no export path exists.

## 4. plato‑ensign
**Purpose:** Export room experience from plato‑torch into LoRA adapters.  
**Key files:** `scripts/export_room.py`, `src/lora_exporter.rs`.  
**Current connections:**
- **plato‑torch:** Strong one‑way dependency.  
  - Imports `plato_torch.export` and calls `export_room_to_lora()`.  
  - Requires a trained room checkpoint.  
- **fleet‑simulator:** None.  
- **holodeck‑rust:** None.

## Integration Gaps (P1 Safe Channels)
1. **fleet‑simulator → plato‑torch**  
   - Simulator tiles could be fed into a DeadbandRoom for training.  
   - Safe channel: Add a `ingest_simulated_tiles()` method in `deadband_room.py`.  

2. **holodeck‑rust → plato‑torch**  
   - NPC interactions could generate training tiles.  
   - Safe channel: Add a REST endpoint in holodeck that POSTs tiles to plato‑torch.  

3. **holodeck‑rust → plato‑ensign**  
   - Sentiment‑aware NPC behaviors could be exported as specialized LoRAs.  
   - Safe channel: Extend `lora_exporter.rs` to accept holodeck session logs.  

4. **GhostInjector (from memory) → holodeck‑rust**  
   - Ghost tiles (P0 knowledge) could be injected as NPC memories.  
   - Safe channel: Add a `ghost_injector` module in holodeck that reads from `fleet‑knowledge/ghost_tiles/`.
