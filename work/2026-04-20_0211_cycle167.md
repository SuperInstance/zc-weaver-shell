# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 167

**Status:** Mapping current connections and identifying integration points.

Based on reading the actual files in the fleet repos, here is a factual map of current integrations and identified gaps.

## 1. Verified Connections (What's Connected)

### plato-torch ↔ plato-ensign (Strong Integration)
- **Connection:** Direct export pipeline.
- **Evidence:** `plato-torch/src/rooms/deadband_room.py` contains `export_to_ensign()` method.
- **Mechanism:** Room training logs are serialized into `.pt` files, which `plato-ensign` consumes to produce LoRA adapters.
- **Data Flow:** Interaction tiles → room state → LoRA weights.

### fleet-simulator ↔ plato-torch (Data Supply)
- **Connection:** Training data generation.
- **Evidence:** `fleet-simulator/src/scenarios/` outputs JSON logs that match the tile schema expected by `plato-torch/rooms/base_room.py`.
- **Mechanism:** Simulator runs agent interactions, produces structured logs, which are ingested as training tiles.
- **Note:** This is a one-way data flow (simulator → plato-torch). No feedback loop observed.

### holodeck-rust ↔ (Planned, Not Fully Connected)
- **Connection:** Sentiment data intended for plato-torch.
- **Evidence:** `holodeck-rust/src/npc/sentiment.rs` defines a `SentimentEngine` that tracks NPC emotional state.
- **Gap:** No observed file or module in `plato-torch` that ingests this sentiment data. The `holodeck-rust` README mentions "sentiment-aware NPCs for training," but no import or adapter exists in the plato-torch codebase.

## 2. Integration Gaps (What's Not Connected)

### Gap 1: holodeck-rust → plato-torch (Sentiment Pipeline)
- **Missing:** Adapter to convert NPC sentiment logs into training tiles.
- **Impact:** Sentiment-aware interactions are not currently feeding the room training system.
- **Potential Integration Point:** Could write a `HolodeckTileAdapter` in `plato-torch/src/adapters/` to parse holodeck JSON and format it as tiles.

### Gap 2: plato-ensign → fleet-simulator (No Feedback Loop)
- **Missing:** Trained ensigns (LoRAs) are not loaded back into the simulator to influence agent behavior.
- **Impact:** Simulation remains static, not improved by trained models.
- **Potential Integration Point:** `fleet-simulator` could load `.safetensors` files from `plato-ensign/output/` and attach them to simulated agents.

### Gap 3: constraint-theory-core (Isolated)
- **Observation:** `constraint-theory-core` is a Rust library for geometric snapping. No imports found in plato-torch, holodeck-rust, or fleet-simulator.
- **Potential Use:** Could be integrated into `holodeck-rust` for spatial reasoning, or into `fleet-simulator` for environment constraints.
- **Current Status:** Standalone library, not wired.

### Gap 4: cudaclaw & flux-runtime (Infrastructure, Not Integrated)
- **Observation:** These are runtime systems. No direct imports in the higher-level repos.
- **Assumption:** They likely sit beneath the stack (e.g., `cudaclaw` may run `plato-torch` on GPU). Not an application-level integration gap.
