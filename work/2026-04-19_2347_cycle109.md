# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 109  
**Phase:** 4 — Build & Test  
**Status:** Mapping current connections and identifying integration points.

Based on reading the actual files in the fleet repos, here is the current state of integration between **plato-torch**, **fleet-simulator**, **holodeck-rust**, and **plato-ensign**.

---

## 1. plato-torch → fleet-simulator
**Connection:** **Direct (via shared data format)**  
- `plato-torch` includes a `fleet-simulator` subdirectory (`plato-torch/fleet-simulator/`).  
- This subdirectory contains simulation logic for generating training data (NPC interactions, agent decisions).  
- The simulation outputs **interaction tiles** in a structured JSON format that `plato-torch` rooms can ingest.  
- **Integration point:** `plato-torch/rooms/deadband_room.py` can load simulator-generated tiles via `TileLoader`.  

**Gap:**  
- No live API or socket connection between the two systems.  
- Tiles are generated offline, then loaded. Real-time simulation → training loop not wired.

---

## 2. plato-torch → holodeck-rust
**Connection:** **Indirect (conceptual, not wired)**  
- Both systems handle NPC interactions and sentiment.  
- `holodeck-rust` is a MUD server with sentiment-aware NPCs; `plato-torch` has training rooms that could benefit from live NPC interactions.  
- **Potential integration:** `holodeck-rust` could serve as a live data source for `plato-torch` rooms, streaming interaction tiles.  

**Gap:**  
- No code references or imports between the two repos.  
- No shared protocol or API calls.  
- `holodeck-rust` outputs are not currently formatted as plato-torch tiles.

---

## 3. plato-torch → plato-ensign
**Connection:** **Direct (export pipeline)**  
- `plato-ensign` is designed to export room experience from `plato-torch` into LoRA adapters.  
- `plato-torch/ensign/` contains the export logic (`export_lora.py`, `adapter_builder.py`).  
- After a room trains on accumulated tiles, `plato-ensign` can snapshot the model and produce a LoRA.  
- **Integration point:** `plato-torch/rooms/base_room.py` has hooks for `ensign.export()`.

**Gap:**  
- Export is manual/trigger-based, not automatic after each training cycle.  
- No continuous integration pipeline from room → ensign → deployment.

---

## 4. fleet-simulator → holodeck-rust
**Connection:** **None observed**  
- `fleet-simulator` simulates fleet agents; `holodeck-rust` is a standalone MUD.  
- No shared data structures, no imports, no references in code.  
- Both generate NPC/agent interactions, but formats differ.

**Gap:**  
- No connection. Could be aligned: simulator could drive holodeck NPCs, or holodeck could provide environment for simulated agents.

---

## 5. fleet-simulator → plato-ensign
**Connection:** **Indirect (via plato-torch)**  
- Simulator tiles train rooms in `plato-torch`.  
- Trained rooms can be exported via `plato-ensign`.  
- No direct simulator → ensign path.

**Gap:**  
- Simulator outputs are not directly convertible to ensign adapters without going through a plato-torch room.

---

## 6. holodeck-rust → plato-ensign
**Connection:** **None observed**  
- `holodeck-rust` does not produce model snapshots or LoRAs.  
- `plato-ensign` does not import holodeck data.

**Gap:**  
- Holodeck interactions could be valuable training data, but no pipeline exists to convert them to ensign adapters.

---
