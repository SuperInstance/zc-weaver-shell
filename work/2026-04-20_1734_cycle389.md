## 1. plato-torch → fleet-simulator
**Connection:** Strong, direct integration via `fleet_simulator` Python module.
- **Evidence:** `plato-torch/rooms/deadband_room.py` imports `from fleet_simulator import FleetSimulator`.
- **Usage:** `DeadbandRoom` instantiates `FleetSimulator` as `self.simulator`. Calls `self.simulator.step()` during room training cycles.
- **Data Flow:** Room passes agent action vectors to simulator; simulator returns next state, reward, done flag.
- **Integration Gap:** None observed — this is a working, coupled integration.

## 2. plato-torch → holodeck-rust
**Connection:** Indirect, via shared protocol (likely WebSocket/JSON) but no direct Python→Rust imports.
- **Evidence:** `plato-torch/configs/holodeck_config.yaml` exists, defining `holodeck_host`, `holodeck_port`, `npc_batch_size`.
- **Usage:** `DeadbandRoom` has `self.holodeck_client` attribute, suggests a client class (`HolodeckClient`) planned or implemented elsewhere.
- **Data Flow:** Config suggests room sends NPC sentiment batches to holodeck; holodeck returns NPC responses.
- **Integration Gap:** No `HolodeckClient` class found in plato-torch. Likely missing implementation — a stub or TODO.

## 3. plato-torch → plato-ensign
**Connection:** Export pipeline, one‑way.
- **Evidence:** `plato-torch/export/ensign_exporter.py` imports `from plato_ensign import LoRAExporter`.
- **Usage:** After room training, `export_room_to_ensign()` calls `LoRAExporter.save_adapter()` with room’s accumulated gradient tiles.
- **Data Flow:** Room tiles → LoRAExporter → output `.safetensors` file.
- **Integration Gap:** None — exporter is implemented and appears functional.
