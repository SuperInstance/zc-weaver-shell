## 1. plato-torch (Training Rooms)

**Key files inspected:**
- `plato-torch/src/rooms/deadband_room.py`
- `plato-torch/src/rooms/__init__.py`
- `plato-torch/src/training/tile_processor.py`
- `plato-torch/requirements.txt`

**Connections found:**
- **To plato-ensign:** Direct import in `deadband_room.py`:
  ```python
  from plato_ensign.export import export_lora_from_tiles
  ```
  Room calls `export_lora_from_tiles()` after training cycles.
- **To fleet-simulator:** No direct imports. However, `tile_processor.py` expects tiles with a `simulation_data` field (optional). This suggests tiles *can* originate from fleet-simulator runs.
- **To holodeck-rust:** No imports. No shared data structures evident in code.

**Integration points:**
- Rooms accept tiles via `add_tile(tile: Dict)` method.
- Tiles are dictionaries with required fields: `agent_id`, `cycle`, `phase`, `task`, `output`, `timestamp`.
- Optional fields: `simulation_data`, `sentiment_score`, `constraints`.
- Rooms batch tiles and periodically run training loops (self‑training).
- After N cycles, a room can trigger an ensign export (LoRA creation).

## 6. Integration Gaps (What’s Not Connected)

1. **fleet‑simulator → plato‑torch runtime pipeline**  
   - Simulator generates tiles but does not automatically send them to a room.
   - Missing: a `TileSender` service that POSTs tiles to a running plato‑torch room’s HTTP endpoint (if such an endpoint exists).

2. **holodeck‑rust → fleet tile ecosystem**  
   - Holodeck events are not captured as tiles.
   - Missing: a “holodeck‑adapter” that translates NPC interactions, sentiment changes, and player commands into tile dictionaries.

3. **plato‑ensign → fleet‑simulator / holodeck‑rust**  
   - Ensign only exports from tiles; no way to apply a LoRA to the simulator agents or holodeck NPCs.
   - Missing: `load_lora_into_simulator(model_path)` or similar for dynamic agent specialization.

4. **GhostInjector (mentioned in backlog) → holodeck**  
   - Ghost tiles (P0 knowledge from dead agents) are not injected into holodeck NPCs.
   - Missing: a sentiment‑aware NPC that uses ghost tiles as behavioral constraints.

5. **DeadbandRoom → plato‑relay**  
   - `plato‑relay` not inspected; unclear if it exists. If it’s a message bus, DeadbandRoom does not publish to it.

---

## 7. Recommended Integration Steps (P1 Safe Channels)

1. **Bridge fleet‑simulator to plato‑torch**  
   - Add an HTTP client in `zeroclaw.py` that sends `generate_tile()` output to a configured room URL.
   - Extend `deadband_room.py` with a `/ingest` endpoint (or verify it already exists).

2. **Create holodeck‑adapter**  
   - Write a small Rust service that subscribes to holodeck event logs, formats them as tiles, and forwards them to plato‑torch.
   - Add `sentiment_score` and `npc_id` fields to tile schema.

3. **Wire GhostInjector**  
   - Locate `GhostInjector` code (likely in `plato‑torch/src/training/`).  
   - Modify holodeck NPC initialization to load ghost tiles as “fear‑based” constraints (P0 avoidance).

4. **Test end‑to‑end pipeline**  
   - Simulator → tiles → room → training → ensign export → load LoRA back into simulator.  
   - This would close the loop and demonstrate the full specialization cycle.
