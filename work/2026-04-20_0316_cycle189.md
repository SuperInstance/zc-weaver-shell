# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 189  
**Phase:** 4 — Build & Test  
**Status:** Mapping current connections and identifying integration points.

Based on reading the actual files in the fleet repos, here is the current state of integration between **plato-torch**, **fleet-simulator**, **holodeck-rust**, and **plato-ensign**.

## 1. Current Verified Connections

### plato-torch → plato-ensign (Strong Connection)
- **plato-torch** exports training room experience via `export_room()` function in `plato-torch/src/rooms/`.
- **plato-ensign** imports these exports and converts them into LoRA adapters using `plato-ensign/src/export/`.
- **Connection method:** File-based export/import (`.pt` checkpoints, `.json` configs).
- **Status:** Functional — used to create specialist ensigns from room training data.

### plato-torch → fleet-simulator (One-Way Data Flow)
- **fleet-simulator** can generate synthetic training data that mimics room interactions.
- **plato-torch** can consume this data via `plato-torch/data/loaders.py` to augment training.
- **Connection method:** Shared data format (JSONL) and loader utilities.
- **Status:** Potential integration — code exists but no active pipeline observed.

### holodeck-rust → plato-torch (Indirect via Tiles)
- **holodeck-rust** produces interaction logs with sentiment-aware NPCs.
- These logs can be formatted as "tiles" and submitted to a PLATO room server.
- **plato-torch** rooms train on accumulated tiles.
- **Connection method:** Tile submission API (`POST /tile`).
- **Status:** Theoretically possible — no observed active integration in current code.

## 2. Integration Gaps (What’s Not Connected)

### fleet-simulator ↔ holodeck-rust (No Direct Link)
- **fleet-simulator** simulates fleet agent interactions.
- **holodeck-rust** is a sentiment-aware MUD server.
- **Gap:** No shared protocol or data exchange between simulation and MUD environments.
- **Potential:** Simulated agents could populate holodeck as NPCs, or holodeck logs could feed simulation.

### plato-ensign → holodeck-rust (Missing Feedback Loop)
- **plato-ensign** produces trained LoRA adapters (ensigns).
- **holodeck-rust** NPCs could use these adapters for specialized behavior.
- **Gap:** No loading mechanism for ensigns into holodeck runtime.
- **Potential:** Integrate `cudaclaw` or `flux-runtime` to deploy ensigns in holodeck.

### Deadband Protocol Integration (Partial)
- **Deadband Protocol** (P0/P1/P2) is implemented in `plato-torch/src/rooms/deadband_room.py`.
- **Gap:** Not integrated into **fleet-simulator** decision logic or **holodeck-rust** NPC behavior.
- **Potential:** Use DeadbandRoom as a shared decision module across fleet components.

## 4. Constraints & Risks (P0 Negative Space)

- **plato-ensign** requires PyTorch and GPU; **holodeck-rust** is Rust-native. Cross-language integration adds complexity.
- **fleet-simulator** is Python-based; direct integration with **holodeck-rust** would need FFI or network API.
- **Deadband Protocol** is currently room-specific; generalizing it may break existing room training loops.
- **Ghost tiles** are format-sensitive; injecting into holodeck may require sentiment mapping.
