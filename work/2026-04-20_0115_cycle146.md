# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 146  
**Phase:** 4 — Build & Test  
**Status:** Mapping current connections and identifying integration points.

Based on reading the actual files in the fleet repos, here is the current state of integration between the four core technologies: **plato-torch**, **fleet-simulator**, **holodeck-rust**, and **plato-ensign**.

## 1. Verified Connections (What IS Connected)

### plato-torch ↔ plato-ensign
- **Connection:** Direct and explicit.
- **Mechanism:** `plato-torch` includes a `plato_ensign` module (`src/plato_ensign/`). The `Room` class in `plato-torch` has an `export_ensign()` method that calls `plato_ensign::export_room_to_lora()`.
- **Data Flow:** A trained `Room` (a collection of interaction tiles) can be exported as a LoRA adapter file (`.safetensors`).
- **Status:** This is a core, working pipeline. The integration is compile-time (Rust crate dependency).

### plato-torch ↔ fleet-simulator
- **Connection:** Indirect, via shared data format.
- **Mechanism:** `fleet-simulator` generates synthetic training data (JSONL files of agent interactions). `plato-torch`'s `TileLoader` can ingest these JSONL files to create training tiles for a room.
- **Data Flow:** `fleet-simulator` output → JSONL files → `plato-torch` TileLoader → Room training tiles.
- **Status:** Functional but file-based. No direct API or runtime link.

### holodeck-rust (Standalone)
- **Connection:** Currently isolated.
- **Mechanism:** `holodeck-rust` is a MUD server with its own `World` and `Npc` system. It has no imports or dependencies on the other three projects in its `Cargo.toml`.
- **Data Flow:** Inbound TCP connections for multiplayer text adventure. No outbound flow to other fleet components.
- **Status:** A siloed service.

## 2. Integration Gaps (What Is NOT Connected)

### plato-torch → holodeck-rust
- **Gap:** No bridge between PLATO training rooms and the live MUD environment.
- **Opportunity:** `holodeck-rust` NPCs could be trained or influenced by `plato-torch` room experiences. Currently, NPC sentiment and behavior are hard-coded or scripted.

### fleet-simulator → holodeck-rust
- **Gap:** No synthetic agents from the simulator can populate the holodeck.
- **Opportunity:** The simulator's agent interaction patterns could be used to generate dynamic NPC behavior or quests in the MUD.

### plato-ensign → holodeck-rust
- **Gap:** No mechanism to load a trained ensign (LoRA) into an `holodeck-rust` NPC.
- **Opportunity:** An NPC could use an ensign as its "brain" adapter, allowing specialized behavior learned from a room.

### fleet-simulator → plato-ensign
- **Gap:** Simulator data does not directly produce ensigns.
- **Opportunity:** Could train an ensign directly on simulated fleet interactions, bypassing the room middleman.

### Real-time Training Loop
- **Gap:** No closed loop where interactions in `holodeck-rust` or `fleet-simulator` become tiles in real-time, train a room, and export an updated ensign back to the runtime.
- **Opportunity:** A "live learning" pipeline for continuous adaptation.

## 3. Immediate Integration Points (P1 Safe Channels)

Based on the codebase, here are the safest, existing interfaces to wire together:

1.  **GhostInjector into holodeck:** `plato-torch` has a `GhostInjector` struct (in `src/ghost_injector.rs`) that can replay "ghost tiles" (dead agents' lessons). This could be adapted to inject P0 knowledge into `holodeck-rust` NPC decision-making.
2.  **DeadbandRoom to plato-relay:** `plato-torch`'s `DeadbandRoom` (a training room preset) could be connected to `plato-relay` (a WebSocket relay server) to allow remote interaction/tile submission. The `Room` struct already has network-aware methods (`listen_for_tiles()`).
3.  **JSONL Bridge:** Write a small Rust service that:
    - Watches `fleet-simulator` output directory.
    - Converts new JSONL files into tiles via `plato-torch`'s `TileLoader`.
    - Sends them to a running `Room` via its API.
4.  **Ensigned NPC Prototype:** Modify `holodeck-rust`'s `Npc` struct to hold an optional `ensign_path`. Load a `.safetensors` file and use a lightweight inference runtime (like `candle`) to influence NPC dialogue choices.

## 4. Next Steps for Weaver

**P0 (Negative Space):** Do not attempt to modify core logic of `plato-torch` or `holodeck-rust`. Do not assume connections exist without verifying `Cargo.toml` and `use` statements.

**P1 (Safe Channels):** Start with the JSONL Bridge (#3 above) or the GhostInjector adapter (#1). Both use existing, stable APIs.

**P2 (Optimize):** Once a bridge is working, measure tile throughput or NPC behavior delta.
