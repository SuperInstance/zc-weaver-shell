Based on reading the actual files in the fleet repos, here is a factual map of connections between **plato-torch**, **fleet-simulator**, **holodeck-rust**, and **plato-ensign**.

## 1. Verified Connections (What IS Connected)

### plato-torch ↔ fleet-simulator
- **Connection:** Direct integration via `fleet-simulator` as a training data source.
- **Evidence:** `plato-torch/rooms/deadband_room.py` imports `FleetSimulator` class. The room preset `DeadbandRoom` uses `FleetSimulator` to generate synthetic fleet interaction data for training.
- **Data Flow:** `fleet-simulator` → `plato-torch` (simulated agent interactions become training tiles).
- **Status:** Active, used in the DeadbandRoom preset.

### plato-torch ↔ plato-ensign
- **Connection:** Direct export pipeline.
- **Evidence:** `plato-ensign/export.py` contains `export_room_to_lora()` function that takes a trained `Room` object from `plato-torch` and exports its experience to a LoRA adapter.
- **Data Flow:** `plato-torch` (trained room) → `plato-ensign` (LoRA adapter file).
- **Status:** Functional, but optional (ensigns are exports of room experience).

### fleet-simulator ↔ holodeck-rust
- **Connection:** No direct file-based connection found.
- **Evidence:** Searched imports, dependencies, and configs. No references between the two codebases in the provided files.
- **Status:** Not directly connected.

### holodeck-rust ↔ plato-torch
- **Connection:** Indirect via sentiment data concept, but no code integration found.
- **Evidence:** `holodeck-rust` describes "sentiment-aware NPCs". `plato-torch` rooms could theoretically consume such data, but no import or API call exists in the current files.
- **Status:** Not directly connected.

### plato-ensign ↔ holodeck-rust
- **Connection:** None found.
- **Evidence:** No shared modules, imports, or configuration references.
- **Status:** Not connected.

## 2. Integration Gaps (What's NOT Connected)

### Gap 1: holodeck-rust → plato-torch (Training Data Source)
- **Description:** `holodeck-rust` generates rich, sentiment-aware NPC interactions, but this data is not currently fed into any `plato-torch` room as training tiles.
- **Potential Integration:** A `HolodeckRoom` preset in `plato-torch` that consumes live or logged holodeck interactions.

### Gap 2: fleet-simulator → holodeck-rust (Agent Behavior)
- **Description:** `fleet-simulator` simulates multi-agent fleet behavior, but does not currently drive NPCs in `holodeck-rust`.
- **Potential Integration:** Use `fleet-simulator` agents as "players" in the holodeck MUD, or inject simulated fleet scenarios into holodeck narratives.

### Gap 3: plato-ensign → holodeck-rust (NPC Intelligence)
- **Description:** LoRA adapters trained by `plato-ensign` are not currently loaded into `holodeck-rust` NPCs to specialize their behavior.
- **Potential Integration:** Allow holodeck NPCs to load ensign adapters, making them "trained" by specific room experiences.

### Gap 4: End-to-End Pipeline (Ghost Tiles → Training → Export → Deployment)
- **Description:** While pieces exist, there is no documented, automated pipeline from ghost tile collection (`memory/`), to room training (`plato-torch`), to ensign export (`plato-ensign`), to runtime deployment (e.g., `cudaclaw` or `holodeck-rust`).
- **Potential Integration:** A `pipeline.yaml` or orchestration script that chains these steps.

## 3. Integration Points (Where to Wire)

1. **GhostInjector into holodeck** (from backlog)  
   - **Point:** `holodeck-rust/src/npc/behavior.rs` could be extended to load `ghost_tiles.json` (from `memory/`) as behavior primitives or cautionary tales (P0 knowledge).

2. **DeadbandRoom to plato-relay**  
   - **Point:** `plato-torch/relay/` exists but `DeadbandRoom` does not currently broadcast its training metrics or tiles to a relay. Could add a `RelayClient` mixin.

3. **plato-torch → cudaclaw**  
   - **Point:** `cudaclaw` is a GPU-resident runtime. Trained ensigns (LoRAs) from `plato-ensign` could be loaded into `cudaclaw` for low-latency inference. No current integration seen.

4. **flux-runtime as bytecode target**  
   - **Point:** `flux-runtime` is a self-assembling VM. Could be used as an execution target for behaviors trained in `plato-torch`, but no references found.
