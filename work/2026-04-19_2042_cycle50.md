## 1. Verified Current Connections

Based on reading the actual fleet repos (plato-torch, fleet-simulator, holodeck-rust, plato-ensign), here are the **verified** connections:

### **plato-torch ↔ fleet-simulator**
- **Connection:** Strong, bidirectional data flow.
- **Evidence:** `plato-torch/src/rooms/deadband_room.py` imports `fleet_simulator` and uses `SimulationEnvironment` to generate training episodes.
- **Data Flow:** Fleet simulator produces agent interaction logs → DeadbandRoom ingests them as training tiles.
- **Purpose:** Simulated fleet data trains the Deadband Protocol (P0/P1/P2) in a Plato room.

### **plato-torch ↔ plato-ensign**
- **Connection:** Direct export pipeline.
- **Evidence:** `plato-ensign/src/exporter.py` imports `Room` from `plato-torch`. It reads room state and tile history, then exports to LoRA adapters.
- **Data Flow:** Room experience (tiles) → plato-ensign → LoRA weights.
- **Purpose:** Specialized ensign models are created from room training.

### **fleet-simulator ↔ holodeck-rust**
- **Connection:** **None verified.** No imports or shared data structures found.
- **Gap:** Fleet simulator generates synthetic agent logs; holodeck-rust is a sentiment-aware MUD. These could connect (simulated agents → NPC personalities) but currently don’t.

### **holodeck-rust ↔ plato-torch**
- **Connection:** **Indirect, potential.**  
- **Evidence:** `holodeck-rust/src/npc/sentiment.rs` defines emotion states that could be training targets for a Plato room, but no direct import of plato-torch exists.
- **Gap:** NPC sentiment logs are not currently fed into any Plato room as tiles.

### **plato-ensign ↔ fleet-simulator**
- **Connection:** **None verified.** Plato-ensign exports room experience, not simulator data.
- **Gap:** Ensigns are trained from room tiles, not raw simulator logs. Could be a missed opportunity for “simulation-to-ensign” pipeline.

## 2. Integration Gaps (Actionable for Phase 4)

### **Gap 1: GhostInjector → holodeck-rust**
- **Problem:** Ghost tiles (dead agents’ P0 lessons) exist in `plato-torch/src/ghost_injector.py` but are not injected into holodeck NPC memory.
- **Action:** Wire GhostInjector to holodeck’s NPC memory system. This would let NPCs “remember” past agent failures (P0 knowledge).
- **Files:** `ghost_injector.py` → `holodeck-rust/src/npc/memory.rs`

### **Gap 2: DeadbandRoom → plato-relay**
- **Problem:** DeadbandRoom trains on simulator data, but its trained P0/P1/P2 decisions are not relayed back to the fleet in real-time.
- **Action:** Connect DeadbandRoom to `plato-relay` (if exists) or a message bus. Let room output influence live fleet decisions.
- **Files:** `deadband_room.py` → (new relay adapter)

### **Gap 3: holodeck-rust sentiment → plato-torch tiles**
- **Problem:** NPC sentiment logs are rich training data but not captured as tiles.
- **Action:** Add a sentiment-tile generator in holodeck that writes to a Plato-compatible tile format.
- **Files:** `holodeck-rust/src/npc/sentiment.rs` → new `sentiment_tile_writer.rs`

### **Gap 4: fleet-simulator → plato-ensign direct export**
- **Problem:** Ensigns only train from room tiles, missing raw simulator patterns.
- **Action:** Create a “simulation ensign” exporter that converts simulator logs directly to LoRA, bypassing room training for speed.
- **Files:** `fleet_simulator/log_writer.py` → new `plato-ensign/src/sim_exporter.py`
