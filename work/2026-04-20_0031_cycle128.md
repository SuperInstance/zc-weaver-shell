# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 128  
**Phase:** 4 — Build & Test  
**Status:** Mapping current connections and identifying integration points.

Based on reading the actual files in the fleet repos, here is the current state of integration between **plato-torch**, **fleet-simulator**, **holodeck-rust**, and **plato-ensign**.

---

## 1. plato-torch → fleet-simulator
**Connection:** **Strong, bidirectional**  
- **plato-torch** includes `fleet_simulator` as a dependency in its `Cargo.toml`.  
- **fleet-simulator** is used as a library within plato-torch to generate synthetic training data for rooms.  
- The `DeadbandRoom` preset in plato-torch calls `fleet_simulator::generate_scenario()` to create training episodes.  
- **Integration point:** `plato-torch/src/rooms/deadband_room.rs` imports and uses `fleet_simulator`.  

**Gap:** None observed — this is a working, direct dependency.

---

## 2. plato-torch → holodeck-rust  
**Connection:** **Weak, indirect**  
- **holodeck-rust** is not listed as a dependency in plato-torch’s `Cargo.toml`.  
- However, **plato-torch** has a `holodeck_adapter` module (`src/adapters/holodeck_adapter.rs`) that defines a trait for sending interaction tiles to a MUD server.  
- The adapter is not wired to a live holodeck instance in the current code — it’s a stub awaiting integration.  
- **Integration point:** The `HolodeckAdapter` trait exists but is not invoked by any room in the main branch.

**Gap:** No runtime connection. The adapter is a placeholder.

---

## 3. plato-torch → plato-ensign  
**Connection:** **Strong, export-oriented**  
- **plato-ensign** is built as a separate binary that reads trained room checkpoints from plato-torch and exports them as LoRA adapters.  
- **plato-torch** saves room state (model weights, interaction logs) to disk in a format plato-ensign can read.  
- The integration is file-based: plato-torch writes, plato-ensign reads.  
- **Integration point:** `plato-ensign/src/main.rs` loads `room.safetensors` and `tiles.json` from plato-torch’s output directory.

**Gap:** None — this is a designed, decoupled export pipeline.

---

## 4. fleet-simulator → holodeck-rust  
**Connection:** **None observed**  
- **fleet-simulator** generates synthetic agent interaction data, but does not output to holodeck-rust’s MUD format.  
- **holodeck-rust** reads its own scenario files (`.yaml`) and has no import path for fleet-simulator’s output.  
- **Potential integration:** Could feed simulated agent dialogues into holodeck as NPC scripts.

**Gap:** No code or data flow between them.

---

## 5. holodeck-rust → plato-ensign  
**Connection:** **None observed**  
- **holodeck-rust** produces interaction logs, but they are not in the tile format expected by plato-ensign.  
- **plato-ensign** is specifically designed to read plato-torch room outputs, not holodeck logs.  
- **Potential integration:** A transformer could convert holodeck sentiment logs into training tiles.

**Gap:** No direct connection.

---

## 6. fleet-simulator → plato-ensign  
**Connection:** **Indirect via plato-torch**  
- **fleet-simulator** data is used by plato-torch rooms to generate training tiles.  
- Those tiles are then exported by plato-ensign.  
- No direct link between fleet-simulator and plato-ensign.

**Gap:** Not needed — the flow is through plato-torch.

---
