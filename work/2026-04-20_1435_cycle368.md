# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 368

**Status:** Direct file inspection of fleet repositories. Focus on actual imports, configuration references, and shared data structures.

## 1. plato-torch → fleet-simulator
**Connection:** Strong, bidirectional integration via shared data format and explicit imports.

**Evidence:**
- `plato-torch/rooms/deadband_room.py` imports `from fleet_simulator.simulator import FleetSimulator` (line 12).
- `fleet-simulator/src/simulator.rs` exports a `FleetSimulator` struct with a `generate_training_data()` method that returns `Vec<TrainingTile>`.
- `plato-torch/tests/test_deadband_room.py` shows the room ingesting `TrainingTile` objects from the simulator to populate its interaction buffer.

**Data Flow:**
1. `FleetSimulator` runs a scenario, produces `Vec<TrainingTile>` (Rust).
2. Python bindings (likely via `pyo3`) expose this to `DeadbandRoom`.
3. Room adds tiles to its buffer for training.

**Gap:** The `pyo3` binding layer is not visible in the inspected files; assumed present but not verified.

## 2. plato-torch → plato-ensign
**Connection:** Direct export pipeline.

**Evidence:**
- `plato-ensign/src/lib.rs` defines `export_lora_adapter(buffer: Vec<TrainingTile>, config: ExportConfig)`.
- `plato-torch/rooms/base_room.py` has a `export_to_ensign()` method that calls the ensign export function via FFI.
- Configuration alignment: both use `ExportConfig` (defined in `plato-ensign/src/config.rs`) specifying rank, alpha, dropout.

**Data Flow:**
1. Room accumulates tiles.
2. When triggered, room calls `export_to_ensign()`.
3. Tiles are serialized and passed to `plato-ensign` to produce a LoRA adapter file.

**Gap:** The FFI interface (C ABI or `ctypes`) is not shown in the Python files; only the method signature is present.

## 3. holodeck-rust → fleet-simulator
**Connection:** Indirect, via shared sentiment/training tile concepts, but no direct imports.

**Evidence:**
- `holodeck-rust/src/npc.rs` defines `SentimentAwareNPC` with `interact()` returning `Interaction`.
- `fleet-simulator/src/events.rs` defines `InteractionEvent` which could map to `Interaction`.
- No `use fleet_simulator;` in holodeck source files.
- Both generate data that could become `TrainingTile` but holodeck does not call the simulator.

**Gap:** No direct pipeline from holodeck NPC interactions to simulator training data. They are separate data sources.

## 4. holodeck-rust → plato-torch
**Connection:** None observed.

**Evidence:**
- `holodeck-rust` is a standalone MUD server (`src/main.rs` starts a TCP server).
- `plato-torch` has no imports of `holodeck` or its types.
- No shared configuration files.

**Gap:** Holodeck interactions are not feeding into plato-torch rooms. This is a missed integration point.

## 5. plato-ensign → Others
**Connection:** Passive library; called by plato-torch, not calling others.

**Evidence:**
- `plato-ensign` has no dependencies on `fleet-simulator` or `holodeck-rust` in its `Cargo.toml`.
- It only defines export logic and config.

## 6. Ghost Tiles & Mirror Play
**Connection:** Conceptual, not yet implemented.

**Evidence:**
- `fleet-knowledge/DEADBAND-PROTOCOL.md` describes ghost tiles and mirror play.
- No source files reference `GhostTile` or `MirrorPlay` as concrete types.
- `plato-torch/rooms/ghost_injector.py` exists but is empty (stub).

**Gap:** Ghost tile injection and mirror play are documented concepts without current integration.
