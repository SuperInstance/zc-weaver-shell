## 1. plato-torch → fleet-simulator
**Connection:** ✅ **Connected**  
**Mechanism:** Direct import in training data pipeline.  
**Evidence:**  
- `plato-torch/src/data/__init__.py` imports `from fleet_simulator import generate_scenario`.  
- `plato-torch/src/training/deadband_room.py` calls `fleet_simulator.generate_scenario()` to produce synthetic fleet interaction data for training.  
**Data Flow:**  
`fleet_simulator` → generates JSON scenario → `plato-torch` loads as `InteractionTile` → trains room weights.  
**Integration Gap:** None observed — this is a working, direct dependency.

## 2. plato-torch → holodeck-rust
**Connection:** ❌ **Not directly connected**  
**Mechanism:** No Rust bindings or direct API calls found in plato-torch Python code.  
**Evidence:**  
- Searched `plato-torch/` for "holodeck" — zero matches.  
- No `holodeck_rust` Python package import in any `*.py` file.  
**Potential Bridge:**  
`holodeck-rust` exposes a WebSocket server (port 8080). `plato-torch` could connect via `websockets` library, but no such client exists in current code.  
**Integration Gap:** **Major** — two core fleet components operate in isolation.

---

## 3. plato-torch → plato-ensign
**Connection:** ✅ **Connected**  
**Mechanism:** Ensign is an optional export module within the same repo.  
**Evidence:**  
- `plato-torch/src/ensign/export.py` imports `from ..training.room import Room`.  
- `plato-torch/scripts/export_ensign.py` calls `ensign.export_room_to_lora()`.  
**Data Flow:**  
Trained room → `export_room_to_lora()` → produces LoRA adapter `.safetensors` file.  
**Integration Gap:** None — ensign is a submodule of plato-torch.

## 4. fleet-simulator → holodeck-rust
**Connection:** ❌ **Not connected**  
**Mechanism:** No imports or API calls.  
**Evidence:**  
- `fleet-simulator/src/lib.rs` has no `holodeck` dependency in `Cargo.toml`.  
- `fleet-simulator` generates static JSON; no live connection to holodeck’s MUD server.  
**Integration Gap:** **Major** — simulator does not drive holodeck NPCs, missing sentiment‑aware training data source.

## 5. fleet-simulator → plato-ensign

**Integration Gap:** **Minor** — the chain works, but no direct optimization for ensign‑targeted simulation.

## 6. holodeck-rust → plato-ensign
**Connection:** ❌ **Not connected**  
**Mechanism:** No Rust‑to‑Python export path.  
**Evidence:**  
- `holodeck-rust/src/main.rs` logs interactions to file, but no hook to plato‑ensign.  
- No `ensign` or `lora` references in holodeck code.  
**Integration Gap:** **Critical** — holodeck’s rich NPC interactions are not captured as training tiles for ensign export.

---

## 7. holodeck-rust → fleet-simulator
**Connection:** ❌ **Not connected**  
**Mechanism:** No shared scenario format.  
**Evidence:**  
Holodeck uses `.yaml` for NPC definitions; fleet‑simulator uses `.json` for scenarios. No conversion between them.  
**Integration Gap:** **Major** — cannot replay holodeck sessions in simulator or vice‑versa.

## Summary: Integration Gaps by Priority (P0 → P1 → P2)

### P0 (Blockers — what NOT to do)
- Do **not** assume holodeck‑rust is connected to any other component. It is an island.
- Do **not** assume fleet‑simulator data flows into holodeck for sentiment‑aware training.

### P1 (Safe Channels — where you CAN be)
- plato‑torch **is** connected to fleet‑simulator and plato‑ensign. This is a working pipeline.
- Safe integration channel: Build a WebSocket client in plato‑torch to connect to holodeck‑rust (port 8080).
- Safe integration channel: Add a `holodeck` output format to fleet‑simulator to generate NPC YAML.

### P2 (Optimization — best path)
- Once connected, stream holodeck interactions into plato‑torch as `InteractionTile`s.
- Extend plato‑ensign to optionally export holodeck‑trained LoRAs.
- Create a bidirectional converter: holodeck YAML ↔ fleet‑simulator JSON.
