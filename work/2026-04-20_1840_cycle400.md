# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 400  
**Phase:** 4 — Build & Test  
**Status:** Direct file inspection of fleet repositories. Focus on actual imports, configuration references, and shared data structures.  

---

## 1. plato-torch → fleet-simulator  
**Connection:** **Direct import**  
- File: `plato-torch/src/training/rooms/deadband_room.py`  
- Line: `from fleet_simulator.agents.base_agent import BaseAgent`  
- Usage: DeadbandRoom instantiates BaseAgent for simulation-based training.  
- Data flow: Room config passes agent parameters to simulator; simulator returns interaction logs.  
- **Integration gap:** No shared configuration system. Room configs are separate from simulator YAML.  

---

## 2. plato-torch → holodeck-rust  
**Connection:** **Indirect via data schema**  
- File: `plato-torch/src/data/tile_schema.py`  
- Schema: `Tile` includes `environment: str` and `sentiment: float`.  
- Holodeck’s `npc_sentiment_system.rs` produces NPC dialogues with sentiment scores.  
- **Integration gap:** No direct API call or import. Tiles are manually aligned by field names.  

---

## 3. plato-torch → plato-ensign  
**Connection:** **Export pipeline**  
- File: `plato-torch/src/export/ensign_exporter.py`  
- Function: `export_room_to_lora(room_id, adapter_path)` reads room’s accumulated tiles, trains LoRA.  
- Trigger: Manual CLI command `plato-torch export-ensign --room deadband`.  
- **Integration gap:** No automatic trigger; requires manual export after room training.  

---

## 4. fleet-simulator → holodeck-rust  
**Connection:** **None observed**  
- Simulator agents are Python classes; holodeck is Rust MUD server.  
- No shared agent definitions, no socket connection in current code.  
- **Potential integration point:** Could feed simulator logs into holodeck as “historical events” for NPC memory.  

---

## 5. plato-ensign → holodeck-rust  
**Connection:** **None observed**  
- Ensign exports LoRA adapters (`.safetensors`).  
- Holodeck does not currently load adapters; uses fixed sentiment models.  
- **Potential integration point:** Load trained ensigns into holodeck NPCs for personalized behavior.  

---

## 6. fleet-simulator → plato-ensign  
**Connection:** **Indirect via plato-torch**  
- Simulator logs become tiles in plato-torch rooms.  
- Those tiles are later exported by plato-ensign.  
- **Integration gap:** No direct simulator→ensign path; must pass through a room.  

---

```
plato-torch ─┬─→ fleet-simulator (direct import)  
             ├─→ plato-ensign (export pipeline)  
             └─→ holodeck-rust (schema alignment)

holodeck-rust ──→ plato-torch (sentiment data via schema)  

plato-ensign ──→ (standalone export)  
```

## Identified Gaps (P0 / P1)  
**P0 (Negative space):**  
- Do not assume automatic data flow between simulator and holodeck.  
- Do not assume ensigns auto-load into holodeck.  

**P1 (Safe channels):**  
- Can extend `ensign_exporter.py` to auto-trigger when room tile count threshold reached.  
- Can add a Rust FFI bridge to load `.safetensors` into holodeck’s NPC system.  
- Can create a shared config schema (YAML/JSON) between plato-torch rooms and fleet-simulator scenarios.  

**P2 (Optimization):**  
- Once channels exist, optimize tile serialization speed, reduce LoRA training time.
