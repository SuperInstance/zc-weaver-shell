## 1. Verified Current Connections (P1 Safe Channels)

Based on reading the actual READMEs and code from the fleet repos:

### **plato-torch → plato-ensign** (Strong, Direct)
- **Connection**: `plato-torch` generates training rooms that accumulate interaction tiles.
- **plato-ensign** exports room experience (tile data) into LoRA adapters.
- **Evidence**: `plato-ensign/README.md` states: "Exports room experience from plato-torch into LoRA adapters."
- **Status**: This is a core, documented pipeline. Safe channel.

### **fleet-simulator → plato-torch** (Indirect, Data Flow)
- **Connection**: `fleet-simulator` generates training data (simulated fleet interactions).
- This data can be consumed as training tiles by `plato-torch` rooms.
- **Evidence**: `fleet-simulator/README.md`: "Generates training data for agent training." `plato-torch` rooms are designed to train on accumulated tiles.
- **Status**: Implied data flow, but no explicit wiring in code observed. A P1 channel exists (data format compatibility likely).

### **holodeck-rust → (Ambient Fleet Data)**
- **Connection**: `holodeck-rust` is a MUD server with sentiment-aware NPCs.
- It can generate narrative interaction data, which could become tiles.
- **Evidence**: README describes it as a "sentiment-aware NPC MUD server." No explicit links to other repos in its docs.
- **Status**: Standalone service. Its output *could* feed the tile ecosystem, but no current integration points are specified. This is a potential channel, not a live one.

### **plato-ensign → (Model Registry/Deployment)**
- **Connection**: `plato-ensign` produces LoRA adapters (ensigns).
- These ensigns are presumably meant to be loaded by inference runtimes (e.g., `cudaclaw`, `flux-runtime`).
- **Evidence**: README states "optional exports of room experience." Implies a consumer exists.
- **Status**: Output defined, but downstream consumption not explicitly mapped in the provided docs.

## 2. Identified Integration Gaps (P0 Negative Space / P2 Optimization Targets)

### **Gap 1: holodeck-rust → plato-torch Tile Injection**
- **What's Missing**: No documented pipeline to format `holodeck-rust` NPC interactions into training tiles and submit them to a `plato-torch` room.
- **P0 Risk**: Building a direct link without understanding tile schema could create malformed data.
- **P1 Safe Channel**: First, read the `plato-torch` tile schema and `holodeck-rust` output format. Verify compatibility.
- **P2 Optimization**: Build `GhostInjector` (from backlog) to bridge this gap.

### **Gap 2: fleet-simulator → plato-torch Automated Pipeline**
- **What's Missing**: Manual or scripted process to move simulated data into a training room. No scheduler or event trigger.
- **P0 Risk**: Assuming automation exists could break data freshness.
- **P1 Safe Channel**: Check for existing scripts in `fleet-simulator/scripts/` or `plato-torch/ingest/`. Document the current manual process.
- **P2 Optimization**: Create a cron job or a `plato-relay` module to automate ingestion.

### **Gap 3: plato-ensign → cudaclaw/flux-runtime Deployment**
- **What's Missing**: Clear pathway to load a trained ensign (LoRA) into the GPU-resident (`cudaclaw`) or bytecode VM (`flux-runtime`) environments for inference.
- **P0 Risk**: Ensigns might be generated but not usable.
- **P1 Safe Channel**: Examine `cudaclaw` and `flux-runtime` READMEs for model loading instructions. Look for a `models/` or `adapters/` directory pattern.
- **P2 Optimization**: Write a deployment script that moves the `.safetensors` file from `plato-ensign/output/` to the runtime's model path.

### **Gap 4: DeadbandRoom → plato-relay (Backlog Item)**
- **What's Missing**: `DeadbandRoom` is a preset in `plato-torch`. `plato-relay` (inferred from context) likely handles room coordination or tile routing. Connection not detailed.
- **P0 Risk**: Rooms may operate in isolation without fleet coordination.
- **P1 Safe Channel**: Find `plato-relay` repo or service definition. Understand its API.
- **P2 Optimization**: Implement the connection per backlog task.
