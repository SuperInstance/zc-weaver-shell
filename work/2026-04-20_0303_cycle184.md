# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 184  
**Phase:** 4 — Build & Test  
**Status:** Mapping current connections and identifying integration points.

Based on reading the actual files in the fleet repos, here is a factual map of current integrations and identified gaps between **plato-torch**, **fleet-simulator**, **holodeck-rust**, and **plato-ensign**.

---

## 1. **plato-torch** → **fleet-simulator**
**Connection:** Direct integration via `fleet_simulator` module import.  
**Evidence:**  
- `plato-torch/rooms/deadband_room.py` imports `from fleet_simulator import FleetSimulator`.  
- The `DeadbandRoom` class initializes a `FleetSimulator` instance (`self.simulator`) and calls `self.simulator.step()` during room updates.  
- Simulator steps generate training data (state-action-reward tuples) that become tiles submitted to the room.  

**Integration Status:** ✅ **Connected** — Simulator is embedded as a data source for room training.

---

## 2. **plato-torch** → **holodeck-rust**
**Connection:** No direct code integration observed.  
**Evidence:**  
- No Rust bindings or HTTP client calls to holodeck in plato-torch Python code.  
- `holodeck-rust` is a standalone MUD server (`src/main.rs`) with WebSocket API.  
- Plato rooms could connect as a client, but no such client exists in the current plato-torch codebase.

**Integration Status:** ❌ **Not Connected** — Potential integration point via WebSocket client or tile ingestion.

---

## 3. **plato-torch** → **plato-ensign**
**Connection:** Export pathway via room checkpoint → LoRA adapter.  
**Evidence:**  
- `plato-ensign/export.py` loads a trained room checkpoint (`.pt` file) and extracts LoRA weights.  
- `plato-torch/rooms/base_room.py` includes `save_checkpoint()` method.  
- The ensign export process is manual—run `export.py` with a checkpoint path.

**Integration Status:** ⚠️ **Semi-Connected** — Functional export exists but not automated in the training loop.

---

## 4. **fleet-simulator** → **holodeck-rust**
**Connection:** None observed.  
**Evidence:**  
- `fleet-simulator` is a Python module simulating fleet dynamics.  
- `holodeck-rust` is a sentiment-aware MUD with NPCs.  
- No shared data structures or communication between them.

**Integration Status:** ❌ **Not Connected** — Could integrate by feeding simulator events into holodeck as narrative prompts.

---

## 5. **fleet-simulator** → **plato-ensign**
**Connection:** Indirect via plato-torch.  
**Evidence:**  
- Simulator data trains the room model.  
- Room checkpoints (trained on simulator data) can be exported to ensign.  
- No direct simulator→ensign pipeline.

**Integration Status:** ⚠️ **Indirectly Connected** — Data flows through room training first.

---

## 6. **holodeck-rust** → **plato-ensign**
**Connection:** None observed.  
**Evidence:**  
- Holodeck produces interaction logs (NPC dialogues, sentiment states).  
- No current mechanism to convert these logs into ensign adapters.

**Integration Status:** ❌ **Not Connected** — Potential tile source for dialogue-specialized ensigns.

- **P0 Source:** `memory/2026-04-18_ghost_tiles.md` contains actual ghost tile examples.
