# Weaver Integration Map — Plato-Torch, Fleet-Simulator, Holodeck-Rust, Plato-Ensign

**Cycle:** 6

**Status:** Mapping current connections between four core fleet technologies.

---

## 1. Plato-Torch (Training Rooms)

**Primary Role:**  
- Provides 25 preset training rooms (including `DeadbandRoom`).  
- Self‑training rooms that accumulate interaction tiles.  
- Exposes a Python API for room creation, tile submission, and training.

**Current Connections:**

| Connected To          | How                                                                                     | Status      |
|-----------------------|-----------------------------------------------------------------------------------------|-------------|
| **Plato‑Ensign**      | Direct export path: room experience → LoRA adapters.                                    | **Active**  |
| **Fleet‑Simulator**   | Not directly linked in code. Potential data‑flow via shared tile format.                | **Indirect** |
| **Holodeck‑Rust**     | No explicit integration found. Both consume/produce tiles but no shared pipeline yet.   | **None**    |

**Key Files:**  
- `plato-torch/rooms/deadband_room.py` – defines room logic and tile handling.  
- `plato-torch/api/room_server.py` – HTTP server for tile submission.  
- `plato-torch/training/trainer.py` – training loop that can export to ensign.

---

## 2. Fleet‑Simulator (Training Data Generation)

**Primary Role:**  
- Simulates fleet interactions to generate training‑data tiles.  
- Outputs tiles in the same JSON format used by plato‑torch rooms.

**Current Connections:**

| Connected To          | How                                                                                     | Status      |
|-----------------------|-----------------------------------------------------------------------------------------|-------------|
| **Plato‑Torch**       | Tiles generated by simulator can be submitted to any plato‑torch room via API.          | **Possible** |
| **Plato‑Ensign**      | No direct link. Simulator tiles must pass through a room first.                         | **Indirect** |
| **Holodeck‑Rust**     | No connection found in code or configs.                                                 | **None**    |

**Key Files:**  
- `fleet-simulator/src/tile_generator.rs` – creates tile‑structured data.  
- `fleet-simulator/config/simulation.toml` – defines agent behaviors and scenarios.

---

## 3. Holodeck‑Rust (Sentiment‑Aware MUD)

**Primary Role:**  
- Multi‑user dungeon server with NPCs that react to player sentiment.  
- Generates interaction tiles from player‑NPC dialogues.

**Current Connections:**

| Connected To          | How                                                                                     | Status      |
|-----------------------|-----------------------------------------------------------------------------------------|-------------|
| **Plato‑Torch**       | Tiles can be sent to a plato‑torch room via HTTP POST (same tile format).               | **Possible** |
| **Fleet‑Simulator**   | No shared code or configuration.                                                        | **None**    |
| **Plato‑Ensign**      | No direct export path. Tiles would need to go through a room first.                     | **Indirect** |

**Key Files:**  
- `holodeck-rust/src/npc/sentiment_engine.rs` – sentiment analysis and tile creation.  
- `holodeck-rust/src/api/tile_submitter.rs` – HTTP client for sending tiles to a room server.

---

## 4. Plato‑Ensign (LoRA Adapter Export)

**Primary Role:**  
- Takes a trained plato‑torch room and exports a LoRA adapter.  
- Optional step; rooms can train without exporting.

**Current Connections:**

| Connected To          | How                                                                                     | Status      |
|-----------------------|-----------------------------------------------------------------------------------------|-------------|
| **Plato‑Torch**       | Direct dependency: imports room checkpoints and runs export script.                     | **Active**  |
| **Fleet‑Simulator**   | No connection. Simulator does not produce room checkpoints.                             | **None**    |
| **Holodeck‑Rust**     | No connection. Holodeck tiles are not directly consumable by ensign.                    | **None**    |

**Key Files:**  
- `plato-ensign/export_lora.py` – loads room model, extracts LoRA weights.  
- `plato-ensign/config/export_config.yaml` – specifies source room and output path.

## Integration Summary

### What’s Connected
1. **Plato‑Torch ↔ Plato‑Ensign** – Direct, active pipeline. Rooms train → ensign exports LoRA adapters.
2. **Fleet‑Simulator → Plato‑Torch** – Indirect, possible. Simulator tiles can be POSTed to a room server.
3. **Holodeck‑Rust → Plato‑Torch** – Indirect, possible. Holodeck tiles can be submitted to a room.

### What’s Not Connected
1. **Fleet‑Simulator ↔ Holodeck‑Rust** – No shared code, configs, or data flow.
2. **Fleet‑Simulator ↔ Plato‑Ensign** – No direct path; simulator doesn’t produce room checkpoints.
3. **Holodeck‑Rust ↔ Plato‑Ensign** – No direct export; holodeck tiles are not room checkpoints.
4. **Any cross‑technology training pipeline** – Each system operates independently; no unified training loop.

### Missing Integration Points (P1 Opportunities)
- **Tile‑routing middleware** – A service that routes tiles from simulator/holodeck to appropriate rooms.
- **Shared tile‑validation library** – Ensure all tiles meet plato‑torch schema before submission.
- **Holodeck‑room bridge** – Direct link between holodeck NPC interactions and a dedicated training room.
- **Simulator‑room auto‑submit** – Config‑driven automatic tile submission from simulator to rooms.
