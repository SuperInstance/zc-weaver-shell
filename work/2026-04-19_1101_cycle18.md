## 1. Verified Connections (What's Connected)

Based on reading the actual fleet repos (`plato-torch`, `fleet-simulator`, `holodeck-rust`, `plato-ensign`), here are the current, factual connections:

### **plato-torch ↔ fleet-simulator**
- **Connection:** Strong, bidirectional data flow.
- **Evidence:** `plato-torch` includes a `FleetSimulatorRoom` preset (one of the 25 training rooms). This room ingests simulation data from `fleet-simulator` to generate training tiles.
- **Mechanism:** The simulator runs agents/scenarios; their interaction logs are formatted into tiles and sent to the PLATO room server for training.
- **Status:** **Live.** This is a core training pipeline.

### **plato-torch ↔ plato-ensign**
- **Connection:** Direct export pipeline.
- **Evidence:** `plato-ensign` is designed explicitly to export a room's accumulated experience (tiles) from `plato-torch` into a LoRA adapter.
- **Mechanism:** After a room trains on a tile corpus, `plato-ensign` can be invoked to create a specialized "ensign" model (a fine-tuned adapter) from that room's state.
- **Status:** **Live.** This is the primary mechanism for creating specialist models from room training.

### **plato-torch (self-contained)**
- **Connection:** Internal loop.
- **Evidence:** `plato-torch` implements "self-training rooms" (e.g., `DeadbandRoom`). These rooms can generate their own synthetic training data and iterate on it without an external source.
- **Mechanism:** Uses rule-based or LLM-generated scenarios to create tiles, trains on them, and repeats. This is the "Mirror Play" concept.
- **Status:** **Live.** A key architectural pattern for bootstrapping.

## 2. Integration Gaps (What's Not Connected)

### **holodeck-rust ↔ plato-torch**
- **Gap:** No direct, automated tile-ingestion pipeline.
- **Evidence:** `holodeck-rust` is a sentiment-aware MUD server that generates rich narrative interactions. `plato-torch` has training rooms that consume structured tile data. There is no preset `HolodeckRoom` or documented API bridge between them.
- **Opportunity:** NPC dialogues and player decisions in the holodeck are a high-quality source of training tiles for narrative, reasoning, and sentiment tasks. This is a major untapped channel.
- **P1 Action:** Build a `HolodeckBridge` service that formats holodeck session logs into tiles and POSTs them to a PLATO room.

### **holodeck-rust ↔ fleet-simulator**
- **Gap:** No shared agent framework or state.
- **Evidence:** Both are simulation environments. `fleet-simulator` focuses on multi-agent fleet dynamics and decision-making. `holodeck-rust` focuses on interactive narrative and character sentiment. Their agent models and state representations are separate.
- **Opportunity:** Fleet agents could have "holodeck sessions" for role-playing or negotiation training. Simulator agents could be given narrative personas.
- **P1 Action:** Design a shared agent profile schema that can be used by both simulators.

### **plato-ensign → holodeck-rust / fleet-simulator**
- **Gap:** Ensigns are not automatically deployed back into simulations.
- **Evidence:** An ensign (LoRA) is created from room experience, but there's no automated pipeline to load that ensign into an agent in `holodeck-rust` or `fleet-simulator` to test/use it.
- **Opportunity:** Close the loop. A room trains on simulator data, creates an ensign, and that ensign is immediately loaded into a new simulator agent to evaluate its performance. This is reinforcement learning.
- **P1 Action:** Create an `EnsignLoader` module for the simulators that can fetch and apply a LoRA from the `plato-ensign` registry.

### **Ghost Tiles (P0 Knowledge) → Systematic Ingestion**
- **Gap:** Ghost tiles are mentioned as a concept (dead agents' lessons), but no system-wide service exists to collect and inject them into all training rooms as P0 constraints.
- **Evidence:** The concept is in the architecture description, but no `GhostInjector` service or room preset is evident in the main repos.
- **Opportunity:** Prevent repeated failures fleet-wide. A central ghost tile store could be subscribed to by all rooms to prepend negative examples to their training corpus.
- **P1 Action:** Build the `GhostInjector` as a standalone service and integrate it with the `plato-torch` room initialization API.
