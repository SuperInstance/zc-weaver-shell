# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 135

Based on reading the actual files in the fleet repos, here is the current state of integration between plato-torch, fleet-simulator, holodeck-rust, and plato-ensign.

## 1. Verified Connections (What IS Connected)

### plato-torch ↔ fleet-simulator
- **Connection:** Direct integration via `fleet-simulator` as a data source for training rooms.
- **Evidence:**  
  - `plato-torch/rooms/deadband_room.py` imports `from fleet_simulator import FleetSimulator`.  
  - The `DeadbandRoom` class initializes a `FleetSimulator` instance (`self.simulator`).  
  - The room uses `self.simulator.step()` to generate simulation states, which become training tiles.  
- **Data Flow:** Fleet simulation states → training tiles → room experience.

### plato-torch ↔ plato-ensign
- **Connection:** Optional export path for trained room experience.
- **Evidence:**  
  - `plato-ensign/README.md` states: "Exports room experience into LoRA adapters."  
  - `plato-torch/export/` directory contains `ensign_exporter.py` with functions to convert room checkpoints to LoRA adapters.  
- **Data Flow:** Room training checkpoints → LoRA adapters (specialist ensigns).

## 2. Integration Gaps (What Is NOT Connected)

### holodeck-rust ↔ plato-torch
- **Gap:** No direct connection found.  
- **Evidence:**  
  - `holodeck-rust/` is a standalone MUD server written in Rust with sentiment-aware NPCs.  
  - `plato-torch/` does not import or reference `holodeck-rust` in any Python files.  
  - No shared data structures or APIs between them in the current codebase.  
- **Potential Integration Point:** NPC interaction logs from holodeck could become training tiles for a "social reasoning" room in plato-torch.

### holodeck-rust ↔ fleet-simulator
- **Gap:** No connection found.  
- **Evidence:**  
  - `fleet-simulator/` simulates fleet dynamics (agents, tasks, resources).  
  - `holodeck-rust/` simulates social interactions in a text-based world.  
  - No shared code or data exchange between these simulators.  
- **Potential Integration Point:** Fleet agents could be represented as NPCs in holodeck for sentiment-aware interaction training.

### holodeck-rust ↔ plato-ensign
- **Gap:** No connection found.  
- **Evidence:**  
  - `plato-ensign` exports from plato-torch rooms only.  
  - No Rust-to-Python pipeline for converting holodeck experience to LoRA adapters.  
- **Potential Integration Point:** If holodeck interactions were logged as tiles, they could be exported to train "social" ensigns.

### fleet-simulator ↔ plato-ensign
- **Gap:** No direct export path from simulator to ensign.  
- **Evidence:**  
  - `plato-ensign` reads from plato-torch room checkpoints, not raw simulator data.  
  - `fleet-simulator` outputs simulation states, not trained model weights.  
- **Potential Integration Point:** Simulator data must pass through a plato-torch room first to become ensign-ready.
