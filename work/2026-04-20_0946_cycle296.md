# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 296  
**Phase:** 4 — Build & Test  
**Status:** Direct file inspection of fleet repositories. Focus on actual imports, configuration references, and shared data structures.

## 1. plato-torch
**Location:** `plato-torch/`  
**Purpose:** Training room presets, self-training rooms.  
**Key Files:**
- `presets/deadband_room.py` — Deadband Protocol training room.
- `presets/__init__.py` — Exports presets.
- `trainer/room_trainer.py` — Orchestrates training loops.
- `utils/tile_processor.py` — Processes interaction tiles.

**Connections:**
- **To plato-ensign:** Exports room experience via `export_to_ensign()` method in `room_trainer.py`. Creates LoRA adapters.
- **To fleet-simulator:** No direct import found. Potential integration via shared tile format.
- **To holodeck-rust:** No direct import. Plato-torch is Python; holodeck is Rust.

**Integration Points:**
- `tile_processor.py` defines `TrainingTile` dataclass — possible shared data structure.
- `deadband_room.py` implements `DeadbandProtocol` (P0, P1, P2) — logic that could be reused.

## 2. plato-ensign
**Location:** `plato-ensign/`  
**Purpose:** Exports room experience into LoRA adapters.  
**Key Files:**
- `exporter/lora_exporter.py` — Converts room checkpoints to LoRA.
- `exporter/__init__.py` — Exports exporter.
- `configs/ensign_config.yaml` — Configuration for adapter creation.

**Connections:**
- **From plato-torch:** Called via `export_to_ensign()` in `room_trainer.py`. Direct dependency.
- **To fleet-simulator:** No import found.
- **To holodeck-rust:** No import found.

**Integration Points:**
- Expects checkpoint files from plato-torch training sessions.
- Outputs `.safetensors` files (LoRA weights) — usable by other fleet components.

## 3. fleet-simulator
**Location:** `fleet-simulator/`  
**Purpose:** Fleet simulation for training data.  
**Key Files:**
- `src/simulator/core.rs` — Core simulation engine.
- `src/agents/mod.rs` — Agent definitions.
- `src/data/tile_generator.rs` — Generates training tiles from simulations.

**Connections:**
- **To plato-torch:** No Rust→Python import. Potential integration via tile format (JSON).
- **To plato-ensign:** No direct connection.
- **To holodeck-rust:** No import found in code.

**Integration Points:**
- `tile_generator.rs` produces `SimulationTile` struct — could align with plato-torch's `TrainingTile`.
- Simulation runs produce logs that could be consumed by training rooms.

## 4. holodeck-rust
**Location:** `holodeck-rust/`  
**Purpose:** MUD server with sentiment-aware NPCs.  
**Key Files:**
- `src/server/mud_server.rs` — Main server loop.
- `src/npc/sentiment_engine.rs` — NPC sentiment logic.
- `src/world/room_manager.rs` — Manages game rooms.

**Connections:**
- **To plato-torch:** No direct connection. Different languages (Rust vs Python).
- **To plato-ensign:** No connection.
- **To fleet-simulator:** No import found.

**Integration Points:**
- Could consume training tiles as NPC dialogue or behavior seeds.
- Sentiment engine might use emotion labels from tile metadata.

## 5. Constraint Theory Core

**Connections:**

**Integration Points:**

## 6. cudaclaw & flux-runtime
**Location:** `cudaclaw/`, `flux-runtime/`  
**Purpose:** GPU-resident agent runtime and self-assembling bytecode VM.  
**Connections:**
- Not directly imported by any of the four core repos inspected.
- Potential low-level infrastructure for plato-torch (GPU training) or fleet-simulator (agent execution).

## Integration Gaps (What's Not Connected)
1. **plato-torch ↔ fleet-simulator:** No shared data pipeline. Tiles generated by simulator not automatically fed to training rooms.
2. **plato-torch ↔ holodeck-rust:** Language barrier. No bridge between Python training and Rust MUD server.
3. **fleet-simulator ↔ holodeck-rust:** Separate Rust codebases with no shared modules.
4. **constraint-theory-core:** Isolated. Not integrated into any higher-level component.
5. **cudaclaw/flux-runtime:** Not wired into application-layer repos.

## Recommended Integration Tasks (P1 Safe Channels)
1. **Define common tile schema** (JSON/protobuf) shared by plato-torch and fleet-simulator.
2. **Create Rust→Python bridge** (PyO3) to allow holodeck to consume training tiles.
3. **Add constraint-theory to holodeck** for spatial puzzles in MUD rooms.
4. **Instrument plato-torch** to optionally use cudaclaw for GPU acceleration.
