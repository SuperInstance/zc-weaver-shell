# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 121  
**Phase:** 4 — Build & Test  
**Status:** Mapping current connections and identifying integration points.

Based on reading the actual files in the fleet repos, here is the current state of integration between **plato-torch**, **fleet-simulator**, **holodeck-rust**, and **plato-ensign**.

---

## 1. **plato-torch** → **plato-ensign**  
**Connection:** ✅ **Strong & Direct**  
- **plato-torch** includes `ensign_exporter.py` which exports room training data to LoRA adapters.  
- The `EnsignExporter` class loads a trained room checkpoint, extracts adapter weights, and saves them as a `.safetensors` file.  
- This is the primary training-to-export pipeline documented in the repo.  
- **Integration Gap:** None — this is a core, working feature.

---

## 2. **plato-torch** → **fleet-simulator**  
**Connection:** 🔄 **Indirect / Data-Only**  
- **fleet-simulator** generates synthetic training data (JSONL files) through agent simulations.  
- **plato-torch** can load these JSONL files via `DataLoader` for training rooms (e.g., `DeadbandRoom`).  
- No runtime API or live connection exists — it's a file-based data pipeline.  
- **Integration Gap:** No real-time simulation → training loop. The simulator runs offline, produces static datasets.

---

## 3. **holodeck-rust** → **plato-torch**  
**Connection:** ⚠️ **Planned / Not Yet Implemented**  
- **holodeck-rust** is a MUD server with sentiment-aware NPCs, intended to generate interactive training episodes.  
- The `README.md` mentions future integration with PLATO rooms for live training.  
- Currently, no code in **plato-torch** references **holodeck-rust**, and vice versa.  
- **Integration Gap:** No live data stream from holodeck to a training room. No adapter to convert MUD interactions into training tiles.

---

## 4. **fleet-simulator** → **holodeck-rust**  
**Connection:** ❌ **None Detected**  
- No shared code, imports, or configuration files linking these two repos.  
- Both generate training data but through different mechanisms (simulated agent dialogues vs. interactive MUD).  
- **Integration Gap:** No cross-pollination of data formats or runtime hooks.

---

## 5. **plato-ensign** → **holodeck-rust** / **fleet-simulator**  
**Connection:** ❌ **None Detected**  
- **plato-ensign** is an export target, not a runtime component.  
- No evidence that ensign adapters are loaded into the simulator or holodeck for inference or fine-tuning.  
- **Integration Gap:** Trained ensigns are not yet consumed by data-generating systems.

---

## 6. **GhostInjector** → **holodeck-rust**  
**Connection:** ⚠️ **Planned / Not Yet Implemented**  
- **GhostInjector** (part of **plato-torch**) is designed to inject "ghost tiles" (P0 lessons from dead agents) into training streams.  
- **holodeck-rust** NPCs could theoretically be seeded with ghost tile knowledge.  
- No current code connects them.  
- **Integration Gap:** No injection mechanism from `ghost_tiles.json` into holodeck's NPC memory.

---

## Summary of Integration Gaps (P1 Safe Channels to Explore)

1. **Live Holodeck → Plato Room Pipeline**  
   - Build a connector that streams NPC interactions from holodeck into a `DeadbandRoom` as training tiles.

2. **Fleet-Simulator Real-Time Mode**  
   - Extend the simulator to emit events via WebSocket or gRPC, allowing plato-torch to train on fresh simulation data.

3. **Ensign Consumption Loop**  
   - Load exported `.safetensors` adapters into holodeck NPCs or simulator agents to test their specialized knowledge.

4. **Ghost Tile Injection**  
   - Implement a `GhostInjector` plugin for holodeck that reads `ghost_tiles.json` and influences NPC behavior.
