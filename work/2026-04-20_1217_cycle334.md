# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 334

---

## 1. plato-torch → fleet-simulator
**Connection:** **Strong**  
- `plato-torch/rooms/deadband_room.py` imports `fleet_simulator` directly:  
  ```python
  from fleet_simulator import FleetSimulator, FleetState
  ```
- Uses `FleetSimulator` to generate training episodes.  
- `FleetState` is passed to the room's `step()` method.  
- Shared data structure: `FleetState` (contains agent positions, velocities, actions).  

**Integration Gap:** None observed. This is a direct, functional dependency.

---

## 2. plato-torch → holodeck-rust
**Connection:** **Weak / Indirect**  
- No direct import of `holodeck-rust` in `plato-torch` Python code.  
- However, `plato-torch/configs/room_presets/deadband_preset.yaml` contains:  
  ```yaml
  environment:
    type: holodeck
    config: holodeck_config.yaml
  ```
- `holodeck_config.yaml` references `holodeck-rust` server endpoints (e.g., `ws://localhost:8080/ws`).  
- Connection is via configuration, not code import. Room likely expects a running `holodeck-rust` server.

**Integration Gap:** No Python client library observed. Integration is socket‑based (WebSocket). Could break if server changes API.

---

## 3. plato-torch → plato-ensign
**Connection:** **Strong**  
- `plato-torch/export/ensign_exporter.py` imports `plato_ensign`:  
  ```python
  from plato_ensign import LoRAAdapter, export_room_experience
  ```
- Called after room training to export LoRA adapters.  
- Shared data: Room experience tiles (serialized interactions).  

**Integration Gap:** None observed. Direct library usage.

---

## 4. fleet-simulator → holodeck-rust
**Connection:** **None**  
- `fleet-simulator/src/lib.rs` has no mention of `holodeck`.  
- Simulator is pure Rust, generates `FleetState` internally.  
- No WebSocket or HTTP client observed.

**Integration Gap:** These two systems are independent. `fleet-simulator` does not drive `holodeck-rust` NPCs.

---

## 5. fleet-simulator → plato-ensign
**Connection:** **None**  
- No cross‑references in code or config.  
- `fleet-simulator` outputs `FleetState` structs, not training tiles.  
- `plato-ensign` works on room‑collected tiles, not simulation states.

**Integration Gap:** No direct link. Integration would require a translation layer (simulator states → training tiles).

---

## 6. holodeck-rust → plato-ensign
**Connection:** **None**  
- `holodeck-rust/src/main.rs` exports no training data.  
- It is a MUD server; sentiment‑aware NPC interactions are not captured as tiles.  
- No `plato_ensign` dependency in `Cargo.toml`.

**Integration Gap:** NPC interactions are currently ephemeral (in‑memory). Not persisted for ensign training.

---

## 7. plato-ensign → Others
**Connection:** **Consumer only**  
- `plato-ensign` reads tiles from `plato-torch` rooms.  
- No dependencies on `fleet-simulator` or `holodeck-rust`.  
- Outputs LoRA adapters (`.safetensors` files).

**Integration Gap:** Only works with `plato-torch` tile format.
