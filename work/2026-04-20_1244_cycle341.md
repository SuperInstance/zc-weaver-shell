## 1. plato-torch (Training Rooms)
**Location:** `plato-torch/`  
**Purpose:** 25 training room presets, self-training rooms.  
**Key Files:**
- `plato-torch/rooms/deadband_room.py` — Deadband Protocol training room.
- `plato-torch/rooms/__init__.py` — Exports room classes.
- `plato-torch/training/trainer.py` — Core training loop.
- `plato-torch/config/room_configs.yaml` — Room configurations.

**Connections Out:**
- **To plato-ensign:** `plato-torch/export/ensign_exporter.py` imports `from plato_ensign.lora_adapter import LoRAAdapter`. Exports room experience to LoRA adapters.
- **To fleet-simulator:** No direct import found. Config `room_configs.yaml` contains `simulation_scenario` keys that *reference* scenario names, but no Python import.
- **To holodeck-rust:** No direct import. No Rust bindings or FFI calls in codebase.

**Data Flow:**  
Rooms generate `interaction_tiles` (JSON). `ensign_exporter.py` reads tiles, converts to LoRA training data.

---

## 2. plato-ensign (LoRA Adapter Export)
**Location:** `plato-ensign/`  
**Purpose:** Exports room experience into LoRA adapters.  
**Key Files:**
- `plato-ensign/lora_adapter.py` — LoRA adapter class.
- `plato-ensign/export.py` — Export utilities.
- `plato-ensign/config/export_config.yaml` — Export settings.

**Connections Out:**
- **To plato-torch:** `plato-ensign/export.py` imports `from plato_torch.rooms.base import BaseRoom`. Dependency is one-way: plato-torch calls ensign, not vice versa.
- **To fleet-simulator:** No import.
- **To holodeck-rust:** No import.

**Data Flow:**  
Accepts `tiles` from plato-torch, produces `.safetensors` files (LoRA weights).

---

## 3. fleet-simulator (Fleet Simulation)
**Location:** `fleet-simulator/`  
**Purpose:** Fleet simulation for training data.  
**Key Files:**
- `fleet-simulator/src/simulator.rs` — Core simulator.
- `fleet-simulator/src/scenarios/mod.rs` — Scenario definitions.
- `fleet-simulator/config/scenarios.yaml` — Scenario config.

**Connections Out:**
- **To plato-torch:** No Rust → Python import. However, `scenarios.yaml` contains `output_format: plato_torch_tile`. Implies data compatibility, not runtime integration.
- **To plato-ensign:** No import.
- **To holodeck-rust:** `fleet-simulator/Cargo.toml` includes `holodeck = { path = "../holodeck-rust" }`. Direct dependency.

**Data Flow:**  
Simulator generates `simulation_events`. Can output JSON tiles compatible with plato-torch. Feeds scenarios to holodeck-rust via Rust API.

---

## 4. holodeck-rust (MUD Server)
**Location:** `holodeck-rust/`  
**Purpose:** MUD server with sentiment-aware NPCs.  
**Key Files:**
- `holodeck-rust/src/server.rs` — MUD server.
- `holodeck-rust/src/npc/sentiment.rs` — Sentiment engine.
- `holodeck-rust/src/lib.rs` — Library exports.

**Connections Out:**
- **To fleet-simulator:** Exported as Rust crate. `fleet-simulator` depends on it.
- **To plato-torch:** No Python bindings (`pyo3` not in Cargo.toml). No direct connection.
- **To plato-ensign:** No connection.

**Data Flow:**  
Accepts scenario definitions from fleet-simulator. Produces NPC interaction logs (JSON). Not currently wired to plato-torch tile ingestion.
