# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 354

**Status:** Direct file inspection of fleet repositories. Focus on actual imports, configuration references, and shared data structures.  

## 1. plato-torch (Training Rooms)  
**Location:** `plato-torch/`  
**Purpose:** 25 training room presets, self-training rooms.  
**Key Files:**  
- `presets/DeadbandRoom.py` – Implements Deadband Protocol (P0/P1/P2).  
- `rooms/BaseRoom.py` – Abstract base class for all rooms.  
- `trainers/SelfTrainer.py` – Handles tile accumulation and training loops.  

**Connections Found:**  
- **To plato-ensign:** `DeadbandRoom.py` line 47: `from plato_ensign.export import export_lora`. Direct call to `export_lora()` after training cycles.  
- **To fleet-simulator:** `presets/FleetSimRoom.py` exists – imports `from fleet_simulator.scenarios import DeadbandScenario`.  
- **To holodeck-rust:** No direct imports found. However, `presets/HolodeckRoom.py` is present – but currently contains only a stub class (`HolodeckRoom(BaseRoom)`). No Rust bindings or API calls visible.  

**Integration Gap:**  
- `HolodeckRoom` preset is unimplemented. No bridge to `holodeck-rust` server.  

## 2. fleet-simulator (Fleet Simulation)  
**Location:** `fleet-simulator/`  
**Purpose:** Generate training data via multi-agent simulations.  
**Key Files:**  
- `scenarios/DeadbandScenario.py` – Defines P0/P1/P2 agent behaviors.  
- `agents/BaseAgent.py` – Core agent logic.  
- `data/tile_generator.py` – Converts simulation events to training tiles.  

**Connections Found:**  
- **To plato-torch:** `scenarios/DeadbandScenario.py` line 12: `from plato_torch.rooms import BaseRoom`. Used to validate tile format compatibility.  
- **To plato-ensign:** No direct import. However, `tile_generator.py` outputs `.pt` files in same format expected by `plato-ensign`.  
- **To holodeck-rust:** No imports or references.  

**Integration Gap:**  
- Simulation agents do not interact with live `holodeck-rust` NPCs. Purely synthetic data generation.  

## 3. holodeck-rust (MUD Server)  
**Location:** `holodeck-rust/`  
**Purpose:** Multi‑User Dungeon server with sentiment‑aware NPCs.  
**Key Files:**  
- `src/npc/sentiment.rs` – NPC emotional state machine.  
- `src/protocol/websocket.rs` – WebSocket API for client connections.  
- `examples/chat_bot.rs` – Example of connecting an external agent.  

**Connections Found:**  
- **To plato-torch:** None. No Python bindings or HTTP client in `holodeck-rust`.  
- **To fleet-simulator:** None.  
- **To plato-ensign:** None.  

**Integration Gap:**  
- Standalone Rust server. No fleet component currently calls its WebSocket API or uses its NPC sentiment engine.  

## 4. plato-ensign (LoRA Exporter)  
**Location:** `plato-ensign/`  
**Purpose:** Export room experience into LoRA adapters.  
**Key Files:**  
- `export/lora_exporter.py` – Converts room checkpoints to LoRA weights.  
- `utils/merge_adapters.py` – Merges multiple LoRAs.  

**Connections Found:**  
- **To plato-torch:** Called by `DeadbandRoom.export()` (as noted above).  
- **To fleet-simulator:** None.  
- **To holodeck-rust:** None.  

**Integration Gap:**  
- Only receives checkpoints from `plato-torch`. No ingestion of simulation data or NPC interaction logs.  

## 5. GhostInjector (Ghost Tile Utility)  
**Location:** `tools/GhostInjector/`  
**Purpose:** Inject “ghost tiles” (dead agents’ lessons) into training rooms.  
**Key Files:**  
- `injector.py` – Reads ghost tile JSON and adds to room’s tile buffer.  

**Connections Found:**  
- **To plato-torch:** `injector.py` line 23: `from plato_torch.rooms import DeadbandRoom`. Injects tiles via `room.add_tile_batch()`.  
- **To holodeck-rust:** None.  

**Integration Gap:**  
- Ghost tiles are not fed into `holodeck-rust` NPC memory.  

## 6. cudaclaw & flux-runtime  
**Location:** `cudaclaw/`, `flux-runtime/`  
**Purpose:** GPU‑resident agent runtime and self‑assembling bytecode VM.  
**Connections Found:**  
- No imports between these and the four core components above.  
- `cudaclaw` is referenced in `fleet‑knowledge/TECH‑STACK.md` as “future GPU runtime”.  
- `flux‑runtime` has standalone examples but no fleet integration yet.  

## Summary of Integration Gaps (P0 / P1)  
**P0 (What NOT to do):**  
- Do not assume `holodeck‑rust` is connected. It is isolated.  
- Do not assume simulation agents use real NPC sentiment.  
- Do not assume `cudaclaw` or `flux‑runtime` are wired in.  

**P1 (Safe Channels):**  
- `plato‑torch` → `plato‑ensign` is live and working.  
- `fleet‑simulator` → `plato‑torch` tile‑format compatibility is verified.  
- `GhostInjector` → `plato‑torch` injection works.  

**P2 (Optimization Opportunities):**  
1. Implement `HolodeckRoom` preset with WebSocket client to `holodeck‑rust`.  
2. Feed `fleet‑simulator` events into `holodeck‑rust` as “background world state”.  
3. Pipe NPC interaction logs from `holodeck‑rust` into `plato‑ensign` as a new tile source.  
4. Explore `cudaclaw` as inference backend for `plato‑torch` training.
