## 1. Verified Connections (Based on fleet-knowledge/)

### plato-torch → plato-ensign
- **Connection:** Direct export pipeline.
- **Mechanism:** `plato-torch` training rooms accumulate interaction tiles. `plato-ensign` can export a room's experience into a LoRA adapter.
- **Evidence:** From FLEET-CONTEXT.md: "plato-ensign: Exports room experience into LoRA adapters" and "Rooms train from accumulated interaction tiles. Ensigns are optional exports of room experience."
- **Status:** Connected, functional.

### plato-torch → fleet-simulator
- **Connection:** Data source for training.
- **Mechanism:** `fleet-simulator` generates training data (likely simulation logs, agent interactions). This data can be formatted into tiles and ingested by a `plato-torch` room for training.
- **Evidence:** Both are listed as "Key Technologies." The architecture implies tiles come from work output, and the simulator is a source of such data.
- **Status:** Connected logically; integration likely via tile formatting.

### holodeck-rust → (Potential Data Consumer)
- **Connection:** Sentiment-aware NPCs could be trained or influenced by room data.
- **Mechanism:** Not explicitly defined in provided context. A plausible link: `holodeck-rust` NPC behavior or dialogue could be adapted using LoRA ensigns exported from `plato-torch`.
- **Evidence:** Both are key technologies. "Ghost tiles: dead agents' lessons persist as P0 knowledge" could be injected into holodeck as NPC memory or behavior constraints.
- **Status:** Not directly connected. A gap.

### fleet-simulator → holodeck-rust
- **Connection:** Unclear from context. The simulator could provide a virtual environment for the holodeck MUD, or generate NPC interaction logs.
- **Evidence:** No direct mention.
- **Status:** Not connected. A gap.

### plato-ensign → holodeck-rust / fleet-simulator
- **Connection:** LoRA adapters (ensigns) could be loaded into agents within the simulator or holodeck NPCs to impart specialized behaviors.
- **Evidence:** Not explicitly stated, but aligns with the "export room experience" purpose.
- **Status:** Not connected. A significant integration opportunity.

## 2. Integration Gaps (P1: Safe Channels for Work)

### Gap 1: Ghost Tile Injection into holodeck-rust
- **Description:** "Ghost tiles" (P0 knowledge from dead agents) are mentioned as persisting. No defined pipeline injects them into `holodeck-rust` NPC sentiment systems.
- **Safe Channel:** Build a `GhostInjector` service that reads from a ghost tile store (e.g., a directory of `.md` files) and converts them into `holodeck-rust` NPC memory entries or behavior flags via its API.
- **Action:** Wire GhostInjector into holodeck (from backlog).

### Gap 2: DeadbandRoom to plato-relay Connectivity
- **Description:** `DeadbandRoom` is a `plato-torch` preset. `plato-relay` (inferred as a communication layer) is not mentioned in context, but connectivity would enable real-time tile streaming or room synchronization.
- **Safe Channel:** Examine `plato-torch` API for room event hooks. Implement a relay client that publishes tile submissions to a `plato-relay` message bus.
- **Action:** Connect DeadbandRoom to plato-relay (from backlog).

### Gap 3: End-to-End Training Pipeline
- **Description:** No documented pipeline from `fleet-simulator` data → `plato-torch` tiles → `plato-ensign` LoRA → deployment to `holodeck-rust` or `cudaclaw`.
- **Safe Channel:** Create a script that: (1) converts simulator logs to tile format, (2) imports them into a room, (3) triggers ensign export, (4) loads the LoRA into a target runtime.
- **Action:** Test end-to-end pipeline (from backlog).

### Gap 4: Constraint Theory Core Utilization
- **Description:** `constraint-theory-core` (Rust geometric snapping) is a key technology but no listed integration with other systems.
- **Safe Channel:** Propose a use case: integrating geometric constraints into `fleet-simulator` for spatial agent navigation, or into `holodeck-rust` for environment interaction.
- **Action:** (New) Draft an integration spec for constraint-theory-core.
