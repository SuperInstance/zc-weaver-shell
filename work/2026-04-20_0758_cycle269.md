## 1. plato-torch
**Repository:** `plato-torch`  
**Purpose:** Training room presets (25 rooms), self-training rooms, tile ingestion and processing.  
**Key Files:**  
- `presets/DeadbandRoom.py` — Primary training room implementing Deadband Protocol.  
- `core/tile_processor.py` — Processes incoming training tiles.  
- `core/room_registry.py` — Manages room instances.  

**Connections:**  
- **plato-ensign:** Exports room experience as LoRA adapters. Direct import in `presets/DeadbandRoom.py`:  
  ```python
  from plato_ensign.export import export_lora_from_room
  ```
- **fleet-simulator:** Used for generating training data. Referenced in `presets/DeadbandRoom.py` as a data source for simulation-based tiles.  
- **holodeck-rust:** Not directly imported. Potential integration via tile data format (sentiment-aware NPC interactions could feed into rooms).  

**Integration Gaps:**  
- No direct import of `holodeck-rust`. Could connect via a shared tile schema or API bridge.  
- `fleet-simulator` is referenced but not imported; likely uses external data files or HTTP API.  

## 2. fleet-simulator
**Repository:** `fleet-simulator`  
**Purpose:** Generates training data through multi-agent fleet simulations.  
**Key Files:**  
- `src/simulation/engine.rs` — Core simulation engine.  
- `src/agents/mod.rs` — Defines agent behaviors.  
- `src/output/tile_generator.rs` — Converts simulation events to training tiles.  

**Connections:**  
- **plato-torch:** Outputs training tiles consumed by `DeadbandRoom`. Tile format matches `plato-torch` schema (JSON with `type`, `content`, `metadata`).  
- **holodeck-rust:** No direct connection observed. Could share agent behavior models or sentiment data.  
- **plato-ensign:** No direct connection; simulation data could be used to train ensigns indirectly via room exports.  

**Integration Gaps:**  
- No Rust-to-Python bridge for direct integration with `plato-torch`. Currently relies on file-based or network tile streaming.  
- `holodeck-rust` integration unexplored—potential for shared NPC/agent logic.  

## 3. holodeck-rust
**Repository:** `holodeck-rust`  
**Purpose:** MUD server with sentiment-aware NPCs, generates interactive narrative tiles.  
**Key Files:**  
- `src/server/mud_server.rs` — MUD server implementation.  
- `src/npc/sentiment_engine.rs` — NPC sentiment modeling.  
- `src/protocol/tile_protocol.rs` — Defines tile format for client-server communication.  

**Connections:**  
- **plato-torch:** Tile format is similar but not identical. No direct import; could be bridged via a custom `TileAdapter`.  
- **fleet-simulator:** No observed connections.  
- **plato-ensign:** No direct connection.  

**Integration Gaps:**  
- Largest gap: `holodeck-rust` is isolated. Its sentiment-aware NPC tiles are not feeding into `plato-torch` rooms.  
- Missing Rust/Python interop for real-time tile streaming.  
- No shared agent definition with `fleet-simulator`.  

## 4. plato-ensign
**Repository:** `plato-ensign`  
**Purpose:** Exports room experience from `plato-torch` into LoRA adapters for specialist ensign models.  
**Key Files:**  
- `src/export/lora_exporter.py` — Core export logic.  
- `src/models/room_adapter.py` — Adapts room state for training.  
- `config/ensign_config.yaml` — Configuration for export jobs.  

**Connections:**  
- **plato-torch:** Direct import and usage. `DeadbandRoom` calls `export_lora_from_room()` at training milestones.  
- **fleet-simulator:** No direct connection.  
- **holodeck-rust:** No connection.  

**Integration Gaps:**  
- Only connected to `plato-torch`. Could be extended to export simulation or NPC-based ensigns.  
- No integration with `holodeck-rust` sentiment models for specialized NPC ensigns.
