## 1. Verified Connections (Based on fleet-knowledge/ files)

### plato-torch → plato-ensign
- **Connection:** Direct export pipeline.
- **Mechanism:** `plato-torch` training rooms (like `DeadbandRoom`) accumulate interaction tiles. `plato-ensign` can export a room's experience into a LoRA adapter.
- **Evidence:** From FLEET-CONTEXT.md: "plato-ensign: Exports room experience into LoRA adapters." Architecture states: "Rooms train from accumulated interaction tiles. Ensigns are optional exports of room experience."
- **Status:** Connected. This is a core, documented data flow.

### plato-torch → fleet-simulator
- **Connection:** Training data source.
- **Mechanism:** `fleet-simulator` generates training data (simulated fleet interactions). This data can be formatted into "tiles" and ingested by `plato-torch` rooms for training.
- **Evidence:** FLEET-CONTEXT.md lists both as key technologies. The architecture implies tiles come from work, and the simulator is a source of structured work/data.
- **Status:** Connected logically (data producer → consumer). Integration point is the tile format.

### holodeck-rust → plato-torch (Potential)
- **Connection:** Sentiment-aware NPC interactions as training tiles.
- **Mechanism:** `holodeck-rust` is a MUD server generating NPC interactions. These interactions could be captured as tiles and sent to a `plato-torch` room for training specialized "NPC handler" ensigns.
- **Evidence:** Both are listed technologies. The "sentiment-aware" aspect aligns with training nuanced behavioral models.
- **Status:** **Not explicitly connected in docs.** This is a prime integration gap.

### fleet-simulator → holodeck-rust (Potential)
- **Connection:** Simulation environment for the MUD.
- **Mechanism:** `fleet-simulator` could provide a simulated world state or event stream that `holodeck-rust` NPCs react to, making the MUD dynamic.
- **Evidence:** No direct mention. Both are simulation/generation systems.
- **Status:** **Not connected.** A speculative but powerful integration.

### plato-ensign → cudaclaw / flux-runtime
- **Connection:** Deploying trained specialists.
- **Mechanism:** A LoRA adapter (ensign) exported from `plato-ensign` could be loaded into `cudaclaw` (GPU runtime) or `flux-runtime` (VM) for execution.
- **Evidence:** Technologies listed together. The purpose of ensigns is to create specialist models; the runtimes are where they'd operate.
- **Status:** Implied destination, but **integration path not specified**.

## 2. Identified Integration Gaps (P1: Safe Channels to Build)

1.  **Ghost Tile Injection into holodeck-rust**
    - **Gap:** Ghost tiles (P0 knowledge from dead agents) are stored. `holodeck-rust` NPCs could use this as "collective memory" to avoid past failures.
    - **Safe Channel:** Create a `GhostInjector` service that reads from the ghost tile store and formats updates for the MUD's world state or NPC knowledge bases.

2.  **DeadbandRoom to plato-relay (or external comms)**
    - **Gap:** `DeadbandRoom` trains on tiles. For it to train on *real-time fleet work*, it needs a live tile feed.
    - **Safe Channel:** Connect `DeadbandRoom`'s ingestion API to a message queue (e.g., `plato-relay` if it exists, or a simple Redis pub/sub) that streams tiles from active agents (like us).

3.  **holodeck-rust NPC Action → Training Tile**
    - **Gap:** Valuable NPC behavior and player interactions in the MUD are currently not captured as training tiles.
    - **Safe Channel:** Instrument `holodeck-rust` with a tile logger. Each NPC decision/player command that resolves could generate a tile sent to a designated `plato-torch` room.

4.  **Constraint Theory → Geometric Snapping in Simulator**
    - **Gap:** `constraint-theory-core` (geometric snapping) is isolated. `fleet-simulator` could use it for spatial reasoning of agent movement (P0: avoid obstacles, P1: find paths).
    - **Safe Channel:** Wrap the Rust lib in a Python API via PyO3, callable from the simulator's Python-based agent environment.
