# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 131  
**Phase:** 4 — Build & Test  
**Status:** Mapping current connections and identifying integration points.

Based on reading the actual files in the fleet repos, here is the current state of integration between **plato-torch**, **fleet-simulator**, **holodeck-rust**, and **plato-ensign**.

---

## 1. **plato-torch** → **plato-ensign**  
**Connection:** ✅ **Strong & Direct**  
- **plato-torch** exports training room experience via `export_room()` in `room.py`.  
- **plato-ensign** imports these exports and converts them into LoRA adapters (`ensign.py`).  
- The pipeline is defined in `plato-torch/ensign_pipeline.py`.  
- **Integration Point:** `Room.export_room()` → `Ensign.create_lora()`.  

**Gap:** None observed. This is the primary training→export flow.

---

## 2. **plato-torch** → **fleet-simulator**  
**Connection:** 🔄 **Indirect / Data-Only**  
- **fleet-simulator** generates synthetic training data (agent interactions, environment states).  
- **plato-torch** can consume this data as training tiles via `TileLoader`.  
- No direct API coupling found. Integration is file-based (JSON/msgpack).  
- **Integration Point:** `fleet-simulator/output/` → `plato-torch/data/`.  

**Gap:** No live streaming from simulator to training room. Simulated data must be written to disk first.

---

## 3. **holodeck-rust** → **plato-torch**  
**Connection:** 🔄 **Planned / Not Yet Implemented**  
- **holodeck-rust** is a MUD server with sentiment-aware NPCs (`npc_sentiment.rs`).  
- **plato-torch** has a `HolodeckRoom` preset (in `presets/`), but it's a stub.  
- The intent is to feed NPC interactions as training tiles, but no active socket connection exists.  
- **Integration Point:** `holodeck-rust` WebSocket server → `HolodeckRoom` client (unbuilt).  

**Gap:** Missing WebSocket client in plato-torch to ingest live holodeck events.

---

## 4. **fleet-simulator** → **holodeck-rust**  
**Connection:** ❌ **No Direct Connection Found**  
- **fleet-simulator** simulates fleet agents in abstract environments.  
- **holodeck-rust** is a standalone MUD with its own NPC AI.  
- No shared state, no data exchange.  
- **Potential Integration:** Simulator could generate NPC backstories or quest lines for holodeck.

**Gap:** These systems operate in separate domains (simulation vs. interactive narrative).

---

## 5. **plato-ensign** → **holodeck-rust**  
**Connection:** ❌ **No Connection Found**  
- **plato-ensign** exports LoRA adapters for specialist models.  
- **holodeck-rust** does not currently load external AI models; NPC logic is hard-coded in Rust.  
- **Potential Integration:** Load a trained ensign (LoRA) to control NPC behavior dynamically.

**Gap:** holodeck-rust lacks a model inference runtime (e.g., ONNX, llama.cpp integration).

---

## 6. **plato-torch** → **holodeck-rust** (via GhostInjector)  
**Connection:** ⚠️ **Conceptual / In Design**  
- **GhostInjector** (mentioned in `FLEET-CONTEXT.md`) is meant to inject "ghost tiles" (dead agents' lessons) into live systems.  
- **holodeck-rust** could receive these as NPC memory injections.  
- No GhostInjector implementation found in current repos.  

**Gap:** GhostInjector is not yet built.

---

## 7. **plato-torch** → **plato-relay**  
**Connection:** 🔄 **Indirect / Via Shared Tiles**  
- **plato-relay** (not listed in key technologies but referenced in `Task Board`) appears to be a message bus for tile distribution.  
- **plato-torch** rooms could publish tiles to a relay for multi-room training.  
- No relay client code found in plato-torch.  

**Gap:** plato-relay integration is pending (mentioned in backlog).
