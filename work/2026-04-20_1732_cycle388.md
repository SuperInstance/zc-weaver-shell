## 1. plato-torch

**Location:** `plato-torch/`  
**Purpose:** Training room presets, self‑training rooms, tile processing.

**Connections found:**

- **To plato‑ensign:**  
  - `plato-torch/src/rooms/deadband_room.py` imports `from plato_ensign import EnsignExporter`.  
  - `DeadbandRoom` has an `export_to_ensign()` method that calls `EnsignExporter.save_lora_adapter()`.  
  - This is a **direct, functional integration** — rooms can export their trained LoRA adapters.

- **To fleet‑simulator:**  
  - No direct imports found in `plato-torch` source files.  
  - However, `plato-torch/config/training_presets.yaml` includes a preset named `fleet_simulator_scenario` that references `"data_source": "fleet-simulator/output/traces/"`.  
  - This is a **configuration‑level integration** — plato‑torch can load training data generated by the simulator.

- **To holodeck‑rust:**  
  - No imports or configuration references found.  
  - **Integration gap:** No direct link between plato‑torch rooms and holodeck’s sentiment‑aware NPCs.

## 2. fleet‑simulator

**Location:** `fleet-simulator/`  
**Purpose:** Generate training traces by simulating fleet interactions.

**Connections found:**

- **To plato‑torch:**  
  - `fleet-simulator/src/output/mod.rs` writes JSONL traces to `output/traces/`.  
  - The format matches the `fleet_simulator_scenario` preset in plato‑torch.  
  - This is a **one‑way data flow** — simulator → plato‑torch training data.

- **To holodeck‑rust:**  
  - No imports or shared data structures found.  
  - **Integration gap:** Simulator does not feed scenarios into holodeck.

- **To plato‑ensign:**  
  - No references found.  
  - **Integration gap:** Simulator traces are not directly used for ensign export.
