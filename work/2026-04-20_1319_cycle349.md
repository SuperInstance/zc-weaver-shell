## 1. plato-torch (Training Rooms)

**Key Files:**  
- `plato-torch/src/rooms/deadband_room.py` — Primary training room implementation  
- `plato-torch/src/rooms/__init__.py` — Room registry  
- `plato-torch/src/tile_server.py` — Receives and stores training tiles  

**Connections Found:**  
1. **To plato-ensign:**  
   - `deadband_room.py` imports `from plato_ensign import export_lora` (line 42)  
   - Room has `export_to_ensign()` method that calls `export_lora(room_state, adapter_path)`  
   - Configurable via `ENABLE_ENSIGN_EXPORT = True` in room settings  

2. **To fleet-simulator:**  
   - No direct imports found in plato-torch source  
   - However, `tile_server.py` accepts JSON tiles with `simulation_id` field (line 89)  
   - Tiles from fleet-simulator can be ingested if they match tile schema  

3. **To holodeck-rust:**  
   - No Rust imports or direct connections  
   - Potential indirect: holodeck could generate tiles that plato-torch ingests  

**Integration Gap:**  
- plato-torch doesn't call fleet-simulator APIs directly  
- No WebSocket or HTTP client to holodeck-rust  
- Room training is isolated unless tiles come from external sources

## 2. fleet-simulator (Fleet Simulation)

**Key Files:**  
- `fleet-simulator/src/simulation/engine.rs` — Core simulation loop  
- `fleet-simulator/src/output/tile_exporter.rs` — Exports simulation events as tiles  
- `fleet-simulator/Cargo.toml` — Dependency declarations  

**Connections Found:**  
1. **To plato-torch:**  
   - `tile_exporter.rs` has `pub fn export_as_tile(events: Vec<SimEvent>) -> Tile`  
   - Tile struct matches plato-torch's `Tile` schema (id, timestamp, agent_id, content, metadata)  
   - Exporter can write to file or send via HTTP POST to tile server  

2. **To holodeck-rust:**  
   - No direct imports in code  
   - Both are Rust projects; could share crates but don't currently  
   - Both use `serde` for JSON serialization (common dependency)  

3. **To plato-ensign:**  
   - No direct connection  
   - Simulation tiles would need to go through plato-torch first to become ensign adapters  

**Integration Gap:**  
- fleet-simulator runs independently  
- Tile export is one-way (simulation → tile)  
- No feedback loop from training rooms to simulation

## 4. plato-ensign (LoRA Exporter)

**Key Files:**  
- `plato-ensign/src/exporter.py` — Main export logic  
- `plato-ensign/src/lora_builder.py` — Creates LoRA adapters from room state  
- `plato-ensign/requirements.txt` — Python dependencies  

**Connections Found:**  
1. **From plato-torch:**  
   - Called by `deadband_room.export_to_ensign()`  
   - Receives `room_state` dict containing tile history and agent interactions  
   - Returns `adapter_path` where LoRA weights are saved  

2. **To fleet-simulator:**  
   - No connection — ensign only works with room state, not simulation data  

3. **To holodeck-rust:**  
   - No connection — Rust/Python boundary not crossed  

**Integration Gap:**  
- Only works with plato-torch room state  
- Cannot create ensigns from simulation or game data directly  
- One-way export (no import/update cycle)
