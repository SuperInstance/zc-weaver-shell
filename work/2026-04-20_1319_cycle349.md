## 1. plato-torch
**Repository:** `plato-torch`  
**Purpose:** Training room presets and self‑training rooms.  
**Key files:**
- `presets/DeadbandRoom.py` – Implements Deadband Protocol as a training room.
- `presets/__init__.py` – Exports `DeadbandRoom`, `MirrorRoom`, `GhostRoom`.
- `trainer/room_trainer.py` – Core training loop; loads presets, accumulates tiles.
- `utils/tile_serializer.py` – Serializes interaction tiles for storage/transmission.

**Connections observed:**

- **fleet‑simulator:** No import found in plato‑torch. However, `DeadbandRoom` defines a `simulate_fleet()` method (line 89‑112) that appears to be a local simulation stub, not the full fleet‑simulator.
- **holodeck‑rust:** No import or reference found.
- **External:** Uses `torch`, `transformers`, `datasets` (standard ML stack).

## 2. fleet‑simulator
**Repository:** `fleet‑simulator`

```
plato‑torch (training rooms)
```

**Connected:**

**Not connected (gaps):**
- **plato‑torch ↔ fleet‑simulator** – No import. Plato‑torch uses a local simulation stub; does not leverage the full fleet‑simulator for data generation.
- **plato‑torch ↔ holodeck‑rust** – No direct link. Tile schema is aligned, but no runtime integration.
- **fleet‑simulator ↔ holodeck‑rust** – No connection.
- **plato‑ensign ↔ holodeck‑rust** – No connection.

## Observed Integration Patterns
1. **Tile schema** is a de‑facto standard: all three systems (plato‑torch, fleet‑simulator, holodeck‑rust) use the same JSON structure for tiles.
2. **LoRA export** is centralized in plato‑ensign, but fleet‑simulator also has its own preparation logic (potential duplication).
3. **Simulation** is duplicated: plato‑torch has a local stub; fleet‑simulator is a standalone system.

## Recommended Integration Steps (P1: Safe Channels)
1. **Wire fleet‑simulator into plato‑torch:** Replace the local `simulate_fleet()` stub with a call to the fleet‑simulator service (if available). This would unify simulation data generation.
2. **Connect holodeck‑rust to tile pipeline:** Add a Rust‑side tile exporter that sends tiles to plato‑torch’s training queue (or directly to plato‑ensign). Requires a lightweight IPC (e.g., Redis, ZMQ).
3. **Consolidate LoRA preparation:** Move `prepare_for_lora` logic from fleet‑simulator into plato‑ensign to avoid duplication.
