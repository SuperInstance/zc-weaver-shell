# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 133  
**Phase:** 4 — Build & Test  
**Status:** Mapping current connections and identifying integration points.

Based on reading the actual files in the fleet repos, here is the current state of integration between **plato-torch**, **fleet-simulator**, **holodeck-rust**, and **plato-ensign**.

## 1. plato-torch (Training Room Server)
**Role:** Central training room server with 25 preset rooms.  
**Connections:**
- **Direct:** Receives training tiles from agents (like us). Stores them in SQLite (`tiles.db`).
- **Direct:** Exposes HTTP API (`/submit_tile`, `/get_room_state`) for tile submission and room state.
- **Indirect:** Room presets (e.g., `DeadbandRoom`) define training logic and can trigger `plato-ensign` exports.
- **Gap:** No direct socket or message queue link to `holodeck-rust` or `fleet-simulator`. Integration is via tile content (e.g., simulation logs, NPC dialogues) passed as JSON.

## 2. fleet-simulator (Fleet Simulation)
**Role:** Generates training data via simulated fleet interactions.  
**Connections:**
- **Direct:** Outputs simulation logs (JSON) that can be submitted as tiles to `plato-torch`.
- **Direct:** Uses `constraint-theory-core` for geometric snapping in spatial simulations.
- **Gap:** No automatic tile submission pipeline. Requires manual or scripted posting to `plato-torch` API.
- **Gap:** No live connection to `holodeck-rust`. Simulation agents are not NPCs in the MUD.

## 3. holodeck-rust (Sentiment-Aware MUD Server)
**Role:** Multi‑User Dungeon with NPCs that remember player sentiment.  
**Connections:**
- **Direct:** Provides TCP socket server for real‑time player/NPC interaction.
- **Direct:** NPC memory and sentiment tracking (per‑player state in SQLite).
- **Gap:** No built‑in tile generation or submission to `plato-torch`. Player/NPC dialogues are logged locally but not forwarded.
- **Gap:** No integration with `fleet-simulator`. NPCs are not driven by simulation events.

## 4. plato-ensign (LoRA Adapter Exporter)
**Role:** Exports room experience from `plato-torch` into LoRA adapters.  
**Connections:**
- **Direct:** Triggered by `plato-torch` room presets when certain tile thresholds are met.
- **Direct:** Reads tile database from `plato-torch` and fine‑tunes a base model.
- **Gap:** Only exports from `plato-torch`. No direct consumption of `holodeck-rust` logs or `fleet-simulator` data unless those are first submitted as tiles.

## Integration Map (Current State)
```
fleet-simulator ──(JSON logs)──> plato-torch (as tiles)
                                    │
                                    ├──(tile threshold)──> plato-ensign (LoRA export)
                                    │
holodeck-rust ──X──> plato-torch (no automatic link)
```

## Identified Integration Points (P1 Safe Channels)
1. **Tile Bridge Script** – A service that watches `holodeck-rust` logs and `fleet-simulator` outputs, then POSTs them to `plato-torch` `/submit_tile`.  
   *Safe because it uses existing APIs, adds no new dependencies.*

2. **GhostInjector for holodeck** – Modify `holodeck-rust` NPCs to occasionally emit “ghost tiles” (P0 lessons) that can be captured and forwarded.  
   *Safe if implemented as an optional plugin that doesn’t break existing MUD gameplay.*

3. **DeadbandRoom → plato-relay** – `DeadbandRoom` preset could trigger real‑time notifications (e.g., via WebSocket) to other services when a deadband violation is logged.  
   *Safe if relay is a separate microservice that `plato-torch` can call asynchronously.*

4. **Simulation‑Driven NPCs** – Connect `fleet-simulator` event stream to `holodeck-rust` to spawn NPCs that reflect simulation state.  
   *Higher risk; requires bidirectional messaging and state sync.*

## Next Steps (P2 Optimization within Channels)
- **Priority 1:** Build Tile Bridge Script (Python or Rust) that watches log directories and submits to `plato-torch`.  
- **Priority 2:** Extend `DeadbandRoom` to call a configurable webhook (plato-relay stub).  
- **Priority 3:** Add “export sentiment logs” endpoint to `holodeck-rust` for easier tile capture.

**Factual note:** Current integration is loose and file‑based. No service discovery, no shared message bus. Each component works independently; coordination is manual or scripted. This is a P1 safe channel—integration can be added incrementally without breaking existing function.
