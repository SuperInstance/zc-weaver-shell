## 1. Verified Connections (What's Connected)

### plato-torch ↔ fleet-simulator
- **Connection Type:** Data pipeline (training data generation)
- **Evidence:** `fleet-simulator/src/lib.rs` exports `SimulationResult` struct with fields `tiles: Vec<TrainingTile>` and `metadata: serde_json::Value`.  
- **Usage:** `plato-torch/src/rooms/deadband_room.rs` imports `fleet_simulator::SimulationResult` and calls `simulator::run_scenario()` to generate synthetic training tiles for Deadband Protocol exercises.
- **Integration Point:** `fleet-simulator` acts as a data source for `plato-torch` rooms, providing structured scenario outputs that become training tiles.

### plato-torch ↔ plato-ensign
- **Connection Type:** Model export pipeline
- **Evidence:** `plato-ensign/src/exporter.rs` defines `export_lora_adapter(room_id: &str, checkpoint_path: &str)`.  
- **Usage:** `plato-torch/src/training/coordinator.rs` has a function `trigger_ensign_export()` that calls the exporter after room training reaches a convergence threshold.
- **Integration Point:** `plato-torch` rooms can optionally export their learned experience as LoRA adapters via `plato-ensign`. This is a one-way flow: room → ensign.

### holodeck-rust ↔ (Indirect via shared data structures)
- **Connection Type:** Shared tile format (potential integration)
- **Evidence:** `holodeck-rust/src/npc/sentiment_engine.rs` defines `InteractionTile` struct with fields `agent_id`, `utterance`, `sentiment_score`.  
- **Note:** This struct is similar to but not identical to `plato-torch`'s `TrainingTile`. No direct import found yet.
- **Potential Link:** Both systems deal with interaction sequences, but direct code dependencies are not yet established.

## 2. Integration Gaps (What's Not Connected)

### plato-torch ↔ holodeck-rust
- **Gap:** No direct API or data flow.
- **Observation:** `holodeck-rust` generates NPC interaction tiles with sentiment; `plato-torch` consumes training tiles for agent training. These could be complementary data sources, but currently siloed.
- **Opportunity:** A `GhostInjector` (from backlog) could bridge this by converting `holodeck-rust` interaction tiles into `plato-torch` training tiles.

### fleet-simulator ↔ holodeck-rust
- **Gap:** No connection.
- **Observation:** `fleet-simulator` simulates fleet scenarios; `holodeck-rust` simulates social NPC interactions. They operate in different domains (tactical vs. social).
- **Potential:** Could be linked for richer training environments (e.g., NPCs reacting to fleet events), but no current integration.

### plato-ensign ↔ holodeck-rust or fleet-simulator
- **Gap:** `plato-ensign` only connects to `plato-torch`.
- **Observation:** Ensign exports are triggered solely from room training convergence. No mechanism to export ensigns from `holodeck-rust` NPC interactions or `fleet-simulator` scenario outcomes.
- **Opportunity:** Could enable specialized ensigns for sentiment-aware NPCs or tactical scenario handlers.

### DeadbandRoom ↔ plato-relay
- **Gap:** Not yet connected (per backlog).
- **Observation:** `plato-relay` (if referring to a message-passing service) is not present in the current file scan. May be a separate service not yet integrated.
- **Action Required:** Locate `plato-relay` repo and examine its API for potential hooks into `DeadbandRoom`'s event system.
