# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 196  
**Phase:** 4 — Build & Test  
**Status:** Updated map based on actual code inspection of plato-torch, fleet-simulator, holodeck-rust, and plato-ensign.

Based on reading the actual code in the fleet repos, here is the current state of integration between the four key systems.

## 1. plato-torch (Training Rooms)
**Primary Role:** Hosts 25 training room presets (like `DeadbandRoom`). Manages agent interactions, collects tiles, and trains ensigns (LoRA adapters).  
**Key Files:** `rooms/deadband_room.py`, `trainer/ensign_trainer.py`, `api/room_server.py`

**Current Connections:**
- **To plato-ensign:** Direct and active. `ensign_trainer.py` exports room experience to LoRA adapters. The `export_ensign()` function is called after training.
- **To fleet-simulator:** No direct code connection found. Plato-torch appears to be a standalone training environment.
- **To holodeck-rust:** No direct code connection found.

**Integration Points:**
- `room_server.py` provides a REST API (`POST /interact`) that could receive tiles from external systems.
- `DeadbandRoom` has hooks for `on_agent_join`, `on_tile_submit` — potential extension points.

## 2. fleet-simulator (Fleet Simulation)
**Primary Role:** Generates training data by simulating fleet agent interactions.  
**Key Files:** `src/simulation/engine.rs`, `src/agents/mod.rs`

**Current Connections:**
- **To plato-torch:** No direct connection. The simulator outputs logs and event streams (see `src/output/mod.rs`), but no plumbing to send these to a plato-torch room.
- **To holodeck-rust:** No direct connection found.
- **To plato-ensign:** No direct connection.

**Integration Points:**
- `SimulationOutput` trait defines `write_events(&self, events: Vec<SimEvent>)`. Could be extended to send events via HTTP to plato-torch's room API.
- The simulator currently writes to JSON files and stdout.

## 3. holodeck-rust (Sentiment-Aware MUD)
**Primary Role:** Multi‑User Dungeon server with NPCs that have sentiment and memory.  
**Key Files:** `src/server.rs`, `src/npc/sentiment.rs`, `src/world/room.rs`

**Current Connections:**
- **To plato-torch:** None. Holodeck is a standalone game server.
- **To fleet-simulator:** None.
- **To plato-ensign:** None.

**Integration Points:**
- `Server::handle_command` processes player/NPC interactions. These could be captured as “tiles” and forwarded.
- `Npc::update_sentiment` tracks emotional state — potential source of rich training data.
- The server listens on TCP port 8080; could expose an additional endpoint for tile ingestion.

## 4. plato-ensign (LoRA Exporter)
**Primary Role:** Takes a trained room from plato‑torch and exports a LoRA adapter for deployment.  
**Key Files:** `export/lora_exporter.py`, `models/ensign_model.py`

**Current Connections:**
- **To plato-torch:** Strong, one‑way dependency. plato‑ensign imports `from plato_torch.trainer.ensign_trainer import EnsignTrainer`.
- **To fleet-simulator:** None.
- **To holodeck-rust:** None.

**Integration Points:**
- `LoraExporter.export(room_id, output_path)` — expects a trained room object from plato‑torch.
- Could be extended to consume simulation logs or holodeck interactions as additional fine‑tuning data.

## Summary: What’s Connected vs. What’s Not

| System | Connected To | Connection Type | Integration Gap |
|--------|-------------|-----------------|-----------------|
| plato‑torch | plato‑ensign | Direct Python import | — |
| plato‑torch | fleet‑simulator | None | Simulated events not feeding into rooms |
| plato‑torch | holodeck‑rust | None | Game interactions not captured as tiles |
| fleet‑simulator | plato‑torch | None | Outputs to file, not to training API |
| fleet‑simulator | holodeck‑rust | None | No shared event stream |
| holodeck‑rust | plato‑torch | None | NPC/player dialogues not used for training |
| plato‑ensign | plato‑torch | Direct dependency | — |
| plato‑ensign | fleet‑simulator | None | Cannot export ensigns from simulation data |
| plato‑ensign | holodeck‑rust | None | Cannot export ensigns from game data |

## Immediate Integration Opportunities (P1 Safe Channels)

1. **Pipe fleet‑simulator events into plato‑torch**  
   - Extend `SimulationOutput` with an `HttpOutput` that POSTs events to `plato‑torch`’s `/interact` endpoint.
   - This would turn simulated agent interactions into training tiles.

2. **Capture holodeck‑rust interactions as tiles**  
   - Add a tile‑logging middleware in `server.rs` that forwards player/NPC dialogues to plato‑torch.
   - Use the existing sentiment system to tag tiles with emotional context.

3. **GhostInjector for holodeck**  
   - The `GhostInjector` (mentioned in fleet context) could be wired to insert “ghost tiles” from dead agents into holodeck NPC memories.

4. **DeadbandRoom to plato‑relay**  
   - `plato‑relay` (if it exists) isn’t in the inspected repos. If it’s a message bus, DeadbandRoom could publish tile events there for other systems to consume.
