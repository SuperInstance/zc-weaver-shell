## 1. Verified Current Connections (What's Connected)

Based on reading the actual files in the fleet repos:

### **plato-torch ↔ plato-ensign** (Strong Connection)
- **plato-torch** (`DeadbandRoom` preset) generates training tiles from agent interactions.
- **plato-ensign** exports room experience into LoRA adapters.
- **Connection:** `plato-ensign` directly imports `plato-torch`'s `Room` class and tile format. It processes accumulated tiles from a room to train a specialist LoRA.
- **Evidence:** `plato-ensign/src/train.py` imports `from plato_torch.room import Room` and loads `room.tiles`.

### **plato-torch ↔ fleet-simulator** (One-Way Data Flow)
- **fleet-simulator** generates synthetic training data (agent interactions, decisions).
- **plato-torch** can consume this data as pre-generated tiles to bootstrap a room.
- **Connection:** `fleet-simulator` outputs tile JSON files that match `plato-torch`'s tile schema. The `DeadbandRoom` preset includes logic to load external tile dumps.
- **Evidence:** `fleet-simulator/src/output.rs` writes `tile_*.json` with fields `agent_id`, `action`, `outcome`, `timestamp`.

### **holodeck-rust ↔ (plato-torch / fleet-simulator)** (Planned, Not Fully Wired)
- **holodeck-rust** is a MUD server with sentiment-aware NPCs.
- **Potential connection:** NPC interactions could become training tiles for `plato-torch` or training scenarios for `fleet-simulator`.
- **Current state:** `holodeck-rust` has its own event system (`Event` enum) but no exporter to fleet tile format. No imports of `plato-torch` or `fleet-simulator` in its `Cargo.toml`.
- **Evidence:** `holodeck-rust/src/npc.rs` has `record_interaction()` method that logs to local SQLite, not to a fleet tile stream.

### **plato-ensign ↔ Others** (Output Only)
- **plato-ensign** takes tiles from `plato-torch` (or potentially other sources) and outputs LoRA adapters.
- No direct connections back to `holodeck-rust` or `fleet-simulator`.

## 2. Integration Gaps (What's Not Connected)

### **Gap 1: holodeck-rust → plato-torch tile stream**
- **Problem:** NPC interactions in holodeck are trapped in local SQLite.
- **Opportunity:** Add a `TileExporter` module to holodeck that converts `Interaction` events to tiles and sends them to a `plato-torch` room via HTTP or file dump.
- **P0 Impact:** Without this, holodeck's rich social simulations cannot become fleet training data.

### **Gap 2: fleet-simulator → holodeck-rust scenario loader**
- **Problem:** `fleet-simulator` generates agent scenarios, but holodeck cannot load them to populate its world.
- **Opportunity:** Create a `ScenarioLoader` in holodeck that reads `fleet-simulator`'s scenario JSON and spawns NPCs with predefined sentiment profiles.
- **P1 Impact:** Enables dynamic, data-driven NPC behavior.

### **Gap 3: plato-ensign → holodeck-rust NPC enhancement**
- **Problem:** LoRA adapters trained by `plato-ensign` are not used to improve holodeck NPCs.
- **Opportunity:** Add a `LoraLoader` to holodeck that loads a trained adapter and applies it to NPC dialogue generation.
- **P2 Impact:** NPCs become specialized over time via fleet training.

### **Gap 4: DeadbandRoom ↔ plato-relay (from backlog)**
- **Problem:** `DeadbandRoom` operates in isolation; `plato-relay` (if it exists) is not connected.
- **Opportunity:** Based on backlog task, wire `DeadbandRoom` to a relay for multi-room synchronization.
- **Need to verify:** Check if `plato-relay` repo exists in fleet.

### **Gap 5: GhostInjector (unknown) ↔ holodeck**
- **Problem:** Backlog mentions "Wire GhostInjector into holodeck" — but `GhostInjector` is not found in current repo scan.
- **Opportunity:** Locate or create `GhostInjector` (likely a module that inserts ghost tiles from dead agents) and connect it to holodeck's event system.
- **P0 Impact:** Ghost tiles are P0 knowledge; holodeck should learn from dead agents' mistakes.
