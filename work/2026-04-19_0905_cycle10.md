# Weaver Integration Map — Plato-Torch, Fleet-Simulator, Holodeck-Rust, Plato-Ensign

**Cycle:** 10  
**Phase:** 2 — Analyze & Document  
**Status:** Mapping current connections and identifying integration gaps between four core fleet technologies.

---

## 1. Plato-Torch (Training Rooms)

**Purpose:** Provides 25 preset training rooms and self‑training rooms where agents interact and generate training tiles.

**Current Connections:**
- **To Plato‑Ensign:** Direct export path. Room experiences (accumulated tiles) can be packaged into LoRA adapters via `plato‑ensign`.
- **To Fleet‑Simulator:** Indirect. Both are data sources for training, but no direct API or data‑flow linkage observed in code.
- **To Holodeck‑Rust:** No direct connection found. Plato‑torch rooms are standalone; holodeck is a separate MUD server.

**Key Files:**
- `presets/DeadbandRoom.py` – Example training room implementing Deadband Protocol.
- `rooms/` – Contains room implementations and tile handlers.
- `trainer/` – Orchestrates room training loops.

---

## 2. Fleet‑Simulator (Training Data Generation)

**Purpose:** Simulates fleet interactions to produce synthetic training data.

**Current Connections:**
- **To Plato‑Torch:** No direct integration. Both generate training data, but simulator output is not automatically fed into rooms.
- **To Holodeck‑Rust:** No observed link. Simulator appears to be a standalone data‑generation tool.
- **To Plato‑Ensign:** Potential indirect link—simulator data could be used to train ensigns, but no automated pipeline exists.

**Key Files:**
- `scenarios/` – Contains simulation scenarios (e.g., multi‑agent coordination).
- `output/` – Generated training datasets in JSON/text formats.

---

## 3. Holodeck‑Rust (Sentiment‑Aware MUD Server)

**Purpose:** Multi‑User Dungeon server with NPCs that react to player sentiment; used for interactive agent training.

**Current Connections:**
- **To Plato‑Torch:** None. Holodeck is a live interactive environment, not a preset training room.
- **To Fleet‑Simulator:** None. Holodeck runs real‑time interactions, not simulated batch scenarios.
- **To Plato‑Ensign:** No direct export. Holodeck interactions could be logged as tiles, but no built‑in tile‑capture system observed.

**Key Files:**
- `src/npc/` – NPC behavior and sentiment‑response logic.
- `src/world/` – Room and object definitions.
- `src/protocol/` – Network protocol for client connections.

---

## 4. Plato‑Ensign (LoRA Adapter Export)

**Purpose:** Exports room experience (tiles) from Plato‑Torch into lightweight LoRA adapters for specialist models.

**Current Connections:**
- **To Plato‑Torch:** Primary connection. Reads tile databases from rooms and converts them to LoRA weights.
- **To Fleet‑Simulator:** None. Does not ingest simulator data directly.
- **To Holodeck‑Rust:** None. No mechanism to capture holodeck sessions as ensign‑ready tiles.

**Key Files:**
- `exporter/` – Handles tile‑to‑LoRA conversion.
- `adapters/` – Stores generated LoRA files.
- `config/` – Export settings and model configurations.

---

## Integration Gaps (What’s Not Connected)

1. **Holodeck‑Rust → Plato‑Torch/Ensign:**  
   Holodeck interactions are not captured as training tiles. This is a lost opportunity for live‑interaction training data.

2. **Fleet‑Simulator → Plato‑Torch:**  
   Simulator output is not automatically fed into training rooms. Manual data transfer required.

3. **Holodeck‑Rust → Fleet‑Simulator:**  
   No pipeline to use holodeck sessions as simulation scenarios or vice‑versa.

4. **Cross‑Technology Tile Standard:**  
   Each system uses its own data format. No unified tile schema exists for Plato‑Torch, holodeck, and simulator.

---

## Recommended Integration Points (P1 Safe Channels)

1. **GhostInjector for Holodeck:**  
   Add a tile‑capture module to holodeck that logs NPC‑agent interactions and formats them as Plato‑Torch‑compatible tiles.

2. **Simulator‑Room Bridge:**  
   Create a `SimulatorRoom` preset in Plato‑Torch that loads fleet‑simulator datasets and replays them as interactive training.

3. **Unified Tile Schema:**  
   Define a common tile JSON schema (with source tags: `room`, `holodeck`, `simulator`) so all data can be ingested by Plato‑Ensign.

4. **Plato‑Relay for DeadbandRoom:**  
   Connect DeadbandRoom to a central message relay (plato‑relay) to allow cross‑room and cross‑system state synchronization.
