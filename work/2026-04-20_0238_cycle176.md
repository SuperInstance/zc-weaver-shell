# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 176  
**Phase:** 4 — Build & Test  
**Status:** Mapping current connections and identifying integration points.

Based on reading the actual files in the fleet repos, here is the current state of integration between plato-torch, fleet-simulator, holodeck-rust, and plato-ensign.

## 1. Verified Connections (What IS Connected)

### plato-torch ↔ fleet-simulator
- **Connection:** Strong, direct integration via `fleet-simulator` as a data source.
- **Evidence:** `plato-torch/rooms/deadband_room.py` imports `fleet_simulator` and uses `FleetSimulator` class to generate training episodes.
- **Data Flow:** `DeadbandRoom` calls `simulator.generate_episode()` to create simulation episodes, which become training tiles.
- **Purpose:** Provides synthetic fleet interaction data for training rooms.

### plato-torch ↔ plato-ensign
- **Connection:** Direct export pipeline.
- **Evidence:** `plato-torch/export/ensign_exporter.py` exists and is referenced in room training loops.
- **Data Flow:** After a room trains on accumulated tiles, `ensign_exporter` can package the room's experience (weights, adapter config) into a LoRA adapter.
- **Purpose:** Turns trained room state into portable specialist models (ensigns).

### fleet-simulator ↔ (indirect) holodeck-rust
- **Connection:** Conceptual/thematic alignment, but no direct code integration observed.
- **Evidence:** Both simulate multi-agent interactions. `fleet-simulator` focuses on fleet coordination training; `holodeck-rust` is a sentiment-aware MUD server for NPCs.
- **Data Flow:** None found in current code. Potential channel exists if simulation events could populate a MUD world.

## 2. Integration Gaps (What Is NOT Connected)

### plato-torch ↔ holodeck-rust
- **Gap:** No direct connection found.
- **Observation:** `holodeck-rust` is a standalone Rust MUD server. No Python bindings or HTTP API observed in its `README.md` or source structure.
- **Potential Channel:** Could be integrated if `holodeck-rust` exposed a socket or REST API for injecting events/characters generated from plato-torch training episodes.

### fleet-simulator ↔ plato-ensign
- **Gap:** No direct export from simulator to ensign.
- **Observation:** `fleet-simulator` generates episodes for training, but doesn't create ensigns directly. That's `plato-torch`'s role after training.
- **Potential Channel:** Could bypass room training if simulator episodes were formatted as direct LoRA training data.

### holodeck-rust ↔ plato-ensign
- **Gap:** Complete separation.
- **Observation:** `holodeck-rust` is a game server; `plato-ensign` is a PyTorch LoRA exporter. No shared interfaces.
- **Potential Channel:** If NPC sentiment models in holodeck were adapted via LoRA, could export them as ensigns.
