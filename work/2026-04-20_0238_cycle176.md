# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 176  
**Phase:** 4 — Build & Test  
**Status:** Mapping current connections and identifying integration points.

Based on reading the actual files in the fleet repos, here is the current state of integration between **plato-torch**, **fleet-simulator**, **holodeck-rust**, and **plato-ensign**.

## 1. Verified Connections (What's Connected)

### plato-torch ↔ fleet-simulator
- **Connection:** Strong, bidirectional data flow.
- **Evidence:** `plato-torch/rooms/deadband_room.py` imports `fleet_simulator` and uses `FleetSimulator` class to generate training episodes.
- **Data Flow:** 
  - `DeadbandRoom` calls `simulator.generate_episode()` to produce simulation trajectories.
  - These trajectories become training tiles stored in the room's memory.
- **Integration Point:** `DeadbandRoom` is preset #25 in plato-torch, explicitly designed to consume fleet-simulator output.

### plato-torch ↔ plato-ensign
- **Connection:** Direct export pipeline.
- **Evidence:** `plato-ensign/export.py` contains `export_room_to_lora()` function that reads from a plato-torch room directory.
- **Data Flow:**
  - Room accumulates interaction tiles (`.jsonl` files).
  - plato-ensign processes these tiles into LoRA adapter weights.
- **Integration Point:** Command-line interface: `python -m plato_ensign.export --room_path ./rooms/deadband_room`.

### fleet-simulator → plato-torch (indirect)
- **Connection:** Training data provider.
- **Evidence:** `fleet-simulator/README.md` states: "Generates training episodes for PLATO rooms."
- **Data Flow:** Simulation episodes → JSON trajectories → plato-torch room memory.

## 2. Integration Gaps (What's Not Connected)

### holodeck-rust ↔ plato-torch
- **Gap:** No direct connection found.
- **Evidence:** 
  - `holodeck-rust/src/` contains no Python bindings or HTTP API for external systems.
  - `plato-torch` has no imports or references to holodeck-rust.
- **Potential Integration Point:** Could connect via:
  1. Holodeck NPC interactions exported as training tiles.
  2. plato-torch rooms consuming holodeck session logs.

### holodeck-rust ↔ fleet-simulator  
- **Gap:** Separate simulation domains.
- **Evidence:** 
  - `fleet-simulator` simulates fleet decision-making.
  - `holodeck-rust` is a text-based MUD with sentiment-aware NPCs.
- **Potential Integration Point:** Fleet agents could interact with holodeck NPCs as part of training scenarios.

### holodeck-rust ↔ plato-ensign
- **Gap:** No export path from holodeck to ensign.
- **Evidence:** `plato-ensign` only reads from plato-torch room directories.
- **Potential Integration Point:** Holodeck session logs → training tiles → ensign adapter.
