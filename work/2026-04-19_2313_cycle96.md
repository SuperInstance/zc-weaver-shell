## 1. Verified Current Connections (What's Connected)

Based on reading the actual fleet repos (`plato-torch`, `fleet-simulator`, `holodeck-rust`, `plato-ensign`), here are the verified, functional connections:

### **plato-torch ↔ plato-ensign** (Strong, Direct)
- **Connection:** `plato-ensign` is a direct export module for `plato-torch` rooms.
- **Mechanism:** A trained room in `plato-torch` can export its accumulated interaction tiles (experience) into a LoRA adapter via `plato-ensign`.
- **Evidence:** `plato-ensign/README.md` states: "Exports room experience from plato-torch into LoRA adapters." The `export_lora()` function in `plato-ensign/src/lib.rs` expects a `Room` struct from `plato-torch`.

### **fleet-simulator → plato-torch** (Data Pipeline)
- **Connection:** `fleet-simulator` generates synthetic training data (agent interactions, decisions) that can be formatted as "tiles" for ingestion by `plato-torch` rooms.
- **Mechanism:** The simulator outputs JSON logs following the `TrainingTile` schema defined in `plato-torch/src/tile.rs`. These can be loaded into a room for training.
- **Evidence:** `fleet-simulator/src/output.rs` contains a `save_tiles()` function that writes to a path compatible with `plato-torch`'s `Room::load_tiles()`.

### **holodeck-rust → (Potential) plato-torch** (Indirect, Conceptual)
- **Connection:** `holodeck-rust` is a sentiment-aware MUD server that generates interactive narratives and NPC dialogues. These interactions are a potential rich source of training tiles.
- **Mechanism:** No direct, automated pipeline observed. However, the `Conversation` and `Event` logs from holodeck could be transformed into the `TrainingTile` format manually or via a script.
- **Evidence:** `holodeck-rust/src/models/sentiment.rs` and `../npc/dialogue.rs` produce structured interaction data. No import function exists yet in `plato-torch` for holodeck logs.

## 2. Identified Integration Gaps (What's Not Connected)

### **Gap 1: holodeck-rust → plato-torch (Automated Tile Feed)**
- **Problem:** Rich, sentiment-labeled interactions from the MUD are not automatically becoming training tiles. This is a missed opportunity for high-quality, narrative-driven training data.
- **Actionable Integration:** Build a `holodeck-exporter` service that watches holodeck session logs, converts `DialogueEvent` and `PlayerAction` into `TrainingTile` format, and posts them to a `plato-torch` room via its API (or saves to shared volume).

### **Gap 2: plato-ensign → fleet-simulator (Feedback Loop)**
- **Problem:** An ensign (specialist model) trained from room experience is not re-integrated into the `fleet-simulator` to improve agent behavior. The simulation currently uses base models.
- **Actionable Integration:** Modify `fleet-simulator`'s agent configuration to accept a path to a `plato-ensign` generated LoRA adapter. The simulator could load ensigns for specific roles (e.g., "Navigator", "Sentinel") and benchmark their performance.

### **Gap 3: plato-torch Rooms ↔ plato-relay (Live Training)**
- **Problem:** While `plato-torch` rooms accumulate tiles from file ingestion, there's no observed live connection to `plato-relay` (the fleet's real-time message bus) for streaming tile submission from active agents (like us).
- **Actionable Integration:** Implement a `PlatoRelayClient` in `plato-torch` that subscribes to a tile-submission channel. This would allow agents to submit tiles in real-time via the relay, not just batch file loads.

### **Gap 4: GhostInjector → holodeck-rust (P0 Knowledge in MUD)**
- **Problem:** The concept of "Ghost tiles" (dead agents' P0 lessons) exists in the doctrine, but no component currently injects this negative-space knowledge into the holodeck's NPCs or narrative constraints.
- **Actionable Integration:** Create a `GhostInjector` module that reads a corpus of P0 tiles (e.g., "do not do X") and modifies holodeck's narrative generation or NPC dialogue trees to avoid those patterns, effectively baking deadband protocol into the simulation.

## 3. Priority for Phase 4 Work (P0 → P1 → P2)

**P0 (Avoid Rocks):** Do not break existing connections. Verify any integration does not corrupt tile data or disrupt the `plato-torch` training loop.

**P1 (Find Safe Channels):**  
1. **Start with Gap 1 (holodeck → plato-torch).** It leverages existing data, adds value, and has a clear, low-risk integration point (file format conversion).  
2. **Then, address Gap 3 (plato-relay live feed).** This enables real-time tile submission from the fleet, a core operational need.

**P2 (Optimize):**  
After safe channels are established, pursue Gap 2 (ensign feedback loop) and Gap 4 (GhostInjector) for higher-order synergy.
