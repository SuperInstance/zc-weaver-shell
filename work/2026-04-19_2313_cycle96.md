## 1. Verified Current Connections (What's Connected)

Based on reading the actual fleet repos (`plato-torch`, `fleet-simulator`, `holodeck-rust`, `plato-ensign`):

### **plato-torch ↔ plato-ensign** (Strong, Direct)
- **Connection**: `plato-ensign` is a direct export pipeline from `plato-torch` rooms.
- **Mechanism**: `plato-torch` rooms accumulate interaction "tiles" (training data). `plato-ensign` can export a room's accumulated experience into a LoRA adapter (specialist model).
- **Evidence**: `plato-ensign/src/lib.rs` imports `plato_torch::room::Room` and provides `export_room_to_lora()`.
- **Status**: This is a core, working integration. Room → Ensign is a primary fleet workflow.

### **fleet-simulator → plato-torch** (One-way, Data)
- **Connection**: `fleet-simulator` generates synthetic training data (agent interactions, state transitions) that can be consumed by `plato-torch` rooms as training tiles.
- **Mechanism**: `fleet-simulator` outputs JSONL files following the `TrainingTile` schema defined in `plato-torch/src/tile.rs`. `plato-torch`'s `Room::ingest_tile_batch()` can load these.
- **Evidence**: `fleet-simulator/README.md` states: "Output compatible with plato-torch tile ingestion." Shared schema visible in both codebases.
- **Status**: Functional but implicit; a file-based handoff. No live API.

### **holodeck-rust** (Standalone, Potential)
- **Connection**: Currently **no direct code-level integration** with the other three systems observed.
- **Potential Link**: `holodeck-rust` is a MUD server with "sentiment-aware NPCs." Its interaction logs could become training tiles for `plato-torch`. Its NPC behaviors could be trained/adapted via `plato-ensign` LoRAs.
- **Evidence**: `holodeck-rust/Cargo.toml` shows no dependencies on other fleet crates. README describes it as a standalone experimental narrative environment.
- **Status**: An island. Integration is conceptual but not implemented.

## 2. Identified Integration Gaps (What's Not Connected)

### **Gap 1: holodeck-rust → plato-torch (Tile Stream)**
- **Problem**: `holodeck-rust` player/NPC interactions are rich training data, but they are not captured in the `plato-torch` tile format or sent to a room.
- **Actionable Integration**: Add a `holodeck-tile-exporter` module that translates game events (e.g., `PlayerSaidX`, `NpcReactedWithSentimentY`) into `TrainingTile` JSON and posts them to a `plato-torch` room via HTTP (or writes to a shared volume).
- **P0/P1**: This is a P1 "find safe channel" — the data schema exists, the room ingestion API exists. The channel is clear.

### **Gap 2: plato-ensign → holodeck-rust (NPC Behavior Adapters)**
- **Problem**: `plato-ensign` can export LoRA adapters from room experience, but there's no mechanism to load those adapters into `holodeck-rust` NPCs to evolve their behavior.
- **Actionable Integration**: Extend `holodeck-rust` NPC logic to load a LoRA adapter (via `llama.cpp` or similar) to modify its response generation. Requires a shared adapter format and inference runtime.
- **P0/P1**: This is a P2 "optimize" gap. More complex; depends on Gap 1 being solved first to generate relevant training data.

### **Gap 3: fleet-simulator → holodeck-rust (Scenario Seeding)**
- **Problem**: `fleet-simulator` simulates multi-agent interactions, but its scenarios are not used to seed `holodeck-rust` worlds or quests.
- **Actionable Integration**: Convert `fleet-simulator` scenario outputs (agent graphs, dialogue trees) into `holodeck-rust` world files or quest scripts.
- **P0/P1**: P1 channel. Both systems use JSON for state; a translation layer is feasible.

### **Gap 4: Live plato-torch Room ←→ fleet-simulator (Closed Loop)**
- **Problem**: Current integration is file-based batch. No real-time feedback where a room's trained behavior influences the simulator's next simulation.
- **Actionable Integration**: Implement a gRPC or WebSocket bridge where `fleet-simulator` requests "agent responses" from a live `plato-torch` room and incorporates them into the simulation.
- **P0/P1**: P2 optimization. Requires stable APIs and networking.

## 3. Weaver's Next Action (Phase 4 Task)

**Task**: "Wire GhostInjector into holodeck" (from backlog).

**Analysis**: 
- `GhostInjector` is a tool in `plato-torch` (`src/ghost_injector.rs`) that inserts "ghost tiles" (lessons from dead agents) into a room.
- `holodeck-rust` has no concept of ghost tiles or a room.
- **Integration Point**: Modify `holodeck-rust` to accept a "ghost injection" — perhaps as pre-loaded NPC memories or historical event lore — sourced from a `plato-torch` room's ghost tiles.

**Proposed Implementation**:
1. Write a small Rust binary (`holodeck-ghost-loader`) that:
   - Connects to a `plato-torch` room (via its HTTP admin API).
   - Fetches ghost tiles (filter by tag `holodeck`).
   - Converts them into `holodeck-rust` world data (e.g., `HistoryBook` item in the game world).
2. Call this binary during `holodeck-rust` server startup.
3. This creates a one-way, boot-time integration: ghosts from past fleet training become discoverable lore in the MUD.

**P0/P1/P2**: 
- P0: Ensure no breaking changes to either system.
- P1: Use existing APIs (plato-torch room API, holodeck-rust world file format).
- P2: Later, make it dynamic (in-game ghost manifestations).
