## 1. Verified Current Connections (What's Connected)

Based on reading the actual files in the fleet repos:

### **plato-torch → plato-ensign** (Strong, direct connection)
- **Connection:** `plato-ensign` is a direct export module of `plato-torch` room experience.
- **Evidence:** `plato-torch/README.md` states: "plato-ensign: Exports room experience into LoRA adapters."
- **Mechanism:** Rooms in `plato-torch` accumulate interaction tiles; `plato-ensign` can package those tiles into a trained LoRA adapter.
- **Status:** This is a core, documented pipeline. Likely functional but may need validation.

### **plato-torch → holodeck-rust** (Indirect, potential data flow)
- **Connection:** `holodeck-rust` is a MUD server with sentiment-aware NPCs. It could serve as a data source or training environment for `plato-torch` rooms.
- **Evidence:** Both are listed under "Key Technologies" as separate projects. No direct import/export code observed yet.
- **Mechanism:** NPC interactions in `holodeck-rust` could generate "interaction tiles" that feed into a `plato-torch` room for training.
- **Status:** Conceptual link; integration not yet wired.

### **fleet-simulator → plato-torch** (Data source connection)
- **Connection:** `fleet-simulator` generates training data for the fleet.
- **Evidence:** `fleet-simulator` description: "Fleet simulation for training data."
- **Mechanism:** Simulation logs or agent interactions could be formatted as tiles and ingested by a `plato-torch` room.
- **Status:** Likely a planned data pipeline; not yet implemented.

### **plato-torch → cudaclaw** (Runtime dependency)
- **Connection:** `cudaclaw` is a GPU-resident agent runtime.
- **Evidence:** Both are key technologies; `plato-torch` likely uses `cudaclaw` for GPU-accelerated training or inference.
- **Mechanism:** `cudaclaw` could host the trained ensigns (LoRA adapters) for low-latency execution.
- **Status:** Implied architectural dependency; integration specifics unclear.

## 2. Integration Gaps (What's Not Connected)

### **Gap 1: holodeck-rust → plato-torch tile ingestion**
- **Problem:** No established pipeline to send NPC interaction logs from `holodeck-rust` to a `plato-torch` room.
- **Impact:** `holodeck-rust` runs in isolation; its sentiment-aware NPC interactions are not captured as training tiles.
- **Actionable Step:** Design a `TileLogger` module for `holodeck-rust` that formats interactions and POSTs them to a `plato-torch` room API.

### **Gap 2: fleet-simulator → plato-torch data format mismatch**
- **Problem:** `fleet-simulator` output format unknown; `plato-torch` expects "interaction tiles" with specific schema.
- **Impact:** Simulation data cannot be directly used for training without transformation.
- **Actionable Step:** Examine `fleet-simulator` log output and define a transformer to tile schema.

### **Gap 3: plato-ensign → holodeck-rust NPC enhancement**
- **Problem:** Trained ensigns (LoRAs) from `plato-ensign` are not deployed back to `holodeck-rust` NPCs.
- **Impact:** NPCs cannot improve over time using fleet-trained adapters.
- **Actionable Step:** Create an `EnsignLoader` in `holodeck-rust` to load and apply LoRA adapters to NPC models.

### **Gap 4: Deadband Protocol ↔ plato-torch room presets**
- **Problem:** The Deadband Protocol (P0/P1/P2) is a fleet doctrine, but not explicitly encoded in `plato-torch` room logic.
- **Impact:** Rooms train without explicit deadband constraint shaping.
- **Actionable Step:** Implement a `DeadbandRoom` preset that uses P0/P1/P2 to filter or weight training tiles.

### **Gap 5: Ghost tiles → holodeck-rust memory**
- **Problem:** "Ghost tiles" (dead agents' lessons) are a fleet concept but not integrated into `holodeck-rust` NPC memory.
- **Impact:** NPCs cannot learn from historical fleet failures.
- **Actionable Step:** Add a `GhostTileMemory` component to `holodeck-rust` that injects P0 negative examples into NPC decision contexts.
