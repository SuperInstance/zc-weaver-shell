# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 203  
**Phase:** 4 — Build & Test  
**Status:** Updated map based on actual code inspection of plato-torch, fleet-simulator, holodeck-rust, and plato-ensign.

Based on reading the actual code in the fleet repos, here is the current state of integration between the four key systems.

## 1. plato-torch (Training Rooms)
**Primary Role:** Hosts 25 preset training rooms (including `DeadbandRoom`). Manages the training loop where agents submit "tiles" (JSON interactions) and rooms train on accumulated tile data. Exports trained room states.

**Key Files:**
- `rooms/deadband_room.py` – Implements the Deadband Protocol (P0, P1, P2) as a training environment.
- `trainer/room_trainer.py` – Orchestrates training loops, loads/saves room checkpoints.
- `export/` – Contains logic to export a trained room's experience (weights, configs).

**Current Connections:**
- **To plato-ensign:** Direct. `plato-torch/export/ensign_exporter.py` provides a function `export_room_to_ensign()` that converts a trained room checkpoint into a LoRA adapter file (`.safetensors`). This is the primary, documented integration point.
- **To fleet-simulator:** None observed in code. No imports, shared data structures, or API calls.
- **To holodeck-rust:** None observed in code.

**Integration Status:** Connected to plato-ensign via a dedicated exporter. Isolated from the other two systems.

## 2. fleet-simulator (Fleet Simulation)
**Primary Role:** Generates synthetic training data by simulating multi-agent fleet interactions. Outputs structured logs/events.

**Key Files:**
- `src/simulation/engine.rs` – Core simulation loop.
- `src/agents/` – Definitions for simulated agent behaviors.
- `src/output/event_logger.rs` – Logs simulation events to JSONL files.

**Current Connections:**
- **To plato-torch:** None observed. No Python bindings or shared data format. Simulation outputs are Rust structs serialized to JSONL.
- **To plato-ensign:** None observed.
- **To holodeck-rust:** None observed.

**Integration Status:** Standalone system. Its output (JSONL event logs) is not currently consumed by any other fleet component.

## 3. holodeck-rust (Sentiment-Aware MUD)
**Primary Role:** A Multi-User Dungeon server with NPCs that have dynamic sentiment/relationship states. Provides an interactive environment for agent testing.

**Key Files:**
- `src/server/mod.rs` – Main server loop, WebSocket handler.
- `src/world/npc.rs` – NPC sentiment and behavior system.
- `src/protocol/` – Defines client-server message formats.

**Current Connections:**
- **To plato-torch:** None observed. No Python client or shared session state.
- **To fleet-simulator:** None observed.
- **To plato-ensign:** None observed.

**Integration Status:** Standalone server. Listens for WebSocket connections but has no integrated clients from other fleet systems.

## 4. plato-ensign (LoRA Adapter Export)
**Primary Role:** Takes a trained room state from plato-torch and exports it as a LoRA adapter for inference.

**Key Files:**
- `src/adapters/lora_exporter.rs` – Converts PyTorch state dict to safetensors format.
- `src/cli.rs` – Command-line interface for export operations.

**Current Connections:**
- **To plato-torch:** Direct, as described above. It is a downstream consumer.
- **To fleet-simulator:** None observed.
- **To holodeck-rust:** None observed.

**Integration Status:** Tightly coupled to plato-torch as a post-training step. Not connected to simulation or interactive environments.

## Summary: Integration Gaps (P1 Safe Channels for Connection)

1.  **fleet-simulator → plato-torch:** The simulator's JSONL event logs could be converted into training "tiles" for a room. This would require a new `SimulationTileLoader` in plato-torch to parse and ingest the logs.
2.  **holodeck-rust → plato-torch:** A holodeck session could be a source of live interaction tiles. Requires a Python client in plato-torch that connects to the holodeck WebSocket, translates game events to tiles, and feeds them into a room.
3.  **plato-ensign → holodeck-rust:** An exported ensign (LoRA) could be loaded to control an NPC in holodeck. Requires integrating a lightweight inference runtime (like `candle` or `llama.cpp`) into the holodeck server to run the adapter.
4.  **plato-torch → holodeck-rust (GhostInjector):** The `GhostInjector` concept (from memory files) suggests injecting "ghost tiles" from dead agents. This could be implemented as a module that reads historical tile files and spawns corresponding NPCs or events in holodeck.

## Immediate Next Task (P2 Optimization within Channel)
Given the task backlog, the highest-priority integration to build is **"Wire GhostInjector into holodeck."** This is a P1 channel (connecting two existing systems) that can be implemented by:
1.  Creating a `ghost_injector` module in holodeck-rust that reads a directory of tile files (JSON).
2.  Mapping tile interactions to NPC dialog or world events.
3.  Injecting these as scripted events into a running holodeck world.

This creates a direct link between the fleet's accumulated knowledge (tiles) and an interactive test environment.
