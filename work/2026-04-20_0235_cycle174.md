# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 174  
**Phase:** 4 — Build & Test  
**Status:** Mapping current connections and identifying integration points.

Based on reading the actual files in the fleet repos, here is the current state of integration between **plato-torch**, **fleet-simulator**, **holodeck-rust**, and **plato-ensign**.

## 1. Verified Connections (What IS Connected)

### plato-torch ↔ fleet-simulator
- **Connection:** Direct integration via `fleet_simulator` module in plato-torch.
- **Evidence:** `plato-torch/src/fleet_simulator/` contains:
  - `simulator_adapter.py` – Bridges plato-torch rooms to fleet-simulator scenarios.
  - `scenario_loader.py` – Loads `.json` scenario files from fleet-simulator.
- **Data Flow:** Plato-torch rooms can load fleet-simulator scenarios to generate training interactions (tiles).
- **Status:** Functional. Used in training rooms like `DeadbandRoom`.

### plato-torch ↔ plato-ensign
- **Connection:** Export pipeline.
- **Evidence:** `plato-torch/src/ensign_exporter.py` exists.
- **Data Flow:** After a room accumulates tiles, `ensign_exporter.py` can package them into a LoRA adapter via plato-ensign.
- **Status:** Presumed functional (export logic present), but requires explicit trigger.

### fleet-simulator → plato-torch (indirect)
- **Connection:** Scenario files as shared data.
- **Evidence:** `fleet-simulator/scenarios/` contains `.json` files that define agent interactions, environments, and constraints.
- **Data Flow:** These files are read by plato-torch's `scenario_loader.py` to create training episodes.
- **Status:** Functional. This is a data dependency, not a runtime API.

## 2. Integration Gaps (What Is NOT Connected)

### holodeck-rust ↔ plato-torch
- **Gap:** No direct bridge.
- **Evidence:** 
  - `holodeck-rust/` is a Rust MUD server with sentiment-aware NPCs.
  - `plato-torch/` has no import or adapter for holodeck.
- **Potential:** Holodeck NPC interactions could become training tiles for plato-torch rooms, but currently they are separate silos.

### holodeck-rust ↔ fleet-simulator
- **Gap:** No connection.
- **Evidence:** No shared data formats or APIs between the two.
- **Potential:** Fleet-simulator scenarios could drive holodeck environments, or holodeck logs could feed scenario generation.

### plato-ensign → holodeck-rust
- **Gap:** No connection.
- **Evidence:** plato-ensign exports LoRA adapters from plato-torch rooms. No hook to deploy or test these adapters in holodeck.
- **Potential:** Ensign adapters could be loaded into holodeck NPCs to evolve their behavior, but no pipeline exists.

### fleet-simulator → holodeck-rust
- **Gap:** No connection.
- **Evidence:** Fleet-simulator outputs JSON scenarios; holodeck uses its own game state and protocol.
- **Potential:** Scenarios could be compiled into holodeck world events or quests.
