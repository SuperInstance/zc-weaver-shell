## 1. plato-torch
**Location:** `plato-torch/`  
**Purpose:** 25 training room presets, self-training rooms.  
**Key Files:**  
- `presets/DeadbandRoom.py` — Implements Deadband Protocol (P0/P1/P2).  
- `presets/__init__.py` — Exports `DeadbandRoom`, `MirrorRoom`, `GhostRoom`.  
- `trainer/room_trainer.py` — Main training loop; loads presets, accumulates tiles.  
- `utils/tile_formatter.py` — Formats interaction data into tiles (JSON).  

**Connections:**  
- **To plato-ensign:** `room_trainer.py` calls `export_to_lora()` from `plato_ensign` if configured.  
- **To fleet-simulator:** Not directly imported. Uses generic `SimulationEnvironment` abstract class; no concrete simulator dependency.  
- **To holodeck-rust:** No imports. Could send tiles via HTTP if `holodeck` exposes an ingestion endpoint (not yet observed).  

**Integration Points:**  
- `config.yaml` includes `ensign_export_path` and `tile_storage_path`.  
- `DeadbandRoom` emits `tile` events containing `agent_id`, `timestamp`, `phase`, `reward`.

## 2. fleet-simulator
**Location:** `fleet-simulator/`  
**Purpose:** Generate training data via multi-agent simulations.  
**Key Files:**  
- `src/simulator/core.rs` — Core simulation engine.  
- `src/agents/mod.rs` — Defines `Agent`, `Environment`, `Action`.  
- `src/output/tile_generator.rs` — Converts simulation steps to tiles (JSON).  

**Connections:**  
- **To plato-torch:** No direct Rust→Python imports. Tiles written to `output/tiles/`; could be read by `plato-torch` via file watcher (not implemented).  
- **To holodeck-rust:** No imports. Both are Rust crates but separate.  
- **To plato-ensign:** No direct link.  

**Integration Points:**  
- Tiles use same JSON schema as `plato-torch` (`agent_id`, `phase`, `reward`).  
- Could be wired via a shared directory or message queue (e.g., Redis). Currently isolated.

**Connections:**  
- **To plato-torch:** None.

- **To plato-ensign:** None.  

**Integration Points:**

## 4. plato-ensign
**Location:** `plato-ensign/`  
**Purpose:** Export room experience into LoRA adapters.  
**Key Files:**  
- `src/exporter.py` — `export_to_lora(tiles, base_model, output_path)`.  
- `src/adapters/lora_builder.py` — Creates LoRA weights from tile sequences.  
- `config/ensign_config.yaml` — Model paths, hyperparameters.  

**Connections:**  
- **From plato-torch:** Called via `room_trainer.py` when `ensign_export_enabled: true`.  
- **To fleet-simulator:** None.  
- **To holodeck-rust:** None.  

**Integration Points:**  
- Expects tiles in same JSON format as `plato-torch` outputs.  
- Outputs `.safetensors` files to `output/adapters/`.  
- Could be used by `holodeck-rust` to load NPC personality adapters (not implemented).

## Summary: What’s Connected vs. What’s Not

### Connected:
1. **plato-torch → plato-ensign** — Direct Python import; tiles flow to LoRA exporter.  
2. **plato-torch & fleet-simulator** — Share tile JSON schema (implicit compatibility).  

### Not Connected (Integration Gaps):
1. **fleet-simulator → plato-torch** — No ingestion pipeline. Tiles sit on disk.  
2. **holodeck-rust ↔ any** — Standalone MUD; no tile ingestion or adapter loading.  
3. **cudaclaw/flux-runtime ↔ training pipeline** — Isolated.  
4. **Ghost tiles (dead agents’ lessons)** — Mentioned in docs but no observed `GhostInjector` component.  
5. **Mirror Play** — Concept exists; no `MirrorRoom` implementation found in `plato-torch/presets/`.
