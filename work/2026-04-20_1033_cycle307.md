## 1. plato-torch → fleet-simulator
**Connection:** **Yes** — Direct import and data flow.
- **Evidence:** `plato-torch/src/rooms/deadband_room.py` imports `from fleet_simulator.simulator import FleetSimulator`.
- **Usage:** `DeadbandRoom` instantiates a `FleetSimulator` object to generate synthetic training data (agent interactions, P0/P1/P2 trajectories).
- **Data Flow:** Simulator → Room. The room calls `simulator.generate_episode()` to create training tiles.
- **Integration Gap:** No bidirectional flow. The simulator does not read back from the room's state or trained models.

---

## 2. plato-torch → plato-ensign
**Connection:** **Yes** — Configuration and pipeline.
- **Evidence:** `plato-torch/config/ensign_config.yaml` defines export settings for LoRA adapters.
- **Usage:** After a room trains on accumulated tiles, an ensign export can be triggered via CLI (`plato-torch export-ensign`). The ensign reads the room's checkpoint and tile history.
- **Data Flow:** Room tiles + model checkpoint → Ensign → LoRA adapter (.safetensors).
- **Integration Gap:** The export is manual/CLI-driven, not automatic per training cycle. No real-time feedback loop.

---

## 3. plato-torch → holodeck-rust
**Connection:** **No** — No direct imports or shared code.
- **Evidence:** Searched `plato-torch/` for "holodeck" — zero matches. No Rust dependencies in `pyproject.toml`.
- **Potential Link:** Both are training environments. `holodeck-rust` is a sentiment-aware MUD; `plato-torch` is a PyTorch-based room server. They could share tile formats or agent personas, but currently do not.
- **Integration Gap:** No shared tile ingestion, no common agent API, no cross-environment telemetry.

---

## 4. fleet-simulator → holodeck-rust
**Connection:** **No** — Separate languages (Python vs Rust), no imports.
- **Evidence:** `fleet-simulator/` has no Rust code or references to holodeck.
- **Potential Link:** Both generate synthetic interactions. The simulator focuses on fleet P0/P1/P2 trajectories; holodeck focuses on NPC sentiment and dialogue. Could be complementary data sources.
- **Integration Gap:** No shared scenario definitions, no character exchange, no unified logging.

---

## 5. fleet-simulator → plato-ensign
**Connection:** **Indirect** — via plato-torch.
- **Evidence:** The simulator feeds tiles to plato-torch rooms; those rooms can export to ensign.
- **Direct Link?** No — `plato-ensign` does not import the simulator.
- **Integration Gap:** Ensign cannot directly read simulator output without going through a room.

---

## 6. holodeck-rust → plato-ensign
**Connection:** **No** — No references in either codebase.
- **Evidence:** `holodeck-rust/src/` contains no Python or PyTorch hooks.
- **Potential Link:** Holodeck NPC interactions could become training tiles for ensigns (e.g., sentiment LoRAs). Not implemented.
- **Integration Gap:** No tile export pipeline from holodeck to plato formats.

---

## 7. GhostInjector (plato-torch) → holodeck-rust
**Connection:** **No** — GhostInjector is a Python class in `plato-torch/src/injectors/ghost_injector.py`.
- **Evidence:** GhostInjector reads "ghost tiles" (dead agent lessons) and injects them into room training. It is specific to plato-torch rooms.
- **Integration Gap:** No mechanism to inject ghost tiles into holodeck’s NPC memory or sentiment engine.

---

## 8. DeadbandRoom → plato-relay
**Connection:** **No** — plato-relay not found in current repo scan.
- **Evidence:** Searched for "relay" in plato-torch — no matches. Not in dependencies.
- **Note:** plato-relay may be a separate service (message passing) not yet integrated.
- **Integration Gap:** Room telemetry and tile submission currently use direct HTTP to room server; no relay layer.

---

## 9. cudaclaw / flux-runtime → Any
**Connection:** **Not observed** in the inspected repos.
- **Evidence:** No imports of `cudaclaw` or `flux-runtime` in plato-torch, fleet-simulator, holodeck-rust, or plato-ensign.
- **Note:** These are likely lower‑level runtimes for GPU‑resident agents and VM orchestration, not yet wired into the training stack.
- **Integration Gap:** Training rooms do not deploy agents via cudaclaw; bytecode VM not used for tile processing.
