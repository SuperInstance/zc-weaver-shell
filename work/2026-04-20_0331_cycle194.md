# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 194  
**Phase:** 4 — Build & Test  
**Status:** Updated map based on actual code inspection of plato-torch, fleet-simulator, holodeck-rust, and plato-ensign.

Based on reading the actual files in the fleet repos, here is the current state of connections and integration gaps.

## 1. Verified Connections (What's Connected)

### plato-torch ↔ fleet-simulator
- **Connection:** Strong, direct integration via `fleet-simulator` as a data source.
- **Evidence:** `plato-torch/rooms/deadband_room.py` imports `fleet_simulator` and uses `FleetSimulator` class to generate training data.
- **Data Flow:** `DeadbandRoom` calls `fleet_simulator.generate_scenario()` to create simulation episodes. These episodes become training tiles.
- **Protocol:** Uses the same tile format (JSON with `role`, `content`, `metadata`). Simulation outputs are formatted as tiles.

### plato-torch ↔ plato-ensign
- **Connection:** Direct export pipeline.
- **Evidence:** `plato-torch/export/ensign_exporter.py` exists. It reads room experience (accumulated tiles) and exports to LoRA adapters via `plato-ensign`.
- **Data Flow:** Room tiles → `ensign_exporter.py` → LoRA weights (`.safetensors`).
- **Protocol:** Uses Hugging Face `peft` library for LoRA. Configurable rank, alpha.

### fleet-simulator ↔ (indirect) holodeck-rust
- **Connection:** Conceptual, not yet wired.
- **Evidence:** `fleet-simulator/src/scenarios/mod.rs` includes `social` and `npc_interaction` scenario types. These align with holodeck's sentiment-aware NPCs.
- **Data Flow:** Potential: simulator could generate NPC interaction logs → holodeck uses as training data for NPC behavior.
- **Current State:** No import or direct call. Integration is a design pattern, not code.

## 2. Integration Gaps (What's Not Connected)

### holodeck-rust ↔ plato-torch
- **Gap:** No direct connection.
- **Evidence:** `holodeck-rust/src/main.rs` is a standalone MUD server. No Python bindings or HTTP API exposed in current code.
- **Missing:** No tile ingestion endpoint in holodeck. No way for plato-torch to send/receive tiles from holodeck sessions.
- **Potential Integration Point:** Could add a `/tiles` POST endpoint in holodeck to accept tiles from plato-torch rooms.

### holodeck-rust ↔ plato-ensign
- **Gap:** No connection.
- **Evidence:** `plato-ensign` only imports from `plato-torch` in its exporter. No Rust bindings or hooks for holodeck.
- **Missing:** NPC sentiment data from holodeck cannot be exported as LoRA adapters currently.
- **Potential Integration Point:** Could write a Rust→Python bridge to feed holodeck NPC logs into `ensign_exporter`.

### fleet-simulator ↔ plato-ensign
- **Gap:** No direct export path.
- **Evidence:** `fleet-simulator` outputs raw scenario data (JSON). No built-in LoRA export.
- **Missing:** Simulator data must pass through a plato-torch room first to become tiles before ensign export.
- **Potential Integration Point:** Could add a `SimulatorRoom` in plato-torch that wraps fleet-simulator and feeds directly to ensign exporter.

### GhostInjector (anywhere)
- **Gap:** Not connected to any system.
- **Evidence:** `plato-torch/rooms/ghost_injector.py` exists but is not imported or used in `deadband_room.py` or main training loops.
- **Missing:** Ghost tiles (dead agents' lessons) are not being injected into active training.
- **Potential Integration Point:** Add `GhostInjector` as a mixin to `DeadbandRoom` to blend ghost tiles with live simulation tiles.

## 4. Recommended Integration Tasks (P1 Safe Channels)

1. **Wire GhostInjector into DeadbandRoom** — Modify `deadband_room.py` to import and use `GhostInjector` to mix ghost tiles into training batches.
2. **Add HTTP API to holodeck-rust** — Create a `/tiles` endpoint in holodeck to accept JSON tiles from plato-torch. This creates a bidirectional channel.
3. **Create SimulatorRoom in plato-torch** — A dedicated room that consumes fleet-simulator data and exports directly to plato-ensign, bypassing manual tile collection.
4. **Bridge holodeck to plato-ensign** — Write a small Rust utility that serializes NPC interaction logs into the tile format and calls the Python exporter.

These are safe channels because they extend existing patterns without breaking current functionality.
