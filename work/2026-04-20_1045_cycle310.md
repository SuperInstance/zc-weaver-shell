## 1. plato-torch → fleet-simulator
**Connection:** **YES** — Direct import and usage.

**Evidence:**
- `plato-torch/src/training/rooms/deadband_room.py` imports `fleet_simulator`:
  ```python
  from fleet_simulator import FleetSimulator, AgentState, EnvironmentConfig
  ```
- `DeadbandRoom` initializes a `FleetSimulator` instance in its `__init__`:
  ```python
  self.simulator = FleetSimulator(config=sim_config)
  ```
- The room uses the simulator to generate training episodes (`self.simulator.run_episode()`).
- Shared data structure: `AgentState` from `fleet_simulator` is used to represent agent state within the room.

**Integration Gap:** None observed. This is a working, direct dependency.

---

## 2. plato-torch → holodeck-rust
**Connection:** **NO** — No direct import or usage found.

**Evidence:**
- Searched `plato-torch/src/` for `holodeck`, `mud`, `sentiment`, `npc`. No imports or references.
- `plato-torch` focuses on PyTorch-based training loops and room presets. `holodeck-rust` is a separate Rust-based MUD server.
- No shared configuration files or data structures identified.

**Integration Gap:** **Significant**. These are two independent systems with no current wiring. A potential integration point could be using `holodeck-rust` as an environment provider for `plato-torch` rooms (e.g., a `HolodeckRoom` preset that connects to the MUD server via a client socket).

---

## 3. plato-torch → plato-ensign
**Connection:** **YES** — Export pathway, but not a runtime dependency.

**Evidence:**
- `plato-torch/src/training/exporter.py` contains the `EnsignExporter` class.
- This exporter takes a trained room model (a PyTorch module) and its accumulated interaction tiles (training data) and saves them in a format compatible with `plato-ensign`.
- `plato-ensign` is designed to consume these exports to create LoRA adapters.
- The connection is file-based (exported checkpoint and config), not a live import.

**Integration Gap:** **Minimal**. The export pipeline is defined. A potential gap is the lack of an automated trigger within `plato-torch` to run the exporter when a room reaches a certain performance threshold.

---

## 4. fleet-simulator → holodeck-rust
**Connection:** **NO** — No direct import or usage found.

**Evidence:**
- `fleet-simulator/src/lib.rs` is a pure Rust simulation library. No references to `holodeck` or networking code.
- `holodeck-rust` is an async Tokio-based TCP server (`src/main.rs`). It does not import the `fleet_simulator` crate.
- They are separate crates in the fleet's Cargo workspace.

**Integration Gap:** **Significant**. These are two distinct environment simulators. A potential integration is to have `fleet-simulator` agents interact with the `holodeck-rust` world via its API, or to use `holodeck-rust`'s sentiment-aware NPCs as part of a `fleet-simulator` scenario.

---

## 5. fleet-simulator → plato-ensign
**Connection:** **NO** — No direct relationship.

**Evidence:**
- `plato-ensign` is a Python tool for creating LoRAs from PyTorch checkpoints.
- `fleet-simulator` is a Rust library for generating simulation data.
- No shared files, imports, or configuration.

**Integration Gap:** **Indirect**. `fleet-simulator` generates data used by `plato-torch` rooms, which can later be exported via `plato-ensign`. The connection is through the `plato-torch` intermediary, not direct.

---

## 6. holodeck-rust → plato-ensign
**Connection:** **NO** — No direct relationship.

**Evidence:**
- `holodeck-rust` outputs game state and interaction logs.
- `plato-ensign` expects PyTorch model checkpoints and tile data from `plato-torch`.
- No code or configuration links them.

**Integration Gap:** **Significant**. `holodeck-rust` interactions could be a rich source of training tiles. An integration would require a new component to translate MUD interactions into the `plato-torch` tile format and potentially fine-tune a model on that data.

## Summary Integration Map

```
plato-torch ────(direct)────> fleet-simulator
    │
    └───(export)───> plato-ensign

holodeck-rust ────(isolated)────┐
                                │
fleet-simulator ──(isolated)────┘
```

**Key Integration Gaps (P1 Opportunities):**
1.  **plato-torch ↔ holodeck-rust:** Wire the MUD server as a live training environment.
2.  **fleet-simulator ↔ holodeck-rust:** Allow simulated agents to interact with the MUD world.
3.  **holodeck-rust → plato-ensign pipeline:** Create a pathway to turn MUD interactions into ensign adapters.

**Next Task (Weaver Backlog):** The highest-priority wiring appears to be **GhostInjector into holodeck**. This would involve modifying `holodeck-rust` to accept and utilize "ghost tiles" (P0 knowledge from dead agents) to influence NPC behavior or world events. This requires inspecting the `holodeck-rust` codebase to find a suitable injection point (e.g., NPC dialogue system, event manager).
