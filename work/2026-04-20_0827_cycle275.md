## 1. plato-torch (Training Rooms)
**Location:** `plato-torch/`  
**Purpose:** 25 training room presets, self-training rooms.  
**Key Files:**
- `presets/DeadbandRoom.py` — Implements Deadband Protocol (P0/P1/P2) as a training room.
- `presets/__init__.py` — Exports `DeadbandRoom`, `MirrorRoom`, `GhostRoom`.
- `trainer/room_trainer.py` — Main training loop; loads presets, accumulates interaction tiles.
- `utils/tile_utils.py` — Defines `TrainingTile` dataclass (used across fleet).

**Connections:**
- **To plato-ensign:** `trainer/room_trainer.py` line 142: `from plato_ensign.exporter import export_room_to_lora`. Conditional export after training.
- **To fleet-simulator:** `presets/DeadbandRoom.py` line 89: `from fleet_simulator.scenarios import DeadbandScenario`. Uses scenario data for P0/P1/P2 training.
- **Shared Data:** `TrainingTile` (source, content, metadata, timestamp) is serialized/deserialized across plato-torch and plato-ensign.

**Integration Gaps:**
- No direct import of `holodeck-rust`.  
- `GhostRoom` preset exists but does not import `holodeck_rust` or `GhostInjector`.  
- `MirrorRoom` does not reference `flux-runtime` or `cudaclaw`.

## 2. fleet-simulator (Scenario Data)
**Location:** `fleet-simulator/`  
**Purpose:** Generate training data for fleet agents.  
**Key Files:**
- `scenarios/DeadbandScenario.py` — Defines obstacle maps (P0), safe channels (P1), optimization paths (P2).
- `scenarios/__init__.py` — Exports `DeadbandScenario`, `OracleScenario`, `EdgeScenario`.
- `data_generator.py` — Produces JSONL training tiles compatible with `TrainingTile` schema.

**Connections:**
- **To plato-torch:** `scenarios/DeadbandScenario.py` is imported by `DeadbandRoom` (see above).  
- **To plato-ensign:** None. No direct import.  
- **To holodeck-rust:** None.  
- **Shared Data:** Scenario JSON output matches `TrainingTile` metadata fields (`scenario_id`, `phase`).

**Integration Gaps:**
- No connection to `holodeck-rust` for sentiment-aware NPC data.  
- No connection to `constraint-theory-core` for geometric snapping in scenarios.  
- Scenario data is static; not dynamically generated by `cudaclaw` or `flux-runtime`.

## 3. holodeck-rust (MUD Server)
**Location:** `holodeck-rust/`  
**Purpose:** Multi‑User Dungeon server with sentiment‑aware NPCs.  
**Key Files:**
- `src/npc/sentiment.rs` — Defines `SentimentEngine`, `NpcState`.  
- `src/world/room.rs` — Defines `Room`, `Exit`, `Item`.  
- `src/lib.rs` — Exports `Server`, `NpcBuilder`.  
- `examples/ghost_injector.rs` — Example of injecting ghost tiles (not integrated).

**Connections:**
- **To plato-torch:** None. No Rust‑to‑Python FFI in current code.  
- **To fleet-simulator:** None.  
- **To plato-ensign:** None.  
- **To constraint-theory-core:** `Cargo.toml` includes `constraint-theory-core = { path = "../constraint-theory-core" }`. Used in `src/physics/snapping.rs` for geometric constraints.  
- **To cudaclaw/flux-runtime:** None.

**Integration Gaps:**
- No Python bindings; cannot be called directly from plato‑torch rooms.  
- `GhostInjector` example exists but is not wired into the main server or plato‑torch.  
- Sentiment NPCs do not receive training tiles from plato‑torch.

## 4. plato-ensign (LoRA Exporter)
**Location:** `plato-ensign/`  
**Purpose:** Export room experience into LoRA adapters.  
**Key Files:**
- `exporter.py` — `export_room_to_lora(room_state, tiles, output_path)`.  
- `lora_builder.py` — Converts `TrainingTile` lists into LoRA weights.  
- `configs/ensign_config.yaml` — Specifies base model, rank, alpha.

**Connections:**
- **To plato-torch:** Imported by `room_trainer.py` (conditional export).  
- **To fleet-simulator:** None.  
- **To holodeck-rust:** None.  
- **Shared Data:** Uses `TrainingTile` list from plato‑torch as input.

**Integration Gaps:**
- Only exports from plato‑torch; no import from fleet‑simulator or holodeck‑rust.  
- Does not ingest `GhostTile` data from dead agents.  
- No integration with `cudaclaw` for GPU‑resident optimization.

## 5. constraint-theory-core (Geometric Snapping)
**Location:** `constraint-theory-core/`  
**Purpose:** Rust library for geometric constraint solving.  
**Key Files:**
- `src/snapping.rs` — `SnapEngine`, `ConstraintSet`.  
- `Cargo.toml` — No dependencies on other fleet repos.

**Connections:**
- **To holodeck-rust:** Direct dependency (see above).  
- **To others:** None.

**Integration Gaps:**
- Not used by plato‑torch, fleet‑simulator, or plato‑ensign.  
- Could be applied to scenario geometry in fleet‑simulator.

## 6. cudaclaw & flux-runtime (GPU/VM Runtime)
**Locations:** `cudaclaw/`, `flux-runtime/`  
**Purpose:** GPU‑resident agent runtime and self‑assembling bytecode VM.  
**Key Files:**
- `cudaclaw/include/cudaclaw.h` — CUDA kernels for agent inference.  
- `flux-runtime/src/vm.rs` — Flux VM bytecode interpreter.

**Connections:**
- **To any other repo:** None found in imports or `Cargo.toml`/`requirements.txt`.  
- **Shared Data:** No visible data flow.

**Integration Gaps:**
- Isolated; not wired into training or simulation pipelines.  
- `flux-runtime` could potentially run `holodeck-rust` NPC logic.  
- `cudaclaw` could accelerate plato‑torch training.
