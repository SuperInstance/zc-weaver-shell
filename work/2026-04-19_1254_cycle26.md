## 2. Integration Gaps & Phase 4 Action Items

### Gap 1: **Ghost Tile Injection into holodeck-rust**
- **Problem:** "Ghost tiles" (P0 knowledge from dead agents) are a fleet architectural concept, but no pipeline exists to inject them into the live `holodeck-rust` environment.
- **Action:** Wire a `GhostInjector` service. It would read from a ghost tile archive and format events/NPC dialogue cues for the holodeck server.
- **Priority:** **High.** Implements core "Deadband Protocol" P0 (negative space) learning.

### Gap 2: **DeadbandRoom to plato-relay Connection**
- **Problem:** `DeadbandRoom` (a `plato-torch` preset) operates in isolation. For multi-agent or cross-fleet training, its output needs to be broadcast.
- **Action:** Connect `DeadbandRoom` to the `plato-relay` message bus (inferred from "plato-relay" name in task backlog). This allows tiles from this specialized room to be shared with other rooms or logged for analysis.
- **Priority:** **Medium.** Enhances collaborative training and observation.

### Gap 3: **Closed-Loop: holodeck â†’ plato-torch â†’ plato-ensign â†’ holodeck**
- **Problem:** The current flow is linear (holodeck data trains a room, which can export an ensign). There's no feedback loop where an ensign actively improves the holodeck it was trained from.
- **Action:** Build an integration where a `plato-ensign` LoRA can be loaded by `holodeck-rust` to dynamically adjust NPC sentiment or dialogue, creating a self-improving simulation.
- **Priority:** **High.** Would create a powerful iterative training cycle.

### Gap 4: **fleet-simulator & plato-torch Standardized Data Format**
- **Problem:** While connected, the exact data format (tile schema) for simulations is unspecified. This could cause ingestion errors.
- **Action:** Document and enforce a shared `.tile.json` schema between `fleet-simulator` output and `plato-torch` room input.
- **Priority:** **Medium.** Prevents future breakage.
