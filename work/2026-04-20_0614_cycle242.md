## 1. plato-torch (Training Rooms)
**Location:** `plato-torch/`  
**Purpose:** 25 training room presets, self-training rooms.  
**Key Files:**  
- `plato-torch/src/rooms/deadband_room.py` — Deadband Protocol implementation.  
- `plato-torch/src/rooms/__init__.py` — Room registry.  
- `plato-torch/src/training/tile_processor.py` — Processes interaction tiles.  

**Connections:**  
- **To plato-ensign:** Exports room experience via `export_to_lora()` method in `deadband_room.py`.  
- **To fleet-simulator:** Not directly imported. Uses `Tile` data structure that *could* be shared.  
- **To holodeck-rust:** No direct imports.  

**Integration Points:**  
- `deadband_room.py` line 47: `from plato_ensign import export_lora` — explicit dependency.  
- `tile_processor.py` expects `Tile` objects with `agent_id`, `timestamp`, `content`.  

## 2. plato-ensign (LoRA Adapter Export)  
**Location:** `plato-ensign/`  
**Purpose:** Exports room experience into LoRA adapters.  
**Key Files:**  
- `plato-ensign/src/exporter.py` — Main export logic.  
- `plato-ensign/src/lora_builder.py` — Builds adapters from tiles.  

**Connections:**  
- **From plato-torch:** Called via `export_lora(room_state, output_path)`.  
- **To fleet-simulator:** No direct imports.  
- **To holodeck-rust:** No direct imports.  

**Integration Points:**  
- `exporter.py` line 12: `from plato_torch.src.rooms.base_room import RoomState` — imports room state structure.  

## 3. fleet-simulator (Fleet Simulation)  
**Location:** `fleet-simulator/`  
**Purpose:** Generates training data via simulated fleet interactions.  
**Key Files:**  
- `fleet-simulator/src/simulator/core.py` — Main simulation loop.  
- `fleet-simulator/src/data/tile_generator.py` — Creates synthetic tiles.  

**Connections:**  
- **To plato-torch:** No direct imports. Generates `Tile` objects with same fields.  
- **To plato-ensign:** No direct imports.  
- **To holodeck-rust:** No direct imports.  

**Integration Points:**  
- `tile_generator.py` uses `Tile` class defined locally, but fields match plato-torch’s `Tile`. Potential for conflict if schemas diverge.

## 5. Integration Gaps (What’s Not Connected)  
**Gap 1:** `fleet-simulator` → `plato-torch` — No shared `Tile` definition. Risk of schema drift.  
**Gap 2:** `holodeck-rust` → `plato-torch` — No bridge for NPC interactions to become training tiles.  
**Gap 3:** `plato-ensign` → `fleet-simulator` — Cannot export simulation data as LoRA adapters.  
**Gap 4:** `holodeck-rust` → `plato-ensign` — NPC sentiment experience not captured for ensign training.  

## 6. Verified Data Flow (Current)  
```
plato-torch (DeadbandRoom) → plato-ensign (export_lora) → LoRA adapter
```
**Missing Flows:**  
- `fleet-simulator` → `plato-torch` (simulated tiles not ingested)  
- `holodeck-rust` → `plato-torch` (NPC interactions not captured)  
- Any system → `holodeck-rust` (no external sentiment input)
