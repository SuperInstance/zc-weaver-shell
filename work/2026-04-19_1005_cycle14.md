## 1. Verified Connections (Based on fleet-knowledge/ files)

### plato-torch → plato-ensign
- **Connection:** Direct export pipeline.
- **Mechanism:** `plato-torch` training rooms accumulate interaction tiles. `plato-ensign` can export a room's experience into a LoRA adapter.
- **Evidence:** From FLEET-CONTEXT.md: "plato-ensign: Exports room experience into LoRA adapters."
- **Status:** Connected. This is a core architectural flow.

### plato-torch → fleet-simulator
- **Connection:** Data source for training.
- **Mechanism:** `fleet-simulator` generates training data (simulated fleet interactions). This data can be formatted into tiles and ingested by `plato-torch` rooms.
- **Evidence:** Both are listed under "Key Technologies" with complementary purposes (training rooms vs. simulation for training data).
- **Status:** Connected in principle (data flow), but specific integration points (APIs, data formats) need verification.

### holodeck-rust → plato-torch
- **Connection:** Potential tile source.
- **Mechanism:** `holodeck-rust` is a MUD server with sentiment-aware NPCs. Agent interactions within the holodeck could be captured as interaction tiles and sent to a `plato-torch` room for training.
- **Evidence:** Both are key technologies; architecture states "Rooms train from accumulated interaction tiles." Holodeck is a natural source of such interactions.
- **Status:** Potential connection. No explicit wiring mentioned.

### fleet-simulator → holodeck-rust
- **Connection:** Unclear direct link.
- **Mechanism:** The simulator could potentially generate scenarios or NPC behavior profiles for the holodeck, or vice-versa.
- **Evidence:** No direct mention in provided context.
- **Status:** Not explicitly connected. A potential gap.

### plato-ensign → (Downstream Use)
- **Connection:** Output model deployment.
- **Mechanism:** LoRA adapters exported by `plato-ensign` could be loaded by agents (like Zeroclaws) or integrated into `cudaclaw`/`flux-runtime`.
- **Evidence:** "Ensigns are optional exports of room experience." Implies use elsewhere.
- **Status:** Connected to the broader fleet ecosystem, but specific consumers not detailed.

## 2. Identified Integration Gaps (Actionable for Phase 3)

### Gap 1: **plato-torch → holodeck-rust Bidirectional Loop**
- **Problem:** No established pipeline for sending holodeck interaction logs to a training room, or for deploying a trained ensign (NPC behavior) back to the holodeck.
- **P0 (Negative Space):** Don't assume a shared data format exists.
- **P1 (Safe Channel):** Check for existing logging in `holodeck-rust` and tile ingestion API in `plato-torch`.
- **P2 (Optimize):** Build a minimal connector service (`holodeck-to-tile`) that formats MUD logs into tiles and POSTs them to a room's endpoint.

### Gap 2: **fleet-simulator → plato-torch Formal Data Contract**
- **Problem:** While a conceptual connection exists, the specifics of how simulation outputs become training tiles are undefined.
- **P0:** Don't modify core data generation of the simulator prematurely.
- **P1:** Examine `fleet-simulator` output formats and `plato-torch` tile schema. Look for an existing adapter or example.
- **P2:** Create a documented, versioned data contract (e.g., a `simulation_tile.proto` or JSON schema) and a reference adapter.

### Gap 3: **Ghost Tile Injection Point**
- **Problem:** "Ghost tiles: dead agents' lessons persist as P0 knowledge." The architectural component (`GhostInjector`) and its integration point are not specified.
- **P0:** Don't corrupt live training data with unvalidated ghost tiles.
- **P1:** Identify where agent lifecycles are logged and where P0 knowledge is stored. Likely connects to `plato-torch` room memory or a dedicated `fleet-memory` service.
- **P2:** Propose a `GhostInjector` service that reads from a dead agent's final state, creates a P0 constraint tile, and injects it into a designated "lessons" room or dataset.

### Gap 4: **Ensign Deployment to cudaclaw/flux-runtime**
- **Problem:** The path for a trained LoRA (from `plato-ensign`) to become active in the GPU-resident (`cudaclaw`) or VM (`flux-runtime`) agent runtime is not mapped.
- **P0:** Don't assume runtime compatibility.
- **P1:** Check `cudaclaw` and `flux-runtime` documentation for model loading interfaces. Check `plato-ensign` for export formats.
- **P2:** Draft a deployment spec: "Ensign Package" (LoRA + manifest) and a loader utility for each target runtime.
