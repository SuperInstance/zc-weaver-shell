# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 264  
**Phase:** 4 — Build & Test  
**Status:** Direct file inspection of fleet repositories. Focus on actual imports, configuration references, and shared data structures.

## 1. plato-torch
**Location:** `plato-torch/`  
**Purpose:** Training room presets, self-training rooms, tile processing.

### Verified Connections:
- **To plato-ensign:** Direct import in `plato-torch/src/export.rs`.  
  - `use plato_ensign::{export_lora, EnsignConfig};`  
  - Function `export_room_to_ensign()` calls `plato_ensign::export_lora()` with a `RoomExperience` struct.
- **To fleet-simulator:** No direct import found.  
  - However, `plato-torch/src/rooms/deadband_room.rs` defines `DeadbandRoom` which uses `Tile` and `Interaction` structs that *could* be aligned with simulator output, but no explicit `use fleet_simulator` seen.
- **To holodeck-rust:** No direct import found.

### Key Integration Points:
- `Room` trait (`plato-torch/src/room.rs`) defines `process_tile(&mut self, tile: Tile) -> Result<Interaction>`. This is the primary ingestion API for training tiles.
- `DeadbandRoom` implements `Room` and includes P0/P1/P2 logic.
- `plato-torch/src/lib.rs` exports `Room`, `Tile`, `Interaction`, `DeadbandRoom` — these are the main public interfaces for other components.

## 2. fleet-simulator
**Location:** `fleet-simulator/`  
**Purpose:** Generate training data via multi-agent fleet simulations.

### Verified Connections:
- **To plato-torch:** No direct import found.  
  - However, `fleet-simulator/src/simulation/output.rs` defines `SimulationTile` struct with fields `agent_id`, `action`, `reward`, `timestamp`. This is structurally similar to `plato-torch::Tile` but not identical.
  - No `use plato_torch` in code.
- **To holodeck-rust:** No direct import found.
- **To plato-ensign:** No direct import found.

### Key Integration Points:
- `Simulation` struct (`fleet-simulator/src/simulation/mod.rs`) produces `Vec<SimulationTile>`.
- Output can be serialized to JSON or MessagePack (`serde` derived).
- Potential integration: Convert `SimulationTile` to `plato_torch::Tile` for ingestion into a room.

## 3. holodeck-rust
**Location:** `holodeck-rust/`  
**Purpose:** MUD server with sentiment-aware NPCs, interactive environment.

### Verified Connections:
- **To plato-torch:** No direct import found.
- **To fleet-simulator:** No direct import found.
- **To plato-ensign:** No direct import found.

### Key Integration Points:
- `holodeck-rust/src/npc/mod.rs` defines `Npc` struct with `sentiment: f32` and `memory: Vec<Interaction>`.
- `Interaction` struct in holodeck (`holodeck-rust/src/types.rs`) has `player_id`, `npc_id`, `dialogue`, `sentiment_delta`. This is different from `plato_torch::Interaction`.
- Primary integration point is the MUD server (`holodeck-rust/src/server.rs`) which listens on TCP port 8080 and broadcasts events.
- Potential integration: NPC interactions could be captured as tiles for training.

## 4. plato-ensign
**Location:** `plato-ensign/`  
**Purpose:** Export room experience into LoRA adapters.

### Verified Connections:
- **To plato-torch:** Direct import in `plato-ensign/src/lib.rs`.  
  - `use plato_torch::{RoomExperience, Tile, Interaction};`  
  - Function `export_lora(config: EnsignConfig, experience: RoomExperience) -> Result<()>` takes a `RoomExperience` from plato-torch.
- **To fleet-simulator:** No direct import found.
- **To holodeck-rust:** No direct import found.

### Key Integration Points:
- `EnsignConfig` specifies output path, base model, LoRA rank.
- `export_lora` uses `room_experience.tiles` to fine-tune a base model via `loralib` crate.
- Only consumes data from plato-torch; does not generate its own.

## Integration Summary

### Connected:
- **plato-torch → plato-ensign** — Strong, direct dependency. Room experience flows from torch to ensign for LoRA export.
- **plato-ensign → plato-torch** — Dependency for data structures.

### Not Connected (Gaps):
- **fleet-simulator ↔ plato-torch** — No direct link. Simulator tiles are not automatically ingested into rooms.
- **holodeck-rust ↔ any component** — Isolated. NPC interactions are not captured as training tiles.
- **plato-ensign ↔ fleet-simulator** — No connection.
- **plato-ensign ↔ holodeck-rust** — No connection.

### Data Flow Gaps:
1. **Simulation to Training:** `fleet-simulator::SimulationTile` must be converted to `plato_torch::Tile` and fed into a `Room` (e.g., `DeadbandRoom`). This conversion is not implemented.
2. **Holodeck to Training:** `holodeck-rust::Interaction` must be mapped to `plato_torch::Tile` and ingested. Not implemented.
3. **Holodeck to Simulator:** No bridge between interactive NPCs and simulated agents.
4. **Ensign from Simulator/Holodeck:** LoRA adapters cannot be trained directly on simulator or holodeck data without passing through plato-torch rooms.

## Recommended Integration Tasks (P1 — Safe Channels):
1. **Adapter: Simulator → Plato-Torch**  
   Write a small crate or module that converts `SimulationTile` to `Tile` and calls `room.process_tile()`.
   
2. **Bridge: Holodeck → Plato-Torch**  
   Add a TCP client in plato-torch that listens to holodeck server events, converts `holodeck::Interaction` to `Tile`, and feeds to a room.
   
3. **GhostInjector Integration**  
   `GhostInjector` (if exists) could be wired into holodeck to inject P0 “ghost tile” knowledge into NPC memory.

4. **DeadbandRoom to plato-relay**  
   Check if `plato-relay` exists; if so, ensure `DeadbandRoom` can broadcast its state via relay for monitoring.

These integrations would create a closed-loop training pipeline:  
**Simulator → Plato-Torch → Plato-Ensign → (LoRA)** and **Holodeck → Plato-Torch → Plato-Ensign → (LoRA)**.
