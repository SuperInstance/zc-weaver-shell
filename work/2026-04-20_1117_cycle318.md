## 1. plato-torch → fleet-simulator
**Connection:** ✅ **Direct import**  
- `plato-torch/rooms/deadband_room.py` imports `from fleet_simulator.simulator import FleetSimulator` (line 12)  
- Used in `DeadbandRoom._step_simulation()` to create a simulator instance for generating training episodes.  
- Simulator provides `step()` method returning `(observation, reward, done, info)`.  
- **Data flow:** plato-torch pulls simulated fleet interactions as training tiles.

**Integration gap:**  
- fleet-simulator appears to be a standalone package; no reciprocal import of plato-torch.  
- Simulator config (`fleet_simulator/config.py`) does not reference plato-torch room presets.  
- Potential P1 channel: allow simulator to read room state to adjust episode generation.

## 2. plato-torch → plato-ensign
**Connection:** ✅ **Direct export path**  
- `plato-torch/export/ensign_exporter.py` provides `export_room_to_lora(room_state, output_path)`.  
- Called from `plato-torch/train/trainer.py` after room training loop completes.  
- Uses `peft` library to save LoRA adapters from trained model.  
- **Data flow:** room experience (tiles) → trained model weights → LoRA adapter file.

**Integration gap:**  
- No automatic trigger; export is manual/optional via CLI flag `--export-ensign`.  
- Ensign adapter format not validated against plato-ensign loader (separate repo).  
- Potential P1 channel: add health check that exported adapter loads in plato-ensign test suite.

## 3. holodeck-rust → (plato-torch, fleet-simulator, plato-ensign)
**Connection:** ❌ **No direct imports found**  
- `holodeck-rust/src/main.rs` is a standalone MUD server with sentiment-aware NPCs.  
- Uses `serde` for JSON config, `tokio` for async networking.  
- No Rust `extern crate` or Python FFI bindings to other fleet repos.  
- **Data flow:** isolated; NPC interactions stay within MUD.

**Integration gap:**  
- holodeck-rust could be a tile source for plato-torch (sentiment training).  
- No bridge between Rust MUD and Python training pipeline.  
- Potential P1 channel: create a `holodeck_bridge` Python package with Rust bindings to stream NPC dialogues as tiles.

## 4. fleet-simulator → holodeck-rust
**Connection:** ❌ **No reference**  
- `fleet-simulator/simulator/agent.py` models agent behaviors but does not call holodeck.  
- Simulator agents are simple rule-based; no sentiment awareness.  
- **Data flow:** none.

**Integration gap:**  
- Could enrich simulator agents with holodeck NPC sentiment models.  
- Potential P1 channel: embed a sentiment score from holodeck into simulator `info` dict.

## 5. plato-ensign → plato-torch
**Connection:** 🔄 **One-way dependency**  
- `plato-ensign/loader.py` expects a PEFT LoRA adapter (created by plato-torch export).  
- It loads the adapter onto a base model for inference.  
- No import of plato-torch; only depends on `transformers` and `peft`.  
- **Data flow:** plato-torch → plato-ensign (adapter file).

**Integration gap:**  
- No round-trip: plato-ensign cannot write back to plato-torch rooms.  
- Potential P1 channel: allow ensign to generate new tiles that feed back into a room.

## 6. constraint-theory-core, cudaclaw, flux-runtime
**Connection:** ❌ **Not integrated with the four core repos**  
- These are separate technologies listed in FLEET-CONTEXT.md.  
- No imports found in plato-torch, fleet-simulator, holodeck-rust, or plato-ensign.  
- **Data flow:** standalone.

**Integration gap:**  
- constraint-theory-core (geometric snapping) could be used for spatial reasoning in fleet-simulator.  
- cudaclaw (GPU runtime) could accelerate plato-torch training.  
- flux-runtime (bytecode VM) could host holodeck NPC logic.  
- Potential P1 channel: identify one concrete hook (e.g., plug cudaclaw into plato-torch device mapping).
