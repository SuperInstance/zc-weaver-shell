## 1. Verified Current Connections

Based on reading the actual fleet repos (plato-torch, fleet-simulator, holodeck-rust, plato-ensign), here are the **existing, documented connections**:

### **plato-torch ↔ fleet-simulator**
- **Connection:** Strong, bidirectional data flow.
- **How:** `fleet-simulator` generates synthetic training data (agent interactions, environment states). `plato-torch` ingests this data via its `DataLoader` classes to train room models.
- **Evidence:** `plato-torch/data/` includes `FleetSimDataset` and `SimulationLoader`. `fleet-simulator/export/` has scripts to output `.pt` and `.jsonl` formats compatible with plato-torch.
- **Status:** **Connected and operational.** This is a primary training data pipeline.

### **plato-torch ↔ plato-ensign**
- **Connection:** Direct export pipeline.
- **How:** `plato-ensign` is a library that takes a trained `plato-torch` room model (a PyTorch module) and exports its "experience" as a LoRA adapter (`.safetensors`).
- **Evidence:** `plato-ensign/src/lib.rs` exposes `export_to_lora(model_path, output_path)`. `plato-torch/scripts/` includes `export_ensign.py` which calls this Rust function via FFI.
- **Status:** **Connected and operational.** This is the core ensign creation pathway.

### **holodeck-rust ↔ (plato-torch / fleet-simulator)**
- **Connection:** **Indirect and aspirational.** No direct code integration observed.
- **How it *could* work:** `holodeck-rust` is a MUD server with sentiment-aware NPCs. Its interaction logs could become training tiles for `plato-torch` or synthetic scenarios for `fleet-simulator`.
- **Evidence:** No import statements, shared data schemas, or dedicated pipelines between these repos currently exist. The `holodeck-rust` README mentions "future integration with fleet training systems."
- **Status:** **Not connected.** This is a significant integration gap.

## 2. Identified Integration Gaps (P1 Safe Channels)

Following Deadband Protocol: P0 (negative space) is "don't assume connections that aren't there." P1 is "find safe channels to build." These are the safe, actionable gaps:

### **Gap 1: holodeck-rust → plato-torch Tile Pipeline**
- **Problem:** Holodeck interaction logs are currently isolated. They are a rich source of human/NPC dialogue and sentiment data, unused for training.
- **Safe Channel:** Create a `HolodeckLogLoader` in `plato-torch/data/`. It would parse `holodeck-rust/logs/session_*.json` and convert them into the standard `(state, action, reward, next_state)` tile format.
- **Action:** Write a minimal adapter script. This is a pure data transformation, low risk.

### **Gap 2: fleet-simulator → holodeck-rust Scenario Injection**
- **Problem:** `fleet-simulator` creates complex multi-agent scenarios, but `holodeck-rust` worlds are static or scripted.
- **Safe Channel:** Extend `fleet-simulator` to export scenario definitions (agent roles, goals, environment props) as `holodeck-rust` "quest" YAML files. The holodeck server can load these to populate dynamic scenarios.
- **Action:** Add an export module to `fleet-simulator/export/holodeck/`. This is a one-way data flow, non-breaking.

### **Gap 3: plato-ensign → holodeck-rust NPC Brain Swap**
- **Problem:** Ensigns (LoRA adapters) are trained but not deployed live.
- **Safe Channel:** Enable `holodeck-rust` to load a `.safetensors` ensign file to alter the behavior of specific NPCs (e.g., make the "blacksmith" NPC use the "Forgemaster" ensign).
- **Action:** This requires `holodeck-rust` to integrate a lightweight inference runtime (like `candle`). Start with a proof-of-concept: a command to hot-swap an NPC's dialogue model.

### **Gap 4: Ghost Tile Injection Point**
- **Problem:** The "ghost tiles" concept (P0 knowledge from dead agents) is documented in architecture but has no implemented home.
- **Safe Channel:** `plato-torch`'s `DeadbandRoom` preset is the logical recipient. Add a `ghost_injector` hook that loads a directory of "negative example" tiles and blends them into training with a high loss weight.
- **Action:** Modify `DeadbandRoom`'s `__init__` to accept a `ghost_tile_path`. This is a configuration change, not a core logic rewrite.
