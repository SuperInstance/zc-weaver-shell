# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 110  
**Phase:** 4 — Build & Test  
**Status:** Mapping current connections and identifying integration points.

Based on reading the actual files in the fleet repos, here is the current state of integration between **plato-torch**, **fleet-simulator**, **holodeck-rust**, and **plato-ensign**.

---

## 1. **plato-torch** → **fleet-simulator**  
**Connection:** **Strong (direct)**  
- `plato-torch/src/rooms/DeadbandRoom.py` imports `fleet_simulator` directly.  
- The `DeadbandRoom` uses `fleet_simulator` to generate synthetic training data (agent trajectories, P0/P1/P2 decisions).  
- The simulator provides a `SimulationEnvironment` class that feeds into the room’s `generate_training_batch()` method.  
- **Integration point:** `DeadbandRoom` config includes `simulator_config` dict.

**Gap:**  
- The simulator runs as a separate service, but the room calls it synchronously. No async or queue-based integration observed.  
- No evidence of simulator state being persisted back to the room’s tile storage.

---

## 2. **plato-torch** → **holodeck-rust**  
**Connection:** **Weak (planned, not wired)**  
- `holodeck-rust/README.md` describes “sentiment-aware NPCs” and a MUD server intended to generate rich interaction tiles.  
- `plato-torch/src/rooms/HolodeckRoom.py` exists but is a stub—only contains placeholder logic and `TODO: integrate with holodeck server`.  
- No actual network calls or Rust FFI bindings found in the codebase.

**Gap:**  
- No live connection between the Python training room and the Rust MUD server.  
- The `HolodeckRoom` does not yet produce real tiles from NPC interactions.

---

## 3. **plato-torch** → **plato-ensign**  
**Connection:** **Strong (export pipeline)**  
- `plato-ensign/src/exporter.py` defines `export_room_to_lora(room_id, output_path)`.  
- `plato-torch/src/management/export_ensign.py` provides a CLI tool that:  
  1. Loads a trained room state (weights + tiles)  
  2. Calls the ensign exporter to produce a LoRA adapter  
  3. Saves to `models/ensigns/`  
- Works for any room that inherits from `BaseRoom`.

**Gap:**  
- Export is manual/CLI-driven; no automatic triggering when room reaches certain performance thresholds.  
- No observed feedback loop where ensigns are re-imported to improve rooms.

---

## 4. **fleet-simulator** → **holodeck-rust**  
**Connection:** **None observed**  
- The simulator generates agent decision paths, not narrative interactions.  
- No Rust dependencies or API calls in the simulator code.  
- The holodeck’s NPC sentiment engine could theoretically consume simulator trajectories as backstory, but no integration exists.

---

## 5. **plato-ensign** → **holodeck-rust**  
**Connection:** **None observed**  
- Ensigns are LoRA adapters for PLATO rooms, not directly usable in the Rust MUD.  
- No Rust code loading `.safetensors` or integrating with PyTorch models.

---

## 6. **fleet-simulator** → **plato-ensign**  
**Connection:** **Indirect**  
- Simulator data trains rooms; rooms can be exported to ensigns.  
- No direct simulator→ensign pipeline.

---
