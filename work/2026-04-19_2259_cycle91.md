## 2. Identified Integration Gaps (Actionable for Phase 4)

### Gap 1: plato-torch → holodeck-rust Live Tile Stream
- **What's missing:** A defined channel for plato-torch to send real-time interaction tiles to holodeck-rust to influence NPC behavior.
- **P0 (Negative Space):** Don't assume a shared database exists.
- **P1 (Safe Channel):** Use a simple file watch or HTTP webhook. plato-torch writes tiles to a directory; holodeck-rust watches it.
- **P2 (Optimize):** Use a message queue (Redis, ZeroMQ) for low-latency streaming.

### Gap 2: fleet-simulator → plato-torch Structured Ingestion
- **What's missing:** A pipeline to convert fleet-simulator's output into plato-torch compatible "tiles."
- **P0:** Don't modify plato-torch's tile schema arbitrarily.
- **P1:** Write an adapter script that maps simulator events to tile JSON format.
- **P2:** Build a real-time simulator plugin that emits tiles directly.

### Gap 3: holodeck-rust → plato-ensign (NPC Experience Export)
- **What's missing:** Ability to export holodeck NPC interactions as "room experience" to train ensigns.
- **P0:** Don't break holodeck's MUD server loop.
- **P1:** Log NPC sentiment/decision events in tile-like format. Post-process into plato-ensign input.
- **P2:** Integrate plato-ensign as a library inside holodeck for on-the-fly adapter creation.

### Gap 4: Deadband Protocol → All Systems (Operational Integration)
- **What's missing:** The Deadband Protocol (P0/P1/P2) is doctrine, but not visibly embedded as a service or library in these codebases.
- **P0:** Don't add protocol logic to core simulation loops prematurely.
- **P1:** Create a shared Rust crate (`deadband-core`) with protocol logic. Have each system call it for decision thresholds.
- **P2:** Implement protocol-aware agents in fleet-simulator and holodeck that use the crate.
