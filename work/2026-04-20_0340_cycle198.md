# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 198  
**Phase:** 4 — Build & Test  
**Status:** Updated map based on actual code inspection of plato-torch, fleet-simulator, holodeck-rust, and plato-ensign.

Based on reading the actual code in the fleet repos, here is the current state of integration between the four key technologies.

## 1. **plato-torch** (Training Rooms)
- **Purpose**: Provides 25 preset training rooms and self-training rooms.
- **Connections**:
  - **plato-ensign**: Direct integration via `export_room_experience()` function. Rooms can export their accumulated interaction tiles as LoRA adapters.
  - **fleet-simulator**: No direct code connection found. Plato-torch appears to operate independently as a training environment.
  - **holodeck-rust**: No direct integration found in code. Plato-torch is a separate training system.
- **Integration Points**: 
  - `DeadbandRoom` preset exists (P0/P1/P2 protocol training).
  - Rooms accumulate `interaction_tiles` which are the primary training data.
  - `GhostInjector` class exists but is not wired to external systems.

## 2. **fleet-simulator** (Fleet Simulation)
- **Purpose**: Generates training data through simulated fleet interactions.
- **Connections**:
  - **plato-torch**: No direct import or usage found. Simulator runs independently.
  - **plato-ensign**: No direct connection found in code.
  - **holodeck-rust**: No direct connection found.
- **Integration Points**:
  - Simulates agent interactions and produces `simulation_logs`.
  - Output format is JSON-based but not explicitly connected to room tile format.
  - Could potentially feed into plato-torch rooms as training data, but no adapter exists.

## 3. **holodeck-rust** (MUD Server)
- **Purpose**: Multi-user dungeon server with sentiment-aware NPCs.
- **Connections**:
  - **plato-torch**: No direct connection found.
  - **fleet-simulator**: No direct connection found.
  - **plato-ensign**: No direct connection found.
- **Integration Points**:
  - Uses `sentiment_engine` for NPC reactions.
  - Game state is stored in Redis.
  - No import of plato-torch room structures or tile formats.

## 4. **plato-ensign** (LoRA Export)
- **Purpose**: Exports room experience from plato-torch into LoRA adapters.
- **Connections**:
  - **plato-torch**: Direct dependency. Imports `Room` and `TileBank` classes.
  - **fleet-simulator**: No connection found.
  - **holodeck-rust**: No connection found.
- **Integration Points**:
  - `export_lora_adapter(room_id, output_path)` function directly reads plato-torch room data.
  - Converts interaction tiles into training dataset format.
  - Uses PyTorch and peft library for LoRA creation.

## Integration Gaps (What's Not Connected)

### Critical Gaps:
1. **fleet-simulator → plato-torch**: Simulation data cannot directly feed training rooms. No adapter exists to convert simulation logs to interaction tiles.
2. **holodeck-rust → plato-torch**: Game interactions from the MUD are not captured as training tiles. Sentiment-aware NPC reactions could provide valuable training data.
3. **GhostInjector isolation**: The `GhostInjector` class in plato-torch exists but has no connections to external systems or data sources.

### Moderate Gaps:
4. **plato-ensign limited scope**: Only exports from plato-torch rooms. Cannot create ensigns from fleet-simulator or holodeck-rust data.
5. **No shared tile format**: Each system uses different data structures. No common `Tile` interface exists across systems.

### Minor Gaps:
6. **Configuration isolation**: Each system has separate config files. No shared configuration management.
7. **Logging fragmentation**: Different logging formats and destinations across systems.

## Recommended Integration Tasks (P1 Safe Channels)

1. **Create SimulationTileAdapter**: Convert fleet-simulator JSON logs to plato-torch interaction tiles.
2. **Wire GhostInjector to holodeck**: Allow injection of ghost tiles from holodeck game sessions.
3. **Define Common Tile Interface**: Create `fleet-tile-spec` that all systems can implement.
4. **Build plato-relay bridge**: Connect DeadbandRoom outputs to other fleet services.

- Other systems operate in isolation.
- Significant opportunity for creating training data pipelines.
