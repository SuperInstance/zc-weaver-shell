# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 112  
**Phase:** 4 — Build & Test  
**Status:** Mapping current connections and identifying integration points.

Based on reading the actual files in the fleet repos, here is the current state of integration between the four core technologies.

## 1. plato-torch (Training Rooms)
**Primary Role:** Hosts 25 training room presets (DeadbandRoom, MirrorRoom, etc.) where agents interact and generate training tiles.  
**Key Files:** `rooms/deadband_room.py`, `rooms/mirror_room.py`, `tile_server.py`, `tile_client.py`  
**Current Connections:**
- **Connected to plato-ensign:** Exports room experience via `export_lora_adapter()` method in `deadband_room.py`. This writes a LoRA adapter to disk after room training.
- **Connected to fleet-simulator:** Not directly. However, `tile_server.py` can receive tiles from any client, which could include a fleet-simulator agent.
- **Connected to holodeck-rust:** No direct connection found. Rooms are Python-based; holodeck is a Rust MUD server.
- **Internal:** Rooms use `tile_client.py` to submit tiles to `tile_server.py` (local or remote). The server stores tiles in `tiles/` directory.

## 2. plato-ensign (LoRA Export)
**Primary Role:** Exports trained room experience as LoRA adapters for specialist models.  
**Key Files:** `ensign/exporter.py`, `ensign/lora_builder.py`  
**Current Connections:**
- **Connected to plato-torch:** Imports `DeadbandRoom` and calls `export_lora_adapter()`. The exporter reads room state and builds a LoRA.
- **Connected to fleet-simulator:** No connection found.
- **Connected to holodeck-rust:** No connection found.
- **Note:** The export is a one-way process: room → LoRA. No feedback loop.

## 3. fleet-simulator (Fleet Training Data)
**Primary Role:** Simulates fleet interactions to generate training data for rooms.  
**Key Files:** `simulator/agent.py`, `simulator/scenario.py`, `simulator/logger.py`  
**Current Connections:**
- **Connected to plato-torch:** Not directly. The simulator logs interactions to JSON files (`logs/`). No automatic tile submission found.
- **Connected to plato-ensign:** No connection found.
- **Connected to holodeck-rust:** No connection found.
- **Internal:** Agents run scenarios and produce log files. These could be manually converted to tiles.

## 4. holodeck-rust (Sentiment-Aware MUD)
**Primary Role:** Multi-user dungeon server with NPCs that react to player sentiment.  
**Key Files:** `src/npc/sentiment.rs`, `src/world/room.rs`, `src/server.rs`  
**Current Connections:**
- **Connected to plato-torch:** No connection found. Different language (Rust vs Python) and protocol (TCP MUD vs HTTP tile server).
- **Connected to plato-ensign:** No connection found.
- **Connected to fleet-simulator:** No connection found.
- **Internal:** Uses a custom TCP protocol. NPCs track player sentiment but don’t export training data.

## Integration Gaps (P1 Safe Channels)
1. **fleet-simulator → plato-torch:** Simulator logs are not automatically converted to tiles. A `LogToTile` converter could bridge this.
2. **holodeck-rust → plato-torch:** No pipeline from MUD interactions to training tiles. A sentiment-aware tile generator could listen to MUD events.
3. **plato-torch → holodeck-rust:** Rooms don’t influence MUD NPCs. A “Deadband NPC” module could import room-trained LoRAs.
4. **plato-ensign feedback loop:** Exported LoRAs are not re-imported into rooms for iterative training.

## P0 Negative Space (What NOT to Do)
- Do not assume these systems are already wired together.
- Do not force incompatible protocols (e.g., direct TCP between Python and Rust without an adapter).
- Do not modify core room logic without testing tile compatibility.
