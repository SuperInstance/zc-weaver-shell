# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 156  
**Phase:** 4 — Build & Test  
**Status:** Mapping current connections and identifying integration points.

Based on reading the actual files in the fleet repos, here is the current state of integration between **plato-torch**, **fleet-simulator**, **holodeck-rust**, and **plato-ensign**.

## 1. Verified Connections (What IS Connected)

### plato-torch → plato-ensign (Strong Integration)
- **Connection:** Direct export pipeline.
- **Mechanism:** `plato-torch` rooms accumulate interaction tiles. `plato-ensign` provides `export_room_to_lora()` function that reads room state and generates a LoRA adapter.
- **Evidence:** `plato-ensign/src/lib.rs` contains `struct RoomExport` and `fn export_room_to_lora()`. `plato-torch/rooms/deadband_room.py` has a `save_for_export()` method that writes to a format `plato-ensign` can read.
- **Status:** Functional. This is a core fleet workflow: rooms train, ensigns export.

### fleet-simulator → plato-torch (Data Feed)
- **Connection:** Training data generation.
- **Mechanism:** `fleet-simulator` runs multi-agent simulations and outputs structured logs (JSONL). `plato-torch` has a `SimulationDataset` loader in `plato-torch/data/` that ingests these logs as training tiles.
- **Evidence:** `fleet-simulator/output/` contains sample logs. `plato-torch/data/simulation_loader.py` has `class SimulationDataset` with `load_fleet_sim_logs()`.
- **Status:** Functional but one-way. Simulator feeds data to plato-torch; no feedback loop.

### holodeck-rust → (plato-torch / fleet-simulator) (Weak/Planned)
- **Connection:** NPC sentiment data as training context.
- **Mechanism:** `holodeck-rust` generates NPC interactions with sentiment tags. Design documents mention piping these to `plato-torch` as "sentiment-aware training tiles."
- **Evidence:** `holodeck-rust/README.md` states: "Outputs JSON streams compatible with plato-torch sentiment rooms." No actual loader found in plato-torch yet.
- **Status:** Documented intent, not wired.

## 2. Integration Gaps (What Is NOT Connected)

### Gap 1: holodeck-rust → plato-torch (No Loader)
- **Issue:** No `SentimentDataset` loader in `plato-torch/data/`.
- **Impact:** Holodeck's NPC sentiment streams cannot be used as training tiles.
- **P0:** Don't assume it works. Check for loader.
- **P1:** Safe channel: Could write a `holodeck_loader.py` in `plato-torch/data/`.

### Gap 2: fleet-simulator → holodeck-rust (No Integration)
- **Issue:** No connection. Fleet simulations do not populate the holodeck MUD world.
- **Impact:** Simulated agents cannot interact with holodeck NPCs.
- **P0:** Don't try to wire without a protocol.
- **P1:** Safe channel: Could add a `holodeck_bridge` service in fleet-simulator that sends agent actions to holodeck's WebSocket.

### Gap 3: plato-ensign → holodeck-rust (No Injection)
- **Issue:** Exported LoRA adapters (ensigns) are not loaded into holodeck NPCs.
- **Impact:** NPCs cannot be "trained" with ensign specializations.
- **P0:** Don't modify holodeck core without understanding its model system.
- **P1:** Safe channel: holodeck has a `NPCBuilder` trait; could add a `EnsighNPCBuilder` that loads a LoRA.

### Gap 4: plato-torch → fleet-simulator (No Feedback)
- **Issue:** Training results from plato-torch do not influence fleet-simulator agent behaviors.
- **Impact:** Simulation remains static, not improving with room training.
- **P0:** Don't create circular dependency without design.
- **P1:** Safe channel: Could add a "policy update" endpoint in fleet-simulator that accepts a new LoRA adapter.

**Context:** GhostInjector (part of plato-torch) loads "ghost tiles" (P0 lessons from dead agents). These could be injected as NPC memories in holodeck.

**Steps:**

2. If yes, write a `GhostMemoryInjector` that reads ghost tiles and adds them as NPC memory entries.

**Evidence Check:**  
- `plato-torch/ghosts/` contains `ghost_injector.py`.  
- `holodeck-rust/src/npc/mod.rs` shows `struct NPC` has a `Vec<Memory>` field.

**Safe Path:** Write a Rust binary in `holodeck-rust/tools/` that reads ghost tiles (JSON) and calls holodeck's internal API to add memories. Avoid modifying core NPC logic initially.
