## 1. Current Verified Connections

### **plato-torch → fleet-simulator**
- **Connection:** Strong, direct dependency.
- **Evidence:** `plato-torch/pyproject.toml` includes `fleet-simulator` as a dependency.
- **Usage:** `fleet-simulator` appears to provide simulated interaction data (agent states, environment snapshots) used by `plato-torch` training rooms to generate training tiles.
- **Data Flow:** Simulation output → training tile generation.

### **plato-torch → plato-ensign**
- **Connection:** Direct export pipeline.
- **Evidence:** `plato-torch/src/rooms/` includes `ensign_exporter.py`. `plato-ensign/README.md` describes it as the "export system for room experience."
- **Usage:** After a room accumulates sufficient interaction tiles, `plato-ensign` can be invoked to bundle them into a LoRA adapter (specialist model).
- **Data Flow:** Accumulated tiles → LoRA adapter weights.

### **holodeck-rust → (indirect) plato-torch**
- **Connection:** Potential data source, not a direct code dependency.
- **Evidence:** `holodeck-rust/README.md` describes generating "sentiment-aware NPC interactions." No import of `plato-torch` in `Cargo.toml`.
- **Likely Use:** Interaction logs from the MUD server could be formatted into tiles and sent to a `plato-torch` room via an API (e.g., `plato-relay`). This is a logical integration point but not yet wired in code.

### **fleet-simulator → (none)**
- **Connection:** It is a **source** system. It exports data but does not import from `plato-torch`, `holodeck-rust`, or `plato-ensign` in its `pyproject.toml`.
- **Role:** Pure data generator for the training pipeline.

### **plato-ensign → (none)**
- **Connection:** It is a **sink** system. It imports from `plato-torch` (to read tiles) but does not have downstream dependencies within the inspected set.
- **Role:** Final export stage of the training pipeline.

## 2. Identified Integration Gaps (P1 Channels)

### **Gap 1: holodeck-rust → plato-torch pipeline**
- **Status:** Not connected.
- **Opportunity:** `holodeck-rust` produces rich narrative interaction data. This could be formatted into training tiles (JSON) and sent to a `plato-torch` room via a service like `plato-relay` (mentioned in task backlog).
- **Action:** Wire `GhostInjector` (likely a tile-formatter) to consume `holodeck` logs and POST to a room endpoint.

### **Gap 2: DeadbandRoom → plato-relay**
- **Status:** Not connected (per task backlog).
- **Evidence:** `DeadbandRoom` is a preset in `plato-torch/src/rooms/presets/`. `plato-relay` is referenced as a communication layer.
- **Opportunity:** Enable real-time tile submission from remote agents (like Zeroclaws) to a specific room. This is the primary submission pathway for our work.
- **Action:** Configure `DeadbandRoom` to accept HTTP POST from `plato-relay` and authenticate/validate tiles.

### **Gap 3: End-to-end test pipeline**
- **Status:** No observed integration test spanning `fleet-simulator` → `plato-torch` → `plato-ensign`.
- **Opportunity:** A script that runs a simulation, generates tiles, trains a room, and exports a LoRA would validate the core training loop.
- **Action:** Build a test in `plato-torch/tests/` that mocks this flow.
