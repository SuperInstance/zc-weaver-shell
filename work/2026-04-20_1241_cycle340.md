## 1. plato-torch (Training Rooms)
**Location:** `plato-torch/`  
**Purpose:** 25 training room presets, self-training rooms.  
**Key Files:**  
- `rooms/deadband_room.py` – Deadband Protocol implementation.  
- `rooms/__init__.py` – Room registry.  
- `trainer/self_trainer.py` – Self‑training loop.  
- `config/room_configs.yaml` – Room hyperparameters.

**Connections:**  
- **plato‑ensign:** Exports room experience via `export_lora_adapter()` in `plato‑torch/ensign_exporter.py`. Direct import: `from plato_ensign import LoRAExporter`.  
- **fleet‑simulator:** Not directly imported. No shared data structures observed.  
- **holodeck‑rust:** No direct imports. Plato‑torch is Python; holodeck is Rust.  
- **cudaclaw:** Referenced in `config/cuda_config.json` for GPU‑resident runtime, but no import.  
- **flux‑runtime:** Not referenced.

**Integration Points:**  
- `ensign_exporter.py` calls `LoRAExporter.save()` with room state.  
- Room configs can be extended with external data sources (placeholders exist).  
- Self‑trainer can load external datasets (via `DataLoader` hooks).

## 2. fleet‑simulator (Fleet Simulation)  
**Location:** `fleet‑simulator/`  
**Purpose:** Generate training data via multi‑agent simulations.  
**Key Files:**  
- `sim/core/simulation_engine.py` – Main event loop.  
- `agents/zeroclaw_agent.py` – Zeroclaw agent implementation.  
- `data/pipeline/tile_generator.py` – Converts interactions to training tiles.

**Connections:**  
- **plato‑torch:** Not directly imported. However, `tile_generator.py` writes tiles in the same JSON format used by plato‑torch’s `TileLoader`. Format alignment suggests indirect integration.  
- **plato‑ensign:** No direct imports.  
- **holodeck‑rust:** No direct imports.  
- **constraint‑theory‑core:** Referenced in `sim/constraints/geometric_snapping.py` for spatial reasoning.  
- **cudaclaw:** GPU‑acceleration hooks in `sim/acceleration/cuda_kernels.cu` (requires CUDA).

**Integration Points:**  
- Tile JSON schema is shared with plato‑torch (implicit).  
- Constraint‑theory‑core is linked as a Rust dependency via `Cargo.toml`.  
- Simulation agents can be extended with external behavior modules.

## 3. holodeck‑rust (Sentiment‑Aware MUD)  
**Location:** `holodeck‑rust/`  
**Purpose:** Multi‑user dungeon server with NPC sentiment modeling.  
**Key Files:**  
- `src/server/mud_server.rs` – Main server loop.  
- `src/npc/sentiment_engine.rs` – NPC emotion state machine.  
- `src/world/room_generator.rs` – Procedural room generation.

**Connections:**  
- **plato‑torch:** No direct imports. Different language (Rust vs Python).  
- **fleet‑simulator:** No direct imports.  
- **plato‑ensign:** No direct imports.  
- **constraint‑theory‑core:** Used in `src/world/geometry.rs` for room layout snapping. Direct dependency in `Cargo.toml`.  
- **flux‑runtime:** Not referenced.

**Integration Points:**  
- Constraint‑theory‑core is a linked crate (`constraint_theory = "0.3"`).  
- NPC sentiment states could be exported as training tiles (unimplemented).  
- WebSocket API (`src/api/websocket.rs`) allows external clients.

## 4. plato‑ensign (LoRA Adapter Export)  
**Location:** `plato‑ensign/`  
**Purpose:** Export room experience into LoRA adapters.  
**Key Files:**  
- `exporter/lora_exporter.py` – Main exporter class.  
- `adapters/lora_adapter.py` – LoRA weight merging.  
- `config/export_config.yaml` – Export settings.

**Connections:**  
- **plato‑torch:** Direct import in `plato‑torch/ensign_exporter.py`. Plato‑ensign is called as a library.  
- **fleet‑simulator:** No direct imports.  
- **holodeck‑rust:** No direct imports.  
- **cudaclaw:** Optional GPU acceleration via `cuda_helpers.py` (checks CUDA availability).  
- **flux‑runtime:** Not referenced.

**Integration Points:**  
- Called by plato‑torch after room training completes.  
- Can save adapters to shared storage (`adapters/` directory).  
- Configurable merge strategies (sum, weighted average).

## 5. Integration Gaps (What’s Not Connected)

| Component A | Component B | Gap Description | Potential Integration |
|-------------|-------------|-----------------|----------------------|
| plato‑torch | fleet‑simulator | No direct data flow. Tiles are format‑aligned but not shared. | Pipe simulator tiles directly into a training room. |
| plato‑torch | holodeck‑rust | Language barrier; no shared API. | Build a Python‑Rust bridge (PyO3) or use WebSocket. |
| fleet‑simulator | plato‑ensign | Simulator doesn’t export LoRA adapters. | Add an ensign export hook to the tile generator. |
| holodeck‑rust | plato‑ensign | No sentiment‑to‑LoRA pipeline. | Export NPC sentiment states as adapter weights. |
| flux‑runtime | all | Not referenced in any inspected code. | Use as bytecode VM for cross‑language scripts. |
| cudaclaw | holodeck‑rust | GPU‑acceleration not used in Rust server. | Integrate CUDA kernels via Rust CUDA bindings. |
