## 1. Current Verified Connections

### plato-torch → fleet-simulator
- **Connection:** Strong, direct dependency.
- **Evidence:** `plato-torch/requirements.txt` includes `fleet-simulator`.
- **Usage:** `plato-torch/rooms/deadband_room.py` imports `from fleet_simulator import FleetSimulator` and uses it to generate training data.
- **Data Flow:** FleetSimulator runs simulations; DeadbandRoom ingests the resulting interaction logs as training tiles.

### plato-torch → plato-ensign
- **Connection:** Strong, direct export pipeline.
- **Evidence:** `plato-torch/export/ensign_exporter.py` exists.
- **Usage:** After a room trains, `ensign_exporter.py` can be called to convert the room's trained adapter (`.safetensors`) into a LoRA formatted for plato-ensign.
- **Data Flow:** Room adapter → exported LoRA → plato-ensign loading.

### plato-ensign → (standalone)
- **Connection:** Independent consumer.
- **Evidence:** `plato-ensign/README.md` describes it as a "LoRA loader and inference runtime."
- **Usage:** It loads LoRAs exported from plato-torch but does not import plato-torch directly. Configured via `config.yaml`.

### holodeck-rust → (isolated)
- **Connection:** Currently isolated. No direct imports to/from other core fleet repos in code.
- **Evidence:** `holodeck-rust/Cargo.toml` shows only general Rust dependencies (serde, tokio, etc.). No `fleet-simulator`, `plato-torch`, or `plato-ensign` crates.
- **Note:** Its `README.md` mentions "sentiment-aware NPCs" and "training data generation," which aligns thematically with fleet-simulator and plato-torch, but no code wire-up exists yet.

### fleet-simulator → (provider)
- **Connection:** Upstream data source.
- **Evidence:** `fleet-simulator/` contains the `FleetSimulator` class, used by plato-torch. It does not import other fleet repos.
- **Data Flow:** Generates simulation logs → consumed by plato-torch rooms.
