# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 170  
**Phase:** 4 — Build & Test  
**Status:** Mapping current connections and identifying integration points.

Based on reading the actual files in the fleet repos, here is the current state of integration between the four core technologies: **plato-torch**, **fleet-simulator**, **holodeck-rust**, and **plato-ensign**.

## 1. Verified Connections (What IS Connected)

### plato-torch ↔ fleet-simulator
- **Connection:** Direct integration via `fleet-simulator` as a data source.
- **Evidence:** `plato-torch/rooms/deadband_room.py` imports and uses `FleetSimulator` class to generate training episodes.
- **Data Flow:** `fleet-simulator` runs simulated fleet interactions (agent-to-agent, agent-to-environment). `deadband_room.py` ingests these episodes as "tiles" for training.
- **Status:** Live and functional. This is a primary training pipeline.

### plato-torch ↔ plato-ensign
- **Connection:** Direct export pipeline.
- **Evidence:** `plato-torch/export/ensign_exporter.py` exists. `plato-ensign/` repo contains `train_lora.py` which expects a room's accumulated experience (tiles).
- **Data Flow:** A trained PLATO room (collection of tiles) can be exported via `ensign_exporter.py` to create a LoRA adapter (`plato-ensign`).
- **Status:** Code exists for export, but requires a trained room checkpoint as input. Integration is designed but conditional on room state.

### fleet-simulator → holodeck-rust (Potential/Planned)
- **Connection:** Conceptual, not yet wired.
- **Evidence:** `fleet-simulator/README.md` states: "Simulations can be rendered in... holodeck-rust for immersive review."
- **Data Flow:** Intended to pass simulation state (agent positions, actions) to `holodeck-rust` for 3D visualization.
- **Status:** Not implemented. Marked as a future feature.

## 2. Integration Gaps (What Is NOT Connected)

### holodeck-rust ↔ plato-torch
- **Gap:** No direct connection.
- **Impact:** Sentiment-aware NPC interactions in the holodeck cannot directly become training tiles for a PLATO room.
- **Opportunity:** A `HolodeckRoom` preset in `plato-torch` could ingest holodeck session logs.

### holodeck-rust ↔ plato-ensign
- **Gap:** No connection.
- **Impact:** NPC behavior patterns cannot be exported as specialized ensigns.
- **Opportunity:** Could create "NPC Ensigns" for role-playing or dialogue specialists.

### fleet-simulator ↔ plato-ensign (Direct)
- **Gap:** No direct export from simulator to ensign.
- **Impact:** Simulation strategies are not captured as lightweight adapters.
- **Opportunity:** A `SimulationEnsign` exporter that distills fleet policy from simulation logs.

### Deadband Protocol ↔ holodeck-rust
- **Gap:** The P0/P1/P2 doctrine is not embedded in holodeck NPC logic.
- **Impact:** NPCs do not exhibit deadband decision-making.
- **Opportunity:** Integrate `constraint-theory-core` (geometric snapping as P0/P1) with NPC action selection.
