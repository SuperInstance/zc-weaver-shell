## 1. Verified Active Connections

### plato-torch → fleet-simulator
- **Connection:** Direct import and usage.
- **Evidence:** In `plato-torch/src/training/rooms/deadband_room.py`:
  ```python
  from fleet_simulator.simulator import FleetSimulator
  ```
- **Purpose:** `DeadbandRoom` uses `FleetSimulator` to generate synthetic training data (agent interaction trajectories) for P0/P1/P2 protocol training.
- **Data Flow:** Simulation runs produce `InteractionTile` objects which are stored in the room's tile buffer.

### plato-torch → plato-ensign
- **Connection:** Optional export pipeline.
- **Evidence:** `plato-torch/src/export/ensign_exporter.py` imports:
  ```python
  from plato_ensign.lora_adapter import LoRAAdapterBuilder
  ```
- **Purpose:** After a room accumulates sufficient tiles, the `ensign_exporter` can package the room's trained weights and tile history into a LoRA adapter file (`.safetensors`).
- **Trigger:** Manual or automated export via room configuration flag `export_to_ensign: bool`.

### holodeck-rust → (indirect, via shared protocol)
- **Connection:** No direct source-level imports between holodeck-rust and plato-torch/fleet-simulator.
- **Evidence:** However, `holodeck-rust/src/protocol/mod.rs` defines a `Tile` struct with fields (`id`, `content`, `sentiment`, `tags`) that aligns with the `InteractionTile` class in `plato-torch/src/core/tile.py`.
- **Purpose:** Structural compatibility suggests intent for future tile exchange. Currently, they operate on similar data concepts but separate runtimes.

## 2. Integration Gaps (What's Not Connected)

### holodeck-rust ←/→ plato-torch
- **Gap:** No Rust/Python interop layer observed.
- **Detail:** `holodeck-rust` is a standalone MUD server (Tokio async). `plato-torch` is a Python training framework. No `pyo3` bindings or gRPC service observed in either codebase.
- **Consequence:** Tiles generated in holodeck (from NPC interactions) cannot directly feed into plato-torch rooms, and vice-versa.

### fleet-simulator → holodeck-rust
- **Gap:** No connection.
- **Detail:** `fleet-simulator` is a Python discrete-event simulator for agent fleets. It outputs trajectory logs. No hooks or adapters to pipe these logs into the holodeck MUD server's event system.
- **Consequence:** Simulated fleet behaviors cannot directly drive or populate the sentiment-aware NPC system.

### plato-ensign → holodeck-rust
- **Gap:** No connection.
- **Detail:** `plato-ensign` produces LoRA adapters (`.safetensors`). No observed loading mechanism in `holodeck-rust` to apply these adapters to its embedded NPC language models (if any).
- **Consequence:** Specialized ensigns trained in plato-torch cannot be used to modulate holodeck NPC behavior.

### GhostInjector (concept) → Any Runtime
- **Gap:** No implementation found.
- **Detail:** "Ghost tiles" are described in architecture docs as dead agents' lessons (P0 knowledge). No `GhostInjector` class or service found in `plato-torch`, `fleet-simulator`, or `holodeck-rust` repositories.
- **Consequence:** P0 knowledge from failed runs remains an architectural concept, not an operational data stream.

## 3. Data Structure Alignment Check

| Component         | Core Tile Structure                            | Match? |
|-------------------|------------------------------------------------|--------|
| plato-torch       | `InteractionTile(id, agent_id, content, phase, tags, embedding)` | Baseline |
| fleet-simulator   | `TrajectoryStep(agent_id, action, observation, reward)`          | Partial (different fields, but `agent_id` common) |
| holodeck-rust     | `Tile { id: Uuid, content: String, sentiment: f32, tags: Vec<String> }` | High (same field names, Rust vs Python) |
| plato-ensign      | Exports `state_dict` + `tile_history` metadata                   | Metadata includes tile IDs |

**Observation:** Tile structures are similar between plato-torch and holodeck-rust, suggesting a **potential integration channel** via a shared serialization format (e.g., JSON lines).

## 4. Recommended Integration Wiring (P1 Safe Channel)

1. **Bridge holodeck-rust ↔ plato-torch** via a simple file watcher or HTTP webhook.
   - **P1 Action:** Create a small service `tile_bridge` that reads `Tile` JSON lines from holodeck's log and posts them to plato-torch's room ingestion endpoint (if exists) or writes to a shared directory monitored by the room.

2. **Connect fleet-simulator logs to holodeck-rust** as "historical event streams".
   - **P1 Action:** Add a `holodeck_exporter` module to fleet-simulator that converts `TrajectoryStep` sequences into holodeck's `Tile` format and writes to a socket or Redis stream that holodeck can consume.

3. **Implement GhostInjector as a plato-torch plugin**.
   - **P1 Action:** Create `GhostInjector` class that reads a directory of archived "ghost tiles" (JSON files) and periodically injects them into a room's tile buffer during training, tagged with `phase: P0`.

These are safe, low-risk channels that avoid deep runtime coupling and respect the current architectural boundaries.
