# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 106

Based on reading the actual fleet repositories (plato-torch, fleet-simulator, holodeck-rust, plato-ensign), here is the current state of connections and integration gaps.

## Verified Connections

### 1. **plato-torch → fleet-simulator**
- **Connection:** Direct integration via `fleet_simulator` module in plato-torch.
- **Evidence:** `plato-torch/src/rooms/deadband_room.py` imports `from fleet_simulator import FleetSimulator`.
- **Purpose:** DeadbandRoom uses FleetSimulator to generate training data (agent interactions) for tile creation.
- **Data Flow:** FleetSimulator runs simulated agent interactions → produces interaction logs → DeadbandRoom converts logs to training tiles.

### 2. **plato-torch → plato-ensign**
- **Connection:** Direct export pipeline.
- **Evidence:** `plato-torch/src/export/ensign_exporter.py` handles conversion of room experience (tiles) to LoRA adapters.
- **Purpose:** After a room accumulates sufficient tiles, plato-ensign can export a specialized model (ensign).
- **Data Flow:** Room tiles → aggregated training data → LoRA adapter generation via plato-ensign.

### 3. **holodeck-rust → (indirect) plato-torch**
- **Connection:** Potential integration via sentiment-aware NPC interactions.
- **Evidence:** `holodeck-rust/src/npc/sentiment.rs` defines NPC behaviors that could generate interaction logs.
- **Gap:** No direct import or API call found in plato-torch to holodeck-rust. Likely planned but not yet wired.
- **Potential Flow:** holodeck NPC interactions → log format compatible with fleet-simulator → feed into DeadbandRoom.

## Integration Gaps (What’s Not Connected)

### 1. **holodeck-rust ↔ fleet-simulator**
- **Status:** Not connected.
- **Evidence:** No shared modules, imports, or data formats between the two codebases.
- **Impact:** Holodeck’s sentiment-aware NPCs cannot currently contribute to fleet simulation training data.
- **Opportunity:** Define a common interaction log schema (JSON) that both can produce/consume.

### 2. **plato-ensign → holodeck-rust**
- **Status:** Not connected.
- **Evidence:** plato-ensign outputs LoRA adapters (.safetensors), but holodeck-rust has no loader for runtime model updates.
- **Impact:** Ensigns (specialist models) cannot be deployed directly into holodeck NPCs.
- **Opportunity:** Add a `model_updater` module in holodeck-rust to hot-swap NPC AI using exported adapters.

### 3. **GhostInjector (plato-torch) → holodeck-rust**
- **Status:** Not connected.
- **Evidence:** `plato-torch/src/rooms/ghost_injector.py` exists but no integration points to holodeck.
- **Impact:** Ghost tiles (P0 knowledge from dead agents) cannot be injected into holodeck NPC memory.
- **Opportunity:** Extend GhostInjector to write to holodeck’s NPC memory store (e.g., via Redis or shared SQLite).

### 4. **DeadbandRoom → plato-relay**
- **Status:** Not connected (per backlog).
- **Evidence:** `plato-relay` is a separate service (message routing) but no subscriber/publisher pattern found in DeadbandRoom.
- **Impact:** Room training progress cannot be broadcast to other fleet components in real-time.
- **Opportunity:** Implement a `RelayClient` in DeadbandRoom to emit tile-acceptance events.
