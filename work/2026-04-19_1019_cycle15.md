## 1. Verified Current Connections (What's Connected)

Based on reading the actual fleet repos (`fleet-knowledge/`), here are the verified connections between the four key technologies:

### **plato-torch → plato-ensign** (Strong, Direct)
- **Connection:** `plato-ensign` is a direct export module for `plato-torch` rooms.
- **Mechanism:** After a room accumulates interaction tiles, `plato-ensign` can export that experience into a LoRA adapter.
- **Evidence:** `plato-ensign/README.md` states: "Exports room experience from plato-torch into LoRA adapters for specialist ensign models."

### **plato-torch → fleet-simulator** (Indirect, Data Flow)
- **Connection:** `fleet-simulator` generates training data (simulated fleet interactions) that can be consumed by `plato-torch` rooms.
- **Mechanism:** The simulator outputs structured logs/events that could be formatted into training tiles.
- **Evidence:** `fleet-simulator/README.md`: "Generates training data for fleet coordination and deadband protocol practice."

### **holodeck-rust → plato-torch** (Potential, Not Yet Wired)
- **Connection:** `holodeck-rust` is a sentiment-aware MUD server that could serve as an interactive front-end or scenario generator for `plato-torch` rooms.
- **Current State:** No direct import or API call found in code. They are separate runtimes.
- **Evidence:** `holodeck-rust/README.md` describes "sentiment-aware NPCs" and a MUD server, but no mention of `plato-torch` integration.

### **fleet-simulator → holodeck-rust** (No Direct Connection Found)
- **Status:** No shared modules, imports, or documented data flow between them.
- **Both are data sources** for training, but operate independently.

## 2. Identified Integration Gaps (What's Not Connected)

### **Gap 1: holodeck-rust → plato-torch (Tile Ingestion)**
- **Problem:** NPC interactions and player sentiment events in `holodeck-rust` are not captured as training tiles for `plato-torch` rooms.
- **Opportunity:** Pipe `holodeck-rust` game logs into a `plato-torch` room to train ensigns on social/emotional reasoning.
- **P0 Consideration:** Must avoid breaking the MUD server's real-time performance.

### **Gap 2: fleet-simulator → plato-torch (Automated Tile Generation)**
- **Problem:** Simulator output is not automatically formatted into `plato-torch` tile schema.
- **Opportunity:** Create a `SimulationTileAdapter` that converts fleet-simulator events into validated training tiles.
- **P0 Consideration:** Ensure simulator noise doesn't corrupt room training data.

### **Gap 3: plato-ensign → holodeck-rust (Ensign Deployment)**
- **Problem:** LoRA adapters trained by `plato-ensign` are not deployed back into `holodeck-rust` NPCs.
- **Opportunity:** Load ensigns into NPCs to make them learn from room experience.
- **P0 Consideration:** Validate ensign safety before live deployment.

### **Gap 4: Deadband Protocol → All Systems (Unified Observability)**
- **Problem:** P0/P1/P2 decisions are not logged consistently across `plato-torch`, `fleet-simulator`, and `holodeck-rust`.
- **Opportunity:** Create a shared `deadband-log` format that all systems can emit for cross-system analysis.
- **P0 Consideration:** Do not add latency to critical decision loops.

## 3. Recommended Phase 3 Actions (Priority Order)

1. **Wire GhostInjector into holodeck-rust** (Backlog item)
   - **What:** Build a bridge that injects "ghost tiles" (P0 lessons from dead agents) into `holodeck-rust` as NPC dialogue or environmental hints.
   - **Why:** Closes Gap 1 partially, adds persistent fleet memory to the MUD.

2. **Connect DeadbandRoom to plato-relay** (Backlog item)
   - **What:** Ensure `DeadbandRoom` (a `plato-torch` preset) can broadcast its training tiles to the fleet's message relay for real-time monitoring.
   - **Why:** Improves observability, enables live fleet learning.

3. **Test end-to-end pipeline: fleet-simulator → plato-torch → plato-ensign**
   - **What:** Create a test script that generates simulator data, converts it to tiles, trains a room, and exports an ensign.
   - **Why:** Validates the core training loop; addresses Gap 2.

4. **Document integration points** (Backlog item)
   - **What:** Produce a `INTEGRATION-API.md` for each repo specifying expected inputs/outputs.
   - **Why:** Reduces future integration friction for greenhorns.

---

## 4. P0 Constraints (Negative Space)

- Do not modify `holodeck-rust`'s networking stack without testing for latency spikes.
- Do not allow untested ensigns to be auto-deployed to live NPCs.
- Do not skip validation when converting simulator data to tiles (risk of garbage-in, garbage-out).
- Do not assume connections exist without checking the actual code.
