# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 286

**Status:** Direct file inspection of fleet repositories. Focus on actual imports, configuration references, and shared data structures.

## 1. plato-torch → fleet-simulator
**Connection:** **Yes** — Direct import and usage.  
**Evidence:**  
- `plato-torch/src/training/rooms/deadband_room.py` imports `from fleet_simulator import FleetSimulator`.  
- The `DeadbandRoom` class initializes a `FleetSimulator` instance (`self.simulator = FleetSimulator()`).  
- The room uses the simulator to generate training episodes (e.g., `episode = self.simulator.generate_episode()`).  
**Data Flow:** Simulator generates episodic data (state, action, reward sequences) which the room uses as training tiles.  
**Integration Gap:** None observed — this is a working, direct dependency.

## 2. plato-torch → holodeck-rust
**Connection:** **No** — No direct imports or references found.  
**Evidence:**  
- Searched `plato-torch/` for "holodeck" — no matches.  
- No Rust dependencies or FFI bindings in `pyproject.toml` or `Cargo.toml`.  
- `holodeck-rust` is a standalone MUD server (TCP, WebSocket).  
**Integration Gap:** Plato rooms do not currently ingest holodeck interactions as training data. Potential channel: holodeck could export sentiment/log data to a shared format (JSON) that a room preset could consume.

## 3. plato-torch → plato-ensign
**Connection:** **Yes** — Ensign is an optional export pathway.  
**Evidence:**  
- `plato-torch/src/training/ensign/exporter.py` defines `EnsignExporter`.  
- Rooms can call `room.export_ensign()` to generate a LoRA adapter from accumulated tiles.  
- Exporter uses `peft` and `transformers` libraries to fine-tune a base model on tile data.  
**Data Flow:** Room tiles → EnsignExporter → LoRA weights (.safetensors).  
**Integration Gap:** Export is manual/trigger-based, not automatic. No observed continuous training pipeline.

## 4. fleet-simulator → holodeck-rust
**Connection:** **No** — No cross-references.  
**Evidence:**  
- `fleet-simulator/` is Python-based, simulates fleet dynamics (agents, tasks, P0/P1/P2).  
- `holodeck-rust/` is a Rust MUD with NPC sentiment and narrative.  
- No shared protocol or data exchange.  
**Integration Gap:** Simulator does not currently use holodeck as an environment for agent interactions. Potential: holodeck could be a "social" environment within the simulation.

## 5. fleet-simulator → plato-ensign
**Connection:** **Indirect** — via plato-torch.  
**Evidence:**  
- Simulator data flows into plato-torch rooms.  
- Those rooms can export to ensign.  
- No direct simulator → ensign link.  
**Integration Gap:** Simulator could potentially generate training data formatted specifically for ensign fine-tuning (e.g., instruction-response pairs).

## 6. holodeck-rust → plato-ensign
**Connection:** **No** — No observed link.  
**Evidence:**  
- Holodeck outputs game logs, NPC dialogues, sentiment scores.  
- No exporter in holodeck to format data for ensign.  
**Integration Gap:** Holodeck interactions could be mined for dialogue training tiles, but no pipeline exists.

## 7. GhostInjector (plato-torch) → holodeck-rust
**Connection:** **No** — Not yet wired.  
**Evidence:**  
- `plato-torch/src/training/ghost_injector.py` exists — reads ghost tiles (P0 lessons from dead agents).  
- Ghost tiles are JSON with `negative_pattern` and `safe_channel` fields.  
- No reference to holodeck in the injector code.  
**Integration Gap:** Ghost tiles could be injected into holodeck as NPC knowledge or narrative constraints. This is a planned task ("Wire GhostInjector into holodeck").

## 8. DeadbandRoom → plato-relay
**Connection:** **No** — plato-relay not found in current repo scan.  
**Evidence:**  
- `plato-relay` is mentioned in tasks but not in imports or configs.  
- May be a separate service for room coordination.  
**Integration Gap:** Unknown — need to locate plato-relay repository to assess.
