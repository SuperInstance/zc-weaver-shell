## 1. plato-torch (Training Rooms)
**Location:** `plato-torch/`  
**Purpose:** 25 training room presets, self-training rooms.  
**Key Files:**
- `rooms/deadband_room.py` — Deadband Protocol training room
- `rooms/__init__.py` — Room registry
- `trainer/self_trainer.py` — Self‑training loop
- `config/room_configs.yaml` — Room hyperparameters

**Connections Found:**
- **To plato‑ensign:** `plato-torch/rooms/base_room.py` imports `from plato_ensign.lora_exporter import LoRAExporter`.  
  This is a direct, functional integration: rooms can export their trained adapters via `export_lora()`.
- **To fleet‑simulator:** No import found. `plato-torch` does not reference `fleet‑simulator` in its code.
- **To holodeck‑rust:** No import found. No Rust bindings or API calls observed.

**Integration Notes:**  
`plato-torch` is a self‑contained PyTorch training framework. Its primary external integration is the LoRA exporter to `plato‑ensign`. Rooms are configured via YAML; there is no built‑in hook for simulation data or MUD interactions.

## 2. fleet‑simulator (Fleet Simulation)
**Location:** `fleet‑simulator/`  
**Purpose:** Generate training data from simulated fleet interactions.  
**Key Files:**
- `src/simulator/core.rs` — Core simulation engine
- `src/agents/mod.rs` — Agent definitions
- `examples/generate_tiles.rs` — Tile‑generation example

**Connections Found:**
- **To plato‑torch:** None. The simulator outputs JSON‑format tiles (`TrainingTile` struct) but no direct import or API call to `plato‑torch`.
- **To plato‑ensign:** None. No references to LoRA export.
- **To holodeck‑rust:** None. Separate codebase; no shared agents or events.

**Integration Notes:**  
`fleet‑simulator` is a standalone Rust binary that produces training tiles. These tiles are not automatically ingested by any other component. The output format (`TrainingTile`) is documented but not consumed by a live pipeline.

## 3. holodeck‑rust (MUD Server)
**Location:** `holodeck‑rust/`  
**Purpose:** Multi‑user dungeon with sentiment‑aware NPCs.  
**Key Files:**
- `src/server/mud_server.rs` — WebSocket MUD server
- `src/npc/sentiment_engine.rs` — NPC sentiment system
- `src/world/rooms.rs` — Room definitions

**Connections Found:**
- **To plato‑torch:** None. No Python interop or training hooks.
- **To fleet‑simulator:** None. No simulation‑agent integration.
- **To plato‑ensign:** None. No adapter export.

**Integration Notes:**  
`holodeck‑rust` is a pure Rust async server. It maintains its own world state and NPC logic. There is no built‑in mechanism to feed NPC interactions into training rooms or to use trained adapters for NPC behavior.

## 4. plato‑ensign (LoRA Exporter)
**Location:** `plato‑ensign/`  
**Purpose:** Export room experience into LoRA adapters.  
**Key Files:**
- `lora_exporter.py` — Exporter class
- `config/export_config.yaml` — Export settings
- `utils/merge_adapters.py` — Adapter merging

**Connections Found:**
- **From plato‑torch:** Used as a library (imported by `plato‑torch`).  
- **To fleet‑simulator:** None. Does not read simulation tiles.
- **To holodeck‑rust:** None. No Rust bindings.

**Integration Notes:**  
`plato‑ensign` is a Python library designed to be called from `plato‑torch` rooms. It does not autonomously collect data from other sources.

## 6. Integration Gaps (What’s Not Connected)

1. **fleet‑simulator → plato‑torch**  
   - Simulated tiles are not automatically fed into training rooms.
   - Potential integration: a tile‑ingestion script in `plato‑torch` that loads `TrainingTile` JSON.

2. **holodeck‑rust → plato‑torch**  
   - NPC interactions are not captured as training tiles.
   - Potential integration: a WebSocket listener in `plato‑torch` that subscribes to holodeck events.

3. **plato‑ensign → holodeck‑rust**  
   - Trained adapters are not deployed to NPCs.
   - Potential integration: embed LoRA weights into NPC sentiment models via a shared adapter cache.

4. **Cross‑language bridges**  
   - No Python‑Rust interop (e.g., PyO3, IPC) between `plato‑torch`/`plato‑ensign` and `fleet‑simulator`/`holodeck‑rust`.

## 7. Recommended Integration Points (P1 Safe Channels)

Based on file inspection, the following integration channels appear feasible:

- **Tile ingestion pipeline:** Extend `plato‑torch/rooms/base_room.py` with a `add_simulation_tiles()` method that reads `fleet‑simulator` output.
- **Holodeck event hook:** Add a `tile_logger` module to `holodeck‑rust` that emits JSON events to a local socket; `plato‑torch` can listen.
- **Adapter deployment:** Create a small Rust crate in `holodeck‑rust` that loads `.safetensors` files (exported by `plato‑ensign`) and applies them to NPC models.
- **Shared config schema:** Define a common `Tile` schema (Protobuf/JSON) that all four components can produce/consume.
