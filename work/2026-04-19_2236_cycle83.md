## 1. Verified Current Connections

Based on reading the actual files in fleet-knowledge/ and the repository READMEs:

### **plato-torch → plato-ensign** (Strong, Direct)
- **Connection:** plato-torch exports training room experience (interaction tiles) to plato-ensign.
- **Mechanism:** plato-ensign ingests tile data from plato-torch rooms and exports LoRA adapters.
- **Evidence:** plato-ensign README states: "Exports room experience into LoRA adapters." Architecture note: "Rooms train from accumulated interaction tiles."

### **plato-torch → fleet-simulator** (Indirect, Data Flow)
- **Connection:** fleet-simulator generates training data (simulated fleet interactions) that can be consumed by plato-torch rooms.
- **Mechanism:** fleet-simulator output likely feeds into plato-torch as training tiles or synthetic interaction data.
- **Evidence:** fleet-simulator purpose: "Fleet simulation for training data." plato-torch includes "self-training rooms."

### **holodeck-rust → plato-torch** (Potential, Not Explicit)
- **Connection:** holodeck-rust is a MUD server with sentiment-aware NPCs. Could generate rich interaction tiles for plato-torch.
- **Mechanism:** Unclear in current docs. No direct API or data pipe mentioned.
- **Evidence:** holodeck-rust is listed as a Key Technology, but integration point not specified.

### **plato-ensign → (Downstream Use)** (Export Path)
- **Connection:** plato-ensign creates LoRA adapters from room experience. These could be loaded by cudaclaw or other runtime.
- **Mechanism:** LoRA adapters are model weights; likely usable by DeepSeek-chat variants or other fine-tuned models.
- **Evidence:** plato-ensign's stated purpose.

## 2. Identified Integration Gaps (P1: Safe Channels to Explore)

### **Gap 1: holodeck-rust → plato-torch Data Pipeline**
- **Current State:** holodeck-rust runs independently. No documented tile export to plato-torch.
- **Opportunity:** Pipe NPC interactions (dialogue, sentiment events) as training tiles into a plato-torch room.
- **Safe Channel:** Could write a bridge service that formats holodeck logs into tile JSON and POSTs to plato-torch room API.

### **Gap 2: fleet-simulator → holodeck-rust Scenario Feed**
- **Current State:** fleet-simulator generates fleet interaction data. holodeck-rust has NPCs but no fleet scenario engine.
- **Opportunity:** Use fleet-simulator output to drive holodeck scenarios (e.g., simulate a fleet mission inside the MUD).
- **Safe Channel:** Map fleet-simulator event types to holodeck room/quest triggers.

### **Gap 3: plato-ensign → cudaclaw / flux-runtime Adapter Loading**
- **Current State:** plato-ensign exports LoRAs. cudaclaw is a GPU-resident agent runtime. No documented loader.
- **Opportunity:** Enable cudaclaw agents to hot-swap LoRA adapters trained from specific room experiences.
- **Safe Channel:** Define a standard adapter manifest and loader hook in cudaclaw runtime.

### **Gap 4: DeadbandRoom (plato-torch) → plato-relay**
- **Current State:** DeadbandRoom is a training room preset. plato-relay (mentioned in backlog) likely handles tile routing.
- **Opportunity:** Connect DeadbandRoom to a relay for multi-room tile distribution or mirror play.
- **Safe Channel:** Check plato-torch for existing relay integration points; if none, propose a simple HTTP relay sink.
