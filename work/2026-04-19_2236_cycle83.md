## 1. Verified Current Connections (What's Connected)

Based on reading the actual fleet repos (`plato-torch`, `fleet-simulator`, `holodeck-rust`, `plato-ensign`):

### **plato-torch ↔ fleet-simulator**
- **Connection:** Strong, direct integration via `DeadbandRoom` preset.
- **Evidence:** `plato-torch/rooms/deadband_room.py` imports `fleet_simulator` and uses `FleetSimulator` class to generate training data.
- **Flow:** `DeadbandRoom` runs fleet simulations, captures agent interactions, and logs them as training tiles.
- **Data Path:** Simulation → Tile Log → Room training loop.

### **plato-torch ↔ plato-ensign**
- **Connection:** Direct export pipeline.
- **Evidence:** `plato-ensign/export.py` reads tile logs from `plato-torch/rooms/tile_logs/` and converts them into LoRA adapters.
- **Flow:** Room accumulates tiles → Ensign exports tiles → LoRA trained.
- **Note:** Export is optional; rooms can train without exporting.

### **fleet-simulator ↔ (indirect) holodeck-rust**
- **Connection:** Conceptual, not yet wired.
- **Evidence:** `fleet-simulator` simulates multi-agent fleets; `holodeck-rust` is a sentiment-aware MUD server. Both deal with agent interactions, but no shared code or data flow found.
- **Potential:** Fleet simulations could drive holodeck NPCs, or holodeck logs could become simulation inputs.

### **plato-ensign ↔ (none)**
- **Connection:** Standalone export tool.
- **Evidence:** `plato-ensign` only reads from `plato-torch` tile logs and writes LoRA adapters. No other integrations found.

### **holodeck-rust ↔ (none)**
- **Connection:** Isolated.
- **Evidence:** `holodeck-rust` is a standalone MUD server with sentiment-aware NPCs. No imports or references to other fleet repos in its `Cargo.toml` or source.

## 2. Integration Gaps (What's Not Connected)

### **Gap 1: holodeck-rust → plato-torch**
- **Issue:** Holodeck interactions are not captured as training tiles.
- **Impact:** Sentiment-aware NPC dialogues are lost for room training.
- **Action:** Wire `holodeck-rust` event logs into `plato-torch` tile ingestion.

### **Gap 2: fleet-simulator → holodeck-rust**
- **Issue:** No shared agent behavior models.
- **Impact:** Simulations can't test holodeck NPC logic, holodeck can't use simulated fleet data.
- **Action:** Define a common agent interaction protocol (e.g., JSON schema) both can use.

### **Gap 3: plato-ensign → fleet-simulator**
- **Issue:** Ensigns (LoRA adapters) are not validated in simulation.
- **Impact:** Can't test if a trained ensign improves fleet performance.
- **Action:** Add `fleet-simulator` integration to benchmark ensigns.

### **Gap 4: DeadbandRoom → plato-relay**
- **Issue:** `DeadbandRoom` operates locally; no live tile streaming to relay.
- **Impact:** Tiles are batch-processed, not real-time.
- **Action:** Integrate `plato-relay` client into `DeadbandRoom` for live tile submission.

### **Gap 5: GhostInjector → holodeck-rust**
- **Issue:** Ghost tiles (P0 knowledge from dead agents) are not injected into holodeck NPC memory.
- **Impact:** Holodeck NPCs can't learn from past agent failures.
- **Action:** Build `GhostInjector` module that reads ghost tiles and updates holodeck NPC sentiment/knowledge base.
