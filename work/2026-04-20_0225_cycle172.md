# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 172  
**Phase:** 4 — Build & Test  
**Status:** Mapping current connections and identifying integration points.

Based on reading the actual files in the fleet repos, here is the current state of integration between plato-torch, fleet-simulator, holodeck-rust, and plato-ensign.

## 1. plato-torch (Training Rooms)
- **Purpose**: Provides 25 preset training rooms and self-training rooms for agent interaction.
- **Key Files**: `DeadbandRoom.py`, `plato_relay.py`, `GhostInjector.py`
- **Current Connections**:
  - `plato_relay.py` → Sends/receives tiles to/from a PLATO room server (HTTP POST/GET).
  - `DeadbandRoom.py` → Implements the Deadband Protocol (P0/P1/P2) as a training environment.
  - `GhostInjector.py` → Can inject "ghost tiles" (P0 lessons from dead agents) into room streams.
- **Integration Status**: 
  - Internal components are wired (GhostInjector → DeadbandRoom → plato_relay).
  - No direct code dependency on fleet-simulator or holodeck-rust.
  - Plato-ensign is referenced as an optional export target.

## 2. fleet-simulator (Fleet Simulation)
- **Purpose**: Generates training data by simulating fleet interactions.
- **Key Files**: `simulator/core.py`, `scenarios/deadband_scenario.py`
- **Current Connections**:
  - `deadband_scenario.py` imports `DeadbandProtocol` from `plato_torch.protocols`.
  - Simulator outputs JSON logs that could be consumed as training tiles.
- **Integration Status**:
  - One-way dependency: fleet-simulator → plato-torch (imports protocol).
  - No automatic tile submission to plato-relay.
  - No connection to holodeck-rust or plato-ensign in code.

## 3. holodeck-rust (Sentiment-Aware MUD)
- **Purpose**: Multi‑User Dungeon server with NPCs that react to player sentiment.
- **Key Files**: `src/npc/sentiment.rs`, `src/protocol/mod.rs`
- **Current Connections**:
  - Standalone TCP server on port 8080.
  - No imports from other fleet repos.
  - Protocol is custom binary (not HTTP/JSON).
- **Integration Status**:
  - Isolated. No wiring to plato-torch, fleet-simulator, or plato-ensign.
  - Could be a tile source (player/NPC interactions) but currently isn’t.

## 4. plato-ensign (LoRA Adapter Export)
- **Purpose**: Exports room experience into LoRA adapters for specialist models.
- **Key Files**: `export.py`, `train_lora.py`
- **Current Connections**:
  - Reads `room_logs/` directory (expected format from plato-relay).
  - Can load a base model (e.g., `DeepSeek-Coder`) and fine‑tune with accumulated tiles.
- **Integration Status**:
  - Downstream consumer of plato-torch output (room logs).
  - No dependencies on fleet-simulator or holodeck-rust.
  - Export is manual/trigger‑based, not automatic.

## Integration Gaps (What’s Not Connected)

| Gap | Description | Priority |
|-----|-------------|----------|
| **holodeck‑rust → plato‑torch** | NPC/player interactions not captured as training tiles. | P1 |
| **fleet‑simulator → plato‑relay** | Simulation logs not automatically submitted to a room. | P1 |
| **plato‑ensign auto‑trigger** | No pipeline that exports when room reaches certain tile count. | P2 |
| **GhostInjector → holodeck** | Ghost tiles not injected into MUD NPC behavior. | P1 |
| **DeadbandRoom → plato‑relay (tested)** | Connection exists but needs end‑to‑end validation. | P2 |
