# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 169  
**Phase:** 4 — Build & Test  
**Status:** Mapping current connections and identifying integration points.

Based on reading the actual files in the fleet repos, here is the current state of integration between **plato-torch**, **fleet-simulator**, **holodeck-rust**, and **plato-ensign**.

---

## 1. **plato-torch** → **plato-ensign** → **fleet-simulator** (Connected)
- **Connection:** Direct pipeline exists.
- **Evidence:**  
  - `plato-torch/rooms/deadband_room.py` exports training tiles via `export_training_tiles()`.
  - `plato-ensign/export_lora.py` imports `DeadbandRoom` from plato-torch, loads tiles, and trains a LoRA adapter.
  - `fleet-simulator/training/` contains scripts that load LoRA adapters from plato-ensign for simulation fine‑tuning.
- **Data flow:**  
  `Room interactions → Tiles → LoRA adapter → Simulation agent`
- **Integration status:** **Live**. This is the primary training pipeline.

---

## 2. **holodeck-rust** → **plato-torch** (Not Connected)
- **Gap:** No direct integration found.
- **Evidence:**  
  - `holodeck-rust/src/` implements a MUD server with sentiment‑aware NPCs, but there is no import of plato‑torch modules.
  - `plato-torch/` does not reference holodeck‑rust or its output format.
- **Potential channel:**  
  Holodeck could emit interaction logs in tile format (JSON lines) that a plato‑torch room could consume. No such adapter exists yet.

---

## 3. **fleet-simulator** → **holodeck-rust** (Not Connected)
- **Gap:** No evidence of connection.
- **Evidence:**  
  - `fleet-simulator/` focuses on multi‑agent simulation with OpenAI/Claude agents.
  - `holodeck-rust/` is a standalone MUD server; its NPCs are not driven by fleet‑simulator agents.
- **Potential channel:**  
  Fleet‑simulator agents could connect to holodeck as clients, or holodeck NPCs could be wrapped as simulator participants.

---

## 4. **plato-ensign** → **holodeck-rust** (Not Connected)
- **Gap:** No integration.
- **Evidence:**  
  - `plato-ensign` exports LoRA adapters for language models.
  - `holodeck-rust` uses its own NPC logic (sentiment state machines) and does not load external adapters.
- **Potential channel:**  
  A holodeck NPC could be backed by a finetuned model from plato‑ensign. Would require a Rust‑Python bridge and adapter loading.

---

## 5. **plato-torch** ↔ **fleet-simulator** (Indirectly Connected)
- **Connection:** Via plato‑ensign (see #1).
- **Evidence:**  
  - `fleet-simulator/training/train_lora.py` imports `peft` and expects a LoRA from plato‑ensign.
  - No direct call from plato‑torch to fleet‑simulator.
- **Integration status:** **One‑way, through adapter**.

---

## 6. **Ghost Tiles & Mirror Play** (Architectural, Not Implemented)
- **Ghost tiles** are described as dead agents’ lessons persisting as P0 knowledge. No file found that implements ghost‑tile injection.
- **Mirror Play** (two PLATO systems iterating screen‑to‑screen) is a concept in `FLEET‑CONTEXT.md` but not yet coded.
