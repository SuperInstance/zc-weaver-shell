# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 382  
**Phase:** 4 — Build & Test  
**Status:** Direct file inspection of fleet repositories. Focus on actual imports, configuration references, and shared data structures.

---

## 1. plato-torch → fleet-simulator
**Connection:** Strong, direct dependency.  
**Evidence:**  
- `plato-torch/requirements.txt` includes `fleet-simulator>=0.3.0`.  
- `plato-torch/src/training/rooms/deadband_room.py` imports `from fleet_simulator import Agent, Environment`.  
- `plato-torch/configs/deadband.yaml` references `simulator_config_path: ../fleet-simulator/configs/basic.yaml`.  
**Data Flow:**  
- DeadbandRoom instantiates a fleet-simulator Environment to generate training episodes.  
- Simulator agents produce interaction logs that become training tiles.  
**Integration Gap:**  
- No bidirectional control flow observed. Plato-torch drives the simulator, but the simulator does not callback into plato-torch.

---

## 2. plato-torch → plato-ensign
**Connection:** Optional export pipeline.  
**Evidence:**  
- `plato-torch/src/export/ensign_exporter.py` exists.  
- It imports `from plato_ensign.core import LoRAAdapter, export_room`.  
- Exporter is not called by default in DeadbandRoom; must be triggered via CLI or room config flag `export_ensign: true`.  
**Data Flow:**  
- After room training, ensign_exporter can package the room’s accumulated tiles and model checkpoints into a LoRA adapter via plato-ensign.  
**Integration Gap:**  
- Integration is one‑way and manual. No automatic triggering based on training metrics.

---

## 3. fleet-simulator → holodeck-rust
**Connection:** None observed.  
**Evidence:**  
- `fleet-simulator/` has no Rust dependencies, no `holodeck` or `mud` references in code or configs.  
- Simulator agents are Python‑based; holodeck‑rust is a standalone MUD server.  
**Potential Integration Point:**  
- Simulator could spawn holodeck‑rust as a subprocess and connect via TCP to use its sentiment‑aware NPCs as part of environment. Not currently implemented.

---

## 4. holodeck-rust → plato-torch
**Connection:** None observed.  
**Evidence:**  
- `holodeck-rust/Cargo.toml` shows no Python or torch bindings.  
- No references to `plato`, `torch`, or `deadband` in source.  
**Potential Integration Point:**  
- Holodeck could emit interaction logs in the same tile format plato‑torch expects, allowing holodeck sessions to become training data. Not implemented.

---

## 5. plato-ensign → Others
**Connection:** Standalone tool.  
**Evidence:**  
- `plato-ensign/` has no imports of plato‑torch, fleet‑simulator, or holodeck‑rust.  
- It expects a pre‑packaged room bundle (tiles + model) as input.  
**Integration Gap:**  
- Only integrated via the exporter in plato‑torch. No direct hooks into simulator or holodeck.

---

## 6. Ghost Tiles & Deadband Protocol
**Connection:** Conceptual, not code‑level.  
**Evidence:**  
- `plato-torch/src/data/ghost_injector.py` reads from `memory/` directory (dead agents’ logs).  
- Injects them as negative‑example tiles (P0 knowledge) into training batches.  
- No fleet‑wide ghost‑tile service observed; each room handles injection locally.  
**Integration Gap:**  
- Ghost tiles are file‑based, not shared via a central service. Simulator and holodeck do not contribute ghost tiles.

```
plato-torch ────(drives)────> fleet-simulator
      │
      └───(optional export)──> plato-ensign
```
