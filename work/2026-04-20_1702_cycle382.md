# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 382

**Status:** Direct file inspection of fleet repositories. Focus on actual imports, configuration references, and shared data structures.

## 1. **plato-torch** → **plato-ensign**  
**Connection:** **Strong** — Direct integration via export pipeline.  
**Evidence:**  
- `plato-torch/src/rooms/deadband_room.py` imports `from plato_ensign import export_lora`.  
- `plato-torch/scripts/train_ensign.py` calls `export_lora(room_state, adapter_path)`.  
- Shared data structure: `RoomState` (defined in `plato-torch/src/core/room_state.py`) is passed to `plato-ensign` for LoRA extraction.  
**Integration Gap:** None observed. This is a working export path.

---

## 2. **plato-torch** → **fleet-simulator**  
**Connection:** **Weak** — Configuration‑only, no runtime coupling.  
**Evidence:**  
- `fleet-simulator/config/training_scenarios.yaml` references `plato-torch` room presets by name (e.g., `deadband_room`).  
- No import of `plato-torch` modules in `fleet-simulator` source code.  
- Simulator generates synthetic interaction logs that *could* be formatted as tiles, but no live tile‑stream observed.  
**Integration Gap:**  
- No direct API or data pipe between `plato-torch` rooms and `fleet-simulator` agents.  
- Simulator outputs are static `.jsonl` files; not consumed by `plato-torch` in real‑time.

---

## 3. **holodeck-rust** → **plato-torch**  
**Connection:** **None** — Separate runtimes, no shared code or data flow.  
**Evidence:**  
- `holodeck-rust/src/main.rs` implements a MUD server with sentiment‑aware NPCs.  
- `plato-torch` is Python‑based and focuses on training rooms.  
- No references to `holodeck` in `plato-torch` codebase.  
- No references to `plato-torch` in `holodeck-rust` codebase.  
**Integration Gap:**  
- Holodeck NPC interactions are not captured as training tiles.  
- No bridge between Rust MUD events and Python room state.

---

## 4. **fleet-simulator** → **holodeck-rust**  
**Connection:** **None** — Separate domains.  
**Evidence:**  
- `fleet-simulator` simulates fleet‑agent interactions.  
- `holodeck-rust` is a standalone multiplayer text‑adventure server.  
- No shared configuration or data structures.  
**Integration Gap:**  
- Simulated agent behaviors do not populate the holodeck world.  
- Holodeck sentiment data does not feed back into simulator.

---

## 5. **plato-ensign** → **fleet-simulator**  
**Connection:** **Indirect** — Ensign adapters *could* be loaded by simulated agents.  
**Evidence:**  
- `fleet-simulator/docs/architecture.md` mentions “optional LoRA loading for agent specialization.”  
- No actual code loading `.safetensors` adapters from `plato-ensign` output.  
**Integration Gap:**  
- Missing `load_adapter()` call in simulator agent initialization.  
- No shared adapter registry or versioning.

---

## 6. **Ghost Tiles & Deadband Protocol**  
**Connection:** **Conceptual** — Not yet wired.  
**Evidence:**  
- `plato-torch/src/core/ghost_injector.py` exists but is not imported anywhere.  
- `DeadbandRoom` preset implements P0/P1/P2 logic but does not call `GhostInjector`.  
- Ghost tiles are described in `FLEET-CONTEXT.md` as “dead agents' lessons.”  
**Integration Gap:**  
- `GhostInjector` is isolated; not connected to room training loops.  
- No persistence layer for ghost tiles across room restarts.

---

## 7. **cudaclaw & flux-runtime**  
**Connection:** **Independent** — Not integrated with the four core repos.  
**Evidence:**  
- `cudaclaw` is a CUDA‑based agent runtime.  
- `flux-runtime` is a bytecode VM.  
- No imports of these in `plato-torch`, `fleet-simulator`, `holodeck-rust`, or `plato-ensign`.  
**Integration Gap:**  
- GPU‑resident agents not used to generate training tiles.  
- Flux bytecode not executed within room contexts.

---

## Summary of Integration Gaps (P0/P1 Channels)  
**P0 (Negative Space — Avoid):**  
- Do not assume data flows between `holodeck-rust` and other systems.  
- Do not expect simulator agents to load ensign adapters without additional wiring.  

**P1 (Safe Channels — Possible Integration Points):**  
- `GhostInjector` → `DeadbandRoom` (inject ghost tiles as P0 examples).  
- `fleet-simulator` → `plato-torch` (stream synthetic interactions as training tiles).  
- `holodeck-rust` → `plato-torch` (capture NPC dialogues as sentiment‑labeled tiles).  
- `plato-ensign` → `fleet-simulator` (load adapters for specialized agent roles).  

**P2 (Optimization — Future):**  
- Build real‑time tile bridges between systems.  
- Create shared adapter registry for ensign‑trained models.
