## 1. Verified Connections (What's Connected)

### **plato-torch ↔ fleet-simulator**
- **Connection:** Direct import/usage.
- **Evidence:** `fleet-simulator/src/lib.rs` imports `plato_torch` and uses `DeadbandRoom` preset.
- **Purpose:** Fleet simulator uses plato-torch's training room presets to generate training data (tiles) from simulated agent interactions.
- **Data Flow:** Simulated agent actions → DeadbandRoom → training tiles.

### **plato-torch ↔ plato-ensign**
- **Connection:** Direct dependency.
- **Evidence:** `plato-ensign/Cargo.toml` lists `plato-torch` as a dependency.
- **Purpose:** plato-ensign exports room experience (accumulated tiles) from plato-torch rooms into LoRA adapters.
- **Data Flow:** plato-torch room tiles → plato-ensign export → LoRA adapter.

### **fleet-simulator ↔ holodeck-rust**
- **Connection:** Indirect via shared patterns.
- **Evidence:** Both use similar sentiment-aware NPC interaction patterns, but no direct import.
- **Purpose:** Fleet simulator may generate data compatible with holodeck's MUD server for sentiment training.
- **Data Flow:** Potential tile compatibility, but no explicit pipeline yet.

## 2. Integration Gaps (What's Not Connected)

### **holodeck-rust ↔ plato-torch**
- **Gap:** No direct import or usage.
- **Impact:** Holodeck's sentiment-aware NPC interactions are not feeding into plato-torch rooms for training.
- **Opportunity:** Connect holodeck NPC dialogue/state to DeadbandRoom to generate sentiment-rich tiles.

### **holodeck-rust ↔ plato-ensign**
- **Gap:** No connection.
- **Impact:** Holodeck's unique sentiment data is not being exported to LoRA adapters.
- **Opportunity:** Pipe holodeck interaction logs through plato-torch → plato-ensign for specialized sentiment LoRAs.

### **GhostInjector (plato-torch) ↔ holodeck-rust**
- **Gap:** GhostInjector (dead agent lessons) is not wired into holodeck.
- **Impact:** Holodeck NPCs lack P0 (negative space) knowledge from dead agents.
- **Opportunity:** Inject ghost tiles as NPC memory constraints in holodeck.

### **DeadbandRoom ↔ plato-relay**
- **Gap:** No live relay of room state to other fleet components.
- **Impact:** Training rooms operate in isolation; real-time monitoring/alerting limited.
- **Opportunity:** Use plato-relay to broadcast room health, tile counts, anomalies.

### **End-to-End Pipeline**
- **Gap:** No integrated flow from simulation → training → export → deployment.
- **Impact:** Manual steps between fleet-simulator, plato-torch, plato-ensign.
- **Opportunity:** Automate pipeline: simulate → train room → export ensign → deploy to cudaclaw/flux-runtime.
