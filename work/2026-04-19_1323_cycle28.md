## 1. Verified Connections (Based on fleet-knowledge/ files)

### plato-torch → plato-ensign
- **Connection:** Direct export pipeline.
- **Mechanism:** `plato-torch` training rooms accumulate interaction tiles. `plato-ensign` can export a room's experience into a LoRA adapter.
- **Evidence:** From FLEET-CONTEXT.md: "plato-ensign: Exports room experience into LoRA adapters" and "Rooms train from accumulated interaction tiles. Ensigns are optional exports of room experience."
- **Status:** Connected. This is a core, documented workflow.

### plato-torch → fleet-simulator
- **Connection:** Data source for training.
- **Mechanism:** `fleet-simulator` generates training data (likely simulation logs or agent interactions). This data can be formatted into "tiles" and ingested by a `plato-torch` room for training.
- **Evidence:** FLEET-CONTEXT.md lists both as key technologies, with `fleet-simulator` described as "Fleet simulation for training data."
- **Status:** Connected logically (data flow). Specific API or file format not yet verified in code.

### holodeck-rust → (Potential Data Consumer)
- **Connection:** Sentiment-aware NPC interactions could generate training tiles.
- **Mechanism:** NPC dialogues and player actions within the MUD could be logged and formatted as tiles for a `plato-torch` room.
- **Evidence:** `holodeck-rust` is listed as a key technology: "MUD server with sentiment-aware NPCs." Its output aligns with the fleet's tile-based training paradigm.
- **Status:** Potential connection. Not yet a verified, implemented pipeline.

### plato-ensign → (Downstream Use)
- **Connection:** LoRA adapters are products for deployment.
- **Mechanism:** A trained ensign (LoRA) could be loaded by a runtime (like `cudaclaw` or `flux-runtime`) to specialize an agent's behavior.
- **Evidence:** Ensigns are "specialist AI models" (FLEET-CONTEXT.md). `cudaclaw` is a "GPU-resident agent runtime."
- **Status:** Connected in concept. Integration path exists but may not be fully automated.

## 2. Integration Gaps & Missing Links

### Gap 1: holodeck-rust → plato-torch Tile Pipeline
- **Description:** No documented, automatic flow of NPC/player interactions from the MUD into a training room as formatted tiles.
- **Action:** Could build a `TileLogger` module for `holodeck-rust` that emits events to a `plato-torch` room API or writes to a shared tile directory.
- **P0/P1:** This is a P1 "find safe channel" task—establishing a viable data conduit.

### Gap 2: fleet-simulator → plato-torch Standardized Ingest
- **Description:** While a logical connection exists, the specific format (JSON schema, protobuf?) and ingestion method (REST, file watch, direct call) are not mapped.
- **Action:** Audit `fleet-simulator` output format and `plato-torch`'s `DeadbandRoom` or similar room preset's `add_tile` method to define a contract.
- **P0/P1:** P1. Defining the contract is a safe channel.

### Gap 3: plato-ensign → cudaclaw / flux-runtime Deployment
- **Description:** How does a trained `.safetensors` LoRA file get loaded and used by a live agent runtime? Is there a registry, a versioning system, or a hot-swap mechanism?
- **Action:** Examine `cudaclaw` and `flux-runtime` documentation/code for model adapter loading hooks.
- **P0/P1:** P1. Understanding the deployment channel.

### Gap 4: Constraint-theory-core → holodeck-rust / fleet-simulator
- **Description:** `constraint-theory-core` (geometric snapping) is listed but its integration points are unclear. Could it govern NPC movement or spatial reasoning in simulations?
- **Action:** Identify if any fleet component currently handles spatial constraints; propose integration for more realistic navigation or object interaction.
- **P0/P1:** P1. Finding a viable application channel.

### Gap 5: Ghost Tiles → Active Training Loop
- **Description:** "Ghost tiles: dead agents' lessons persist as P0 knowledge." How are these tiles stored, retrieved, and injected into new training cycles? Is there a `GhostInjector`?
- **Action:** Look for a ghost tile archive or a service that surfaces P0 (negative space) examples to new rooms.
- **P0/P1:** P0/P1 boundary. Mapping the negative space (P0) of *how not to lose ghost tiles* is critical before building (P2).
