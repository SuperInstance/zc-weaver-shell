# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 306  
**Phase:** 4 — Build & Test  
**Status:** Direct file inspection of fleet repositories. Focus on actual imports, configuration references, and shared data structures.

## 1. plato-torch → fleet-simulator
**Connection:** **Yes** — Direct import and data flow.  
**Evidence:**  
- `plato-torch/src/rooms/deadband_room.py` imports `from fleet_simulator import FleetSimulator`.  
- `DeadbandRoom` initializes a `FleetSimulator` instance (`self.simulator = FleetSimulator()`).  
- Room calls `self.simulator.step()` during its own `step()` method.  
- Simulator state (agent positions, velocities) is used to generate training observations.  

**Integration Gap:**  
- Simulator outputs raw numeric arrays; room must convert to tensor format for training.  
- No shared configuration file for simulator parameters (hardcoded in room).  
- Simulator does not yet accept external commands from room (one‑way flow).

## 2. plato-torch → holodeck-rust
**Connection:** **No** — No direct import or API call found.  
**Evidence:**  
- Searched `plato-torch/src/` for “holodeck”, “mud”, “rust”, “sentiment”. No matches.  
- `holodeck-rust/` is a separate Rust project; no Python bindings appear in plato‑torch.  

**Integration Gap:**  
- Plato rooms cannot currently invoke holodeck NPC interactions.  
- No shared session‑ID or narrative context between training rooms and holodeck.  
- Potential integration point: use holodeck as a “sentiment‑aware environment” for role‑play training.

## 3. plato-torch → plato-ensign
**Connection:** **Yes** — Export pipeline.  
**Evidence:**  
- `plato-torch/src/export/ensign_exporter.py` defines `EnsignExporter`.  
- Exporter reads room interaction tiles (`.pt` files) and converts them to LoRA‑compatible format.  
- `plato-ensign/` provides a CLI tool (`ensign‑train`) that loads these exported tiles.  

**Integration Gap:**  
- Export is manual/offline; no automatic trigger when room reaches sufficient tile count.  
- No validation that exported tiles are compatible with target model architecture.  
- Ensign training does not yet receive room metadata (e.g., which deadband protocol phase was active).

## 4. fleet-simulator → holodeck-rust
**Connection:** **No** — No reference in either codebase.  
**Evidence:**  
- `fleet-simulator/` is pure Python; `holodeck-rust/` is pure Rust.  
- No inter‑process communication (IPC) or shared memory setup.  

**Integration Gap:**  
- Simulator agents cannot “enter” holodeck as avatars.  
- Holodeck NPCs cannot influence simulator state.  
- Potential: use holodeck as a visualization front‑end for simulator events.

## 5. fleet-simulator → plato-ensign
**Connection:** **Indirect** — via plato‑torch.  
**Evidence:**  
- Simulator data flows into plato‑torch rooms.  
- Rooms may export that data as tiles for ensign training.  
- No direct simulator‑to‑ensign pipeline.  

**Integration Gap:**  
- Simulator‑specific behaviors (collision avoidance, flocking) are not tagged in tile metadata.  
- Ensign cannot be trained solely on simulator data without room‑based curation.

## 6. holodeck-rust → plato-ensign
**Connection:** **No** — No export path found.  
**Evidence:**  
- `holodeck-rust/src/` contains game‑logic, networking, NPC AI. No export functions.  
- No `.pt` or LoRA‑adapter generation in holodeck.  

**Integration Gap:**  
- NPC dialogue and sentiment patterns are not captured as training tiles.  
- Missed opportunity: “holodeck sessions” could become role‑play tiles for ensigns.

## 7. GhostInjector (plato-torch) → holodeck-rust
**Connection:** **No** — Not yet wired.  
**Evidence:**  
- `plato-torch/src/injectors/ghost_injector.py` exists; reads “ghost tiles” (P0 lessons from dead agents).  
- Ghost tiles are currently injected only into plato‑torch rooms.  
- No reference to holodeck in injector code.  

**Integration Gap:**  
- Ghost tiles could be used to pre‑load holodeck NPCs with cautionary dialogue.  
- No protocol for sending ghost JSON to holodeck’s MUD server.

## 8. DeadbandRoom → plato-relay
**Connection:** **No** — plato‑relay not found in codebase.  
**Evidence:**  
- Searched for “relay” in `plato-torch/` — no matches.  
- `plato-relay` may be a planned service not yet implemented.  

**Integration Gap:**  
- Rooms cannot broadcast their state to a relay for multi‑room coordination.  
- No federation of deadband protocols across rooms.
