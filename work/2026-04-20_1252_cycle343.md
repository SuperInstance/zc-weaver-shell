## 1. plato-torch → fleet-simulator
**Connection:** **Strong** (direct import, shared data structures)  
- `plato-torch/src/rooms/deadband_room.py` imports `from fleet_simulator import FleetSimulator`  
- `DeadbandRoom` initializes a `FleetSimulator` instance in its constructor  
- Simulator provides `step()` method that returns `(observation, reward, done, info)`  
- Room passes simulator observations through `_format_observation()`  
- **Data flow:** FleetSimulator → DeadbandRoom → training loop  
- **Integration point:** `DeadbandRoom` acts as RL environment wrapper for FleetSimulator

**Gap:** No bidirectional control flow observed. Simulator doesn't call back into plato-torch.

---

## 2. plato-torch → plato-ensign  
**Connection:** **Strong** (direct export pipeline)  
- `plato-torch/src/export/ensign_exporter.py` contains `EnsignExporter` class  
- Exporter reads room training history (tiles) from `room.history`  
- Converts interaction tiles into LoRA training format  
- Uses `peft` library to create and save adapters  
- **Data flow:** Room.history → EnsignExporter → LoRA adapter files  
- **Integration point:** Optional export triggered after room training

**Gap:** No observed import of plato-ensign into plato-torch (ensign appears to be downstream consumer).

---

## 3. fleet-simulator → holodeck-rust
**Connection:** **None observed**  
- No Rust imports in Python fleet-simulator code  
- No Python bindings found in holodeck-rust  
- Separate technology stacks: Python vs Rust  
- **Potential integration:** Could connect via REST API or shared data files, but no current implementation found

## 6. Ghost Tiles Integration
**Connection:** **Partial**  
- `plato-torch/src/data/ghost_injector.py` exists with `GhostInjector` class  
- Can load "ghost tiles" (dead agents' lessons) from JSON files  
- Injects them as P0 negative examples into training  
- **Integration gap:** Not wired into holodeck or fleet-simulator observed  
- Ghost tiles currently file-based, not runtime-integrated
