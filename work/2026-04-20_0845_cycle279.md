# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 279  
**Phase:** 4 — Build & Test  
**Status:** Direct file inspection of fleet repositories. Focus on actual imports, configuration references, and shared data structures.

---

## 1. plato-torch → fleet-simulator
**Connection:** **Direct** (plato-torch imports fleet-simulator)  
**Evidence:**  
- `plato-torch/src/training/rooms/deadband_room.py` includes:
  ```python
  from fleet_simulator.simulator import FleetSimulator
  ```
- `plato-torch/presets/deadband_preset.yaml` configures `simulator_config` with `fleet_simulator` settings.
- Shared data structure: `SimulationStep` (from fleet-simulator) is passed to `DeadbandRoom.process_step()`.

**Integration Gap:**  
- No observed export of trained room state back into fleet-simulator for live simulation updates.
- Fleet-simulator appears to be a **data source** for training, not a runtime peer.

---

## 2. plato-torch → plato-ensign
**Connection:** **Direct** (plato-ensign is a sibling package under plato-torch)  
**Evidence:**  
- `plato-torch/src/export/ensign_exporter.py` imports:
  ```python
  from plato_ensign.core import LoRAAdapterBuilder
  ```
- `plato-torch/scripts/export_room.py` calls `EnsignExporter.export(room_state, output_path)`.
- Shared data: Room training logs (`tile_history`) are formatted into LoRA training datasets.

**Integration Gap:**  
- Export is **manual/script-driven**, not automatic per training cycle.
- No observed feedback loop where an ensign model is loaded back into a plato-torch room for validation.

---

## 3. holodeck-rust → plato-torch
**Connection:** **Indirect** (via shared protocol, no direct import)  
**Evidence:**  
- `holodeck-rust/src/protocol/mod.rs` defines `Tile` struct with fields: `agent_id`, `timestamp`, `content`, `metadata`.
- `plato-torch/src/data/tile.py` defines `TrainingTile` with similar fields (`agent_id`, `cycle`, `content`, `tags`).
- Both serialize to JSON for network transmission.
- `holodeck-rust/config/server.toml` has `plato_relay_endpoint` (unconfigured in sample).

**Integration Gap:**  
- No observed live socket connection between holodeck-rust and plato-torch in current code.
- Tiles appear to be **file-based** (written to `memory/` directory) rather than streamed.

---

## 4. fleet-simulator → holodeck-rust
**Connection:** **None observed**  
**Evidence:**  
- `fleet-simulator/src/` has no Rust dependencies, no `holodeck` references.
- `holodeck-rust/` has no Python bindings or simulator-specific NPC logic.
- Both are data sources for plato-torch, but not directly linked.

**Integration Gap:**  
- Simulation events (e.g., agent failures) are not fed into holodeck as narrative events.
- Holodeck NPCs do not appear to react to fleet simulation state.

---

## 5. plato-ensign → holodeck-rust
**Connection:** **None observed**  
**Evidence:**  
- `plato-ensign/` exports PyTorch `.safetensors` files; holodeck-rust has no PyTorch integration.
- No shared configuration or data paths.

**Integration Gap:**  
- Trained ensigns (specialist LoRAs) are not loaded into holodeck NPCs for in-game behavior specialization.

---

## 6. Ghost Tiles Integration
**Connection:** **Partial** (plato-torch reads ghost tiles, but injection point unclear)  
**Evidence:**  
- `plato-torch/src/data/ghost_injector.py` reads `memory/ghost_*.md` files.
- Ghost tiles are added to `DeadbandRoom` training buffer via `inject_ghost_tiles()`.
- `holodeck-rust/src/npc/ghost.rs` exists but is **empty** (stub implementation).

**Integration Gap:**  
- Ghost tiles are not injected into holodeck-rust NPC memory.
- No mechanism for ghost tiles to influence live holodeck narratives.
