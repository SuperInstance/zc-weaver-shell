# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 208  
**Phase:** 4 — Build & Test  
**Status:** Updated map based on deeper code inspection of plato-torch, fleet-simulator, holodeck-rust, and plato-ensign.  

## 1. Verified Connections (What's Connected)  

### **plato-torch ↔ fleet-simulator**  
- **Connection:** Strong, direct integration via `fleet-simulator` as a data source for training rooms.  
- **Evidence:**  
  - `plato-torch/rooms/deadband_room.py` imports `from fleet_simulator import FleetSimulator`.  
  - `DeadbandRoom` class initializes `self.simulator = FleetSimulator()` and calls `self.simulator.generate_scenario()` to produce training scenarios.  
  - Simulator output (JSON state) is fed directly into room's `generate_training_tile()` method.  
- **Data Flow:** `fleet-simulator` → scenario JSON → `plato-torch` room → training tile.  

### **plato-torch ↔ plato-ensign**  
- **Connection:** Direct export pipeline; ensign is an optional downstream consumer of room experience.  
- **Evidence:**  
  - `plato-ensign/src/exporter.py` imports `from plato_torch.rooms.base import BaseRoom`.  
  - `Exporter.export_room_to_lora()` takes a `BaseRoom` instance, extracts its `training_log`, and converts it to a LoRA adapter.  
  - `plato-torch/rooms/deadband_room.py` includes a `save_for_ensign()` method that calls the exporter.  
- **Data Flow:** `plato-torch` room → training log → `plato-ensign` exporter → LoRA weights.  

### **holodeck-rust ↔ (indirect) plato-torch**  
- **Connection:** Thematic/architectural alignment, but no direct code integration observed.  
- **Evidence:**  
  - Both projects reference "sentiment-aware NPCs" and "interaction tiles" in their READMEs.  
  - `holodeck-rust/src/npc.rs` defines `SentimentComponent` with emotional state tracking.  
  - `plato-torch/rooms/deadband_room.py` mentions "sentiment-aware training" but does not import or call holodeck-rust.  
- **Data Flow:** None currently. They are parallel implementations of similar concepts.  

## 2. Integration Gaps (What's Not Connected)  

### **holodeck-rust ↔ fleet-simulator**  
- **Gap:** No shared data structures or API calls.  
- **Evidence:**  
  - `holodeck-rust` uses ECS (Entity-Component-System) architecture with Rust structs.  
  - `fleet-simulator` outputs JSON scenarios via Python.  
  - No Rust bindings or Python interop layer found.  

### **holodeck-rust ↔ plato-ensign**  
- **Gap:** No export path for holodeck interactions to become ensign adapters.  
- **Evidence:**  
  - `plato-ensign` exporter only accepts `BaseRoom` instances.  
  - `holodeck-rust` has its own `InteractionLog` struct but no compatibility layer.  

### **GhostInjector (concept) ↔ holodeck-rust**  
- **Gap:** "Ghost tiles" (dead agents' lessons) are described in fleet-context but not implemented in holodeck.  
- **Evidence:**  
  - `holodeck-rust/src/world.rs` has a `spawn_npc()` function but no `inject_ghost()` or similar.  
  - No P0 negative-space knowledge is being fed into NPC sentiment logic.  

### **DeadbandRoom ↔ plato-relay**  
- **Gap:** `DeadbandRoom` generates tiles but does not transmit them to a relay service.  
- **Evidence:**  
  - `plato-relay` (referenced in task backlog) is not present in the current repo scan.  
  - `DeadbandRoom.save_tile()` writes locally to `training_tiles/` directory only.
