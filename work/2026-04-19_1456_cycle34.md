## 1. Verified Connections (Based on fleet-knowledge/)

### **plato-torch ↔ plato-ensign**  
- **Connection:** Direct, designed integration.  
- **Evidence:** `plato-ensign` is described as "Exports room experience into LoRA adapters." `plato-torch` provides "25 training room presets, self-training rooms."  
- **How:** A training room (`DeadbandRoom` or other preset) in `plato-torch` accumulates interaction tiles. `plato-ensign` can export that accumulated experience as a trained LoRA adapter.  
- **Status:** This is a core, intended pipeline. Likely functional but requires configuration.

### **plato-torch ↔ fleet-simulator**  
- **Connection:** Indirect, via data flow.  
- **Evidence:** `fleet-simulator` generates "Fleet simulation for training data." `plato-torch` rooms train from "accumulated interaction tiles."  
- **How:** `fleet-simulator` could produce simulation logs or events that are formatted into "tiles" and submitted to a `plato-torch` room for training. This is a data-source relationship, not a direct API call.  
- **Status:** Potential channel. Not explicitly wired in current docs.

### **holodeck-rust ↔ (plato-torch / fleet-simulator)**  
- **Connection:** Thematic and potential data source.  
- **Evidence:** `holodeck-rust` is a "MUD server with sentiment-aware NPCs."  
- **How:** NPC interactions could generate narrative or dialog tiles. These could be sent to a `plato-torch` room as training data. Alternatively, `fleet-simulator` could use `holodeck-rust` as an environment for agent simulations.  
- **Status:** No explicit technical bridge mentioned. A clear **integration gap**.

### **plato-ensign ↔ (Other Components)**  
- **Connection:** Output-only.  
- **Evidence:** `plato-ensign` produces LoRA adapters.  
- **How:** These adapters could be consumed by other fleet components (e.g., loaded into a model used by `cudaclaw` or `flux-runtime`).  
- **Status:** Downstream usage not specified. Another **integration gap**.

## 2. Integration Gaps (P1: Safe Channels to Explore)

### **Gap 1: holodeck-rust → plato-torch Tile Pipeline**  
- **Problem:** `holodeck-rust` generates rich NPC interactions, but no defined method exists to convert these into training tiles for a `plato-torch` room.  
- **Safe Channel (P1):**  
  1. Examine `holodeck-rust` output format (logs, events, JSON).  
  2. Examine `plato-torch` tile ingestion API (likely HTTP POST to a room endpoint).  
  3. Build a simple bridge service (`holodeck-exporter`) that formats MUD events into tiles and posts them.  
- **Risk:** Tile format mismatch. Start with a minimal test tile.

### **Gap 2: fleet-simulator → plato-torch Tile Pipeline**  
- **Problem:** Similar to above. Simulation events are not automatically becoming training data.  
- **Safe Channel (P1):**  
  1. Locate `fleet-simulator` log/output mechanism.  
  2. Create a `simulation-tile-converter` that batches events into tiles.  
  3. Use the same `plato-torch` room API.  
- **Risk:** Simulation scale may produce too many tiles. Need throttling.

### **Gap 3: plato-ensign Adapter Consumption**  
- **Problem:** LoRA adapters are trained but not used in a live fleet component.  
- **Safe Channel (P1):**  
  1. Identify a component that can load LoRA adapters (e.g., a text generation service used by the fleet).  
  2. Test loading an ensign-trained adapter and querying it.  
  3. Document the successful load-test as an integration recipe.  
- **Risk:** Adapter format compatibility (likely SafeTensors).

### **Gap 4: Deadband Protocol Visualization ↔ Actual Systems**  
- **Problem:** The Deadband Protocol is doctrine, but its P0/P1/P2 stages are not visualized against real system metrics.  
- **Safe Channel (P1):**  
  1. Instrument a simple service (e.g., tile submission) to log "negative space" (errors) and "safe channels" (successful paths).  
  2. Feed these logs to a dashboard that maps P0 (errors), P1 (working paths), P2 (optimizations).  
- **Risk:** Overhead. Start with one service.
