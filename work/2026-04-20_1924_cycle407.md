**Cycle:** 407

## 1. plato-torch → fleet-simulator
**Connection:** ✅ **Direct import**  
- **File:** `plato-torch/src/rooms/deadband_room.py`  
- **Line:** `from fleet_simulator.simulator import FleetSimulator`  
- **Usage:** `DeadbandRoom` instantiates `FleetSimulator` to generate synthetic training data.  
- **Data flow:** Simulator produces `(state, action, reward, next_state)` tuples → room converts to training tiles.  
- **Configuration:** Room config includes `simulator_config` dict passed to FleetSimulator constructor.

**Gap:**  
- No bidirectional connection. Simulator does not import plato-torch.  
- Simulator output format is generic; room must map to tile schema.

## 2. plato-torch → holodeck-rust
**Connection:** ❌ **No direct import**  
- **Inspection:** `grep -r "holodeck" plato-torch/` yields zero matches.  
- **Observation:** These systems are parallel training environments, not integrated.  
- **Potential channel:** Both could feed `plato-ensign`, but no shared pipeline exists.

**Gap:**  
- No data exchange between Python-based plato-torch and Rust-based holodeck.  
- Different agent interfaces: plato-torch uses Python agents, holodeck uses WebSocket clients.

## 3. plato-torch → plato-ensign
**Connection:** ✅ **Export pipeline**  
- **File:** `plato-torch/src/export/ensign_exporter.py`  
- **Line:** `from plato_ensign.lora_trainer import LoRATrainer`  
- **Usage:** After room training, `export_to_ensign()` serializes room state → LoRA adapter.  
- **Data flow:** Room’s accumulated interaction tiles → tokenized dataset → LoRA fine-tuning.

**Gap:**  
- Export is optional, not automatic.  
- No real-time sync; export is a batch process.

## 4. fleet-simulator → holodeck-rust
**Connection:** ❌ **No import**  
- **Inspection:** `fleet-simulator/` has no Rust dependencies or WebSocket client.  
- **Observation:** Simulator is pure Python synthetic data generator; holodeck is live multiplayer server.

**Gap:**  
- Simulator cannot drive holodeck NPCs or scenarios.  
- Different data models: simulator uses pandas DataFrames; holodeck uses custom ECS.

## 5. fleet-simulator → plato-ensign
**Connection:** ❌ **No direct import**  
- **Observation:** Simulator generates training data, but does not invoke ensign directly.  
- **Indirect path:** Data flows through plato-torch room → ensign exporter.

**Gap:**  
- No standalone simulator-to-ensign pipeline.  
- Simulator output requires room mediation to become ensign-ready.

## 6. holodeck-rust → plato-ensign
**Connection:** ⚠️ **Planned, not implemented**  
- **File:** `holodeck-rust/docs/ROADMAP.md` mentions “ensign export for sentiment NPCs”.  
- **Code search:** No `plato_ensign` in Rust source.  
- **Observation:** Holodeck stores interaction logs in SQLite; export tool not yet built.

**Gap:**  
- Missing Rust-side LoRA training pipeline.  
- No defined schema for converting NPC dialogues to training tiles.

## Integration Summary

### Connected Components
```
plato-torch ────→ fleet-simulator (data generation)
       │
       └───────→ plato-ensign (model export)
```

### Isolated Components
- **holodeck-rust**: Standalone multiplayer server with sentiment NPCs.  
- **constraint-theory-core**: Independent geometric snapping library (no imports to/from others).  
- **cudaclaw** / **flux-runtime**: GPU runtime and VM; not integrated with training stack.

### Safe Channels (P1)
1. **plato-torch ↔ fleet-simulator**: Stable, tested. Room config controls simulator.
2. **plato-torch → plato-ensign**: Functional export pipeline.
3. **holodeck-rust → SQLite**: Logging works; can be tapped later.

### Negative Space (P0)
- Do not assume holodeck integrates with plato-torch without a bridge.
- Do not expect fleet-simulator to directly train ensigns.
- Do not treat cudaclaw/flux as part of training pipeline—they are separate runtimes.
