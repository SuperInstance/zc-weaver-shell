## 1. **plato-torch** → **fleet-simulator**
**Connection:** ✅ **Connected via `fleet_simulator` Python module import**  
**Evidence:**  
- `plato-torch/rooms/deadband_room.py` imports:  
  ```python
  from fleet_simulator.simulator import FleetSimulator
  ```
- `plato-torch/configs/deadband_config.yaml` contains:  
  ```yaml
  simulator:
    enabled: true
    config_path: "../fleet-simulator/configs/training_scenario.yaml"
  ```
**Data Flow:**  
DeadbandRoom can instantiate a FleetSimulator object to generate synthetic training interactions (tiles).  
**Integration Gap:**  
No observed export of simulator results back to a shared data store; tiles appear to stay in memory within the room instance.

---

## 2. **plato-torch** → **holodeck-rust**
**Connection:** ⚠️ **Indirect / Config-Driven**  
**Evidence:**  
- `plato-torch/configs/holodeck_bridge.yaml` exists, referencing:  
  ```yaml
  holodeck_host: "localhost:8080"
  sentiment_npc_enabled: true
  ```
- No direct Rust imports in Python code (expected).  
- `holodeck-rust/src/protocol.rs` defines a JSON-over-WebSocket protocol that could be consumed by a Python client (not yet observed in plato-torch).  
**Integration Gap:**  
No active client bridge in plato-torch codebase. The config suggests planned integration, but no `holodeck_client.py` or similar is present in the repo.

---

## 3. **plato-torch** → **plato-ensign**
**Connection:** ✅ **Connected via explicit export pipeline**  
**Evidence:**  
- `plato-torch/export/ensign_exporter.py` imports:  
  ```python
  from plato_ensign.lora_adapter import LoRAAdapterBuilder
  ```
- `plato-torch/rooms/base_room.py` has method:  
  ```python
  def export_to_ensign(self, adapter_path: str) -> bool:
      # calls ensign exporter with accumulated tiles
  ```
**Data Flow:**  
Rooms accumulate interaction tiles → passed to ensign exporter → generates LoRA adapters.  
**Integration Gap:**  
The export is optional and manual; no automatic triggering based on tile count or quality metrics observed.

---

## 4. **fleet-simulator** → **holodeck-rust**
**Connection:** ❌ **No direct connection found**  
**Evidence:**  
- `fleet-simulator/` contains only Python simulation logic and scenario YAMLs.  
- No references to `holodeck`, `rust`, `websocket`, or `sentiment` in code or config.  
- `holodeck-rust/` does not reference `fleet_simulator`.  
**Integration Gap:**  
Simulated agent interactions do not currently feed into the holodeck MUD for sentiment‑NPC reactions, and holodeck events do not drive simulation scenarios.

---

## 5. **plato-ensign** → **Other components**
**Connection:** 🔄 **One‑way import from plato-torch only**  
**Evidence:**  
- `plato-ensign/setup.py` shows no dependencies on `fleet-simulator` or `holodeck-rust`.  
- It is a pure PyTorch/LoRA utility library called by plato-torch’s exporter.  
**Integration Gap:**  
Ensign adapters are not automatically loaded back into rooms or the simulator for fine‑tuned inference.

---

## 6. **Ghost Tiles & Mirror Play**
**Connection:** 🔍 **Referenced conceptually, not yet wired**  
**Evidence:**  
- `plato-torch/rooms/ghost_injector.py` exists but only reads static JSON dumps (e.g., `ghost_tiles/2026-04-18.json`).  
- No automatic ingestion of ghost tiles into active rooms.  
- `mirror_play` mentioned in `FLEET-CONTEXT.md` but no corresponding module found in inspected repos.  
**Integration Gap:**  
Ghost tiles are a dead‑store archive; not actively injected into training loops. Mirror Play is not implemented.

---

## 7. **cudaclaw & flux-runtime**
**Connection:** ❌ **No visible integration with the four core repos**  
**Evidence:**  
- `cudaclaw/` and `flux-runtime/` are separate Rust/CUDA projects.  
- No Python imports or config references in plato-torch, fleet-simulator, holodeck-rust, or plato-ensign.  
**Integration Gap:**  
GPU‑resident agent runtime and self‑assembling bytecode VM are isolated from the current training‑room‑simulator‑ensign pipeline.
