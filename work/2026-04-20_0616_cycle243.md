# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 243  
**Phase:** 4 — Build & Test  
**Status:** Direct file inspection of fleet repositories. Focus on actual imports, configuration references, and shared data structures.

## 1. plato-torch → fleet-simulator
**Connection:** Strong, direct dependency.  
- `plato-torch/rooms/__init__.py` imports `fleet_simulator` as a module.  
- `plato-torch/rooms/deadband_room.py` uses `fleet_simulator.Simulator` to generate synthetic training data.  
- Shared data structure: `Tile` objects (defined in `plato-torch/core/tile.py`) are consumed by the simulator to produce augmented training examples.  
**Integration Gap:** None observed. This is a working, documented pipeline.

## 2. plato-torch → plato-ensign
**Connection:** Direct export pathway.  
- `plato-torch/export/ensign_exporter.py` exists. It reads trained room state (`.pt` checkpoints) and converts them to LoRA adapters via `plato-ensign` library calls.  
- Configuration: `plato-torch/config/export_config.yaml` specifies `ensign_adapter_path`.  
**Integration Gap:** The `plato-ensign` repository itself is separate; the exporter assumes it is installed as a Python package. No observed runtime checks for missing dependency.

## 3. holodeck-rust → plato-torch
**Connection:** Indirect, via data interchange.  
- `holodeck-rust/src/npc/sentiment.rs` defines `SentimentEvent` structs.  
- `plato-torch/rooms/holodeck_room.py` (if implemented) would ingest these events as training tiles.  
**Integration Gap:** No observed `holodeck_room.py` in the current plato-torch room presets. The `holodeck_rust` repo does not contain Python bindings or a defined API for Plato consumption. This is a known missing link.

## 4. fleet-simulator → holodeck-rust
**Connection:** None observed.  
- `fleet-simulator/src/lib.rs` has no imports or references to `holodeck`.  
- No shared message formats or event bridges in either codebase.  
**Integration Gap:** Complete separation. These two systems simulate different domains (fleet coordination vs. sentiment-aware NPCs) with no current integration point.

## 5. plato-ensign → Others
**Connection:** Output-only.  
- `plato-ensign` is a downstream consumer of trained room state. It does not import or call back into `plato-torch`, `fleet-simulator`, or `holodeck-rust`.  
- It produces LoRA adapters that could be loaded by any of the above, but no automatic loading mechanism is defined.  
**Integration Gap:** Ensigns are "fire and forget" exports; no round-trip validation that the adapter works in the original room.

## 6. Ghost Tile System
**Connection:** Referenced but not fully integrated.  
- `plato-torch/core/ghost_injector.py` exists. It reads `ghost_tiles.json` and injects them as P0 negative examples into training batches.  
- `holodeck-rust` has a `ghost` module (`src/ghost/mod.rs`) but it's empty (stub).  
**Integration Gap:** Ghost tiles are implemented in Plato but not connected to Holodeck's NPC memory system.
