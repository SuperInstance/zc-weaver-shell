# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 370  
**Phase:** 4 — Build & Test  
**Status:** Direct file inspection of fleet repositories. Focus on actual imports, configuration references, and shared data structures.

## 1. plato-torch (Training Rooms)
**Location:** `plato-torch/`  
**Purpose:** 25 training room presets, self-training rooms.  
**Key Files:**
- `plato-torch/rooms/deadband_room.py` — Primary Deadband Protocol training room.
- `plato-torch/rooms/__init__.py` — Exports room classes.
- `plato-torch/trainer/room_trainer.py` — Orchestrates training loops.
- `plato-torch/data/tile_loader.py` — Loads interaction tiles for training.

**Connections Found:**
- **To plato-ensign:** `plato-torch/rooms/base_room.py` imports `from plato_ensign.export import export_lora`. This is a direct integration point for exporting room experience as LoRA adapters.
- **To fleet-simulator:** No direct imports found. However, `plato-torch/data/tile_loader.py` expects tile data in a specific JSON format. The `fleet-simulator` could generate compatible tiles, but no explicit import or configuration link exists in code.
- **To holodeck-rust:** No direct imports or references found.

**Integration Gap:** plato-torch does not import or configure fleet-simulator or holodeck-rust. It is a standalone training system that consumes tiles and can export to plato-ensign.

## 2. fleet-simulator (Fleet Simulation)
**Location:** `fleet-simulator/`  
**Purpose:** Generate training data via multi-agent simulation.  
**Key Files:**
- `fleet-simulator/src/simulation/engine.rs` — Core simulation loop.
- `fleet-simulator/src/agents/mod.rs` — Agent definitions and behaviors.
- `fleet-simulator/src/output/tile_writer.rs` — Writes interaction tiles to JSON.

**Connections Found:**
- **To plato-torch:** The `tile_writer.rs` outputs JSON files with a structure that matches the expected tile format in `plato-torch/data/tile_loader.py`. This is a *data compatibility* link, not a code dependency.
- **To holodeck-rust:** No imports found. Both are Rust projects but separate.
- **To plato-ensign:** No references.

**Integration Gap:** fleet-simulator operates independently, producing tile files. It does not directly trigger plato-torch training or plato-ensign exports.

## 3. holodeck-rust (Sentiment-Aware MUD)
**Location:** `holodeck-rust/`  
**Purpose:** Multi-User Dungeon server with NPCs that have sentiment memory.  
**Key Files:**
- `holodeck-rust/src/server/mod.rs` — MUD server implementation.
- `holodeck-rust/src/npc/sentiment.rs` — NPC sentiment tracking.
- `holodeck-rust/Cargo.toml` — Dependencies.

**Connections Found:**
- **To plato-torch:** None.
- **To fleet-simulator:** None.
- **To plato-ensign:** None.
- **To constraint-theory-core:** `Cargo.toml` includes `constraint-theory-core = { path = "../constraint-theory-core" }`. This is a direct dependency for geometric snapping in the game world.

**Integration Gap:** holodeck-rust is isolated from the training pipeline. It uses constraint-theory-core but does not connect to plato-torch, fleet-simulator, or plato-ensign.

## 4. plato-ensign (LoRA Export)
**Location:** `plato-ensign/`  
**Purpose:** Export room experience from plato-torch into LoRA adapters.  
**Key Files:**
- `plato-ensign/export/lora_exporter.py` — Main export logic.
- `plato-ensign/config/export_config.yaml` — Configuration for exports.

**Connections Found:**
- **To plato-torch:** As noted, plato-torch imports `plato_ensign.export`. This is a one-way dependency: plato-torch calls plato-ensign.
- **To fleet-simulator:** None.
- **To holodeck-rust:** None.

**Integration Gap:** plato-ensign is a downstream tool for plato-torch only. It does not interact with fleet-simulator or holodeck-rust.

## 5. Summary of Current Connections
```
plato-torch → plato-ensign (direct import, export capability)
fleet-simulator → (data) → plato-torch (JSON tile compatibility)
holodeck-rust → constraint-theory-core (direct Rust dependency)
```

**No connections exist between:**
- plato-torch and holodeck-rust
- fleet-simulator and holodeck-rust
- fleet-simulator and plato-ensign
- holodeck-rust and plato-ensign

## 6. Integration Opportunities (P1 Safe Channels)

2. **Connect DeadbandRoom to plato-relay:** `plato-relay` (if it exists) could stream training metrics or tile submissions. Check for `plato-relay` repo.
3. **End-to-end pipeline:** Automate: `fleet-simulator` → tile files → `plato-torch` training → `plato-ensign` export. This needs a orchestrator script or CI job.
4. **Shared tile schema:** Define a common tile schema (protobuf/JSON schema) used by `fleet-simulator`, `plato-torch`, and potentially `holodeck-rust` for consistency.
