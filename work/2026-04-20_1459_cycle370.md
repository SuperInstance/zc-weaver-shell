# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 370  
**Phase:** 4 — Build & Test  
**Status:** Direct file inspection of fleet repositories. Focus on actual imports, configuration references, and shared data structures.

## 1. plato-torch → fleet-simulator
**Connection:** Strong, direct integration via `SimulationRoom` preset.
- **plato-torch/rooms/simulation_room.py** imports `fleet_simulator` and uses `FleetSimulator` class.
- **fleet-simulator/src/lib.rs** exposes Rust structs via PyO3 bindings; Python wrapper in `fleet_simulator/__init__.py`.
- **Data flow:** `SimulationRoom` generates training episodes (state-action-reward sequences) using the simulator, converts them to interaction tiles, submits to PLATO room server.
- **Shared structures:** `SimulationConfig` (Rust) ↔ `SimulationParams` (Python) for scenario definition.

**Gap:** No observed two-way real-time control loop. The simulator runs episodic batches; no continuous telemetry stream back to influence room behavior mid‑episode.

## 2. plato-torch → holodeck-rust
**Connection:** Indirect, via tile format.
- **holodeck-rust/src/tile.rs** defines `Tile` struct with fields: `id`, `agent_id`, `room_id`, `content`, `sentiment`, `timestamp`.
- **plato-torch/rooms/base.py**’s `submit_tile()` serializes to same JSON schema expected by holodeck’s `/tile` endpoint.
- **Integration point:** Holodeck’s MUD server can ingest tiles from any PLATO room, including those generated by plato‑torch.

**Gap:** No direct import of holodeck‑rust in plato‑torch codebase. Connection is protocol‑based (HTTP/REST). No shared Rust/Cargo dependencies.

## 3. plato-torch → plato-ensign
**Connection:** Direct, via export pipeline.
- **plato-ensign/src/lib.rs** provides `export_room_to_lora(room_id: String)` function.
- **plato-torch/rooms/deadband_room.py** includes `export_experience()` method that calls plato‑ensign’s export API.
- **Data flow:** Room accumulates tiles → triggers export → ensign processes tiles into LoRA adapter weights → saves `.safetensors` file.

**Gap:** Export is manual/trigger‑based; no automatic continuous export when room reaches certain tile count.

## 4. fleet-simulator → holodeck-rust
**Connection:** None observed.
- **fleet-simulator/src/** has no references to “holodeck”, “tile”, or “MUD”.
- **holodeck-rust/src/** has no references to “fleet”, “simulator”, or “episode”.
- **Potential channel:** Could route simulator‑generated episodes through a PLATO room first, then to holodeck as tiles.

## 5. fleet-simulator → plato-ensign
**Connection:** None observed.
- No direct calls from simulator to ensign export.
- Simulator data must pass through a PLATO room to become tiles before ensign can process them.

## 6. holodeck-rust → plato-ensign
**Connection:** Possible but not yet implemented.
- **plato-ensign/src/lib.rs** expects tile JSON; holodeck stores tiles in SQLite.
- No existing endpoint in holodeck to trigger ensign export.
- Could be wired by adding an `/export` route in holodeck that calls ensign.
