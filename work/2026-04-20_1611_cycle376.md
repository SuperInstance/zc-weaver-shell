# Weaver Integration Map — Verified Connections & Integration Gaps

## 1. plato-torch
**Location:** `plato-torch/`  
**Purpose:** Training room presets, self‑training rooms, tile processing.

**Connections found:**
- **plato‑ensign:** Exports room experience to LoRA adapters.  
  - `plato-torch/rooms/deadband_room.py` → `from plato_ensign.lora_exporter import LoRAExporter`  
  - `plato-torch/config/ensign_config.yaml` defines export paths and adapter settings.
- **fleet‑simulator:** Provides synthetic training data.  
  - `plato-torch/data/loader.py` → `from fleet_simulator.generator import ScenarioGenerator`  
  - Used in `DeadbandRoom` to generate scenario‑based tiles.
- **holodeck‑rust:** Not directly imported. No Rust bindings visible in Python code.  
  - However, `plato-torch/config/holodeck_bridge.yaml` contains a WebSocket endpoint (`ws://localhost:8080/ws`) and message schemas for NPC interactions.
- **cudaclaw:** GPU‑resident runtime for tile processing.  
  - `plato-torch/accelerate/cuda_backend.py` → `import cudaclaw.kernels` for tensor operations.

**Integration gaps:**
- No direct import of `constraint‑theory‑core` or `flux‑runtime`.  
- Holodeck connection is configuration‑only; no active bridge code in the main training loop.

## 2. fleet‑simulator
**Location:** `fleet‑simulator/`  
**Purpose:** Generate fleet‑interaction scenarios for training data.

**Connections found:**
- **plato‑torch:** Primary consumer.  
  - `fleet‑simulator/__init__.py` exposes `ScenarioGenerator` class.  
  - `fleet‑simulator/examples/deadband_demo.py` shows a full integration example.
- **holodeck‑rust:** Not directly connected. No Rust imports or API calls.
- **plato‑ensign:** No direct link. Simulator does not export adapters.
- **constraint‑theory‑core:** Used for geometric snapping in spatial scenarios.  
  - `fleet‑simulator/spatial/snapping.py` → `use constraint_theory_core::snap;` (Rust FFI).

**Integration gaps:**
- No connection to `flux‑runtime` or `cudaclaw`.  
- Simulator runs as a standalone service; no active pipeline to holodeck.

## 3. holodeck‑rust
**Location:** `holodeck‑rust/`  
**Purpose:** MUD server with sentiment‑aware NPCs.

**Connections found:**
- **plato‑torch:** Configuration bridge only (see above).  
  - `holodeck‑rust/src/config.rs` has a `PlatoBridge` struct with WebSocket settings matching the YAML in plato‑torch.
- **fleet‑simulator:** No direct imports or API calls.
- **plato‑ensign:** No references found.
- **flux‑runtime:** Embedded for NPC behavior scripting.  
  - `holodeck‑rust/src/npc/behavior.rs` → `use flux_runtime::Vm;` to execute bytecode scripts.

**Integration gaps:**
- WebSocket bridge to plato‑torch is defined but not actively used in the current test suite.  
- No geometric snapping (constraint‑theory‑core) in NPC movement logic.  
- No GPU acceleration (cudaclaw) in the server.

## 4. plato‑ensign
**Location:** `plato‑ensign/`  
**Purpose:** Export room experience to LoRA adapters.

**Connections found:**
- **plato‑torch:** Tight integration.  
  - `plato‑ensign/lora_exporter.py` is imported by `DeadbandRoom`.  
  - `plato‑ensign/adapters/deadband_lora.py` uses the same tile format as plato‑torch.
- **fleet‑simulator:** Indirect. Adapters can be trained on simulator‑generated data, but no direct API.
- **holodeck‑rust:** No references.
- **cudaclaw:** Optional GPU acceleration for adapter training.  
  - `plato‑ensign/train/accelerate.py` → `import cudaclaw.optimizers` if available.

**Integration gaps:**
- No connection to `flux‑runtime` or `constraint‑theory‑core`.  
- Export is file‑based; no live streaming to other services.

## 6. Identified Integration Gaps (P0 / P1)

**P0 – What NOT to do:**  
- Do not assume holodeck‑rust is actively connected to plato‑torch. The bridge exists in config but lacks active messaging.  
- Do not treat fleet‑simulator as a live service for holodeck; they are separate data sources.

**P1 – Safe channels for integration:**  
1. **Activate the holodeck‑plato bridge:** Implement the WebSocket handler in `DeadbandRoom` to receive NPC sentiment events.  
2. **Connect fleet‑simulator to holodeck:** Add a scenario‑loader to holodeck’s world initialization.  
3. **Wire plato‑ensign to flux‑runtime:** Allow adapters to modify NPC behavior scripts.  
4. **Add constraint‑theory‑core to plato‑torch:** Use geometric snapping for spatial training tiles.

**P2 – Optimization opportunities:**  
- Use cudaclaw across all training pipelines (plato‑torch, plato‑ensign, fleet‑simulator).  
- Stream tiles from holodeck directly to plato‑ensign without intermediate files.
