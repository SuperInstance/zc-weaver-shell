# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 337  
**Phase:** 4 — Build & Test  
**Status:** Direct file inspection of fleet repositories. Focus on actual imports, configuration references, and shared data structures.

## 1. plato-torch (Training Rooms)
**Location:** `plato-torch/`  
**Purpose:** 25 training room presets, self-training rooms, tile ingestion and processing.

**Connections Found:**
- **plato-ensign:** Direct import in `plato-torch/src/export.rs`. Contains `EnsignExporter` struct that serializes room state (tiles, interactions) into LoRA-compatible datasets.
- **fleet-simulator:** No direct import found. However, `plato-torch/src/rooms/deadband_room.rs` defines a `DeadbandRoom` that could theoretically ingest simulation logs, but no current import of fleet-simulator types.
- **holodeck-rust:** No direct import. No shared types or configuration references in code.
- **Shared Data Structure:** `Tile` struct (`plato-torch/src/tile.rs`) is the core unit. Contains `agent_id`, `cycle`, `phase`, `content`, `metadata`. This is the universal training data format.

**Integration Gap:** plato-torch ingests tiles but does not directly call fleet-simulator or holodeck-rust. It is a sink, not an orchestrator.

## 2. fleet-simulator (Fleet Simulation)
**Location:** `fleet-simulator/`  
**Purpose:** Generate training data via multi-agent simulation.

**Connections Found:**
- **plato-torch:** No direct import. However, `fleet-simulator/src/simulation/output.rs` defines `SimulationOutput` which could be transformed into `Tile` format. No exporter to plato-torch currently implemented.
- **holodeck-rust:** No direct import. Separate domain (simulation vs. interactive MUD).
- **plato-ensign:** No direct import.
- **Shared Data Structure:** None with other repos. Internal `AgentState`, `Environment`.

**Integration Gap:** fleet-simulator runs in isolation. Its output is not automatically fed into plato-torch rooms. Manual pipeline step missing.

## 3. holodeck-rust (Sentiment-Aware MUD)
**Location:** `holodeck-rust/`  
**Purpose:** Multi-User Dungeon server with NPCs that have sentiment memory.

**Connections Found:**
- **plato-torch:** No direct import. However, `holodeck-rust/src/npc/memory.rs` defines `MemoryFragment` which resembles a `Tile` (agent_id, content, timestamp). Could be mapped.
- **fleet-simulator:** No direct import.
- **plato-ensign:** No direct import.
- **GhostInjector:** Referenced in `holodeck-rust/README.md` as a planned feature to inject "ghost tiles" (dead agents' lessons) into NPC memory. Not yet implemented.
- **Shared Data Structure:** `MemoryFragment` (content, embedding, valence). Not aligned with `Tile` but convertible.

**Integration Gap:** holodeck-rust operates as a standalone interactive server. No automatic export of interactions to training rooms.

## 4. plato-ensign (LoRA Adapter Export)
**Location:** `plato-ensign/`  
**Purpose:** Export room experience from plato-torch into LoRA adapters.

**Connections Found:**
- **plato-torch:** Direct dependency. `plato-ensign/Cargo.toml` includes `plato-torch = { path = "../plato-torch" }`. Uses `Tile` and `Room` types.
- **fleet-simulator:** No connection.
- **holodeck-rust:** No connection.
- **Shared Data Structure:** Uses `plato-torch::Tile` directly.

**Integration Gap:** plato-ensign only works with plato-torch rooms. Cannot export from fleet-simulator or holodeck-rust directly.

## 5. cudaclaw & flux-runtime (GPU/VM Infrastructure)
**Location:** `cudaclaw/`, `flux-runtime/`  
**Purpose:** GPU-resident agent runtime and self-assembling bytecode VM.

**Connections Found:**
- **plato-torch:** No direct import. Potential future use for on-GPU training.
- **fleet-simulator:** Could host simulated agents, but no current integration.
- **holodeck-rust:** Could run NPCs, but no integration.
- **Shared Data Structure:** None.

**Integration Gap:** These are low-level runtimes not yet wired into the high-level training/data generation stack.

## 6. constraint-theory-core (Geometric Snapping)
**Location:** `constraint-theory-core/`  
**Purpose:** Rust geometric constraint solver.

**Connections Found:** None with plato-torch, fleet-simulator, holodeck-rust, or plato-ensign. Appears to be a standalone library for spatial reasoning tasks.

## Identified Integration Points (P1 Safe Channels)

1. **fleet-simulator → plato-torch:** Build a `SimulationToTile` adapter in `plato-torch/src/ingest/` that converts `SimulationOutput` to `Tile` stream.
2. **holodeck-rust → plato-torch:** Implement `GhostInjector` that reads `MemoryFragment` from holodeck, converts to `Tile`, and posts to a DeadbandRoom.
3. **holodeck-rust ← plato-torch:** Inject `Tile` content (ghost tiles) into NPC memory as `MemoryFragment`.
4. **cudaclaw → fleet-simulator:** Host simulated agents on GPU runtime for speed.
5. **plato-ensign ← holodeck-rust:** Allow exporting NPC memory datasets as LoRA adapters (specialized character models).
