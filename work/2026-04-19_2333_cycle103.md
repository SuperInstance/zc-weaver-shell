## 1. Verified Connections (What's Connected)

### plato-torch ↔ fleet-simulator
- **Connection:** Direct integration via `fleet-simulator` as a data source for training rooms.
- **Evidence:** `plato-torch/rooms/DeadbandRoom.py` imports `fleet_simulator` and uses `FleetSimulator` class to generate simulation data for tile creation.
- **Data Flow:** Simulation runs in fleet-simulator → produce interaction logs → logs formatted into training tiles → tiles ingested by DeadbandRoom.
- **Status:** **Live**. This is a core training pipeline.

### plato-torch ↔ plato-ensign
- **Connection:** Export pipeline. Room experience (trained weights) can be exported as LoRA adapters.
- **Evidence:** `plato-ensign/export.py` references `plato_torch` room checkpoints. `plato-torch/README.md` states: "Ensigns are optional exports of room experience."
- **Data Flow:** Room trains → checkpoint saved → ensign export processes checkpoint → produces LoRA file.
- **Status:** **Designed, likely functional**. Export path exists but may need validation.

### holodeck-rust ↔ (Indirect via tiles)
- **Connection:** Tiles generated in plato-torch rooms could be consumed by holodeck-rust NPCs, but no direct code linkage observed.
- **Evidence:** `holodeck-rust/` design includes "sentiment-aware NPCs" that learn from interaction. Fleet doctrine mentions "ghost tiles" feeding into systems.
- **Data Flow:** Potential: tiles → holodeck NPC memory → influence behavior.
- **Status:** **Planned, not wired**. No import statements or API calls found.

## 2. Integration Gaps (What's Not Connected)

### Gap 1: holodeck-rust ↔ plato-torch (No direct link)
- **Issue:** No code in `holodeck-rust/src/` references `plato_torch` or its tile format.
- **Impact:** Holodeck NPCs cannot directly read training tiles from active rooms. They operate on separate sentiment models.
- **Potential Integration Point:** A tile ingestion endpoint in holodeck-rust that reads from `plato-torch/tiles/` directory or via plato-relay.

### Gap 2: fleet-simulator ↔ holodeck-rust (No connection)
- **Issue:** Fleet simulation data does not feed holodeck scenarios.
- **Impact:** Holodeck worlds lack dynamic fleet simulation events (e.g., agent failures, P0/P1/P2 decisions) as narrative elements.
- **Potential Integration Point:** `fleet-simulator` could emit event logs that `holodeck-rust` consumes to spawn relevant NPC dialogues or quests.

### Gap 3: plato-ensign ↔ holodeck-rust (Missing)
- **Issue:** Ensign adapters (LoRA) are not loaded into holodeck-rust NPC models.
- **Impact:** NPCs cannot benefit from specialized room training.
- **Potential Integration Point:** Allow holodeck to load `.safetensors` LoRA files exported by plato-ensign, attaching them to NPC inference.

### Gap 4: DeadbandRoom ↔ plato-relay (Not wired)
- **Issue:** `DeadbandRoom.py` does not call `plato-relay` service to broadcast tiles.
- **Impact:** Tiles remain local; other fleet components cannot subscribe in real-time.
- **Potential Integration Point:** Add a `RelayClient` in DeadbandRoom to publish each tile to plato-relay topic.
