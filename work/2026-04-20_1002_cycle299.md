# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 299

## 1. plato-torch
**Repository:** `plato-torch`  
**Purpose:** 25 training room presets, self-training rooms.  
**Key Files:**  
- `rooms/deadband_room.py` — Deadband Protocol training room.  
- `rooms/__init__.py` — Exports room classes.  
- `trainer/self_trainer.py` — Self‑training loop.  
- `configs/room_configs.yaml` — Room hyperparameters.  

**Connections:**  
- **To plato‑ensign:** `plato‑torch/rooms/base_room.py` imports `from plato_ensign import export_lora` (conditional). Rooms can export experience to LoRA adapters.  
- **To fleet‑simulator:** No direct import found. Potential integration via shared training data format (tiles).  
- **To holodeck‑rust:** No direct import. Plato‑torch rooms could consume tile data generated by holodeck, but no explicit link in code.  

**Integration Gap:**  
- Rooms are isolated training environments. No live connection to holodeck’s NPCs or fleet‑simulator’s agent simulations.

## 2. fleet‑simulator
**Repository:** `fleet‑simulator`  
**Purpose:** Fleet simulation for training data.  
**Key Files:**  
- `src/simulator/core.rs` — Core simulation engine.  
- `src/agents/mod.rs` — Agent definitions.  
- `src/output/tiles.rs` — Exports simulation events as tiles.  

**Connections:**  
- **To plato‑torch:** No Rust‑to‑Python import. However, `tiles.rs` writes tile data in JSON format that matches `plato‑torch/tile_schema.json`. This is a file‑based integration.  
- **To holodeck‑rust:** No direct import. Both are Rust crates but no `Cargo.toml` dependency observed.  
- **To plato‑ensign:** No connection.  

**Integration Gap:**  
- Simulator runs independently. Tiles must be manually moved to plato‑torch for training. No automatic pipeline.

## 3. holodeck‑rust
**Repository:** `holodeck‑rust`  
**Purpose:** MUD server with sentiment‑aware NPCs.  
**Key Files:**  
- `src/server/mud_server.rs` — MUD server loop.  
- `src/npc/sentiment.rs` — NPC sentiment engine.  
- `src/events/mod.rs` — In‑game events.  

**Connections:**  
- **To plato‑torch:** No direct import. NPC interactions could generate tiles, but no exporter is implemented.  
- **To fleet‑simulator:** No direct import.  
- **To plato‑ensign:** No connection.  

**Integration Gap:**  
- Holodeck is a standalone game server. No tile‑export mechanism exists. Sentiment‑aware NPCs are not used as training data sources.

## 4. plato‑ensign
**Repository:** `plato‑ensign`  
**Purpose:** Exports room experience into LoRA adapters.  
**Key Files:**  
- `src/exporter/lora_exporter.py` — LoRA adapter creation.  
- `src/utils/room_loader.py` — Loads plato‑torch room checkpoints.  

**Connections:**  
- **To plato‑torch:** Direct import in `room_loader.py`: `from plato_torch.rooms import DeadbandRoom`. Plato‑ensign is designed to post‑process plato‑torch training runs.  
- **To fleet‑simulator:** No connection.  
- **To holodeck‑rust:** No connection.  

**Integration Gap:**  
- Only works with plato‑torch rooms. Cannot export experience from holodeck or fleet‑simulator.

## 5. cudaclaw & flux‑runtime
**Repositories:** `cudaclaw`, `flux‑runtime`  
**Purpose:** GPU‑resident agent runtime and self‑assembling bytecode VM.  
**Connections:**  
- No imports found between these and the four core repos above.  
- These appear to be lower‑level infrastructure, not yet wired into the training/data pipeline.

## Integration Summary

### Connected:
1. **plato‑torch → plato‑ensign** — Room checkpoints can be exported to LoRA adapters.  
2. **fleet‑simulator → plato‑torch** — File‑based tile data sharing (JSON schema match).  

### Not Connected:
1. **holodeck‑rust** — Isolated MUD server.  
2. **cudaclaw / flux‑runtime** — Infrastructure not integrated.  
3. **Live training pipeline** — No automatic flow from holodeck or fleet‑simulator to plato‑torch rooms.
