## 1. Verified Current Connections (What's Connected)

Based on reading the actual fleet repos:

### **plato-torch ↔ plato-ensign** (Strong, Direct)
- **Connection:** `plato-ensign` is a direct export module of `plato-torch`.
- **Mechanism:** `plato-torch` accumulates interaction "tiles" in a training room. `plato-ensign` can be invoked to export that room's experience into a LoRA adapter (specialist model).
- **Evidence:** `plato-ensign/README.md` states: "Exports room experience from plato-torch into LoRA adapters." The `export.py` script imports from `plato_torch.rooms`.

### **plato-torch ↔ fleet-simulator** (Indirect, Data Flow)
- **Connection:** `fleet-simulator` generates training data (simulated fleet interactions). This data is intended to become "tiles" for `plato-torch` rooms.
- **Mechanism:** The simulator's output format (JSON logs of agent states and decisions) aligns with the tile schema expected by `plato-torch`'s `DeadbandRoom`.
- **Evidence:** `fleet-simulator/src/lib.rs` exports a `SimulationLog` struct. `plato-torch/plato_torch/rooms/deadband.py` has a `add_tile` method expecting a dict with `role`, `content`, and `metadata`.

### **holodeck-rust ↔ (plato-torch / fleet-simulator)** (Weak, Conceptual)
- **Connection:** `holodeck-rust` is a sentiment-aware MUD server. Its NPC interactions could be a source of training tiles or a test environment for trained ensigns.
- **Mechanism:** No direct import or API call found. Connection is documented as a potential integration point in architecture descriptions.
- **Evidence:** `FLEET-CONTEXT.md` lists it as a key technology. `holodeck-rust/README.md` describes "sentiment-aware NPCs" but does not mention specific fleet integrations.

## 2. Identified Integration Gaps (What's Not Connected)

### **Gap 1: No Automated Tile Pipeline from fleet-simulator to plato-torch**
- **Status:** Manual. Simulator logs must be formatted and loaded into a room.
- **Actionable Task:** Write a `SimLogToTile` adapter in `plato-torch/tools/` that consumes `SimulationLog` JSON and calls `room.add_tile()`.

### **Gap 2: holodeck-rust Operates in Isolation**
- **Status:** Standalone game server. No hooks for fleet data in or out.
- **Actionable Task:** Wire the **GhostInjector** (from `plato-torch`'s ghost tile system) into holodeck. Modify `holodeck-rust/src/npc/behavior.rs` to read from a `ghost_tiles.json` and influence NPC dialogue.

### **Gap 3: plato-ensign Export is Manual/Offline**
- **Status:** Requires explicit command to export a LoRA. No trigger based on room maturity or fleet events.
- **Actionable Task:** Connect `DeadbandRoom` to `plato-relay` (if exists) or a fleet event bus. Auto-export when tile count passes a threshold.

### **Gap 4: Missing End-to-End Test Pipeline**
- **Status:** No integrated test showing: Simulator → Tiles → Room Training → Ensign Export → Deployment.
- **Actionable Task:** Build a test in `plato-torch/tests/integration/` that mocks the simulator, populates a room, exports an ensign, and validates the output adapter file.
