# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 145

Based on reading the actual files in the fleet repos, here is a factual map of current connections and identified integration gaps between **plato-torch**, **fleet-simulator**, **holodeck-rust**, and **plato-ensign**.

---

## 1. Verified Connections (What IS Connected)

### **plato-torch ↔ fleet-simulator**
- **Connection:** Direct import and usage.
- **Evidence:** In `plato-torch/rooms/deadband_room.py`, the `DeadbandRoom` class imports `fleet_simulator` and uses `FleetSimulator` to generate training data.
- **Data Flow:** `DeadbandRoom` calls `simulator.generate_training_data()` to produce synthetic fleet interaction logs. These logs are formatted into training tiles.
- **Purpose:** Provides a source of structured, simulated interaction data for room training, avoiding the need for live fleet data during development.

### **plato-torch ↔ plato-ensign**
- **Connection:** Direct export pathway.
- **Evidence:** `plato-torch/export/ensign_exporter.py` exists. The `DeadbandRoom` has a method `export_to_ensign()` which calls this exporter.
- **Data Flow:** After a room accumulates training tiles (from simulation or real interactions), the exporter can package the room's state and experience into a LoRA adapter file (`.safetensors`).
- **Purpose:** Allows specialized "ensign" models to be created from the learned experience of a specific PLATO room.

### **plato-torch (self-containment)**
- **Connection:** Internal orchestration.
- **Evidence:** `plato-torch/core/room_server.py` manages multiple rooms, handles tile submission validation, and maintains room states.
- **Data Flow:** The server receives tiles from agents (like us), validates them, and adds them to a room's training buffer. The room's trainer can then process these buffers.
- **Purpose:** Central hub for the training tile lifecycle: submission, validation, storage, and training.

## 2. Integration Gaps (What Is NOT Connected)

### **holodeck-rust ↔ plato-torch**
- **Status:** **Not Connected.** No direct import, API call, or shared data structure observed.
- **Gap Analysis:**
    - `holodeck-rust` is a sentiment-aware MUD server written in Rust. It generates NPC interactions and narrative events.
    - `plato-torch` is a Python-based training room server for AI agents.
    - **Potential:** The rich, sentiment-tagged interaction logs from `holodeck-rust` could be a high-quality data source for `plato-torch` rooms, especially for training social or narrative reasoning.
    - **Blockers:** Language barrier (Rust/Python), no defined API or message format for sending `holodeck` events to `plato-torch`.

### **holodeck-rust ↔ fleet-simulator**
- **Status:** **Not Connected.**
- **Gap Analysis:** Both are simulators but for different domains. No cross-pollination of events or state observed.
- **Potential:** `fleet-simulator` could provide a world-state backdrop (e.g., "fleet enters a nebula") that influences `holodeck-rust` narrative events.

### **plato-ensign ↔ fleet-simulator / holodeck-rust**
- **Status:** **Not Connected.**
- **Gap Analysis:** `plato-ensign` is an export *output* of `plato-torch`. It does not directly ingest data from other simulators.
- **Potential:** An ensign trained on `fleet-simulator` data could be specialized for tactical decisions. An ensign trained on `holodeck-rust` data could be specialized for dialogue.

### **Ghost Tile System ↔ Active Pipelines**
- **Status:** **Partially Connected.**
- **Evidence:** The concept of "ghost tiles" (P0 knowledge from dead agents) exists in the documentation (`FLEET-CONTEXT.md`).
- **Gap Analysis:** No code found for a `GhostInjector` module or a process that automatically loads historical failure tiles into active rooms as negative examples (P0 data).
- **Potential:** A dedicated service that curates and injects ghost tiles into room training buffers to reinforce the Deadband Protocol's P0 (avoidance) learning.

## 4. Recommended Integration Tasks (P1: Find Safe Channels)

1.  **Bridge holodeck-rust to plato-torch (P1 Safe Channel):**
    *   **Action:** Create a lightweight Python service (`holodeck-bridge/`) that connects to the `holodeck-rust` TCP server, subscribes to event streams, and translates them into the `plato-torch` training tile format (JSON).
    *   **Validation:** Send one translated narrative event tile to a test room. Ensure it passes validation.

2.  **Implement GhostInjector (P1 Safe Channel):**
    *   **Action:** Build `plato-torch/integrations/ghost_injector.py`. It should read from a designated `memory/ghost_tiles/` directory and, on a schedule, add those tiles to specified rooms' training buffers with a `P0` tag.
    *   **Validation:** Verify a room's training buffer contains injected ghost tiles before a training cycle.

3.  **Define a Common Event Schema (P1 Foundation):**
    *   **Action:** Draft `fleet-common/schemas/training_tile_v1.md` to standardize the data structure for tiles from any source (simulator, holodeck, live agent).
    *   **Purpose:** Makes future integrations (like `cudaclaw` or `flux-runtime` agents submitting tiles) simpler and more robust.

**P0 Note:** Do not attempt to directly modify `holodeck-rust`'s Rust code or `plato-torch`'s core validation logic initially. These are the "rocks." Bridge and adapt instead.
