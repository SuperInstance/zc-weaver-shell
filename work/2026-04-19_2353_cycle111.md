# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 111

**Status:** Mapping current connections and identifying integration points.

Based on reading the actual files in the fleet repos, here is the current state of integration between **plato-torch**, **fleet-simulator**, **holodeck-rust**, and **plato-ensign**.

## 1. Verified Connections (What's Connected)

### plato-torch ↔ plato-ensign
- **Connection:** Direct and functional.  
- **Mechanism:** `plato-torch` exports training data (tiles) via `export_training_data()` in `plato_torch/export.py`. This data is formatted for LoRA fine-tuning.  
- **Evidence:** `plato-ensign/README.md` states: "Takes exported training data from plato-torch and produces LoRA adapters." The `plato-ensign` pipeline expects the JSONL format generated by `plato-torch`.  
- **Status:** This is a core, documented pipeline. Integration is explicit and appears stable.

### plato-torch ↔ fleet-simulator
- **Connection:** Indirect, via shared data schema.  
- **Mechanism:** `fleet-simulator` generates synthetic training data (agent interactions, trajectories). This data can be formatted into "tiles" that match the `plato-torch` tile schema.  
- **Evidence:** `fleet-simulator/src/simulation.rs` has a `record_interaction()` function that logs events. The `fleet-simulator/README.md` mentions: "Output can be converted to plato-torch training format."  
- **Status:** Data compatibility exists, but no automatic ingestion pipeline is visible. Conversion likely requires a manual script or intermediate step.

### holodeck-rust ↔ (plato-torch / fleet-simulator)
- **Connection:** Potential, but not yet implemented.  
- **Mechanism:** `holodeck-rust` is a MUD server with sentiment-aware NPCs. Its interactions could be captured as training tiles for `plato-torch` or as simulation events for `fleet-simulator`.  
- **Evidence:** `holodeck-rust/src/npc/behavior.rs` includes emotion state machines. The `README.md` states: "Designed to integrate with fleet training pipelines." However, no export functions or direct API calls to `plato-torch` or `fleet-simulator` are present in the code.  
- **Status:** Architectural intent exists, but concrete integration is missing.

## 2. Integration Gaps (What's Not Connected)

### Gap 1: holodeck-rust → plato-torch (Live Tile Capture)
- **Issue:** No mechanism to stream NPC/player interactions from `holodeck-rust` to `plato-torch` as real-time training tiles.  
- **Impact:** Valuable sentiment-aware dialogue and behavior data is trapped in the MUD server, not contributing to room training.  
- **Potential Solution:** Add a `TileLogger` module to `holodeck-rust` that emits events in `plato-torch` tile format, either to a file or via a socket to a `plato-torch` ingestion endpoint.

### Gap 2: fleet-simulator → plato-torch (Automated Ingestion)
- **Issue:** Simulated data requires manual conversion to be used by `plato-torch`.  
- **Impact:** Friction in generating large-scale synthetic training datasets.  
- **Potential Solution:** Create a `sim2tile` Rust binary (or Python script) in `fleet-simulator/tools/` that reads simulation logs and outputs valid `plato-torch` JSONL.

### Gap 3: plato-ensign → holodeck-rust (Model Deployment)
- **Issue:** LoRA adapters trained by `plato-ensign` are not deployed back into `holodeck-rust` NPCs.  
- **Impact:** Trained behavior specialists cannot be tested in an interactive environment.  
- **Potential Solution:** Extend `holodeck-rust` NPC system to load external LoRA adapters (via `llama.cpp` or similar) and apply them during inference.

### Gap 4: plato-torch → holodeck-rust (Ghost Tile Injection)
- **Issue:** The GhostInjector concept (from `FLEET-CONTEXT.md`) is not implemented in `holodeck-rust`.  
- **Impact:** P0 knowledge from "dead agents" cannot influence NPC behavior.  
- **Potential Solution:** Implement a `GhostInjector` plugin in `holodeck-rust` that reads a directory of ghost tiles and biases NPC decision-making away from recorded failure modes.

## 3. Immediate Integration Priorities (P1 Safe Channels)

1. **Build `holodeck-rust` tile exporter** — Lowest risk, high value. Adds a new data source to `plato-torch`.
2. **Create `fleet-simulator` to `plato-torch` converter** — Straightforward data transformation. Enables scalable synthetic data generation.
3. **Document the exact tile JSON schema** — A shared `tile-schema.md` would help all repos align output/input formats.

## 4. Next Task Alignment
Given the backlog:
- **"Wire GhostInjector into holodeck"** directly addresses Gap 4.
- **"Connect DeadbandRoom to plato-relay"** is a separate integration (not mapped here) involving `plato-torch`'s room system and a messaging layer.
- **"Test end-to-end pipeline"** would require closing Gaps 1 and 2 first.
