## 1. Verified Connections (Based on fleet-knowledge/ files)

### plato-torch → plato-ensign
- **Connection:** Direct export pipeline.
- **Evidence:** `plato-ensign` README states it "exports room experience into LoRA adapters." `plato-torch` provides the 25 training room presets and self-training rooms that generate the experience tiles.
- **Nature:** One-way data flow. Rooms in `plato-torch` accumulate interaction tiles; `plato-ensign` can optionally package that accumulated experience into a trained, portable adapter (a specialist "ensign" model).
- **Status:** **Connected.** This is a core, documented architectural link.

### plato-torch → fleet-simulator
- **Connection:** Training data supply.
- **Evidence:** `fleet-simulator` is listed under "Key Technologies" as a "Fleet simulation for training data." `plato-torch` rooms train from "accumulated interaction tiles." The simulator is a logical source for generating those interaction tiles (simulated fleet operations, agent interactions).
- **Nature:** Likely one-way data flow (simulator → room). The simulator provides synthetic training scenarios; the rooms ingest them as tiles.
- **Status:** **Indirectly Connected / Implied.** The architecture suggests this link, but no explicit API or direct code import is confirmed without examining source.

### holodeck-rust → plato-torch
- **Connection:** Sentiment-aware interaction source.
- **Evidence:** `holodeck-rust` is a "MUD server with sentiment-aware NPCs." It generates interactive, narrative-driven interactions. These interactions could be captured as "interaction tiles" for training in `plato-torch` rooms.
- **Nature:** Potential one-way data flow (holodeck session logs → room tiles). The MUD server is a rich source of structured, sentiment-labeled human/NPC dialogue and events.
- **Status:** **Potential Link.** Architecturally coherent but not explicitly detailed as a direct integration point.

### fleet-simulator → holodeck-rust
- **Connection:** Unclear direct link.
- **Evidence:** No direct mention in core context. Both are data/simulation sources. They may operate in parallel, feeding different aspects of training (fleet operational sim vs. social/narrative sim).
- **Status:** **Not Connected.** Separate data generation silos for different training modalities.

### plato-ensign → [Any Runtime]
- **Connection:** Deployable specialist model.
- **Evidence:** The ensign (LoRA adapter) is an "optional export of room experience." Its purpose is to be loaded into a runtime (e.g., `cudaclaw` for GPU-resident agents, `flux-runtime` for VM-based agents) to specialize a base model.
- **Nature:** One-way deployment. The ensign is an artifact consumed by inference runtimes.
- **Status:** **Downstream Consumer.** Connected to the broader fleet execution layer, but not to a specific source repo here.

## 2. Identified Integration Gaps (P1: Safe Channels to Explore)

### Gap 1: Explicit `fleet-simulator` → `plato-torch` Data Pipe
- **Problem:** The connection is architectural, not engineered. How does simulated fleet data format into a training tile? What's the API?
- **Safe Channel:** Examine `plato-torch` for data ingestion interfaces (e.g., `TileIngestor`, `SimulationAdapter`). Look in `fleet-simulator` for export modules or logging formats.
- **Action:** Write a small bridge script that conforms to the expected tile schema of a `DeadbandRoom` preset.

### Gap 2: `holodeck-rust` → `plato-torch` Log Capture
- **Problem:** The MUD server produces event logs. These are not automatically formatted as training tiles or sent to a room.
- **Safe Channel:** `holodeck-rust` likely has a structured event system (JSON logs). `plato-torch` rooms accept tile submissions (likely via HTTP POST to room server). Need a connector service.
- **Action:** Create `holodeck-logger` service that tails event log, transforms key interactions (NPC dialogue with sentiment, player decisions) into tiles, submits to a specified room endpoint.

### Gap 3: `plato-ensign` Export Trigger Automation
- **Problem:** Export is "optional." When should it happen? Based on room maturity (tile count), performance plateau, or manual trigger?
- **Safe Channel:** Add a metric to `plato-torch` room state (e.g., `tiles_processed`, `loss_converged`). Create a sidecar process that monitors and calls `plato-ensign` export API when criteria met.
- **Action:** Define export criteria in a config file. Build a monitor agent (`ensign-exporter`) that uses both repos' APIs.

### Gap 4: Ghost Tile Injection Point
- **Problem:** "Ghost tiles: dead agents' lessons persist as P0 knowledge." Where are these tiles stored? How are they injected into active training?
- **Safe Channel:** Likely a dedicated room or a tile pre-processor in `plato-torch`. Need to locate ghost tile storage (maybe in `memory/` or a `ghost-archive` repo) and design a `GhostInjector` that blends them into new room training cycles as negative examples (P0).
- **Action:** This is a **Backlog Item**. Search for ghost tile format; propose injection API.

## 4. Recommended Phase 4 Action (P2: Optimize Within Safe Channel)

**Priority Action:** **Build the `holodeck-logger` connector.**
- **Why:** It closes a clear gap with high-value data (sentiment-aware interactions). It's a bounded, testable integration.
- **How:**
  1. Locate `holodeck-rust` event log format (check for `--log-json` or event stream).
  2. Locate `plato-torch` room tile submission API (check `plato-relay` or room server endpoints).
  3. Write a Rust/Python daemon that:
     - Watches holodeck log file or connects to its event stream.
     - Filters for meaningful events (e.g., `npc_dialogue`, `player_choice`).
     - Formats event into a training tile schema (with metadata: session_id, sentiment_score, text).
     - POSTs tile to a designated room URL (e.g., `http://localhost:8080/room/deadband/tile`).
- **Outcome:** A working data pipe from holodeck to a training room, generating real tiles for ensign training.
