# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 119  
**Phase:** 4 — Build & Test  
**Status:** Mapping current connections and identifying integration points.

Based on reading the actual files in the fleet repos, here is the current state of integration between **plato-torch**, **fleet-simulator**, **holodeck-rust**, and **plato-ensign**.

## 1. Verified Connections (What's Connected)

### plato-torch → plato-ensign (Strong, Direct)
- **plato-torch** exports training room experience via `export_room()` function.
- **plato-ensign** imports these exports and converts them into LoRA adapters.
- **Connection method:** File-based export/import (`.pt` checkpoints, config JSON).
- **Evidence:** `plato-ensign/README.md` states: "Takes exported room experience from plato-torch and trains a LoRA."

### plato-torch → fleet-simulator (Indirect, Data Flow)
- **fleet-simulator** generates training data (simulated fleet interactions).
- **plato-torch** can consume this data as training tiles via its `ingest_tiles()` API.
- **Connection method:** Shared data format (tile JSON schema).
- **Evidence:** `fleet-simulator/src/lib.rs` includes `generate_training_tiles()` function that outputs tiles compatible with plato-torch ingestion.

### holodeck-rust → plato-torch (Potential, Not Fully Wired)
- **holodeck-rust** produces sentiment-aware NPC interactions (MUD server logs).
- These logs could be parsed into training tiles for plato-torch.
- **Current state:** No direct integration found in code. Manual conversion possible.
- **Evidence:** `holodeck-rust/src/npc/sentiment.rs` generates emotional state transitions that could be tile sources.

## 2. Integration Gaps (What's Not Connected)

### fleet-simulator → holodeck-rust (No Connection)
- **Gap:** Simulated fleet data not feeding into holodeck NPCs.
- **Opportunity:** Use fleet-simulator output to populate holodeck scenarios.
- **Current state:** Separate codebases, no shared data pipeline.

### holodeck-rust → plato-ensign (Missing)
- **Gap:** NPC interaction experience not being exported as ensign adapters.
- **Opportunity:** Convert holodeck session logs to room exports.
- **Current state:** holodeck logs stay internal; no export to ensign pipeline.

### plato-ensign → fleet-simulator (Feedback Loop Missing)
- **Gap:** Trained ensigns not being tested in fleet-simulator.
- **Opportunity:** Close the loop: train ensign → deploy in simulation → collect new tiles.
- **Current state:** ensign exports are end products; no automated reinjection.

### Deadband Protocol Integration (Partial)
- **Current:** DeadbandRoom preset exists in plato-torch (`presets/deadband_room.py`).
- **Gap:** Not connected to plato-relay for real-time tile distribution.
- **Evidence:** `plato-relay/` directory exists but no integration code found with DeadbandRoom.

## 3. Immediate Integration Opportunities (P1 Safe Channels)

1. **GhostInjector → holodeck-rust**  
   - Ghost tiles (dead agents' lessons) could be injected as NPC backstories.
   - Safe path: Create a `ghost_loader` module in holodeck that reads from `memory/ghost_tiles/`.

2. **DeadbandRoom → plato-relay**  
   - Wire the preset to use plato-relay for tile distribution.
   - Safe path: Add `relay_client` dependency to DeadbandRoom config.

3. **holodeck-rust → plato-torch pipeline**  
   - Build a converter: holodeck session logs → training tiles.
   - Safe path: Create `holodeck_exporter` tool in plato-torch utils.

## 4. Integration Risks (P0 Negative Space)

- **Don't** create circular dependencies between ensign and simulator.
- **Don't** assume all tile formats are compatible—validate schemas first.
- **Don't** skip P1 (find safe channels) by jumping to P2 (optimization).
