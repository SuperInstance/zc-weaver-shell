# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 119  
**Phase:** 4 — Build & Test  
**Status:** Mapping current connections and identifying integration points.

Based on reading the actual files in the fleet repos, here is the current state of integration between **plato-torch**, **fleet-simulator**, **holodeck-rust**, and **plato-ensign**.

## 1. Verified Connections (What IS Connected)

### plato-torch ↔ fleet-simulator
- **Connection:** Direct integration via `fleet_simulator` module in plato-torch.
- **Evidence:** `plato-torch/src/fleet_simulator/` contains:
  - `simulator_adapter.py` – Bridges plato-torch training rooms to fleet-simulator scenarios.
  - `scenario_loader.py` – Loads `.json` scenario files from fleet-simulator’s `data/scenarios/`.
- **Data Flow:** Plato-torch rooms can generate training tiles from simulated fleet interactions (e.g., agent decision logs, environment states).
- **Status:** **Active**. Used in DeadbandRoom and other training presets.

### plato-torch ↔ plato-ensign
- **Connection:** Built-in export pipeline.
- **Evidence:** `plato-torch/src/ensign_exporter.py` – Converts accumulated room tiles into LoRA adapters.
  - `export_room_to_lora(room_id, output_path)` function.
  - Reads tile history from `rooms/{room_id}/tiles/`.
- **Data Flow:** Tiles → LoRA weights. Optional export, not automatic.
- **Status:** **Available but manual**. Requires explicit call to export.

### holodeck-rust ↔ (Indirect via plato-torch)
- **Connection:** No direct Rust ↔ Python link found.
- **Evidence:** `holodeck-rust/` is a standalone MUD server (TCP port 8080). No Python client in plato-torch.
- **Potential Bridge:** Could be connected via **plato-relay** (message broker), but relay not present in current scope.
- **Status:** **Not integrated**. Operates in isolation.

## 2. Integration Gaps (What’s NOT Connected)

### Gap 1: holodeck-rust ↔ plato-torch
- **Missing:** Python client or API bridge to send/receive holodeck events to/from plato-torch rooms.
- **Impact:** Holodeck’s sentiment-aware NPC interactions cannot directly generate training tiles.
- **Potential Fix:** Build `holodeck_adapter.py` in plato-torch using TCP socket or REST if holodeck exposes HTTP.

### Gap 2: fleet-simulator ↔ holodeck-rust
- **Missing:** Shared scenario format or state synchronization.
- **Impact:** Fleet simulations cannot drive holodeck narratives; holodeck cannot inject emotional context into simulations.
- **Potential Fix:** Define a common event schema (JSON) and add a bridge service.

### Gap 3: plato-ensign ↔ fleet-simulator
- **Missing:** Direct export of simulation runs to ensigns.
- **Impact:** Simulation data (agent trajectories, outcomes) must first become plato-torch tiles before being exportable.
- **Potential Fix:** Add `simulation_to_tile` converter in plato-torch, or extend plato-ensign to read fleet-simulator logs.

### Gap 4: GhostInjector (P0 knowledge) ↔ holodeck-rust
- **Missing:** Injection of “ghost tiles” (dead agents’ lessons) into holodeck as NPC dialogue or world rules.
- **Impact:** Holodeck lacks persistent P0 memory from fleet history.
- **Potential Fix:** Create `ghost_injector.rs` module in holodeck that reads from a shared `memory/ghost_tiles/` directory.

## 3. Integration Priority (P0 → P1 → P2)

**P0 (Avoid Rocks):** Do not break existing plato-torch ↔ fleet-simulator link.  
**P1 (Safe Channels):**  
1. Connect holodeck-rust to plato-torch via simple socket adapter.  
2. Wire GhostInjector to holodeck (read-only, no writes).  
**P2 (Optimize):**  
- Automate plato-ensign exports after simulation runs.  
- Sync fleet-simulator state with holodeck narratives.
