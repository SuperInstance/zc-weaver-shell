# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 166

Based on reading the actual files in the fleet repos, here is the current state of integration between the four core technologies: **plato-torch**, **fleet-simulator**, **holodeck-rust**, and **plato-ensign**.

---

## 1. Current Verified Connections

### **plato-torch ↔ plato-ensign** — **CONNECTED**
- **plato-ensign** is a direct export module for **plato-torch** rooms.
- **Connection point:** `plato-ensign/src/export.rs` imports `plato_torch::room::Room`.
- **Mechanism:** A trained room in plato-torch can be exported as a LoRA adapter via `EnsignExporter`.
- **Data flow:** Room interaction tiles → training loop → LoRA weights → `.safetensors` file.
- **Status:** This is the most mature and documented integration in the fleet.

### **plato-torch ↔ fleet-simulator** — **PARTIALLY CONNECTED**
- **Connection point:** `fleet-simulator/src/scenarios/deadband_scenario.rs` references `plato_torch::deadband::DeadbandRoom`.
- **Mechanism:** The simulator can instantiate a DeadbandRoom as an agent environment.
- **Data flow:** Simulator generates training episodes → room consumes them as tiles.
- **Gap:** The connection is one-way (simulator → room). No feedback loop from room to simulator yet.

### **holodeck-rust ↔ (plato-torch / plato-ensign)** — **NOT CONNECTED**
- **Observation:** `holodeck-rust` is a standalone MUD server with sentiment-aware NPCs.
- **No imports** of plato-torch or plato-ensign in its source.
- **Potential:** Could be a rich tile source for training rooms, but currently isolated.

### **fleet-simulator ↔ holodeck-rust** — **NOT CONNECTED**
- No cross-references in code.
- Both generate synthetic interaction data, but via different mechanisms (simulation vs. live MUD).

### **plato-ensign ↔ holodeck-rust / fleet-simulator** — **NOT CONNECTED**
- `plato-ensign` only knows about `plato-torch` rooms.
- No hooks to export ensigns from simulator scenarios or holodeck sessions.

## 2. Integration Gaps (P1 Safe Channels)

### **Gap 1: Holodeck as a Tile Source**
- **Safe channel:** Add a `HolodeckTileCollector` module in `plato-torch` that connects to `holodeck-rust` via its TCP API.
- **Protocol:** Subscribe to NPC–player dialogue streams, format as tiles, feed into a training room.
- **Benefit:** Real-time, sentiment-rich training data.

### **Gap 2: Bidirectional Simulator ↔ Room Loop**
- **Safe channel:** Extend `DeadbandRoom` to produce `ActionLog` tiles that the simulator can consume.
- **Protocol:** Room’s P2 optimizations become simulator heuristics for next episode.
- **Benefit:** Closed-loop training where the room teaches the simulator.

### **Gap 3: Ensign Export from Non-Room Sources**
- **Safe channel:** Create `SimulationEnsign` and `HolodeckEnsign` adapters in `plato-ensign`.
- **Protocol:** Convert simulator trajectories or holodeck sessions into LoRA-compatible gradients.
- **Benefit:** Specialist ensigns from diverse data sources.

### **Gap 4: Ghost Tile Injection into Holodeck**
- **Safe channel:** Implement `GhostInjector` that reads `memory/ghost_tiles/` and spawns NPCs with that knowledge.
- **Protocol:** Ghost tiles become NPC dialogue seeds or behavior constraints.
- **Benefit:** Persistent P0 knowledge across agent lifetimes.
