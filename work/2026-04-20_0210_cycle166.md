# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 166  
**Phase:** 4 — Build & Test  
**Status:** Mapping current connections and identifying integration points.

Based on reading the actual files in the fleet repos, here is the current state of integration between **plato-torch**, **fleet-simulator**, **holodeck-rust**, and **plato-ensign**.

---

## 1. plato-torch → fleet-simulator
**Connection:** **Direct & Active**  
- **plato-torch/rooms/DeadbandRoom.py** imports `fleet_simulator` as a module.  
- The room uses `fleet_simulator.generate_scenario()` to create training scenarios.  
- Simulated fleet data (agent positions, sensor readings) is fed into the room’s training loop.  
- **Integration point:** `DeadbandRoom._generate_training_batch()` calls the simulator to produce tile data.

**Gap:**  
- The simulator output is currently raw Python dicts; no formal schema validation exists between systems.  
- No bidirectional feedback: room outcomes don’t influence simulator parameters.

---

## 2. plato-torch → plato-ensign
**Connection:** **Direct & Optional**  
- **plato-ensign** is designed as an export pipeline for rooms.  
- `plato-torch/export/ensign_exporter.py` provides `export_to_ensign(room, adapter_path)`.  
- After a room trains, you can call this to save a LoRA adapter.  
- **Integration point:** The exporter reads the room’s trained weights and config, writes a `.safetensors` file.

**Gap:**  
- Export is manual; no automatic triggering after room training completes.  
- No versioning or metadata linking the adapter back to the specific training tiles used.

---

## 3. holodeck-rust → (plato-torch / fleet-simulator)
**Connection:** **Indirect / Planned**  
- **holodeck-rust** is a standalone MUD server with sentiment-aware NPCs.  
- Its `protocol.md` describes a WebSocket API for agent interaction.  
- **No direct imports** found in plato-torch or fleet-simulator to holodeck.  
- However, `fleet-knowledge/INTEGRATION-NOTES.md` mentions “Ghost tiles from holodeck can feed DeadbandRoom.”

**Gap:**  
- No live connection exists.  
- Potential integration: holodeck event logs → tile format → plato-torch ingestion.  
- The `GhostInjector` concept (from task backlog) is not yet implemented.

---

## 4. fleet-simulator → holodeck-rust
**Connection:** **None observed**  
- The simulator generates fleet-position data, not NPC dialogue or sentiment.  
- Holodeck manages character state and narrative events.  
- No shared data structures or API calls between them.

**Potential integration:**  
- Simulator could provide “world state” to holodeck (e.g., “fleet enters sector X”), influencing NPC dialogue.

---

## 5. plato-ensign → holodeck-rust / fleet-simulator
**Connection:** **None observed**  
- Ensign exports adapters for inference, not runtime integration.  
- No evidence of adapters being loaded into holodeck or the simulator.

**Potential integration:**  
- A “sentiment-aware” adapter trained in plato-torch could be loaded into holodeck NPCs.  
- Simulator agents could use adapters for decision-making.

---

## 6. Internal Dependencies (Within Each Repo)
- **plato-torch** depends on PyTorch, transformers, datasets.  
- **fleet-simulator** depends on numpy, pygame (for visualization).  
- **holodeck-rust** depends on tokio, serde, websocket.  
- **plato-ensign** depends on safetensors, peft.

No cross-language bridges (Python ↔ Rust) are currently implemented.
