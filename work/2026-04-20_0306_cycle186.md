# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 186

Based on reading the actual files in the fleet repos, here is the current state of integration between **plato-torch**, **fleet-simulator**, **holodeck-rust**, and **plato-ensign**.

## Verified Connections (What's Connected)

### 1. **plato-torch ↔ fleet-simulator**
- **Connection Type:** Data pipeline (training data generation)
- **How it works:** `fleet-simulator` generates synthetic training scenarios (agent interactions, decision trees). These scenarios are formatted as **interaction tiles** and ingested by `plato-torch` rooms.
- **Evidence:** `fleet-simulator/src/lib.rs` exports a `Simulation` struct with a `generate_tiles()` method that returns `Vec<Tile>`. `plato-torch/rooms/deadband_room.py` imports `Tile` from `fleet_simulator` (via Python bindings).
- **Status:** **Active** — tiles flow from simulator to training rooms.

### 2. **plato-torch ↔ plato-ensign**
- **Connection Type:** Model export pipeline
- **How it works:** `plato-ensign` hooks into a trained `plato-torch` room, extracts its learned weights (LoRA adapters), and serializes them for deployment.
- **Evidence:** `plato-ensign/src/exporter.rs` has a `RoomExporter` struct that takes a `room_id` (from plato-torch) and calls `export_lora()`.
- **Status:** **Active** — but only triggered manually via CLI. No automatic export on room maturity.

### 3. **holodeck-rust ↔ fleet-simulator** (Partial)
- **Connection Type:** Shared data schema (NPC sentiment)
- **How it works:** `holodeck-rust` uses a `Sentiment` enum (`Positive`, `Neutral`, `Negative`) for NPC reactions. `fleet-simulator` generates dialogue trees that include sentiment tags.
- **Evidence:** Both repos define compatible `Sentiment` types. `fleet-simulator`'s `DialogueNode` has a `sentiment: Sentiment` field.
- **Status:** **Schema-aligned** — but no runtime integration. Simulator doesn't push NPC data to holodeck.

## Integration Gaps (What's Not Connected)

### 1. **holodeck-rust ↔ plato-torch**
- **Gap:** No direct connection.
- **Impact:** NPC interactions in holodeck are not captured as training tiles for plato-torch rooms.
- **Opportunity:** Wire a `TileLogger` in holodeck that sends player-NPC exchanges to a plato-torch room (e.g., `npc_dialogue_room`).

### 2. **plato-ensign ↔ fleet-simulator**
- **Gap:** Ensign exports are not validated against simulator scenarios.
- **Impact:** A trained ensign might perform well in its room but fail in simulated edge cases.
- **Opportunity:** Add a `validate_against_simulator(sim_scenarios)` step in `plato-ensign` export pipeline.

### 3. **holodeck-rust ↔ plato-ensign**
- **Gap:** No integration.
- **Impact:** NPC behavior cannot be improved via ensign adapters.
- **Opportunity:** Allow holodeck to load a `plato-ensign` LoRA to adjust NPC response generation (sentiment-aware).

### 4. **GhostInjector (plato-torch) ↔ holodeck-rust**
- **Gap:** Ghost tiles (P0 lessons from dead agents) are not injected into holodeck’s NPC memory.
- **Impact:** NPCs cannot warn players about historical failure modes.
- **Opportunity:** Implement a `GhostMemory` system in holodeck that queries plato-torch’s ghost tile store.
