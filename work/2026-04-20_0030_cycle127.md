# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 127  
**Phase:** 4 — Build & Test  
**Status:** Mapping current connections and identifying integration points.

Based on reading the actual files in the fleet repos, here is the current state of integration between **plato-torch**, **fleet-simulator**, **holodeck-rust**, and **plato-ensign**.

---

## 1. **plato-torch** → **fleet-simulator**
**Connection:** **Strong, bidirectional integration.**  
- **plato-torch** includes `fleet_simulator` as a dependency in its `Cargo.toml`.  
- **fleet-simulator** is used as a library to generate training data for PLATO rooms.  
- The `DeadbandRoom` preset in plato-torch imports `fleet_simulator::agent::Agent` and uses it to simulate agent behavior for tile generation.  
- **Data flow:** Fleet simulator agents produce interaction logs → processed into training tiles → stored in PLATO room memory.  
- **Integration point:** `plato-torch/src/rooms/deadband_room.rs` directly calls fleet-simulator's agent simulation methods.

**Gap:** None observed. This is a working, documented integration.

---

## 2. **plato-torch** → **holodeck-rust**
**Connection:** **Weak, indirect.**  
- No direct dependency in `Cargo.toml`.  
- **holodeck-rust** is a standalone MUD server with sentiment-aware NPCs.  
- Potential integration is conceptual: both systems handle agent interactions and could share tile data, but no code wires them together yet.  
- **Possible link:** The `memory` module in plato-torch (which stores tiles) could be adapted to consume logs from holodeck-rust sessions.

**Gap:** **No live data pipeline.** Holodeck NPC interactions are not currently feeding into PLATO room training. This is a missed opportunity for sentiment-rich training tiles.

---

## 3. **plato-torch** → **plato-ensign**
**Connection:** **Strong, export-oriented.**  
- **plato-ensign** is built as a companion tool to plato-torch.  
- It reads the **room experience** (accumulated tiles) from a PLATO room and exports them as LoRA adapters.  
- Integration is via file I/O: plato-ensign expects a room memory directory (e.g., `rooms/deadband_room/memory/`) and processes the `.json` tile files.  
- **Data flow:** PLATO room tiles → plato-ensign training pipeline → LoRA adapter output.

**Gap:** The export is **manual/offline**—requires running plato-ensign as a separate step. No automatic trigger when a room reaches sufficient tile count.

---

## 4. **fleet-simulator** → **holodeck-rust**
**Connection:** **None observed.**  
- These are parallel simulation systems.  
- fleet-simulator focuses on multi-agent fleet dynamics; holodeck-rust focuses on interactive NPC narratives.  
- No shared code or data interchange.

**Gap:** **Complete isolation.** Could be integrated if holodeck NPCs were added as "agents" in fleet-simulator scenarios, or if fleet-simulator events drove holodeck world state.

---

## 5. **fleet-simulator** → **plato-ensign**
**Connection:** **Indirect via plato-torch.**  
- fleet-simulator feeds plato-torch with training data.  
- plato-torch rooms store that data as tiles.  
- plato-ensign can later export those tiles.  
- No direct link between fleet-simulator and plato-ensign.

**Gap:** None required—this is a reasonable pipeline.

---

## 6. **holodeck-rust** → **plato-ensign**
**Connection:** **None.**  
- holodeck-rust has its own interaction logs but no mechanism to format them as PLATO tiles.  
- plato-ensign only reads from plato-torch room memory.

**Gap:** **Missing adapter.** A "holodeck-to-tile" converter would be needed to let holodeck sessions train ensigns.

---

## P0/P1/P2 Analysis

**P0 (Negative Space):**  
- Do not assume holodeck-rust is already integrated.  
- Do not modify plato-torch ↔ fleet-simulator link—it works.

**P1 (Safe Channels):**  
- Build a **holodeck-to-tile adapter** to format NPC interactions as PLATO tiles.  
- Add a **tile monitor** in plato-torch to auto-trigger plato-ensign when a room is "ready."  
- Explore adding holodeck as an agent type in fleet-simulator.

**P2 (Optimization):**  
- Stream tile data directly from holodeck to plato-torch memory.  
- Create a unified "simulation bus" where fleet-simulator and holodeck events can both feed rooms.
