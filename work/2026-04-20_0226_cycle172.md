# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 172

Based on reading the actual files in the fleet repos, here is the current state of integration between **plato-torch**, **fleet-simulator**, **holodeck-rust**, and **plato-ensign**.

## 1. Verified Connections (What's Connected)

### plato-torch ↔ fleet-simulator
- **Connection:** Direct integration via `fleet-simulator` as a data source for training rooms.
- **Evidence:**  
  - `plato-torch/rooms/deadband_room.py` imports `from fleet_simulator.simulator import FleetSimulator`.  
  - The `DeadbandRoom` class initializes a `FleetSimulator` instance (`self.simulator`).  
  - The room's `step()` method calls `self.simulator.step()` to generate simulation state used for tile creation.  
- **Data Flow:** Simulation state → room observations → training tiles.  
- **Status:** **Active and functional.**

### plato-torch ↔ plato-ensign
- **Connection:** Optional export pathway; rooms can be packaged as LoRA adapters.  
- **Evidence:**  
  - `plato-ensign/export.py` provides `export_room_to_lora()` function.  
  - `plato-torch/README.md` states: "Rooms train from accumulated interaction tiles. Ensigns are optional exports of room experience."  
  - `plato-torch/rooms/base_room.py` includes a `save()` method that could be extended for ensign export.  
- **Data Flow:** Trained room state → LoRA weights file.  
- **Status:** **Designed but not automatically wired.** Requires explicit call to export function.

### fleet-simulator → holodeck-rust (Potential)
- **Connection:** Fleet simulation could provide NPC sentiment or world state to holodeck.  
- **Evidence:**  
  - `holodeck-rust/src/npc.rs` defines `SentimentAwareNPC` struct with `update_sentiment()` method.  
  - `fleet-simulator/simulator.py` generates agent states with metrics (efficiency, risk).  
  - No direct import found yet, but architecture suggests simulation data could fuel NPC behavior.  
- **Data Flow:** Simulation agent metrics → NPC sentiment values.  
- **Status:** **Not yet implemented.** Identified as an integration opportunity.

## 2. Integration Gaps (What's Not Connected)

### holodeck-rust ↔ plato-torch
- **Gap:** No direct link between the MUD server and training rooms.  
- **Evidence:**  
  - `holodeck-rust` is a standalone Rust binary (`src/main.rs`).  
  - `plato-torch` has no imports of `holodeck_rust` or references to MUD protocols.  
  - NPC interactions in holodeck are not currently captured as training tiles.  
- **Opportunity:** Connect holodeck event stream to a dedicated training room (e.g., `HolodeckRoom`).

### plato-ensign → holodeck-rust
- **Gap:** LoRA adapters from ensign are not loaded into holodeck NPCs.  
- **Evidence:**  
  - `plato-ensign` exports `.safetensors` files.  
  - `holodeck-rust` does not currently load external model weights; NPC logic is hardcoded in Rust.  
- **Opportunity:** Enable dynamic NPC personality injection via LoRA adapters.

### fleet-simulator → plato-ensign
- **Gap:** Simulation data not directly used to create ensigns.  
- **Evidence:**  
  - `plato-ensign` exports room experience, not raw simulation data.  
  - `fleet-simulator` output is consumed by rooms, not by ensign export directly.  
- **Opportunity:** Create a `SimulationRoom` that trains exclusively on simulator data, then export as ensign.

### GhostInjector (Missing Component)
- **Gap:** Referenced in task backlog ("Wire GhostInjector into holodeck") but no corresponding repo found.  
- **Evidence:**  
  - `fleet-knowledge/FLEET-CONTEXT.md` mentions "Ghost tiles: dead agents' lessons persist as P0 knowledge."  
  - No `GhostInjector` code in scanned repos.  
- **Opportunity:** Build a service that reads ghost tiles and injects them as NPC dialogue or room observations.
