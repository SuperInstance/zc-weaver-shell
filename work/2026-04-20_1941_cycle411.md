# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 411  
**Phase:** 4 — Build & Test  
**Status:** Direct file inspection of fleet repositories. Focus on actual imports, configuration references, and shared data structures.

## 1. plato-torch → fleet-simulator
**Connection:** **YES** — Direct import and data flow.  
**Evidence:**  
- `plato-torch/src/rooms/deadband_room.py` imports `from fleet_simulator import FleetSimulator`.  
- `DeadbandRoom` initializes a `FleetSimulator` instance (`self.simulator = FleetSimulator()`).  
- The room calls `self.simulator.step()` during its training loop to generate simulation data.  
- Simulation state (agent positions, actions) is logged and can be converted into training tiles.

**Integration Gap:**  
- No observed two-way control: the simulator runs as a data source but doesn’t receive commands back from the room mid‑step.  
- Configuration is hard‑coded; no external config file mapping simulator parameters to room presets.

## 2. plato-torch → holodeck-rust
**Connection:** **NO** — No import or direct reference in code.  
**Evidence:**  
- Searched `plato-torch/` for “holodeck”, “mud”, “sentiment”, “npc”. No matches.  
- `holodeck-rust/` is a standalone Rust MUD server; no Python bindings or HTTP API mentioned in its `README.md`.  

**Integration Gap:**  
- No bridge between Python training rooms and Rust MUD server.  
- Potential channel: both could produce interaction tiles, but currently no shared tile format or ingestion pipeline.

## 3. plato-torch → plato-ensign
**Connection:** **YES** — Export pathway defined.  
**Evidence:**  
- `plato-torch/src/export/ensign_exporter.py` exists.  
- `DeadbandRoom` has an `export_to_ensign()` method that calls the exporter.  
- Exporter uses `peft` to create a LoRA adapter from accumulated room tiles.  

**Integration Gap:**  
- Export is manual/optional; no automatic trigger based on tile count or performance threshold.  
- No visible feedback loop where an ensign is reloaded into the room for further training.

## 4. fleet-simulator → holodeck-rust
**Connection:** **NO** — No cross‑references.  
**Evidence:**  
- `fleet-simulator/` is pure Python, no Rust interop.  
- `holodeck-rust/` does not mention simulation or fleet agents.  

**Integration Gap:**  
- Separate domains: simulator is for training‑data generation; holodeck is for interactive narrative. No shared agent or environment model.

## 5. fleet-simulator → plato-ensign
**Connection:** **INDIRECT** — via plato‑torch.  
**Evidence:**  
- Simulator data becomes tiles in `DeadbandRoom`.  
- Those tiles can be exported to an ensign.  
- No direct simulator‑to‑ensign pipeline.

**Integration Gap:**  
- Simulator could theoretically produce tiles in ensign‑ready format, but currently requires the room as an intermediary.

## 6. holodeck-rust → plato-ensign
**Connection:** **NO** — No references.  
**Evidence:**  
- `holodeck-rust/` outputs JSON logs of NPC interactions.  
- `plato-ensign` expects PyTorch state‑dict format.  

**Integration Gap:**  
- No adapter to convert holodeck sentiment/event logs into training tiles for ensign export.

## 7. Ghost Tiles & Mirror Play
