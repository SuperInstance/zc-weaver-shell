## 1. plato-torch → fleet-simulator
**Connection:** **Direct, via import**  
- `plato-torch/src/rooms/deadband_room.py` imports `fleet_simulator` (line 9)  
- Uses `fleet_simulator.Simulator` class to generate training data  
- Simulator provides `generate_scenario()` method that returns a dict with `state`, `actions`, `rewards`  
- DeadbandRoom calls `simulator.generate_scenario()` during `_generate_training_example()`  
- **Data flow:** Simulator → Room → Training tile  

**Gap:**  
- No reverse dependency (fleet-simulator doesn’t import plato-torch)  
- Simulator is standalone; could be used by other rooms but currently only DeadbandRoom uses it  
- No shared configuration file; simulator parameters hardcoded in room  

## 2. plato-torch → plato-ensign  
**Connection:** **Direct, via export pipeline**  
- `plato-torch/src/export/ensign_exporter.py` exists  
- Exporter reads room checkpoints (`room.pt`) and converts to LoRA adapters  
- Uses `peft` library to extract LoRA weights  
- Command-line tool: `python -m plato_torch.export.ensign_exporter --room path/to/room.pt --output adapter.safetensors`  
- **Data flow:** Room training → checkpoint → exporter → LoRA file  

**Gap:**  
- Not automatically triggered; manual export step  
- No integration with training loop (post‑training only)  
- No validation that exported adapter works with target model  

## 3. holodeck-rust → (any)  
**Connection:** **None found**  
- `holodeck-rust/` is a standalone MUD server written in Rust  
- No imports of plato-torch, fleet-simulator, or plato-ensign  
- Has its own `src/npc/sentiment.rs` for NPC emotion system  
- Listens on TCP port 4000, uses custom binary protocol  
- **Data flow:** Client connections → game state updates → sentiment‑aware responses  

**Gap:**  
- No integration with fleet training pipeline  
- NPC sentiment system could be training target, but no connector exists  
- No shared data structures with other repos  

## 4. fleet-simulator → holodeck-rust  
**Connection:** **None found**  
- `fleet-simulator/src/lib.rs` defines `Simulator` struct with `scenario` module  
- No mention of holodeck, MUD, or TCP  
- Simulator generates abstract state/action/reward tuples, not game‑specific data  

**Gap:**  
- Simulator could generate holodeck scenarios (e.g., NPC dialogues, player actions) but doesn’t  
- No adapter to convert simulator output to holodeck protocol  

## 5. plato-ensign → holodeck-rust  
**Connection:** **None found**  
- `plato-ensign/` is Python‑based LoRA exporter  
- No Rust dependencies, no FFI bindings  
- Output is `.safetensors` file, not directly loadable by holodeck  

**Gap:**  
- Could embed adapted NPC behavior in holodeck, but no loader exists  
- Missing: `holodeck‑rust/src/ai/lora_adapter.rs`  

## 6. GhostInjector (mentioned in backlog)  
**Location:** `plato-torch/src/data/ghost_injector.py`  
**Status:** Standalone module, not integrated

- **Not wired into:**  
  - DeadbandRoom (could augment training data)
