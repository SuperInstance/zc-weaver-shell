# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 155  
**Phase:** 4 — Build & Test  
**Status:** Mapping current connections and identifying integration points.

Based on reading the actual files in the fleet repos, here is the current state of integration between **plato-torch**, **fleet-simulator**, **holodeck-rust**, and **plato-ensign**.

---

## 1. plato-torch → fleet-simulator
**Connection:** ✅ **Connected**  
**How:**  
- `plato-torch/rooms/DeadbandRoom.py` imports `fleet_simulator` as a module.  
- The `DeadbandRoom` class uses `fleet_simulator.Simulator` to generate training data.  
- Simulator runs in a separate thread, feeding `(state, action, reward, next_state)` tuples into the room’s replay buffer.  
- This is a **live data pipeline** — not a one-time export.

**Integration Quality:**  
- Direct Python import.  
- Simulator config is passed via `room_config`.  
- Room pulls simulator states and converts them to training tiles.

**Gaps:**  
- No bidirectional control — room cannot *steer* the simulator mid‑episode.  
- Simulator runs fixed scenarios; no dynamic scenario injection from the room.

---

## 2. plato-torch → holodeck-rust
**Connection:** ❌ **Not directly connected**  
**How:**  
- `holodeck-rust` is a standalone MUD server written in Rust.  
- `plato-torch` has no Rust bindings or HTTP client to talk to holodeck.  
- No imports, no API calls, no shared data structures.

**Potential Integration Point:**  
- `holodeck-rust` could expose a WebSocket or REST API for sentiment‑aware NPC interactions.  
- `plato-torch` rooms could query holodeck for NPC dialogue/state to enrich training tiles.

**Blockers:**  
- Language barrier (Python ↔ Rust).  
- No defined protocol between them in the current code.

---

## 3. plato-torch → plato-ensign
**Connection:** ✅ **Connected**  
**How:**  
- `plato-ensign` is a CLI tool that loads a trained room and exports a LoRA adapter.  
- It imports `plato_torch` and calls `room.export_adapter()`.  
- Works on any room that inherits from `BaseRoom` and implements the export method.

**Integration Quality:**  
- Clean separation — ensign is a post‑training export step.  
- Uses PyTorch’s `state_dict` serialization.

**Gaps:**  
- Export is manual (CLI), not automatic after training.  
- No continuous integration — rooms don’t auto‑export ensigns when they reach a performance threshold.

---

## 4. fleet-simulator → holodeck-rust
**Connection:** ❌ **Not connected**  
**How:**  
- `fleet-simulator` is a Python‑based multi‑agent simulation.  
- `holodeck-rust` is a text‑based MUD.  
- No shared communication or data exchange.

**Potential Integration Point:**  
- Simulator agents could “enter” the MUD as players, using holodeck as an environment layer.  
- Sentiment‑aware NPCs in holodeck could influence simulator agent behavior.

**Blockers:**  
- No API from holodeck for external agent control.  
- Simulator would need to parse MUD text output (non‑trivial).

---

## 5. fleet-simulator → plato-ensign
**Connection:** ❌ **Not connected**  
**How:**  
- `plato-ensign` only works on `plato-torch` rooms, not directly on simulator data.  
- Simulator data must first pass through a room to become tiles before ensign can export.

**Integration Quality:**  
- Indirect connection via plato‑torch room.

**Gaps:**  
- No direct ensign export from raw simulator trajectories.

---

## 6. holodeck-rust → plato-ensign
**Connection:** ❌ **Not connected**  
**How:**  
- `holodeck-rust` produces game logs, not PyTorch state dicts.  
- `plato-ensign` expects a trained room, not Rust game state.

**Potential Integration Point:**  
- Could train a room on holodeck interaction logs, then export an ensign.  
- Would require a log‑to‑tile conversion pipeline.

**Blockers:**  
- No existing pipeline from holodeck logs to plato‑torch tiles.

## Identified Integration Gaps (P1 Safe Channels)

1. **plato-torch ↔ holodeck-rust**  
   - Build a Rust‑Python bridge (e.g., PyO3) or HTTP API.  
   - Define a protocol for NPC interaction tiles.

2. **fleet-simulator → holodeck-rust**  
   - Let simulator agents act as MUD clients.  
   - Use holodeck as a richer environment layer.

3. **Auto‑export ensigns**  
   - Add a callback in plato‑torch to auto‑export when validation loss plateaus.

4. **Bidirectional room‑simulator control**  
   - Allow room to send scenario parameters back to simulator.
