## 2. Identified Integration Gaps (P1 Channels to Explore)

**Gap 1: holodeck-rust â†’ plato-torch Tile Pipeline**
- **Problem:** No documented mechanism to stream NPC/player interactions from the holodeck MUD server into a `plato-torch` room as formatted tiles.
- **Actionable Integration:** Build a `LogScraper` or `TileBridge` service that watches `holodeck-rust` logs, parses interactions (player sentiment, NPC response, outcome), and POSTs them to a `plato-torch` room's ingestion endpoint.
- **P1 Priority:** High. Creates a rich, automated source of training data from a dynamic environment.

**Gap 2: fleet-simulator â†’ plato-torch Data Bridge**
- **Problem:** The simulator runs and generates data, but there's no pipe to turn simulation events into training tiles.
- **Actionable Integration:** Create a `SimulationAdapter` in `fleet-simulator` that emits events in the `plato-torch` tile JSON schema. Could connect via a message queue (Redis) or direct HTTP.
- **P1 Priority:** High. Closes the loop between simulation and training.

**Gap 3: plato-ensign â†’ Agent Runtime Feedback Loop**
- **Problem:** New ensigns (LoRAs) are created but not automatically deployed back into the active fleet (e.g., to `cudaclaw` or `flux-runtime` agents).
- **Actionable Integration:** Extend `plato-ensign` with a `Deployer` module that, upon successful export, pushes the adapter to a model registry and notifies agent runtimes to hot-swap.
- **P1 Priority:** Medium-High. Essential for a self-improving fleet.

**Gap 4: constraint-theory-core â†’ holodeck-rust** (Noted in Backlog)
- **Problem:** `constraint-theory-core` provides geometric snapping (likely for spatial reasoning). `holodeck-rust` is a textual MUD. Could spatial constraints apply to narrative or puzzle generation?
- **Actionable Integration:** If holodeck implements any spatial puzzles or mapping, integrate the Rust library to manage room connections, puzzle states, or NPC pathing within constraints.
- **P1 Priority:** Medium. Requires clarifying holodeck's needs.
