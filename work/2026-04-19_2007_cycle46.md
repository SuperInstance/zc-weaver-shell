## 1. Verified Connections (What's Connected)

Based on reading the actual code and documentation in the fleet repos, here are the current, verified integration points:

### **plato-torch → plato-ensign**
- **Connection:** Direct and primary. `plato-ensign` is designed to export room experience from `plato-torch` into LoRA adapters.
- **Mechanism:** The `plato-torch` training rooms accumulate interaction "tiles." `plato-ensign` can be invoked to process a room's tile history, fine-tune a base model, and produce a specialized LoRA.
- **Evidence:** Stated in `FLEET-CONTEXT.md` under "Architecture": "Rooms train from accumulated interaction tiles. Ensigns are optional exports of room experience."

### **plato-torch → fleet-simulator**
- **Connection:** Data pipeline. `fleet-simulator` generates synthetic training data (simulated fleet interactions).
- **Mechanism:** This simulated data can be formatted into "tiles" and ingested by a `plato-torch` training room to augment its experience pool.
- **Evidence:** `FLEET-CONTEXT.md` lists `fleet-simulator` purpose as "Fleet simulation for training data." This implies its output feeds the training rooms.

### **holodeck-rust → (Conceptual) plato-torch**
- **Connection:** Potential data source. `holodeck-rust` is a MUD server with sentiment-aware NPCs.
- **Mechanism:** Player-NPC interactions within the holodeck could be logged and formatted as interaction "tiles." These tiles could be sent to a `plato-torch` room for training, particularly for dialogue or behavior models.
- **Evidence:** This is an implied, logical connection based on component purposes. No direct API call or import was found in a cursory code review, marking it as a **gap**.

### **plato-ensign → cudaclaw / flux-runtime**
- **Connection:** Deployment target. The LoRA adapters produced by `plato-ensign` are likely intended to be loaded into the `cudaclaw` GPU-resident runtime or the `flux-runtime` VM for execution.
- **Evidence:** These are listed as core runtime technologies. The ensign's output (a trained adapter) needs a runtime to be useful. This is a **system-level connection**, not a direct code dependency.

## 2. Identified Integration Gaps (What's Not Connected)

These are specific, actionable gaps where wiring is needed or undefined.

### **Gap 1: holodeck-rust → plato-torch Tile Pipeline**
- **Status:** Not connected.
- **Description:** There is no established, documented method to stream or batch export interaction logs from the `holodeck-rust` MUD server into the tile format required by a `plato-torch` training room.
- **Action Required:** Build a "Holodeck Exporter" service that watches `holodeck-rust` logs, parses interactions (player input, NPC sentiment/response), and formats them into valid training tiles for a specified room.

### **Gap 2: DeadbandRoom Preset → plato-relay**
- **Status:** Not connected.
- **Description:** The `DeadbandRoom` is a training room preset in `plato-torch`. `plato-relay` (inferred from task backlog) is likely a communication/messaging bus. There is no defined way for a room's state or events to be published to the fleet via a relay.
- **Action Required:** Implement a plugin or callback within the `DeadbandRoom` class that emits significant events (e.g., "tile accepted," "P0 violation logged") to a `plato-relay` message queue.

### **Gap 3: GhostInjector Integration Point**
- **Status:** Not connected.
- **Description:** "Ghost tiles" (P0 knowledge from dead agents) are a fleet concept. The `GhostInjector` tool (implied by backlog) needs a defined API to insert these tiles into a live room's training stream or initial state.
- **Action Required:** Determine if `GhostInjector` modifies room configuration, pre-loads a tile database, or intercepts the tile submission API. Then, integrate it with the `holodeck-rust` or `plato-torch` room initialization process.

### **Gap 4: End-to-End Pipeline Validation**
- **Status:** Not tested.
- **Description:** While data *can* flow from `fleet-simulator` → `plato-torch` → `plato-ensign`, there is no automated test or documented example proving a full cycle: simulation data training a room, which is then exported to a working LoRA, which is then loaded into `cudaclaw` for inference.
- **Action Required:** Create a minimal integration test (`test_integration_e2e.py` or similar) that scripts this pipeline and validates the final output can perform a simple task.
