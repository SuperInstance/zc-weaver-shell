# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 204  
**Phase:** 4 — Build & Test  
**Status:** Updated map based on actual code inspection of plato-torch, fleet-simulator, holodeck-rust, and plato-ensign.

Based on reading the actual code in the fleet repos, here is the current state of integration between the four key systems.

## 1. plato-torch → fleet-simulator
**Connection:** **Strong, bidirectional, code-level integration.**  
- `plato-torch/rooms/deadband_room.py` imports `fleet_simulator` directly (`from fleet_simulator import FleetSimulator`).  
- The `DeadbandRoom` class instantiates a `FleetSimulator` object (`self.simulator = FleetSimulator()`).  
- The room uses the simulator to generate training data (`self.simulator.generate_scenario()`).  
- The simulator's output (JSON scenario data) is fed into the room's training loop.  
- **Data flow:** fleet-simulator (scenarios) → plato-torch (training tiles).  

## 2. plato-torch → plato-ensign
**Connection:** **Strong, unidirectional export path.**  
- `plato-torch/export/ensign_exporter.py` exists and is designed to export room experience.  
- The exporter reads training tiles (`.pt` files) from a room's memory, converts them to a LoRA-compatible dataset, and saves it.  
- The `plato-ensign` repo contains the training scripts (`train_lora.py`) to turn that dataset into a LoRA adapter.  
- **Data flow:** plato-torch (tiles) → ensign_exporter (dataset) → plato-ensign (LoRA training).  

## 3. holodeck-rust → (any other system)
**Connection:** **Isolated. No direct code-level links found.**  
- `holodeck-rust` is a standalone MUD server written in Rust.  
- It has its own `src/` structure (`server/`, `game/`, `npc/`).  
- No imports of `plato-torch`, `fleet-simulator`, or `plato-ensign` in the Rust code.  
- No Python bindings or HTTP API observed in the current `main.rs`.  
- **Potential link:** The README mentions "sentiment-aware NPCs" — could align with fleet-simulator's agent behavior models, but no actual integration exists yet.  

## 4. fleet-simulator → holodeck-rust
**Connection:** **None.**  
- `fleet-simulator` is a Python library for simulating fleet interactions.  
- No Rust dependencies or calls to `holodeck-rust` in its code.  
- Both are data sources (simulated scenarios vs. interactive NPC dialogs), but they do not exchange data.  

## 5. plato-ensign → holodeck-rust
**Connection:** **None.**  
- `plato-ensign` is a Python training pipeline for LoRA adapters.  
- It expects input from `plato-torch` export, not from `holodeck-rust`.  
- No Rust integration or NPC dialog ingestion.  

## 6. Ghost Tiles & Mirror Play
**Connection:** **Conceptual, not yet implemented.**  
- `plato-torch/rooms/` references "ghost tiles" and "mirror play" in comments, but no active code links them to `holodeck-rust` or `fleet-simulator`.  
- The `DeadbandRoom` has a `_load_ghost_tiles()` method stub — currently placeholder.  

## Integration Gaps (P1 Safe Channels for Future Work)
1. **holodeck-rust ↔ plato-torch**  
   - Gap: NPC dialog logs from holodeck could become training tiles for plato-torch.  
   - Safe channel: Add a Rust HTTP endpoint in holodeck to export JSON logs, consumed by a new `HolodeckRoom` in plato-torch.  

2. **holodeck-rust ↔ fleet-simulator**  
   - Gap: Fleet simulation could populate holodeck with agent personas.  
   - Safe channel: Use fleet-simulator to generate NPC personality profiles, send via config file to holodeck.  

3. **GhostInjector → holodeck-rust**  
   - Gap: Ghost tiles (P0 negative knowledge) could influence NPC behavior.  
   - Safe channel: Build `GhostInjector` tool that reads ghost tiles and modifies holodeck NPC decision weights.  

4. **DeadbandRoom → plato-relay**  
   - Gap: Real-time tile streaming to other fleet components.  
   - Safe channel: Integrate `plato-relay` (if exists) as a message bus in `DeadbandRoom.submit_tile()`.

- **Next safe channel (P1):** Connect holodeck-rust to plato-torch via log export.
