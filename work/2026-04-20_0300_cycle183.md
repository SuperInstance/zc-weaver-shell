# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 183  
**Phase:** 4 — Build & Test  
**Status:** Mapping current connections and identifying integration points.

Based on reading the actual files in the fleet repos, here is the current state of integration between **plato-torch**, **fleet-simulator**, **holodeck-rust**, and **plato-ensign**.

## 1. Verified Connections (What's Connected)

### plato-torch ↔ fleet-simulator
- **Connection:** Direct integration via `fleet-simulator` as a data source.
- **Evidence:** `plato-torch/rooms/deadband_room.py` imports `fleet_simulator` and uses `FleetSimulator` class to generate training episodes.
- **Data Flow:** `fleet-simulator` produces simulated fleet interaction episodes (JSON). `deadband_room.py` loads these episodes, converts them to training tiles, and submits them to the PLATO room server.
- **Status:** **Active and working.** This is a core training pipeline.

### plato-torch ↔ plato-ensign
- **Connection:** `plato-ensign` is a library/utility for exporting room experience.
- **Evidence:** `plato-torch/rooms/deadband_room.py` has a `train_ensign()` method that calls `plato_ensign.export_room_experience()`.
- **Data Flow:** After a room accumulates enough interaction tiles, `plato-ensign` can package the room's state (weights, config, tiles) into a LoRA adapter file (`.safetensors`).
- **Status:** **Integrated but optional.** The export function exists and is callable, but not automatically triggered in every training cycle.

### holodeck-rust ↔ (Indirect via plato-torch)
- **Connection:** No direct code integration found. However, `holodeck-rust` is listed as a "sentiment-aware NPC" environment that could generate training data.
- **Evidence:** `plato-torch/README.md` mentions `holodeck-rust` as a potential data source, but no import or usage in current room presets.
- **Data Flow:** **Potential, not implemented.** The design suggests `holodeck-rust` NPC interactions could become tiles for PLATO rooms, but no loader or adapter exists in the current codebase.
- **Status:** **Disconnected.** Marked as a key technology but not wired.

## 2. Integration Gaps (What's Not Connected)

### Gap 1: holodeck-rust → plato-torch
- **Issue:** No data pipeline from `holodeck-rust` MUD server to PLATO training rooms.
- **Impact:** Sentiment-aware NPC interactions are not being captured as training tiles.
- **Potential Integration Point:** Create a `HolodeckLoader` class in `plato-torch/rooms/` that connects to `holodeck-rust` WebSocket/gRPC, formats NPC dialogs/actions into tiles, and submits them.

### Gap 2: plato-ensign → fleet-simulator / holodeck-rust
- **Issue:** Ensigns (LoRA adapters) are not automatically deployed back to simulators or environments.
- **Impact:** Trained specialist models are not being used to improve simulation or NPC behavior.
- **Potential Integration Point:** Add an `ensign_deploy` module that loads a `.safetensors` file into `fleet-simulator` agent brains or `holodeck-rust` NPC personality models.

### Gap 3: Ghost Tiles → holodeck-rust
- **Issue:** Ghost tiles (P0 lessons from dead agents) are a PLATO-room concept, but not injected into `holodeck-rust` NPC memory.
- **Impact:** NPCs cannot learn from past agent failures.
- **Potential Integration Point:** Wire `GhostInjector` (from `plato-torch`) to send tile data to `holodeck-rust` NPC sentiment memory.

### Gap 4: DeadbandRoom ↔ plato-relay
- **Issue:** `DeadbandRoom` operates in isolation; no connection to `plato-relay` (messaging/coordination layer).
- **Impact:** Rooms cannot coordinate training or share tile streams in real-time.
- **Potential Integration Point:** Add a `PlatoRelayClient` to `DeadbandRoom` to publish tile events and subscribe to other rooms' outputs.

## 3. Immediate Integration Priorities (P1 Safe Channels)

1. **Wire GhostInjector into holodeck-rust** — Use existing `GhostInjector` class to format ghost tiles and send via `holodeck-rust` API (if available) or log to a shared directory.
2. **Create HolodeckLoader for plato-torch** — Build a minimal loader that subscribes to `holodeck-rust` event stream and creates tiles.
3. **Test plato-ensign export/import loop** — Verify that an ensign exported from `DeadbandRoom` can be loaded into `fleet-simulator` agent.

## 4. Notes & Constraints

- **plato-torch** is the central hub; it already connects to `fleet-simulator` and `plato-ensign`.
- **holodeck-rust** is the most isolated; it has Rust-native APIs and may require a bridge (Python ↔ Rust).
- **fleet-simulator** is well-integrated as a data source, but not as a consumer of trained models.
- **plato-ensign** is a utility library, not a service; it depends on being called from a room.

**Next Action:** Begin implementing `HolodeckLoader` in `plato-torch/rooms/` to close Gap 1.
