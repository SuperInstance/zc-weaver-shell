# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 160  
**Phase:** 4 — Build & Test  
**Status:** Mapping current connections and identifying integration points.

Based on reading the actual files in the fleet repos, here is the current state of integration between the four core technologies: **plato-torch**, **fleet-simulator**, **holodeck-rust**, and **plato-ensign**.

## 1. Verified Connections (What IS Connected)

### plato-torch ↔ plato-ensign (Strong, Direct)
- **Connection:** Direct code dependency and data flow.
- **Evidence:** `plato-torch/src/ensign_exporter.rs` exists. It exports room experience (tiles) into LoRA adapters via `plato-ensign` library calls.
- **Data Flow:** Training rooms in plato-torch accumulate interaction tiles → passed to ensign exporter → generates `.safetensors` LoRA files.
- **Status:** This is the primary "training to export" pipeline documented in fleet architecture.

### plato-torch ↔ fleet-simulator (Indirect, Planned)
- **Connection:** Shared data schema and potential future integration.
- **Evidence:** `fleet-simulator/src/data/tile_schema.rs` defines `TrainingTile` struct that matches the tile format used in plato-torch rooms.
- **Current Use:** fleet-simulator can generate synthetic training tiles that *could* be consumed by plato-torch rooms, but no import code exists yet.
- **Status:** Data compatibility is established; runtime integration is not implemented.

### holodeck-rust ↔ (None Yet) (Isolated)
- **Connection:** No direct code links found to the other three repos.
- **Evidence:** `holodeck-rust/Cargo.toml` shows no dependencies on plato-torch, fleet-simulator, or plato-ensign. It is a standalone MUD server.
- **Note:** The README mentions "sentiment-aware NPCs" and potential for "ghost tile injection," but no current integration points are wired.

## 2. Integration Gaps (What Is NOT Connected)

### Gap 1: holodeck-rust ↔ plato-torch
- **Missing:** No way for holodeck NPC interactions to become training tiles in a plato-torch room.
- **Potential Integration Point:** A `HolodeckTileInjector` service that watches holodeck event logs, formats them as tiles, and submits them to a plato-torch room via its API.

### Gap 2: fleet-simulator → plato-torch (Runtime Feed)
- **Missing:** While data schemas align, there is no running service that pipes synthetic tiles from the simulator into live training rooms.
- **Potential Integration Point:** A `SyntheticTileFeeder` that uses the plato-torch room API to inject simulator-generated tiles, useful for bootstrapping new rooms.

### Gap 3: plato-ensign → holodeck-rust
- **Missing:** No mechanism to load a trained ensign (LoRA) into holodeck NPCs to alter their behavior.
- **Potential Integration Point:** An `EnsignNPCAdapter` in holodeck that loads `.safetensors` and adjusts NPC response generation based on the fine-tuned model.

### Gap 4: Deadband Protocol ↔ All Systems
- **Missing:** The Deadband Protocol (P0/P1/P2) is documented but not codified as a shared library or service that these systems can call.
- **Potential Integration Point:** A `deadband-core` Rust crate with decision functions, integrated into plato-torch room logic, fleet-simulator agent logic, and holodeck NPC logic.
