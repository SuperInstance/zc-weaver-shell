# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 129  
**Phase:** 4 — Build & Test  
**Status:** Mapping current connections and identifying integration points.

Based on reading the actual files in the fleet repos, here is the current integration state between **plato-torch**, **fleet-simulator**, **holodeck-rust**, and **plato-ensign**.

---

## 1. plato-torch → fleet-simulator
**Connection:** ✅ **Connected**  
**How:**  
- `plato-torch/src/rooms/deadband_room.py` imports `fleet_simulator` as a module.  
- The `DeadbandRoom` class uses `fleet_simulator` to generate simulated fleet interaction data for training.  
- The simulator provides structured JSON logs that are parsed into training tiles.  
**Integration Point:** `DeadbandRoom._generate_simulation_tiles()` calls the simulator’s API.

**Gap:**  
- The connection is one‑way: plato‑torch consumes simulator output, but the simulator does not receive feedback from trained models.  
- No real‑time coupling; runs as offline data generation.

---

## 2. plato-torch → plato-ensign
**Connection:** ✅ **Connected**  
**How:**  
- `plato-torch/src/export/ensign_exporter.py` provides a `export_to_ensign()` function.  
- It converts a trained room’s experience (accumulated tiles) into a LoRA adapter via `plato-ensign`’s training scripts.  
**Integration Point:** The exporter calls `plato_ensign.train_lora()` with tile data and hyperparameters.

**Gap:**  
- Export is manual/optional; not automatically triggered by room training.  
- No version tracking between room checkpoints and exported ensigns.

---

## 3. holodeck-rust → plato-torch
**Connection:** ⚠️ **Partial**  
**How:**  
- `holodeck-rust/src/npc/sentiment.rs` defines sentiment‑aware NPCs that could generate interaction tiles.  
- No direct import of plato‑torch in the Rust code, but the `Tile` struct in `holodeck-rust/src/protocol/tile.rs` matches the tile schema expected by plato‑torch.  
- Potential integration via file output or network socket (unimplemented).  
**Integration Point:** Hypothetical — NPC interactions could be logged as tiles and sent to a DeadbandRoom.

**Gap:**  
- No actual data flow between holodeck and plato‑torch in current code.  
- Tile schema alignment suggests planned integration, but no connector exists.

---

## 4. fleet-simulator → holodeck-rust
**Connection:** ❌ **Not connected**  
**How:**  
- `fleet-simulator` outputs JSON logs of simulated agent interactions.  
- `holodeck-rust` expects real‑time MUD commands and sentiment‑tagged dialogue.  
- No shared protocol or adapter between the two.

**Gap:**  
- Simulator data cannot drive holodeck NPCs.  
- Holodeck cannot replay simulator scenarios.

---

## 5. plato-ensign → holodeck-rust
**Connection:** ❌ **Not connected**  
**How:**  
- `plato-ensign` produces LoRA adapters (`.safetensors` files).  
- `holodeck-rust` has no model‑loading or inference runtime (no torch integration).  
- Ensigns are trained for text‑generation tasks, not directly usable in a Rust MUD.

**Gap:**  
- No mechanism to load an ensign into holodeck NPCs.  
- Missing `cudaclaw` or `flux‑runtime` bridge to run adapters in Rust.

---

## 6. fleet-simulator → plato-ensign
**Connection:** ⚠️ **Indirect**  
**How:**  
- Simulator data can be turned into tiles (`plato-torch`).  
- Tiles can be exported to an ensign (`plato-ensign`).  
- No direct simulator‑to‑ensign pipeline.

**Gap:**  
- Extra step required (room training) before ensign creation.  
- Simulator output format not directly compatible with ensign training.
