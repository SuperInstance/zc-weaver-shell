## 1. plato-torch
**Repo:** `plato-torch`  
**Purpose:** 25 training room presets, self-training rooms.  
**Key files:**
- `rooms/deadband_room.py` — Deadband Protocol training room.
- `rooms/__init__.py` — Exports room classes.
- `trainer/self_trainer.py` — Self‑training loop.
- `config/room_configs.yaml` — Room hyperparameters.

**Connections found:**
- **To plato‑ensign:** `plato‑torch/rooms/base_room.py` imports `from plato_ensign.lora_exporter import LoRAExporter`. Used in `export_ensign()` method.
- **To fleet‑simulator:** No direct import found in code. However, `config/room_configs.yaml` contains a key `simulator_data_path: "../fleet-simulator/data/trajectories/"`. This is a configuration‑level dependency.
- **To holodeck‑rust:** No import or config reference found.

**Integration notes:**  
plato‑torch is designed to export trained room experiences to plato‑ensign via a direct Python import. It expects fleet‑simulator data to be present at a relative path, but does not import the simulator as a module. No current link to holodeck‑rust.

## 2. fleet‑simulator
**Repo:** `fleet‑simulator`  
**Purpose:** Generate training trajectories for fleet agents.  
**Key files:**
- `src/simulator/core.rs` — Main simulation engine.
- `src/agents/mod.rs` — Agent definitions.
- `data/trajectories/` — Output directory for training data.

**Connections found:**
- **To plato‑torch:** No Rust → Python import possible. The connection is one‑way: plato‑torch reads the `data/trajectories/` directory.
- **To plato‑ensign:** No reference found.
- **To holodeck‑rust:** No import or config reference found.

**Integration notes:**  
fleet‑simulator is a standalone Rust binary that writes trajectory files. It is a data source for plato‑torch, but there is no feedback loop from plato‑torch back to the simulator.

## 6. Integration Gaps (P1 Safe Channels)
1. **holodeck‑rust ↔ plato‑torch** — No bridge exists. A safe channel could be a simple JSON log of NPC interactions written by holodeck and read by a new plato‑torch room preset.
2. **fleet‑simulator → plato‑ensign** — Simulator trajectories are not directly used to create LoRAs. Could add a pipeline to convert trajectories into ensign‑compatible formats.
3. **plato‑torch → fleet‑simulator** — No feedback loop. Trained agents cannot influence simulation. A safe channel: plato‑torch could write agent policies that the simulator loads.
4. **GhostInjector** — Mentioned in task backlog. Not found in any repo. Likely a planned component to inject “ghost tiles” (dead agents’ lessons) into holodeck‑rust NPCs.
