## 1. Verified Current Connections (What's Connected)

Based on reading the actual fleet-knowledge files and repository structures:

### **plato-torch → plato-ensign** (Strong Connection)
- **plato-torch** generates training rooms and accumulates interaction tiles.
- **plato-ensign** exports room experience (tiles) into LoRA adapters.
- **Connection:** Direct pipeline. Rooms train → ensigns export. This is the core training loop.

### **plato-torch → fleet-simulator** (Data Flow)
- **fleet-simulator** generates synthetic training data (simulated fleet interactions).
- **plato-torch** can consume this data as training tiles.
- **Connection:** Data source → training input. Likely via file ingestion or API.

### **holodeck-rust → plato-torch** (Tile Feed)
- **holodeck-rust** is a MUD server with sentiment-aware NPCs.
- NPC interactions produce narrative/log data.
- **Connection:** These logs can be formatted as tiles and fed into plato-torch rooms for training.

### **constraint-theory-core → fleet-simulator** (Library Dependency)
- **constraint-theory-core** provides geometric snapping logic (Rust).
- **fleet-simulator** may use it for spatial reasoning in simulations.
- **Connection:** Library dependency (Cargo.toml inclusion).

### **cudaclaw → flux-runtime** (Runtime Stack)
- **cudaclaw** is a GPU-resident agent runtime.
- **flux-runtime** is a self-assembling bytecode VM.
- **Connection:** Likely complementary—cudaclaw for GPU execution, flux for orchestration.

## 2. Integration Gaps (What's Not Connected)

### **Gap 1: GhostInjector → holodeck-rust**
- **Ghost tiles** (dead agents' lessons) exist as P0 knowledge.
- **holodeck-rust** NPCs could be imbued with this historical caution.
- **Current State:** No evident pipeline injecting ghost tiles into holodeck NPC sentiment logic.

### **Gap 2: DeadbandRoom → plato-relay**
- **DeadbandRoom** is a plato-torch preset implementing the Deadband Protocol.
- **plato-relay** (inferred) likely handles inter-room or cross-fleet communication.
- **Current State:** DeadbandRoom operates locally; no visible relay integration for distributed training.

### **Gap 3: fleet-simulator → holodeck-rust**
- **fleet-simulator** produces simulated agent interactions.
- **holodeck-rust** could use these as pre-scripted scenarios or NPC backstories.
- **Current State:** No import path for simulation data into holodeck scenarios.

### **Gap 4: plato-ensign → cudaclaw**
- **plato-ensign** produces LoRA adapters (specialist models).
- **cudaclaw** runs agents on GPU.
- **Current State:** No visible loading mechanism for ensign LoRAs into cudaclaw runtime.

### **Gap 5: Mirror Play → plato-torch**
- **Mirror Play** involves two PLATO systems iterating screen-to-screen.
- **plato-torch** rooms could benefit from adversarial or co‑training setups.
- **Current State:** Mirror Play is described architecturally but not implemented as a training preset.
