## 1. Verified Connections (What's Connected)

### plato-torch → fleet-simulator
- **Connection:** Direct import in training data pipeline.
- **Evidence:** `plato-torch/src/data/__init__.py` imports `from fleet_simulator import generate_scenario`.
- **Purpose:** Uses fleet-simulator to generate synthetic training scenarios (agent interactions, environment states) as data source for room training.
- **Data Flow:** Scenarios → Tile format → Room training batches.

### plato-torch → plato-ensign
- **Connection:** Export pipeline via explicit hooks.
- **Evidence:** `plato-torch/src/rooms/deadband_room.py` contains method `export_to_ensign()` that calls `plato_ensign.create_adapter()`.
- **Purpose:** After room training converges, the room's learned weights can be exported as a LoRA adapter file.
- **Data Flow:** Trained room state → LoRA adapter file (.safetensors).

### holodeck-rust → (indirect) plato-torch
- **Connection:** Shared tile format specification.
- **Evidence:** `holodeck-rust/src/protocol/tile.rs` defines `Tile` struct with fields (`id`, `agent_id`, `content`, `metadata`). This matches the tile schema expected by `plato-torch/src/data/tiles.py`.
- **Purpose:** NPC interactions in holodeck produce tiles that can, in principle, be ingested by plato-torch rooms. No direct import, but format compatibility suggests intended pipeline.
- **Data Flow:** NPC dialogue/actions → Tile JSON → Potential plato-torch ingestion.

## 2. Integration Gaps (What's Not Connected)

### fleet-simulator → holodeck-rust
- **Gap:** No import or reference found.
- **Inspection:** `fleet-simulator/src/lib.rs` has no mention of `holodeck`. `holodeck-rust` does not import fleet-simulator.
- **Potential Use:** Fleet simulation scenarios could drive holodeck environment generation or NPC behavior trees. Currently separate.

### plato-ensign → holodeck-rust
- **Gap:** No connection.
- **Inspection:** `plato-ensign/src/lib.rs` exports adapter creation but has no hooks for runtime integration with holodeck NPCs.
- **Potential Use:** Ensign adapters (specialist models) could be loaded into holodeck NPCs to give them specialized behavior. Not implemented.

### plato-torch → holodeck-rust (runtime)
- **Gap:** No live connection.
- **Inspection:** plato-torch training is offline/batch. Holodeck is a live MUD server. No socket or API connection observed in code.
- **Potential Use:** Trained rooms could serve as live "oracles" for holodeck NPCs (e.g., sentiment analysis, dialogue generation). Would require a service bridge.

### GhostInjector (concept) → Any System
- **Gap:** GhostInjector is referenced in fleet context as a way to inject "ghost tiles" (dead agents' lessons), but no implementation found in scanned repos.
- **Location:** Not in plato-torch, holodeck-rust, fleet-simulator, or plato-ensign main branches.
- **Potential Use:** P0 negative knowledge injection into training data. Unimplemented.
