# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 143  
**Phase:** 4 — Build & Test  
**Status:** Mapping current connections and identifying integration points.

Based on reading the actual files in the fleet repos, here is the current state of integration between **plato-torch**, **fleet-simulator**, **holodeck-rust**, and **plato-ensign**.

---

## 1. **plato-torch** → **fleet-simulator**
**Connection:** ✅ **Connected**  
**How:**  
- `plato-torch/src/rooms/deadband_room.py` imports `fleet_simulator` as a module.  
- The `DeadbandRoom` class uses `fleet_simulator` to generate synthetic training data (agent interactions, P0/P1/P2 decisions).  
- Simulated agents in `fleet_simulator` produce "interaction tiles" that feed directly into the room's training loop.  

**Integration Point:**  
`DeadbandRoom._generate_training_batch()` calls `fleet_simulator.simulate_episode()` and formats the output into tiles.

**Gap:**  
- The connection is one‑way: plato‑torch consumes simulator data, but the simulator does not receive feedback from trained ensigns.  
- No live parameter tuning of the simulator based on room performance.

---

## 2. **plato-torch** → **plato-ensign**
**Connection:** ✅ **Connected**  
**How:**  
- `plato-torch/src/export/ensign_exporter.py` provides the `export_ensign()` function.  
- After a room trains on accumulated tiles, the exporter can save the trained weights as a LoRA adapter (`.safetensors`).  
- The exporter reads the room’s `training_log.json` and model checkpoints.

**Integration Point:**  
`plato-torch` → `plato-ensign` is a pipeline: room training → adapter export.

**Gap:**  
- Export is manual/optional; no automatic triggering when a room reaches a performance threshold.  
- No validation loop that tests the exported ensign back in the simulator.

---

## 3. **holodeck-rust** → **plato-torch**
**Connection:** ⚠️ **Partial**  
**How:**  
- `holodeck-rust/src/npc/sentiment_engine.rs` defines NPCs that generate dialogue and actions.  
- These NPC interactions *could* become training tiles, but there is no current code that sends them to a plato‑torch room.  
- The `holodeck-rust` README mentions “sentiment‑aware NPCs for training data,” but the plumbing is not yet implemented.

**Integration Point:**  
None in code. Design doc suggests `holodeck-rust` → `plato-torch` as a data source.

**Gap:**  
- No TCP/HTTP API or file‑based handoff from holodeck to plato‑torch.  
- NPC interactions are currently trapped inside the MUD server.

---

## 4. **fleet-simulator** → **holodeck-rust**
**Connection:** ❌ **Not Connected**  
**How:**  
- `fleet-simulator` simulates abstract agent decision‑making.  
- `holodeck-rust` is a concrete, interactive MUD environment.  
- No imports, no shared data structures, no messaging between them.

**Potential Integration:**  
Simulator agents could be given “avatars” in the MUD, allowing their decisions to drive NPC behavior. Not implemented.

---

## 5. **plato-ensign** → **fleet-simulator** / **holodeck-rust**
**Connection:** ❌ **Not Connected**  
**How:**  
- An ensign (LoRA) is a trained specialist model.  
- It could be loaded into the simulator to guide synthetic agents, or into holodeck‑rust to control NPCs.  
- No code exists to load `.safetensors` adapters into either system.

**Gap:**  
Missing “ensign deployment” step. Ensigns are exported but not used downstream.

---

## 6. **Ghost Tiles & Mirror Play**
**Connection:** 🔄 **Conceptual**  
**How:**  
- Ghost tiles (dead agents’ lessons) are stored as P0 examples in `memory/`.  
- Mirror Play is described as two PLATO systems iterating screen‑to‑screen.  
- In code: `plato-torch/src/data/ghost_injector.py` can load ghost tiles into a room’s training buffer.

**Integration Point:**  
Ghost tiles can be injected, but there is no automatic harvesting of new ghost tiles from dead agents.

**Gap:**  
No daemon that watches agent logs, extracts P0 failures, and writes them to the ghost‑tile pool.

---

## Summary: Integration Gaps (P1 Channels)

| Gap | Systems Involved | Priority |
|-----|------------------|----------|
| Holodeck → Plato‑torch data pipe | holodeck‑rust → plato‑torch | High |
| Ensign deployment loop | plato‑ensign → fleet‑simulator / holodeck‑rust | High |
| Simulator feedback from room performance | plato‑torch → fleet‑simulator | Medium |
| Automatic ghost‑tile harvesting | agent logs → ghost_injector | Medium |
| Mirror‑Play implementation | plato‑torch ↔ plato‑torch (second instance) | Low |
