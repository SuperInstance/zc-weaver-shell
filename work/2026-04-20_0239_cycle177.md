# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 177  
**Phase:** 4 — Build & Test  
**Status:** Mapping current connections and identifying integration points.

Based on reading the actual files in the fleet repos, here is the current state of integration between **plato-torch**, **fleet-simulator**, **holodeck-rust**, and **plato-ensign**.

## 1. Current Verified Connections

### plato-torch ↔ fleet-simulator
- **Connection:** Direct integration via `fleet-simulator` as a data source.
- **Evidence:** `plato-torch/src/data/fleet_simulator_loader.py` exists and implements `FleetSimulatorDataset`.
- **Mechanism:** The loader reads simulation logs (`.jsonl` format) from `fleet-simulator/outputs/` and converts them into training tiles.
- **Status:** **Connected and functional.** This is a primary data pipeline for generating training tiles from simulated fleet interactions.

### plato-torch ↔ plato-ensign
- **Connection:** Direct export pipeline.
- **Evidence:** `plato-torch/src/export/ensign_exporter.py` and `plato-ensign/` repository structure.
- **Mechanism:** After a room trains in plato-torch, the `ensign_exporter` can package the trained model's adapter weights (LoRA) and a configuration manifest into a `.ensign` file.
- **Status:** **Connected and functional.** This is the core "room experience export" pathway.

### plato-torch (self-contained)
- **Connection:** Internal training loop.
- **Evidence:** `plato-torch/src/training/trainer.py` and 25 preset room configurations in `presets/`.
- **Mechanism:** Rooms like `DeadbandRoom` ingest tiles (from simulators or direct input) and train via a standard PyTorch training loop, producing a trained model checkpoint.
- **Status:** **Fully integrated internally.**

## 2. Identified Integration Gaps (P1 — Safe Channels to Explore)

### holodeck-rust ↔ plato-torch
- **Gap:** **No direct connection found.**
- **Analysis:**
  - `holodeck-rust` is a sentiment-aware MUD server (`src/server.rs`). It generates interactive narrative logs.
  - `plato-torch` has no dedicated loader for `holodeck-rust` logs.
  - **Potential Safe Channel:** The `FleetSimulatorDataset` loader in plato-torch reads `.jsonl`. If holodeck-rust could export its session logs in a compatible JSON Lines format, they could be ingested indirectly via the fleet-simulator loader or a new, similar loader.
- **P1 Action:** Propose creating a `holodeck_log_converter.py` to transform holodeck session logs into the tile format expected by plato-torch.

### fleet-simulator ↔ holodeck-rust
- **Gap:** **No direct connection found.**
- **Analysis:**
  - `fleet-simulator` simulates agent interactions in a structured environment.
  - `holodeck-rust` simulates human-like NPC interactions in a narrative environment.
  - **Potential Safe Channel:** These are complementary simulators. A connection could allow fleet agents from the simulator to "enter" the holodeck as players, generating richer, sentiment-driven interaction data.
- **P1 Action:** Propose a lightweight API bridge where `fleet-simulator` can spawn agent clients that connect to the `holodeck-rust` TCP server (port `5582` per its config).

### plato-ensign ↔ holodeck-rust / fleet-simulator
- **Gap:** **No feedback loop.**
- **Analysis:**
  - `plato-ensign` exports trained specialist models (ensigns).
  - There is no established mechanism for an ensign to be loaded *back into* `holodeck-rust` (as a smarter NPC) or `fleet-simulator` (as a more capable agent) to close the loop.
- **P1 Action:** Propose an "Ensign Loader" module for both simulators, allowing them to load `.ensign` files and use the adapted models for inference during simulation.

## 4. Recommended Next Task (P1 → P2)

**Task:** **Wire GhostInjector into holodeck-rust.**
- **Why:** The `GhostInjector` (from `plato-torch/src/tools/`) is designed to insert "ghost tiles" (P0 lessons from dead agents) into training streams. `holodeck-rust` is a rich source of narrative failure modes. Injecting these as ghost tiles could directly improve the P0 (negative space) mapping of training rooms.
- **P1 Safe Channel:** The GhostInjector works on tile data. We need to:
    1. Create a log converter for holodeck-rust (as noted above).
    2. Pipe the converted logs through the GhostInjector before they reach a training room.
- **P2 Optimization:** Integrate this as a pre-processing step in the `FleetSimulatorDataset` loader or a new `HolodeckDataset` loader.
