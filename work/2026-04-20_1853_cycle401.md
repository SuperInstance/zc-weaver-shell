# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 401

## 1. plato-torch → fleet-simulator
**Connection:** ✅ **Direct import**  
- `plato-torch/rooms/deadband_room.py` imports `from fleet_simulator.simulator import FleetSimulator`  
- Used in `DeadbandRoom._step_simulation()` to create a simulator instance and step it.  
- Simulator provides `get_observations()` which returns agent states (position, velocity, battery).  
- **Data flow:** Simulator state → room observations → training tiles.

**Integration gap:**  
- Simulator configuration is hardcoded in `DeadbandRoom` (e.g., `num_agents=5`).  
- No shared configuration file or environment variable mapping.  
- Simulator’s internal event system not exposed to room’s reward/penalty logic.

---

## 2. plato-torch → plato-ensign
**Connection:** ✅ **Direct export path**  
- `plato-torch/export/ensign_exporter.py` provides `export_room_to_lora()`.  
- Called from `plato-torch/train.py` after room training loop completes.  
- Uses `PeftModel` and `LoraConfig` from Hugging Face `peft` library.  
- **Data flow:** Room checkpoint → LoRA adapter → saved to `outputs/ensigns/`.

**Integration gap:**  
- Export is optional and manual (must be triggered via CLI flag `--export-ensign`).  
- No automatic quality threshold (e.g., only export if reward > X).  
- No versioning or metadata linking ensign back to specific room session.

---

## 3. plato-torch → holodeck-rust
**Connection:** ❌ **No direct import found**  
- Searched `plato-torch/` for `holodeck`, `mud`, `sentiment`, `npc` – no matches.  
- `holodeck-rust` is a separate binary (MUD server) with its own TCP/WebSocket interface.  
- **Potential link:** Could be connected via `plato-relay` (message routing), but not observed in current code.

**Integration gap:**  
- No shared session state between PLATO rooms and holodeck NPCs.  
- No sentiment or narrative context passed to training rooms.  
- Holodeck’s “ghost tile” concept not yet wired into plato-torch’s tile ingestion.

---

## 4. fleet-simulator → holodeck-rust
**Connection:** ❌ **No direct import found**  
- `fleet-simulator` is pure Python, no Rust bindings observed.  
- `holodeck-rust` does not reference simulator types or data.  
- Both are independent data sources for training.

**Integration gap:**  
- Simulator’s geometric agent paths not used for NPC navigation in holodeck.  
- No common coordinate system or world state synchronization.

---

## 5. fleet-simulator → plato-ensign
**Connection:** ⚠️ **Indirect via plato-torch**  
- Simulator data influences room training.  
- Room training produces checkpoints.  
- Checkpoints can be exported to ensigns.  
- No direct simulator→ensign pipeline.

**Integration gap:**  
- Cannot create an ensign purely from simulator data (no room intermediary).  
- Simulator’s internal decision logic not extractable as a lightweight adapter.

---

## 6. holodeck-rust → plato-ensign
**Connection:** ❌ **No observed pathway**  
- `holodeck-rust` outputs JSON logs and tile files.  
- `plato-ensign` expects a PyTorch checkpoint from a PLATO room.  
- No tool currently converts holodeck sessions to ensigns.

**Integration gap:**  
- NPC sentiment models cannot be exported as LoRA adapters.  
- Holodeck’s “ghost tiles” remain isolated from ensign training pipeline.

## 7. GhostInjector (mentioned in tasks)

**Integration gap:**  
- GhostInjector not wired into holodeck’s tile stream.  
- No mechanism for holodeck to generate ghost tiles directly.
