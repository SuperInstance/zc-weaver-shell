# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 373  
**Phase:** 4 — Build & Test  
**Status:** Direct file inspection of fleet repositories. Focus on actual imports, configuration references, and shared data structures.

## 1. plato-torch → fleet-simulator
**Connection:** **Direct import**  
- `plato-torch/rooms/deadband_room.py` imports `fleet_simulator` (line 4)  
- Uses `fleet_simulator.FleetSimulator` to generate training data  
- Simulator provides `generate_training_data()` method returning a list of interaction dicts  
- Room ingests these dicts as training tiles  

**Integration Gap:**  
- Simulator output format not explicitly validated against room’s expected tile schema  
- No shared constants or schema definition file  
- Room assumes dict keys (`agent_id`, `action`, `reward`, `state`) exist  

## 2. plato-torch → plato-ensign  
**Connection:** **Optional export pipeline**  
- `plato-torch/export/ensign_exporter.py` exists  
- Exporter reads room’s accumulated tiles (from `room.training_log`)  
- Converts tiles to LoRA‑compatible dataset (via `transform_to_lora_dataset()`)  
- Outputs `.jsonl` and adapter config  

**Integration Gap:**  
- Exporter is separate script, not integrated into room’s training loop  
- No automatic trigger—requires manual run after room training  
- No versioning or checksum linking between room state and exported adapter  

## 3. holodeck-rust → (plato-torch / fleet-simulator)  
**Connection:** **None found**  
- `holodeck-rust/src/` shows no imports of Python modules  
- No Python bindings (`pyo3`, `cffi`) in `Cargo.toml`  
- No shared data files or schema references  

**Potential Integration Point:**  
- `holodeck-rust` could be a tile *source* for `plato-torch` (NPC interactions → training tiles)  
- Would require a bridge (RPC, Redis, shared JSON log)  
- Currently isolated as standalone MUD server  

## 4. fleet-simulator → plato-ensign  
**Connection:** **Indirect via plato-torch**  
- Simulator tiles flow into room → room tiles exported to ensign  
- No direct simulator‑to‑ensign path  

**Integration Gap:**  
- Simulator tiles lack metadata needed for LoRA (e.g., instruction‑response pairs)  
- Ensign exporter expects transformed format, not raw simulator output  

## 5. plato-ensign → holodeck-rust  
**Connection:** **None found**  
- No Rust code loading `.safetensors` or LoRA adapters  
- No references to `plato-ensign` in `holodeck-rust`  

**Potential Integration Point:**  
- Trained ensign (LoRA) could be loaded into holodeck NPCs for sentiment‑aware behavior  
- Would require `llama.cpp`‑style Rust inference + adapter merging  

## 6. Internal Dependencies (within plato-torch)  
**Connection:** **Tightly coupled**  
- `deadband_room.py` imports `training_logger`, `tile_validator`, `room_base`  
- All share `Tile` dataclass (in `models/tile.py`)  
- Logger writes to `room.training_log` (list of Tile objects)  

**Integration Gap:**  
- Tile schema defined only as Python dataclass—no protobuf/JSON schema for cross‑language use

```
plato-torch ←── fleet-simulator (direct import, tiles)
      │
      ├─→ plato-ensign (optional export)
      │
holodeck-rust (isolated)
```
