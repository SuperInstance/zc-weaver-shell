# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 373  
**Phase:** 4 — Build & Test  
**Status:** Direct file inspection of fleet repositories. Focus on actual imports, configuration references, and shared data structures.

## 1. plato-torch
**Location:** `plato-torch/`  
**Purpose:** Training room presets (25 rooms), self-training rooms, tile ingestion and processing.

### Verified Connections:
- **plato-ensign:** Direct import.  
  `plato-torch/ensign_exporter.py` imports `from plato_ensign import export_lora`.  
  `plato-torch/rooms/deadband_room.py` calls `export_lora()` after training cycles.
- **fleet-simulator:** Indirect via shared tile format.  
  `plato-torch/tile_formats.py` defines `SimulationTile` used by `fleet-simulator/tile_ingest.py`.
- **holodeck-rust:** No direct import. Connection is through tile ingestion pipeline.  
  `plato-torch/ingest/holodeck_ingest.py` reads JSON logs from `holodeck-rust/logs/`.

### Integration Gaps:
- No live API connection to `holodeck-rust`. Ingestion is file‑based (batch).
- `plato-torch` does not call `fleet-simulator` directly; tiles are passed via shared storage.

## 2. fleet-simulator
**Location:** `fleet-simulator/`  
**Purpose:** Generate simulation data for training, emulate multi‑agent fleet interactions.

### Verified Connections:
- **plato-torch:** Provides `SimulationTile` format.  
  `fleet-simulator/tile_ingest.py` imports `from plato_torch.tile_formats import SimulationTile`.
- **holodeck-rust:** No direct import. Simulated agent behaviors are inspired by holodeck sentiment patterns, but no code linkage.
- **plato-ensign:** No direct connection. Simulation tiles can be consumed by plato‑torch, which may later export an ensign.

### Integration Gaps:
- Stand‑alone simulator; not triggered by other systems automatically.
- No real‑time feedback loop from plato‑torch to adjust simulation parameters.

## 3. holodeck-rust
**Location:** `holodeck-rust/`  
**Purpose:** MUD server with sentiment‑aware NPCs, generates interaction logs.

### Verified Connections:
- **plato-torch:** Logs are read by `plato-torch/ingest/holodeck_ingest.py`.  
  `holodeck-rust/src/logger.rs` writes JSON to `logs/` directory.
- **fleet-simulator:** No direct connection.
- **plato-ensign:** No direct connection.

### Integration Gaps:
- Logs are passive; no API for plato‑torch to query holodeck state live.
- No mechanism for plato‑torch to inject scenarios or NPC traits into holodeck.

## 4. plato-ensign
**Location:** `plato-ensign/`  
**Purpose:** Export room experience (LoRA adapters) from plato‑torch training.

### Verified Connections:
- **plato-torch:** Directly called by `plato-torch/ensign_exporter.py`.  
  `plato-ensign/export.py` provides `export_lora(room_state, output_path)`.
- **fleet-simulator:** No connection.
- **holodeck-rust:** No connection.

### Integration Gaps:
- Only exports from plato‑torch; cannot create ensigns from raw holodeck or simulator data directly.
- No versioning or registry of exported ensigns.

## 5. Cross‑System Data Flow
```
holodeck-rust → (JSON logs) → plato-torch → (training tiles) → plato-ensign
fleet-simulator → (SimulationTile) → plato-torch → (training tiles) → plato-ensign
```
**Missing Links:**
- No pipeline from holodeck‑rust → fleet‑simulator (could enrich simulation with real interaction patterns).
- No feedback from plato‑ensign back to plato‑torch to guide next training cycle.
- No shared configuration or service discovery across repos.
