# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 147  
**Phase:** 4 — Build & Test  
**Status:** Mapping current connections and identifying integration points.

Based on reading the actual files in the fleet repos, here is the current state of integration between **plato-torch**, **fleet-simulator**, **holodeck-rust**, and **plato-ensign**.

---

## 1. **plato-torch** → **plato-ensign**  
**Connection:** **Direct and documented**  
- `plato-torch` exports training room experience via `export_room()` function.  
- `plato-ensign` ingests these exports to create LoRA adapters (`ensign.py`).  
- File: `plato-ensign/ensign.py` shows explicit import from `plato_torch.rooms`.  
- This is the primary training-to-export pipeline.

**Integration Gap:** None — this is the core designed flow.

---

## 2. **plato-torch** → **fleet-simulator**  
**Connection:** **Indirect via shared data format**  
- `fleet-simulator` generates synthetic training data (JSONL format).  
- `plato-torch` can load external datasets via `DataLoader` classes.  
- No direct API calls between them, but they share a common tile format.

**Integration Gap:**  
- No automated pipeline from simulator → training room.  
- Manual step required to move generated data into a room's tile store.

---

## 3. **holodeck-rust** → **plato-torch**  
**Connection:** **Planned but not implemented**  
- `holodeck-rust` is a MUD server with sentiment-aware NPCs.  
- `plato-torch` has a `HolodeckRoom` preset (in `presets/`).  
- The preset exists but the actual socket connection to a running holodeck server is not wired.

**Integration Gap:**  
- `HolodeckRoom` is a stub — does not connect to a live holodeck.  
- No tile ingestion from holodeck interactions into plato-torch.

---

## 4. **fleet-simulator** → **holodeck-rust**  
**Connection:** **None observed**  
- `fleet-simulator` generates agent interaction logs.  
- `holodeck-rust` has its own internal event system.  
- No shared data format or API between them.

**Integration Gap:**  
- Simulated agent behaviors cannot drive holodeck NPCs.  
- Holodeck events cannot feed back into the simulator.

---

## 5. **plato-ensign** → **holodeck-rust**  
**Connection:** **None observed**  
- `plato-ensign` outputs LoRA adapters (`.safetensors`).  
- `holodeck-rust` does not currently load external AI models.  
- No hook for injecting an ensign into the MUD’s NPC logic.

**Integration Gap:**  
- Trained specialist models cannot be deployed to holodeck NPCs.

---

## 6. **Ghost Tiles & Deadband Protocol**  
**Connection:** **Conceptual, not implemented**  
- Deadband Protocol (P0/P1/P2) is documented in `DEADBAND-PROTOCOL.md`.  
- `plato-torch` has a `DeadbandRoom` preset.  
- Ghost tiles (P0 knowledge from dead agents) are mentioned but no storage/retrieval system exists.

**Integration Gap:**  
- No actual ghost tile storage or injection mechanism.  
- DeadbandRoom is a visual demo, not integrated with real P0/P1/P2 logic.

---
