# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 397

**Status:** Direct file inspection of fleet repositories. Focus on actual imports, configuration references, and shared data structures.

---

## 1. plato-torch → fleet-simulator
**Connection:** ✅ **Direct import**  
- **File:** `plato-torch/src/training/rooms/deadband_room.py`  
- **Line:** `from fleet_simulator.simulator import FleetSimulator`  
- **Usage:** `DeadbandRoom` initializes a `FleetSimulator` instance to generate training episodes.  
- **Data flow:** Simulator produces episode trajectories (state-action-reward sequences) → Room ingests them as training tiles.  
- **Shared config:** `configs/deadband_config.yaml` references `fleet_simulator` settings for episode length and agent count.

**Integration gap:**  
- No bidirectional control. Room triggers simulator, but simulator cannot push data asynchronously.  
- Simulator output format is rigid; room expects a specific `Episode` schema.

---

## 2. plato-torch → plato-ensign
**Connection:** ✅ **Direct export pipeline**  
- **File:** `plato-torch/src/export/ensign_exporter.py`  
- **Line:** `from plato_ensign.lora_adapter import LoRAAdapter`  
- **Usage:** After room training completes, `ensign_exporter` serializes the trained model’s adapter weights and metadata into a LoRA file.  
- **Data flow:** Room training checkpoints → Exporter → `.safetensors` + `adapter_config.json` → plato-ensign repository.

**Integration gap:**  
- Export is a one-time batch process, not incremental.  
- No live streaming of adapter updates during training.

---

## 3. holodeck-rust → (plato-torch, fleet-simulator, plato-ensign)
**Connection:** ❌ **No direct imports found**  
- **File inspection:** `holodeck-rust/src/` shows no Python imports or FFI bindings to the other three repos.  
- **Configuration:** `holodeck-rust/config.toml` contains only internal MUD settings (port, NPC sentiment ranges, room definitions).  
- **Potential indirect link:** The `memory/` directory in fleet-knowledge contains logs that mention holodeck sessions, but no code-level integration.

**Integration gap:**  
- Holodeck operates as a standalone sentiment-aware MUD.  
- No shared episode data with fleet-simulator.  
- No training tile ingestion from plato-torch.  
- No adapter import/export with plato-ensign.

---

## 4. fleet-simulator → plato-ensign
**Connection:** ❌ **No direct link**  
- Simulator outputs raw episode data, not model adapters.  
- No `plato_ensign` import in simulator source.

**Integration gap:**  
- Simulated episodes cannot directly fine-tune an ensign without passing through a plato-torch room first.

---

## 5. GhostInjector (plato-torch) → holodeck-rust
**Connection:** ⚠️ **Planned but not implemented**  
- **File:** `plato-torch/src/training/ghost_injector.py` exists.  
- **Code inspection:** Contains placeholder `inject_to_holodeck()` method with `# TODO: connect to holodeck API` comment.  
- **Hypothesis:** Ghost tiles (P0 negative examples) are meant to be injected into holodeck as NPC dialogue or event triggers.

**Integration gap:**  
- No actual networking or API call to holodeck’s running MUD server.  
- Ghost tile format (`GhostTile` class) does not match holodeck’s `Event` schema.

---

## 6. plato-relay → DeadbandRoom
**Connection:** ❌ **Not found**  
- `plato-relay` repository exists but is not imported in `deadband_room.py`.  
- `DeadbandRoom` uses local simulator only.

**Integration gap:**  
- Room cannot receive remote training tiles or distribute workloads via relay.
