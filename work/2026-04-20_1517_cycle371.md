## 1. Repository Structure & Dependencies

Based on reading the actual files in the fleet-knowledge directory and cross-referencing with repository READMEs:

### **plato-torch** (Training Rooms)
- **Purpose**: 25 training room presets, self-training rooms.
- **Key Files**: `DeadbandRoom.py`, `GhostInjector.py`, `plato_relay.py`.
- **Dependencies**: `torch`, `transformers`, `datasets` (PyTorch ecosystem).
- **Exports**: Training tiles (JSON/msgpack) to room server.

### **plato-ensign** (LoRA Export)
- **Purpose**: Exports room experience into LoRA adapters.
- **Key Files**: `export_lora.py`, `train_ensign.py`.
- **Dependencies**: `peft`, `accelerate`, `plato-torch` (imports room data).
- **Input**: Tiles from plato-torch rooms.
- **Output**: `.safetensors` adapter files.

### **holodeck-rust** (Sentiment-Aware MUD)
- **Purpose**: MUD server with NPCs that react to player sentiment.
- **Key Files**: `src/npc/sentiment.rs`, `src/world/mod.rs`.
- **Dependencies**: `tokio`, `serde`, `tracing` (Rust async stack).
- **Data Flow**: Receives events, updates NPC state, emits logs.

### **fleet-simulator** (Training Data Generator)
- **Purpose**: Generates synthetic fleet interaction data.
- **Key Files**: `simulate.py`, `agent_behavior.py`.
- **Dependencies**: `numpy`, `pandas`, `networkx`.
- **Output**: JSONL files of simulated agent interactions.

## 2. Verified Connections (What's Connected)

### **plato-torch → plato-ensign** (Strong, Direct)
- **Evidence**: `plato-ensign/train_ensign.py` imports `from plato_torch.rooms import DeadbandRoom`.
- **Data Flow**: Room tiles (saved as `.pt` or `.json`) are loaded by ensign training script.
- **Integration Point**: Shared tile schema (agent_id, timestamp, interaction, reward).

### **plato-torch → holodeck-rust** (Planned, Not Yet Implemented)
- **Evidence**: `plato-torch/GhostInjector.py` contains placeholder `inject_to_holodeck()` function.
- **Status**: Commented stub. No actual network call or Rust FFI.
- **Intent**: Inject ghost tiles (dead agent lessons) into holodeck NPC memory.

### **fleet-simulator → plato-torch** (Indirect, via Data)
- **Evidence**: `fleet-simulator/README.md` states "output compatible with plato-torch tile format".
- **Data Flow**: Simulator writes JSONL → plato-torch data loader reads it.
- **Integration Point**: Shared tile schema validation.

## 3. Integration Gaps (What's Not Connected)

### **holodeck-rust ←/→ plato-torch** (Missing Bidirectional Link)
- **Gap**: No Rust Python bindings (PyO3/maturin) or HTTP API exposed by holodeck.
- **Consequence**: GhostInjector cannot send data; holodeck cannot trigger training events.

### **plato-ensign ←/→ holodeck-rust** (No Connection)
- **Gap**: Ensign exports LoRA adapters, but holodeck has no loader for `.safetensors`.
- **Consequence**: Trained behavior models cannot be deployed to NPCs.

### **fleet-simulator → holodeck-rust** (No Direct Feed)
- **Gap**: Simulator outputs static files; holodeck consumes live socket connections.
- **Consequence**: Synthetic data cannot drive live NPCs without a bridge service.

### **DeadbandRoom → plato-relay** (Unverified)
- **Gap**: `plato_relay.py` exists but import relationship with DeadbandRoom is unclear.
- **Action Needed**: Check if `DeadbandRoom` calls `plato_relay.submit()` or if relay is separate.
