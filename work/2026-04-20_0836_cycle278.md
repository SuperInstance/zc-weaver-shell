## 1. plato-torch → fleet-simulator
**Connection:** **Yes** — Direct import and data flow.

**Evidence:**
- `plato-torch/rooms/deadband_room.py` imports `from fleet_simulator.simulator import FleetSimulator`.
- `DeadbandRoom` initializes a `FleetSimulator` instance in its `__init__` method.
- The room uses the simulator to generate training episodes (`simulator.generate_episode()`).
- Episode data (state, action, reward sequences) is passed back to the room for tile creation.

**Integration Point:** `DeadbandRoom` treats `FleetSimulator` as a synthetic data source for training tiles. This is a **producer-consumer** connection.

---

## 2. plato-torch → plato-ensign
**Connection:** **Yes** — Configuration and pipeline path.

**Evidence:**
- `plato-torch/config/ensign_config.yaml` defines export settings for LoRA adapters.
- `plato-torch/export/ensign_exporter.py` contains the `EnsignExporter` class, which reads trained room checkpoints and calls `plato-ensign`'s conversion utilities.
- The export process is triggered via a CLI command (`plato-torch export --room deadband --target ensign`).

**Integration Point:** `plato-torch` rooms train, save checkpoints; `plato-ensign` converts checkpoints to LoRA adapters. This is a **post-training export** connection.

## 4. fleet-simulator → holodeck-rust

- No shared data formats or cross-references.

**Integration Gap:** Could connect if simulator agents interact with holodeck NPCs for richer narrative training data. Currently **isolated**.

**Integration Point:** Data flows `fleet-simulator` → `plato-torch` → `plato-ensign`. No direct link.

## 7. plato-ensign → Others
**Connection:** **Output-only** — produces LoRA adapters.

**Evidence:**
- `plato-ensign/README.md` states it "exports room experience into LoRA adapters".
- Adapters are saved to disk; can be loaded by inference servers or `cudaclaw` runtime.
- No inbound dependencies from other fleet components.

**Integration Point:** `plato-ensign` is a **terminal node** in the training pipeline.
