## 1. plato-torch → fleet-simulator
**Connection:** **YES** (direct dependency)  
- `plato-torch/requirements.txt` includes `fleet-simulator>=0.1.0`  
- `plato-torch/src/training/rooms/deadband_room.py` imports `from fleet_simulator.agents import BaseAgent`  
- `plato-torch/src/training/rooms/__init__.py` registers `DeadbandRoom` which uses `fleet_simulator` for agent simulation  
- **Data flow:** `DeadbandRoom` instantiates simulator agents, steps through simulation cycles, collects interaction tiles.

**Integration gap:**  
- No observed use of `fleet-simulator`'s `Scenario` or `Environment` classes directly in plato-torch. Only `BaseAgent` import suggests limited integration.  
- Simulator configuration (agent count, step limits) appears hardcoded in `deadband_room.py` rather than shared config.

## 2. plato-torch → holodeck-rust
**Connection:** **NO** (no direct imports or references)  
- `plato-torch` Python codebase contains no Rust bindings or `holodeck` module imports.  
- `holodeck-rust` is a separate Rust MUD server with its own TCP/WebSocket API.  
- **Potential indirect link:** Both systems produce "interaction tiles" (JSON), but no observed shared tile format or ingestion pipeline.

**Integration gap:**  
- `holodeck-rust` sentiment-aware NPC interactions are not wired into `plato-torch` training rooms.  
- No shared session state or tile exchange mechanism between the two systems.

## 3. plato-torch → plato-ensign
**Connection:** **YES** (explicit export pipeline)  
- `plato-torch/src/export/ensign_exporter.py` exists, imports `torch`, `peft` (LoRA).  
- `ensign_exporter.py` defines `EnsignExporter` class with `export_lora_adapter(tiles, base_model, output_dir)`.  
- `plato-torch/scripts/export_ensign.py` provides CLI to export a trained room's tile history to a LoRA adapter.  
- **Data flow:** Room tile history → `EnsignExporter` → LoRA weights saved to disk.

**Integration gap:**  
- Export is manual (CLI script), not automatic after room training.  
- No observed integration where `plato-ensign` adapters are reloaded into `plato-torch` for further training (no cyclic improvement).

## 4. fleet-simulator → holodeck-rust
**Connection:** **NO**  
- `fleet-simulator` is Python-based agent simulation for training data generation.  
- `holodeck-rust` is a live multiplayer text‑based world.  
- No shared agent definitions, communication protocols, or tile formats.

**Integration gap:**  
- Simulated agents cannot enter the holodeck MUD; holodeck NPCs cannot be used as simulator agents.

## 5. fleet-simulator → plato-ensign
**Connection:** **INDIRECT** (via plato-torch)  
- `fleet-simulator` generates agent interaction tiles during `DeadbandRoom` runs.  
- Those tiles are stored in the room’s tile history, then can be exported by `plato-ensign`.  
- No direct simulator‑to‑ensign export observed.

## 6. holodeck-rust → plato-ensign
**Connection:** **NO**  
- `holodeck-rust` stores interaction logs in its own database (`holodeck.db`).  
- No observed exporter that converts holodeck logs to the tile format expected by `plato-ensign`.  
- No Rust‑to‑Python pipeline for LoRA training.

## Summary of Integration Gaps (P0 / P1)
**P0 (what NOT to do):**  
- Do not assume holodeck‑rust and plato‑torch share tile formats—they do not.  
- Do not assume fleet‑simulator agents can interact with holodeck NPCs—no bridge exists.

**P1 (safe channels for integration):**  
- **Channel A:** Extend `plato-torch` to ingest `holodeck-rust` interaction logs (convert to tiles).  
- **Channel B:** Wire `holodeck-rust` sentiment‑aware NPCs as agents in `fleet-simulator` (requires Rust‑Python bridge).  
- **Channel C:** Automate `plato-ensign` export after each room training session (trigger from room completion).  
- **Channel D:** Create shared tile schema (`tile_spec.json`) across plato‑torch, holodeck‑rust, fleet‑simulator.
