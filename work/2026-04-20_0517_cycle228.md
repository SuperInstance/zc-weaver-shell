## 1. plato-torch
**Location:** `fleet-knowledge/plato-torch/`  
**Purpose:** Training room presets, self-training rooms, tile processing.

### Verified Connections:
- **To plato-ensign:** Direct import in `plato-torch/src/ensign_exporter.py`.  
  - Exports room experience (tiles) to LoRA adapters via `export_to_lora()`.
  - Uses `plato_ensign.LoRAAdapterBuilder` class.
- **To fleet-simulator:** No direct import found.  
  - However, `plato-torch/presets/DeadbandRoom.py` contains a `SimulationEnvironment` class that could be a placeholder or interface for future integration.
- **To holodeck-rust:** No direct import or reference in Python files.  
  - Potential indirect link: both use "sentiment" and "interaction" concepts, but no code-level connection.

### Integration Gaps:
- No live data pipeline from `fleet-simulator` to `plato-torch` training rooms.
- No explicit hook for `holodeck-rust` NPC interactions to generate tiles.

## 2. fleet-simulator
**Location:** `fleet-knowledge/fleet-simulator/`  
**Purpose:** Generate synthetic training data through multi-agent simulation.

### Verified Connections:
- **To plato-torch:** None.  
  - `fleet-simulator/src/main.rs` outputs logs and JSON events, but no direct pipe to `plato-torch`.
  - Could be manually curated into training tiles.
- **To holodeck-rust:** None.  
  - Separate Rust projects, no shared crates observed.
- **To plato-ensign:** None.

### Integration Gaps:
- Simulation events not automatically formatted as `plato-torch` tiles.
- No real-time training loop where simulator data feeds room presets.

## 3. holodeck-rust
**Location:** `fleet-knowledge/holodeck-rust/`  
**Purpose:** MUD server with sentiment-aware NPCs, interactive environment.

### Verified Connections:
- **To plato-torch:** None.  
  - NPC interactions and player sentiment are logged but not exported as tiles.
- **To fleet-simulator:** None.
- **To plato-ensign:** None.

### Integration Gaps:
- Rich interaction data trapped in server logs; not harvested for training.
- No `plato-torch` preset that consumes holodeck session data.

## 4. plato-ensign
**Location:** `fleet-knowledge/plato-ensign/`  
**Purpose:** Export room experience into LoRA adapters.

### Verified Connections:
- **To plato-torch:** Directly imported by `plato-torch` as a library.  
  - `plato_ensign` provides `LoRAAdapterBuilder` and `export_room()`.
- **To fleet-simulator:** None.
- **To holodeck-rust:** None.

### Integration Gaps:
- Only consumes data from `plato-torch` rooms.  
- No adapter training from `fleet-simulator` or `holodeck-rust` data sources.

## 5. Overall Integration Map

```
plato-torch (training rooms)
    │
    ├──► plato-ensign (LoRA export)   ✅ CONNECTED
    │
    ├─── fleet-simulator (sim data)   ❌ NOT CONNECTED
    │
    └─── holodeck-rust (NPC logs)     ❌ NOT CONNECTED
```

**Connected:**
- `plato-torch` → `plato-ensign` (one-way, room → LoRA).

**Not Connected:**
- `fleet-simulator` → any.
- `holodeck-rust` → any.
- No round-trip or closed-loop training pipelines.
