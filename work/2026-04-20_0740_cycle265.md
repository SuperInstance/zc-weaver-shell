## 1. plato-torch
**Location:** `plato-torch/`  
**Purpose:** Training room presets, self-training rooms, tile processing.

### Verified Connections:
- **plato-ensign:** Exports room experience to LoRA adapters.  
  - `plato-torch/export_ensign.py` imports `plato_ensign` and calls `export_room_to_lora()`.  
  - Shared config: `training_config.yaml` references `ensign_output_dir`.  
- **fleet-simulator:** Provides training data.  
  - `plato-torch/data_loader.py` imports `fleet_simulator.simulate_episode`.  
  - Uses `SimulationEpisode` data structure from `fleet_simulator/types.py`.  
- **holodeck-rust:** Sentiment-aware NPC interactions as training tiles.  
  - `plato-torch/tile_ingest.py` reads from `holodeck-rust/logs/npc_interactions.json`.  
  - No direct Rust imports; file-based integration.

### Integration Gaps:
- **cudaclaw:** No GPU-resident runtime hooks in plato-torch.  
  - Training loops are CPU/ PyTorch only; no `cudaclaw` API calls.  
- **flux-runtime:** No bytecode VM integration.  
  - Room presets are pure Python; no `flux` module imports.  
- **constraint-theory-core:** No geometric snapping in training logic.  
  - Only used in separate `constraint-theory-demos`.

## 2. fleet-simulator
**Location:** `fleet-simulator/`  
**Purpose:** Generate fleet simulation data for training.

### Verified Connections:
- **plato-torch:** Primary consumer of simulation episodes.  
  - `fleet-simulator/__init__.py` exposes `simulate_episode()` function.  
  - Shared `Episode` data class used by both.  
- **holodeck-rust:** NPC sentiment data influences simulation.  
  - `fleet-simulator/agent_behavior.py` reads `holodeck-rust/out/sentiment_scores.csv`.  
  - Conditional logic: `if sentiment < threshold: agent.flee()`.

### Integration Gaps:
- **plato-ensign:** No direct export of simulations to LoRA.  
  - Simulations feed plato-torch, but no `plato_ensign` import in fleet-simulator.  
- **cudaclaw:** Simulation runs on CPU; no GPU agent runtime integration.  
- **flux-runtime:** No bytecode-driven simulation steps.

## 3. holodeck-rust
**Location:** `holodeck-rust/`  
**Purpose:** MUD server with sentiment-aware NPCs.

### Verified Connections:
- **plato-torch:** NPC interaction logs become training tiles.  
  - `holodeck-rust/src/logger.rs` writes `npc_interactions.json` to shared `logs/` directory.  
  - JSON schema includes `user_input`, `npc_response`, `sentiment_score`.  
- **fleet-simulator:** Sentiment scores influence agent behavior.  
  - `holodeck-rust/src/sentiment/analyzer.rs` outputs CSV to `out/sentiment_scores.csv`.  
  - File watched by `fleet-simulator/agent_behavior.py`.

### Integration Gaps:
- **plato-ensign:** No direct LoRA export from NPC interactions.  
  - Logs are consumed by plato-torch, not ensign.  
- **cudaclaw:** No GPU-resident NPC logic.  
- **flux-runtime:** MUD server does not use self-assembling bytecode.  
- **GhostInjector:** Not wired in.  
  - `holodeck-rust/src/` has no `ghost_injector` module or imports.

## 4. plato-ensign
**Location:** `plato-ensign/`  
**Purpose:** Export room experience into LoRA adapters.

### Verified Connections:
- **plato-torch:** Sole source of room experience data.  
  - `plato-ensign/lora_exporter.py` imports `plato_torch.RoomExperience`.  
  - Called via `export_room_to_lora(room_id, config)` from plato-torch.

### Integration Gaps:
- **fleet-simulator:** No direct simulation-to-LoRA path.  
- **holodeck-rust:** NPC interactions not exported as LoRA adapters.  
- **cudaclaw:** LoRA training runs on PyTorch, not GPU runtime.  
- **flux-runtime:** No bytecode involvement in export process.

## 6. Identified Integration Gaps (P1 Safe Channels)

1. **GhostInjector not in holodeck-rust**  
   - Gap: Dead agents' lessons (ghost tiles) not injected into NPC behavior.  
   - Safe channel: Add `ghost_injector` module to `holodeck-rust/src/`, reading from `memory/ghost_tiles/`.

2. **DeadbandRoom not connected to plato-relay**  
   - Gap: Room preset exists but no relay for cross-room tile sharing.  
   - Safe channel: Extend `plato-torch/rooms/deadband_room.py` with `plato_relay.send_tile()`.

3. **No cudaclaw integration in training pipeline**  
   - Gap: GPU-resident runtime unused in plato-torch or fleet-simulator.  
   - Safe channel: Add `cudaclaw.inference()` call to `plato-torch/training_loop.py` for GPU-accelerated forward passes.

4. **flux-runtime isolated**  
   - Gap: Self-assembling bytecode VM not used by any core repo.  
   - Safe channel: Create `flux_bridge.py` in `plato-torch/` to compile room logic to bytecode.
