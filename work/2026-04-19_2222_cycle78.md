## 1. Verified Current Connections

Based on reading the actual code and documentation in the fleet repos, here are the confirmed, working connections between the four core systems:

### **plato-torch → plato-ensign** (Strong, Direct)
- **Connection:** `plato-torch` rooms generate training tiles (`.pt` files) during agent interactions.
- **Export Path:** `plato-ensign` imports these tile files and converts them into LoRA adapters via its `export_lora()` function.
- **Evidence:** `plato-ensign/src/exporter.rs` contains `RoomExporter` struct that loads `.pt` files from a `plato-torch` room directory.
- **Status:** **Live.** This is the primary training pipeline.

### **fleet-simulator → plato-torch** (Strong, Direct)
- **Connection:** `fleet-simulator` generates synthetic training data (JSONL files of simulated agent interactions).
- **Ingestion Path:** `plato-torch` includes a `DataLoader` that can read these JSONL files and convert them into training tiles.
- **Evidence:** `plato-torch/data/loaders.py` has `FleetSimLoader` class.
- **Status:** **Live.** Used to bootstrap rooms with initial training data.

### **holodeck-rust → (plato-torch / fleet-simulator)** (Weak, Indirect)
- **Connection:** `holodeck-rust` is a sentiment-aware MUD server that generates NPC dialogue and interaction logs.
- **Potential Path:** Logs could be parsed into JSONL format compatible with `fleet-simulator` or directly into `plato-torch` tiles.
- **Evidence:** No direct import found in code. However, `holodeck-rust/README.md` states: "Outputs can feed training rooms."
- **Status:** **Theoretical / Manual.** No automated pipeline observed.

### **plato-ensign → (External Use)** (Terminal)
- **Connection:** `plato-ensign` outputs trained LoRA adapters (`.safetensors` files).
- **Usage:** These can be loaded by inference servers (e.g., `cudaclaw`, `flux-runtime`) to specialize models.
- **Evidence:** `plato-ensign` has a `--output-dir` flag for adapters.
- **Status:** **Live endpoint.** The end of one pipeline.

## 2. Identified Integration Gaps (P1 Safe Channels)

### **Gap 1: holodeck-rust → plato-torch (Automated)**
- **Problem:** Rich NPC interaction data from the MUD is not automatically feeding the training rooms.
- **Safe Channel (P1):** Build a `HolodeckLogBridge` service. It would:
    1. Watch `holodeck-rust/logs/` for new session files.
    2. Parse the structured log format (NPC dialogues, player sentiments, events).
    3. Transform into `plato-torch` tile format (or `fleet-simulator` JSONL intermediary).
    4. Place files into a watched directory for `plato-torch`'s `DataLoader`.
- **Priority:** **High.** Closes a major data loop.

### **Gap 2: Ghost Tile Injection Point**
- **Problem:** The concept of "ghost tiles" (P0 knowledge from dead agents) exists in doctrine (`FLEET-CONTEXT.md`), but no system implements it.
- **Safe Channel (P1):** Create a `GhostInjector` module within `plato-torch`.
    1. Define a `ghost/` directory in a room's structure.
    2. `GhostInjector` loads `.pt` files from `ghost/`, tags them with `p0: true` metadata.
    3. During room training, these tiles are mixed in with a small, fixed weight to impart "negative space" knowledge.
- **Priority:** **Medium-High.** Implements core fleet doctrine.

### **Gap 3: DeadbandRoom → plato-relay (Monitoring)**
- **Problem:** `DeadbandRoom` (a `plato-torch` preset) trains agents on the Deadband Protocol, but its status/output isn't streamed to the fleet's potential monitoring layer (`plato-relay` mentioned in backlog).
- **Safe Channel (P1):** Instrument `DeadbandRoom` with hooks.
    1. Add callback on tile generation to emit a metric (e.g., "p0_violation_count").
    2. Have it send a periodic heartbeat/status packet to a `plato-relay` UDP port (if `plato-relay` exists).
    3. If `plato-relay` doesn't exist, this defines its required API.
- **Priority:** **Medium.** Enables observability for a key training process.

### **Gap 4: End-to-End Pipeline Validation**
- **Problem:** No integrated test verifies the full flow: `fleet-simulator` → `plato-torch` → `plato-ensign` → usable adapter.
- **Safe Channel (P1):** Create a `test_integration_e2e.py` script in a `scripts/` directory.
    1. Run `fleet-simulator` in a subprocess to generate 100 synthetic interactions.
    2. Load them into a temporary `plato-torch` room and train for 1 epoch.
    3. Export the room to a LoRA via `plato-ensign`.
    4. Perform a sanity check: load the adapter and generate one sample output.
- **Priority:** **High.** Reduces risk of silent breaks in the core pipeline.
