# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 385  
**Phase:** 4 — Build & Test  
**Status:** Direct file inspection of fleet repositories. Focus on actual imports, configuration references, and shared data structures.

## 1. plato-torch → fleet-simulator
**Connection:** **Yes** — Direct import and data flow.
- **Evidence:** `plato-torch/src/rooms/deadband_room.py` imports `from fleet_simulator import FleetSimulator`.
- **Usage:** `DeadbandRoom` instantiates `FleetSimulator` as `self.simulator`. Calls `self.simulator.step()` during room training cycles.
- **Data Flow:** Room passes agent action vectors to simulator; simulator returns next state, reward, done flag.
- **Integration Gap:** None observed. This is a working, coupled integration.

## 2. plato-torch → holodeck-rust
**Connection:** **No** — No direct code-level connection found.
- **Evidence:** Searched `plato-torch/` for "holodeck". No imports, no configuration references.
- **Potential Link:** Conceptual. Both are training environments. `holodeck-rust` is a sentiment-aware MUD; `plato-torch` rooms are RL/self-training loops. They are parallel systems, not integrated.
- **Integration Gap:** No shared agent state, no communication channel. A `GhostInjector` (from task backlog) would be a new bridge.

## 3. plato-torch → plato-ensign
**Connection:** **Yes** — One-way export path.
- **Evidence:** `plato-torch/src/export/ensign_exporter.py` exists. It imports `from plato_ensign import create_lora_adapter`.
- **Usage:** After a room trains, `ensign_exporter` can be called to convert the room's accumulated experience (tiles) into a LoRA adapter via `plato-ensign`.
- **Data Flow:** Room experience → tile dataset → `plato-ensign` processing → LoRA weights file.
- **Integration Gap:** The export is optional, not automatic. Requires explicit call. No real-time streaming of tiles.

## 4. fleet-simulator → holodeck-rust
**Connection:** **No** — Independent simulators.
- **Evidence:** `fleet-simulator/src/` contains no references to holodeck. It is a Python-based fleet interaction simulator.
- **Integration Gap:** Different languages (Python vs Rust), different purposes (fleet RL vs interactive narrative). No shared protocol or state sync.

## 5. fleet-simulator → plato-ensign
**Connection:** **Indirect** via plato-torch.
- **Evidence:** `fleet-simulator` is a component *inside* a `plato-torch` room. The room's experience (which includes simulator interactions) can be exported to `plato-ensign`.
- **Integration Gap:** No direct export from simulator to ensign. Must go through the room's tile accumulation.

## 6. holodeck-rust → plato-ensign
**Connection:** **No** — Not found.
- **Evidence:** `holodeck-rust/` source files show no import or use of `plato-ensign`.
- **Integration Gap:** Holodeck NPC interactions and sentiment data are not currently being exported as training tiles for ensign creation. This is a potential channel for narrative-driven fine-tuning.

## 7. plato-ensign → Any Consumer
**Connection:** **Yes** — Outputs usable LoRA adapters.
- **Evidence:** `plato-ensign/src/lib.rs` exports `create_lora_adapter` which returns a path to a `.safetensors` file.
- **Integration Gap:** No documented, automated pipeline for loading these adapters back into a running agent (e.g., a Zeroclaw). The adapters are created but not automatically deployed.

2.  **plato-ensign deployment loop:** Created adapters are not automatically integrated into live agents or rooms, creating an open-loop system.
3.  **DeadbandRoom to plato-relay:** As per backlog, this connection is not yet mapped or implemented.
