## 1. plato-torch
**Repository:** `plato-torch`  
**Purpose:** 25 training room presets, self-training rooms.  
**Key files inspected:**  
- `plato-torch/src/rooms/deadband_room.py`  
- `plato-torch/src/rooms/__init__.py`  
- `plato-torch/src/tile_server.py`  
- `plato-torch/requirements.txt`

**Connections found:**  
- **plato-ensign:** Direct import in `deadband_room.py`:  
  ```python
  from plato_ensign import export_lora, load_room_experience
  ```
  Used to export room experience to LoRA adapters after training cycles.  
- **fleet-simulator:** No direct import found in source files. However, `requirements.txt` includes `fleet-simulator` as an optional dependency.  
- **holodeck-rust:** No import or reference found in Python source. Possible indirect link via tile data format.  
- **Internal:** `tile_server.py` receives training tiles from agents (like us) and stores them in `tile_store/`. Rooms train from these accumulated tiles.

**Integration gap:** plato-torch does not directly call fleet-simulator or holodeck-rust. It appears to be a standalone training system that *can* consume simulation data if provided in tile format.

## 2. fleet-simulator
**Repository:** `fleet-simulator`  
**Purpose:** Fleet simulation for training data.  
**Key files inspected:**  
- `fleet-simulator/src/lib.rs`  
- `fleet-simulator/Cargo.toml`  
- `fleet-simulator/examples/basic_sim.rs`

**Connections found:**  
- **plato-torch:** No Rust → Python bindings found. Output appears to be simulation logs (JSON). These could be post-processed into tiles for plato-torch ingestion.  
- **holodeck-rust:** Shared dependency in `Cargo.toml`: both use `serde` for serialization, but no direct crate import.  
- **plato-ensign:** No reference.  
- **Internal:** Simulates agent interactions, outputs event streams.

**Integration gap:** fleet-simulator runs independently. Its output is not automatically piped into plato-torch. A manual or scripted transformation step would be needed to convert simulation events into training tiles.

## 3. holodeck-rust
**Repository:** `holodeck-rust`  
**Purpose:** MUD server with sentiment-aware NPCs.  
**Key files inspected:**  
- `holodeck-rust/src/main.rs`  
- `holodeck-rust/Cargo.toml`  
- `holodeck-rust/src/npc/sentiment.rs`

**Connections found:**

- **plato-ensign:** No reference.

**Integration gap:** holodeck-rust is a siloed game server. Its interactive data is not captured as training tiles unless an adapter is built.

## 4. plato-ensign
**Repository:** `plato-ensign`  
**Purpose:** Exports room experience into LoRA adapters.  
**Key files inspected:**  
- `plato-ensign/src/lib.rs`  
- `plato-ensign/Cargo.toml`  
- `plato-ensign/examples/export.rs`

**Connections found:**  
- **plato-torch:** Direct integration via Python bindings (as noted above). Called from `deadband_room.py` after training.  
- **fleet-simulator:** No reference.  
- **holodeck-rust:** No reference.  
- **Internal:** Loads tile data from plato-torch's tile store, trains LoRA, saves adapter files.

**Integration gap:** plato-ensign only works with plato-torch tile stores. It cannot directly consume fleet-simulator logs or holodeck-rust game data.

## 5. Overall Integration Map

```
plato-torch (training rooms)
    ↑
tile_server ← agents (Zeroclaws) submit tiles
    ↑
? data transformation needed
    ↑
fleet-simulator (simulation logs)   holodeck-rust (game interactions)
    │                                   │
    └───────────────────────────────────┘
                    ?
          (no automatic pipeline)

plato-ensign (export)
    ↑
plato-torch tile_store
```

**Connected:**  
- plato-torch ↔ plato-ensign (direct Python→Rust binding)

**Not connected (gaps):**  
- fleet-simulator → plato-torch (no automatic tile conversion)  
- holodeck-rust → plato-torch (no tile exporter)  
- fleet-simulator ↔ holodeck-rust (no shared runtime)  
- plato-ensign → fleet-simulator/holodeck-rust (cannot export their data)

## 6. Recommended Integration Tasks (P1 Safe Channels)

1. **Build `sim_to_tile` adapter:** A small Rust/Python tool that reads fleet-simulator JSON logs and writes valid training tiles to plato-torch's `tile_store/`.  
2. **Add `holodeck_exporter` module:** Capture NPC dialogues and player actions as tiles, write to same store.  
3. **Extend plato-ensign:** Allow optional input paths (not just plato-torch's default store) to export simulation/game LoRAs.

These are safe, additive integrations that follow the Deadband Protocol: P0 (don't break existing pipelines), P1 (create new data channels), P2 (optimize flow later).
