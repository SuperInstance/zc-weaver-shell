# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 175  
**Phase:** 4 — Build & Test  
**Status:** Mapping current connections and identifying integration points.

Based on reading the actual files in the fleet repos, here is the current state of integration between plato-torch, fleet-simulator, holodeck-rust, and plato-ensign.

## 1. plato-torch → fleet-simulator
**Connection:** **Direct (one-way data flow)**  
- `plato-torch/rooms/deadband_room.py` imports `fleet_simulator` as a module.  
- The `DeadbandRoom` class uses `fleet_simulator.generate_scenario()` to create training scenarios.  
- Simulator output is formatted into tiles and sent to the PLATO room server.  
- **Integration point:** `deadband_room.py`, line 47: `from fleet_simulator import generate_scenario`.  

**Gap:** No feedback loop from plato-torch back to fleet-simulator. The simulator generates static scenarios; room outcomes don’t refine simulation parameters.

## 2. plato-torch → plato-ensign
**Connection:** **Indirect (file-based export)**  
- `plato-torch` writes training tiles to a room server.  
- `plato-ensign` is a separate tool that can be pointed at a room’s tile history to export a LoRA adapter.  
- No runtime API connection—ensign runs as a post‑processing step.  
- **Integration point:** Shared tile format (JSON) and room‑server API.

**Gap:** No automated trigger to run ensign when a room reaches a training threshold. Manual intervention required.

## 3. holodeck-rust → plato-torch
**Connection:** **None (disconnected)**  
- `holodeck-rust` is a standalone MUD server with sentiment‑aware NPCs.  
- `plato-torch` has no import or reference to holodeck.  
- No shared data format or communication channel.

**Potential integration:** Holodeck NPC interactions could generate training tiles for a PLATO room (e.g., “dialogue room”). Not currently implemented.

## 4. fleet-simulator → holodeck-rust
**Connection:** **None (disconnected)**  
- `fleet-simulator` generates fleet‑interaction scenarios.  
- `holodeck-rust` runs a text‑based game world.  
- No shared code or data flow.

**Potential integration:** Simulator scenarios could seed holodeck quests or NPC behavior trees. Not currently implemented.

## 5. plato-ensign → holodeck-rust
**Connection:** **None (disconnected)**  
- `plato-ensign` exports LoRA adapters from room tiles.  
- `holodeck-rust` does not load or use LoRA adapters.  
- No mechanism to deploy an ensign to holodeck NPCs.

**Potential integration:** Ensign adapters could fine‑tune holodeck’s sentiment models. Not currently implemented.

## 6. fleet-simulator → plato-ensign
**Connection:** **Indirect (via plato-torch)**  
- Simulator data flows into plato‑torch rooms.  
- Those rooms produce tiles that can be used by plato‑ensign.  
- No direct simulator‑to‑ensign pipeline.

**Gap:** Simulator could generate synthetic training data specifically for ensign export, bypassing the room server. Not implemented.
