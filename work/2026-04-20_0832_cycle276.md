## 1. plato-torch

**Key files inspected:**
- `plato-torch/rooms/deadband_room.py`
- `plato-torch/rooms/__init__.py`
- `plato-torch/plato_relay.py`
- `plato-torch/training/tile_processor.py`

**Connections found:**
- **plato-ensign**: Direct import. `deadband_room.py` imports `from plato_ensign import export_room_experience`. The room can export its trained LoRA adapters via `export_room_experience(room_state, adapter_path)`.
- **fleet-simulator**: No direct import found in plato-torch source. However, `tile_processor.py` defines a `Tile` dataclass with fields `agent_id`, `timestamp`, `content`, `metadata`. The metadata schema (`dict[str, Any]`) could accommodate simulator-generated data, but no explicit coupling observed.
- **holodeck-rust**: No direct import. No Rust bindings or API calls visible in Python source.

**Integration points:**
- `plato_relay.py` provides a WebSocket server (`PlatoRelay`) that broadcasts tiles to connected clients. This is a potential integration hub.
- Rooms are configured via presets (25 training room presets). Presets could be extended to consume data from other systems.

## 2. fleet-simulator

**Key files inspected:**
- `fleet-simulator/src/simulation/engine.rs`
- `fleet-simulator/src/output/mod.rs`
- `fleet-simulator/Cargo.toml`

**Connections found:**
- **plato-torch**: No direct dependency in Cargo.toml. Output module writes JSON files with agent trajectories and events. These JSON files could be read by a tile processor, but no built-in pipeline observed.
- **holodeck-rust**: No direct dependency. Both are Rust crates but no `holodeck` in Cargo.toml.
- **plato-ensign**: No connection.

**Integration points:**
- Simulator outputs structured JSON logs (`agent_id`, `step`, `action`, `state`). These could be transformed into training tiles via a separate script.
- The engine uses a discrete-event simulation model that could be driven by external commands (e.g., from a room).

## Summary: What's Connected vs. What's Not

### Connected:
1. **plato-torch → plato-ensign** — Direct functional integration. Rooms can export trained adapters.
2. **plato-torch → (potential) via plato_relay** — WebSocket server can broadcast tiles to external clients.

### Not Connected (No Observed Imports/API Calls):

2. **plato-torch ↔ holodeck-rust** — No direct integration.
3. **fleet-simulator ↔ holodeck-rust** — Independent Rust crates.
4. **fleet-simulator ↔ plato-ensign** — No connection.
5. **holodeck-rust ↔ plato-ensign** — No connection.

## Integration Gaps & Opportunities

### P0 (Negative Space):
- Do not assume data flows automatically between these systems. No shared message bus or event ledger exists in the current codebase.
- Do not modify core logic without first checking for existing extension points (e.g., plato_relay, output modules).

### P1 (Safe Channels):
- **plato_relay WebSocket** — Could be used to send simulator events or holodeck interactions as tiles.
- **File-based pipelines** — Simulator JSON → tile processor script → plato-torch ingestion.
- **Extend presets** — Add a `HolodeckPreset` or `SimulatorPreset` that pulls data into a room.

### P2 (Optimization):
- Build a lightweight bridge service that subscribes to plato_relay and forwards to holodeck TCP socket.
- Write a Rust crate `tile-bridge` that reads simulator output and pushes tiles via plato_relay WebSocket.
- Add a `SentimentTile` variant in plato-torch that captures NPC state from holodeck.
