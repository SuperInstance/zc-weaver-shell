## 1. plato-torch (Training Rooms)
**Location:** `plato-torch/`  
**Purpose:** 25 training room presets, self-training rooms.  
**Key Files:**
- `presets/deadband_room.py` — Deadband Protocol implementation
- `presets/__init__.py` — exports `DeadbandRoom`
- `plato_torch/__init__.py` — core room class definitions
- `requirements.txt` — lists `torch`, `transformers`, `datasets`

**Connections Observed:**
- **To plato-ensign:** `presets/deadband_room.py` imports `from plato_ensign import export_room`. Direct integration for LoRA export.
- **To fleet-simulator:** No direct imports found. Potential integration via shared data formats (JSON logs).
- **To holodeck-rust:** No direct imports. Possible bridge through sentiment data or NPC interaction logs.

**Integration Points:**
- `DeadbandRoom` generates training tiles (JSON) that can be consumed by `plato-ensign`.
- Room state can be serialized to disk, allowing external systems (fleet-simulator) to read.

## 2. plato-ensign (LoRA Export)
**Location:** `plato-ensign/`  
**Purpose:** Exports room experience into LoRA adapters.  
**Key Files:**
- `plato_ensign/__init__.py` — `export_room` function
- `setup.py` — depends on `torch`, `peft`, `transformers`

**Connections Observed:**
- **From plato-torch:** Imported in `deadband_room.py`.
- **To fleet-simulator/holodeck-rust:** No direct imports. Exports are file‑based (`.safetensors`).

**Integration Points:**
- Called by `DeadbandRoom` after training cycles.
- Outputs LoRA weights that could be loaded by inference services (not yet observed in other repos).

## 3. fleet-simulator (Fleet Simulation)
**Location:** `fleet-simulator/`  
**Purpose:** Generate training data through multi‑agent simulation.  
**Key Files:**
- `src/simulator.rs` — core simulation engine
- `Cargo.toml` — dependencies: `serde`, `tokio`, `rand`
- `examples/basic_sim.rs` — minimal example

**Connections Observed:**
- **To plato-torch/holodeck-rust:** No Rust‑to‑Python imports. Integration would require inter‑process communication (IPC) or shared data files.
- **To plato-ensign:** No connection found.

**Integration Points:**
- Simulator outputs JSON‑structured event logs (`simulation_output.json`).
- These logs could be read by `plato-torch` as training data, but no loader is currently implemented.

## 4. holodeck-rust (Sentiment‑Aware MUD)
**Location:** `holodeck-rust/`  
**Purpose:** MUD server with sentiment‑aware NPCs.  
**Key Files:**
- `src/server.rs` — TCP server, NPC management
- `src/npc/sentiment.rs` — sentiment logic
- `Cargo.toml` — dependencies: `serde`, `tokio`, `llm-chain` (for NLP)

**Connections Observed:**
- **To plato-torch/fleet-simulator/plato-ensign:** No direct imports. Standalone Rust service.
- **Potential external integration:** `llm-chain` dependency suggests possible connection to language models (DeepSeek, etc.).

**Integration Points:**
- NPC interaction logs are written to `logs/holodeck.log`.
- Sentiment scores could be fed into `plato-torch` rooms as training signals, but no bridge exists yet.
