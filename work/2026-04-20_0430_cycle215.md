## 2. Verified Connections (Actual Imports/References)

### A. **plato-torch → plato-ensign** ✅ **CONNECTED**
- **Evidence:** `plato-torch/presets/DeadbandRoom.py` line 47:
  ```python
  from ensign.export import export_to_lora
  ```
- **Usage:** DeadbandRoom can export its trained tile cache to a LoRA adapter via `export_to_lora(tile_cache, output_path)`.
- **Data flow:** `Tile` objects (from plato-torch) → LoRA weights (via plato-ensign).

### B. **fleet-simulator → holodeck-rust** ❌ **NOT CONNECTED**
- **Evidence:** No import of `holodeck` in `fleet-simulator/src/`.  
- **Evidence:** No shared `Cargo.toml` dependency.  
- **Gap:** Simulated agent interactions do not feed into the live MUD server. NPCs in holodeck cannot learn from simulated fleet events.

### C. **holodeck-rust → plato-torch** ❌ **NOT CONNECTED**
- **Evidence:** No Rust FFI bindings to Python in `holodeck-rust/`.  
- **Evidence:** No `pyo3` or `cpython` dependencies.  
- **Gap:** NPC sentiment data and player interactions are not captured as training tiles for plato-torch rooms.

### D. **plato-ensign → fleet-simulator** ❌ **NOT CONNECTED**
- **Evidence:** No Rust code in `plato-ensign/`; no Python imports in `fleet-simulator/`.  
- **Gap:** LoRA adapters cannot be directly loaded into the Rust-based simulator for agent behavior tuning.

### E. **plato-torch → fleet-simulator** ❌ **NOT CONNECTED**
- **Evidence:** Separate language stacks (Python vs Rust).  
- **Gap:** Training rooms cannot ingest simulated events as tile data.
