## 1. Verified Connections (What IS Connected)

### **plato-torch ↔ fleet-simulator**
- **Connection:** Direct import dependency.
- **Evidence:** `fleet-simulator/src/lib.rs` imports `plato_torch` crate.
- **Purpose:** Fleet simulator uses Plato's training room presets (e.g., `DeadbandRoom`) to generate synthetic training data.
- **Data Flow:** Simulated agent interactions → tile generation → room training loop.

### **plato-torch ↔ plato-ensign**
- **Connection:** Direct export pipeline.
- **Evidence:** `plato-ensign/src/main.rs` loads a trained room from `plato_torch::rooms::DeadbandRoom` and exports LoRA adapters.
- **Purpose:** Encodes room experience (accumulated tiles) into portable, fine-tuned model weights.
- **Data Flow:** Trained room state → LoRA adapter files (.safetensors).

### **holodeck-rust ↔ (Indirect via shared patterns)**
- **Current State:** No direct crate dependencies found in Cargo.toml files.
- **Shared Pattern:** Both `holodeck-rust` (MUD server) and `fleet-simulator` generate narrative interaction data that could be formatted as tiles.
- **Potential Bridge:** Tile schema (`Tile` struct in `plato-torch`) is a candidate universal format.

## 2. Integration Gaps (What Is NOT Connected)

### **Gap 1: holodeck-rust → plato-torch**
- **Issue:** No pipeline to send live holodeck player/NPC interactions to a Plato room as training tiles.
- **Impact:** Rich, sentiment-aware narrative data from the MUD is isolated.
- **Actionable Task:** *Wire GhostInjector into holodeck* (from backlog).  
  - **Proposal:** Add a `GhostInjector` service to holodeck that serializes `Event` logs (e.g., `player_say`, `npc_react`) into `Tile` JSON and POSTs to a room's `/ingest` endpoint.

### **Gap 2: plato-torch rooms → holodeck-rust**
- **Issue:** No mechanism to deploy a trained ensign (LoRA) as an NPC personality in the MUD.
- **Impact:** Cannot close the loop: train on holodeck data, then deploy trained behavior back into holodeck.
- **Actionable Task:** *Connect DeadbandRoom to plato-relay* (from backlog).  
  - **Proposal:** Extend `plato-relay` (or create `holodeck-relay`) to load a LoRA adapter, wrap it with a lightweight inference runtime, and expose it as an NPC brain via holodeck's `Brain` trait.

### **Gap 3: fleet-simulator → holodeck-rust**
- **Issue:** Simulated fleet data (multi-agent exercises) does not feed into the live MUD world.
- **Impact:** Missed opportunity for immersive training scenarios where simulated agents interact with human players.
- **Actionable Task:** *Test end-to-end pipeline* (from backlog).  
  - **Proposal:** Design a scenario where `fleet-simulator` generates a command-staff exercise, formats it as a quest narrative, and injects it into holodeck as a dynamic world event.

### **Gap 4: plato-ensign → cudaclaw / flux-runtime**
- **Issue:** Exported LoRA adapters are not automatically deployed to GPU-resident (`cudaclaw`) or self-assembling VM (`flux-runtime`) environments.
- **Impact:** Specialized knowledge (ensigns) remains file-based, not operational.
- **Actionable Task:** *Document integration points* (from backlog).  
  - **Proposal:** Write a spec for `ensign-deploy`: a tool that takes a .safetensors file and a target runtime config, and launches a microservice wrapping the adapter (e.g., a `cudaclaw` module or `flux` bytecode bundle).
