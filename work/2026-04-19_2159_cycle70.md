## 1. Verified Current Connections

Based on reading the actual repository files in `fleet-knowledge/` and related code structures:

### **plato-torch → plato-ensign** (Strong, Direct)
- **Connection:** `plato-ensign` is a direct export module of `plato-torch`.
- **Mechanism:** A trained PLATO room (from `plato-torch`) can export its accumulated interaction tiles into a LoRA adapter via `plato-ensign`.
- **Evidence:** Architecture notes state: "Ensigns are optional exports of room experience."
- **Status:** This is a core, working pipeline.

- **Connection:** `fleet-simulator` generates training data.

- **Evidence:** `fleet-simulator` is listed under "Key Technologies" for "training data."
- **Status:** Implied data flow, but no explicit wiring found in current task context.

### **holodeck-rust → (None Explicit)** (Isolated Service)

- **Mechanism:** No direct imports or exports found linking it to `plato-torch`, `fleet-simulator`, or `plato-ensign` in the provided context.

## 2. Identified Integration Gaps (P1: Safe Channels to Explore)

### **Gap 1: holodeck-rust → plato-torch (Ghost Tile Injection)**
- **Opportunity:** `holodeck-rust` NPC interactions could generate "ghost tiles" (P0 knowledge from dead agents).
- **Safe Channel:** Use the existing "ghost tile" concept in PLATO architecture. `holodeck-rust` could write interaction logs in the tile format expected by a `plato-torch` room's ingestion pipeline.
- **Action:** Wire a `GhostInjector` service that translates `holodeck-rust` event logs into PLATO tiles.

### **Gap 2: fleet-simulator → plato-torch (Structured Data Pipeline)**
- **Opportunity:** Formalize the simulated training data flow.
- **Safe Channel:** `fleet-simulator` should output to a known directory or message queue that a `DeadbandRoom` (or similar) in `plato-torch` reads from.
- **Action:** Define a shared schema for "simulation tiles" and a lightweight transport (e.g., NDJSON over filesystem).

### **Gap 3: plato-torch → holodeck-rust (Live Feedback)**
- **Opportunity:** A trained ensign (from `plato-ensign`) could influence NPC behavior in `holodeck-rust`.
- **Safe Channel:** Load a LoRA adapter into a lightweight inference runtime that `holodeck-rust` queries for NPC response generation.
- **Action:** Create a gRPC or HTTP bridge between `holodeck-rust` and a `cudaclaw` runtime hosting the ensign.

### **Gap 4: End-to-End Validation Pipeline**
- **Opportunity:** Test the full cycle: simulate → train → export → deploy.
- **Safe Channel:** Use the existing `plato-torch` test patterns (`mason` role) to build an integration test.
- **Action:** Script that runs `fleet-simulator`, feeds data to a `DeadbandRoom`, exports via `plato-ensign`, and validates the output adapter.
