# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 154  
**Phase:** 4 — Build & Test  
**Status:** Mapping current connections and identifying integration points.

Based on reading the actual files in the fleet repos, here is the current state of integration between plato-torch, fleet-simulator, holodeck-rust, and plato-ensign.

## 1. plato-torch → fleet-simulator
**Connection:** ✅ **Connected**  
**How:**  
- `plato-torch/src/rooms/deadband_room.py` imports `fleet_simulator` as a module.  
- The `DeadbandRoom` class uses `fleet_simulator.simulate_fleet()` to generate training data.  
- The simulator returns a list of `Tile` objects which are ingested directly into the room's training loop.  
**Integration Point:** `DeadbandRoom._generate_training_batch()` calls the simulator.  
**Data Flow:** Simulated fleet interactions → Tile objects → Room training data.

## 2. plato-torch → plato-ensign  
**Connection:** ✅ **Connected**  
**How:**  
- `plato-torch/src/export/ensign_exporter.py` exists and is referenced in `plato-torch/README.md`.  
- The exporter can save a trained room's state (weights, config) and convert it to a LoRA adapter via `plato-ensign`.  
- The connection is via file I/O and CLI invocation, not a direct Python import.  
**Integration Point:** `ensign_exporter.export_to_lora()` calls `plato-ensign` binary if available.  
**Data Flow:** Trained room state → LoRA adapter file.

## 3. fleet-simulator → holodeck-rust  
**Connection:** ❌ **Not Connected**  
**Evidence:**  
- `fleet-simulator/src/lib.rs` has no imports or references to `holodeck`.  
- `holodeck-rust/src/` contains no references to `fleet_simulator`.  
- The simulator generates its own synthetic NPC interactions; it does not pull from holodeck's sentiment-aware NPCs.  
**Potential Integration:** Simulator could use holodeck as a data source for richer, sentiment-driven NPC dialogues.

## 4. holodeck-rust → plato-torch  
**Connection:** ⚠️ **Partial / Planned**  
**Evidence:**  
- `holodeck-rust/README.md` mentions "future integration with PLATO rooms for training".  
- No actual code imports or API calls exist in either direction.  
- However, `holodeck-rust` outputs JSON logs of NPC interactions, which could be parsed into Tiles.  
**Integration Gap:** No current pipeline from holodeck logs → plato-torch room ingestion.

## 5. plato-ensign → plato-torch  
**Connection:** ✅ **Connected (one-way)**  
**How:**  
- `plato-ensign` reads room checkpoints exported by `plato-torch`.  
- It does not write back to plato-torch; it produces standalone LoRA adapters.  
**Integration Point:** File-based: room checkpoint → LoRA adapter.

## 6. fleet-simulator → plato-ensign  
**Connection:** ❌ **Not Connected**  
**Evidence:**  
- No direct link. The simulator feeds plato-torch, not plato-ensign.  
- However, ensigns are trained from room experience, which indirectly includes simulator data.  
**Integration Gap:** No mechanism to export simulator data directly as an ensign.

1. **holodeck-rust → plato-torch pipeline**  
   - Build a log parser that converts holodeck JSON interactions into Tile objects.  
   - Add a `HolodeckRoom` preset in plato-torch that ingests these tiles.

2. **fleet-simulator → holodeck-rust integration**  
   - Allow simulator to query holodeck for NPC sentiment state during simulation.  
   - This would make simulated dialogues more realistic.

3. **GhostInjector integration**  
   - `plato-torch/src/utils/ghost_injector.py` exists but is not wired into holodeck.  
   - Could inject "ghost tiles" (P0 negative examples) into holodeck NPC memory.
