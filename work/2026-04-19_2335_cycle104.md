## 1. Verified Connections (What's Connected)

### plato-torch ↔ fleet-simulator
- **Connection:** Direct integration via `fleet-simulator` as a training data source.
- **Evidence:** `plato-torch/rooms/deadband_room.py` imports `fleet_simulator` and uses `FleetSimulatorEnv` to generate simulation episodes.
- **Purpose:** Fleet simulation provides structured interaction episodes (state, action, reward) that become training tiles for PLATO rooms.
- **Data Flow:** Simulation episodes → tile format → room training buffer.

### plato-torch ↔ plato-ensign
- **Connection:** Optional export pipeline.
- **Evidence:** `plato-ensign/export.py` imports `Room` from `plato-torch` and calls `room.export_experience()` to extract LoRA adapters.
- **Purpose:** Rooms accumulate experience (tiles); ensigns package that experience into portable specialist models.
- **Data Flow:** Room training buffer → experience snapshot → LoRA weights.

### holodeck-rust ↔ (Indirect via tiles)
- **Connection:** No direct code imports found between holodeck-rust and the other three repos.
- **Indirect Link:** Holodeck is designed as a sentiment-aware MUD server that could generate interaction tiles. These tiles could be consumed by plato-torch rooms, but no current ingestion code exists.
- **Potential Bridge:** The `Tile` format (likely defined in plato-torch) would need to be implemented in holodeck-rust for tile compatibility.

## 2. Integration Gaps (What's Not Connected)

### Gap 1: holodeck-rust → plato-torch (No tile ingestion)
- **Issue:** holodeck-rust produces NPC interactions and sentiment logs, but these are not formatted as PLATO tiles or sent to a room.
- **Missing:** A `TileWriter` or API endpoint in holodeck-rust that converts interactions to tile format and submits to a plato-torch room server.
- **Impact:** Holodeck's rich social simulations are isolated from fleet training pipelines.

### Gap 2: fleet-simulator → holodeck-rust (No shared environment)
- **Issue:** fleet-simulator models fleet dynamics; holodeck-rust models social NPCs. No shared environment or character exchange.
- **Missing:** A common entity/agent representation that could move between simulation (fleet-simulator) and social world (holodeck-rust).
- **Impact:** Fleet agents cannot "enter" the holodeck; social NPCs cannot influence fleet simulations.

### Gap 3: plato-ensign → holodeck-rust (No ensign deployment)
- **Issue:** Ensigns (LoRA adapters) exported from plato-ensign are not deployed to holodeck-rust NPCs.
- **Missing:** A mechanism to load a `.safetensors` LoRA into holodeck-rust's NPC AI models.
- **Impact:** Specialist knowledge trained in rooms cannot enhance holodeck NPC behavior.

### Gap 4: Deadband Protocol ↔ holodeck-rust (No P0/P1/P2 integration)
- **Issue:** The Deadband Protocol (map negative space, find safe channels) is implemented in plato-torch's `DeadbandRoom` but not in holodeck-rust's decision loops.
- **Missing:** Sentiment-aware NPCs do not use deadband reasoning for action selection.
- **Impact:** Holodeck NPCs may not exhibit the same caution/optimization patterns as fleet agents.
