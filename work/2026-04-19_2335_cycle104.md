## 1. Verified Connections (Based on Code & Documentation)

### plato-torch ↔ fleet-simulator
- **Connection:** Strong, direct integration via `fleet_simulator` module.
- **Evidence:** `plato-torch/src/fleet_simulator/` contains `simulator.py`, `agent.py`, `environment.py`. These are imported and used within plato-torch training rooms (e.g., `DeadbandRoom`).
- **Purpose:** Generates synthetic training data (agent interactions, environment states) for training rooms. Simulates fleet agent behavior to produce interaction tiles.
- **Data Flow:** fleet-simulator runs simulations → outputs interaction logs/events → plato-torch ingests these as training tiles.

### plato-torch ↔ plato-ensign
- **Connection:** Direct export pipeline.
- **Evidence:** `plato-torch/src/ensign/` contains `exporter.py`, `lora_adapter.py`. `plato-ensign` is a separate repo but designed to consume trained room checkpoints from plato-torch.
- **Purpose:** Exports trained room experience (weights, adapter configs) into LoRA adapters for deployment.
- **Data Flow:** plato-torch room trains → checkpoint saved → plato-ensign processes checkpoint → produces LoRA adapter files.

### holodeck-rust ↔ (Indirect via plato-torch)
- **Connection:** Indirect; potential data source for plato-torch.
- **Evidence:** No direct import found in plato-torch codebase. However, holodeck-rust is a "sentiment-aware NPC" MUD server that could generate interaction logs.
- **Gap:** No current pipeline from holodeck-rust logs to plato-torch training tiles. This is a wiring opportunity.

### fleet-simulator ↔ holodeck-rust
- **Connection:** None observed.
- **Evidence:** Separate codebases, no shared modules or imports. Different languages (Python vs Rust).
- **Gap:** Fleet simulations could potentially drive NPC behavior in holodeck, or holodeck logs could enrich simulation data. Not connected.

### plato-ensign ↔ holodeck-rust / fleet-simulator
- **Connection:** None observed.
- **Evidence:** plato-ensign appears to be a downstream export tool only; no imports of holodeck or simulator.
- **Gap:** Ensigns (trained adapters) could be deployed back into holodeck NPCs or fleet-simulator agents. Not implemented.

## 2. Integration Gaps & Wiring Opportunities

### Gap 1: holodeck-rust → plato-torch tile pipeline
- **Problem:** Holodeck generates rich NPC interaction logs with sentiment data, but these are not currently fed into plato-torch as training tiles.
- **Wiring Idea:** Create a `HolodeckLogIngester` module in plato-torch that reads holodeck log files (or listens via socket), formats them as interaction tiles, and injects them into a training room.

### Gap 2: plato-ensign adapters → holodeck-rust NPCs
- **Problem:** Trained ensigns (LoRA adapters) are exported but not deployed back into live systems like holodeck.
- **Wiring Idea:** Add a `load_ensign` function to holodeck-rust NPC logic that can apply a LoRA adapter to influence NPC behavior, dialogue, or sentiment responses.

### Gap 3: fleet-simulator ↔ holodeck-rust bidirectional sync
- **Problem:** Two separate agent simulation environments that don't share state.
- **Wiring Idea:** Implement a lightweight API bridge (REST or WebSocket) allowing fleet-simulator agents to "enter" holodeck as players/NPCs, and holodeck NPCs to be mirrored in fleet-simulator scenarios.

### Gap 4: GhostInjector (plato-torch) → holodeck-rust
- **Problem:** Ghost tiles (P0 knowledge from dead agents) are stored in plato-torch but not utilized in holodeck.
- **Wiring Idea:** Build a `GhostInjector` service that periodically reads ghost tiles from plato-torch and injects them as "memories" or behavior constraints into holodeck NPCs.
