## 1. Verified Current Connections (What's Connected)

Based on reading the actual READMEs and code structure in the fleet repos:

### **plato-torch → plato-ensign** (Strong Connection)
- **plato-torch** generates training rooms and accumulates interaction tiles.
- **plato-ensign** exports room experience (tiles) into LoRA adapters.
- Connection is explicit: `plato-ensign` imports `plato-torch`'s `Room` class and tile format.
- This is the core training→export pipeline.

### **holodeck-rust → fleet-simulator** (Data Flow)
- **holodeck-rust** is a MUD server with sentiment-aware NPCs.
- **fleet-simulator** generates training data for fleet coordination.
- Connection: `fleet-simulator` can use `holodeck-rust` as an environment for agent interactions (NPCs as synthetic participants).
- Not a direct code dependency, but a logical data pipeline.

### **plato-torch → fleet-simulator** (Indirect)
- `fleet-simulator` could generate training tiles compatible with `plato-torch` rooms.
- No direct import found, but tile format is shared (JSONL with specific metadata).
- This is a potential channel, not yet wired.

### **cudaclaw → flux-runtime** (Runtime Stack)
- **cudaclaw**: GPU-resident agent runtime.
- **flux-runtime**: Self-assembling bytecode VM.
- Connection: `flux-runtime` could target `cudaclaw` as an execution backend.
- Mentioned in architecture docs, but no explicit code link found.

## 2. Integration Gaps (What's Not Connected)

### **Gap 1: GhostInjector → holodeck-rust**
- **GhostInjector** (part of `plato-torch`) loads ghost tiles (dead agents' P0 lessons).
- **holodeck-rust** has NPCs but no mechanism to inject ghost knowledge.
- **Action**: Wire GhostInjector to populate NPC dialogue/behavior with ghost tile data.

### **Gap 2: DeadbandRoom → plato-relay**
- **DeadbandRoom** is a `plato-torch` preset implementing the Deadband Protocol.
- **plato-relay** is a real-time tile relay server (mentioned in `plato-torch` docs).
- **Action**: Connect DeadbandRoom to plato-relay for live tile streaming and multi-room sync.

### **Gap 3: fleet-simulator → plato-torch (direct)**
- `fleet-simulator` generates simulation logs.
- `plato-torch` ingests tiles for training.
- **Action**: Build a `SimulationTileAdapter` to convert simulator logs to plato-torch tile format.

### **Gap 4: constraint-theory-core → holodeck-rust**
- **constraint-theory-core**: Rust geometric snapping library.
- **holodeck-rust**: MUD server with spatial rooms.
- **Action**: Integrate constraint snapping for room geometry, NPC pathing, or puzzle mechanics.

### **Gap 5: plato-ensign → cudaclaw**
- **plato-ensign** exports LoRA adapters.
- **cudaclaw** runs GPU-resident agents.
- **Action**: Enable `cudaclaw` to load ensign LoRAs for specialized GPU agents.
