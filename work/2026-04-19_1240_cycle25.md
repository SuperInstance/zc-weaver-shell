## 2. Identified Integration Gaps (P1 Safe Channels to Explore)

### Gap 1: holodeck-rust → plato-torch Tile Pipeline
- **Problem:** NPC interactions and player sentiment logs are lost as training data.  
- **Safe Channel:** Add a `TileLogger` module to holodeck-rust that formats interactions (NPC dialogue, player commands, sentiment scores) into the plato-torch tile schema. Write to a shared volume or message queue.  
- **Action:** Prototype a Rust struct `HolodeckTile` that implements `Serialize` to JSON matching plato-torch's expected ingestion format.

### Gap 2: fleet-simulator → plato-torch Direct Feed
- **Problem:** Simulation events (agent decisions, fleet maneuvers) require manual curation to become tiles.  
- **Safe Channel:** Create a `SimulationTileAdapter` in fleet-simulator that taps event streams, filters for pedagogically useful moments (e.g., Deadband Protocol violations), and emits them as tiles.  
- **Action:** Examine fleet-simulator output format; define a mapping from simulation event → training tile metadata.

### Gap 3: plato-ensign LoRA → Runtime Deployment
- **Problem:** Trained ensigns (LoRA adapters) are not loaded into any runtime (cudaclaw, flux-runtime) for inference.  
- **Safe Channel:** Design a `EnsignRegistry` service that stores, versions, and serves LoRA weights. Extend cudaclaw or flux-runtime to dynamically pull adapters for specialized tasks.  
- **Action:** Draft API spec for EnsignRegistry (POST /upload, GET /{ensign_id}/weights).

### Gap 4: Deadband Protocol Visualization ↔ Actual Rooms
- **Problem:** The Deadband Protocol is doctrine, but no room preset visually demonstrates P0/P1/P2 in action.  
- **Safe Channel:** Create a `DeadbandVisualRoom` preset in plato-torch that uses a simple 2D grid environment. Agents navigate, and their phase transitions (P0→P1→P2) are logged as annotated tiles.  
- **Action:** Fork an existing plato-torch room preset (e.g., GridWorld) and inject Deadband phase tracking.
