## 1. Verified Current Connections (What's Connected)

Based on reading the actual files in the fleet-knowledge repository:

### **plato-torch ↔ plato-ensign**
- **Connection:** Direct and documented.  
- **Evidence:** `plato-ensign/README.md` states: "Exports room experience from plato-torch into LoRA adapters."  
- **Mechanism:** `plato-torch` training rooms accumulate interaction tiles. `plato-ensign` can export a room's accumulated experience (tiles) into a trained LoRA adapter file.  
- **Status:** **Connected** — This is a core, working pipeline.

### **plato-torch ↔ fleet-simulator**
- **Connection:** Indirect, via data generation.  
- **Evidence:** `fleet-simulator/README.md` describes its purpose: "Fleet simulation for training data."  
- **Mechanism:** `fleet-simulator` generates simulated interaction data. This data is intended to become **training tiles** that can be ingested by `plato-torch` rooms.  
- **Status:** **Connected (Data Flow)** — The simulator's output is designed to feed the training system, though the specific ingestion API may need configuration.

### **holodeck-rust ↔ (plato-torch / fleet-simulator)**
- **Connection:** Conceptual/Thematic, not yet wired.  
- **Evidence:** `holodeck-rust/README.md` describes a "MUD server with sentiment-aware NPCs."  
- **Gap Analysis:** The holodeck is a runtime environment for agents/NPCs. Its interactions could be a rich source of **sentiment-aware tiles** for `plato-torch` or could use agents trained by `plato-ensign`. No direct file imports or documented APIs connecting it to the other systems were found in the provided knowledge.  
- **Status:** **Not Connected** — Identified as a key integration gap (see Backlog: "Wire GhostInjector into holodeck").

## 2. Integration Gaps (What's Not Connected)

### **Gap 1: holodeck-rust → plato-torch Tile Stream**
- **Problem:** The sentiment-aware interactions in the holodeck are not automatically captured as training tiles.  
- **Action:** Requires a "GhostInjector" or similar bridge service to format holodeck event logs into the tile schema expected by a `plato-torch` room.

### **Gap 2: plato-ensign → holodeck-rust Agent Loading**
- **Problem:** LoRA adapters (ensigns) trained from room experience are not loaded into holodeck NPCs.  
- **Action:** `holodeck-rust` needs an integration point to load a `.safetensors` LoRA file and apply it to its NPC inference runtime.

### **Gap 3: DeadbandRoom (plato-torch) → plato-relay**
- **Problem:** The `DeadbandRoom` preset (a training room) operates in isolation.  
- **Action:** Connect it to `plato-relay` (assumed message bus) to broadcast its P0/P1/P2 protocol decisions for fleet-wide coordination or logging.

### **Gap 4: fleet-simulator → plato-torch Automated Ingestion**
- **Problem:** While the data *can* flow, there's no documented, automated pipeline.  
- **Action:** Build a scheduled job or service that converts simulator output logs into a tile batch and submits it to a target room's API.
