# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 109  
**Phase:** 4 — Build & Test  
**Status:** Mapping current connections and identifying integration points.

Based on reading the actual files in the fleet repos, here is the current state of integration between **plato-torch**, **fleet-simulator**, **holodeck-rust**, and **plato-ensign**.

## 1. Verified Connections (What's Connected)

### plato-torch ↔ fleet-simulator
- **Connection:** Strong, bidirectional data flow.
- **Evidence:**  
  - `plato-torch/src/rooms/deadband_room.py` imports `fleet_simulator` and uses `FleetSimulator` class to generate training data.  
  - `fleet-simulator/src/lib.rs` exports a Python module via `pyo3`, exposing `FleetSimulator` to Python.  
  - The `DeadbandRoom` preset in plato-torch calls `simulator.generate_scenario()` to create P0/P1/P2 training examples.
- **Data Flow:** Fleet simulation scenarios → plato-torch training tiles.

### plato-torch ↔ plato-ensign
- **Connection:** Direct export pipeline.
- **Evidence:**  
  - `plato-ensign/src/main.rs` reads `training_tiles.json` produced by plato-torch rooms.  
  - `plato-torch/src/export.py` contains `export_to_ensign()` function that converts room experience to LoRA adapters.  
  - The ensign CLI has a `--room-dir` argument pointing to plato-torch room output directories.
- **Data Flow:** plato-torch room training logs → plato-ensign LoRA adapters.

### holodeck-rust ↔ (Indirect via plato-torch)
- **Connection:** Planned but not yet implemented.
- **Evidence:**  
  - `holodeck-rust/README.md` states: "Sentiment-aware NPCs for training rooms."  
  - `plato-torch/docs/ROADMAP.md` lists "Integrate holodeck NPCs as training partners" as a future feature.  
  - No actual imports or API calls between them in current code.
- **Data Flow:** None currently.

## 2. Integration Gaps (What's Not Connected)

### holodeck-rust ↔ fleet-simulator
- **Gap:** No connection.
- **Evidence:**  
  - `holodeck-rust/src/main.rs` is a standalone MUD server with no imports from fleet-simulator.  
  - `fleet-simulator` has no references to holodeck or NPCs in its code.
- **Impact:** NPCs cannot participate in fleet simulation scenarios.

### holodeck-rust ↔ plato-ensign
- **Gap:** No connection.
- **Evidence:**  
  - `plato-ensign` only processes plato-torch tile formats.  
  - `holodeck-rust` outputs game state logs in different format (JSON lines).
- **Impact:** NPC interactions cannot be exported as ensign training data.

### fleet-simulator ↔ plato-ensign
- **Gap:** Indirect only via plato-torch.
- **Evidence:**  
  - `plato-ensign` reads tiles, not raw simulation data.  
  - No direct export path from fleet-simulator to ensign adapters.
- **Impact:** Simulation scenarios must pass through plato-torch rooms to become training data.
