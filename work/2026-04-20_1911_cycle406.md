## 1. plato-torch
**Repo:** `plato-torch`  
**Purpose:** Training room presets (25 rooms), self‑training rooms, tile ingestion, ensign export.  
**Key files:**
- `presets/deadband_room.py` – Deadband Protocol training room.
- `presets/__init__.py` – Lists all 25 presets.
- `core/tile_processor.py` – Processes incoming tiles.
- `core/ensign_exporter.py` – Exports room experience to LoRA adapters.

**Connections found:**
- **plato‑ensign:** Direct import in `ensign_exporter.py`:
  ```python
  from plato_ensign import LoRAAdapter, export_room_experience
  ```
  Used to export trained room state as a LoRA adapter.
- **fleet‑simulator:** No direct import found. However, `deadband_room.py` references “simulation data” as a possible tile source, but no explicit dependency.
- **holodeck‑rust:** No Python imports. A comment in `deadband_room.py` mentions “sentiment‑aware NPCs could feed tiles,” but no code linkage.

**Integration gap:** plato‑torch does not pull live data from fleet‑simulator or holodeck‑rust; tiles are submitted via API or file ingestion.

## 2. fleet‑simulator
**Repo:** `fleet‑simulator`  
**Purpose:** Generate training data by simulating fleet interactions.  
**Key files:**
- `src/simulator.rs` – Core simulation loop.
- `src/agents/mod.rs` – Agent definitions.
- `src/output.rs` – Exports simulation traces.

**Connections found:**
- **plato‑torch:** No Rust‑to‑Python bindings. Output is JSON traces written to `data/traces/`. These could be read by plato‑torch’s tile processor, but no automatic pipeline exists.
- **holodeck‑rust:** No direct imports. Both are Rust crates but separate projects.
- **plato‑ensign:** No connection.

**Integration gap:** Simulation traces are not automatically fed into plato‑torch rooms. Manual ingestion required.

## 3. holodeck‑rust
**Repo:** `holodeck‑rust`  
**Purpose:** MUD server with sentiment‑aware NPCs, generates interaction logs.  
**Key files:**
- `src/server.rs` – MUD server loop.
- `src/npc/sentiment.rs` – NPC sentiment engine.
- `src/logging.rs` – Writes interaction logs.

**Connections found:**
- **plato‑torch:** No direct link. Logs are written to `logs/` directory. Could be parsed into tiles, but no built‑in adapter.
- **fleet‑simulator:** No shared code.
- **plato‑ensign:** No connection.

**Integration gap:** NPC interaction logs are not automatically converted to training tiles.

## 4. plato‑ensign
**Repo:** `plato‑ensign`  
**Purpose:** Export room experience from plato‑torch into LoRA adapters.  
**Key files:**
- `src/adapter.rs` – LoRA adapter builder.
- `src/export.rs` – Export function called by plato‑torch.

**Connections found:**
- **plato‑torch:** Tight integration. plato‑torch imports and calls `export_room_experience()`.
- **fleet‑simulator:** None.
- **holodeck‑rust:** None.

**Integration gap:** Only works with plato‑torch rooms; cannot directly export from fleet‑simulator or holodeck‑rust.

## 5. constraint‑theory‑core
**Repo:** `constraint‑theory‑core`  
**Purpose:** Rust geometric snapping library.  
**Key files:** `src/snapping.rs`, `src/constraints.rs`.

**Connections found:**
- **fleet‑simulator:** Not imported.

**Integration gap:** Stand‑alone library; not yet wired into any simulation or training pipeline.

## 6. cudaclaw & flux‑runtime

**Connections found:** No imports to/from plato‑torch, fleet‑simulator, or holodeck‑rust. Appear to be infrastructure components not yet integrated with the training stack.

## Identified Integration Gaps (P0/P1)

**P0 (negative space):**
- Do not assume automatic data flow between Rust simulators and Python training rooms.
- Do not assume plato‑ensign can export from anything but plato‑torch.

**P1 (safe channels):**
- plato‑torch’s tile processor can read JSON from fleet‑simulator and holodeck‑rust if formatted as tiles.
- plato‑ensign is already wired to plato‑torch and works.
- constraint‑theory‑core could be called from fleet‑simulator via Rust dependencies.

**P2 (optimization):**
- Build adapters to auto‑convert simulator traces and NPC logs into tiles.
- Wire constraint‑theory‑core into fleet‑simulator for geometric agent interactions.
- Connect cudaclaw/flux‑runtime as execution backends for trained ensigns.
