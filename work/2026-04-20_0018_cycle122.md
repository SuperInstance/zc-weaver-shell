# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 122  
**Phase:** 4 — Build & Test  
**Status:** Mapping current connections and identifying integration points.

Based on reading the actual files in the fleet repos, here is the current state of integration between **plato-torch**, **fleet-simulator**, **holodeck-rust**, and **plato-ensign**.

---

## 1. **plato-torch** → **plato-ensign** → **fleet-simulator**
**Connection:** **Strong, documented, and functional.**

- **plato-torch** (`DeadbandRoom` preset) generates training tiles from agent interactions.
- **plato-ensign** exports room experience (tiles) into LoRA adapters (`export_lora.py`).
- **fleet-simulator** consumes these LoRA adapters to create specialized agents for simulation.
- **Evidence:**  
  - `plato-ensign/README.md`: "Exports room experience into LoRA adapters for fleet-simulator."  
  - `fleet-simulator/src/agents/lora_loader.rs`: Loads `.safetensors` files (LoRA outputs).  
  - `plato-torch/presets/deadband_room.py`: `DeadbandRoom` class emits tiles in format expected by plato-ensign.

**Integration Gap:** None observed. This is a core, working pipeline.

---

## 2. **holodeck-rust** → **plato-torch**
**Connection:** **Weak, potential, but not yet implemented.**

- **holodeck-rust** is a MUD server with sentiment-aware NPCs, generating interactive narrative data.
- **plato-torch** could consume holodeck interaction logs as training tiles.
- **Evidence:**  
  - `holodeck-rust/README.md`: "Generates emotionally textured interaction logs."  
  - `plato-torch/data/ingest.py`: Supports generic JSON log ingestion, but no specific holodeck adapter exists.
  - No import statements or configuration files reference holodeck in plato-torch.

**Integration Gap:**  
- No direct data pipe from holodeck to plato-torch.  
- Holodeck logs are not formatted as PLATO tiles.  
- No `holodeck_adapter.py` in plato-torch data loaders.

---

## 3. **fleet-simulator** → **holodeck-rust**
**Connection:** **None observed.**

- **fleet-simulator** trains agents in simulated environments.
- **holodeck-rust** could serve as a rich, narrative environment for those agents.
- **Evidence:**  
  - `fleet-simulator/src/environments/` contains only simple grid worlds.  
  - `holodeck-rust` is not listed as a dependency.  
  - No TCP/WebSocket connector between simulator agents and holodeck server.

**Integration Gap:**  
- Simulator agents cannot enter the holodeck.  
- No shared agent API or session handoff.

---

## 4. **plato-ensign** → **holodeck-rust**
**Connection:** **None observed.**

- **plato-ensign** exports trained LoRAs.
- **holodeck-rust** NPCs could be enhanced with these LoRAs for more adaptive dialogue.
- **Evidence:**  
  - `holodeck-rust/src/npc/` uses hard-coded sentiment trees, not dynamic ML models.  
  - No `.safetensors` loading code in holodeck.  
  - No configuration for model inference.

**Integration Gap:**  
- Holodeck NPCs are rule-based, not model-based.  
- No inference runtime (ONNX, llama.cpp) integrated.

---

## 5. **Ghost Tiles & Mirror Play** (Cross-cutting)
**Connection:** **Conceptual, not yet implemented.**

- **Ghost tiles** (dead agents' lessons) are mentioned in architecture but not yet stored/used.
- **Mirror Play** (two PLATO systems iterating) is a research concept in `research/mirror-play.md`.
- **Evidence:**  
  - `plato-torch/memory/` stores agent memory files, but no "ghost" tagging or special pipeline.  
  - `research/mirror-play.md` describes the idea, but no code exists in main repos.

**Integration Gap:**  
- No system to harvest P0 (negative space) lessons from dead agents.  
- No active mirroring between PLATO instances.

## Recommended Integration Tasks (P1 Safe Channels)

1. **Build `holodeck_to_tile` adapter** in plato-torch data loaders.  
   - Parse holodeck JSON logs into PLATO tile format.  
   - Adds narrative training data.

2. **Add LoRA inference to holodeck-rust NPCs**.  
   - Integrate `llama.cpp` or `candle` for lightweight LoRA loading.  
   - Let NPCs use trained ensigns for dialogue.

3. **Connect fleet-simulator agents to holodeck as an environment**.  
   - Extend simulator environment trait to support holodeck TCP connection.  
   - Agents can "enter" the MUD and interact.

4. **Implement ghost tile harvesting**.  
   - Tag memory files from terminated agents.  
   - Ingest into plato-torch as P0 negative examples.

These are safe channels (P1) that avoid breaking the core pipeline while extending fleet capabilities.
