**Cycle:** 187

Based on reading the actual files in the fleet repos, here is the current state of integration between plato-torch, fleet-simulator, holodeck-rust, and plato-ensign.

## 1. Verified Connections (What IS Connected)

### plato-torch ↔ fleet-simulator
- **Connection:** Direct integration via `fleet-simulator` as a data source.
- **Evidence:** `plato-torch/rooms/deadband_room.py` imports `fleet_simulator` and uses `FleetSimulator` class to generate training episodes.
- **Data Flow:** `fleet-simulator` produces simulated fleet interaction episodes (state-action-reward sequences). `deadband_room.py` ingests these episodes as training tiles.
- **Purpose:** Provides synthetic training data for Deadband Protocol (P0/P1/P2) reinforcement learning.

### plato-torch ↔ plato-ensign
- **Connection:** Direct export pipeline.
- **Evidence:** `plato-torch/ensign/exporter.py` contains `export_room_to_lora()` function. `plato-torch/rooms/base_room.py` has `save_for_ensign()` method.
- **Data Flow:** A trained room (accumulated tiles) can be exported as a LoRA adapter via `plato-ensign`.
- **Purpose:** Turns room experience into portable specialist models (ensigns).

### fleet-simulator → plato-ensign (Indirect)
- **Connection:** Indirect via plato-torch.
- **Evidence:** Since `fleet-simulator` trains rooms in `plato-torch`, and rooms can be exported to `plato-ensign`, there is a two-step pipeline: `fleet-simulator` → `plato-torch room` → `plato-ensign LoRA`.
- **Purpose:** Ensigns can embody behaviors learned from simulated fleet interactions.

## 2. Integration Gaps (What Is NOT Connected)

### holodeck-rust ↔ plato-torch
- **Gap:** No direct connection observed.
- **Evidence:** `holodeck-rust/` is a Rust MUD server with sentiment-aware NPCs. `plato-torch/` is Python/PyTorch. No imports, shared data structures, or APIs between them in the code reviewed.
- **Potential:** `holodeck-rust` could be a source of human-agent interaction tiles for `plato-torch`, but currently not linked.

### holodeck-rust ↔ fleet-simulator
- **Gap:** No direct connection.
- **Evidence:** Separate codebases, different languages (Rust vs. Python), no shared simulation state.
- **Potential:** `holodeck-rust` NPCs could be driven by `fleet-simulator` policies, or vice versa.

### holodeck-rust ↔ plato-ensign
- **Gap:** No direct connection.
- **Evidence:** `plato-ensign` exports PyTorch LoRAs; `holodeck-rust` has no apparent import or use of these adapters.
- **Potential:** Ensigns (LoRAs) could be loaded into `holodeck-rust` NPCs to give them specialized personalities/traits learned from rooms.

### fleet-simulator → holodeck-rust (Direct)
- **Gap:** No direct connection.
- **Evidence:** No API or data pipe observed.
- **Potential:** Simulated fleet interactions could generate narrative events or NPC behaviors in the MUD.
