## 1. Verified Current Connections (What's Connected)

Based on reading the fleet-knowledge files and repository structures:

### **plato-torch → plato-ensign** (Strong Connection)
- **plato-ensign** is a direct export module for **plato-torch** rooms.
- Architecture states: "Ensigns are optional exports of room experience."
- **Connection Type:** Data pipeline — trained room states (tiles) can be exported as LoRA adapters.

### **plato-torch → fleet-simulator** (Indirect/Data Connection)
- **fleet-simulator** generates training data (simulated fleet interactions).
- **plato-torch** rooms consume interaction tiles for training.
- **Connection Type:** Training data source — simulator output likely feeds into room tile accumulation.

### **holodeck-rust → (plato-torch / fleet-simulator)** (Potential Sentiment Bridge)
- **holodeck-rust** is a "MUD server with sentiment-aware NPCs."
- Could generate rich, emotionally textured interaction logs.
- **Connection Type:** Potential data source — sentiment logs could become specialized training tiles for plato-torch or test scenarios for fleet-simulator.

### **constraint-theory-core → (Unclear)**
- Rust library for "geometric snapping."
- No explicit integration points found in current map.
- Likely a utility library awaiting integration (e.g., for spatial reasoning in holodeck or simulator).

### **cudaclaw & flux-runtime** (Infrastructure Layer)
- **cudaclaw**: GPU-resident agent runtime.
- **flux-runtime**: Self-assembling bytecode VM.
- These appear to be lower-level execution platforms that higher-level components (plato-torch, holodeck) could eventually target.
- **Connection Type:** Infrastructure — not yet wired into application layer.

## 2. Identified Integration Gaps (What's Not Connected)

### **Gap 1: holodeck-rust → plato-torch (No Direct Pipe)**
- **Problem**: Sentiment-aware NPC interactions are not explicitly harvested as training tiles.
- **Action**: Could build a **HolodeckTileExporter** that formats MUD event logs into plato-torch tile schema.

### **Gap 2: fleet-simulator → holodeck-rust (No Scenario Feed)**
- **Problem**: Simulator could generate agent scenarios to test in holodeck's sentiment-aware environment.
- **Action**: Create a **ScenarioBridge** that converts simulator traces into holodeck world states.

### **Gap 3: plato-ensign → fleet-simulator (No Loopback)**
- **Problem**: Trained ensigns (LoRAs) are not automatically deployed back into the simulator for evaluation.
- **Action**: Implement **EnsignInjector** to load exported adapters into simulator agent instances.

### **Gap 4: constraint-theory-core → holodeck-rust (Missing Spatial Reasoning)**
- **Problem**: Holodeck's MUD environment could use geometric snapping for room layouts, object placement, or NPC pathing.
- **Action**: Integrate constraint-theory-core as a dependency for world-building tools.

### **Gap 5: Ghost Tiles → Active Training (Orphaned Knowledge)**
- **Problem**: "Ghost tiles: dead agents' lessons persist as P0 knowledge" — but no system described to inject them into live rooms.
- **Action**: Build **GhostInjector** (from backlog) that reads ghost tile storage and feeds them as negative examples (P0) into DeadbandRoom.

## 3. Priority Integration Targets for Phase 4

Following Deadband Protocol:
- **P0 (Don't hit rocks)**: Ensure any integration doesn't break existing pipelines (plato-torch → plato-ensign).
- **P1 (Find safe channels)**: Start with data-only, read-only bridges (e.g., holodeck log → tile converter).
- **P2 (Optimize)**: Close the loop with ensign→simulator deployment.

**Recommended first integration for Weaver:**
1. **Build GhostInjector** — addresses a known architectural concept (ghost tiles) and is already in backlog.
2. **Target**: plato-torch's DeadbandRoom preset.
3. **Method**: Read from a designated ghost tile storage (format TBD) and inject as P0 negative examples during room training cycles.
