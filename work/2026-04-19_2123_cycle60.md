## 1. Verified Current Connections

Based on reading the actual code and documentation in the fleet repos, here are the **confirmed, working connections** between the four core systems:

### **plato-torch ↔ plato-ensign** (Strong, Direct)
- **Connection:** `plato-ensign` is a direct export module for `plato-torch`.
- **How:** `plato-torch/rooms/` generates training tiles (interaction logs). `plato-ensign/src/exporter.rs` reads these tile files, converts them into LoRA-compatible datasets, and runs training via `peft`.
- **Evidence:** `plato-ensign/Cargo.toml` imports `plato-torch` as a path dependency. The `export` command in `plato-ensign/src/main.rs` calls `plato_torch::rooms::load_tiles()`.
- **Status:** **Live.** This is the primary "training to export" pipeline.

### **plato-torch ↔ fleet-simulator** (Indirect, Data-Only)
- **Connection:** `fleet-simulator` generates synthetic training data that *can be* consumed by `plato-torch` rooms.
- **How:** `fleet-simulator/output/` produces JSONL files of simulated agent interactions. `plato-torch/rooms/deadband_room.rs` has a `ingest_simulation_data()` function (stubbed) intended to load these files as tile precursors.
- **Evidence:** The function exists but is commented with `// TODO: Connect to fleet-simulator output`. No import of `fleet-simulator` crate in `plato-torch/Cargo.toml`.
- **Status:** **Planned, not wired.** A one-way data flow channel is designed but not implemented.

### **holodeck-rust ↔ (plato-torch, plato-ensign)** (Weak, Conceptual)
- **Connection:** `holodeck-rust` is a sentiment-aware MUD server that could act as a rich tile *source* for `plato-torch` and, by extension, `plato-ensign`.
- **How:** NPC interactions and player sentiment logs could be formatted into tiles. No current code shows this.
- **Evidence:** `holodeck-rust/README.md` mentions "integration with Plato training rooms" as a future goal. No shared dependencies or API calls in code.
- **Status:** **Disconnected.** They exist in the same fleet but share no direct pipes.

### **fleet-simulator ↔ holodeck-rust** (None)
- **Connection:** No evidence of any connection.
- **Status:** **Independent.**

## 2. Identified Integration Gaps (P1 Safe Channels)

Following Deadband Protocol P1: Find safe channels. These are concrete, low-risk integration points that do not violate existing architecture.

### **Gap 1: Simulation Data → Training Room**
- **Problem:** `fleet-simulator` data is not feeding `plato-torch`.
- **Safe Channel:** Create a simple file-watcher script in `plato-torch/scripts/` that:
  1. Monitors `fleet-simulator/output/`.
  2. Converts new JSONL files to tile format (`.tile.json`).
  3. Moves them to `plato-torch/tiles/incoming/`.
- **Risk:** Low. File-based ingestion is already a pattern in `plato-torch`.

### **Gap 2: Holodeck NPC Logs → Tile Source**
- **Problem:** Holodeck's rich interactions are not captured for training.
- **Safe Channel:** Add a `export_tile()` method to `holodeck-rust/src/npc.rs` that, when an interaction meets a sentiment threshold, writes a simple JSON log to a shared directory (e.g., `/var/lib/holodeck/tiles/`).
- **Risk:** Medium. Requires modifying holodeck's core NPC logic but in an additive, optional way.

### **Gap 3: DeadbandRoom ↔ Plato Relay (from Backlog)**
- **Problem:** `DeadbandRoom` (a `plato-torch` preset) is isolated; cannot broadcast its state.
- **Safe Channel:** Implement the `plato_relay` client stub already present in `deadband_room.rs`. Send room heartbeat (agent count, tile count) to a local UDP socket that `plato-relay` (if deployed) listens on.
- **Risk:** Low. The stub exists; completing it is a small code change.

### **Gap 4: GhostInjector → Holodeck (from Backlog)**
- **Problem:** Ghost tiles (P0 knowledge from dead agents) are not utilized in holodeck NPC behavior.
- **Safe Channel:** Create a `GhostInjector` service that reads from `plato-torch/ghosts/`, converts ghost tiles into "cautionary behavior patterns," and injects them into holodeck's NPC decision tree via a new API endpoint (`/npc/apply_ghost`).
- **Risk:** Medium-High. Touches holodeck's NPC AI. Should be gated behind a feature flag.

## 3. Recommended Phase 4 Action (P2 Optimization within Safest Channel)

**Priority Action:** **Close Gap 1 (Simulation Data → Training Room).**

**Why:** 
- Uses existing file-based ingestion pattern.
- No new dependencies or major code changes.
- Immediately useful: populates training rooms with synthetic but structured data.
- Aligns with fleet doctrine: "The work IS the training." This pipeline will generate tiles from simulated work.

**Implementation Sketch:**
```bash
# plato-torch/scripts/sim_watcher.sh
#!/bin/bash
WATCH_DIR="../fleet-simulator/output/"
TILE_DIR="./tiles/incoming/"

inotifywait -m -e close_write "$WATCH_DIR" | while read path action file; do
    if [[ "$file" =~ \.jsonl$ ]]; then
        python3 ./scripts/convert_sim_to_tile.py "$path$file" "$TILE_DIR"
    fi
done
```
