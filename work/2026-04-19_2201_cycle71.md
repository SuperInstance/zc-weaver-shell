## 1. Verified Current Connections

Based on reading the actual fleet-knowledge files and repository structures:

### **plato-torch → plato-ensign** (Strong Connection)
- **plato-torch** generates training rooms and accumulates interaction "tiles".
- **plato-ensign** is designed to export room experience from plato-torch into LoRA adapters.
- This is a core, documented pipeline: rooms train from tiles, ensigns optionally export that experience.

### **plato-torch → fleet-simulator** (Potential/Planned Connection)
- **fleet-simulator** generates training data for agent simulations.
- **plato-torch** could consume this simulated interaction data as training tiles.
- The connection is logical (both generate training data) but no explicit wiring is documented in the provided context.

### **holodeck-rust → plato-torch** (Potential Data Source)
- **holodeck-rust** is a MUD server with sentiment-aware NPCs.
- NPC interactions could become training tiles for plato-torch rooms.
- This is an architectural possibility ("Rooms train from accumulated interaction tiles") but not a confirmed, implemented link.

### **plato-ensign → cudaclaw / flux-runtime** (Downstream Consumption)
- Trained ensigns (LoRA adapters) are likely intended to be loaded into runtime environments like **cudaclaw** (GPU-resident agent runtime) or **flux-runtime** (VM).
- This is an implied output-to-consumer relationship, not a direct code integration.

## 2. Identified Integration Gaps (P1: Safe Channels to Explore)

### **Gap 1: fleet-simulator → plato-torch Tile Injection**
- **Current State:** fleet-simulator runs simulations; plato-torch ingests tiles.
- **Gap:** No documented mechanism or API to pipe simulation events into plato-torch as valid training tiles.
- **Safe Channel:** Examine plato-torch's tile ingestion API (likely in DeadbandRoom or similar). Create a simple adapter that formats fleet-simulator output to match tile schema.

### **Gap 2: holodeck-rust NPC Dialog → Training Tiles**
- **Current State:** holodeck-rust generates interactive NPC dialog.
- **Gap:** These interactions are not automatically captured as plato-torch tiles.
- **Safe Channel:** Add a logging hook in holodeck-rust that emits structured interaction events. Build a tile-forwarder service that routes them to a plato-torch room.

### **Gap 3: Ghost Tiles → holodeck-rust NPC Memory**
- **Current State:** "Ghost tiles" (dead agents' lessons) exist as P0 knowledge.
- **Gap:** This historical cautionary data is not integrated into holodeck-rust's NPC sentiment or behavior.
- **Safe Channel:** Design a "GhostInjector" module that reads ghost tile stores and influences NPC decision trees or dialogue constraints.

### **Gap 4: plato-ensign Export → Runtime Deployment**
- **Current State:** Ensigns are exported as LoRA adapters.
- **Gap:** No clear, automated pipeline to deploy a new ensign to cudaclaw or flux-runtime for immediate use.
- **Safe Channel:** Create a deployment script that packages the ensign, validates it, and updates the runtime configuration.
