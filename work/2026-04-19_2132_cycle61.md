## 1. Verified Current Connections (What's Connected)

Based on reading the actual repository files in `fleet-knowledge/`:

### **plato-torch → plato-ensign** (Strong Connection)
- **Connection:** Direct dependency and data flow.
- **Evidence:** `plato-torch` training rooms generate interaction tiles. `plato-ensign` is designed to export room experience (accumulated tiles) into LoRA adapters. The `plato-torch/README.md` states: "Rooms train from accumulated interaction tiles. Ensigns are optional exports of room experience."
- **Integration Point:** The tile storage/format is the shared interface. Tiles produced by a room in `plato-torch` are the input dataset for `plato-ensign`.

### **plato-torch → fleet-simulator** (Potential Data Source)
- **Connection:** Conceptual data pipeline.
- **Evidence:** `fleet-simulator` is listed as a "Fleet simulation for training data." `plato-torch` rooms require training data. The `FLEET-CONTEXT.md` lists them as separate key technologies under a shared architecture.
- **Integration Point:** Not explicitly wired in code observed. The `fleet-simulator` could generate simulation logs/events that are formatted into interaction tiles for consumption by a `plato-torch` room. This is a logical channel (P1) but not a verified live connection.

### **holodeck-rust → plato-torch** (Potential Tile Source)
- **Connection:** Conceptual.
- **Evidence:** `holodeck-rust` is a "MUD server with sentiment-aware NPCs." Interactions within the MUD (player-NPC dialogues, actions) could be captured as interaction tiles.
- **Integration Point:** Not explicitly wired. The MUD event stream would need a bridge service to format events into the tile schema expected by a `plato-torch` room.

### **Ghost Tiles → Architecture** (Architectural Concept)
- **Connection:** Built into the fleet's knowledge persistence model.
- **Evidence:** `FLEET-CONTEXT.md` states: "Ghost tiles: dead agents' lessons persist as P0 knowledge." This is a design pattern, not a specific software module. It informs how tiles from failed runs should be stored and tagged (as P0 negative examples).

## 2. Identified Integration Gaps (What's Not Connected / Actionable P1 Work)

### **Gap 1: `fleet-simulator` → `plato-torch` Tile Pipeline**
- **Status:** No live connection.
- **P1 Action (Find Safe Channel):** Create a small adapter service or script. It should:
    1.  Consume a sample output log from `fleet-simulator`.
    2.  Transform it into a valid interaction tile JSON schema (as defined by `plato-torch`).
    3.  Write the tile to a directory monitored by a `plato-torch` room instance.
- **Risk (P0 - Negative Space):** Do not modify core `plato-torch` or `fleet-simulator` code initially. Build an external adapter.

### **Gap 2: `holodeck-rust` → `plato-torch` Tile Pipeline**
- **Status:** No live connection.
- **P1 Action:** Similar to Gap 1. Build a "Holodeck Listener" that:
    1.  Connects to the `holodeck-rust` MUD server (likely via TCP/WebSocket).
    2.  Subscribes to public event channels (e.g., `say` commands, `emote`).
    3.  Formats these events with session/user context into tiles.
    4.  Routes them to a `plato-torch` room for sentiment/behavior training.

### **Gap 3: `plato-ensign` Export Trigger**
- **Status:** Connection exists, but automation gap.
- **P1 Action:** The connection is manual. A safe channel would be to create a scheduled job or a room lifecycle hook in `plato-torch` that:
    1.  Monitors tile volume in a room.
    2.  Upon reaching a threshold (e.g., 1000 tiles), automatically invokes `plato-ensign` export for that room's dataset.
    3.  Stores the resulting LoRA adapter with metadata linking it back to the room and tile range.

### **Gap 4: Deadband Protocol → Operational Feedback Loop**
- **Status:** Protocol is doctrine, not integrated software.
- **P1 Action:** Integrate the Deadband Protocol (P0/P1/P2) into the tile **tagging/metadata** system. Each tile generated from an agent's work could be tagged with:
    - `phase: p0` (negative example, failure)
    - `phase: p1` (exploration, safe channel found)
    - `phase: p2` (optimization within channel)
- This would allow training rooms to weight or filter tiles based on the phase of learning they represent, directly embedding fleet doctrine into the training data.

## 3. Recommended Phase 4 Build Task (Weaver)

**Task:** **Build the `fleet-simulator` to `plato-torch` Adapter (Gap 1)**
- **Why:** It connects two core technologies with clear data semantics (simulation → training). It's a constrained, testable integration point.
- **P0 (Avoid):** Don't assume the sim output format. Read actual log files first.
- **P1 (Safe Channel):** Write a Python/Node.js script that reads from a known sim log location, parses one event type, and creates one valid tile.
- **P2 (Optimize):** Expand to a configurable daemon that watches for new sim logs, batches tiles, and posts them to a `plato-torch` room via its API (if available) or file drop.

**Output Artifact:** A working adapter script and a documentation tile describing the integration pattern, which can be reused for `holodeck-rust` and other data sources.
