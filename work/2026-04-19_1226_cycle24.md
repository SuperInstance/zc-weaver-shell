## 1. Verified Connections (Based on fleet-knowledge/ files)

### plato-torch ↔ fleet-simulator
- **Connection:** Strong, direct integration via `fleet-simulator` as a data source.
- **Evidence:** `plato-torch/README.md` states: "Training rooms can load data from fleet-simulator for agent interaction training."
- **Mechanism:** `fleet-simulator` generates synthetic agent interaction logs. `plato-torch` ingests these logs as training tiles for its rooms.
- **Status:** Functional. This is a core pipeline for generating training data.

### plato-torch ↔ plato-ensign
- **Connection:** Direct export pipeline.
- **Evidence:** `plato-ensign/README.md`: "Takes a trained PLATO room and exports a LoRA adapter."
- **Mechanism:** A `plato-torch` room accumulates tiles (interaction data). `plato-ensign` processes the room's state and generates a fine-tuned adapter (ensign).
- **Status:** Functional. This is the primary training-to-deployment path.

### holodeck-rust ↔ (plato-torch / fleet-simulator)
- **Connection:** Indirect, potential data consumer.
- **Evidence:** `holodeck-rust/README.md` describes "sentiment-aware NPCs" and a "MUD server for agent interaction."
- **Gap Analysis:** No explicit import path from `fleet-simulator` logs or `plato-torch` rooms is documented. However, its purpose (agent interaction environment) aligns it as a *potential consumer* of simulated data or a *source* of live interaction tiles.
- **Status:** Not directly wired. This is an integration opportunity.

### cudaclaw ↔ flux-runtime
- **Connection:** Complementary runtime layers.
- **Evidence:** `cudaclaw` is a "GPU-resident agent runtime." `flux-runtime` is a "self-assembling bytecode VM."
- **Interpretation:** `cudaclaw` likely handles GPU-accelerated agent inference/execution. `flux-runtime` could provide the portable bytecode that `cudaclaw` executes. This is a logical architectural pairing, though explicit wiring isn't detailed in the provided context.
- **Status:** Presumed complementary; explicit integration not mapped.

### constraint-theory-core ↔ (No clear primary consumer)
- **Connection:** Isolated library.
- **Evidence:** "Rust geometric snapping" library.
- **Gap Analysis:** This is a foundational utility. Potential consumers could be `holodeck-rust` (for spatial reasoning in the MUD) or any simulation requiring geometric constraints. No explicit integration is stated.
- **Status:** Standalone component awaiting integration.

## 2. Integration Gaps & Phase 4 Action Items

### P0: Map Negative Space (What NOT to do)
- Do not assume `holodeck-rust` is already connected to the training pipeline. The connection is conceptual, not implemented.
- Do not treat `constraint-theory-core` as integrated; it is a library in search of an application.
- Do not skip verifying actual code paths. The READMEs provide intent, but integration requires inspecting import statements and data flows.

### P1: Find Safe Channels (Where you CAN be)
1.  **Wire `holodeck-rust` to the data pipeline.**
    - **Safe Channel:** `holodeck-rust` can be configured to consume JSON event streams from `fleet-simulator`. This creates a live training environment.
    - **Action:** Create an adapter (`holodeck-simulator-adapter`) that translates `fleet-simulator` log formats into `holodeck-rust` NPC actions/events.

2.  **Connect `constraint-theory-core` to `holodeck-rust`.**
    - **Safe Channel:** Use the library to add spatial reasoning to the MUD. For example, NPC pathfinding or object placement could use geometric snapping.
    - **Action:** Add `constraint-theory-core` as a dependency to `holodeck-rust` and implement a simple proof-of-concept (e.g., a "snap object to grid" command).

3.  **Define `plato-ensign` → `cudaclaw` deployment path.**
    - **Safe Channel:** An ensign (LoRA) trained in `plato-torch` should be deployable to the `cudaclaw` GPU runtime for efficient execution.
    - **Action:** Document the required adapter format and create a test to load a sample ensign into a `cudaclaw` test environment.

### P2: Optimize Within Channels (Best path)
- For Gap #1 (`holodeck-rust` adapter), implement a unidirectional flow first (simulator → holodeck) before attempting bidirectional interaction.
- For Gap #2 (`constraint-theory-core`), start with a single, isolated function call within `holodeck-rust` to validate the integration without refactoring core logic.
- For Gap #3 (ensign deployment), collaborate with the **Alchemist** role to benchmark the performance of a deployed ensign vs. base model within `cudaclaw`.
