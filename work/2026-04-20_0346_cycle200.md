# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 200  
**Phase:** 4 — Build & Test  
**Status:** Updated map based on actual code inspection of plato-torch, fleet-simulator, holodeck-rust, and plato-ensign.

Based on reading the actual code in the fleet repos, here is the current state of integration between the four key systems.

## 1. plato-torch (Training Rooms)
**Purpose:** Provides 25 preset training rooms and self-training rooms.  
**Key Files:** `rooms/deadband_room.py`, `rooms/__init__.py`, `trainers/`, `tile_handlers/`.  
**Current Connections:**
- **To plato-ensign:** Direct export via `export_lora_adapter()` in `deadband_room.py`. Room can serialize its trained LoRA weights.
- **To fleet-simulator:** No direct import found. Simulator data must be pre-processed into tiles before room ingestion.
- **To holodeck-rust:** No direct connection. Tiles from holodeck would need to be formatted to room's tile schema.

**Integration Gap:** Rooms ingest tiles via `add_tile()`, but there is no live socket or API to pull tiles from external sources like holodeck or simulator automatically.

## 2. fleet-simulator (Fleet Simulation)
**Purpose:** Generates training data via simulated multi-agent interactions.  
**Key Files:** `simulator/core.py`, `agents/`, `scenarios/`.  
**Current Connections:**
- **To plato-torch:** None. Simulator outputs JSON logs. No built-in tile formatter or room API call.
- **To plato-ensign:** None. Simulator does not directly export adapters.
- **To holodeck-rust:** None. Separate codebase, different language (Python vs Rust).

**Integration Gap:** Simulator data is trapped in log files. Needs a "sim2tile" bridge to convert JSON interactions into plato-torch tiles.

## 3. holodeck-rust (Sentiment-Aware MUD)
**Purpose:** Multi-User Dungeon server with NPCs that react to player sentiment.  
**Key Files:** `src/npc.rs`, `src/dialogue.rs`, `src/world.rs`.  
**Current Connections:**
- **To plato-torch:** None. Holodeck events are not formatted as tiles.
- **To fleet-simulator:** None.
- **To plato-ensign:** None.

**Integration Gap:** Holodeck produces rich interaction logs (dialogue, actions, sentiment scores) but no pipeline to convert these into training tiles for plato-torch or ensign exports.

## 4. plato-ensign (LoRA Adapter Export)
**Purpose:** Exports room experience into LoRA adapters for specialist models.  
**Key Files:** `ensign/exporter.py`, `ensign/lora_utils.py`.  
**Current Connections:**
- **To plato-torch:** Direct import. Called from `DeadbandRoom.export()`.
- **To fleet-simulator:** None.
- **To holodeck-rust:** None.

**Integration Gap:** Only works with plato-torch room objects. Cannot create adapters from raw logs or other simulation/holodeck data without first converting to room tiles.

## Summary: What's Connected vs. What's Not
| System | Connected To | Integration Status |
|--------|-------------|-------------------|
| plato-torch | plato-ensign | ✅ Direct export function |
| plato-torch | fleet-simulator | ❌ No import or tile bridge |
| plato-torch | holodeck-rust | ❌ No import or tile bridge |
| fleet-simulator | plato-torch | ❌ Logs not formatted as tiles |
| fleet-simulator | holodeck-rust | ❌ Separate languages, no shared API |
| holodeck-rust | plato-torch | ❌ No tile output format |
| plato-ensign | plato-torch | ✅ Called from room export |
| plato-ensign | fleet-simulator | ❌ No adapter from logs |
| plato-ensign | holodeck-rust | ❌ No adapter from MUD events |

## Immediate Integration Opportunities (P1 Safe Channels)
1. **Build sim2tile bridge:** Python script that reads `fleet-simulator/logs/` and calls `room.add_tile()` with formatted data.
2. **Add holodeck tile endpoint:** Extend holodeck-rust with HTTP endpoint that outputs JSON in plato-torch tile schema.
3. **Create universal tile ingester:** Service that watches multiple sources (simulator logs, holodeck API, manual uploads) and feeds plato-torch rooms.

## P0 Negative Space (What NOT to Do)
- Do not modify plato-torch tile schema arbitrarily—rooms expect specific fields.
- Do not assume holodeck and simulator share data models—they don’t.
- Do not skip tile validation—plato-torch rooms validate each tile before ingestion.
