## 1. plato-torch → fleet-simulator
**Connection:** **Direct import**  
**Evidence:** `plato-torch/src/rooms/deadband_room.py` imports `fleet_simulator`:
```python
from fleet_simulator import FleetSimulator, AgentState, EnvironmentGrid
```
**Usage:**  
- `DeadbandRoom` instantiates a `FleetSimulator` object (`self.simulator`)  
- Calls `simulator.step()` to advance simulation  
- Reads `simulator.get_agent_states()` to get training data  
- Uses `simulator.reset()` between training episodes  

**Data Flow:**  
Simulation states → Room training batches → Tile generation  

**Integration Gap:**  
- No bidirectional control: room cannot modify simulator parameters mid‑episode  
- Simulator uses its own internal RNG; no seed synchronization with torch  

---

## 2. plato-torch → plato-ensign
**Connection:** **Optional export pipeline**  
**Evidence:** `plato-torch/src/export/ensign_exporter.py`:
```python
from plato_ensign import LoRAAdapter, export_room_experience
```
**Usage:**  
- After room training, `ensign_exporter` can be called  
- It serializes room’s `history_buffer` (interaction tiles)  
- Passes buffer to `plato_ensign.export_room_experience()`  
- Returns a `LoRAAdapter` object that can be saved to disk  

**Data Flow:**  
Room tiles → LoRA weights  

**Integration Gap:**  
- Export is manual/offline; no automatic triggering  
- No validation that exported LoRA matches room’s current policy

## 3. fleet-simulator → holodeck-rust
**Connection:** **None found**  
**Evidence:**  
- `fleet-simulator/src/` has no Rust imports, no `holodeck` references  
- `holodeck-rust/src/` has no Python bindings, no simulator data structures  
- Separate concerns: simulator is pure Python grid‑based; holodeck is Rust MUD with sentiment NPCs  

**Potential Integration Point:**  
Simulator could feed agent trajectories to holodeck as “storylines” for NPCs, but no code exists yet.

## 4. holodeck-rust → plato-torch
**Connection:** **None found**  
**Evidence:**  
- `holodeck-rust` outputs JSON over TCP (port 8080)  
- `plato-torch` has no client for that port, no `holodeck` import  
- Different languages: Rust vs Python/Torch  

**Potential Integration Point:**  
A bridge service (e.g., `holodeck‑adapter`) could translate holodeck JSON into training tiles for a room.

## 5. plato-ensign → Others
**Connection:** **Standalone library**  
**Evidence:**  
- `plato‑ensign/src/lib.rs` is pure Rust, no Python deps  
- Exports `LoRAAdapter` struct and `export_room_experience` function  
- Called only by `plato‑torch/export/ensign_exporter.py` (see above)  

**Integration Gap:**  
- No direct consumption of ensign adapters by `fleet‑simulator` or `holodeck‑rust`  
- No automatic loading of LoRA into a running room

## 6. Ghost Tiles & Mirror Play
**Connection:** **Architectural pattern, not code**  
**Evidence:**  
- `fleet‑knowledge/DEADBAND‑PROTOCOL.md` describes ghost tiles (P0 knowledge from dead agents)

**Potential Integration Point:**
