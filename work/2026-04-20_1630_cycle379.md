# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 379  
**Phase:** 4 — Build & Test  
**Status:** Direct file inspection of fleet repositories. Focus on actual imports, configuration references, and shared data structures.

## 1. plato-torch → fleet-simulator
**Connection:** **Yes** — Direct import and data flow.
- **Evidence:** `plato-torch/rooms/deadband_room.py` imports `from fleet_simulator.simulator import FleetSimulator`.
- **Usage:** The `DeadbandRoom` class instantiates a `FleetSimulator` object (`self.simulator = FleetSimulator()`). It calls `self.simulator.step()` during room updates to generate simulation state.
- **Data Flow:** Simulation state (agent positions, actions) is returned from the simulator and can be used to create training tiles within the room.
- **Integration Gap:** The simulator's output format and the room's expected tile format are not explicitly documented. No shared schema file exists.

## 2. plato-torch → plato-ensign
**Connection:** **Yes** — Functional pipeline.
- **Evidence:** `plato-torch/exporters/ensign_exporter.py` exists. `plato-torch/rooms/base_room.py` has a `train_ensign()` method that calls this exporter.
- **Usage:** After a room accumulates training tiles, `room.train_ensign()` can be called. This exports the room's experience (tiles) into a LoRA adapter via the `ensign_exporter`.
- **Data Flow:** Room tiles → EnsignExporter → LoRA weights file (.safetensors).
- **Integration Gap:** The export process is optional and manual. No automatic triggering based on tile count or performance metrics.

## 3. plato-torch → holodeck-rust
**Connection:** **No** — No direct import or API call found.
- **Evidence:** Searched `plato-torch/` for "holodeck". No imports or references found in Python files.
- **Potential Link:** Both systems generate training data (tiles vs. NPC interactions), but they are separate data silos.
- **Integration Gap:** No shared session ID, no unified tile format, no bridge service. This is a major isolation point.

## 4. fleet-simulator → holodeck-rust
**Connection:** **No** — Separate codebases, different languages (Python vs. Rust).
- **Evidence:** `fleet-simulator/` contains no Rust code or references to holodeck.
- **Potential:** Both simulate agent behavior, but for different purposes (fleet training vs. sentiment-aware NPCs).
- **Integration Gap:** No shared agent definition or event protocol.

## 5. fleet-simulator → plato-ensign
**Connection:** **Indirect via plato-torch**.
- **Evidence:** The simulator's output becomes tiles in a plato-torch room, which can then be exported to an ensign.
- **Direct Link:** None. `fleet-simulator/` does not import or reference `plato-ensign`.
- **Integration Gap:** Simulator data must pass through a plato-torch room to become an ensign. No direct "simulation-to-ensign" pipeline.

## 6. holodeck-rust → plato-ensign
**Connection:** **No** — No found reference.
- **Evidence:** `holodeck-rust/` Cargo.toml and source files show no dependency on Python or plato-ensign.
- **Potential:** NPC interaction logs could be formatted as training tiles, but no exporter exists.
- **Integration Gap:** Holodeck data is currently trapped in its own ecosystem.

## 7. Ghost Tiles & GhostInjector
**Status:** **Isolated utility**.
- **Location:** `plato-torch/utils/ghost_injector.py`.
- **Function:** Reads "ghost tile" JSON files and injects them into a room's tile history as P0 (negative space) examples.
- **Connections:** Used by `DeadbandRoom` (has a `load_ghosts()` method). Not connected to `holodeck-rust` or `fleet-simulator`.
- **Integration Gap:** Ghost tiles are a plato-torch concept only. Not integrated with holodeck's NPC memory or simulator agent logs.

## 8. cudaclaw & flux-runtime
**Connection:** **Not observed** in the inspected repos.
- **Evidence:** No imports of `cudaclaw` or `flux-runtime` in `plato-torch`, `fleet-simulator`, `holodeck-rust`, or `plato-ensign`.
- **Note:** These are listed as key technologies but appear to be separate runtime components, not integrated with these four core data-generation systems.

## Identified Integration Gaps (P1 Channels for Work)
1. **plato-torch ↔ holodeck-rust bridge**: Most significant gap. Two training data sources are separate.
2. **Direct fleet-simulator → plato-ensign path**: Bypassing room intermediary for pure simulation ensigns.
3. **Ghost tile sharing**: Inject holodeck NPC "lessons" as ghost tiles into plato-torch.
4. **Unified tile schema**: No common `tile.json` format across systems.
5. **cudaclaw/flux integration**: These runtimes are not leveraged by the training data stack.
