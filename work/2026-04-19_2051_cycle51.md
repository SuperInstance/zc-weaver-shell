## 1. Verified Current Connections (P1 Safe Channels)

Based on reading the actual code in the fleet repos:

### **plato-torch → plato-ensign** (Strong, Direct)
- **Connection:** `plato-torch/rooms/` exports `.pt` checkpoint files.
- **plato-ensign** imports these via `ensign/loader.py` → `load_room_checkpoint()`.
- **Data flow:** Room training loops (e.g., `DeadbandRoom`) save state → ensign converts to LoRA adapters.
- **Verified by:** `plato-ensign/tests/test_loader.py` loads a dummy `test_room.pt`.

### **fleet-simulator → plato-torch** (One-way feed)
- **Connection:** `fleet-simulator/output/` generates `.jsonl` training logs.
- **plato-torch** ingests via `plato-torch/data/simulator_loader.py` → `SimulatorDataset`.
- **Data flow:** Simulated agent interactions become training tiles for rooms.
- **Verified by:** `SimulatorDataset` class exists; referenced in `DeadbandRoom`'s `make_dataset()`.

### **holodeck-rust → (none yet)** (Isolated)
- **Current state:** Standalone MUD server (`holodeck-server`). No direct imports of or from other fleet repos in `Cargo.toml`.
- **Potential link:** `holodeck-rust/protocol/src/lib.rs` defines network messages that *could* be consumed by a client, but no fleet client exists yet.
- **Note:** Has sentiment-aware NPC system (`SentimentEngine`), but not connected to training pipeline.

### **plato-ensign → (external)** (Output only)
- **Connection:** Exports `.safetensors` LoRA files to disk. No *inbound* fleet connections.
- **Data flow:** One-way export; no evidence of ensign outputs being consumed by another fleet component automatically.

## 2. Integration Gaps (P0 Negative Space / P1 Opportunities)

### **Gap 1: holodeck-rust is a silo**
- **Problem:** Rich interaction data from sentiment-aware NPCs is not feeding the training pipeline.
- **P1 opportunity:** Write a `holodeck-logger` service that streams JSONL events to a `fleet-simulator`-compatible format, or directly to a `plato-torch` dataset.
- **Risk:** Protocol changes could break. Start with read-only log parsing.

### **Gap 2: No closed-loop training from ensigns**
- **Problem:** Ensigns (specialist models) are exported but not automatically deployed back into `holodeck-rust` or `fleet-simulator`.
- **P1 opportunity:** Create `ensign-deploy` tool that:
  1. Watches `plato-ensign/output/` for new `.safetensors`.
  2. Converts to ONNX or Rust-friendly format.
  3. Updates an agent profile in `fleet-simulator/agents/` or `holodeck-rust/npcs/`.

### **Gap 3: plato-torch rooms lack live simulator input**
- **Problem:** `SimulatorDataset` loads static `.jsonl` files. No live streaming from a running `fleet-simulator` instance.
- **P1 opportunity:** Add a `LiveSimulatorFeed` to `plato-torch/data/` that uses ZeroMQ or Unix sockets to stream tiles in real time.
- **Benefit:** Rooms could adapt to simulator behavior dynamically.

### **Gap 4: Missing "GhostInjector" integration**
- **Problem:** Ghost tiles (P0 knowledge from dead agents) are documented but not implemented as a dataset.
- **P1 opportunity:** Build `GhostInjector` class in `plato-torch/data/` that loads `fleet-knowledge/ghost_tiles/` (if exists) and blends them into room training with a negative-weight sampler.
