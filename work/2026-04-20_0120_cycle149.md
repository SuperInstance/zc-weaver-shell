# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 149

Based on reading the actual files in the fleet repos, here is the current state of integration between **plato-torch**, **fleet-simulator**, **holodeck-rust**, and **plato-ensign**.

## 1. Verified Connections (What IS Connected)

### plato-torch ↔ fleet-simulator
- **Connection:** Strong, bidirectional data flow.
- **Evidence:**  
  - `plato-torch/training_rooms/deadband_room.py` imports `fleet_simulator` and uses `FleetSimulator` class to generate training data.  
  - `fleet_simulator/__init__.py` exports `FleetSimulator` and `SimulationResult`.  
  - The DeadbandRoom preset runs simulations via `self.simulator.run()` and ingests the resulting `SimulationResult` objects as training tiles.  
- **Data Flow:**  
  `fleet-simulator` → generates simulation traces → `plato-torch` rooms consume as training data.

### plato-torch ↔ plato-ensign
- **Connection:** Direct export pipeline.
- **Evidence:**  
  - `plato-ensign/export.py` provides `export_room_to_lora()` function.  
  - `plato-torch/training_rooms/base.py` has a `save()` method that calls `export_room_to_lora()` if the `plato-ensign` package is installed.  
  - `plato-torch/README.md` states: "Rooms can optionally export their experience as LoRA adapters via plato-ensign."  
- **Data Flow:**  
  `plato-torch` room accumulates tiles → `plato-ensign` packages them into a LoRA adapter file.

### holodeck-rust ↔ (Indirect via future integration)
- **Connection:** Not directly wired, but designed for sentiment‑aware NPCs that could feed training data.
- **Evidence:**  
  - `holodeck-rust/README.md` describes "sentiment‑aware NPCs" and a MUD server that could generate interaction logs.  
  - `plato-torch/training_rooms/` includes a `holodeck_room.py` stub (currently empty) suggesting planned integration.  
  - No actual imports or calls found in current code.

## 2. Integration Gaps (What Is NOT Connected)

### holodeck-rust ↔ plato-torch
- **Gap:** No live data pipeline.  
- **Evidence:**  
  - `holodeck_room.py` exists but contains only a placeholder class with no simulator binding.  
  - No import of `holodeck` modules in any working room preset.  
  - The `holodeck-rust` server runs independently; its output is not consumed by any training room.

### holodeck-rust ↔ fleet-simulator
- **Gap:** No shared simulation context.  
- **Evidence:**  
  - `fleet-simulator` simulates fleet‑level coordination; `holodeck-rust` simulates individual NPC interactions.  
  - No common data structures or APIs between the two.

### fleet-simulator ↔ plato-ensign
- **Gap:** No direct export path.  
- **Evidence:**  
  - `fleet-simulator` outputs `SimulationResult` objects, but these are not directly convertible to LoRA adapters.  
  - `plato-ensign` expects a trained `plato-torch` room as input, not raw simulation results.

### plato-ensign ↔ holodeck-rust
- **Gap:** No mechanism to export NPC interactions as adapters.  
- **Evidence:**  
  - `plato-ensign` is tightly coupled to `plato-torch` room objects.  
  - `holodeck-rust` has no Python interface or adapter‑export functionality.
