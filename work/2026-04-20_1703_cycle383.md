## 1. plato-torch (Training Rooms)
**Location:** `plato-torch/`  
**Purpose:** Provides 25 training room presets, including DeadbandRoom. Self-training rooms that accumulate interaction tiles.

**Key Files:**
- `presets/deadband_room.py` – Implements Deadband Protocol (P0/P1/P2) as a training environment.
- `rooms/base_room.py` – Base class for all rooms. Defines `submit_tile(tile_data)` method.
- `tiles/tile_schema.py` – Defines JSON schema for training tiles (agent_id, cycle, phase, task, output, metadata).

**Connections:**
- **plato-ensign:** Exports room experience. `deadband_room.py` has `export_to_lora()` method that calls `plato_ensign.export(room_state)`.
- **fleet-simulator:** Not directly imported. However, `presets/simulator_room.py` exists, suggesting a dedicated room for simulator training.
- **holodeck-rust:** No direct imports. No Rust bindings visible in Python code.
- **cudaclaw:** Not referenced in Python files. Likely separate runtime.
- **flux-runtime:** Not referenced.

**Data Flow:**  
Rooms collect tiles via `submit_tile()`. Tiles are stored in `room_state['tiles']`. `export_to_lora()` packages tiles for ensign training.

## 2. plato-ensign (LoRA Adapter Export)
**Location:** `plato-ensign/`  
**Purpose:** Exports room experience into LoRA adapters for specialist models.

**Key Files:**
- `export.py` – Main export pipeline. Accepts `room_state` dict, extracts tiles, trains LoRA, saves adapter.
- `lora_train.py` – Uses PyTorch and Hugging Face `peft` library. Configurable rank, alpha.
- `config/ensign_config.yaml` – Specifies base model (e.g., `deepseek-ai/deepseek-coder-6.7b-instruct`), output path.

**Connections:**
- **plato-torch:** Direct import. `export.py` expects `room_state` from a plato-torch room.
- **fleet-simulator:** Not referenced.
- **holodeck-rust:** Not referenced.
- **cudaclaw/flux:** Not referenced.

**Integration Point:**  
`plato-torch` → `plato-ensign` is a **verified, working connection**. Room calls `export_to_lora()`, which invokes `plato_ensign.export()`.

## 3. fleet-simulator (Fleet Simulation)
**Location:** `fleet-simulator/`  
**Purpose:** Generates training data via simulated fleet interactions.

**Key Files:**
- `simulator/core.py` – Simulates agent interactions, produces event logs.
- `output/log_parser.py` – Parses logs into structured tiles.
- `config/sim_config.json` – Defines agent types, interaction rules.

**Connections:**
- **plato-torch:** Indirect. `simulator_room.py` in plato-torch likely consumes simulator output, but no import found in core files.
- **plato-ensign:** Not referenced.
- **holodeck-rust:** Not referenced.
- **cudaclaw/flux:** Not referenced.

**Data Flow:**  
Simulator generates logs → parsed into tiles → could be fed to a plato-torch room. **No direct import found**; likely a manual or configuration-based pipeline.

## 4. holodeck-rust (Sentiment-Aware MUD)
**Location:** `holodeck-rust/`  
**Purpose:** Multi‑User Dungeon server with NPCs that react to sentiment.

**Key Files:**
- `src/npc/sentiment.rs` – Sentiment analysis for NPC dialogue.
- `src/world/room.rs` – Room management.
- `Cargo.toml` – Dependencies: `serde`, `tokio`, `llm-chain` (for NLP).

**Connections:**
- **plato-torch:** None. Pure Rust, no Python bindings.
- **plato-ensign:** None.
- **fleet-simulator:** None.
- **cudaclaw/flux:** None.
- **GhostInjector:** Referenced in `src/ghost/injector.rs` – placeholder for injecting ghost tiles.

**Integration Gap:**  
holodeck‑rust is **isolated** from the Python‑based training pipeline. It has a GhostInjector stub but no live connection to plato‑torch tile storage.

## 5. cudaclaw & flux-runtime
**Location:** `cudaclaw/`, `flux-runtime/`  
**Inspection:** Both are separate runtime systems. No imports found in plato‑torch, plato‑ensign, fleet‑simulator, or holodeck‑rust. Likely used for GPU‑resident agents and bytecode execution, not directly integrated with the training tile flow.
