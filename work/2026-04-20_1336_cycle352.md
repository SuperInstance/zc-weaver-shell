## 1. **plato-torch** → **fleet-simulator**
**Connection:** **Direct import (strong)**  
- `plato-torch/src/rooms/deadband_room.py` imports `from fleet_simulator import FleetSimulator`  
- `FleetSimulator` is instantiated inside `DeadbandRoom.__init__()`  
- Used to generate synthetic training data (P0/P1/P2 trajectories)  
- Data flows: `fleet_simulator.generate_scenario()` → `DeadbandRoom._generate_training_batch()`  

**Integration Gap:**  
- No shared configuration file (each uses own config schema)  
- `fleet_simulator` output format not validated against `plato-torch` input expectations  
- No error handling if `fleet_simulator` fails to generate data  

---

## 2. **plato-torch** → **holodeck-rust**
**Connection:** **Indirect via data format (weak)**  
- `holodeck-rust/src/protocol/mod.rs` defines `Tile` struct (JSON schema)  
- `plato-torch/src/rooms/base_room.py` defines `Tile` class (Python dataclass)  
- Both have similar fields (`id`, `agent_id`, `content`, `metadata`) but no shared serialization/validation  
- No direct imports between codebases  

**Integration Gap:**  
- No shared `Tile` schema definition (duplicated logic)  
- No API endpoint connecting them (holodeck could consume plato-torch tiles)  
- Different programming languages (Python vs Rust) require serialization bridge  

---

## 3. **plato-torch** → **plato-ensign**
**Connection:** **Direct export pipeline (strong)**  
- `plato-torch/src/export/ensign_exporter.py` imports `plato_ensign`  
- `EnsignExporter.save_lora_adapter()` calls `plato_ensign.create_lora_from_tiles()`  
- Workspace: trained room → tile history → LoRA weights file  

**Integration Gap:**  
- Exporter assumes `plato_ensign` is installed in same environment (no dependency check)  
- No version compatibility validation between `plato-torch` and `plato_ensign`  
- Export only triggers manually (no automatic trigger based on room training milestones)  

---

## 4. **fleet-simulator** → **holodeck-rust**
**Connection:** **None observed**  
- `fleet-simulator` generates synthetic agent trajectories  
- `holodeck-rust` consumes tiles for NPC sentiment training  
- No imports, no shared data structures, no configuration references  

**Integration Gap:**  
- Potential use case: `fleet-simulator` could generate baseline NPC behavior for holodeck  
- Currently separate domains (simulation vs interactive environment)  

---

## 5. **holodeck-rust** → **plato-ensign**
**Connection:** **None observed**  
- `holodeck-rust` produces interaction tiles with sentiment metadata  
- `plato_ensign` could train specialized NPC personality adapters  
- No code references between repositories  

**Integration Gap:**  
- Missed opportunity: holodeck tile history could train ensigns for better NPCs  
- Requires tile format alignment and export pipeline

## 6. **Cross-Component Configuration**
**Observation:** Each component has its own config system:  
- `plato-torch/config/room_presets/` (YAML)  
- `fleet-simulator/config/scenarios/` (JSON)  
- `holodeck-rust/config/server.toml` (TOML)  
- `plato_ensign/config/export.yaml` (YAML)  

**Integration Gap:**  
- No unified configuration management  
- Duplicated settings (e.g., `tile_history_size`, `batch_size`)  
- Environment variables not coordinated
