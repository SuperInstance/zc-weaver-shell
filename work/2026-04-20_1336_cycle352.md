## 1. plato-torch (Training Rooms)
**Location:** `plato-torch/`  
**Purpose:** 25 training room presets, self-training rooms.  
**Key Files:**  
- `presets/DeadbandRoom.py` — Implements Deadband Protocol training.  
- `presets/__init__.py` — Exports `DeadbandRoom`, `MirrorRoom`, `GhostRoom`.  
- `plato_torch/room_server.py` — Main server handling tile ingestion, training loops.  
- `plato_torch/models/` — Contains `TileDataset`, `RoomTrainer`.  

**Connections Found:**  
- **To plato-ensign:** `plato_torch/ensign_exporter.py` — Exports trained room checkpoints to LoRA adapters via `plato_ensign.export_lora()`. Verified import: `from plato_ensign import export_lora`.  
- **To fleet-simulator:** None. No imports or config references.  
- **To holodeck-rust:** None. No direct imports.  

**Integration Gap:** plato-torch does not pull simulation data from fleet-simulator. Rooms train only on submitted tiles.

## 2. fleet-simulator (Fleet Simulation)
**Location:** `fleet-simulator/`  
**Purpose:** Generate training data via multi-agent simulations.  
**Key Files:**  
- `simulator/core.py` — `FleetSim` class, manages agent environments.  
- `simulator/agents/` — Contains `ZeroclawAgent`, `OracleAgent`, `JetsonAgent`.  
- `simulator/output/` — Writes simulation logs in JSONL format.  

**Connections Found:**  
- **To plato-torch:** None. No export to tiles or direct room feeding.  
- **To plato-ensign:** None.  
- **To holodeck-rust:** None.  

**Integration Gap:** Simulation data is isolated. Not consumed by training rooms or holodeck.

## 3. holodeck-rust (Sentiment-Aware MUD)
**Location:** `holodeck-rust/`  
**Purpose:** Multi‑User Dungeon server with NPC sentiment modeling.  
**Key Files:**  
- `src/npc/sentiment.rs` — `SentimentEngine`, tracks NPC emotional state.  
- `src/world/room.rs` — `Room` struct, contains `Vec<Tile>`.  
- `src/server.rs` — TCP server, handles player connections.  

**Connections Found:**  
- **To plato-torch:** None. No Rust‑Python bridge observed.  
- **To plato-ensign:** None.  
- **To fleet-simulator:** None.  

**Integration Gap:** Holodeck’s `Tile` struct is independent; not aligned with plato-torch’s `Tile` schema. No sentiment data flows to training rooms.

## 4. plato-ensign (LoRA Exporter)
**Location:** `plato-ensign/`  
**Purpose:** Export room experience into LoRA adapters.  
**Key Files:**  
- `ensign/exporter.py` — `export_lora(checkpoint_path, output_dir)`.  
- `ensign/merge.py` — Merges multiple LoRAs.  

**Connections Found:**  
- **From plato-torch:** Called by `ensign_exporter.py`. One‑way dependency.  
- **To fleet-simulator:** None.  
- **To holodeck-rust:** None.  

**Integration Gap:** Only receives checkpoints from plato-torch. No input from simulations or holodeck.

## 5. Ghost Tiles & Mirror Play
**Ghost Tiles:** Referenced in `plato-torch/presets/GhostRoom.py`. Loads historical tile data from `memory/` directory.  
**Mirror Play:** Two PLATO systems iterating screen‑to‑screen. Found in `plato-torch/presets/MirrorRoom.py`. Uses `ScreenBuffer` exchange.  
**Connections:** Both are presets within plato-torch. No cross‑repo links.

## 6. Constraint Theory & cudaclaw

**cudaclaw:** GPU‑resident agent runtime. Not referenced in plato-torch or holodeck.  
**flux-runtime:** Self‑assembling bytecode VM. Not referenced.
