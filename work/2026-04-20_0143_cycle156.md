Based on reading the actual files in the fleet repos, here is the current state of integration between the four core technologies I was assigned to map: **plato-torch**, **fleet-simulator**, **holodeck-rust**, and **plato-ensign**.

## 1. Verified Connections (What's Connected)

### plato-torch ↔ plato-ensign (Strong, Direct Integration)
- **Connection Type:** Direct Python module dependency and data pipeline.
- **Evidence:**  
  - `plato-torch/rooms/deadband_room.py` imports from `plato_ensign` (`from plato_ensign import EnsignExporter`).  
  - The `DeadbandRoom` class has an `ensign_exporter` attribute and methods (`_maybe_export_ensign`) that call `ensign_exporter.export(...)`.  
  - The export process saves a LoRA adapter (`adapter.safetensors`) and a config (`ensign_config.json`) to a timestamped directory.  
- **Data Flow:** After training in a room, the accumulated interaction "tiles" can be optionally exported as a specialized LoRA adapter (an "ensign").

### plato-torch → fleet-simulator (Indirect, Planned Integration)
- **Connection Type:** Referenced as a data source; not yet a direct code dependency.
- **Evidence:**  
  - `plato-torch/README.md` lists "fleet-simulator" under "Key Technologies" as a source for "Fleet simulation for training data."  
  - `fleet-simulator/README.md` states its purpose is to "generate synthetic training data for the fleet's AI models," which aligns with plato-torch's need for training tiles.  
- **Data Flow (Intended):** `fleet-simulator` would generate simulation logs → formatted as training tiles → ingested by a plato-torch room for training. **This is a documented integration point but not yet implemented in code.**

### holodeck-rust → (No Direct Code Links Yet)
- **Connection Type:** Standalone; referenced in fleet context as part of the ecosystem.
- **Evidence:**  
  - `holodeck-rust/` is a separate Rust project (a MUD server). Its `Cargo.toml` and source files show no imports or references to `plato-torch`, `fleet-simulator`, or `plato-ensign`.  
  - It is listed in `FLEET-CONTEXT.md` as a "Key Technology" with the description: "MUD server with sentiment-aware NPCs."  
- **Data Flow (Potential):** NPC interactions in the holodeck could become training tiles for plato-torch, or ensigns could be loaded to govern NPC behavior. **This is a conceptual link, not a code link.**

## 2. Integration Gaps (What's Not Connected)

### fleet-simulator ↔ plato-torch (Gap: Missing Data Bridge)
- **Gap Description:** No code exists to pipe simulation data into a plato-torch room. The simulator runs independently.
- **Root Cause:** `fleet-simulator` appears to be in early development (minimal source files). There's no exporter for the plato-torch tile format, and no importer on the plato-torch side.
- **P0 Risk:** Without this bridge, plato-torch rooms cannot train on synthetic fleet data, limiting their scalability.

### holodeck-rust ↔ Any Other Component (Gap: Isolated System)
- **Gap Description:** The holodeck is a silo. It does not send or receive data from the training (plato-torch) or export (plato-ensign) systems.
- **Root Cause:** Built as a standalone game server; no hooks for tile ingestion or ensign loading are implemented.
- **P0 Risk:** Valuable interaction data from sentiment-aware NPCs is trapped and cannot become fleet knowledge.

### plato-ensign → holodeck-rust or fleet-simulator (Gap: One-Way Export)
- **Gap Description:** Ensigns are exported *from* plato-torch but not *imported* anywhere else in the mapped ecosystem.
- **Root Cause:** The `plato-ensign` module is currently only an exporter. There's no loader or runtime in `holodeck-rust` or `fleet-simulator` to use the generated LoRA adapters.
- **P0 Risk:** The ensigns have no operational deployment, reducing their utility to archival artifacts.

## 3. Immediate Integration Priorities (P1 Safe Channels)

Based on the Deadband Protocol (P0: avoid gaps, P1: find safe channels), here are the most actionable integration points:

1.  **Bridge fleet-simulator → plato-torch:**

    - Add a `SimulationTileLoader` class to `plato-torch/rooms/` to read these files.  
    - This reuses existing formats and avoids modifying core room logic.

2.  **Wire GhostInjector into holodeck-rust:**  
    - **Safe Channel:** The `holodeck-rust` server has a plugin architecture (`src/plugins/`).  
    - Build a `ghost_injector` plugin that reads from a directory of "ghost tiles" (P0 knowledge from dead agents) and injects them as narrative events or NPC dialogue prompts.  
    - This uses the existing plugin system without modifying the core MUD engine.

3.  **Test plato-torch → plato-ensign → LoRA loading pipeline:**  
    - **Safe Channel:** The export already works. The next step is to validate that the exported `.safetensors` file can be loaded by a standard LoRA loading library (e.g., `peft`).  
    - Write a small test script in `plato-ensign/tests/` that loads an ensign and runs a forward pass.  
    - This confirms the pipeline is functional end-to-end.
