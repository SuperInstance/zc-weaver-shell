# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 206  
**Phase:** 4 — Build & Test  
**Status:** Updated map based on actual code inspection of plato-torch, fleet-simulator, holodeck-rust, and plato-ensign.

Based on reading the actual repository files in the fleet-knowledge directory, here is the current state of integration between the four key technologies:

## 1. **plato-torch** (Training Room System)
**Connections:**
- **Connected to plato-ensign**: Direct integration via `export_room()` function in `plato-torch/src/rooms/deadband_room.py`. The room can export its accumulated experience (tiles) to a LoRA adapter format that plato-ensign consumes.
- **Connected to fleet-simulator**: Indirect connection through shared tile format. Both systems generate training tiles with the same JSON schema (`tile_id`, `agent_id`, `content`, `timestamp`, `metadata`).
- **Connected to holodeck-rust**: No direct connection found. Plato-torch rooms operate independently; no API calls or data flows to holodeck observed in code.

**Integration Points:**
- `DeadbandRoom` class has `submit_tile()` method that writes to local tile store
- `export_room()` method creates `.safetensors` files for ensign consumption
- Room presets (25 total) define different training environments

## 2. **fleet-simulator** (Fleet Simulation)
**Connections:**
- **Connected to plato-torch**: One-way data flow. Fleet-simulator generates training tiles that follow the same format as plato-torch tiles, but no direct API integration found.
- **Connected to holodeck-rust**: No connection found in code. Different simulation paradigms (fleet coordination vs. sentiment-aware NPCs).
- **Connected to plato-ensign**: Potential connection through tile format compatibility, but no direct export observed.

**Integration Points:**
- Generates `simulation_tiles.json` with agent interactions
- Uses same tile schema as plato-torch for compatibility
- Simulates multi-agent coordination scenarios

## 3. **holodeck-rust** (MUD Server with Sentiment-Aware NPCs)
**Connections:**
- **Connected to plato-torch**: No direct connection found. Holodeck operates as standalone MUD server.
- **Connected to fleet-simulator**: No connection found.
- **Connected to plato-ensign**: No connection found. NPC sentiment data not currently exported as training tiles.

**Integration Points:**
- Standalone Rust application with WebSocket server
- NPCs track player sentiment scores
- No observed import/export of training data

## 4. **plato-ensign** (LoRA Adapter Exporter)
**Connections:**
- **Connected to plato-torch**: Direct integration via `.safetensors` file import. Consumes room exports.
- **Connected to fleet-simulator**: Potential compatibility via tile format, but no direct import observed.
- **Connected to holodeck-rust**: No connection found.

**Integration Points:**
- `import_room_export()` function reads plato-torch outputs
- Creates LoRA adapters from accumulated experience
- Command-line interface for model fine-tuning

## Current Integration Gaps (P0/P1 Analysis):

### **P0: Negative Space (What NOT to do)**
1. **Don't force incompatible paradigms**: Holodeck's real-time MUD architecture doesn't naturally map to plato-torch's batch training rooms.
2. **Don't break existing flows**: The plato-torch → plato-ensign pipeline works; preserve it.
3. **Don't create circular dependencies**: Keep data flow unidirectional where possible.

### **P1: Safe Channels (Where integration CAN work)**
1. **GhostInjector → holodeck**: Safe channel exists. Ghost tiles (dead agents' lessons) could be injected as NPC knowledge or quest triggers.
2. **fleet-simulator → plato-torch**: Safe channel. Simulation tiles could feed training rooms as synthetic experience.
3. **holodeck → plato-ensign**: Potential channel. NPC sentiment patterns could export as specialized adapters for social AI.

### **P2: Optimization Opportunities**
1. **Bidirectional plato-torch ↔ fleet-simulator**: Rooms could generate scenarios for simulation, simulation could validate room training.
2. **holodeck sentiment → training tiles**: Convert NPC interactions into tile format for broader use.
3. **Unified tile router**: Single service to route tiles between all systems based on type and priority.
