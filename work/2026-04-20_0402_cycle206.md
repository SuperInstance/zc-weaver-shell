## 1. Verified Connections (What's Connected)

### plato-torch ↔ fleet-simulator
- **Connection:** Direct import and usage.
- **Evidence:** `fleet-simulator/src/lib.rs` exports `FleetSimulator` struct and `SimulationResult`. `plato-torch/src/rooms/deadband_room.rs` imports `fleet_simulator::FleetSimulator` and uses it to generate training data.
- **Data Flow:** `DeadbandRoom` calls `FleetSimulator::new().run_scenario("basic")` to produce `SimulationResult` (containing `tiles: Vec<Tile>`). These tiles are then passed to the room's training pipeline.
- **Integration Point:** Tight coupling for data generation. The simulator acts as a data source for the training room.

### plato-torch ↔ plato-ensign
- **Connection:** Direct export pathway.
- **Evidence:** `plato-torch/src/rooms/mod.rs` shows room structs have an `export_to_ensign` method (or similar pattern). `plato-ensign/src/lib.rs` defines `EnsignBuilder` which accepts a `room: &impl Room` to extract and compile LoRA adapters.
- **Data Flow:** A trained room's state (weights, interaction history) is passed to `plato-ensign` to create a `.safetensors` adapter file.
- **Integration Point:** Optional post-training export. Not required for room operation, but a defined output channel.

### holodeck-rust (Standalone)
- **Current Status:** No direct code-level connections found to the other three components.
- **Evidence:** `holodeck-rust/Cargo.toml` shows no dependencies on `plato-torch`, `fleet-simulator`, or `plato-ensign`. Its `src/` focuses on MUD server architecture, NPC sentiment systems (`src/npc/sentiment.rs`), and network protocols.
- **Note:** It is part of the fleet's "Key Technologies" but operates in its own domain (sentiment-aware simulation). It is a potential *consumer* of training data or *producer* of interaction tiles, but no wiring exists yet.

## 2. Integration Gaps (What's Not Connected)

### Gap 1: holodeck-rust ↔ plato-torch
- **Missing:** Mechanism to send `holodeck-rust` interaction logs (player/NPC dialogues, actions) to a `plato-torch` room as training tiles.
- **Potential Integration Point:** A `HolodeckTileGenerator` adapter in `plato-torch` that consumes `holodeck-rust` log streams or event bus messages, converting them to the `Tile` format.

### Gap 2: holodeck-rust ↔ fleet-simulator
- **Missing:** Shared simulation state or scenario definitions. They simulate different domains (fleet operations vs. social MUD).
- **Potential Integration Point:** `fleet-simulator` could provide background "world state" (e.g., fleet events) that influences `holodeck-rust` NPC behavior or room descriptions.

### Gap 3: plato-ensign → holodeck-rust
- **Missing:** Ability to load a trained ensign (LoRA) into `holodeck-rust` NPCs to specialize their behavior.
- **Potential Integration Point:** `holodeck-rust` NPC system could be extended to load `.safetensors` adapters, allowing NPCs to embody specific room-trained personalities or knowledge.

### Gap 4: Ghost Tile Injection
- **Missing:** Systematic wiring of "ghost tiles" (P0 lessons from dead agents) into the training pipeline of `plato-torch` rooms.
- **Potential Integration Point:** A `GhostInjector` module that reads from a fleet memory store and pre-seeds rooms with negative example tiles before training begins.

## 3. Current End-to-End Pipeline (As Found)
```
fleet-simulator (generates tiles)
        ↓
plato-torch::DeadbandRoom (trains on tiles)
        ↓
plato-ensign (exports room to LoRA)
```
**holodeck-rust runs in parallel, unconnected.**

## 4. Recommended Next Integration Task
**Wire GhostInjector into holodeck-rust (from backlog).**  
- **Why:** `holodeck-rust` NPC sentiment system could use P0 ghost tiles as "cautionary memories" to avoid negative interaction patterns.  
- **How:** Create a `ghost_injector` crate that reads ghost tile files and exposes an API to query "negative examples" for a given context. `holodeck-rust`'s NPC decision-making can call this API before choosing actions.  
- **Outcome:** Creates a direct link between fleet memory (P0 knowledge) and live simulation behavior, closing a key integration gap.
