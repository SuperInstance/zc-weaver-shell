## 1. Verified Current Connections

Based on reading the actual fleet repos:

### **plato-torch ↔ fleet-simulator**
- **Connection:** Strong, bidirectional data flow.
- **Evidence:** `fleet-simulator/src/export.rs` exports simulation logs in `Tile` format (struct defined in `plato-torch/src/tile.rs`).
- **Direction:** fleet-simulator → plato-torch (training data generation).
- **Status:** **Connected and functional.** Simulation runs produce tiles that can be ingested by training rooms.

### **plato-torch ↔ plato-ensign**
- **Connection:** Direct, core dependency.
- **Evidence:** `plato-ensign/src/lib.rs` imports `plato_torch::{Room, Tile}`. Its `export_room` function takes a trained `Room` and produces a LoRA adapter.
- **Direction:** plato-torch → plato-ensign (export pipeline).
- **Status:** **Connected and functional.** This is the primary ensign creation pathway.

### **holodeck-rust ↔ (none yet)**
- **Connection:** Currently isolated.
- **Evidence:** `holodeck-rust/Cargo.toml` shows no dependencies on other fleet crates. Its `NPC` and `SentimentEngine` are self-contained.
- **Direction:** No active data flow to/from other core fleet components.
- **Status:** **Not connected.** Identified as a major integration gap.

### **fleet-simulator ↔ holodeck-rust**
- **Connection:** None.
- **Evidence:** No shared types, imports, or data pathways in either codebase.
- **Status:** **Not connected.**

### **plato-ensign ↔ fleet-simulator**
- **Connection:** Indirect via plato-torch.
- **Evidence:** Ensigns can be trained from rooms that consumed simulator tiles, but no direct link.
- **Status:** **Connected indirectly** through the plato-torch intermediary.

## 2. Identified Integration Gaps (P1 Safe Channels)

### **Gap 1: holodeck-rust → plato-torch**
- **Problem:** NPC interactions and sentiment logs are not captured as training tiles.
- **Safe Channel:** Add a `Tile` exporter to `holodeck-rust/src/logger.rs`. Follow the same `Tile` struct schema from `plato-torch`.
- **Action:** Create `holodeck-to-tile` bridge module. Low risk; follows existing pattern.

### **Gap 2: fleet-simulator → holodeck-rust**
- **Problem:** Simulated fleet events cannot drive NPC behavior or sentiment shifts.
- **Safe Channel:** Expose a lightweight gRPC or WebSocket endpoint in `holodeck-rust` to accept `SimulationEvent` messages (from `fleet-simulator/src/events.rs`).
- **Action:** Design event adapter. Medium risk; new protocol needed.

### **Gap 3: plato-ensign → holodeck-rust**
- **Problem:** Trained ensigns (LoRAs) cannot be loaded to influence NPC dialogue or decision weights.
- **Safe Channel:** Extend `holodeck-rust`'s `NPC` to allow swapping a `BehaviorModel` trait object, where one implementation wraps a loaded LoRA.
- **Action:** Requires `plato-ensign` to expose a runtime loader. Higher risk; needs careful testing.

### **Gap 4: Deadband Protocol ↔ All Components**
- **Problem:** The Deadband Protocol (P0/P1/P2) is documented but not instrumented in code.
- **Safe Channel:** Introduce a `deadband` crate with `Phase` enum and `DeadbandLogger` that all components can use to tag their operations.
- **Action:** Cross-cutting concern. Start with logging, then integrate into decision points.
