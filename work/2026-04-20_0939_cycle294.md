**Integration Points:**

## 2. plato-ensign
**Repo:** `plato-ensign`  
**Purpose:** Exports room experience into LoRA adapters.  
**Key Files:**  
- `export/lora_exporter.py` — Contains `export_lora_adapter()` function.  
- `utils/tile_processor.py` — Processes interaction tiles from rooms.

**Connections Inbound:**  
- **From plato-torch:** `export_lora_adapter()` is called by `DeadbandRoom.export_to_lora()`.  
- **From fleet-simulator:** None found.  
- **From holodeck-rust:** None found.

**Connections Outbound:**  
- Exports LoRA adapters (`.safetensors` files) to disk. No direct API calls to other fleet components.

**Integration Points:**  
- Single integration: `plato-torch` → `plato-ensign` for LoRA export.  
- Tile format expected: List of interaction dictionaries with `agent_id`, `action`, `reward`, `state`.

## Verified Gaps (P0: What's Not Connected)
1. **fleet-simulator → plato-torch**: No pipe to feed simulated tiles into training rooms.  
2. **holodeck-rust → plato-torch/plato-ensign**: No bridge from MUD interactions to training pipeline.  
3. **GhostInjector** (mentioned in backlog) not found in any repo.  
4. **plato-relay** not found in any repo.  
5. **cudaclaw, flux-runtime, constraint-theory-core** are isolated infrastructure.

## Safe Channels (P1: Where Integration Could Work)
1. **File-based tile exchange**: `fleet-simulator` writes tiles to `.jsonl`, `plato-torch` reads them.  
2. **HTTP endpoint**: `holodeck-rust` POSTs tiles to `plato-torch` room API.  
3. **GhostInjector** could be a module in `plato-torch` that loads ghost tiles from memory.  
4. **plato-relay** could be a message bus (Redis/pub-sub) connecting rooms, simulator, holodeck.
