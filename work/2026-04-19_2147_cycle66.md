## 1. Verified Current Connections

Based on reading the actual code and documentation in the fleet repos:

### **plato-torch → plato-ensign** (Strong, Direct)
- **Connection:** `plato-torch` rooms generate training tiles (JSONL files).
- **Mechanism:** `plato-ensign` imports these tiles via `ensign/tile_loader.py` and converts them into LoRA adapters.
- **Evidence:** `plato-ensign/README.md` states: "Exports room experience from plato-torch into LoRA adapters."
- **Status:** **Connected and functional.** This is the core training pipeline.

### **plato-torch → fleet-simulator** (Indirect, Data Flow)
- **Connection:** `fleet-simulator` generates synthetic training data (simulated agent interactions).
- **Mechanism:** This data can be formatted as tiles and ingested by `plato-torch` rooms for training.
- **Evidence:** `fleet-simulator/README.md`: "Generates training data for fleet AI." `plato-torch` rooms accept external tile files.
- **Status:** **Potentially connected** but not explicitly wired. A manual export/import step is likely required.

### **holodeck-rust → plato-torch** (Weak, Conceptual)
- **Connection:** `holodeck-rust` is a MUD server with sentiment-aware NPCs.
- **Mechanism:** NPC interactions could be logged and formatted as tiles for `plato-torch`.
- **Evidence:** No import/export code found. `holodeck-rust` is standalone.
- **Status:** **Not connected.** This is a prime integration gap.

### **plato-ensign → fleet-simulator / holodeck-rust** (None)
- **Connection:** Ensigns (trained LoRAs) could be loaded into simulators or game engines.
- **Mechanism:** No evidence of runtime integration.
- **Status:** **Not connected.** Ensigns are currently static exports.

### **constraint-theory-core, cudaclaw, flux-runtime** (Isolated)
- These are core technologies referenced in the fleet context but show no direct code-level integration with the training stack (`plato-torch`, `plato-ensign`).
- **Status:** **Independent components.** Potential future integration points.

## 2. Identified Integration Gaps (P1: Safe Channels to Build)

### **Gap 1: holodeck-rust → plato-torch Tile Bridge**
- **Problem:** Valuable NPC interaction data is trapped in the MUD.
- **Safe Channel:** Build a logging middleware in `holodeck-rust` that outputs JSONL in `plato-torch` tile format.
- **Action:** Create `holodeck-rust/src/tile_logger.rs` and a configurable export path.

### **Gap 2: fleet-simulator → plato-torch Automated Pipeline**
- **Problem:** Simulated data requires manual processing to become tiles.
- **Safe Channel:** Add a `--export-tiles` flag to `fleet-simulator` that writes directly to a `plato-torch` room's tile directory.
- **Action:** Modify simulator output module, coordinate directory structure with `plato-torch/rooms/`.

### **Gap 3: plato-ensign → holodeck-rust Live NPC Adapter**
- **Problem:** Trained ensigns (specialist personalities) cannot be used to drive NPCs.
- **Safe Channel:** Create a `EnsightNPC` wrapper in `holodeck-rust` that loads a `.safetensors` LoRA and interfaces with the sentiment engine.
- **Action:** Research Rust + PyTorch C++ bindings or ONNX runtime for inference.

### **Gap 4: DeadbandRoom Visual Demo → plato-relay**
- **Problem:** The Deadband Protocol demo (Tinker's task) is isolated.
- **Safe Channel:** Connect the demo to `plato-relay` (if it exists) to show real-time tile flow.
- **Action:** Locate `plato-relay` service, implement WebSocket or SSE stream from demo.
