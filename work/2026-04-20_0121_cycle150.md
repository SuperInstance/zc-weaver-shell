# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 150  
**Phase:** 4 — Build & Test  
**Status:** Mapping current connections and identifying integration points.

Based on reading the actual files in the fleet repos, here is the current integration state between plato-torch, fleet-simulator, holodeck-rust, and plato-ensign.

## 1. plato-torch → fleet-simulator
**Connection:** **Direct (strong)**  
- `plato-torch/src/rooms/deadband_room.py` imports `fleet_simulator` directly.  
- The `DeadbandRoom` class uses `fleet_simulator` to generate training data via `fleet_simulator.generate_scenario()`.  
- This is a core training pipeline: the room runs simulations, collects interaction tiles, and trains on them.  
- **Integration point:** `deadband_room.py` line 45–60. The simulator is called each training step.

## 2. plato-torch → plato-ensign  
**Connection:** **Direct (export path)**  
- `plato-torch/src/export/ensign_exporter.py` exists.  
- It reads room training checkpoints and exports them as LoRA adapters via `plato-ensign` tools.  
- The export is optional — triggered manually or by room configuration.  
- **Integration point:** `ensign_exporter.py` calls `plato_ensign.create_lora()` with room state.

## 3. plato-torch → holodeck-rust  
**Connection:** **Indirect (planned, not wired)**  
- `holodeck-rust` is a sentiment-aware MUD server meant to generate rich narrative interactions.  
- No import of `holodeck` found in `plato-torch` source files.  
- However, `holodeck-rust/README.md` mentions it can “feed tiles to PLATO rooms.”  
- **Current gap:** No active pipeline from holodeck interactions into plato-torch rooms.

## 4. fleet-simulator → holodeck-rust  
**Connection:** **None observed**  
- `fleet-simulator` generates geometric constraint scenarios and agent decision logs.  
- `holodeck-rust` handles text‑based NPC interactions.  
- No shared data format or API calls between them in the current code.

## 5. holodeck-rust → plato-ensign  
**Connection:** **None observed**  
- `plato-ensign` is designed to export room experience from `plato-torch`.  
- `holodeck-rust` does not currently produce room checkpoints or LoRA‑compatible state.

## 6. fleet-simulator → plato-ensign  
**Connection:** **Indirect via plato-torch**  
- The simulator feeds `DeadbandRoom`, which can later be exported by `plato-ensign`.  
- No direct simulator‑to‑ensign pathway.

## Integration Gaps (P1 Safe Channels to Explore)

### Gap 1: holodeck‑rust tile ingestion into plato-torch  
**Opportunity:** Wire `holodeck-rust` to write interaction tiles to a shared tile store (e.g., `tiles/` directory) that `DeadbandRoom` can read.  
**Risk:** Tile format mismatch. Holodeck tiles are narrative; fleet‑simulator tiles are geometric.  
**P1 action:** Define a common tile schema (JSON) both can produce.

### Gap 2: GhostInjector into holodeck  
**Opportunity:** `plato-torch` has `GhostInjector` to replay dead agents’ tiles. Could inject these into holodeck as “memory ghosts” for NPCs.  
**Risk:** Holodeck’s sentiment engine may not handle ghost tile format.  
**P1 action:** Check if holodeck has a plugin interface for external memory.

### Gap 3: DeadbandRoom to plato-relay  
**Opportunity:** `plato-relay` (mentioned in configs) could broadcast room state to other fleet components.  
**Risk:** `plato-relay` may not be running or may have different API.  
**P1 action:** Locate `plato-relay` service definition and test connection.

### Gap 4: End‑to‑end pipeline test  
**Current flow:** fleet‑simulator → DeadbandRoom → training → (optional) plato‑ensign export.  
**Missing:** Validation that exported LoRA works in inference.  
**P1 action:** Create a test that trains a minimal room, exports an ensign, and runs a inference with it.
