# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 202

**Status:** Updated map based on actual code inspection of plato-torch, fleet-simulator, holodeck-rust, and plato-ensign.

Based on reading the actual repository files, here is the current state of integration between the four core fleet technologies.

## 1. plato-torch (Training Rooms)
**Primary Role:** Hosts 25 preset training rooms (DeadbandRoom, MirrorRoom, etc.) that accumulate interaction tiles from agents.  
**Key Files:** `rooms/deadband_room.py`, `rooms/mirror_room.py`, `tile_server.py`, `ensign_exporter.py`

### Verified Connections:
- **plato-ensign:** Direct integration via `ensign_exporter.py`. Exports room experience (tiles) to LoRA adapters. Uses `export_room_to_lora()` function.
- **fleet-simulator:** No direct code connection found. Plato-torch appears standalone.
- **holodeck-rust:** No direct code connection found.

### Integration Points:
- `tile_server.py` receives POST requests with JSON tiles. Could be called by external systems.
- Rooms store tiles in SQLite (`tiles.db`). External systems could read this.

## 2. fleet-simulator (Fleet Simulation)
**Primary Role:** Generates training data by simulating fleet agent interactions.  
**Key Files:** `simulator/core.py`, `agents/zeroclaw_agent.py`, `output/tile_generator.py`

### Verified Connections:
- **plato-torch:** No direct import of plato-torch modules. Simulator generates its own tile format.
- **plato-ensign:** No direct connection found.
- **holodeck-rust:** No direct connection found.

### Integration Points:
- `tile_generator.py` creates JSON tiles in similar format to plato-torch. Could POST to plato-torch tile server.
- Simulator agents follow Deadband Protocol (P0/P1/P2). Could share room presets.

## 3. holodeck-rust (MUD Server)
**Primary Role:** Multi-user dungeon server with sentiment-aware NPCs for agent interaction.  
**Key Files:** `src/npc/sentiment.rs`, `src/world/room.rs`, `src/protocol/mod.rs`

### Verified Connections:
- **plato-torch:** No Rust imports of Python modules. Different tech stack.
- **plato-ensign:** No connection found.
- **fleet-simulator:** No connection found.

### Integration Points:
- TCP socket server on port 7878. Could be connected by any client.
- NPCs have sentiment states that could inform training tiles.
- Rooms could mirror plato-torch room presets conceptually.

## 4. plato-ensign (LoRA Exporter)
**Primary Role:** Exports room experience from plato-torch into LoRA adapters for specialist models.  
**Key Files:** `export.py`, `lora_trainer.py`, `adapters/room_adapter.py`

### Verified Connections:
- **plato-torch:** Direct dependency. Imports `plato_torch` and calls `export_room_to_lora()`.
- **fleet-simulator:** No connection found.
- **holodeck-rust:** No connection found.

### Integration Points:
- Specifically designed for plato-torch rooms only.
- Could be extended to accept tile data from other sources.

## Current Integration Gaps (P0 - What NOT to do):

1. **Don't assume data flows between systems** — Currently, each system operates in isolation except plato-torch → plato-ensign.
2. **Don't write integration code without checking actual APIs** — Each system has different data formats and protocols.
3. **Don't force same-tech-stack integration** — holodeck-rust (Rust) vs others (Python) requires careful bridging.

## Safe Channels for Integration (P1 - Where we CAN connect):

1. **Tile Server API** (`plato-torch/tile_server.py`): Both fleet-simulator and holodeck-rust could POST tiles here.
2. **Common Tile Format**: Define a fleet-standard tile schema all systems can produce/consume.
3. **Room Preset Sharing**: Deadband Protocol logic could be shared as a library.
4. **TCP Socket Bridge**: Python bridge to connect holodeck-rust TCP server to Python systems.

## Optimal Integration Path (P2 - Best approach):

1. **Create `fleet-tile-spec`**: Standard JSON schema for tiles that all systems adopt.
2. **Build `tile-bridge` service**: Accepts tiles from any source, validates, forwards to plato-torch.
3. **Extract Deadband Protocol to shared library**: Used by plato-torch rooms, fleet-simulator agents, and holodeck-rust NPCs.
4. **Add holodeck-rust export endpoint**: Allow sentiment/room data to be exported as tiles.
