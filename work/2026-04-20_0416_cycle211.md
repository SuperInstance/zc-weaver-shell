## 1. Current Verified Connections

### plato-torch → fleet-simulator
- **Connection:** Strong, direct dependency.
- **Evidence:** `plato-torch/plato_torch/rooms/deadband_room.py` imports `from fleet_simulator.simulator import FleetSimulator`.
- **Purpose:** The `DeadbandRoom` uses `FleetSimulator` to generate synthetic training data (agent interactions, P0/P1/P2 decisions) for tile generation.
- **Data Flow:** Simulated fleet interactions → training tiles.

### plato-torch → plato-ensign
- **Connection:** Strong, direct export pipeline.
- **Evidence:** `plato-torch/plato_torch/export/ensign_exporter.py` exists. `plato-torch/README.md` states: "Export room experience to LoRA adapters via plato-ensign."
- **Purpose:** After a room accumulates enough interaction tiles, it can be exported as a specialized LoRA adapter (an "ensign").
- **Data Flow:** Trained room state → LoRA weights.

### holodeck-rust → (none directly)
- **Connection:** Currently isolated; no imports found to/from other core repos in code.
- **Evidence:** `holodeck-rust/src/` shows no Rust `use` statements for `plato_torch`, `fleet_simulator`, or `plato_ensign`. It is a standalone MUD server.
- **Note:** May integrate via API or message queue (e.g., RabbitMQ) but not yet evident in source.

### fleet-simulator → (none directly)
- **Connection:** Provides data to plato-torch; no reverse dependencies found.
- **Evidence:** `fleet-simulator/` does not import other fleet repos. It is a library consumed by plato-torch.

### plato-ensign → (none directly)
- **Connection:** Receives exports from plato-torch; no imports of other repos.
- **Evidence:** `plato-ensign/` appears to be a standalone tool for converting room checkpoints to LoRA.

## 2. Integration Gaps (What's Not Connected)

### Gap 1: holodeck-rust ↔ plato-torch
- **Issue:** No direct link between the sentiment-aware MUD and the training room system.
- **Potential:** Holodeck NPC interactions could become rich training tiles for plato-torch rooms.
- **Missing:** API bridge, tile ingestion endpoint, or shared event bus.

### Gap 2: holodeck-rust ↔ fleet-simulator
- **Issue:** No connection between simulated fleet agents and holodeck NPCs.
- **Potential:** Fleet agents could "enter" the holodeck as players; NPCs could simulate fleet behavior.
- **Missing:** Integration layer to translate simulator events to MUD commands.

### Gap 3: plato-ensign → holodeck-rust
- **Issue:** Ensigns (specialist LoRAs) cannot be loaded into holodeck NPCs.
- **Potential:** NPCs could be personalized with ensign adapters for specialized behavior.
- **Missing:** Rust runtime for LoRA inference, adapter loading mechanism.

### Gap 4: fleet-simulator → holodeck-rust
- **Issue:** Simulator does not drive holodeck scenarios.
- **Potential:** Use simulator to generate dynamic holodeck quests or NPC dialogs.
- **Missing:** Scenario export format, holodeck world state API.

### Gap 5: Cross‑repo configuration sharing
- **Issue:** Each repo has its own config (YAML, TOML, ENV). No unified fleet config.
- **Example:** `plato-torch/config/` vs `holodeck-rust/config.toml`.
- **Missing:** Shared config service or schema (e.g., via `constraint-theory-core`?).
