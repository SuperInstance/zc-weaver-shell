## 1. plato-torch → fleet-simulator
**Connection:** Strong, direct dependency.  
**Evidence:**  
- `plato-torch/requirements.txt` includes `fleet-simulator>=0.1.0`  
- `plato-torch/src/training/rooms/deadband_room.py` imports `from fleet_simulator import FleetSimulator`  
- `plato-torch/configs/room_presets/deadband.yaml` references `simulator_config_path: ../fleet-simulator/configs/basic_fleet.yaml`  

**Data Flow:**  
DeadbandRoom instantiates a FleetSimulator object. Room steps call `simulator.step()`. Agent observations from the simulator are wrapped into training tiles.

**Integration Gap:**  
No bidirectional control flow observed. The simulator does not call back into plato-torch. It's a data source, not an integrated runtime.

## 2. plato-torch → plato-ensign
**Connection:** Optional export pipeline.  
**Evidence:**  
- `plato-torch/src/training/exporter.py` defines `EnsignExporter` class  
- `plato-torch/scripts/export_ensign.py` loads a trained room checkpoint and calls `exporter.export_to_lora()`  
- `plato-ensign/README.md` states: "Import a trained room from plato-torch via the export script."

**Data Flow:**  
Post-training, a room's accumulated interaction tiles and model weights can be serialized into a LoRA adapter file (`.safetensors`). This is a one-time export, not a live connection.

**Integration Gap:**  
No observed live API between plato-torch and plato-ensign during training. The ensign is an offline artifact.
