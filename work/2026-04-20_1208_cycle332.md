# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 332

**Status:** Direct file inspection of fleet repositories. Focus on actual imports, configuration references, and shared data structures.

## 1. plato-torch → fleet-simulator
**Connection:** **Strong**  
- `plato-torch/rooms/deadband_room.py` imports `fleet_simulator` directly:  
  ```python
  from fleet_simulator import FleetSimulator
  ```
- `DeadbandRoom` instantiates `FleetSimulator` as `self.simulator`.  
- Simulator provides `step()` method returning `(observation, reward, done, info)`.  
- Room uses simulator state to generate training tiles.  
- **Data flow:** Simulator observations → Room tile generation → Plato server.

**Integration Gap:** None observed. This is a core, wired dependency.

## 2. plato-torch → plato-ensign
**Connection:** **Direct via export API**  
- `plato-torch/rooms/base_room.py` defines `export_to_ensign()` method.  
- Method serializes room’s `self.memory` (list of interaction tiles) to a LoRA‑compatible format.  
- `plato-ensign/` provides `train_lora.py` which expects exactly this export format.  
- **Data flow:** Room memory → Ensign export → LoRA training.

**Integration Gap:** No automatic trigger observed. Export is manual (call `room.export_to_ensign()`). Could be scheduled or event‑driven.

## 3. plato-torch → holodeck-rust
**Connection:** **None observed**  
- No import of `holodeck` in any `plato-torch` Python file.  
- No shared configuration files referencing holodeck.  
- Plato rooms are Python‑based; holodeck is a Rust MUD server.  
- **Potential link:** Both produce “interaction tiles,” but formats differ.

**Integration Gap:** **Major**. No direct wiring. Tiles from holodeck cannot currently feed into plato‑torch rooms without an adapter.

## 4. fleet-simulator → holodeck-rust
**Connection:** **None observed**  
- `fleet-simulator` is pure Python, no Rust bindings.  
- No shared protocol or message format found.  
- Simulator models fleet‑level dynamics; holodeck models individual NPC sentiment.  
- **Potential link:** Simulator could provide world‑state context to holodeck NPCs.

**Integration Gap:** **Major**. Separate domains, no current bridge.

## 5. holodeck-rust → plato-ensign
**Connection:** **Theoretical**  
- `holodeck-rust` produces “sentiment‑aware” interaction logs.  
- `plato-ensign` can train LoRAs from any tile‑structured data.  
- No observed Rust‑to‑Python export pipeline in code.  
- **Data flow:** Could be: holodeck logs → tile conversion → ensign training.

**Integration Gap:** **Missing adapter**. Need a script to convert holodeck JSON logs to plato‑ensign input format.

## 6. fleet-simulator → plato-ensign
**Connection:** **Indirect via plato-torch**  
- Simulator data flows through `DeadbandRoom` to become tiles.  
- Those tiles can be exported to ensign.  
- No direct simulator‑to‑ensign observed.  
- **Data flow:** Simulator → Room → Ensign.

**Integration Gap:** No direct link required; room acts as intermediary.

## 7. Ghost Tiles & Deadband Protocol Integration
**Connection:** **Partially implemented**  
- `plato-torch/rooms/deadband_room.py` references `ghost_tiles` in `self.memory` initialization.  
- Ghost tiles are loaded from file if present (P0 negative‑space knowledge).  
- **Integration Gap:** No observed “GhostInjector” component. Ghost tiles are static file‑based, not dynamically injected.
