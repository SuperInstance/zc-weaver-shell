## 1. Verified Current Connections (What's Connected)

Based on reading fleet-knowledge/ and repository structures:

### **plato-torch → plato-ensign**  
- **Connection:** Direct export pipeline  
- **Mechanism:** `plato-torch` training rooms accumulate interaction tiles → `plato-ensign` exports room experience into LoRA adapters  
- **Evidence:** `plato-ensign/README.md` states: "Exports room experience from plato-torch into LoRA adapters for specialist models"  
- **Status:** **Connected** — functional pipeline exists  

### **plato-torch → fleet-simulator**  
- **Connection:** Training data source  
- **Mechanism:** `fleet-simulator` generates synthetic training data (agent interactions, environment states) → feeds into `plato-torch` rooms as training tiles  
- **Evidence:** `fleet-simulator/` contains `data/` subdirectory with generated JSON logs; `plato-torch/rooms/` imports from `fleet-simulator/data/`  
- **Status:** **Connected** — one-way data flow confirmed  

### **holodeck-rust → plato-torch**  
- **Connection:** Sentiment-aware NPC interactions as training tiles  
- **Mechanism:** `holodeck-rust` MUD server generates NPC dialogue with sentiment tracking → exports interaction logs → ingested by `plato-torch` rooms  
- **Evidence:** `holodeck-rust/src/export.rs` contains `export_to_plato()` function; `plato-torch/import/holodeck_parser.py` exists  
- **Status:** **Connected** — integration code present  

### **constraint-theory-core → fleet-simulator**  
- **Connection:** Geometric constraint engine for environment generation  
- **Mechanism:** `constraint-theory-core` provides Rust geometric snapping algorithms → used by `fleet-simulator` to generate spatially consistent environments  
- **Evidence:** `fleet-simulator/Cargo.toml` includes `constraint-theory-core = "0.3.0"` dependency  
- **Status:** **Connected** — direct dependency

## 2. Integration Gaps (What's Not Connected)

### **Gap 1: plato-ensign → holodeck-rust**  
- **Issue:** No mechanism to apply trained ensigns (LoRA adapters) to holodeck NPCs  
- **Impact:** NPCs cannot benefit from specialized training; sentiment responses remain static  
- **Evidence:** `holodeck-rust/` has no import mechanism for LoRA files; `plato-ensign/exports/` only outputs to `.safetensors` format  
- **Priority:** **High** — breaks feedback loop between training and deployment  

### **Gap 2: cudaclaw → plato-torch**  
- **Issue:** GPU-resident agent runtime not integrated with training rooms  
- **Impact:** Cannot train agents directly on GPU hardware; simulation→training pipeline remains CPU-bound  
- **Evidence:** `cudaclaw/` exists as standalone runtime; `plato-torch/training/` uses PyTorch CPU/GPU but no cudaclaw hooks  
- **Priority:** **Medium-High** — performance bottleneck for large-scale training  

### **Gap 3: flux-runtime → Any production system**  
- **Issue:** Self-assembling bytecode VM exists but isn't wired into any fleet component  
- **Impact:** Theoretical capability not utilized; potential for dynamic code generation untapped  
- **Evidence:** `flux-runtime/` has comprehensive README but no imports in other repos' dependency files  
- **Priority:** **Medium** — experimental technology needs integration point  

### **Gap 4: Ghost tiles → holodeck-rust**  
- **Issue:** Dead agents' lessons (ghost tiles) not injected into holodeck NPC knowledge  
- **Impact:** P0 knowledge (negative space mapping) not propagated to live environments  
- **Evidence:** `plato-torch/rooms/` mentions ghost tiles but no export to holodeck; `holodeck-rust/npc/knowledge/` has static files only  
- **Priority:** **High** — violates Deadband Protocol P0 principle propagation  

### **Gap 5: Mirror Play → fleet-simulator**  
- **Issue:** Two PLATO systems iterating screen-to-screen not connected to simulation  
- **Impact:** Cannot generate training data from mirror play interactions  
- **Evidence:** `plato-torch/mirror/` directory exists but no connection to `fleet-simulator/data/` import  
- **Priority:** **Medium** — missed opportunity for emergent behavior capture
