## 1. Plato-Torch (Training Rooms)
**Location:** `plato-torch/`  
**Purpose:** 25 training room presets, self-training rooms, tile ingestion and processing.

### Verified Connections:
- **To Plato-Ensign:** Direct export via `export_ensign()` function in `plato-torch/src/rooms/deadband_room.py`. Calls `plato_ensign.export.export_lora()` with room state and tile history.
- **To Fleet-Simulator:** Referenced in `plato-torch/configs/training_presets.yaml` under `data_source: fleet_simulator`. Simulator data can be loaded as training tiles.
- **To Holodeck-Rust:** No direct import found. However, `plato-torch/README.md` mentions "sentiment-aware NPCs from holodeck" as a future integration target.
- **To Constraint-Theory-Core:** Not referenced in code. Potential integration point for geometric training data.

### Integration Points:
- `plato-torch/src/rooms/base_room.py` — Base class for all rooms, handles tile ingestion.
- `plato-torch/src/export/ensign_exporter.py` — Bridge to plato-ensign.
- `plato-torch/configs/` — YAML configs define data sources and room parameters.

## 2. Fleet-Simulator (Training Data Generation)
**Location:** `fleet-simulator/`  
**Purpose:** Generate synthetic training data for fleet scenarios.

### Verified Connections:
- **To Plato-Torch:** Outputs JSON tile format compatible with `plato-torch/src/data/tile_loader.py`. Used as a data source in training presets.
- **To Holodeck-Rust:** No direct import. Separate simulation domains (fleet vs. NPC interactions).
- **To Plato-Ensign:** No direct link. Data flows through plato-torch rooms first.
- **To Cudaclaw:** Simulator can generate GPU-resident agent trajectories (mentioned in `fleet-simulator/docs/design.md`).

### Integration Points:
- `fleet-simulator/output/tiles/` — Generated tiles follow plato-torch schema.
- `fleet-simulator/src/scenarios/deadband_scenario.rs` — Scenario that aligns with Deadband Protocol.

## 3. Holodeck-Rust (MUD Server with Sentiment-Aware NPCs)
**Location:** `holodeck-rust/`  
**Purpose:** Multi-User Dungeon server with NPCs that have emotional state.

### Verified Connections:
- **To Plato-Torch:** Not directly connected. README mentions future integration for "NPC dialogue as training tiles".
- **To Plato-Ensign:** No direct link. NPC interactions could be exported as LoRA adapters for character personalities.
- **To Fleet-Simulator:** Separate domain. No shared code.
- **To Constraint-Theory-Core:** Used in `holodeck-rust/src/world/geometry.rs` for spatial snapping of NPC movement.

### Integration Points:
- `holodeck-rust/src/npc/sentiment.rs` — Emotional state engine.
- `holodeck-rust/src/api/` — REST API for external integration (unused by other repos currently).

## 4. Plato-Ensign (LoRA Adapter Export)
**Location:** `plato-ensign/`  
**Purpose:** Export room experience into LoRA adapters for specialist models.

### Verified Connections:
- **To Plato-Torch:** Direct import in `plato_ensign/export/export_lora.py`. Receives room state and tiles.
- **To Fleet-Simulator:** No direct connection. Requires plato-torch as intermediary.
- **To Holodeck-Rust:** No connection found.
- **To Flux-Runtime:** `plato-ensign/src/adapters/flux_adapter.rs` — Experimental integration for bytecode-aware LoRA.

### Integration Points:
- `plato-ensign/export/export_lora.py` — Main export function.
- `plato-ensign/configs/lora_config.yaml` — LoRA hyperparameters.

## 5. Constraint-Theory-Core (Geometric Snapping)
**Location:** `constraint-theory-core/`  
**Purpose:** Rust library for geometric constraint solving.

### Verified Connections:
- **To Holodeck-Rust:** Used as dependency in `Cargo.toml`. Applied for NPC spatial snapping.
- **To Plato-Torch:** Not referenced. Potential for geometric training data.
- **To Fleet-Simulator:** Not referenced. Could be used for fleet formation constraints.

### Integration Points:
- `constraint-theory-core/src/snapping/` — Core snapping algorithms.

## 6. Cudaclaw (GPU-Resident Agent Runtime)
**Location:** `cudaclaw/`  
**Purpose:** Run agents directly on GPU memory.

### Verified Connections:
- **To Fleet-Simulator:** Referenced in design docs as a target runtime for simulated agents.
- **To Plato-Torch:** No direct link. Could accelerate room inference.
- **To Flux-Runtime:** `cudaclaw/src/flux_bridge.cu` — Bridge to flux bytecode VM.

### Integration Points:
- `cudaclaw/include/agent_kernel.h` — GPU agent kernel interface.

## 7. Flux-Runtime (Self-Assembling Bytecode VM)
**Location:** `flux-runtime/`  
**Purpose:** VM that rewrites its own bytecode.

### Verified Connections:
- **To Cudaclaw:** Bridge exists for GPU execution.
- **To Plato-Ensign:** Experimental adapter for LoRA on flux bytecode.
- **To Plato-Torch:** No direct connection.

### Integration Points:
- `flux-runtime/src/vm/` — Core VM components.
