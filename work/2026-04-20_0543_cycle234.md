## 1. plato-torch → fleet-simulator
**Connection:** **Yes** — Direct import and usage.

**Evidence:**
- `plato-torch/plato_torch/rooms/deadband_room.py` imports `fleet_simulator`:
  ```python
  from fleet_simulator import FleetSimulator
  ```
- `DeadbandRoom` instantiates `FleetSimulator` in its `__init__`:
  ```python
  self.simulator = FleetSimulator()
  ```
- The room uses the simulator to generate training data (`self.simulator.generate_training_data()`).

**Integration Point:**  
`DeadbandRoom` uses `fleet_simulator` as a data source for agent interaction simulations. The simulator provides synthetic fleet interaction logs that become training tiles.

**Gap:**  
No observed reverse dependency (fleet-simulator does not import plato-torch). The simulator appears to be a standalone data generator.

## 3. plato-torch → plato-ensign
**Connection:** **Yes** — Direct export pathway.

**Evidence:**
- `plato-torch/plato_torch/ensign_exporter.py` exists.
- `DeadbandRoom` has an `export_to_ensign()` method that calls the exporter.
- Exporter uses PyTorch to save LoRA adapters from room training history.

**Integration Point:**  
`plato-torch` rooms can optionally export their trained weights to `plato-ensign` format (LoRA adapters). This is a one-way export, not a runtime dependency.

**Gap:**  
`plato-ensign` does not appear to be a runtime library; it's an output format. No observed imports from `plato-ensign` into `plato-torch`.
