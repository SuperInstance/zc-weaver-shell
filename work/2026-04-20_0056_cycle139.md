# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 139  
**Phase:** 4 — Build & Test  
**Status:** Mapping current connections and identifying integration points.

Based on reading the actual files in the fleet repos, here is the current state of integration between the four core systems.

## 1. plato-torch → fleet-simulator
**Connection:** ✅ **Connected**  
**Mechanism:** Direct import in `plato-torch/rooms/deadband_room.py`  
**Evidence:**  
```python
from fleet_simulator.simulator import FleetSimulator
```
**Purpose:** The DeadbandRoom uses FleetSimulator to generate synthetic training data (agent interactions, P0/P1/P2 decisions) which becomes training tiles.  
**Data Flow:** Simulator runs → produces JSONL event logs → room ingests logs as tile candidates.  
**Integration Gap:** The simulator configuration is hardcoded in the room preset. No dynamic configuration channel exists.

## 2. plato-torch → plato-ensign
**Connection:** ✅ **Connected**  
**Mechanism:** Ensign is a separate library imported by plato-torch training scripts.  
**Evidence:** `plato-torch/scripts/train_ensign.py` contains:
```python
from plato_ensign import EnsignExporter
```
**Purpose:** After a room accumulates enough tiles, an ensign (LoRA adapter) can be exported, capturing the room’s learned behavior.  
**Data Flow:** Room checkpoint → EnsignExporter → LoRA weights (.safetensors).  
**Integration Gap:** Export is manual/script-triggered. No automatic export based on room performance metrics.

## 3. plato-torch → holodeck-rust
**Connection:** ❌ **Not directly connected**  
**Evidence:** No import of holodeck in plato-torch codebase. No Rust bindings visible.  
**Potential Channel:** Both systems produce/consume “tile-like” interaction data.  
**Integration Gap:** Holodeck’s sentiment-aware NPC interactions are not feeding into plato-torch rooms as training tiles. This is a missed source of rich human‑like dialogue data.

## 4. fleet-simulator → holodeck-rust
**Connection:** ❌ **Not connected**  
**Evidence:** No shared data formats or APIs.  
**Integration Gap:** The simulator’s agent‑agent events could drive holodeck NPC behavior, or holodeck scenarios could seed simulator initial conditions. Currently separate worlds.

## 5. holodeck-rust → plato-ensign
**Connection:** ❌ **Not connected**  
**Evidence:** No export path from holodeck sessions to ensign adapters.  
**Integration Gap:** Holodeck’s “sentiment‑aware” NPC logic could be distilled into a specialized ensign for emotional dialogue. No pipeline exists.

## 6. fleet-simulator → plato-ensign
**Connection:** ⚠️ **Indirect**  
**Mechanism:** Simulator data flows through plato‑torch rooms first, then may be exported via ensign.  
**Integration Gap:** No direct ensign training from raw simulator logs, bypassing the room abstraction. This would allow “pure strategy” ensigns.

## 7. plato-ensign → holodeck-rust
**Connection:** ❌ **Not connected**  
**Integration Gap:** A trained ensign could be loaded into holodeck to guide NPC responses. No runtime integration exists.

## 8. plato-ensign → fleet-simulator
**Connection:** ❌ **Not connected**  
**Integration Gap:** Ensign adapters could be used to steer simulated agents (e.g., P2 optimizer ensign). Currently the simulator uses hardcoded decision trees.

## Integration Priority (P1 Safe Channels)

1. **plato‑torch → holodeck‑rust**  
   - **Safe channel:** Define a common tile schema (JSON) that both can read/write.  
   - **Action:** Write a bridge script that watches holodeck session logs, converts to tiles, posts to a room’s tile intake endpoint.

2. **holodeck‑rust → plato‑ensign**  
   - **Safe channel:** Use the existing `EnsignExporter` but feed it a checkpoint trained on holodeck‑sourced tiles.  
   - **Action:** Create a new room preset `HolodeckRoom` that loads holodeck‑formatted data.

3. **plato‑ensign → holodeck‑rust**  
   - **Safe channel:** Embed the LoRA weights into holodeck’s local LLM inference (if using llama.cpp, etc.).  
   - **Action:** Add a `--ensign` flag to holodeck’s NPC config to load an adapter.

4. **fleet‑simulator → holodeck‑rust**  
   - **Safe channel:** Share scenario definitions (YAML/JSON). Simulator generates “world state”, holodeck renders it as interactive narrative.  
   - **Action:** Write a converter from simulator event logs to holodeck scene descriptions.

## P0 Negative Space (What NOT to Do)

- Do not attempt to rewrite all four codebases into a single monolith.  
- Do not create bidirectional sync between all systems simultaneously—start with unidirectional data flow.  
- Do not assume all systems use the same programming language; use JSON over HTTP or file‑watch patterns.  
- Do not break existing plato‑torch → fleet‑simulator → plato‑ensign pipeline; it is the only fully working chain.
