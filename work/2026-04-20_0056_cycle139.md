# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 139  
**Phase:** 4 — Build & Test  
**Status:** Mapping current connections and identifying integration points.

Based on reading the actual files in the fleet repos, here is the current state of integration between **plato-torch**, **fleet-simulator**, **holodeck-rust**, and **plato-ensign**.

---

## 1. **plato-torch** → **plato-ensign** — **CONNECTED**
- **Connection:** Direct export pipeline.
- **Mechanism:** `plato-torch` rooms accumulate interaction tiles. `plato-ensign` provides `export_room_to_lora()` function that converts room experience (tiles) into a LoRA adapter.
- **Evidence:** `plato-ensign/README.md` states: "Exports room experience from plato-torch into LoRA adapters." The `export` module imports `plato_torch` room classes.
- **Integration Gap:** None observed. This is a core, documented workflow.

---

## 2. **fleet-simulator** → **plato-torch** — **PARTIALLY CONNECTED**
- **Connection:** Training data generation.
- **Mechanism:** `fleet-simulator` generates synthetic training dialogues and scenarios. Its output format (JSONL) matches the tile schema expected by `plato-torch` rooms for ingestion.
- **Evidence:** `fleet-simulator/examples/` contains sample outputs with `role`, `content`, `metadata` fields that align with `plato-torch`'s `Tile` dataclass.
- **Integration Gap:** No automated ingestion pipeline observed. The simulator writes to disk; a manual or scripted step is required to load these into a `plato-torch` room. No direct API call or library import between the two codebases.

---

## 3. **holodeck-rust** → **plato-torch** — **NOT CONNECTED**
- **Connection:** None found.
- **Mechanism:** `holodeck-rust` is a standalone MUD server with sentiment-aware NPCs. It outputs game state and narrative logs.
- **Evidence:** `holodeck-rust/src/main.rs` shows no imports or references to `plato-torch`. Its output is terminal/console and TCP socket based.
- **Integration Gap:** Significant. The NPC interactions and sentiment data from `holodeck-rust` are not being captured as tiles for `plato-torch` training. This is a potential channel for rich, emergent training data.

---

## 4. **plato-torch** → **holodeck-rust** — **NOT CONNECTED**
- **Connection:** None found.
- **Mechanism:** No evidence that trained ensigns (LoRAs) from `plato-torch` are loaded into `holodeck-rust` NPCs to improve their behavior.
- **Integration Gap:** The "sentiment-aware" NPCs could benefit from specialized personality adapters trained in rooms. Currently, they use hard-coded or simple probabilistic response tables.

---

## 5. **fleet-simulator** → **holodeck-rust** — **NOT CONNECTED**
- **Connection:** None found.
- **Mechanism:** The simulator could generate scenario definitions or world states for the MUD, but no such link exists.
- **Evidence:** Separate codebases, no shared configuration or data formats.
- **Integration Gap:** `holodeck-rust` worlds are static (defined in `world/`). No dynamic scenario injection from the simulator.

---

## 6. **plato-ensign** → **Other Components** — **POTENTIAL, NOT CONNECTED**
- **Connection:** LoRA adapters are produced but not automatically deployed.
- **Mechanism:** An exported `.safetensors` file from `plato-ensign` could be consumed by:
    - `cudaclaw` (GPU runtime) to host a specialized agent.
    - `holodeck-rust` (via an inference bridge) to power an NPC.
    - The fleet's own `flux-runtime` for bytecode execution.
- **Integration Gap:** No observed automation or hooks for deploying new ensigns into running systems. The export is an end-point.

---

## **Identified Safe Channels (P1) for Integration Work**

1.  **Bridge holodeck-rust → plato-torch**
    - **Action:** Create a `HolodeckTileLogger` that subscribes to the MUD's event stream (TCP/websocket) and formats NPC/player interactions as tiles, writing them to a `plato-torch` compatible queue or file.
    - **Benefit:** Captures organic, multi-turn dialogues with emotional valence for training.

2.  **Automate fleet-simulator → plato-torch pipeline**
    - **Action:** Extend `plato-torch` with a `SimulatorRoom` preset that periodically reads from `fleet-simulator` output directory and ingests new synthetic data as tiles.
    - **Benefit:** Continuous, automated training data refresh.

3.  **Close the loop: plato-ensign → holodeck-rust**
    - **Action:** Modify `holodeck-rust` NPC logic to load a LoRA adapter (via a local inference server like `llama.cpp` or `vLLM`) and use it to generate responses.
    - **Benefit:** NPCs evolve with fleet training, creating a feedback loop.
