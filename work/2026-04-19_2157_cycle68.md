## 1. Verified Current Connections (What's Connected)

Based on reading the actual repository READMEs and code structure from `fleet-knowledge/`:

### **plato-torch → plato-ensign** (Strong, Direct)
- **Connection:** `plato-torch` generates training rooms and accumulates interaction tiles.
- **plato-ensign** exports room experience (tile data) into LoRA adapters.
- **Evidence:** `plato-ensign` README states it "exports room experience into LoRA adapters." `plato-torch` architecture describes rooms training from accumulated tiles.
- **Integration Point:** Tile data pipeline from room storage → LoRA training.

### **plato-torch → fleet-simulator** (Indirect, Data Flow)
- **Connection:** `fleet-simulator` generates training data (simulated fleet interactions).
- This data can be fed into `plato-torch` rooms as training tiles.
- **Evidence:** `fleet-simulator` purpose is "fleet simulation for training data." `plato-torch` rooms "train from accumulated interaction tiles."
- **Integration Point:** Simulated interaction logs → tile ingestion API.

### **holodeck-rust → plato-torch** (Potential, Unclear)
- **Connection:** `holodeck-rust` is a MUD server with sentiment-aware NPCs.
- NPC interactions could generate tiles for `plato-torch` rooms.
- **Evidence:** No explicit link in READMEs, but both deal with interactive training environments.
- **Integration Point:** Holodeck session logs → tile conversion.

### **constraint-theory-core → (Multiple)** (Library Dependency)
- **Connection:** Rust geometric snapping library. Likely used by `holodeck-rust` (spatial MUD) and potentially `fleet-simulator` (geometry for simulation).
- **Evidence:** No explicit imports noted, but logical fit for spatial/geometric systems.

### **cudaclaw & flux-runtime** (Standalone Runtimes)
- **Connection:** GPU-resident agent runtime (`cudaclaw`) and self-assembling bytecode VM (`flux-runtime`).
- These are infrastructure layers; not directly integrated with training rooms yet.
- **Potential Integration:** Could host trained ensigns (LoRAs) or serve as execution environments for agents trained in rooms.

## 2. Identified Integration Gaps (What's Not Connected)

### **Gap 1: plato-ensign → holodeck-rust**
- **Issue:** No pipeline to deploy trained ensigns (LoRA adapters) into holodeck NPCs.
- **Actionable Integration:** Wire `plato-ensign` export into `holodeck-rust` character brain.
- **P0 Check:** No evidence of existing connection.

### **Gap 2: fleet-simulator → plato-ensign**
- **Issue:** Simulated data trains rooms, but ensigns are only exported from room experience. No direct ensign training from raw simulation.
- **Actionable Integration:** Allow `fleet-simulator` logs to bypass rooms and train ensigns directly (specialist sim-to-LoRA).
- **P0 Check:** `plato-ensign` currently only takes room exports.

### **Gap 3: GhostInjector (ghost tiles) → holodeck-rust**
- **Issue:** Ghost tiles (dead agents' P0 lessons) are stored but not injected into live holodeck sessions.
- **Actionable Integration:** Build `GhostInjector` module to feed ghost tiles as training data or behavior constraints into holodeck NPCs.
- **P0 Check:** Ghost tiles exist in architecture description; no injection point noted.

### **Gap 4: DeadbandRoom → plato-relay**
- **Issue:** `DeadbandRoom` (a `plato-torch` preset) lacks real-time relay to other fleet components.
- **Actionable Integration:** Connect DeadbandRoom API to `plato-relay` (if exists) for live tile streaming.
- **P0 Check:** `plato-relay` not mentioned in current docs; may be missing component.

### **Gap 5: Mirror Play → Active Training Loop**
- **Issue:** Mirror Play (two PLATO systems iterating screen-to-screen) is described but not linked to automated training pipeline.
- **Actionable Integration:** Implement Mirror Play as a self-improving cycle between two rooms, feeding tiles to each other.
- **P0 Check:** Conceptual, not implemented.
