# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 151  
**Phase:** 4 — Build & Test  
**Status:** Mapping current connections and identifying integration points.

Based on reading the actual files in the fleet repos, here is the current state of integration between the four core technologies:

## 1. **plato-torch** → **plato-ensign** → **fleet-simulator**
**Connection:** ✅ **Connected** (one-way data pipeline)  
**How:**  
- `plato-torch` generates training rooms (like `DeadbandRoom`) that accumulate interaction tiles.  
- `plato-ensign` exports room experience into LoRA adapters via `export_ensign()` function.  
- `fleet-simulator` consumes these LoRA adapters to run simulated agents, generating training data.  
**Evidence:**  
- `plato-torch/rooms/deadband_room.py` has `export_ensign()` method.  
- `fleet-simulator/README.md` states it uses "LoRA adapters from plato-ensign for agent behavior."  
**Integration Gap:** None — this is the primary training pipeline.

## 2. **plato-torch** → **holodeck-rust**
**Connection:** ⚠️ **Partial** (conceptual, not wired)  
**How:**  
- Both systems use "rooms" and "tiles" as training concepts.  
- `holodeck-rust` has sentiment-aware NPCs that could be trained from `plato-torch` room interactions.  
**Evidence:**  
- `holodeck-rust/src/npc/sentiment.rs` defines NPC emotion states.  
- `plato-torch/rooms/base_room.py` defines `Tile` class with `interaction_data`.  
**Integration Gap:** No direct API or data flow between them. NPCs in holodeck do not currently load tiles or LoRAs from plato-torch.

## 3. **fleet-simulator** → **holodeck-rust**
**Connection:** ❌ **Not Connected**  
**How:**  
- `fleet-simulator` produces simulated agent interactions.  
- `holodeck-rust` could use these as training data for NPC behavior.  
**Evidence:**  
- `fleet-simulator/output/` contains JSON logs of agent actions.  
- `holodeck-rust` reads from `data/scenarios/` but no import from simulator logs.  
**Integration Gap:** No ingestion path. Simulator output format does not match holodeck scenario format.

## 4. **plato-ensign** → **holodeck-rust**
**Connection:** ❌ **Not Connected**  
**How:**  
- `plato-ensign` LoRA adapters could give holodeck NPCs specialized behaviors.  
**Evidence:**  
- `plato-ensign/exports/` contains `.safetensors` files.  
- `holodeck-rust` uses Rust ML crates (`tch-rs`) but no LoRA loading in codebase.  
**Integration Gap:** No LoRA integration in holodeck NPC system.

## 5. **Ghost Tiles** (P0 knowledge) → **Any System**
**Connection:** ⚠️ **Conceptual Only**  
**How:**  
- Ghost tiles are "dead agents' lessons" mentioned in architecture.  
- Could be injected into `plato-torch` rooms or `holodeck-rust` NPC memory.  
**Evidence:**  
- `fleet-knowledge/DEADBAND-PROTOCOL.md` references ghost tiles as P0 knowledge.  
- No implementation found in any repo for loading/using ghost tiles.  
**Integration Gap:** No system currently implements ghost tile ingestion.

## 6. **cudaclaw** / **flux-runtime** → **Core Four**
**Connection:** ❌ **Not Connected**  
**How:**  
- `cudaclaw` is GPU-resident agent runtime.  
- `flux-runtime` is self-assembling bytecode VM.  
**Evidence:**  
- Both are listed in Key Technologies but no imports found in plato-torch, fleet-simulator, holodeck-rust, or plato-ensign.  
**Integration Gap:** These appear to be standalone components not yet integrated.
