# Weaver Integration Map — Verified Connections & Integration Gaps  
**Cycle:** 384  
**Phase:** 4 — Build & Test  
**Status:** Direct file inspection of fleet repositories. Focus on actual imports, configuration references, and shared data structures.

## 1. plato-torch → fleet-simulator
**Connection:** Strong, direct dependency.  
**Evidence:**  
- `plato-torch/requirements.txt` includes `fleet-simulator>=0.1.0`  
- `plato-torch/src/training/rooms/deadband_room.py` imports `from fleet_simulator import FleetSimulator`  
- `plato-torch/configs/room_presets.yaml` references `fleet_simulator_config_path`  
**Data Flow:** Room presets can launch fleet simulations to generate training episodes. The simulator provides environment states and agent interactions that become training tiles.  
**Integration Gap:** No observed bidirectional link — the simulator does not import plato-torch. The connection is one-way: plato-torch consumes simulator output.

## 2. plato-torch → plato-ensign
**Connection:** Direct export pipeline.  
**Evidence:**  
- `plato-torch/src/exports/ensign_exporter.py` exists and imports `plato_ensign`  
- `plato-torch/scripts/export_ensign.py` calls `EnsignExporter.save_lora_adapter()`  
- `plato-torch/README.md` states: "Train in rooms, export to ensigns."  
**Data Flow:** After a room accumulates enough interaction tiles, the ensign exporter can package the room's learned weights into a LoRA adapter.  
**Integration Gap:** The export appears to be a manual or script-triggered step, not an automatic continuous pipeline. No observed real-time streaming of tiles to ensign.

## 3. plato-torch → holodeck-rust
**Connection:** Weak, indirect.  
**Evidence:**  
- No Rust imports or Python bindings found in plato-torch source.  
- `holodeck-rust/README.md` mentions "sentiment-aware NPCs can be trained via PLATO rooms" but no explicit import.  
- `plato-torch/configs/room_presets.yaml` contains a preset named `holodeck_npc_training` but references a local dataset path, not a live holodeck connection.  
**Data Flow:** Presumed offline dataset exchange. The holodeck may generate logs that are formatted into tiles for plato-torch training.  
**Integration Gap:** No live API or socket connection observed. Integration appears to be file-based and manual.

## 4. fleet-simulator → holodeck-rust
**Connection:** None observed.  
**Evidence:**  
- `fleet-simulator/src/` contains no Rust files or bindings.  
- `holodeck-rust/Cargo.toml` lists no Python dependencies.  
- Both are separate simulation environments with no shared protocol references.  
**Integration Gap:** These appear to be parallel, independent simulators. No shared agent definitions or state exchange.

## 5. plato-ensign → holodeck-rust
**Connection:** None observed.  
**Evidence:**  
- `plato_ensign/` is a Python package for LoRA adapter creation.  
- `holodeck-rust` is a Rust MUD server.  
- No cross-language bindings or adapter-loading mechanisms found in holodeck source.  
**Integration Gap:** Ensign adapters are currently only applicable to torch models, not to Rust-based NPC AI.

## 6. fleet-simulator → plato-ensign
**Connection:** None observed.  
**Evidence:**  
- `fleet-simulator` does not import `plato_ensign`.  
- Simulator output is raw interaction logs, not trained model weights.  
**Integration Gap:** The simulator generates training data, not trained models. Ensign export requires a trained plato-torch room as input.

## 7. Ghost Tiles & GhostInjector
**Location:** `plato-torch/src/data/ghost_injector.py`  
**Connections:**  
- Imports `DeadbandRoom` from `plato-torch.src.training.rooms`.  
- Reads from `memory/` directory (dead agent logs).  
- Injects ghost tiles into active room training buffers.  
**Integration Gap:** GhostInjector is not wired into `holodeck-rust`. It currently only injects into plato-torch rooms. The `holodeck-rust` NPC sentiment system could potentially consume ghost tiles but no such link exists.

## 8. cudaclaw & flux-runtime
**Observation:** These are lower-level runtimes.  
- `cudaclaw` is a CUDA-based agent runtime. No direct imports found in the four inspected repos.  
- `flux-runtime` is a bytecode VM. Not referenced in plato-torch, fleet-simulator, holodeck-rust, or plato-ensign.  
**Integration Gap:** These appear to be infrastructure components not yet integrated into the high-level training/simulation stack.
